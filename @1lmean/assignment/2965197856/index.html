[4팀 한선민] Chapter 1-2. AI와 테스트를 활용한 안정적인 기능 개발

by 1lmean2025.10.31

과제 체크포인트

필수 스펙

  1. 반복 유형 선택
  • 일정 생성 또는 수정 시 반복 유형을 선택할 수 있다.
  • 반복 유형은 다음과 같다: 매일, 매주, 매월, 매년
    • 31일에 매월을 선택한다면 → 매월 마지막이 아닌, 31일에만 생성하세요.
    • 윤년 29일에 매년을 선택한다면 → 29일에만 생성하세요!
  • 반복일정은 일정 겹침을 고려하지 않는다.
  1. 반복 일정 표시
    • 캘린더 뷰에서 반복 일정을 아이콘을 넣어 구분하여 표시한다.
  2. 반복 종료
    • 반복 종료 조건을 지정할 수 있다.
    • 옵션: 특정 날짜까지
      • 예제 특성상, 2025-12-31까지 최대 일자를 만들어 주세요.
  3. 반복 일정 수정
    1. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      • 반복일정을 수정하면 단일 일정으로 변경됩니다.
      • 반복일정 아이콘도 사라집니다.
    2. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      • 이 경우 반복 일정은 유지됩니다.
      • 반복일정 아이콘도 유지됩니다.
  4. 반복 일정 삭제
    1. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      1. 해당 일정만 삭제합니다.
    2. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      1. 반복 일정의 모든 일정을 삭제할 수 있다.

기본 과제

공통 제출

  • 테스트를 잘 작성할 수 있는 규칙 명세
  • 명세에 있는 기능을 구현하기 위한 테스트를 모두 작성하고 올바르게 구현했는지
  • 명세에 있는 기능을 모두 올바르게 구현하고 잘 동작하는지

기본 과제(Easy)

  • AI 코드를 잘 작성하기 위해 추가로 작성했던 지침
  • 커밋별 올바르게 단계에 대한 작업
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

심화 과제

  • 모든 질문에 대해 꼼꼼하게 정리했는지

과제 셀프회고

롤백을 세 번 정도해서 백업하고 easy로 제출하는 과제인데, 다음주에는 리셋 증후군이 발동하지 않았으면 좋겠습니다.

기술적 성장

  • TDD 사이클 스스로가 orchestrator를 자처하여 사이클을 확실히 익힐 수 있었습니다.

  • 프롬프팅 기법

  • 각 에이전트의 프롬프트 초안을 입력/출력 예시와 명확한 지침 중심으로 구성했는데, 나중에서야 이 방식이 Few-shot 학습 기법에 해당한다는 것을 알게 되었습니다. Zero-shot의 개념은 이해하고 있으나, 실제로 어떤 상황에서 활용하는지에 대해서는 아직 명확히 파악하지 못했습니다.
  • 에이전트에 **페르소나(persona)**를 부여한 뒤, 해당 에이전트가 작성한 테스트 코드와 일반 테스트 코드를 GPT에게 비교하도록 했는데, 어떤 것이 페르소나 버전인지 따로 설명하지 않았음에도 GPT는 페르소나를 부여한 에이전트의 코드를 정확히 더 나은 테스트 코드라고 말해서 제법 흥미로웠습니다.
  • 이후 **Chain of Thought (CoT)**를 적용해 보다 유의미한 변화가 있을 것으로 기대했으나, 결과적으로 테스트 코드에 간단한 주석만 추가된 수준에 그쳤다. 이는 추가 개선이 필요 없다고 판단한 결과인지, 아니면 CoT가 의도한 대로 **사고 과정(log)**을 기록하지 못했기 때문인지는 확실하지 않다.
  • Cursor Rules .cursor/rules/ 내에서는 자동으로 문서를 전역 참조하지만 "rtl", "test"등의 단어가 존재하면 알아서 전역 참조를 해제하여 참조를 필수적으로 넣어줘야 한다는 잘못된 문서를 보고 테스트 관련 문서까지 해당 디렉토리에 넣고 에이전트에 일을 시켰었는데 그래서 에이전트의 성능에 방해가 되었던 것 같기도 합니다.

해당 디렉토리의 문서는 아래의 링크로부터 마크다운을 생성하여 주입하였습니다.

AI가 주입하기 좋은 컨텍스트는 어떤 포맷의 문서인지에 대한 ... 궁금증이 있습니다. 지피티도 에이전트를 수정하라고 할 때마다 다른 형식의 마크다운을 주어서 cot처럼 "여기다 이거 추가해" 했습니다.

코드 품질

  • 테스트 코드의 유의미성 확보 지난 과제에서는 testTimeout으로 인한 실패 사례가 빈번하여 통합 테스트를 완전히 마무리하지 못한 채 제출하였습니다. AI가 생성한 테스트 코드임에도 불구하고 이러한 문제가 반복적으로 발생하였고, 그럼에도 유의미한 테스트를 작성하기 위해 고민하며 프롬프트에서 세부적인 요구사항을 계속적으로 추가했던 것 같습니다.

과제 피드백

  • 초반에 각 에이전트의 역할 분리가 어디까지 되어야 하는건지에 대한 기준을 잡는게 어려웠습니다.
  • 이전 회사에서는 PRD 등 기획·명세 문서를 작성하거나 검토할 기회가 거의 없었기 때문에, 문서의 구조와 구성 체계를 이해하는 과정도 어려웠습니다.

다음 과제에서는 문서 자동화를 이해해서 문서를 읽고 쓰고 이해하지 않아도 되는 사람이 되었으면 좋겠습니다 ....

리뷰 받고 싶은 내용

  • TDD 과정에서 RED → GREEN으로 넘어가기 전, 테스트 코드의 타당성은 어떻게 판단할 수 있을까요! 기능 구현이 제대로 됐음에도 먼저 작성한 테스트 코드를 통과하지 못해서 테스트 코드를 수정해야 하는 경우가 많았는데, 이게 TDD의 원칙에 위반되는 일이라는 생각이 들어서 혼란스러웠습니다.
