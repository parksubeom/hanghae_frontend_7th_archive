과제 회고

Before 패키지에서 발견한 문제점

1. God Component 문제

  • ManagementPage 하나가 데이터 로딩, CRUD 로직, 통계 계산, 전체 UI 렌더링을 모두 담당
  • UI, 도메인 로직, 상태 관리가 한 파일에 섞여 있어 읽기/수정/테스트가 어려움
  • 재사용이 불가능한 구조

2. 스타일링 문제

  • 인라인 스타일 범람: 레이아웃, 카드, 탭 버튼, 통계 카드 등 거의 모든 UI가 style={{ ... }}로 작성
  • 하드코딩 색상: #1976d2, #e3f2fd, #e8f5e9 같은 값이 직접 박혀 있어 테마 변경/다크모드 대응 불가
  • 반복 패턴 미추출: 통계 카드 4개, 탭 구조 등이 복붙 패턴으로 반복

3. 폼 상태 관리 문제

  • formData를 useState<any>로 선언하고 필드 이름을 문자열로 직접 매칭
  • 타입 안전성 없음, 유효성 검증/에러 처리가 분산되어 있음
  • 서비스 호출 결과에 따라 Alert/Modal 상태를 직접 토글하는 등 비즈니스 로직과 뷰 상태가 강하게 결합

4. 도메인(user/post) 로직 혼재

  • entityType에 따라 user/post 처리 로직이 if/else로 곳곳에 분기
  • 컬럼 정의, 폼 필드, 통계 계산이 한 컴포넌트에 뒤섞여 있음

5. 매직 스트링 사용

  • 상태 값("active", "inactive", "published" 등)이 문자열로 곳곳에 흩어져 있음
  • enum/타입으로 관리되지 않아 오타/변경에 취약

개편 과정에서 집중한 부분

1. 컴포넌트 아키텍처 개편

기존 경험들이 UI·로직 분리를 명확히 요구하는 구조가 아니었기 때문에, 처음에는 파일 단위로 모듈화 기준을 잡기 어려웠습니다. hooks나 context 같은 역할이 명확한 파일들만 일단 분류하여 모듈화를 진행하였고, 나머지 컴포넌트들은 디렉토리에 대-애충 묶어놓고 AI에 아키텍처 개선안을 요청하여 진행하였습니다. 아키텍처라는게 팀 내 컨벤션의 영역이고 취향의 영역이라고도 생각하지만, 스스로 느끼기엔 취향도 경험에 기반한다고 생각해서 이렇게 작은 기능 단위일 때, 다른 분들의 레포를 보며 다양한 아키텍처 구조를 쌔비지하고 경험해보려 합니다!

2. Context + Hooks 패턴으로 상태 관리

처음에는 일단 Prop Drilling으로 해결 가능한 범위라고 판단 하였지만, 모듈화를 진행할수록 불필요한 props 전달이 계속 생겨 상태관리를 먼저 해야겠다는 생각이 들었습니다. 전역 상태 관리 라이브러리도 고려했으나, 이번 과제 규모에서는 React 기본 기능만으로 충분하다고 판단해 Context + Custom Hook 패턴을 사용했습니다. 현업에서 전역상태 관리 라이브러리로 때려 박아서 관리하다가 렌더링 속도가 너무 느려져 최적화에 애 먹었던 경험이 있었는데, 같은 팀 소은님이 현업에서도 context만으로 상태관리를 하셨다는 얘기를 듣고 이번 기회에 경험해보는 게 좋을 것 같다고 생각했습니다.

Before: Prop Drilling

<ManagementStats 
  entityType={entityType} 
  data={data} 
/>

After: Context + Hook

// ManagementStats 내부
const { entityType } = useManagementTab();
const { data } = useManagementData();

3. React Hook Form + Zod 폼 검증

현업에서는 훅폼만 사용해서 유효성 검증을 처리 했는데 (상당히 하기 싫어하는 파트이기도 했습니다 ㅎㅎ), 팀원분들의 Zod 예찬론을 듣고 써봐야겠다는 생각을 했습니다. 회사 다닐 때 알았더라면 보다 덜 예민스럽게 일했을 것 같았습니다.

const userSchema = z.object({
  username: z.string().min(2, "사용자명은 2자 이상"),
  email: z.string().email("유효한 이메일 주소를 입력하세요"),
  role: z.enum(["admin", "moderator", "user"]),
  status: z.enum(["active", "inactive", "suspended"]),
});

const { register, handleSubmit, formState: { errors } } = useForm({
  resolver: zodResolver(userSchema),
  mode: "onBlur",
});

4. 디자인 시스템 구축

디자이너와 협업해본 경험이 없어서 AI를 통해 적용해보았습니다. Radix Colors 기반으로 접근성이 보장되는 색상 토큰을 정의했고, TailwindCSS 4의 CSS Variables 기능을 조합하여 Light/Dark 테마를 모두 지원하도록 구성했다.

Radix Colors 기반 디자인 토큰

/* 접근성 검증된 색상 팔레트 */
--primary: var(--blue-9);
--destructive: var(--red-9);
--muted: var(--slate-3);
--border: var(--slate-6);

Q) Context vs Zustand: 현재 프로젝트 규모에서는 Context + Hooks로 충분했지만, 규모가 커지면 Zustand 같은 상태 관리 라이브러리 도입이 필수적일까요?

A) UI를 다룰 때에는 굳이 Zustand 같은걸 사용할 필요는 없다고 생각해요 ㅎㅎ 상태관리 라이브러리의 경우 "도메인 로직"을 다룰 때 더 빛을 발휘하니까요!

A) 지금 정도의 수준이 적당하다고 생각합니다 ㅎㅎ 아니면 Form 자체를 Compound Component 로 묶어서 사용한다거나!?

Q) 집계 로직과 useMemo 최적화 기준

통계 계산 로직은 useMemo로 메모이제이션할 수 있으나, 현재 데이터 크기와 렌더 빈도를 고려했을 때 과도한 최적화라고 판단해 이번 과제에서는 가독성을 우선했습니다. 실무에서는 이런 “가벼운 집계 로직”에 대해 어느 정도 수준부터 useMemo 같은 최적화를 적용하시는지, 그리고 팀 차원에서 사용하는 기준이나 룰이 있는지 궁금합니다.

A)

저희 팀은 "판단 기준"을 생각하는 것 자체가 커뮤니케이션 비용이라고 생각하여 그냥 값을 다룰 때에는 useMemo를 사용하는 편이에요.

다만, 제가 개인적으로 가지고 있는 기준은 결과값이 "원시타입"일 때에는 useMemo를 안 써도 되고, 객체나 배열 등의 값을 반환하는 경우에는 useMemo를 씌우는 편입니다.

이게 "연산 최적화" 때문이라기보단, useCallback이나 useEffect 등의 dependencies 에서 사용할 때 문제가 될 수 있다보니, 이런 기준을 적용하고 있어요!

Q) Form 모듈화와 useImperativeHandle 선택

Modal에 submit 버튼이 있고 Form이 필드를 가지고 있는 구조라 useImperativeHandle + forwardRef를 사용해 부모에서 submit을 트리거하는 패턴도 고려했습니다. 다만 프로젝트 규모 대비 복잡도가 높다고 판단해, handleSubmit을 부모(Modal)에서 주입하고 Form은 register, errors 등 입력 책임에만 집중하도록 구성했습니다.

<ManagementModal onConfirm={handleSubmit(onSubmit)}>
  <UserForm register={register} errors={errors} />
</ManagementModal>

이런 구조 선택이 적절했는지, 혹은 실무에서 모달 + 폼을 분리할 때 더 선호하시는 패턴(useImperativeHandle, 전역 폼 상태, 별도 FormContainer 등)이 있다면 조언을 듣고 싶습니다.

A) 이러한 방식도 좋고, 개인적으로는 이럴 때 Context를 사용하면 어떨까!? 라는 생각을 하는 편입니다 ㅎㅎ useImperativeHandle의 경우 생각보다 이 흐름을 이해하는게, 작성자가 아니면 어렵다고 많이 느꼈어요. 제가 어떤 의도를 가지고 만들었는지를 다른 사람이 이해하기가 쉽지 않달까... ㅠㅠ

여튼, 제일 직관적인 방식은 Context 라고 생각해요!
