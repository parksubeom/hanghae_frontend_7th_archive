과제 셀프회고

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

  1. FSD 폴더 구조에 대한 이해
  • 단순히 폴더 나누기가 아니라 데이터 흐름을 기준으로 책임을 분리하는 방식이라는 걸 이해함
  • 평일 Q&A에서 다른 분들도 같은 고민을 많이 하셨던 것 같은데, 초반에는 entities와 features의 구분을 명확히 하지 못해서 구조를 여러 번 수정했고, 그 과정에서 내가 어떤 기준으로 판단해야 하는지 스스로 정리하는 데 시간을 많이 썼다.
  1. hook을 동작단위로 분리하는 패턴
  • 이전에는 공통 ui별로 훅을 나눠 (useAlert, useModal 등) 모든 기능에 대해 같은 훅을 끌어다 썼던 것 같은데, 기존의 개발 스타일과는 다르게 동작단위로 hook을 분리하는 패턴이라 처음에는 적응이 어려웠음
  • 그러나 상태 + 행동을 하나의 인터페이스로 관리하니까 훨씬 읽기 쉽다고 느꼈음.

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

  1. FSD에 대한 이해
  • 스스로 데이터 흐름과 책임의 경계를 명확히 하기 위해 노력함.
  • 게시글 추가/수정 기능을 구현하면서 유효성 검사를 UI, feature, entity 중 어디에 두는 것이 맞는지에 대해 가장 많은 고민을 함
    • 처음에는 다이얼로그 내부(features/*/ui/)에서 검증을 처리하려 했지만, 비슷한 로직이 반복되는 것에 대한 문제를 느낌.
    • 반대로 공통의 훅 내부 (feature/*/model/)에 검증을 두는 경우, 해당 규칙이 사용자 액션에 종속된 로직이 맞는지에 대한 의문이 생김.
    • 최종적으로는 api호출을 entities/*/api/ 에 두는 기준과 동일하게 유효성 검증 자체는 도메인 규칙으로 두고, 사용자 액션에 종속됐다고 판단한 에러표출 부분만 ui로 표출함.
  • API 처리 역시 query/mutation 구분과 관계없이, “도메인에 속한 책임인가”라는 동일한 기준으로 entities 레이어에 배치함.
  • 이 기준을 통해 “어디서든 동일하게 지켜져야 하는 규칙”과 “어떤 타이밍에 어떻게 막을 것인가”를 분리할 수 있었고, 구조적으로도 entities와 features의 구분을 좀 더 명확히 할 수 있었음.
  1. useDialog 사용 방식과 책임 분리
  • 기존에는 useDialog, useAlert 같은 공통 훅을 전역적으로 재사용하는 방식에 익숙했기 때문에, 다이얼로그를 기능 단위로 소유하는 구조가 처음에는 다소 생소하게 느껴졌음.
  • 이번 과제에서는 다이얼로그를 단순 UI 요소가 아니라, 하나의 사용자 행동 흐름(생성, 수정, 상세 진입)에 속한 상태로 바라보려고 했음.
    • 게시글 생성/수정 → 각각의 feature에서 다이얼로그 제어
    • 게시글 상세 + 댓글 관련 액션 → post-detail 위젯 내부에서 상태를 통합 관리
  • 또한 JSX에서 조건문으로 다이얼로그를 제어하는 방식이 불편하다고 느꼈고, 이를 줄이기 위해 상태 + 행동을 하나의 훅(usePostDetail)으로 묶는 구조를 선택함
    • 이 방식 덕분에 상위 페이지는 "무엇을 보여줄지”만 결정하고 “어떻게 열리고 닫히는지”는 위젯/훅 내부로 숨길 수 있었음.

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

  1. (여전히) FSD 폴더 구조의 경계
  • 과제의 볼륨이 실무 프로젝트에 비해 작고, 혼자 작업하는 환경이었기 때문에 구조의 경계가 완벽히 확신이 서지 않더라도 구현을 진행할 수 있었던 것 같음.
  • 하지만 실제로 작업하면서도 이 로직이 entities인지, features인지, widgets로 끌어올려야 하는지에 대해 매번 판단이 필요했고, 이 기준을 세우는 데 가장 많은 시간을 사용함.
  • 특히 post-detail / comment 관련 로직을 어디까지 위젯에서 가져가야 하는지, 혹은 features 단위로 쪼개는 것이 맞는지에 대한 고민이 반복적으로 발생함.
  • 이번 과제에서는 “지금 이 기능이 사용자 행동 단위인가, 도메인 규칙인가”라는 개인적인 기준을 세워 정리했지만, 실무처럼 여러 명이 동시에 작업하는 환경에서는 이 기준을 팀 내에서도 사람마다의 느끼는 차이 때문에 정의하는 것에 꽤 많은 리소스가 필요할 것 같다고 생각함.
  1. hook 상태 개수 관리 문제
  • usePostDetail 훅을 구현하면서, 다이얼로그 상태 + 선택된 게시글 + 댓글 목록 + 댓글 생성/수정 상태까지 하나의 훅에 많은 상태가 모이게 되었고, 결과적으로 훅이 다시 비대해지고 있다는 느낌을 받음.
  • JSX에서 조건문을 줄이기 위해 상태와 행동을 훅으로 묶는 선택은 만족스러웠지만, 상태 개수가 늘어났을 때 어디까지를 하나의 훅으로 유지해야 하는지, 혹은 어느 시점에서 다시 쪼개는 것이 좋은지에 대한 기준은 아직 모호함.
  • 실제로 구현 과정에서도 댓글 관련 상태를 별도 훅으로 분리할지, post-detail 훅이 다이얼로그 제어까지 가져가는 것이 맞는지를 여러 번 고민했으나, 이번 과제에서는 “사용자 흐름 단위”라는 기준으로 하나로 유지하는 쪽을 선택함.
  • 다만 이 방식이 항상 최선인지, 혹은 규모가 커질수록 오히려 다른 형태의 복잡도를 만들지는 않을지에 대해서는 앞으로 실무 경험을 통해 더 검증이 필요하다고 느꼈음.

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

  • 현업에서 개발을 진행할 때, 초기 기획과 실제 구현 사이의 간극이 점점 벌어지면서 디렉토리 구조 역시 그 변화에 맞춰 정리되지 못한 채 누적된 경험이 많았음.
    • 그 결과, 명확한 기준 없이 파일이 추가되고 구조가 흐트러진 상태에서도 “일단 돌아가니까”라는 이유로 그대로 두고 개발을 이어간 적이 많았다고 느꼈음.
  • 이번 과제를 통해 FSD를 적용해보면서, 처음에는 구조가 복잡하고 추상화 단계가 많아 보인다는 인상을 받았고 오히려 응집도가 높은 것인지, 낮은 것인지 판단하기 어려운 순간들도 있었음.
    • 다만 구현을 진행하면서 점점 느낀 점은, FSD가 “처음부터 완벽한 구조를 만드는 방법”이라기보다는 변화가 생겼을 때도 책임의 경계를 다시 조정할 수 있게 도와주는 틀에 가깝다는 점이었음.
    • 앞으로는 새로운 프로젝트를 시작할 때부터 FSD를 그대로 정답처럼 적용하기보다는, 이번 과제에서 정리한 기준(도메인 규칙 / 사용자 행동 / 화면 구성)을 바탕으로 변화에 따라 구조를 조정할 수 있는 최소한의 기준선으로 활용해보고 싶음.
  • 또한 아직은 FSD 외에 더 단순하거나 직관적인 구조, 혹은 상황에 따라 더 적합한 아키텍처 패턴이 있을 수도 있다고 생각하기 때문에, 하나의 구조에 집착하기보다는
  • “왜 이 구조를 쓰는지 설명할 수 있는 설계”를 목표로 계속 탐색해보고 싶음.

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

  • 더티코드를 접했을 때 어떤 기분이었나요? ^^; 클린코드의 중요성, 읽기 좋은 코드란 무엇인지, 유지보수하기 쉬운 코드란 무엇인지에 대한 생각을 공유해주세요
    • 사실 회사 다니는 시절에 비슷한 맥락의 코드를 많이 봐왔는데요, 다시 마주하니 감회가 새로웠습니다. 당시에는 작성자에게 구조 개선을 요구하기 보다, 시간이 생기면 직접 고쳐쓰는 방식을 선택했던 것 같아요. (그렇다고 제 코드도 클린했다고 자신할 수는 없지만요…)
    • 그때는 "클린코드"라는 개념을 컨벤션 잘 지키고, 변수명을 깔끔하게 짓는 정도로만 생각했던 것 같은데, 과제를 하면서 읽기 좋은 코드란, 맥락 하나하나 추적하지 않아도, 작성자가 아닌 사람이 봐도 흐름을 이해하고 수정하기 쉬운 코드 라는 생각이 들었습니다.
    • 그래야 자연스럽게 유지보수도 쉬워진다는 점을 과제를 통해 더 체감했던 것 같아요.

    • 당시 개발하던 웹앱이 대시보드 성격이라 화면에 보여줘야 할 데이터가 많고, 서로 얽힌 상태도 너무 많았는데 "일단 기능개발 먼저"에 집중하다 보니, 자연스럽게 전역으로 쌓이던 상태였습니다.
    • 그러다 결국 렌더링 문제로 한 번 크게 구조를 손봐야 했던 적도 있는데, 지금 생각해보면 그때는 개념적인 로직이나 책임 분리에 관한 이해가 많이 부족했던 것 같습니다.
    • 디자인 패턴을 배우면서 순수함수로 로직을 분리하고, 컴포넌트를 역할 단위로 나누면서 경계를 명확하게 하는 것이 얼마나 중요한지 체감한 시간이었어요.
    • 그때 이런 개념들을 알고 조금만 덜 전역으로, 조금만 덜 엮어서 썼다면 유지보수 할 때 훨씬 덜 힘들지 않았을까 하는 ... 아쉬움이 남았습니다.

응집도 높이기: 서버상태관리, 폴더 구조

  • "이 코드는 대체 어디에 둬야 하지?"라고 고민했던 시간, FSD를 적용해보면서의 느낌, 나만의 구조를 만들어가는 과정, TanStack Query로 서버 상태를 분리하면서 느낀 해방감(?)등을 공유해주세요
    • 이번 과제를 하면서 가장 많이 했던 생각 중 하나가 **“이 코드는 대체 어디에 둬야 하지?”**였던 것 같습니다.
    • 기능을 추가할 때마다 entities / features / widgets 중 어디에 두는 게 맞는지 계속 헷갈렸고, 특히 유효성 검사나 데이터 가공 로직처럼 UI도 아니고, API도 아닌 코드들이 가장 고민이 됐던 것 같아요.
    • 처음에는 빠르게 구현하려고 페이지나 위젯 쪽에 로직을 몰아넣기도 했는데, 기능이 조금만 늘어나도 코드가 금방 복잡해지는 걸 느꼈습니다.
    • FSD를 적용해보면서 “이 로직은 도메인 규칙인가?”, “이건 사용자 행동에 종속된 흐름인가?”를 기준으로 코드를 하나씩 밀어내듯 정리하게 되었고, 그 과정 자체가 응집도를 높이는 연습이 됐다고 생각합니다.
    • 사실 지금도 FSD가 엄청 응집도를 높여주는지에 대한 말이 바로 체감이 되진 않는 것 같습니다. 오히려 폴더/레이어가 늘어나면서 코드가 흩어졌다는 생각이 드는 것 같기도 해요ㅠㅠ
    • 최소한 각 레이어가 왜 존재하는지, 그리고 이 코드가 여기 있는 이유를 설명할 수 있는 구조를 만들려고 노력했던 점이 이번 과제에서 가장 큰 수확이었던 것 같습니다.

리뷰 받고 싶은 내용이나 궁금한 것에 대한 질문

  • feature/model 훅의 책임 범위 usePostDetail, usePostCreate처럼 상태와 행동을 함께 가지는 훅을 작성하면서 편하긴 했지만 상태가 많아질수록 다시 복잡해지는 느낌도 받았습니다. 이 정도 복잡도에서 더 나은 분리 기준(훅 분리, 내부 reducer화 등)이 있는지 궁금합니다.

  • 유효성 검증 위치에 대한 판단 검증 로직을 entities(lib)로 두고, 에러 표현만 feature/ui에서 처리하는 구조를 선택했는데 실무 기준에서도 괜찮은 선택인지, 혹은 feature 단에서 검증을 끝내는 게 더 일반적인지 피드백을 받고 싶습니다.

  • 다이얼로그 제어 방식 JSX 조건부 렌더링을 줄이기 위해 다이얼로그 상태를 훅으로 감싸는 방식을 선택했는데, 이 방식이 오히려 추상화를 과하게 만든 건 아닌지도 궁금합니다.

과제 피드백

잘했어요 선민! 셀프회고를 읽으면서 FSD를 단순히 폴더 구조로 받아들이지 않고 "왜 이렇게 나눠야 하는지"를 끊임없이 질문한 흔적이 느껴졌습니다.

특히 유효성 검사를 어디에 둘지 고민하면서 "어디서든 동일하게 지켜져야 하는 규칙"과 "어떤 타이밍에 어떻게 막을 것인가"를 분리했다는 부분이 인상적이었어요. 이건 스스로 기준을 세우고 그 기준에 따라 일관되게 배치한 좋은 예시입니다. "왜 이 구조를 쓰는지 설명할 수 있는 설계를 목표로 하겠다"는 마무리도 정확히 맞는 방향이에요.

다만 선민이가 느낀 "폴더/레이어가 늘어나면서 코드가 흩어졌다는 느낌"은 자연스러운 감각입니다. FSD가 응집도를 높여준다기보다는, 규칙을 명확히 해서 나중에 찾기 쉽게 만들어주는 거예요. 인간은 거대한 프로젝트의 세부적인 것들을 다 외우지 못하지만 규칙은 잘 외우거든요. 그래서 핵심은 경계를 고민하는 게 아니라, 규칙을 세워서 그 규칙대로만 하면 분리가 다 되느냐 아니냐를 발견하는 겁니다.

Q) feature/model 훅의 책임 범위 - usePostDetail, usePostCreate처럼 상태와 행동을 함께 가지는 훅을 작성하면서 편하긴 했지만 상태가 많아질수록 다시 복잡해지는 느낌도 받았습니다. 이 정도 복잡도에서 더 나은 분리 기준이 있는지 궁금합니다.

훅이 비대해지는 건 자연스러운 현상이에요. 일단은 쪼개보세요. 쪼개놓고 나면 항상 비슷한 단위, 비슷한 모양들이 보이기 시작하거든요. 그게 패턴이에요. 패턴을 발견하는 게 먼저입니다. 그러다 보면 다시 적절한 단위로 합칠 수 있는 지점이 보이고, 그때 응집도를 높일 수 있어요. 매번 이렇게 분리를 하는데 안 한고 합쳐도 모두 일관되게 만들 수 있네? 그러면 여기까지 분리는 안해도 되겠구나 하면서 적절한 크기감을 찾아가세요. 일단 분리하는 법을 이해하되 최대한 덜 분리하는 식으로 만드는게 요령입니다.

Q) 유효성 검증 위치에 대한 판단 - 검증 로직을 entities(lib)로 두고, 에러 표현만 feature/ui에서 처리하는 구조를 선택했는데 실무 기준에서도 괜찮은 선택인지 궁금합니다.

좋은 선택이에요. 검증 규칙 자체는 도메인 로직이니까 entities에 두고, 언제 어떻게 보여줄지는 feature에서 결정하는 게 맞습니다. 중요한 건 이 기준을 프로젝트 전체에서 일관되게 적용하는 거예요.

Q) 다이얼로그 제어 방식 - JSX 조건부 렌더링을 줄이기 위해 다이얼로그 상태를 훅으로 감싸는 방식을 선택했는데, 이 방식이 오히려 추상화를 과하게 만든 건 아닌지도 궁금합니다.

과하지 않아요. 오히려 좋은 방향입니다. 다이얼로그 열림/닫힘이 여러 곳에서 제어되면 버그 추적이 어려워지거든요. 한 곳에서 관리하는 게 맞습니다.

3챕터 동안 수고했어요. 스스로 기준을 세우고 그 기준을 설명할 수 있게 된 것, 그게 이번 과제의 가장 큰 수확입니다!
