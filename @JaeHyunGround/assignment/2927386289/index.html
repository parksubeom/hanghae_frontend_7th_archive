질문

Q. handlersUtils에 남긴 질문에 답변해주세요. A. 각 테스트마다 독립적인 msw 서버를 생성해야겠다는 생각이 들었습니다. 하나의 msw 서버에서 여러 테스트가 진행된다면 의도치 않은 데이터 꼬임이 발생할 수 있다고 생각했습니다. handlers.ts 는 HTTP 메서드에 대한 기본 동작 방식을, handlersUtils.ts는 server.resetHandlers() 메서드로 기존 환경의 핸들러를 초기화 + 해당 환경에서 사용할 메서드를 정의하고, 클로저를 활용해 각 환경마다 독립적인 데이터를 가질 수 있도록 했습니다. 처음에는 감이 잡히지 않아 많이 헤맸는데, 아고라에 올라온 질문들과 학습 메이트분들께서 클로저의 개념을 사용해보면 좋겠다는 말씀을 듣고 방향성을 잡을 수 있었습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요. A. vitest의 beforeEach, afterEach, beforeAll, afterAll 메서드를 사용해 코드 중복을 줄임과 동시에 각 테스트의 데이터를 분리하여 독립적인 환경에서 테스트가 구동될 수 있도록 했습니다. handlersUtils.ts 파일에서 api 모킹 테스트 시, 독립적인 msw 서버를 가질 수 있도록 구성했습니다.

과제 셀프회고

이번 과제를 통해 테스트 코드를 처음 작성해보게 됐는데, 초반에는 테스트 코드는 어떻게 짜야 좋은 테스트 코드일까? 에 대한 고민이 많이 들었습니다. 테스트를 위한 input, output을 스스로 정의하는 것이 맞는걸까? 억지로 끼워맞추는 거 아닐까?, 이 값도 테스트 해야할 것 같고, 저 값도 테스트 해야할 것 같고..... 테스트 범위를 어떻게 적절하게 조절하지?, 중복 코드를 줄이기 위해선 어떻게 코드를 구성해야할까? 와 같은 고민들에 휩싸여 초반엔 정말 답답했습니다...

하지만 많은 분들께서 양질의 질문들을 아고라를 통해 공유해주셨고, 멘토님의 QnA 세션을 통해 다른 분들은 고민을 어떻게 해결하고 계시는지 많은 노하우를 얻을 수 있었습니다. 특히 테스트 코드를 명세서 관점에서 접근하기, 테스트 작성 프로세스는 [환경 구성 -> 검증 상황 발생 -> 그 상황 검증] 경우가 많다 내용이 저에게 도움이 많이 됐습니다.

아직까지 부족한 부분이 많지만, 과제 시작 전과 후를 비교했을 땐 조금은 성장했다는 것에 의의를 두려고 합니다 !! 과제의 의미와 목표를 체화하여 더욱 성장할 수 있도록 노력해야겠습니다🔥 (많은 분들의 내용 공유와 도움이 없었더라면 중간에 포기했을지도 몰라요... 다들 감사합니다 😊)

해당 테스트가 꼭 필요할까?, 어떤 테스트가 추가되면 좋을까? 에 대한 고민이 쉽지 않았습니다. 과제 후반부에는 테스트 필요 유무를 판단하기 보단 적혀있는 테스트를 통과하는 것에 집중하여 고민해보지 못한 것이 아쉬운 부분입니다...

기술적 성장

코드 품질

테스트 환경을 고정된 시간으로 두어 언제 테스트를 실행하든 실패하지 않도록 구성


it('달력에 1월 1일(신정)이 공휴일로 표시되는지 확인한다', async () => {
    vi.useFakeTimers();
    vi.setSystemTime(new Date('2025-01-01T11:00'));

    renderApp();

    const monthView = screen.getByTestId('month-view');
    expect(within(monthView).getByText('신정')).toBeInTheDocument();

    vi.useRealTimers();

    debug();
  });

와 같이 테스트를 돌릴 때마다 고정된 환경을 제공하고자 vi.useFakeTimers() 을 활용해 가짜 타이머를 설정했습니다. 이를 통해 항상 동일한 테스트 결과가 나올 수 있도록 코드를 작성했습니다.

(기존에는 캘린더의 < 아이콘으로 월을 바꾸는 방식으로 코드를 짰었는데 10월은 9번을 눌러야 1월이고 11월은 10번을 눌러야 1월이 되기 때문에 날짜가 변경되면 테스트가 실패했었습니다.)

중복 코드가 많다

테스트 코드를 작성하면서 매번 이벤트 생성 코드, render 등과 같은 환경 세팅 코드 등을 중복해서 사용한 경향이 있는 것 같습니다. beforeEach, afterEach 메서드를 사용해 중복 코드를 줄이고자 노력했지만 테스트 전체를 아우르는 데이터를 세팅 하는 것이 쉽지 않았습니다.

description이 명확하지 않다.

description을 보고 어떤 테스트에서 문제가 발생하였고 기대 결과값이 무엇인지 까지 파악할 수 있도록 작성하는 것이 좋은 description인 것 같다고 오프 코치님께서 말씀주셨는데, 제 테스트 코드의 description은 그렇지 않은 것 같아 아쉬움이 남습니다.

학습 효과 분석

테스트 코드의 필요성, 도입하였을 때 불러올 긍정 효과에 대해 저만의 생각을 정리해볼 수 있었습니다.

이전까진 테스트 코드 작성까지 하면 일이 2배로 늘어난다., 서비스 코드가 바뀌면 테스트 코드도 바꿔야 하니 효율성이 좋지 않다. 와 같은 생각을 가지고 있었는데, 해당 과제를 진행하면서 테스트 코드에 대한 입장이 바뀌게 되었습니다.

테스트 코드는 명세서 역할을 할 수 있다. 이야기를 듣고 규모가 큰 프로젝트나 프로젝트에 중간에 합류하게 되었을 때 잘 작성된 테스트 코드가 있다면 코드를 이해하는데 도움을 줄 수 있겠다는 생각을 할 수 있었고, 서비스 코드가 바뀌면 테스트 코드도 바꿔야 한다 ? 가 아닌 서비스 코드를 수정했을 때 테스트가 실패한다면 이는 명세가 깨진 것으로 볼 수 있고, 이는 서비스 에러를 초래할 수 있다는 점에서 테스트 코드의 중요성을 알 수 있었습니다.

또한, 테스트 코드 작성 방법에 대해서도 학습할 수 있었습니다. 테스트 환경 세팅 -> 검증하고 싶은 상황 발생 -> 결과 검증 이라는 큰 틀을 잡고나니 테스트 코드를 작성하는데 조금은 수월해진 것 같습니다.

좋은 테스트 코드 작성하기, 중복 코드 줄이기, 명확한 description을 작성하는 방법에 대해선 아직까지 부족하지만 앞으로 이어질 테스트 코드 과제를 통해 부족한 부분들을 채워가고 싶습니다.

과제 피드백

리뷰 받고 싶은 내용

1. 테스트 전체를 아우르는 데이터 세팅하기

테스트 코드를 작성하면서 데이터 세팅 부분을 beforeEach, afterEach 메서드로 분리해보려 했으나, 각 테스트마다 필요한 데이터의 형태가 조금씩 다른 부분이 있어 매 테스트마다 필요한 데이터를 하나하나 설정한 경우가 많았습니다. 따라서 중복 코드가 많아져 코드 가독성이 떨어지는 문제가 발생했습니다.

코치님은 필요한 데이터를 세팅하실 때 어떻게 구성하시는 지 노하우가 궁금합니다 :)

2. 현재 작성된 테스트 코드는 명세서의 역할을 잘 하고 있는가?

특히 통합 테스트 부분은 명세서의 역할을 한다기 보단 또 따른 하나의 구현체라는 생각이 많이 듭니다. 한편으론 ui 테스트를 위해선 추가되어야 할 코드가 많기 때문에 테스트 코드가 길어지는 건 어쩔 수 없는 부분이라는 생각도 듭니다.

3. hook 간 검증을 위해선?

medium.integration.spec.tsx에서 팀 회의'를 검색하면 해당 제목을 가진 일정이 리스트에 노출된다 테스트에 대해


it("'팀 회의'를 검색하면 해당 제목을 가진 일정이 리스트에 노출된다", async () => {
    const event1 = createEvent({
      id: '1',
      title: '축구',
      date: '2025-10-22',
      startTime: '12:00',
      endTime: '14:00',
      description: '축구 경기',
      location: '축구장',
      category: '개인',
    });
    const event2 = createEvent({
      id: '2',
      title: '팀 회의',
      date: '2025-10-24',
      startTime: '12:00',
      endTime: '14:00',
      description: '팀 정기 미팅',
      location: '회의실',
      category: '업무',
    });

    const events = [event1, event2];

    setupMockHandlerCreation(events);

    renderApp();

    const user = userEvent.setup();
    const { result: eventOperations } = renderHook(() => useEventOperations(false));
    const { result: search } = renderHook(() => useSearch(events, new Date('2025-10-22'), 'month'));

    // hook 검증
    await act(async () => eventOperations.current.fetchEvents());
    await act(async () => search.current.setSearchTerm('팀 회의'));

    expect(search.current.filteredEvents).toEqual([event2]);

    // ui 검증
    const searchBar = screen.getByLabelText('일정 검색');
    await user.clear(searchBar);
    await user.type(searchBar, '팀 회의');

    const searchResult = screen.getByTestId('event-list');
    expect(within(searchResult).getByText('팀 회의')).toBeInTheDocument();
  });

와 같이 코드를 기존에 작성을 했었는데, hook 간의 검증이라기 보단 하나의 테스트에서 2개의 hook이 각각 잘 동작하는 지 검증하는 코드라는 생각이 듭니다.

사실 hook 간의 검증을 위해선 코드를 어떻게 작성해야 할 지 감이 안오는데, hook 간의 검증을 위해선 어떤 부분을 고려하여 코드를 작성해야 하는지 코치님의 노하우가 궁금합니다.

이 외에도 부족한 부분, 개선할 점에 대해 많은 피드백 받고 싶습니다 💪

// to-be
