기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.

  1. 타입에 따른 캘린더 뷰 렌더링 (캘린더 UI - 주간, 월간)

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

팀 전략 회의 정리 노션 첨부합니다 :)

[ 내가 생각한 최적의 테스트 전략 ]

팀 전략 회의를 하기 전까진 통합 테스트로 기능 대부분을 커버할 수 있지 않을까? 라는 생각을 가지고 있었습니다. 하지만 회의 과정에서 여러 가지 기능이 한 페이지 안에서 일어나고 있고, 기능 추가 시 스낵바 + 캘린더 반영 등 여러 기능이 종속되어 복잡한 UI 적용 구조를 가지고 있다. 는 의견에 크게 공감했습니다. 저는 단순히 통합 테스트로 각 기능이 잘 동작하는 지 확인하면 충분하지 않을까? 라고 판단했지만, 제가 생각하지 못했던 부분 or 신경쓰지 못했던 부분을 팀원분들을 통해 알 수 있어 많은 인사이트를 얻어갈 수 있었습니다. 프로젝트를 바라보는 시각이나 팀원분들의 작업 방식 (프로젝트 시작 전 어떻게 계획을 하시는지 등등) 을 배울 수 있었습니다.

[ 합의한 최적의 테스트 전략 ]

  1. 핵심 사용자 플로우에 대한 E2E 테스트를 우선으로 작성한다.
  2. 통합 테스트로 부족한 기능 검증을 진행한다

로 결론을 내릴 수 있었습니다. 프로젝트 진행 상 단위 + 통합 테스트가 일정부분 작성이 되어 있었고, 배포를 앞둔 실무 상황이라고 가정했을 때 팀원들 모두 핵심 플로우에 대한 검증이 필요할 것 같다., 사용자 관점의 시나리오 검증이 중요하다 는 공통의 의견을 가지고 있었습니다. 따라서, 핵심 사용자 플로우에 대한 E2E 테스트를 우선으로 작성하되 부족한 기능 검증을 통합 테스트에서 진행한다 는 결론을 낼 수 있었습니다.

[ 추가한 테스트 ]

  1. 드래그 엔 드롭 E2E 테스트
  2. 날짜 선택 시 왼쪽 폼 반영 E2E + 통합 테스트

를 추가했습니다.

드래그 엔 드롭, 날짜 선택 시 왼쪽 폼 반영 이 신규 기능으로 추가되었기 때문에, 두 기능을 테스트 타겟으로 잡았습니다. D&D의 경우 브라우저 이벤트를 사용하기 때문에 통합 테스트 환경에서 모킹을 통해 테스트를 작성한다면 신뢰도가 낮아질 수 있다는 의견이 있었습니다. 따라서 D&D 기능은 E2E 테스트로 검증을 했고, 날짜 선택 시 왼쪽 폼 적용 기능의 경우에는 통합 + E2E 모두 작성하여 기능 검증과 실제 사용자 시나리오를 검증하고자 했습니다.

E2E 테스트 항목 D&D

  • 월간, 주간 D&D 성공 시 스낵바 출력 + 일정 수정
  • D&D 시 일정이 겹칠 경우 취소 or 계속 진행 을 눌렀을 때 시나리오
  • 반복 일정을 D&D 할 시 예 or 아니오 을 눌렀을 때 시나리오

날짜 선택

  • 월간, 주간 캘린더 날짜 선택 시 왼쪽 폼 날짜 반영 + 일정 생성 시나리오

통합 테스트 항목 날짜 선택

  • 날짜 클릭 시 폼 반영 여부
  • 일정 수정 중일 땐 날짜 클릭해도 적용 X

과제 셀프회고

테스트 코드를 손으로 다 작성해야 한다 vs 테스트 코드 작성은 AI에게 맡긴다

이번 과제를 진행하면서 들었던 생각 중 하나는 내가 직접 하나하나 테스트 코드를 치기엔 시간이 오래 걸릴 것 같고, AI에게 위임하자니 실력 향상 안됨 + 100% 신뢰가 가능할까? 입니다. 실력 향상 안됨은 현재 배우고 있는 입장이기 때문에 든 생각인 것 같기도 합니다. 테스트 코드 작성법(문법 + 메서드)을 전부 다 외워두는게 좋을까 ? 라고 생각해보면 뭔가 아닌거 같고, 그렇다고 AI에게 위임을 하자니 내가 검토는 하지만 테스트 코드를 작성할 때마다 매번 AI에게 시키는 것이 과연 바람직한 방식인지 고민이 많이 됩니다.

아마도 제가 생각하는 이상적인 개발자가 도움 안받고 내 머리만으로 코드를 뚝딱뚝딱 작성하는 개발자 라고 생각해서 위와 같은 고민이 드는 것 같습니다. 예전에는 구글링, 지금은 AI를 활용하여 많은 개발자들이 개발할 때 도움을 받곤 하는데 제가 너무 구시대적인 발상을 가지고 있는 것인지도 모르겠습니다. 욕심 때문일까요 ? 강박 때문일까요 ? 정답을 보고 코드를 작성하는 건 내 성장에 도움이 안돼. 내 머리로 직접 생각해서 직접 작성해야 진정한 내 것이야 라는 생각이 자꾸만 저 스스로를 압박하는 느낌입니다.

본론으로 돌아와 코치님은 테스트 코드를 손으로 다 작성해야 한다 vs 테스트 코드 작성은 AI에게 맡긴다 + 결과물 확인을 사람이 에 대해 어떻게 생각하시는 지 궁금합니다 !!

사고력 기르기

개발자에게 기록 + 사고력 기반의 프로젝트 파악 능력은 정말 중요하다는 것을 느낄 수 있었습니다. 과제 수행을 위한 방법을 찾고 있던 저와는 달리, 과제 시작 전 프로젝트 전체 구조를 파악하고 설계해나가는 팀원분들의 모습을 보면서 많이 배울 수 있었습니다. 과제 마무리에 급급하여 코치님께서 의도한 학습 목표에 부합하지 못하는 모습을 보이는 것 같은데, 마무리 보다는 배움에 초점을 맞춰 마음의 여유를 가지고 과제를 할 수 있도록 노력해야겠습니다...

(과제의 의도를 파악하기, 무작정 코드부터 치지말고 과제 내용 파악 + 진행 방향성 설계해보기, 머리로만 생각하지 말고 내용, 구조가 엉망이더라도 문서로 기록하기 등등..)

기술적 성장

테스트의 종류와 장단점 파악

프론트엔드 개발에서 사용되는 테스트의 종류에 대해서 배울 수 있었습니다. 어떤 테스트가 있고 어떤 상황에서 사용되는지 파악할 수 있었고, 각각의 장단점에 대해 생각해보면서 상황에 따른 테스트 전략 선택에 대해 고민해볼 수 있었습니다.

단위 테스트

  • 정의 : 함수나 메서드 등 작은 단위를 독립적으로 검증하는 테스트 (주어진 input에 대해 올바른 output을 내는지)
  • 장점 : 빠른 실행 속도
  • 단점 : 모듈 간 상호작용 or 전체 시스템에 대한 검증이 어려움

통합 테스트

  • 정의 : 여러 모듈, 컴포넌트가 올바르게 상호작용 하는지 확인, 핵심 비지니스 기능의 로직 검증
  • 장점 : 모듈 간 검증 가능
  • 단점 : 모킹에 의존하게 되면 신뢰도가 낮아질 수 있다

E2E 테스트

  • 정의 : 사용자 관점에서 시나리오 기반 전체 워크플로우를 검증
  • 장점 : 사용자 시나리오 기반 테스트이다 보니, 핵심 비지니스 기능에 대한 시각적 확인 가능
  • 단점 : 실제 API 사용을 위해선 백엔드와 협의가 필요, 테스트 실행 시간 증가

시각적 회귀 테스트

  • 정의 : 공통 컴포넌트 등의 UI의 변경 사항을 시각적으로 검증
  • 장점 : 직접 개발 환경을 실행하지 않아도 작업 내용을 실시간으로 확인 가능
  • 단점 : 비용과 관리에 대한 부담이 존재

스냅샷 테스트

  • 정의 : HTML DOM 구조를 직렬화하여 이전 스냅샷과 비교 검증
  • 장점 : 빠른 컴포넌트 구조 변경 사항 확인
  • 단점 : 의도된 변경 사항인지 매번 확인 필요, 스냅샷 업데이트 남발 가능성

무작정 테스트를 추가하는 것이 좋은게 아니라, 각 프로젝트의 특성과 상황을 분석하여 적합한 테스트 전략을 세우는 것이 중요하다는 것을 배울 수 있었습니다.

코드 품질

독립적인 스토리 제공

스토리북을 작성할 때 고민했던 부분은 하나의 스토리에서 스토리북 하단 옵션을 조절하며 UI를 확인하는 방식 vs 상황 별로 스토리를 생성해 UI를 확인하는 방식 였습니다.

전자의 경우, 개발자가 아닌 디자이너 혹은 기획자가 스토리북을 통해 작업물을 확인할 때 각각 옵션(props)에 대한 Description이 자세하게 적혀있지 않다면 스토리북을 제대로 활용하지 못할 수 있겠다는 생각이 들었습니다. (스토리북은 개발자만 사용하는 것이 아닌 팀 전체에 작업물을 공유하기 위해 사용하는 것으로 이해했기 때문입니다 !)

따라서, InputForm을

사진과 같이 타입 별, 에러 유무 등에 따른 스토리를 각각 생성하여 개발자가 아닌 팀원들도 UI를 쉽게 확인할 수 있도록 작성해봤습니다.

학습 효과 분석

  • 시각적 회귀 테스트 (storybook + chromatic) : 컴포넌트 변경 사항을 UI로 직접 검증
  • E2E 테스트 (playwright) : 핵심 기능에 대한 사용자 시나리오 기반 테스트

테스트 코드는 프로젝트의 구조와 핵심 기능을 파악할 수 있는 방법 중 하나라는 것을 다시 한 번 느낄 수 있었습니다.

  • 시각적 회귀 테스트 코드를 작성하면서 프로젝트에는 이런 컴포넌트들이 있고, 각 컴포넌트가 상황 별로 어떻게 출력되는지 다시 한 번 파악할 수 있었습니다.
  • E2E 테스트 코드를 작성하면서 프로젝트 핵심 기능은 무엇이고, 어떤 흐름으로 동작을 하는지 다시 한 번 파악할 수 있었습니다.

과제 피드백

좋은 점 : 팀 단위 전략 회의에서, 제가 미처 생각하지 못한 부분에 대해 다른 분들의 의견과 아이디어를 듣는 과정에서 많은 인사이트를 얻을 수 있었습니다. 어려운 점 : 항상 느끼는 것이지만, 특정 Tool을 사용하기 위한 환경 세팅이 쉽지 않은 것 같습니다.

리뷰 받고 싶은 내용

1. 통합 테스트와 E2E 테스트 구분

통합 테스트와 E2E 테스트의 목적이 큰 틀에서는 비슷하다고 생각했습니다.

사실 이건 성향에 가까운 것 같아요 ㅎㅎ 처음부터 큰 그림을 보는게 어려울 수 있달까... 제가 그런 사람이거든요. 무조건 다 파악하고 시작하기보단, 시작하면서 점진적으로 파악하는거죠.

그래서 나에게 맞는 방법을 찾으면 된다고 생각해요!

Q1) 통합 테스트와 E2E 테스트 구분 통합 테스트와 E2E 테스트의 목적이 큰 틀에서는 비슷하다고 생각했습니다.

  • 통합 테스트 : 핵심 비지니스 기능에 대한 코드 로직 검증
  • E2E 테스트 : 핵심 비지니스 기능에 대한 사용자 관점 시나리오 검증

핵심 비지니스 기능에 대해서 통합 테스트와 E2E 테스트를 모두 작성한다면 중복 검증이 되어 비효율적일까요? 아니면

  • 통합 테스트 : 로직 검증
  • E2E : 시나리오 검증

와 같이 검증 포인트를 다르게 두고 접근한다면 더 신뢰성 높은 테스트를 진행할 수 있는 것일까요?

A1) 이건 선택의 문제라고 생각해요 ㅎㅎ 뭐가 맞다라기보단...? 통합 테스트의 경우 실행속도가 빠르기 때문에 모든 단계의 ci에서 실행해도 무방한데 e2e 테스트의 경우 지금은 작은 프로젝트라서 실행 시간이 오래 걸리지 않지만, 실제 프로젝트에서 e2e를 실행한다고 치면 굉장히 오래걸릴 수 있답니다. 그래서 e2e는 배포 직전 처럼 특정 시점에 실행하여 문제를 발견하도록 할 수 있어요.

즉, 피드백 타이밍이 다르다는거죠. 대신 테스트 작성 난이도는 통합 테스트가 더 높아요.

시간적 여유가 있다면 e2e와 통합테스트 모두 작성하면 당연히 좋겠지만..!? 그게 어렵다면 무언가 하나는 포기해야겠죠 ㅎㅎ

저는 특히 이게 프로젝트의 성격에 따라 갈라진다고 생각하는데, 가령 제가 이전 회사에서 만들었던 "웹 포털 서비스"의 경우 통합서비스가 큰 의미가 없어요. 대신 e2e를 통해 한 번에 검증하는게 제일 효과적이었죠. 단위테스트도 큰 의미가 없었어요.

사용자의 인터랙션을 기반으로 진행되는 서비스가 아니라, 거의 단뱡항으로 데이터를 공급하여 보여주는 방식이기 때문이죠.

그런데 쇼핑몰 같은 프로젝트에서는 통합테스트가 유의미할 수 있고 지금 저희 팀에서 만들어가는 노코드 빌더의 경우 통합테스트나 e2e를 작성하는 난이도가 너무 높아서 단위 테스트를 통해 단계별 검증을 하는게 제일 효과적입니다.

그래서 어떤 프로젝트의 어떤 기능을 검증하냐에 따라 다르다고 생각해요!

추가로 코치님께서 스토리북을 작성하실 땐 어떤 기준을 가지고 작성하시는지 노하우를 얻고 싶습니다 !!

  • 사이트를 만들고 템플릿을 선택하고
