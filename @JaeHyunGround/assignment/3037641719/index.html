아토믹 디자인이란?

잼민이 결과물

웹사이트나 앱을 만들 때 페이지 단위로 디자인 하는 것이 아니라 가장 작은 부품(원자)부터 시작해서 점점 큰 단위로 조립해 나가는 디자인 방법론

원자 (Atoms)

더 이상 쪼갤 수 없는 가장 기초적인 블록

  • [검색] 버튼 하나, 검색어 입력창 하나
분자 (Molecules)

두개 이상의 원자가 결합하여 하나의 기능을 수행하는 그룹

  • 검색어 입력창 + [검색] 버튼 ⇒ 검색창
유기체 (Organisms)

원자와 분자들이 모여 만들어진 복잡하고 독립적인 인터페이스 영역

  • 로고(원자) + 네비게이션 메뉴(분자) + 검색창(분자) = 헤더(Header)
템플릿 (Templates)

유기체들을 배치하여 페이지의 레이아웃을 잡은 상태

  • 헤더는 위에, 상품 리스트는 중간에, 푸터는 아래에 배치된 빈 화면 구조.
페이지 (Pages)

템플릿에 실제 컨텐츠를 채워 넣은 최종 결과물

  • 사용자가 보게 될 화면

Atomic Design 개념 이해 후 개편하는 것이 아니라 Atomic Design의 문제점을 파악하고 폴더 구조를 개편해보는 것이 과제였습니다....! 하마터면 잘못된 방향으로 과제를 진행할 뻔했습니다. 제가 생각한 Atomic Design의 가장 큰 문제는 사람마다 분리 기준이 너무나 다르다는 것입니다. Atomic Design 분류 기준을 맞추기 위한 팀원들과의 커뮤니케이션 비용이 너무나 크다고 생각했습니다. 또한 컴포넌트를 생성할 때마다 이건 atoms인지 molecules인지 고민을 해야하는 것이 비효율적이리고 느껴졌습니다.

디자인 시스템이란, 서비스를 개발하는 데 필요한 모든 요소(원칙, 컴포넌트, 가이드라인) 를 통합적으로 모아 놓은 집합체

Button 컴포넌트를 리팩토링하면서 들었던 의문점

  • pagination 과 같이 모든 버튼 타입에 대한 variant를 선언하는게 맞는 방식일까?
    • pagination은 페이지네이션 하단바 한 곳에서만 쓰는데 인라인으로 넘겨주는게 더 나을까?
    • 그러면 관리 방식이 투트랙이 되니깐 오히려 더 유지보수하기 힘들어질까? cva 안에 넣는게 맞는거 같기도 하고…?
  • Button 컴포넌트를 사용한 버튼만 리팩토링하는 건가?
    • 일반 button 태그를 사용한 버튼은 뭔가 스팩이 중복되지 않아서 공통으로 빼기가 애매한데...
  • 한 군데에서만 쓰이는 스타일 (.alert-close)
    • 이것도 button 공통 컴포넌트 cva 안에서 다뤄야할 지 아님 인라인으로 넘기게 할지

위 고민들을 공통 Button 컴포넌트를 생성 후 CloseButton 등 기능별 버튼 컴포넌트를 생성하는 방식으로 해결해보고자 합니다!

  • 공통 Button 컴포넌트는 컬러, 사이즈와 같은 값을 주로 다룰 수 있도록 !

Before 패키지에서 발견한 문제점

1. 스타일링 혼용

인라인 스타일링, css class 스타일링 방식이 혼용되어 있는 문제점이 있었습니다. => 문제를 해결하기 위해 tailwindcss 기반 스타일링 기법으로 로직을 수정하였고, 반복되는 색상은 토큰화하여 색상 코드를 하드 코딩하는 경우를 줄였습니다. cva 라이브러리를 사용하여 사용단에서 variants 만으로 정해진 스타일을 불러올 수 있도록 하여 컴포넌트 사용성을 높힐 수 있었습니다.


import * as React from "react";
import { Slot } from "@radix-ui/react-slot";
import { cva, type VariantProps } from "class-variance-authority";

import { cn } from "@/lib/utils";

const buttonVariants = cva(
  "inline-block leading-normal border-solid font-sans rounded-[3px] cursor-pointer border whitespace-nowrap disabled:cursor-not-allowed disabled:opacity-60",
  {
    variants: {
      variant: {
        none: "bg-transparent border-none",
        blue: "bg-blue-200 text-white border-blue-300 hover:bg-blue-300",
        gray: "bg-gray-100 text-gray-500 border-gray-300 hover:bg-gray-200",
        red: "bg-red-100 text-white border-red-200 hover:bg-red-200",
        green: "bg-green-100 text-white border-green-200 hover:bg-green-200",
      },
      size: {
        none: "",
        sm: "py-[6px] px-[12px] text-[13px]",
        md: "py-[10px] px-[20px] text-[14px]",
        lg: "py-[12px] px-[24px] text-[15px]",
        fullwidth: "w-full py-[10px] px-[20px] text-[14px]",
      },
    },
    defaultVariants: {
      variant: "blue",
      size: "md",
    },
  }
);

export function Button({
  className,
  variant,
  size,
  asChild = false,
  type = "button",
  ...props
}: React.ComponentProps<"button"> &
  VariantProps<typeof buttonVariants> & {
    asChild?: boolean;
    type?: "button" | "submit" | "reset";
  }) {
  const Comp = asChild ? Slot : "button";

  return (
    <Comp
      data-slot="button"
      type={type}
      className={cn(buttonVariants({ variant, size, className }))}
      {...props}
    />
  );
}


// 사용단

<Button
    variant="blue"
    size="md"
    onClick={() => setIsCreateModalOpen(true)}
  >
    새로 만들기
</Button>

CloseButton, PaginationButton 등 특정 기능을 하는 버튼의 경우 새롭게 컴포넌트를 생성해서 공통 Button 컴포넌트를 래핑하여 사용하고자 했습니다.


import { cn } from "@/lib/utils";
import { Button } from "./button";

export const CloseButton = ({ ...props }: React.ComponentProps<"button">) => {
  return (
    <Button
      variant="none"
      size="none"
      className={cn([
        "text-[24px] w-[24px] h-[24px] shrink-0 leading-1 outline-none focus:outline-none rounded-full",
        props.className,
      ])}
      {...props}
    >
      {props.children || "×"}
    </Button>
  );
};

2. UI 로직과 비즈니스 로직 혼용

컴포넌트 내에 비즈니스 로직과 UI 로직이 혼용되어 있어 확장성이 떨어졌고 비즈니스 로직이 변경될 때 UI 로직도 수정해야 하는 잠재적인 문제가 있었습니다. => Headless 패턴을 참고하여 비즈니스 로직을 Custom hook으로 분리하고자 했고, 컴포넌트는 props로 넘어오는 값을 활용해 출력하는 일에만 집중할 수 있도록 로직을 구성하고자 했습니다.

개편 과정에서 집중한 부분

1. UI 컴포넌트와 비즈니스 로직 분리

UI 컴포넌트와 비즈니스 로직을 분리하는 것에 집중하고자 했습니다. Before 패키지의 컴포넌트는 상위 컴포넌트로 부터 도메인 관련 props를 받아와 컴포넌트 내부에서 어떤 요소를 렌더링 할 지 판별하는 로직이 포함되어 있었습니다. 이로 인해 발생할 수 있는 문제점은 확장성이 떨어진다, 비즈니스 로직이 추가될 때마다 컴포넌트 내부 로직도 수정을 해줘야 한다 (렌더링을 위한 if문 분기 처리 등) 등이 있습니다.

문제점을 해결하기 위해 비즈니스 로직을 커스텀 hook으로 분리하여 UI와 로직을 분리하고자 했고 컴포넌트에 출력해야 할 데이터 등을 넘겨줌으로써 컴포넌트는 UI 출력만 담당할 수 있도록 구성하고자 했습니다.

2. 일관된 스타일 시스템 구축

자주 사용되는 색상 컬러를 디자인 토큰으로 분리하여 인라인 스타일링 + 하드 코딩된 색상 코드 문제를 해결하고자 했습니다. 또한 TailwindCSS로 스타일을 재구성하여 혼용된 스타일 문법 문제를 해결하고자 했습니다.

패딩 간격, 폰트 사이즈 등 더 세부적인 디자인 토큰은 나누지 못했지만 자주 사용되는 사이즈 값들을 디자인 토큰화하여 사용한다면 더욱 견고한 디자인 시스템 구축을 할 수 있을 것이라 생각합니다.

사용한 기술 스택 경험

1. tailwindCSS + CVA

: 유틸리티 퍼스트의 장점을 느낄 수 있었습니다. css 파일을 별도로 생성하지 않아도 된다는 점, cva를 통해 공통 스타일 코드를 줄일 수 있다는 점, 사용단에서 variants만으로 컴포넌트의 스타일을 정할 수 있는 편리함 등이 장점이라고 생각합니다.

단점은 초반에 tailwindCSS 문법을 익히는데 시간이 걸린다 정도인 것 같습니다. 하지만 문법에 익숙해진다면 강력한 스타일링 도구가 될 수 있다고 생각합니다.

2. shadCN

: Headless UI를 과제를 통해 처음 사용해봤는데, 사용 경험이 좋았습니다. 컴포넌트에서 대부분의 기능을 제공하기 때문에 사용자는 스타일링에만 집중할 수 있어 좋았습니다. 또한 shadCN은 기본 스타일링도 제공했기에 Headless UI의 단점인 스타일링을 제로에서 부터 시작해야 한다는 단점을 상쇄시켜 준다고 느껴졌습니다.

3. React-hook-form + zod

: React-hook-form의 경우, form을 다룰 때 자주 사용해봤지만 zod의 경우 사용 경험이 많지 않았습니다. React-hook-form + zod 조합이 굉장히 좋다는 이야기를 많이 들어왔는데, 사용해보니 정말 궁합이 좋았습니다. 유효성 검사 로직을 한 곳에 모아 볼 수 있어 유지보수 할 때도 편해보였습니다.

어려웠던 점과 해결 방법

첫번째는 UI 로직과 비즈니스 로직을 분리하는 것이 쉽지 않았습니다. 특히 Table, Form 부분은 복잡하고 어려워서 AI로 선구현 후 코드를 이해하는 방식을 채택했습니다.

제가 생각한 UI 로직과 비즈니스 로직 분리의 핵심은 컴포넌트 내에서 렌더링 요소를 판별하지 않는다. 입니다. 컴포넌트 내에서 렌더링 할 요소를 판별해버리게 된다면 특정 기능에서만 사용 가능한 확장성이 떨어지는 컴포넌트가 될 수 있다고 생각했습니다. 따라서 컴포넌트는 props로 넘어오는 값들을 그대로 출력하도록 구성하고 데이터 세팅, 핸들러 함수 등은 상위 영역(페이지 단)에서 관리를 하는 것이 중요한 포인트라고 느껴졌습니다.

어려움을 극복한 방법은, 제가 분리할 수 있을 것 같은 컴포넌트는 커스텀 hook을 생성해 최대한 분리해보고자 했고, 막히는 부분이 있다면 AI를 활용해 어떤 방식으로 리팩토링을 진행할 수 있는지 정리 문서를 생성한 뒤 해당 문서를 바탕으로 리팩토링 해보고자 했습니다. 그럼에도 막히는 부분이 있다면 AI를 활용해 단계적으로 리팩토링을 진행하고 작성된 코드를 이해해보는 방향으로 어려움을 해결하고자 했습니다. 리팩토링 방법의 다양한 방향성에 대해 알 수 있었고, AI로 작성된 코드는 제가 다시 작성해보며 방법론을 체화하고자 합니다 :)

두번째는 디자인 토큰 정의 기준 정하기입니다. 처음에는 프로젝트 내에 사용되고 있는 컬러, 사이즈를 전부 분석하여 반복적으로 쓰이고 있는 값들을 토큰화 해야 하나? 하는 생각이 들어 프로젝트 코드 내에서 사용되고 있는 값들을 분석했었습니다. 사용하고 있는 컬러, 사이즈가 제각각이어서 디자인 토큰화 구분에 대한 명확한 기준을 세울 수가 없어 혼란스럽던 찰나에, 화요일 평일 QnA를 통해 어느정도 감을 잡을 수 있었습니다.

디자인 시안을 기반으로 우선 디자인 토큰을 정의하고 이를 활용해 스타일을 적용하는 것이 일반적인 워크플로우라고 이해했습니다. 과제처럼 리팩토링이 필요한 상황에서는 기존 스타일을 그대로 유지하기보다 토큰을 재정의해 일관된 스타일로 재구성하는 방향을 설정하면서, 제가 가진 고민을 정리할 수 있었습니다.

세번째는 컴포넌트 생성 기준입니다.


<button onClick={onClose} className="alert-close">
    ×
</button>

버튼이 있을 때, alert-close 클래스는 저 버튼 한 곳에서만 쓰이는 상황에서 alert-close 스타일을 공통 버튼 컴포넌트의 variants로 뺄지 아니면 AlertCloseButton 이라는 컴포넌트를 만들어야할지 고민이 고민이 됐습니다.

이는 모각코 같이 했던 분들, 코치님의 답변을 들으면서 정답이 없는 고민이면서 저만의 컴포넌트 분리 기준을 세워볼 수 있었습니다.

저는 공통 버튼 컴포넌트를 생성한 뒤, 특정 기능 or 추가 디자인이 포함되어야 하는 버튼의 경우 컴포넌트를 새로 생성해 공통 버튼 컴포넌트를 래핑해서 사용하는 방식을 채택했습니다. 채택한 이유는 공통 버튼 컴포넌트 내부에 닫기 기능과 같은 특정 기능에 대한 옵션을 추가하게 된다면 해당 버튼을 공통으로 쓰일 수 있을까? 하는 생각이 들었습니다. 따라서 공통 버튼 컴포넌트는 정말 어디든지 사용할 수 있는 형태로 두고 특정 기능이나 디자인이 추가되어야 하는 경우는 새로 컴포넌트를 생성함으로써 공통 컴포넌트의 재사용성을 높이고자 했습니다.

지금 폴더 구조는 /components/ui 경로에 모든 컴포넌트를 담고 있는데, 추후에는 공통 컴포넌트와 특정 기능이 포함된 컴포넌트가 존재하는 경우에 Button 과 같은 공통 컴포넌트 단위로 폴더를 구성하고, 해당 컴포넌트와 관련된 요소들을 폴더 단위로 묶어 관리하는 형태로 마이그레이션하고자 합니다.

리뷰받고 싶거나 질문하고 싶은 내용

디자인 토큰의 레이어 나누기

발제 자료에서는 디자인 토큰을 Primitive / Semantic / Component 레벨로 나누어 관리하는 방식이 소개되고 있는데, 저는 디자인 토큰을 레이어화 해서 관리해야 할까? 라는 의문이 들었습니다. 색상, 사이즈와 같은 단위의 Primitive 디자인 토큰만으로도 저는 충분히 프로젝트를 관리할 수 있을 것 같다는 생각이 드는데, 코치님의 의견이 궁금합니다 ! 또한 코치님은 디자인 토큰을 어떻게 관리하시는지도 궁금해요 !

디자인 토큰을 레이어화 해서 관리해야 할까? 라는 의문이 들었던 이유는 예를 들어


// Primitive
--blue-200: #1976d2;

// Semantic
--color-primary: var(--blue-200);

// Component
--color-btn-primary: var(--color-primary);

라는 레이어 토큰이 있을 때 Primitive - Semantic 단계까진 이해할 수 있지만 Component 단계까지 쪼갠다면 너무 과하지 않을까? 라는 생각이 들었습니다. 컴포넌트 사용단에서 "여기엔 primary 색상이 들어가야해!" 라고 한다면 bg-primary , bg-blue-200 으로 하면 충분하지 않을까? 하는 생각이 자꾸만 들더라구요...!!

Primitive / Semantic / Component 레벨로 나누어 디자인 토큰을 관리했을 때 이점이 무엇인지 아직까진 감이 잘 안잡히는 것 같습니다. (디자인 시스템을 만들거나 사용해본 적이 없고, 규모가 큰 프로젝트를 해보지 못해서 생기는 의문점이기도 한 것 같아요)

Q) 디자인 토큰의 레이어 나누기 발제 자료에서는 디자인 토큰을 Primitive / Semantic / Component 레벨로 나누어 관리하는 방식이 소개되고 있는데, 저는 디자인 토큰을 레이어화 해서 관리해야 할까? 라는 의문이 들었습니다. 색상, 사이즈와 같은 단위의 Primitive 디자인 토큰만으로도 저는 충분히 프로젝트를 관리할 수 있을 것 같다는 생각이 드는데, 코치님의 의견이 궁금합니다 ! 또한 코치님은 디자인 토큰을 어떻게 관리하시는지도 궁금해요 !

A) 저희 팀에서도 3단계의 레이어를 구분해서 사용하곤 한답니다!

범용적으로 활용할 수 있는 기본적인 디자인 토큰을 먼저 정의한 다음에, 프로젝트 단위의 메인컬러/서브컬러 혹은 무언가 의미가 있는 토큰을 분리해서 관리하는거죠 마지막으로 컴포넌트 레이어에서 활용할 수 있는 토큰을 정의해놓는 편이에요.

만약 "회사의 브랜딩 컬러"가 변경되었을 때 2차 레이어에 있는 토큰만 변경하면 일괄 수정 될 수 있어요 ㅎㅎ

컴포넌트로 단위로 분리하는 이유는, 각 컴포넌트마다 독립적인 색상을 사용할 수 있기 때문이라고 생각해요. 그치만 말씀해주신 것 처럼 컴포넌트 레이어의 토큰을 무조건 분리할 필요는 없다고 생각합니다!
