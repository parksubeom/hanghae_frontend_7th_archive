과제를 하면서 내가 알게된 점, 좋았던 점은 무엇인가요?

1. 여러 코드들이 응집되어 있는 코드를 분리하면서 역할 & 관심사에 따라 코드를 분리하는 것에 대한 중요성을 체감했습니다.

1100줄 가량의 App.tsx 코드를 처음 봤을 때 막막하다는 생각이 가장 먼저 들었습니다. state, 함수, UI 로직이 한 파일에 응집 되어있어 함수의 역할이나 UI가 어떤 state를 가져다 사용하는지 파악이 어려웠습니다. 따라서 크게 Header, Notification, Page 단 3 덩어리로 우선 코드를 분리한 뒤, Page 단 하위 컴포넌트를 나눠가는 방식으로 과제를 수행했습니다.

해당 프로젝트의 핵심 도메인은 cart, product, 'coupon, 'notification 4가지라고 판단했고 도메인 별 로직을 최대한 분리 하면서 코드를 작성하고자 했습니다. 도메인 별 state, 액션 함수는 커스텀 hook으로 분리하고 계산 함수는 유틸 함수로 분리함으로써 함수의 용도를 명확히 구분 하고자 했습니다.

각 도메인별로 비즈니스 로직과 UI 로직을 분리한 결과, 로직 구조 파악이 한층 명확해졌습니다. 또한 특정 엔티티 기능을 수정할 때 해당 엔티티 영역의 파일만 확인하면 되어, 유지보수 효율성이 크게 향상될 수 있다고 생각했습니다.

2. 전역 상태 관리를 도입하는 이유에 대해서 생각해볼 수 있었습니다.

저는 과제를 하면서도 전역 상태 관리를 도입하는 이유가 props drilling을 없애기 위함이라고 생각했습니다. 과제를 마무리 한 후 다른 분들의 PR을 보면서 전역 상태 관리를 사용하는 진짜 이유는 props drilling 해결이 아닌 계층 분리 라는 내용을 보며 전역 상태 관리를 도입하는 이유가 뭘까? 에 대해서 고민해보려 했습니다.

UI 컴포넌트가 도메인 props에 의존하는 것이 아닌, 전역 store에서 상태를 가져옴으로써 UI와 도메인의 계층을 분리하는 것인가? 하는 생각을 할 수 있었습니다. 또한 서버 상태와 클라이언트 상태를 구분하여 관리하기 위해 전역 상태 관리를 도입한다는 의견도 보며 전역 상태 관리를 도입하는 이유가 무엇일지에 대해서 여러 고민을 해볼 수 있었습니다.

아쉬운 부분, 개선 Point

1. useProductForm, useCouponForm을 굳이 hook으로 분리할 필요가 있었을까 ?

이번 과제에서 useProductForm, useCouponForm 커스텀 hook을 분리했는데, 지금 생각해보면 굳이 나눌 필요가 있었을까? 하는 생각이 들었습니다. 커스텀 hook으로 분리한 이유는 컴포넌트 안에 state, 핸들러 함수를 분리하기 위함이었습니다. 하지만 form 내용을 form 이외에 다른 곳에서 사용하지는 않기 때문에 hook으로 분리하는게 과도한 추상화였나? 하는 생각이 들기도 합니다.

만약 다시 하라고 하면 분리하지 않았을 것 같네요...!! form 로직의 관리 포인트가 2개로 늘어나는 느낌이라 로직 수정을 해야할 때 번거로울 것 같기도 하고 form 관련 코드는 form 컴포넌트 안에서만 쓰이니깐 !

2. 파일 네이밍 통일성 유지

과제가 끝난 후 결과물을 봤을 때 파일 네이밍의 통일성이 없다는 느낌을 받았습니다. 구조를 미리 생각하고 한 것이 아닌 그때그때 필요할 때마다 파일을 분리하다 보니 생긴 문제점인 것 같습니다.

예로, AdminCouponView - ProductCard 과 같이 데이터 뷰를 담당하는 컴포넌트의 파일명이 어떤 건 ~~View, 어떤 건 ~~Card 로 통일성을 지키지 못한 것이 아쉽게 느껴집니다.

어떻게 폴더 구조를 짤 지, 어떤 항목을 나눠야할 지를 미리 고민해봤다면 해결할 수 있지 않았을까? 하는 마음에 개선 Point로 선정해봤습니다.

명확한 구분 기준 세우기

로직을 분리할 때 명확한 구분 기준을 세워보고자 했습니다. 제가 세웠던 기준은 아래와 같습니다.

  • state와 state를 조작하는 함수는 custom hook으로 분리
    • custom hook으로 분리하는 조건 : 여러 컴포넌트에서 사용되는 경우 (state가 여러 곳에서 공유), 도메인 데이터를 다루는 경우
  • 파라미터로 값을 받아 특정 값을 return 하는 함수는 유틸 함수로 분리
    • 파라미터로 도메인과 관련된 데이터를 받는 경우 도메인 유틸 함수로 분리
    • 그렇지 않은 경우는 공통 유틸 함수로 분리

폴더 구조는 해당 자료를 참고하여 구성하고자 했습니다.

폴더 구조의 경우, 이전까지는 자주 사용하는 프로젝트 폴더 구조 와 같은 아티클을 찾아보며 따라하는 형태로 많이 해왔습니다. 그랬을 때 생겼던 문제점은 프로젝트를 길게 가져가거나 규모가 커졌을 때 나의 분리 기준이 아닌 남의 분리 기준을 따라 했다보니, 이 파일은 어디다가 둬야하지? 와 같은 고민들이 반복적으로 생겼습니다.

이번 과제를 하면서는 내가 가장 잘 이해할 수 있는 형태의 폴더 구조, 나만의 폴더 구조를 찾기 위해 노력했던 것 같습니다. 명확한 답변을 내리진 못했지만, 더 많은 고민도 해보고 다른 분들은 어떻게 하셨는지 분석해보면서 생각의 폭을 넓히고 언젠가는 나만의 기준을 만들 수 있었으면 좋겠습니다 !

커스텀 hook을 분리할 때 어떤 함수까지 hook 안에 넣고 관리할 지에 대해서도 명확한 기준을 세워보고자 노력했던 것 같습니다.

제가 설정한 기준은 state를 직접 조작하는 함수는 커스텀 hook 안에 배치하고, 파라미터로 도메인 데이터를 받아 계산 후 특정 값을 return 하는 함수는 유틸 함수로 분리하자 였습니다.


// useProduct 커스텀 hook

  const [products, setProducts] = useLocalStorage("products", INITIAL_PRODUCTS);

  const addProduct = useCallback(
    (newProduct: Omit<ProductWithUI, "id">) => {
      const product: ProductWithUI = {
        ...newProduct,
        id: `p${Date.now()}`,
      };
      setProducts((prev) => [...prev, product]);
      addNotification("상품이 추가되었습니다.", "success");
    },
    [addNotification]
  );



// filterProducts 유틸 함수

export const filterProducts = (
  products: ProductWithUI[],
  debouncedSearchTerm: string
) => {
  if (debouncedSearchTerm) {
    return products.filter(
      (product) =>
        product.name
          .toLowerCase()
          .includes(debouncedSearchTerm.toLowerCase()) ||
        (product.description &&
          product.description
            .toLowerCase()
            .includes(debouncedSearchTerm.toLowerCase()))
    );
  } else {
    return products;
  }
};

product를 예로 들면 addProduct는 product state를 직접 조작하는 함수이니 커스텀 hook 내부에 배치하여 파일 한 곳에서 관리하고자 했고, filterProducts는 파라미터로 입력을 받아 계산 후 특정 값을 return 하니 유틸 함수로 분리하되 도메인 관련 파라미터를 받으니 /utils/product/filterProducts.ts 경로에 배치하고자 했습니다.

1. 계산 함수와 액션 함수 개념

아직까지 계산 함수와 액션 함수의 개념을 정확하게 파악하지 못한 것 같습니다.

제가 생각한 개념은

  • 계산 함수 : 파라미터로 들어온 데이터를 뚝딱뚝딱해서 특정 값을 return 하는 함수
  • 액션 함수 : state 변화 등으로 UI에 변경을 불러 일으키는 함수 로 이해하고 있는데 맞을지, 아니라면 계산 함수와 액션 함수가 어떤 것인지 궁금해요....!!

2. 함수 구분 기준

제가 설정한 기준은 state를 직접 조작하는 함수는 커스텀 hook 안에 배치하고, 파라미터로 도메인 데이터를 받아 계산 후 특정 값을 return 하는 함수는 유틸 함수로 분리하자 였습니다.

코치님의 시선에서 함수 구분 기준이 타당한지, 납득이 되는지 궁금합니다 !
