기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

  • 처음엔 이미 handlers.ts 파일로 api모킹할때 쓰면 되지 않나? 생각했습니다. handlersUtils은 아예 다른 기능(과제 기준 CRUD 외의 기능)을 구현하라는 건가 싶었지만, (수많은 질문과 탐색을 통해) 각각의 테스트에서 모킹 데이터가 공유되어 테스트의 독립성이 보장되지 못하는 상황을 방지하기 위해 매 테스트마다 새로운 데이터 환경을 구축해줄 handlersUtils 모듈을 따로 만들어야 함을 알게 됐습니다. (과제를 진행할 땐 한참 집중하다보니 이해가 된 느낌이 들었지만, PR을 쓰고있는 현 시점에서는 애매하게 감만 남아 있을 뻔 했습니다. 운이 너무 좋게도, 디스코드 아고라에 적힌 토픽의 댓글에서 다시 한 번 개념을 정리해봤습니다 ㅎ..)
  /* A test code */ server.use(handlerForA); // A 핸들러가 지정됨
  /* B test code */ server.use(handlerForB); // B 핸들러가 지정됨
  /* A test code */ fetch(); // B 핸들러가 호출됨
  /* B test code */ fetch(); // B 핸들러가 호출됨
  /* A test code */ server.resetHandlers();
  /* B test code */ server.resetHandlers();

==> A/B테스트 모두 동일한 환경에서 resetHandlers()가 실행될 수 있으니, 각 테스트의 간섭을 방지하기 위한 장치가 필요함 ====> server.use()tl handlerUtils가 모킹 데이터를 스코프 내부로 감싸서 처리될 수 있게 해야함.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

  • 테스트에서 계산되는 날짜의 시점을 통일시켜줬습니다. 실제, useNotification 관련 테스트 진행 중, 테스트 코드 내의 시간과 커스텀 훅 내의 시간이 서로 다르게 찍히는 것을 발견했고, 또 그 둘의 시간대가 달라 시간 차이를 구하는 로직에서 테스트 로직이 꼬이는 점을 발견했습니다. 그래서 setupTest.ts 파일에
    vi.useFakeTimers()와 vi.setSystemTime()을 셋업 단계에 추가하여 가상의 날짜로 시점을 제어하고, vi.stubEnv('TZ', 'UTC')로직을 통해 테스트 소스와 커스텀 훅간 시간대도 통일시켰습니다

심화 과제

  • App 컴포넌트 적절한 단위의 컴포넌트, 훅, 유틸 함수로 분리했는가?
  • 해당 모듈들에 대한 적절한 테스트를 2개 이상 작성했는가?

과제 셀프회고

일단, 테스트 구현을 해본적도, 본적도 없는 일주일 전의 나를 생각하면 과제 완료를 찍었다는 것 만으로 가슴이 벅차오르네요.. 첫 날만해도 암흑 속을 걷는 기분이 들었지만, 점점 부딪혀보고 도움도 건네받고 하니 테스트 자체의 필요성과 목적, 테스트 코드 작성 요령 등에 자신감이 생겼던 것 같습니다. 특히 마지막 handler 파트는 당장이라도 포기하고 medium으로 내려가고 싶었지만, 꾹 참고 끝까지 도전한 스스로에게 박수를 보내봅니다.. (도와주신 모든 분들 감사해요)

기술적 성장

아무래도 가장 큰 소득은 테스트 API를 나의 의도에 맞게 사용할 수 있게 된 점이고, 이를 통해 단위 테스트, 통합 테스트에서 어떻게 설계해야 주어진 디스크립션과 나의 의도를 충족시킬 수 있는지 잘 이해하게 된 것 같습니다.(실제로 그렇게 했는지는 잘 모르겠네요..ㅎ)

코드 품질

  • 가장 만족스러운 부분은, 위에서 언급한 시점/시간대 통일 로직을 매 테스트 마다 실행하지 않고 setupTest파일에 올려둔 점인 것 같습니다. 이 로적 덕에 코드 수를 많이 줄일 수 있었네요.
  • 가장 불만인건 이 외에도 공통화처리 할 부분이 많이 남아있는 것 같습니다. 특히 통합테스트 소스에서 매 테스트마다 해들러를 주입하고 APP컴포넌트를 랜더링하고.. 하루만 더 시간이 생겼으면 통테를 비롯한 여러 곳에서의 로직 간소화를 하지 않았을가..
  • 총평 : 테오 코치님께서 멘토링 때 조언해주셨던 '나만의 패턴'이 아직 정립되지 않음을 절실히 느끼게 됐습니다. 바쁘고 정신없으니, 여기서는 공통화, 저기서는 하드코딩... 코치님 말씀대로 수없이 패턴화 하여 나만의 설계 패턴을 완성시켜보려 합니다.

학습 효과 분석

-- 비동기 렌더링

통테를 마지막에 진행해서 제대로 분석하진 못했지만, 렌더링, 내부 훅, 유저 이벤트 등 비동기로 처리해야되는 여러 상황에 따라 전후 데이터가 달라질 수 있기 때문에(심지어는 쿼링이 안될 수 있음) 테스트의 의도에 부합하도록 expect 실행 시점을 잘 제어해야 함을 경험했습니다.

-- 애매한 디스크립션

간혹 "적당한 시간, "올바른 조건" 등 디스크립션에 구체적이지 않은 테스트 조건을 제시된게 종종 있었는데, 테스트 코드를 작성함에 있어 구체적인 초기 환경 설정 및 세팅이 중요하다고 생각하게 됐습니다.

과제 피드백

통합 테스트 관련

리뷰 받고 싶은 내용

궁금한 점

1번

통테 시나리오 중 '일정을 삭제하고 더 이상 조회되지 않는지 확인한다'부분에서, 통합테스트에서 "데이터 변화에 의한 분기처리" 가능한지 궁금합니다.

코드를 짜다보니 자연스럽게 화면의 변화 전/후를 쿼링하여 테스트 결과를 예측했는데, 해당 시나리오에서는 'filteredEvents'배열의 길이를 기준으로 분기를 치는게 더 명확한 검증이 될 것 같다고 느꼈습니다.

  • 렌더링 > 기존 이벤트 삭제
    1. 이벤트가 0개일 때 --> "검색 결과가 없습니다." 표기돼야 함 --> getByText로 쿼링 시도
    1. 이벤트가 1개 이상일 때 --> "삭제 버튼" 갯수로 쿼링 시도

이런 흐름대로 분기처리를 시도하려 했는데, 이벤트가 0개가 되거나 1개 이상이 될 때 각각의 반대상황에서 조건으로 잡은 쿼링 타겟들이 화면상 사라져 제 의도대로 로직을 설계할 수 없었습니다.

그래서, 아래 두 가지가 궁금합니다.

  • 통합 테스트 코드 작성 시, UI변경 말고 데이터로 조건 분기할 수 있는 방법
  • DOM요소로 제가 설계한 의도대로 처리하려 할 때의 방법

과제를 수행하면서 느낀점

  • 너무 막막하고, 슥 해보다가 medium 통과라도 하자

과제 제출 후 생각

과제 피드백

수고 많았어요 지훈! 이번 과제는 테스트를 하는 다양한 방법들을 접하면서 테스트 코드를 작성한다는 그 심리적 장벽을 허물고 친숙해지는데 있었습니다. Hard까지 완주하시고 회고도 정말 깊이 있게 작성해주셨네요. 잘했습니다. 합격입니다!

"첫 날만해도 암흑 속을 걷는 기분이 들었지만" → "어려운게 아니라 낯선거였다! 해보면 안될게 없다"로의 변화가 정말 인상적입니다. 특히 "마지막 handler 파트는 당장이라도 포기하고 medium으로 내려가고 싶었지만, 꾹 참고 끝까지 도전한 스스로에게 박수를 보내봅니다" 이 말이 정말 와닿네요. 저도 박수 보냅니다 ㅋㅋ 과제의 취지에 맞게 생소함을 극복하고 끝까지 해줘서 고맙습니다. 테스트의 진짜 어려움은 그 생소함과 왜 해야 하는거지? 를 극복하는거라고 생각해요.

"테오 코치님께서 멘토링 때 조언해주셨던 '나만의 패턴'이 아직 정립되지 않음을 절실히 느끼게 됐다. 바쁘고 정신없으니 여기서는 공통화, 저기서는 하드코딩"이라고 하셨는데, 그걸 벌써 의식해주고 적용해주고 있다니 너무 고맙습니다. 나의 코드 일관성을 의식하다보면 둘다 만족할만한 패턴을 뭘까를 고민하면서 적재적소에 좋은 코드를 쓸 수 있을거에요. 9주뒤가 기대가 됩니다!

Q) 통테 시나리오 중 '일정을 삭제하고 더 이상 조회되지 않는지 확인한다'부분에서, 통합테스트에서 "데이터 변화에 의한 분기처리" 가능한지 궁금합니다.

통합테스트에서 데이터 기반 분기 처리에 대한 질문이 정말 좋습니다. 답변드리자면, 통합테스트는 기본적으로 "사용자가 보는 UI"를 기준으로 검증하는 게 원칙이에요. 그래서 filteredEvents 배열 길이를 검증하기 보다는 눈에 보이는 그대로 화면에 "검색 결과가 없습니다"라는 글자가 보이는지 테스트를 하는거죠. 테스트를 INPUT과 OUTPUT을 고정시켜서 구현을 변경했을때 잘 되던게 안되는 회귀 테스트의 목적이 강하므로 분기를 하도록 하기 보다는 각 케이스마다 간결하고 명세와 사용자에게 노출되는 UI를 근거로 작성되는 편이 좋습니다.

남은 9주간 많은 경험을 하면서 자기만의 패턴들을 잘 정립해 나가길 바랍니다. 정말 잘했습니다. 다음 과제도 화이팅입니다!
