사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

Before 패키지에서 발견한 문제점

  1. '기능'이 결합된 컴포넌트 구조

기존 UI 컴포넌트들은 순수하게 디자인적 요소만을 리턴하는 것이 아니라 props요소를 가지고 내부 로직이 들어가있다는 점에서 유연하게 활용되지 못한 부분이 있습니다.

예를 들어, Button 컴포넌트에서 entityTyper과 role에 따른 비활성화 로직이 들어가있습니다.

// /before/src/components/atoms/Button.tsx
if (entityType && action && entity) {
    if (entityType === 'user' && action === 'delete' && entity.role === 'admin') {
        actualDisabled = true;
    }
    
    if (entityType === 'post' && action === 'publish' && entity.status === 'published') {
        actualDisabled = true;
    }
}

이러한 로직은 다른 페이지 및 컴포넌트에서 재사용하기 힘들어진다는 단점을 안고가야 합니다. (확장성 X)

  1. 일관성 없는 컴포넌트 API

컴포넌트마다 prop 네이밍이 제각각이라 개발 시 컴포넌트 세부 코드를 일일이 확인해봐야하는 불편함이 있습니다.

// /before/src/components/atoms/Button.tsx
interface ButtonProps {
    variant?: 'primary' | 'secondary' | 'danger' | 'success'; // 이름: variant
    ...
}

// /before/src/components/atoms/Badge.tsx
interface BadgeProps {
    type?: 'primary' | 'secondary' | 'success' | 'danger' ...; // 이름: type (Button과 다름)
    pill?: boolean;
    ...
}

위처럼, 버튼 컴포넌트에선 variant, 뱃지 컴포넌트에선 type으로 서로 다른 네이밍 규칙을 갖고 있습니다.

  1. 아토믹 디자인 시스템의 잘못된 사용 예시(?)

organisms 폴더에 있는 컴포넌트들이 단순히 원자(Atom)와 분자(Molecule)의 조합이 아니라, 특정 페이지를 위한 거대한 덩어리가 되어 재사용성이 떨어지는 것 같습니다.

// /before/src/components/organisms/Table.tsx
const renderCell = (row: any, columnKey: string) => {
    if (entityType === 'user') {
        if (columnKey === 'role') {
            return <Badge userRole={value} showIcon />;
        }
        // ...
    }
    
    if (entityType === 'post') {
        if (columnKey === 'status') {
            return <Badge status={value} showIcon />;
        }
        // ...
    }
};

테이블은 데이터만 받아서 보여줘도 됐을 것 같은데, 내부에서 user 타입인지 post 타입인지에 따라 렌더링 분기 로직이 들어가있어 컴포넌트 활용 범위가 제한된다 생각합니다.

개편 과정에서 집중한 부분

  1. headless UI를 활용해 컴포넌트로 분리
  • 기존의 내부 로직까지 결속돼있던 부분을 View적인 요소에만 집중하는 구조로 리팩토링 합니다.
  • (사실 이건 의도하지 않았어도, 과제 요구사항(shadcn 사용)을 지키다 보면 자연스레 충족된다고 생각함..)
  1. CVA를 활용한 일관된 API 설계
  • CVA 라이브러리르 사용하여 스타일 prop 데이터 양식을 통일합니다.
  • (이거도 위에와 마찬가지인듯..?)
  1. 실용적인 아토믹 디자인 (컴포넌트 디렉토리 구조 개편)
  • 물리적인(?) 구조 (atoms, molecules, organisms)에 종속되지 않고, 역할을 기반으로 디렉토리 구조를 개편합니다.
    • 기본 UI 컴포넌트들(shadcn) : /components/ui
    • 내부 로직 혹은 컴포넌트들 간 조합이 이루어진 것들 : /components 하위

사용한 기술 스택 경험

어려웠던 점과 해결 방법

  1. 그동안 누군가 설계해준 디자인 시스템을 '이용'만 해봤지, 직접 설계를 해본적은 처음이다보니, 어디서 부터 시작해야될지 몰랐고, 특히 설치한 라이브러리들이 잘 적용되는걸 확인하는 것 조차도 많이 버벅거린 것 같습니다.
