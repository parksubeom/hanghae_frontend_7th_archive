기본과제

질문

Q. medium.useEventOperations.spec.tsx > 아래 toastFn과 mock과 이 fn은 무엇을 해줄까요?

toastFn과 mock은 테스트 환경에서 useSnacbar() 훅을 가짜로 사용하기 위한 설정이다. 실제 enqueueSnackbar 함수가 아니라 테스트 환경에서 제어 가능한 mock 함수가 대신 호출된다. -> 메시지 검증 가능

Q. medium.integration.spec.tsx > 여기서 ChakraProvider로 묶어주는 동작은 의미있을까요? 있다면 어떤 의미일까요?

MUI 컴포넌트들이 정삭적으로 렌더링돼 동작할 수 있는 환경을 제공하기 위해서 Provider로 감싸준다.

Q. handlersUtils > 아래 여러가지 use 함수는 어떤 역할을 할까요? 어떻게 사용될 수 있을까요?

setupTest.ts에서 설정한 MSW 서버에 각 테스트에서 필요한 목 api 응답을 등록하는 역할을 한다. 실제 API 서버를 호출하지 않고도 진짜 호출하는 것처럼 테스트 하기 위해 사용한다.

Q. setupTests.ts > 왜 이 시간을 설정해주는 걸까요?

목데이터가 10월 기준으로 되어 있어서 테스트코드에서 new Date()를 실행할 때 2025-10-01로 고정시킨다. 현재 날짜가 바뀌어도 테스트 결과는 변하지 않는다.

과제 셀프회고

처음에는 블로그 글을 참고하며 테스트 코드를 따라 쳐보는 정도였고, 직접 작성하는 것은 이번이 처음이었다. 공식 문서와 다른 개발자들의 블로그를 참고하면서 학습하니, Easy 단계의 훅 테스트나 유닛 테스트는 방향만 잡히면 비교적 빠르게 익숙해질 수 있었다.

하지만 Medium 과제부터는 MSW(Mock Service Worker) 를 사용하다 보니 확실히 난이도가 높게 느껴졌다. 퇴근 후 집에 도착하면 이미 8시쯤이라, 테스트를 공부하고 작성하기에 시간이 부족하다는 점도 어려웠다.

결국 Hard 난이도 대신 Medium으로 조정했지만, 다행히 Medium 과제는 완성할 수 있었다. 다만 테스트 코드 작성이 이번이 처음이라, 내가 구현한 로직이 충분히 올바른지에 대한 확신은 아직 부족하다고 느꼈다. 그래서 빠르게 피드백을 받아 개선해보고 싶다.

기술적 성장

Vitest로 테스트 코드를 작성하면서 유닛 테스트와 통합 테스트의 차이점을 명확히 이해하게 되었다. 또한 각 테스트가 서로 독립적으로 동작해야 한다는 것의 중요성을 직접 느낄 수 있었다. 추가적으로 mock 데이터로 의존성을 분리하는 법까지 배울 수 있었다.

코드 품질

Easy 과제는 전반적으로 만족스러웠다. 테스트의 목적을 명확히 정의하려고 노력했고, 그에 맞게 테스트 설명(Description)도 세밀하게 수정했다.

리팩토링이 필요한 부분은 주석에도 언급했듯이 타이머 관련 로직이라고 생각한다. 현재 setupTests에서 타이머를 설정하면 각 테스트에 상호 영향을 미칠 수 있어, 이를 테스트별로 독립적으로 관리할 수 있는 방법에 대해 좋은 방법을 알고싶다.

학습 효과 분석

가장 큰 배움은 직접 테스트 코드를 작성해 본 경험이었다. 테스트를 통해 내가 구현한 로직이 의도한 대로 동작하는지 명확한 기댓값으로 검증할 수 있다는 점이 인상 깊었다. 현재 연말정산 관련 업무를 담당하고 있는데, 계산 로직이 많은 특성상 이러한 테스트 방식을 실무에 적용한다면 더 완성도 높고 신뢰할 수 있는 제품을 만드는 데 큰 도움이 될 것이라고 생각한다.

과제 피드백

Easy 과제를 진행하면서, 저처럼 테스트 코드를 처음 작성하는 사람들도 테스트의 첫걸음을 쉽게 밟을 수 있는 구조라고 느꼈다. 다만 통합 테스트 부분에서는 테스트 개념이 충분히 잡혀 있지 않은 상태라 다소 어려움을 겪었다.

리뷰 받고 싶은 내용

setupTests에서 전역적으로 타이머를 설정했을 때, 각 테스트 내에서 타이머를 독립적으로 관리하려면 어떤 방식이 가장 안전할까요? 특히 테스트 간 타이머 충돌 문제를 방지하는 방법이 궁금합니다.
