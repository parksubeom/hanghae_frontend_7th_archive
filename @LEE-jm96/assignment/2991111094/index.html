기본과제

상품목록

상품 목록 로딩

  • 페이지 접속 시 로딩 상태가 표시된다
  • 데이터 로드 완료 후 상품 목록이 렌더링된다
  • 로딩 실패 시 에러 상태가 표시된다
  • 에러 발생 시 재시도 버튼이 제공된다

상품 목록 조회

  • 각 상품의 기본 정보(이미지, 상품명, 가격)가 카드 형태로 표시된다

한 페이지에 보여질 상품 수 선택

  • 드롭다운에서 10, 20, 50, 100개 중 선택할 수 있으며 기본 값은 20개 이다.
  • 선택 변경 시 즉시 목록에 반영된다

상품 정렬 기능

  • 상품을 가격순/이름순으로 오름차순/내림차순 정렬을 할 수 있다.
  • 드롭다운을 통해 정렬 기준을 선택할 수 있다
  • 정렬 변경 시 즉시 목록에 반영된다

무한 스크롤 페이지네이션

  • 페이지 하단 근처 도달 시 다음 페이지 데이터가 자동 로드된다
  • 스크롤에 따라 계속해서 새로운 상품들이 목록에 추가된다
  • 새 데이터 로드 중일 때 로딩 인디케이터와 스켈레톤 UI가 표시된다
  • 홈 페이지에서만 무한 스크롤이 활성화된다

상품을 장바구니에 담기

  • 각 상품에 장바구니 추가 버튼이 있다
  • 버튼 클릭 시 해당 상품이 장바구니에 추가된다
  • 추가 완료 시 사용자에게 알림이 표시된다

상품 검색

  • 상품명 기반 검색을 위한 텍스트 입력 필드가 있다
  • 검색 버튼 클릭으로 검색이 수행된다
  • Enter 키로 검색이 수행된다
  • 검색어와 일치하는 상품들만 목록에 표시된다

카테고리 선택

  • 사용 가능한 카테고리들을 선택할 수 있는 UI가 제공된다
  • 선택된 카테고리에 해당하는 상품들만 표시된다
  • 전체 상품 보기로 돌아갈 수 있다
  • 2단계 카테고리 구조를 지원한다 (1depth, 2depth)

카테고리 네비게이션

  • 현재 선택된 카테고리 경로가 브레드크럼으로 표시된다
  • 브레드크럼의 각 단계를 클릭하여 상위 카테고리로 이동할 수 있다
  • "전체" > "1depth 카테고리" > "2depth 카테고리" 형태로 표시된다

현재 상품 수 표시

  • 현재 조건에서 조회된 총 상품 수가 화면에 표시된다
  • 검색이나 필터 적용 시 상품 수가 실시간으로 업데이트된다

장바구니

장바구니 모달

  • 장바구니 아이콘 클릭 시 모달 형태로 장바구니가 열린다
  • X 버튼이나 배경 클릭으로 모달을 닫을 수 있다
  • ESC 키로 모달을 닫을 수 있다
  • 모달에서 장바구니의 모든 기능을 사용할 수 있다

장바구니 수량 조절

  • 각 장바구니 상품의 수량을 증가할 수 있다
  • 각 장바구니 상품의 수량을 감소할 수 있다
  • 수량 변경 시 총 금액이 실시간으로 업데이트된다

장바구니 삭제

  • 각 상품에 삭제 버튼이 배치되어 있다
  • 삭제 버튼 클릭 시 해당 상품이 장바구니에서 제거된다

장바구니 선택 삭제

  • 각 상품에 선택을 위한 체크박스가 제공된다
  • 선택 삭제 버튼이 있다
  • 체크된 상품들만 일괄 삭제된다

장바구니 전체 선택

  • 모든 상품을 한 번에 선택할 수 있는 마스터 체크박스가 있다
  • 전체 선택 시 모든 상품의 체크박스가 선택된다
  • 전체 해제 시 모든 상품의 체크박스가 해제된다

장바구니 비우기

  • 장바구니에 있는 모든 상품을 한 번에 삭제할 수 있다

상품 상세

상품 클릭시 상세 페이지 이동

  • 상품 목록에서 상품 이미지나 상품 정보 클릭 시 상세 페이지로 이동한다
  • URL이 /product/{productId} 형태로 변경된다
  • 상품의 자세한 정보가 전용 페이지에서 표시된다

상품 상세 페이지 기능

  • 상품 이미지, 설명, 가격 등의 상세 정보가 표시된다
  • 전체 화면을 활용한 상세 정보 레이아웃이 제공된다

상품 상세 - 장바구니 담기

  • 상품 상세 페이지에서 해당 상품을 장바구니에 추가할 수 있다
  • 페이지 내에서 수량을 선택하여 장바구니에 추가할 수 있다
  • 수량 증가/감소 버튼이 제공된다

관련 상품 기능

  • 상품 상세 페이지에서 관련 상품들이 표시된다
  • 같은 카테고리(category2)의 다른 상품들이 관련 상품으로 표시된다
  • 관련 상품 클릭 시 해당 상품의 상세 페이지로 이동한다
  • 현재 보고 있는 상품은 관련 상품에서 제외된다

상품 상세 페이지 내 네비게이션

  • 상품 상세에서 상품 목록으로 돌아가는 버튼이 제공된다
  • 브레드크럼을 통해 카테고리별 상품 목록으로 이동할 수 있다
  • SPA 방식으로 페이지 간 이동이 부드럽게 처리된다

사용자 피드백 시스템

토스트 메시지

  • 장바구니 추가 시 성공 메시지가 토스트로 표시된다
  • 장바구니 삭제, 선택 삭제, 전체 삭제 시 알림 메시지가 표시된다
  • 토스트는 3초 후 자동으로 사라진다
  • 토스트에 닫기 버튼이 제공된다
  • 토스트 타입별로 다른 스타일이 적용된다 (success, info, error)

심화과제

SPA 네비게이션 및 URL 관리

페이지 이동

  • 어플리케이션 내의 모든 페이지 이동(뒤로가기/앞으로가기를 포함)은 하여 새로고침이 발생하지 않아야 한다.

상품 목록 - URL 쿼리 반영

  • 검색어가 URL 쿼리 파라미터에 저장된다
  • 카테고리 선택이 URL 쿼리 파라미터에 저장된다
  • 상품 옵션이 URL 쿼리 파라미터에 저장된다
  • 정렬 조건이 URL 쿼리 파라미터에 저장된다
  • 조건 변경 시 URL이 자동으로 업데이트된다
  • URL을 통해 현재 검색/필터 상태를 공유할 수 있다

상품 목록 - 새로고침 시 상태 유지

  • 새로고침 후 URL 쿼리에서 검색어가 복원된다
  • 새로고침 후 URL 쿼리에서 카테고리가 복원된다
  • 새로고침 후 URL 쿼리에서 옵션 설정이 복원된다
  • 새로고침 후 URL 쿼리에서 정렬 조건이 복원된다
  • 복원된 조건에 맞는 상품 데이터가 다시 로드된다

장바구니 - 새로고침 시 데이터 유지

  • 장바구니 내용이 브라우저에 저장된다
  • 새로고침 후에도 이전 장바구니 내용이 유지된다
  • 장바구니의 선택 상태도 함께 유지된다

상품 상세 - URL에 ID 반영

  • 상품 상세 페이지 이동 시 상품 ID가 URL 경로에 포함된다 (/product/{productId})
  • URL로 직접 접근 시 해당 상품의 상세 페이지가 자동으로 로드된다

상품 상세 - 새로고침시 유지

  • 새로고침 후에도 URL의 상품 ID를 읽어서 해당 상품 상세 페이지가 유지된다

404 페이지

  • 존재하지 않는 경로 접근 시 404 에러 페이지가 표시된다
  • 홈으로 돌아가기 버튼이 제공된다

AI로 한 번 더 구현하기

  • 기존에 구현한 기능을 AI로 다시 구현한다.
  • 이 과정에서 직접 가공하는 것은 최대한 지양한다.

과제 셀프회고

기술적 성장

저는 프로젝트의 상태관리를 크게 2가지로 관리했습니다.

  1. URL SearchParams (상품 목록, 필터 상태) 관리하는 상태: 검색어 (search), 카테고리 필터 (category1, category2), 정렬 조건 (sort), 페이지네이션 (page, limit) 특징: 새로고침 시 URL에서 자동 복원, 브라우저 히스토리 지원 (뒤로가기/앞으로가기)

  2. CartStore (장바구니 상태) 관리하는 상태: 장바구니 아이템 목록 (items). 선택된 아이템 ID 목록 (selectedItems) 특징: Observer 패턴으로 상태 변경 알림, localStorage 자동 동기화, 새로고침 시 localStorage에서 복원, 싱글톤 패턴

class CartStore {
  constructor() {
    this.state = {
      items: [], // 장바구니 아이템
      selectedItems: [], // 선택된 아이템 ID
    };
    this.listeners = new Set();
    
    // localStorage에서 장바구니 복원
    this.loadFromStorage();
  }
}

먼저 상품 목록, 필터 상태 등을 URL로 관리하면서 새로고침 시 데이터가 그대로 보존됐고 브라우저 히스토리 기능을 활용하기 편했습니다. 뒤로가기/앞으로가기가 자연스럽게 동작하였습니다. UI 파라미터를 굳이 store에 넣지 않아도 된다고 판단했습니다.

검색 필터와는 달리 장바구니는 URL로 관리하기에 적합하지 않다고 생각했습니다. 사용자의 앱 내부 상태이기 때문에 더 안정적이고 지속적인 저장 방식이 필요했기 때문입니다. 그래서 저는 CartStore를 직접 구현했습니다. 구독을 하면 store 변화를 자동으로 감지하므로 UI가 최신 상태로 유지됐고, 직접 설계하면서 “상태가 변화하면 UI를 자동 갱신한다”는 개념을 보다 깊게 이해하게 됐습니다.

자랑하고 싶은 코드

subscribe(listener) {
  this.listeners.add(listener);
  
  // 구독 해제 함수 반환
  return () => {
    this.listeners.delete(listener);
  };
}

// Store 구독 (상태 변경 시 자동 리렌더링)
if (unsubscribe) {
  unsubscribe(); // 이전 구독 해제 → listeners에서 제거
}
unsubscribe = cartStore.subscribe(() => {
  renderCartModal();
});

모달을 한번 열고 구독 해제를 하지 않은 후 다시 모달을 열면 총 2번 렌더링이 됩니다. → cartStore.listeners = [renderCartModal 함수1, renderCartModal 함수2]

이를 방지하기 위해 구독 해제 함수를 반환해 항상 렌더를 1번만 유지할 수 있도록 구현하였습니다. → unsubscribe() 실행 → renderCartModal 함수1 제거 → cartStore.listeners = [] → 새로 구독 → cartStore.listeners = [renderCartModal 함수2]

개선이 필요하다고 생각하는 코드

이벤트 핸들러 파일(eventHandlers.js)이 지나치게 커졌는데 이 부분에서 개선이 필요하다고 생각했습니다.

프로젝트 중반 이후 가장 크게 체감했던 문제 중 하나는 eventHandlers.js가 점점 커지면서 유지보수가 어려워졌다는 점입니다. 특히 handleClick 하나에 350줄 이상의 if-else 체인이 들어가면서, 코드를 파악하기 위해 스크롤을 수십 번 해야 하는 상황이 자주 발생했습니다. 또 찾고자 하는 버튼에 걸린 이벤트 로직을 찾기도 너무 불편했습니다. 이벤트는 관심사별로 쪼갤수록 유지보수성이 올라간다는 것을 알게 됐고. 다음에 이런 상황이 온다면 핸들러를 분리해야겠다고 생각했습니다.

학습 효과 분석

이번 프로젝트에서 장바구니 상태 관리를 위해 라이브러리를 쓰지 않고, 직접 Store를 설계해서 Observer 패턴으로 구현해보는 시도를 했습니다.

처음에는 간단한 장바구니를 만든다는 가벼운 생각으로 임했는데, 막상 구현하고 유지보수하면서 생각보다 훨씬 많은 것을 배울 수 있었습니다. 특히 라이브러리가 자동으로 해주는 일을 직접 구현해보니 당연하게 느껴졌던 여러 동작들이 실제로는 어떤 메커니즘으로 작동하는지 직접 경험할 수 있었습니다. Redux나 Zustand를 사용할 때는 listeners를 어떻게 저장할지, notify를 어느 시점에서 호출할지 setState에서 어떤 방식으로 merge할지를 깊게 생각하지 않았는데 이 작은 Store를 만들면서 라이브러리가 감춰놓은 많은 디테일을 꽤 깊게 이해하게 되었습니다. 단순히 써보는 것과 직접 만들어보는 것 사이의 차이를 실감했습니다.

또 상태 변화의 흐름을 ‘내가 설계했다’는 좋은 경험을 하게 됐습니다. React는 상태 변화 → 리렌더링을 알아서 처리해주지만, Observer 패턴은 모든 흐름을 제가 직접 설계해야 했습니다. 상태가 언제 바뀌는지, 무엇 때문에 바뀌는지, 누가 이 변경을 받아야 하는지 변경이 퍼져나가는 순서와 방식이 모든 것을 직접 구현하다 보니 앱 전체의 데이터 흐름이 더 디테일하게 이해되기 시작했습니다.

결론적으로 라이브러리가 대신해주던 부분을 직접 만들어보는 경험은 사고 방식을 한 단계 더 깊게 만들어 주었습니다.

과제를 진행하면서 전체 렌더링과 부분 렌더링의 설계를 고민해보았습니다.

이번 프로젝트를 처음 설계할 때는 화면 렌더링을 “전체 렌더링(Full Rendering)” 방식으로 구현했습니다. 특정 경로에 진입하면 #root 영역 전체를 통째로 교체하는 구조였습니다. renderFullHomePage()에서 로딩 화면을 먼저 그리고, 데이터를 받아오면 다시 전체 화면을 렌더링하는 식으로 내부적으로는 renderToRoot()라는 유틸리티가 사용되는데, 이 함수는 단순히 root.innerHTML을 새로운 컴포넌트 문자열로 완전히 갈아끼웠습니다.

이 방식은 구조적으로 단순하고 UI 상태를 크게 신경 쓰지 않아도 된다는 장점이 있었습니다. 항상 완전히 새로운 DOM을 만들기 때문에 일종의 “리셋”이 자동으로 이루어졌고, 오류 발생 가능성도 적었습니다. 하지만 실제로 검색 기능과 필터 기능을 반복적으로 사용하면서 문제점이 드러났습니다.

우선, 검색하거나 필터를 적용할 때 화면이 깜빡였습니다. 전체 DOM을 교체하기 때문에 헤더, 푸터, 검색창, 리스트 등 화면의 모든 요소가 재생성되었고, 그 과정에서 사용자 경험이 끊기는 느낌을 받았습니다. 또한 검색창에 포커스를 두고 입력하던 중 검색을 적용하면 포커스가 사라지고 스크롤이 초기화되는 문제도 있었습니다. 기능은 정상적으로 동작했지만, 사용 경험은 매끄럽지 못했습니다.

이 문제를 해결하기 위해, 기존의 전체 렌더링 방식과 병행할 부분 렌더링(Partial Rendering) 방법을 도입했습니다. 전체를 갈아엎는 renderToRoot() 대신, 특정 DOM 요소만 업데이트할 수 있는 renderTo()라는 새로운 유틸리티 함수를 추가했습니다. 이 함수는 특정 셀렉터를 받아 해당 영역만 교체하도록 설계했습니다. 이렇게 하면 검색 결과 리스트나 상품리스트처럼 자주 변경되는 영역만 다시 렌더링할 수 있고, 전체 화면을 덮어쓰지 않기 때문에 헤더, 검색 폼, 스크롤 위치 등 나머지 상태는 그대로 유지할 수 있었습니다.

부분 렌더링을 도입해서 얻은 것들

  1. 화면 깜빡임 제거

검색 후 Enter를 누르거나 필터를 변경해도 전체 화면이 재생성되지 않기 때문에, 유저 입장에서 화면이 더 이상 깜빡이지 않습니다.

  1. 매끄러운 사용자 경험

리스트나 일부 UI만 부드럽게 갱신되기 때문에 SPA에 가까운 UX를 제공할 수 있습니다.

결과적으로 검색·필터 기능이 훨씬 자연스럽게 동작하게 됐습니다. 이렇게 전체 렌더링과 부분 렌더링을 분리하는 전략을 도입하면서, 프로젝트의 구조적 단순함은 유지하면서도 사용자 경험을 크게 개선할 수 있었다는 점이 이번 과제에서 좋았던 부분이었습니다.

리뷰 받고 싶은 내용

리액트를 쓰지 않고 바닐라 JS로 작업하다 보니 불편한 부분이 있었습니다. 이벤트를 최상단의 eventHandler.js에서 모두 관리하다 보니, 버튼 클릭 시 closest로 영역을 찾아 처리하는 방식이 번거롭다는 생각을 했습니다.

리액트에서는 onClick 안에 바로 이벤트 로직을 넣을 수 있어서, 버튼 이름만 보고도 관련 코드를 빠르게 찾을 수 있었는데 바닐라에서는 이게 잘 안 돼서 불편했습니다. 최상단에서 한 번만 container에 click을 걸고 event.target.closest()로 분기하는 이벤트 위임 방식보다 더 좋은 방법이 있는지 궁금합니다!

수고했어요 정민! 프레임워크 없이 SPA를 직접 만들어보면서 상태 관리의 본질과 렌더링 전략을 깊이 체험하는 시간이 되었기를 바랍니다.

회고 내용 중 상태 관리를 URL SearchParams와 CartStore로 명확히 분리한 근거를 설명하는 부분이 인상깊네요. 같은 전역 데이터가 필요한 경우더라도 "검색 필터는 URL로, 장바구니는 Store로"라는 설계에 대한 근거를 대는 부분이 좋았습니다. URL로 관리하면 새로고침 시 자동 복원되고 브라우저 히스토리도 자연스럽게 동작하죠. 반면 장바구니는 사용자의 앱 내부 상태이기 때문에 localStorage 기반의 지속적 저장이 필요하다는 통찰, 맞습니다. 같은 개념이더라도 상황에 따른 더 나은 방법을 선택하고 특히 그 근거를 들어 설명할 수 있다는 점이 아주 중요합니다.

전체 렌더링과 부분 렌더링을 구분해서 구현한 것도 좋았습니다. 처음에는 renderToRoot()로 전체를 갈아엎는 방식으로 시작했다가, 검색할 때마다 화면이 깜빡이고 포커스가 사라지는 문제를 겪으면서 renderTo()를 추가해 특정 영역만 업데이트하도록 개선한 과정. 이게 바로 React가 왜 가상 DOM을 만들었는지, Vue가 왜 반응형 시스템을 택했는지 이해하는 출발점입니다. 프레임워크들도 결국 같은 문제를 해결하기 위해 각자의 방식을 선택했거든요.

Q) 이벤트 위임 방식보다 더 좋은 방법이 있을까요?

=> 사실 이벤트 위임 자체는 좋은 패턴입니다. 동적으로 추가되는 요소에도 이벤트가 작동하고, 메모리 효율도 좋죠. 지금 설계의 문제는 모든 이벤트를 한 곳에서 처리하려다 보니 서로 영향을 주지 않는 코드가 한 곳에 있어서 복잡하게 느껴지는 것입니다. 설계에서 중요한 것은 서로 영향이 없다면 같은 자리에 두지 않는다에요.

개선 방향은 컴포넌트 단위로 이벤트를 분리하는 겁니다. 각 컴포넌트(ProductCard, SearchForm, CartModal)가 자기 영역의 이벤트를 직접 관리하도록 하는 거죠. 렌더링 후 해당 컴포넌트에 이벤트를 바인딩하고, 리렌더링 시 재등록하는 방식입니다.

이미 React를 쓰는 동안 익숙한 방법이죠? React 도한 이벤트 위임 방식으로 구현되어 있지만 각자의 컴포넌트에서 이벤트 등록을 하고 있죠. 다음 시간에는 React는 실제로 이러한 방식을 어떻게 활용했을지 파악해볼 수 있는 시간이 되길 바래요
