Chapter3-1. UI 컴포넌트 모듈화와 디자인 시스템

과제 목표

레거시 코드베이스를 현대적인 디자인 시스템으로 개편하는 실무 경험

  1. 정리되지 않은 레거시 코드의 문제점 식별 및 분석
  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

Before 패키지 분석 후 After 패키지 개편

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

  • 개선이 필요한 부분과 그 이유 정리

3. 컴포넌트 개편

과제 회고

과제를 진행하면서 느낀 점, 배운 점을 자유롭게 작성해주세요.

회사에서는 퍼블리싱 팀이 별도로 있어서, 그동안 실무에서는 CSS를 직접 다루는 경험이 많지 않았습니다. 이번 과제를 하면서 처음으로 Shadcn UI나 CVA 같은 기술을 접하게 되었는데, 처음엔 이런 스타일링 방식이 낯설어서 어떻게 접근해야 할지 고민이 있었습니다.

하지만 막상 시작해보니 Shadcn UI는 MUI나 Bootstrap처럼 명령어 기반으로 컴포넌트를 설치한 후 import해서 바로 사용할 수 있다는 점이 익숙하게 느껴졌고, Tailwind 기반 컴포넌트를 “필요한 것만 가져오는 방식”이라 프로젝트 전체 스타일을 강제하지 않고 유연하게 커스터마이징 가능하다는 점이 매력적이었습니다. 또, 디자인 토큰이 명확하게 정의돼 있어 일관된 UI 스타일을 유지하면서도 수정이 쉽다는 점도 장점으로 느꼈습니다.

CVA 또한 이번에 처음 사용했지만, 사용해보니 컴포넌트의 스타일 로직을 정리하기에 매우 적합했습니다. 특히 variant나 compoundVariants 기능을 이용해 props에 따라 스타일을 안전하게 변경할 수 있고, 스타일을 TS 로직 안에서 관리하다 보니 클래스가 복잡하게 흩어지지 않고 구조적으로 깔끔해진다는 장점을 크게 느꼈습니다. CSS 파일을 일일이 만들지 않아도 되고, 스타일 변경이 필요한 경우에도 함수 형태로 관리되어 확장성과 유지보수성이 높다는 점이 개인적으로 좋았습니다.

이번 과제를 통해 처음 접한 기술들이었지만, Shadcn UI와 CVA의 구조와 장점을 이해하게 되면서 CSS를 직접 다루지 않았던 경험적인 공백을 어느 정도 채울 수 있었고, 새로운 스타일링 패턴에 대한 이해도 확실히 넓어진 것 같습니다.

Before 패키지에서 발견한 문제점

  1. 컴포넌트들이 개별 css에 지나치게 의존하는 문제가 있습니다.

동일한 버튼, 폼, 테이블이라도 컴포넌트마다 색·폰트·패딩 정의가 제각각이라 디자인 일관성이 깨집니다. 특히 다크 모드나 브랜드 색상 교체 같은 글로벌 변경을 적용할 때, 모든 컴포넌트를 일일이 찾아 수정해야 합니다. 또 토큰 없이 #1976d2, font-size: 14px 같은 값이 하드코딩되면, 스타일을 공유하거나 동적 테마를 만들기 어렵습니다. 토큰이 있어야 Tailwind, Storybook 등 간에 동일한 원천을 재사용할 수 있습니다.

  1. ui와 비즈니스 로직이 같이 관리되고 있습니다.

같은 검증 로직이나 상태 전환을 다른 화면에서도 써야 할 때, UI 컴포넌트를 통째로 끌어다 쓰거나 로직을 복붙해야 합니다. 로직이 서비스/도메인 레이어로 분리돼 있으면 여러 뷰에서 공통으로 호출하기 쉬운데, 컴포넌트 안에 박혀 있으면 재사용이 거의 불가능합니다. 또 UI는 “어떻게 보여줄지”, 비즈니스 로직은 “무엇을 할지”를 담당하는데, 둘이 한 파일에 섞여 있어서 이해하기 어렵습니다.

개편 과정에서 집중한 부분

after 프로젝트는 각 컴포넌트가 독립적인 CSS를 가지는 대신, 토큰 기반 디자인 시스템을 도입해 재사용 가능하고 일관된 스타일을 만드는 것에 집중하였습니다.

사용한 기술 스택 경험

  1. 중앙 집중식 토큰 관리 src/tokens/colors.css, src/tokens/typography.css에 색상, 타이포그래피, spacing, radius 등을 CSS 변수로 정의했습니다. 각 컴포넌트는 하드코딩된 값(#1976d2, 14px) 대신 토큰(hsl(var(--primary)), var(--font-size-sm))을 참조합니다. 디자인 변경 시 토큰 파일만 수정하면 전체 UI에 일괄 반영되도록 하였습니다.

  2. 테마 일관성 보장 라이트/다크 모드는 .dark 블록에서 토큰만 오버라이드해 처리합니다. 컴포넌트 코드 변경 없이 테마 전환이 가능하고, 모든 요소가 동일한 팔레트를 공유해 시각적 일관성을 유지하도록 하였습니다.

  3. Tailwind와의 통합 @theme에서 토큰을 Tailwind 유틸리티(bg-primary, text-foreground)로 매핑했습니다. 토큰을 직접 기억할 필요 없이 의미 있는 클래스명으로 스타일을 적용할 수 있습니다.

  4. 디자인 시스템 문서화 Storybook에서 토큰을 시각화하고, 컴포넌트별 사용 예시를 생성하였습니다.

어려웠던 점과 해결 방법

  1. Storybook Controls에서 fullWidth를 토글해도 버튼 너비가 변하지 않는 현상이 있었습니다. Button 컴포넌트에서 fullWidth를 구조분해하지 않고 buttonVariants() 호출에도 포함하지 않아 발생했습니다. fullWidth를 명시적으로 받아 buttonVariants({ variant, size, fullWidth, className })에 전달하도록 수정했습니다.

  2. 다크 모드에서 일부 색상/텍스트가 사라지는 문제가 있었습니다. tokens/colors.css의 다크 모드 팔레트를 재조정하고, ManagementView, Table 등에서 하드코딩된 색상(#333, #fff 등)을 모두 CSS 변수(hsl(var(--foreground)))로 교체했습니다. 테이블에는 .dark .table th, .dark .table td { color: #f8fafc; } 오버라이드를 추가하여 해결했습니다.

리뷰받고 싶거나 질문하고 싶은 내용

이번 과제를 진행하면서 PR에 적혀있는 after 구조를 기반으로 프로젝트 구조를 설계했습니다. 그런데 과제 중 팀원들의 이야기를 들으면서 3계층 레이어드 아키텍처나 FSD 패턴 같은 구조적 패턴을 처음 접하게 되었고, 과제 완성에 집중하느라 이번에는 이론만 공부하고 실제 적용은 하지 못했습니다.

그래서 다음과 같은 궁금증이 생겼습니다.

프로젝트 규모나 성격에 따라 3계층 레이어드나 FSD처럼 명확한 구조를 반드시 적용해야 하는 기준이 있는지, 혹은 제가 작업한 구조로 진행해도 괜찮은지 알고 싶습니다.
