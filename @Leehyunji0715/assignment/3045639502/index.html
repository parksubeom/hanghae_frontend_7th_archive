Chapter3-1. UI 컴포넌트 모듈화와 디자인 시스템

과제 목표

레거시 코드베이스를 현대적인 디자인 시스템으로 개편하는 실무 경험

  1. 정리되지 않은 레거시 코드의 문제점 식별 및 분석
  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

Before 패키지 분석 후 After 패키지 개편

개편 목표

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

필수 과제

1. 디자인 시스템 구축

  • TailwindCSS 설정 및 디자인 토큰 정의
  • shadcn/ui 컴포넌트 설치 (Button, Input, Select, Card, Table 등)
  • CVA를 활용한 variants 패턴 적용
  • 일관된 스타일 시스템 구축

2. Before 패키지 분석

  • Before 패키지 실행 및 전체 코드 탐색
  • 스타일링, 컴포넌트 설계, 폼 관리 측면에서 문제점 파악
  • 개선이 필요한 부분과 그 이유 정리

3. 컴포넌트 개편

  • UI와 비즈니스 로직 분리
  • 순수한 UI 컴포넌트로 재구성
  • 일관된 컴포넌트 API 설계
  • 적절한 컴포넌트 구조 설계

심화 과제

  • Dark Mode 완전 지원 (CSS Variables + Tailwind)
  • Design Token 시스템 고도화 (색상 팔레트, 타이포그래피 스케일)
  • 뷰와 비즈니스로직이 분리되도록

과제 회고

과제를 진행하면서 느낀 점, 배운 점을 자유롭게 작성해주세요.

  • 처음에는 atom -> molcules -> organisms 구조보고, 오 체계적이고 좋은데? 라고 생각했다. 하지만 Badge를 mocules 레벨로 분리하면서 그 생각이 싹 사라졌다. UI + 1개의 역할이 있는 컴포넌트 (mocules)로 Badge를 분리하는데, CategoryBadge, StatusBadge, UserStatusBadge, RoleBadge 등등 여러개가 나오고, 하나씩 생성할 때마가 index.ts (배럴)에다가 계속 추가해야했다. 나중에는 과제 방향성 보고 해당 atomic 구조를 모두 없앴지만, 잠깐 맛본 atom 구조는 생각보다 관리하는데 불편했다.
  • 저는 과제 순서를 shadcn 컴포넌트 설치 => Shadcn 컴포넌트 기존 App에 적용 => (리팩토링) => components.css 를 Shadcn 컴포넌트 스타일로 이식 => 이를 기반으로 디자인시스템 설계 시작! 했습니다. 토큰부터 정하라하셨지만.. 정하는데도 시간이 많이 걸릴것 같아 우선 스타일 이식부터 진행했어요! 그러다보니 실제 디자인시스템을 목요일부터 설계하게 되어서 부랴부랴 과제를 진행하게된것 같습니다. (개인적으로 조금 아쉬웠던 부분이었어요..! 디자인시스템부터 할껄,,,)

Before 패키지에서 발견한 문제점

  • 기존 Badge와 Button 컴포넌트에 비즈니스로직과 UI가 섞여있었고, 이를 분리하는 과정에서 앱에서 전혀 쓰이지 않는 데이터(Props) 발견 및 제거했다.
  • ManagementPage.tsx 에 User과 Post를 한번에 처리하려는 로직이 대다수였다. UI와 일부 데이터 타입은 비슷할지라도, 비슷하다고 같이 공통으로 묶어 쓰려니까 SRP 원칙에도 위배되고, 코드가 불분명하게 섞여져 유지보수를 어렵게 만들었다. any 타입도 많이 보았다. AnyScript
  • 스타일링: 스타일 값들이 클래스 내부에서 직접 하드코딩되어 쓰이고있다 (토큰화가 되어있지 않음)
  • 컴포넌트 설계: 컴포넌트의 Props의 형식이 일관적이지 않다. Badge와 Button 의 size 가 Badge는 'sm', 'md' 같이 축양형이지만, Button의 경우에는 'small', 'medium'같이 풀네임으로 썼다.
  • 폼 관리 측면: FormData 타입을 any로 User과 Post를 한번에 처리한다. 안정성이 떨어진다. UI상에서는 필수 '*' 표시가 되어있지만, validation 로직이 없는지 빈 값을 넣어도 Post가 추가되는 문제가 발생했다.

개편 과정에서 집중한 부분

  • 비즈니스로직과 UI를 분리: Shadcn 컴포넌트를 쓰니까 절반은 작업된거다보니 훨씬 수월하게 작업할 수 있었다.
  • Post와 User 분리: 둘이 섞여있는 것을 PostForm/UserForm 과 PostTable/UserTable 로 분리했다.
  • 타입의 명확화: any로 쓴것과 string 타입이라고 범위가 넓게 잡힌 타입들을 각 컴포넌트 도메인에 맞게 범위를 좁혔다.

사용한 기술 스택 경험

  • Shadcn + Tailwind + CVA: CVA를 이용해 variants를 설정하는데 참 유용하다 생각했다. 체계적으로 컴포넌트의 프로퍼티?를 세팅할 수 있게 한것 같다. Shadcn은 이미 다 세팅된 컴포넌트를 가져와서 거의 그대로 쓰는것이다보니 편리하고, 확실히 커스텀하기가 수월하다는 생각이 들었다.
  • React hook form + zod: 폼 UI와 또 그것을 검증하는 로직이 분리된것을 보고 참 잘 만들었구나 싶다. 타입 추론도 훌륭하고...

어려웠던 점과 해결 방법

  • 디자인 토근 설계. 이론을 들어도 막상 실전으로 적용하고 시작하려니 막막했다. 머릿속에 bg-primary, bg-danger... 이거밖에 생각이 안났는데, AI의 도움과 KRDS 디자인패턴 사이트를 보고 많이 참고했다. 그래서 3단계 토큰으로 나누었다. palette.json -> semantic.json -> component.json
    • palette.json 에는 말 그대로 앱에서 쓰일 색상들 스테틱한 값을 정의한 것.
    • semantic.json 는 테일윈드처럼 bg-primary, bg-danger, text-neutral 등등 유틸같은 클래스를 정의한것. 여기서 palette.json 값을 참조한다!
    • component.json는 컴포넌트 레벨에서 해당 컴포넌트에서만 쓰일 클래스들을 정의한것. ex) badge-bg-primary, card-text-error 등등. 여기서는 주로 semantic.json 에서 정의한 클래스 값을 참조한다.
  • 설계한 토큰을 실제 css로 어떻게 가져올 것인가? 일일이 옮겨적는것은 아닌것 같고.. 그래서 찾아본 결과 Style Dictionary 라는게 존재했다. 내가 설계한 토큰 .js .json 파일을 .css로 변환해주는 라이브러리이다. 이를 이용해 빌드할 때 자동으로 css로 변환해 기존 components.css에 포함되도록 만들었다 (생각보다 변환 빌드 과정이 복잡했는데, AI로 뚝딱했다)
  • 토큰 light/dark theme을 구분하면서 문제가 발생했다! 원래 palette.json 은 한가지 색으로하고, semantic.json 을 light와 dark로 구분했었는데, dark 환경에서 light 색상을 쓰게 되면서 눈뽕이슈가 발생해버렸다. 부랴부랴 palette.json을 Light/dark 두가지 색상 버전으로 이분화했다. semantic.json 은 그럼 이분화 하지 않아도 될 것 같긴 한데, 시간 부족 이슈로 그대로 dark, light 구조로 이분화시켜두었다,,,
