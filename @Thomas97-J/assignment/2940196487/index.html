질문

Q. medium.useEventOperations.spec.tsx > 아래 toastFn과 mock과 이 fn은 무엇을 해줄까요?

enqueueSnackbar을 뜻하는 게 맞다면 목 내부에서 실행되는 함수를 spy해 대체한다고 이해했다. 이를 통해 직접 상태가 공유되지 않는 목킹함수 대신 성공 실패 여부를 따져보는 용도로 사용할 수 있었다.

Q. medium.integration.spec.tsx > 여기서 ChakraProvider로 묶어주는 동작은 의미있을까요? 있다면 어떤 의미일까요? 해당부분 구현을 못했습니다...

Q. handlersUtils > 아래 여러가지 use 함수는 어떤 역할을 할까요? 어떻게 사용될 수 있을까요?

미디움의 구현 기반으로는 각 함수별로 상태를 공유하지 않아, 업데이트한 결과를 패치로 확인하는 등의 활용은 할 수 없었다. 이를 각 함수들이 상태를 공유하도록 수정해보고 싶었으나, 부족한 시간으로 인해 시도할 수 없었다.

Q. setupTests.ts > 왜 이 시간을 설정해주는 걸까요?

테스트가 시간이 지나 틀어지는것을 막기 위해 2025년 10월1일로 시스템 시간을 고정하는 것으로 이해했다.

과제 셀프회고

기술적 성장

제대로 완주하지 못한 과제에도 성장의 가능성은 있다고 생각한다. 엉망진창으로 작성한 코드를 통해 멀게만 느껴졌던 테스트 코드가 가깝고 안쓰럽게 느껴졌다. 내 코드를 보며 다음에는 이것보다는 더 잘 할 수 있겠다고 생각하며 약간의 자신감을 얻었다고 하면 이상한가? 나 역시 그렇다.

코드 품질

코드 품질은 심히 엉망이라고 할 수 있겠다. 살면서 이렇게 정리 안된 코드를 짜본게 얼마나 오랜만인가 하는 생각이 들 정도였다. 세션을 통해 테스트 코드 안에서 복사 붙여넣기를 하는 것이 보통의 코드에서 코드량을 늘리는 것과는 조금 다른 의미라는 것을 배웠으니, 이정도까지도 용납이 되는걸까?를 고민하게 되었다.

학습 효과 분석

완주에 목표를 두었고, 완주조차 하지 못했으나, 퇴근하고 다시 노트북을 잡는 일에 점차 적응하고 있다. 이것만으로 절반의 성공이라고 생각한다. 또 테스트 코드는 조금 거창한 것 이라고 생각했는데, 엉망이지만 어쨌든 내 손으로 작성되는 코드들을 보며 막연한 두려움을 조금 덜 수 있었다. 과제에 투자할 시간을 조금 더 내어야 할 것 같다는 생각이 들었다.

과제 피드백

개인적인 사정으로 퇴근하고 두시간에서 세시간 정도만을 투자할 수 있었는데, 멘토링이나 세션이 있는 날이면 더 과제에 시간을 사용하기 힘들었고, 시간이 많이 부족하게 느껴졌습니다. 새삼 상당히 안일한 마음으로 코스를 신청했구나 하는 생각이 들었습니다. 그러나 타성에 젖은 생활에서 벗어나기 위해서는 이렇게 조금 속상하고 빠듯한 것이 좋지 않을까? 라는 느낌입니다. 통합테스트부터는 구현하지 못해 아쉬움이 남지만 다양한 유닛테스트를 작성해보며 이리 저리 생각해볼 기회가 되었습니다.

리뷰 받고 싶은 내용

'새로 정의된 'title', 'endTime' 기준으로 적절하게 일정이 업데이트 된다' 에 대한 테스트 과정에서 useEventOperations의 onSave 함수를 사용해서 테스트를 할 수 있지 않을까 하는 생각이 들었습니다. 일반적으로 테스트 대상이 되는 코드에 enqueueSnackbar처럼 스파이 하기 좋은 함수가 항상 딸려있지는 않을 것 같은데 이런경우의 테스트는 어떻게 진행되는지 궁금합니다. 목킹을 조금 더 고도화하는 방법으로 가게 되는지, 그 외의 방법이 있을까요?

과제 피드백

민수님 한 주간 고생 많으셨습니다 작성해주신 테스트를 보니 각 순서에 맞게 필요한 코드들을 잘 작성해주셨어요 어느정도 자주 작성을 하다보니 어떤 패턴으로 테스트가 운영이 되는건지 감을 잡으셨으면 좋았겠네요 ㅎㅎ 이벤트는 필요에따라 5개나 운영을 하고 있는데, 이게 꼭 다 필요한 이벤트인지는 고민해보면 좋을것 같습니다. 컴포넌트 테스트도 꼭 작성해보시면 좋을것 같아요!!

'새로 정의된 'title', 'endTime' 기준으로 적절하게 일정이 업데이트 된다' 에 대한 테스트 과정에서 useEventOperations의 onSave 함수를 사용해서 테스트를 할 수 있지 않을까 하는 생각이 들었습니다. 일반적으로 테스트 대상이 되는 코드에 enqueueSnackbar처럼 스파이 하기 좋은 함수가 항상 딸려있지는 않을 것 같은데 이런경우의 테스트는 어떻게 진행되는지 궁금합니다. 목킹을 조금 더 고도화하는 방법으로 가게 되는지, 그 외의 방법이 있을까요?

결국 모킹하는 방식은 다 거기서 거기라고 생각하는 편인데요. 지금처럼 스파이 하기 좋은 함수를 넣기 좋게 만드는 방법도 있을것 같고, 모듈 자체에 대한 모킹을 재정의할수도 있구요. 내가 작성하는 코드라면 테스트하기 좋은 검증방향으로 리팩토링할 수도 있구요. 말씀하시는것처럼 고도화라기보다는 약간약간씩 틀어서 다른 방법으로 하는거니 크게 걱정하지 않으시고 그 상황이 닥치면 고민해보는걸로 하셔도 충분할겁니다 ㅎㅎ
