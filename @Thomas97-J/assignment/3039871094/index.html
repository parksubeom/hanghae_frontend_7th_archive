과제 목표

레거시 코드베이스를 현대적인 디자인 시스템으로 개편하는 실무 경험

  1. 정리되지 않은 레거시 코드의 문제점 식별 및 분석
  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

Before 패키지 분석 후 After 패키지 개편

개편 목표

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

필수 과제

  • TailwindCSS 설정 및 디자인 토큰 정의

  • CVA를 활용한 variants 패턴 적용

  • 스타일링, 컴포넌트 설계, 폼 관리 측면에서 문제점 파악
  • 개선이 필요한 부분과 그 이유 정리

3. 컴포넌트 개편

  • UI와 비즈니스 로직 분리
  • 순수한 UI 컴포넌트로 재구성
  • 일관된 컴포넌트 API 설계
  • 적절한 컴포넌트 구조 설계

과제 회고

목요일 회식이 예정되어 있어 수요일까지 과제를 마치는 것을 목표로 잡았다. 때문에 소스코드 분석 레벨부터 ai를 적극적으로 활용했다. 덕분에 수요일 저녁에 과제를 마무리 할 수 있었고, 지금 나는 PR을 적고있으니 성공적인 계획이라 할 수 있겠다. 과제를 진행하며 느낀 것은 많은 제약이 결국 일관된 스타일을 만들고, 높은 재사용성을 만든다는 것이었다. cva를 사용해 베리언트를 추가하며 이렇게 복잡하게 조건부 스타일을 세팅해야 할까? 이정도 프로젝트를 위해 이렇게까지 코드를 늘려야 할까? 하는 불편함을 계속 느꼈지만, 작업이 후반부로 갈 수록 이런 제약들이 스토리북을 작성하고, 이 ui 컴포넌트가 사용되는 패턴을 고정하는 효과가 있다는 것을 느낄 수 있었다.

우리 회사에 유지보수되지 않는 UI 라이브러리가 있다. 토요일 세션에서 언급된 그 라이브러리인데, 비슷한 디자인으로 구현된 여러개의 프로덕트의 스타일을 통일시키고 재사용성을 높히고자 만들어진 라이브러리지만 유지보수가 되지 않아 지금은 이미 추가된 아이콘들을 간단하게 꺼내 쓰는 용도 정도로만 사용되고 있다. 개발 당시부터 디자인시스템에 대해 깊은 고려가 없이 진행된 터라 ui 라이브러리 속 컴포넌트들이, 기존 컴포넌트가 사용하던 옵션들, 예를들어 버튼의 경우 도메인에 대한 정보를 옵션으로 받아 정해진 스타일을 리턴하는 등 순수한 ui 컴포넌트로 관리되지 않고 있는 상황이었다. 이번 과제를 진행하며 기존 ui 라이브러리가 정확히 어떤 점에서 문제인지 조금 더 명확하게 이해할 수 있었다. 아직 해결 방안까지는 떠오르지 않지만, 이걸 고친다면 어떻게 고칠 것인지, 제거한다면 어떻게 제거할 것인지 갈피를 잡아가고 있다.

Before 패키지에서 발견한 문제점

우선 ManagementPage 컴포넌트가 atoms, molecules, organisms의 컴포넌트를 가져다 사용하고 있었지만. 각 페이지의 기능별 섹션으로는 전혀 컴포넌트가 나누어져 있지 않았다. 이를 organisms 에 추가해야 되나 생각했지만, 공용으로 사용하지 않는 컴포넌트를 components 폴더 하단에 추가하고 싶지 않아, YAGNI을 지키기 위해 ManagementPage를 쪼개고 그 하위에 폴더를 만들어 넣었다. 명확하지 않은 아토믹 패턴 등의 이야기는 세션과 자료에서 다루었으므로 별도로 기록하지 않겠다.

개편 과정에서 집중한 부분

작업을 시작할때 만들었던 chadcn을 어떻게 하면 더 많이 사용할 수 있을까를 고민했다. 레이아웃이 눈에 보이지 않는 래퍼들을 제외하고 대부분의 컴포넌트들을 이 구조는 미리 정의한 chadcn 컴포넌트 중 어디에 해당할까를 생각하며 작업을 하다 보니 조금 불편하게 느껴졌지만, 제약 안에서 디자인을 하고 레이아웃을 구현해야 일관적인 형태를 만들 수 있다는 것을 이해할 수 있었다.

사용한 기술 스택 경험

오랜만에 테일윈드를 사용하니 페이지 레이아웃을 잡는데만도 꽤 많은 시간을 사용했다. 거기에 더해 레거시에서 사용하던 components.css 의 css 중 와일드카드로 지정된 패딩, 마진 0 스타일이 테일윈드를 덮어써 스타일링을 방해해 애를 먹었다. 분명 와일드카드보다 클래스로 특정한 스타일이 더 높은 우선순위를 가져야 할것이라 생각했는데, 찾아보니 테일윈드와 css를 섞어서 사용할 때는 @layer 키워드 세팅을 제대로 해야 한다는 것을 알 수 있었다. 이 경우에서는 이미 사용하지 않던 css가 불필요한 영향을 준 것이므로 삭제했지만, 앞으로 비슷한 경우를 만날 때 도움이 될 것 같다.

어려웠던 점과 해결 방법

테스트 코드를 통과하는 과정에서 select 관련 이슈가 있었다. 세션 중 공유된 내용을 기반으로 네이티브 셀렉터를 사용하도록 해서 이슈를 해결했다.

리뷰받고 싶거나 질문하고 싶은 내용

디자이너들이 자주 바뀌었던 회사라, 어드민 시스템에도 여기저기 디자인이 다른 상황입니다. 이미 기준이 되는 디자인이 없으니 새로 온 디자이너들은 다시 기존 디자인이 마음에 들지 않는다며 조금씩 다른 디자인을 제시할 때가 많고, 이럴 때마다 이것과 가장 유사한 용도, 디자인을 기존 페이지에서 찾아 이것으로 대체하는 것이 어떠냐고 제안을 하는 것이 일상이 되었습니다. 디자인시스템이 확립되지 않은 회사에서 디자인 시스템을 도입하려면 무엇부터 시작하는게 좋을까요? 어떤 공통의 룰을 정한다거나, 사용하고 있는 컴포넌트들을 정리해 스토리북을 만드는 것이 선행되면 좋겠다고 막연히 생각하고 있습니다. 디자이너분들과 사이는 좋아 무엇인가를 시도해보고자 하면 같이 시도할 수는 있을 것 같습니다. 감사합니다.

과제 피드백

잘했어요 민수! 이번 과제는 레거시 코드를 현대적인 디자인 시스템으로 개편하면서 최신 기술의 방향성을 체득하는 것이 목표였습니다. AI를 적극 활용해 빠르게 완료한 것도 좋지만, 더 인상적인 건 과제를 진행하면서 "많은 제약이 결국 일관된 스타일을 만든다"는 걸 체득한 부분이 좋네요. 모든 기술들은 저마다의 이유가 있는데 그걸 이론적으로가 아니라 진짜로 그렇구나 이해하는건 어려운거라서요.

그렇다고 해서 이 모든 기술들이 내 실무에도 도움이 되거나 공감대가 있는 기술이 아닐수도 있습니다. 그래서 기술의 이점과 원칙과 실무에서의 제약과 상황을 항상 함께 같이 고려를 해야 하는게 개발자의 기술결정 능력 부분에서도 중요한 능력이라고 생각합니다.

Q) 디자인시스템이 확립되지 않은 회사에서 디자인 시스템을 도입하려면 무엇부터 시작하는게 좋을까요?

그래서 디자인 시스템을 도입하다는 생각보다는 우선 필요의 공감과 함께 이 과정이 모두에게 도움이 되어야 하도록 만들어야 합니다. 아무도 나에게 도움이 되지 않은 업무가 추가가 되는 건 반기지 않으니까요. 또한 문제에 대한 공감대를 가지는 것과 실제로 업무외에 그 작업을 할건가 라는 건 또 다릅니다. 그래서 업무를 기왕하는 김에 더 편해지는 방향으로 가기 위해 하는 거라는 인식과 방향을 만들어야 해요.

디자이너들은 다양한 디자인을 하고 많이 개편을 해보고 싶어해요. 그러니까 디자인 수정을 제한하는 형태로 느껴지는게 아니라 더 디자인을 잘 반영하기 위해서 하는 거라는 입장을 취하면서도 "제약과 약속"이 더 좋은 디자인을 만든다라는 부분들을 함께 이야기를 나누면서 디자인은 자유롭게 그러나 디자인에 쓰이는 숫자나 팔레트등은 통일하고 정리해보자는 식으로 논의를 해보면 좋겠습니다. 개발자가 농담으로 17px은 안 쓰고 싶다 하면 그런 공감대는 디자이너도 어떤 건지 아니까 공감을 해주더라구요.

저도 이런식으로 우선 수치나 색상등을 제한하고 토큰화 시키면 디자인도 하기 편해지고 나중에 수정하기도 편해진다고 figma 토큰 시스템을 알려드렸고 이러면서 다크모드 자동 전환 같은 것도 보여드리면서 토큰에 익숙해지면 variant나 component의 네이밍이나 종류등을 정해두고 디자인을 다양하게 만드는 방식으로 함께 진행했었습니다. 스토리북도 좋지만 디자인 시스템은 사실 디자이너들의 것이리기에 디자인의 체계가 정리가 되는게 먼저라고 생각해요.

이후에 디자인의 체계가 갖춰지면 스토리북으로 만들어서 보여주면서 variant등도 알려주면서 뭔가가 갖춰져 가는 것들을 보여주면 디자이너들도 뿌듯하고 이게 포트폴리오가 될 수 있다는 점도 어필하면 나중에는 함께 또 으샤으쌰가 되고 그랬습니다
