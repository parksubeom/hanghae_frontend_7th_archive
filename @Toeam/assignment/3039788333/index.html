과제 목표

레거시 코드베이스를 현대적인 디자인 시스템으로 개편하는 실무 경험

  1. 정리되지 않은 레거시 코드의 문제점 식별 및 분석
  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

Before 패키지 분석 후 After 패키지 개편

개편 목표

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

필수 과제

1. 디자인 시스템 구축

  • TailwindCSS 설정 및 디자인 토큰 정의
  • shadcn/ui 컴포넌트 설치 (Button, Input, Select, Card, Table 등)
  • CVA를 활용한 variants 패턴 적용
  • 일관된 스타일 시스템 구축

2. Before 패키지 분석

  • Before 패키지 실행 및 전체 코드 탐색
  • 스타일링, 컴포넌트 설계, 폼 관리 측면에서 문제점 파악
  • 개선이 필요한 부분과 그 이유 정리

3. 컴포넌트 개편

  • UI와 비즈니스 로직 분리
  • 순수한 UI 컴포넌트로 재구성
  • 일관된 컴포넌트 API 설계
  • 적절한 컴포넌트 구조 설계

심화 과제

  • Dark Mode 완전 지원 (CSS Variables + Tailwind)
  • Design Token 시스템 고도화 (색상 팔레트, 타이포그래피 스케일)
  • 뷰와 비즈니스로직이 분리되도록

과제 회고

Before 패키지에서 발견한 문제점

Automic Design 패턴

  • automic design 구조를 사용하고 있지만 각각의 역할 이 명확하지 않다 ex) UI 컴포넌트임에도 도메인 로직 포함, 렌더링 로직의 하드코딩 등

관심사의 분리 부족

  • UI 컴포넌트가 비즈니스 로직과 도메인 지식을 포함하고 있어 컴포넌트의 재사용성이 떨어지고 테스트가 어려우며 유지보수 비용이 증가한다.

단일 책임 원칙 위반

  • 하나의 컴포넌트가 여러 책임을 가지고 있음

스타일링 방식의 불일치

  • 인라인 스타일과 css클래스가 혼합되어 사용되고 있다.

css 파일 구조의 문제

  • 하나의 파일에 모든 컴포넌트 스타일이 포함되어 있음
  • 스타일을 찾기가 어려워 유지보수가 어려우며 충돌 가능성이 높다.
  • 색상, 간격, 폰트 크기 등 하드코딩이 되어 있어 유지보수의 어려움이 있다.

반응현 디자인의 부재

  • 반응형 디자인이 고려되어있지 않다.

스타일 일관성의 부족

  • 크기 단위 혼용
  • 간격 시스템의 부재
  • 컴포넌트별로 다른 네이밍 컨벤션

구조적 문제점

페이지 컴포넌트의 과도한 책임

  • ManagementPage.tsx 가 너무 많은 기능을 가짐

상태 관리의 복잡성

  • 관련된 상태들이 분산되어 있음
  • 상태 업데이트 로직의 중복

디렉토리 구조 문제

  • 유틸리티 함수 분리의 부족
  • Hooks 디렉토리 부재

코드중복

  • 특정 로직들이 반복되어 있다.

재사용 불가능한 컴포넌트

  • 컴포넌트들이 특정 도메인에 결합되어있다.
  • 컴포넌트의 역할이 모호하며 독립적으로 사용하기 어려운 구조를 가지고 있음.

개편 과정에서 집중한 부분

  • 일단 먼저 디자인 시스템 구조에서 유지보수의 편리함을 제일 우선시로 생각하여 진행하였다. components.css파일에 하드 코딩된 값들이 많아 ai를 사용하여 디자인 토큰화를 실행하여 1차적으로 정리를 하였다. 그 후에 각각의 컴포넌트를 살펴보며 어떤식으로 개선을 할 지 고민을 하였다. 우선 스타일에 있어서 재사용 가능성이 높고 유지보수를 하는데 편리함을 우선시 하여 css의 나머지 것 들을 전부 토큰화를 하였고 shadcn을 통해 props에 따라 각각의 컴포넌트들의 스타일들이 반영되도록 작성하였다.
  • 이어서 컴포넌트에서 기능들을 분리시켜 단일 책임 원칙을 지키는 것을 그 다음 우선 순위로 잡아 구조를 나누고 그 후에 리팩토링을 할 계획이었지만 이번주 일정이 바빠서 진행을 하지 못하였습니다.

사용한 기술 스택 경험

  • 직장에서는 tailwindcss와 NextUI를 사용한 적만 있고 shadCn, cva에 대한 경험이 없어서 이번 기회에 알게 되었다. 또한 디자인 토큰 작성 방식의 3단계 패턴을 알게 되었다. shadcn을 작성하면서 디자인 토큰화만 잘 진행되어 있다면 shadcn의 varient의 경우만 잘 분리한다면 초기 작성에만 시간이 걸리고 그 이후에는 유지보수도 쉽고 재사용성이 높아서 되게 편리하다는 생각이 들었다.

어려웠던 점과 해결 방법

  • 초반에 디자인 토큰을 생성할 때 어떤 식으로 작성을 해야될 지 방향성을 못잡아 고민을 많이 하고 문서들도 많이 찾아봣지만 primitive, semantic, component 이 3단계로 나눠서 사용하기엔 현재 프로젝트의 구조상 primitive의 단계로 작성만 한다면 충분히 이용이 가능할 것 같아서 (3단계로 나누기엔 낭비(?)라는 생각이 들었다.) 기본 값들을 디자인 토큰으로 치환하는 방식으로 진행하였습니다.

잘했어요 새듬! 이번 과제는 레거시 코드를 현대적인 디자인 시스템으로 개편하면서 컴포넌트 설계 원칙을 체득하는 것이 목표였습니다. Before 패키지 분석이 꼼꼼했어요. Atomic Design 구조를 쓰고 있지만 역할이 명확하지 않다는 점, 관심사 분리 부족, 단일 책임 원칙 위반, 스타일링 방식 불일치까지 문제점을 체계적으로 정리했네요.

"디자인 토큰화만 잘 되어 있다면 shadcn의 variant만 잘 분리하면 초기 작성에만 시간이 걸리고 이후에는 유지보수도 쉽고 재사용성이 높다"는 깨달음도 좋았습니다. 이게 디자인 시스템의 핵심이에요. 처음에 체계를 잡는 데 투자하면 나중에 그 이상으로 돌려받는 거죠.

다만 아쉬운 부분이 있어요. 토큰을 3단계(primitive → semantic → component) 대신 primitive만으로 충분하다고 판단했는데, 이번 과제에서는 오히려 2단계까지 가보는 게 학습에 도움이 됐을 거예요. --blue-600만 있으면 "이 파란색이 버튼용인지, 링크용인지, 브랜드 컬러인지" 코드만 봐서는 알기 어렵거든요. --color-brand-primary: var(--blue-600)처럼 semantic 레이어를 하나 두면 의도가 명확해지죠. 실무에서 "이 색상 바꿔주세요" 할 때 primitive만 있으면 모든 사용처를 찾아다녀야 하지만, semantic이 있으면 한 곳만 바꾸면 되니까요.

그리고 컴포넌트 분리와 리팩토링을 시간 부족으로 못 했다고 했는데, 이 부분이 사실 이번 과제의 핵심이었어요. UI 컴포넌트가 도메인 로직을 품고 있을 때 어떤 문제가 생기는지, 분리하면 어떻게 달라지는지를 직접 경험해보는 게 목표였거든요. 다음 과제에서는 엔티티별로 컴포넌트를 분리하는 관점을 꼭 경험해보길 바랍니다.
