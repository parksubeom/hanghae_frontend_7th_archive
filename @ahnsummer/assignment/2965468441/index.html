과제 체크포인트

필수 스펙

    1. 반복 유형 선택
    • 일정 생성 또는 수정 시 반복 유형을 선택할 수 있다.
    • 반복 유형은 다음과 같다: 매일, 매주, 매월, 매년
      • 31일에 매월을 선택한다면 → 매월 마지막이 아닌, 31일에만 생성하세요.
      • 윤년 29일에 매년을 선택한다면 → 29일에만 생성하세요!
    • 반복일정은 일정 겹침을 고려하지 않는다.
  1. 반복 일정 표시
    • 캘린더 뷰에서 반복 일정을 아이콘을 넣어 구분하여 표시한다.
  2. 반복 종료
    • 반복 종료 조건을 지정할 수 있다.
    • 옵션: 특정 날짜까지
      • 예제 특성상, 2025-12-31까지 최대 일자를 만들어 주세요.
  3. 반복 일정 수정
    1. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      • 반복일정을 수정하면 단일 일정으로 변경됩니다.
      • 반복일정 아이콘도 사라집니다.
    2. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      • 이 경우 반복 일정은 유지됩니다.
      • 반복일정 아이콘도 유지됩니다.
  4. 반복 일정 삭제
    1. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      1. 해당 일정만 삭제합니다.
    2. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      1. 반복 일정의 모든 일정을 삭제할 수 있다.

과제 진행 로그

에이전트를 체계적으로 구축하기 위해 먼저 프로젝트의 요구사항을 파악하고 프로젝트를 어떤 순서로 진행할 것인가, 어떤 에이전트들을 구성할 것이고 어떤 프로세스를 거쳐 작업이 진행되게 할 것인가, 그리고 어떤 산출물들을 산출해내게 할 것인가를 계획했습니다.

예시 : note.md 내용 일부

- Agent 구현 명세 문서 또는 코드

과제의 요구사항

과제는 최종적으로 캘린더의 반복기능을 만드는 것이었으나, 본질은 정확하게 동작하는 에이전트 구현에 있었다고 생각합니다. 함수가 어떤 input에 대해 특정한 output을 반환하듯이 에이전트에게도 어떠한 input을 줬을 때 그에 대응되는 output을 줄 수 있는 구조를 만드는 게 가장 중요한 목표라고 생각했습니다.

과제 풀이의 핵심 컨셉

저는 이번 과제를 진행하면서 이상적인 모습의 Top-Down 회사의 업무 프로세스를 생각했습니다. 이런 환경에서 실무자들은 업무를 진행하고, 이에 대한 업무일지와 함께 상사에게 결재를 받게 되는데요, 비슷하게 에이전트들이 각 업무를 수행한 이후 worklog를 작성하게 하여 이를 통한 업무 내역 확인 및 다음 작업자에게 업무 인수인계를 할 수 있도록 하였습니다.

또한 업무 프로세스를 명확히 짜서 안정적인 업무 퍼포먼스를 낼 수 있도록 각 에이전트가 업무를 진행하는 단계별 작업, 각 단계에서 지켜야 하는 지침들, 산출해야 하는 산출물의 양식과 예시 등을 제공하고 각 단계의 성공 여부 검증을 위한 체크리스트를 제공하여 안정성을 높이려 했습니다.

파일 분리

에이전트에 대한 지침, 에이전트가 작업을 진행하는 각 스텝에 대한 지침, 에이전트가 참고해야 하는 산출물 및 예시 등을 파일로 각각 분리하여 링크하는 형태로 구성하였습니다. 이는 아래와 같은 이점이 있을 것이라고 예상하고 진행하였습니다.

  • 파일 분리를 통한 가독성 개선
  • 에이전트로 하여금 작업을 수행하면서 필요한 파일들만 참조하도록 유도 -> 비용 절감 및 컨텍스트의 단축으로 정확도 개선
  • 여러 번 참조되는 내용의 경우 중복 코드 방지 및 SSOT의 역할

또한 해당 양식에 대한 예시를 제공하여 양식만으로 전달할 수 없는 텍스트의 톤과 같은 부분들을 에이전트가 이해할 수 있도록 하였습니다.

## 🔍 Root Cause 분석

**문제**: `getFilteredEvents` 함수가 반복 일정을 날짜 범위로 필터링하면서 제외시킴

**시나리오**:

1. 10월 3일 시작 → 11월 30일까지 반복되는 일정 생성
2. 이 일정의 `date` 필드는 `"2025-10-03"`
3. 11월 뷰로 이동 → `filterEventsByDateRangeAtMonth`가 11월 1~30일 범위로 필터링
4. `date: "2025-10-03"`는 11월 범위가 아니므로 **제외됨**
5. `filteredEvents`에 없으므로 `expandRecurringEvents`에 전달 안 됨
6. 결과: 11월에 일정이 표시되지 않음

**해결책**: 반복 일정(`repeat.type !== 'none'`)은 날짜 범위 필터링에서 제외해야 합니다.

수정하겠습니다.

  • 의미론적 태그: <identity>, <constraints> 등 목적이 명확한 태그
  • 속성을 통한 메타데이터: <step n="1">, <ok>true</ok>
  • 주석을 통한 설명: <!-- 이 섹션은... -->

  • 테스트 코드 작성 및 로직 구현 시에 다소 복잡한 로직을 작성해야 할 때 에이전트가 고전하는 경우가 많았는데, 저의 경우 이런 상황에서 대부분 의도적으로 동작을 중지시키고 디버깅을 위한 정보를 직접 제공해주는 형태로 문제를 해결했습니다. 에이전트가 고전하는 이유를 생각해봤을 때, 코드를 넓게 보지 못 하고 단편적으로 오류가 나는 부분만 확인하다보니 실제로 문제의 원인이 아닌 부분을 문제라고 인식하고 억지로 고치려고 하는 경우가 많았던 것 같습니다. 또한 이를 통해 새로운 이슈가 발생하기도 했구요. 프롬프트를 더 잘 작성했다면 에이전트가 Root Cause를 더 잘 정의하고 이러한 현상을 방지할 수 있었을 것 같은데, 이러한 AI의 시야가 좁아지는 현상..?과 관련하여 코치님께서 경험하신 노하우나 팁이 있는 지 궁금합니다!

AI와 테스트를 활용한 안정적인 기능 개발 리포트

사용하는 도구를 선택한 이유가 있을까요? 각 도구의 특징에 대해 조사해본적이 있나요?

Cursor를 선택한 이유

AI에게 개발을 "위임"하기보다 "협업"하는 방식이 안정성과 효율성 측면에서 더 나은 접근이라고 생각했습니다.

CLI 도구(Aider, Cursor CLI 등)는 전체 작업을 AI에게 온전히 맡기는 형태로, 코드 리뷰나 중간 개입이 어렵고 잘못된 방향으로 진행될 경우 되돌리기가 어렵다고 판단했습니다.

반면 Cursor(IDE + AI 통합)는 다음과 같은 장점이 있다고 생각했습니다:

  • 실시간 코드 확인: AI가 코드를 작성하는 중간에도 파일을 직접 열어 확인할 수 있습니다.
  • 컨텍스트 관리: @파일명, @폴더명 등을 통해 선택적으로 컨텍스트를 제공할 수 있습니다.
  • 단계별 개입: 에이전트가 각 스텝을 완료할 때마다 리뷰하고 피드백할 수 있습니다.
  • 협업 워크플로우: 보고 → 컨펌 → 실행 사이클을 통해 인간-AI 협업이 가능합니다.

실제로 이번 과제에서 Implementation Engineer가 달력 이동 시 반복 일정이 표시되지 않는 이슈를 혼자 해결하지 못했을 때, "useSearch에서 문제가 있을 것 같다"고 힌트를 주자 바로 Root Cause를 찾아낸 경험이 있었습니다. 이러한 협업 방식은 CLI 도구로는 구현하기 어렵다고 생각합니다.

테스트를 기반으로 하는 AI를 통한 기능 개발과 없을 때의 기능개발은 차이가 있었나요?

워크플로우 관점

테스트 없이 개발했을 때는 "사용자 → AI 지시 → 코드 생성 → 수동 테스트 → 버그 발견 → 수정 요청 → 다른 부분 파손 → 무한 루프"의 형태였습니다.

반면 TDD와 에이전트 시스템을 도입한 이번 과제에서는 "PO가 PRD 작성 → QA Engineer가 테스트 케이스 문서 작성 → Test First Engineer가 테스트 코드 작성(RED) → Implementation Engineer가 로직 구현(GREEN) → Refactoring Engineer가 코드 개선(REFACTOR)"의 명확한 단계를 거쳤습니다.

완료 기준 관점

테스트가 없을 때는 "잘 동작하는 것 같은데?"라는 주관적 판단에 의존했습니다. 테스트 기반 개발에서는 "172개 테스트 모두 통과"라는 객관적이고 명확한 완료 기준이 있었습니다.

안정성 관점

리팩토링 단계에서 150줄의 중복 코드를 제거했을 때, 테스트가 있어서 안심하고 진행할 수 있었습니다. 리팩토링 후에도 172개 테스트가 모두 통과하여 기존 기능이 파손되지 않았음을 즉시 확인할 수 있었습니다.

AI 정확도 관점

테스트가 있으면 AI가 "무엇을 구현해야 하는지" 명확히 이해할 수 있었습니다. 특히 경계값 케이스(매월 31일, 윤년 2/29)를 테스트로 명시하니 AI가 정확히 구현했습니다.

디버깅 효율성 관점

테스트 실패 메시지가 정확한 원인을 알려주었습니다. 예를 들어 "Expected: ["2025-01-31", "2025-03-31", "2025-05-31"], Received: ["2025-01-31", "2025-02-31", "2025-03-31"]"라는 메시지를 통해 2월 31일이 포함되어선 안 된다는 것을 즉시 파악할 수 있었습니다.

AI의 응답을 개선하기 위해 추가했던 여러 정보(context)는 무엇인가요?

고도화 방안: 각 에이전트의 성공/실패 이력을 수집하여 제약사항과 단계를 점진적으로 개선할 수 있을 것 같습니다.

장점: 각 단계가 파일로 분리되어 있어 여러 에이전트가 공통 스텝을 재사용할 수 있었습니다. 또한 특정 단계에서 문제가 발생하면 해당 지침만 개선하면 되었습니다.

고도화 방안: 각 단계의 성공률을 측정하여 어떤 지침이 효과적인지 정량적으로 평가할 수 있을 것 같습니다.

<constraints>

  <constraint>테스트 실행 시간이 합리적이어야 함 (< 5초)</constraint>

<success-criteria>

  <criterion>테스트 실행 시간이 합리적</criterion>
  <criterion>커버리지가 목표치 이상 (선택사항)</criterion>
</success-criteria>

채택 이유: 각 단계의 완료 여부를 객관적으로 검증할 수 있는 기준을 제공하기 위해 제약사항과 성공 기준을 명시했습니다.

장점: 에이전트가 작업 완료를 주관적으로 판단하지 않고 명확한 기준에 따라 작업을 완료할 수 있었습니다.

고도화 방안: 성공 기준 항목에 자동 검증 스크립트를 연결하여 수동 확인 없이도 완료 여부를 판단할 수 있을 것 같습니다.

파일 분리 전략

처음에는 하나의 파일에 모든 내용을 담았지만(예: implementation-engineer.md), 이를 에이전트 정의, 단계별 지침(steps/), 좋은/나쁜 예시(examples/)로 분리했습니다. 이를 통해 AI가 필요한 부분만 참조하여 컨텍스트 길이가 단축되었고, 여러 에이전트가 공통 스텝을 재사용할 수 있었으며, SSOT(Single Source of Truth)를 유지할 수 있었습니다.

선택적 컨텍스트 제공

전체 .cursor/ 폴더를 참조하도록 하는 대신, 필요한 파일만 정확히 지정하도록 했습니다. 예를 들어 "@implementation-engineer.md 구현 개발자님, @docs/worklog/worklog-test-first-engineer-v1.md 를 참고하여 @src/tests/recurring-events.integration.spec.tsx 테스트를 통과시켜주세요"와 같이 작업에 필요한 파일만 지정했습니다. 이를 통해 불필요한 컨텍스트 로딩을 방지하고 AI가 관련 정보에 집중할 수 있었습니다.

명시적 제약사항 지정

초기에는 Implementation Engineer가 테스트를 GREEN으로 만드는 대신 테스트 코드를 수정해버리는 문제가 있었습니다. 이를 해결하기 위해 에이전트 정의에 <rule>테스트 코드를 절대 수정하지 마세요</rule>, <rule>리팩토링은 하지 마세요 (GREEN만 달성)</rule>과 같은 명시적인 규칙을 추가했습니다. 이러한 제약사항을 명확히 함으로써 에이전트가 작업 범위를 벗어나는 것을 방지할 수 있었습니다.

보고 및 컨펌 프로세스 도입

초기에는 AI가 혼자 판단해서 작업을 진행했는데, 중간에 멈추고 보고하도록 변경했습니다. "구현 전에 어떤 문제들이 있는지 리스트업 하고 해결 방안을 제시해서 컨펌 받으세요"와 같이 지시하여, AI가 작업을 시작하기 전에 이해한 내용을 확인받도록 했습니다. 이 과정을 통해 AI가 잘못된 방향으로 작업하는 것을 사전에 방지할 수 있었습니다.

예시 기반 학습 활용

단순히 "PRD 작성해줘"라고 지시하는 대신, templates/prd/prd.md.hbs 템플릿과 templates/prd/prd.md.example 예시를 함께 제공했습니다. AI가 예시를 참고하여 톤, 구조, 상세도 수준을 학습할 수 있었고, 이를 통해 일관된 품질의 문서를 생성할 수 있었습니다.

점진적 개선 프로세스

에이전트 지침과 산출물도 버전 관리하면서 개선했습니다. 예를 들어 worklog-implementation-engineer-v1.md부터 worklog-implementation-engineer-v6.md까지 각 버전에서 발생한 문제점을 분석하여 다음 버전에서 개선했습니다. 이러한 반복적인 개선 과정을 통해 에이전트의 성능을 점진적으로 향상시킬 수 있었습니다.

템플릿 및 지침 준수 평가

AI의 결과물을 평가할 때 가장 중요하게 본 기준은 제공한 템플릿과 지침을 얼마나 잘 따랐는가였습니다.

산출물 형식 준수 여부

각 에이전트가 생성한 산출물(PRD, Worklog, 테스트 케이스 문서 등)이 템플릿에서 정의한 형식을 따르는지 확인했습니다. 예를 들어 Worklog의 경우 "작성자", "업무 지시 내용", "참고자료", "산출물", "업무 과정", "다음 작업자에게 남기는 코멘트" 섹션이 모두 포함되어 있는지 체크했습니다.

초기에는 일부 섹션이 누락되거나 순서가 바뀌는 경우가 있었습니다. 이럴 때는 템플릿에 각 섹션의 필수 여부와 순서를 더 명확히 명시하도록 개선했습니다.

작업 단계 준수 여부

에이전트가 정의된 단계를 순서대로 수행했는지 확인했습니다. 예를 들어 Implementation Engineer의 경우 "PRD 확인 → 테스트 실패 원인 분석 → 로직 구현 → 테스트 통과 확인" 단계를 따라야 합니다.

초기에는 단계를 건너뛰거나 순서를 바꾸는 경우가 있었습니다. 이를 개선하기 위해 각 단계마다 명시적인 체크리스트를 추가하고, 다음 단계로 넘어가기 전에 현재 단계의 완료 여부를 확인하도록 지침을 강화했습니다.

제약사항 준수 여부

에이전트가 정의된 제약사항을 지켰는지 확인했습니다. 예를 들어 Implementation Engineer는 "테스트 코드 수정 금지", "리팩토링 금지(GREEN만 달성)" 제약을 따라야 합니다.

초기에는 Implementation Engineer가 테스트를 통과시키기 위해 테스트 코드를 수정하는 문제가 있었습니다. 이를 해결하기 위해 <rule> 섹션에 제약사항을 더 명확히 명시하고, 제약사항 위반 시 경고를 표시하도록 개선했습니다.

산출물 품질 기준 준수 여부

각 산출물이 품질 기준을 만족하는지 확인했습니다. 예를 들어 PRD는 "사용자 시나리오 명시", "경계값 케이스 정의", "테스트 가능한 수용 기준" 등의 기준을 만족해야 합니다.

초기 버전의 PRD에서는 경계값 케이스가 누락되는 경우가 있었습니다. 이를 개선하기 위해 PO 에이전트의 지침에 "경계값 케이스는 반드시 포함해야 함"을 명시하고, 체크리스트에도 추가했습니다.

결과물 개선 프로세스

지침을 따르지 못한 경우에는 다음과 같은 과정을 거쳤습니다:

  1. 문제 파악: 어떤 지침을 따르지 못했는지, 왜 따르지 못했는지 분석했습니다.
  2. 지침 개선: 해당 지침을 더 명확하게 작성하거나, 예시를 추가했습니다.
  3. 재작업 요청: 개선된 지침을 바탕으로 에이전트에게 재작업을 요청했습니다.
  4. 검증: 재작업된 결과물이 지침을 잘 따르는지 다시 확인했습니다.

예를 들어 Worklog의 "다음 작업자에게 남기는 코멘트" 섹션이 형식적으로 작성되는 문제가 있었습니다. 이를 개선하기 위해 "구체적인 주의사항이나 인사이트를 3개 이상 포함해야 함"이라는 기준을 추가했고, 좋은 예시와 나쁜 예시를 함께 제공했습니다.

최종 만족도

대부분의 산출물이 템플릿과 지침을 잘 따랐습니다. 특히 반복적인 작업(테스트 코드 생성, Worklog 작성)에서는 일관된 품질을 유지할 수 있었습니다. 다만 복잡한 로직 구현이나 창의적인 문제 해결이 필요한 경우에는 여전히 인간의 개입과 방향 제시가 필요했습니다.

하나의 작업은 독립적으로 완료 가능해야 합니다. 예를 들어 "generateRecurringDates 함수 구현"은 독립적인 작업이지만, "generateRecurringDates의 윤년 처리 부분만"은 부분적인 작업이므로 적절하지 않습니다.

기준 2: 테스트 가능성

작업 완료 여부를 객관적으로 검증할 수 있어야 합니다. "이 테스트 파일이 통과하면 완료"는 명확한 기준이지만, "잘 동작하게 해줘"는 주관적이므로 적절하지 않습니다.

기준 3: 에이전트의 전문성

각 에이전트의 역할에 맞는 범위를 설정했습니다. 예를 들어 PO는 PRD 1개 작성, QA Engineer는 테스트 케이스 문서 1개 작성, Test First Engineer는 테스트 파일 1개 작성, Implementation Engineer는 유틸 모듈 1개 구현, Refactoring Engineer는 특정 코드 스멜 제거와 같이 각 역할의 특성에 맞는 작업 단위를 지정했습니다.

내가 생각하는 적절한 단위

레벨 1(함수 단위): generateRecurringDates 함수 구현과 같이 단일 함수를 구현하는 수준입니다.

레벨 2(모듈 단위): recurringUtils.ts 전체를 구현하는 것처럼 5~6개의 함수로 이루어진 모듈을 구현하는 수준입니다.

레벨 3(기능 단위): 반복 일정 생성 기능처럼 여러 파일이 연관된 기능을 구현하는 수준입니다.

레벨 4(에픽 단위): 반복 일정 전체 기능처럼 대규모 기능입니다. 이 경우 여러 에이전트에게 분할하여 위임하는 것이 적절합니다.

경험상 레벨 2~3이 가장 적절했습니다. AI가 충분히 맥락을 이해할 수 있으면서도 방향을 잃지 않을 정도의 범위였고, 테스트로 완료 여부를 명확히 검증할 수 있었습니다.

XML을 활용한 구조화된 지침

AI 에이전트에게 지침을 전달할 때 Markdown보다 XML을 사용하는 것이 더 효과적이었습니다. Cursor 공식 문서의 에이전트 예시에서도 XML을 활용한 구조화된 표현을 권장하고 있었는데, 실제로 적용해보니 AI가 역할, 제약사항, 단계를 더 명확히 이해하는 것을 확인할 수 있었습니다.

<agent>
  <name>PO</name>
  <role>PRD 작성</role>
  <constraints>
    <constraint>측정 불가한 목표 인정 안 함</constraint>
  </constraints>
</agent>

가독성은 Markdown이 더 좋지만, AI의 이해도와 정확도를 고려하면 XML이 더 적합하다고 판단됩니다.

컨텍스트 선택의 중요성

전체 프로젝트를 참조하도록 하는 것보다 필요한 파일만 정확히 지정하는 것이 중요합니다. "@src/utils/recurringUtils.ts 이 파일만 보고 @src/tests/unit/easy.recurringUtils.spec.ts 이 테스트를 통과시켜주세요"와 같이 작업에 필요한 최소한의 파일만 지정하면 AI가 관련 정보에 집중할 수 있고 비용도 절감할 수 있었던 것 같습니다.

Few-Shot Learning의 효과

단순히 규칙을 설명하는 것보다 좋은 예시와 나쁜 예시를 함께 제공하는 것이 훨씬 효과적이었습니다. Anthropic 공식 문서에서도 Few-Shot Learning을 권장하고 있는데, 실제로 적용해보니 AI가 의도를 정확히 파악하고 일관된 품질의 산출물을 생성하는 것을 확인할 수 있었습니다.

<examples>
  <success-case>
    <input>prd-login-v1.md</input>
    <reason>기능명과 버전이 명확</reason>
  </success-case>
  <failure-case>
    <input>prd-로그인-v0.md</input>
    <reason>
      1. 한글 사용 (영문 권장)
      2. 버전은 1부터 시작
    </reason>
  </failure-case>
</examples>

파일 분리 전략

하나의 큰 파일보다 역할별로 분리된 작은 파일들이 더 효과적이었습니다. 에이전트 정의(80줄), 단계별 지침(steps/), 예시(examples/)로 분리하여 AI가 필요한 부분만 참조하도록 했습니다. 이를 통해 컨텍스트 길이를 단축하고 여러 에이전트가 공통 스텝을 재사용할 수 있었습니다.

AI가 잘하는 것과 못하는 것에 대해 고민한 적이 있나요? 내가 생각하는 지점에 대해 작성해주세요.

AI가 잘하는 것

1. 반복적이고 패턴화된 작업

템플릿과 예시를 제공하면 일관된 품질로 산출물을 생성할 수 있었습니다. 이번 프로젝트에서 6개 에이전트가 각각 여러 버전의 worklog를 작성했는데 모두 일관된 형식을 유지했습니다.

2. 코드 생성 및 보일러플레이트

90개 사용자 시나리오를 21개 통합 테스트로 변환하고, Given-When-Then 구조를 일관되게 유지하며, 반복적인 setup/teardown 코드를 자동으로 생성하는 작업에서 AI가 효과적이었습니다.

// AI가 생성한 테스트 코드 (일관된 구조)
describe('반복 일정 생성', () => {
  it('매일 반복 일정을 생성한다', async () => {
    // Given
    const { user } = setup();

    // When
    await saveSchedule(user, {
      title: '매일 미팅',
      repeat: { type: 'daily', interval: 1 },
    });

    // Then
    expect(screen.getByText('매일 미팅')).toBeInTheDocument();
  });
});

3. 문서 작성 및 구조화

요구사항을 90개 사용자 시나리오로 구체화하고, 템플릿에 맞춰 일관된 구조를 유지하며, 경계값 케이스를 포함하여 PRD를 작성하는 작업에서 AI가 유용했습니다.

4. 단순 로직 구현

isLeapYear(윤년 판별), getDaysInMonth(월별 일수 계산), getRepeatText(텍스트 변환)와 같은 단순 로직을 정확히 구현했습니다.

// AI가 한 번에 정확히 구현
function isLeapYear(year: number): boolean {
  return year % 4 === 0 && (year % 100 !== 0 || year % 400 === 0);
}

5. 리팩토링

150줄의 중복된 건너뛰기 처리 로직을 findNextValidDate 함수로 추출하고, 테스트를 깨뜨리지 않고 안전하게 리팩토링하는 작업에서 AI가 효과적이었습니다.

AI가 못하는 것

1. 복잡한 로직의 전체 맥락 파악

달력 이동 시 반복 일정이 표시되지 않는 이슈가 발생했을 때, AI는 6번의 시도에도 불구하고 근본 원인을 찾지 못했습니다. expandRecurringEvents, generateRecurringDates, App.tsx의 필터링, useCalendarView, 날짜 계산 로직을 각각 수정해보았지만 모두 실패했습니다.

"useSearch의 filterEventsByDateRangeAtMonth가 반복 일정을 날짜 범위로 필터링해서 제외시키는 것 같다"는 힌트를 제공하자 AI가 즉시 문제를 해결했습니다.

AI는 부분적인 증상에만 집중하고 전체 데이터 흐름을 파악하기 어려워하는 것 같습니다. 특히 3개 이상의 파일이 연관된 로직에서 이러한 한계가 두드러졌습니다.

2. 암묵적 요구사항 파악

명시하지 않았을 때 AI는 "매일 반복 일정을 365개 Event로 저장하겠습니다"라고 제안했습니다. 이는 데이터베이스에 365개 레코드를 생성하고, 수정/삭제 시 365개 모두 업데이트해야 하는 비효율적인 방식입니다.

"하나의 반복 일정 세트에 대해 하나의 데이터가 저장되도록 해주세요. 뷰에서 전개하는 방식으로 처리하세요"라고 명시하자 올바르게 구현했습니다. AI는 비효율성을 스스로 판단하기 어려운 것 같습니다.

3. 트레이드오프 판단

반복 종료일을 어떻게 처리할지에 대해 AI는 세 가지 옵션을 제시했지만 최종 결정을 내리지 못하고 물어왔습니다. (1. endDate를 '9999-12-31'로 설정, 2. endDate를 빈 문자열로 설정, 3. endDate를 optional로 변경)

AI는 여러 옵션을 제시할 수 있지만 최종 결정은 인간이 내려야 하는 것 같습니다.

4. 창의적 문제 해결

반복 일정의 특정 날짜만 수정하는 로직에 대해 AI는 예외 날짜 목록을 별도로 관리하거나 수정된 일정을 별도 배열에 저장하는 방식을 제안했습니다. 이는 복잡도를 증가시키고 데이터 일관성 관리가 어려워지는 문제가 있었습니다.

"해당 날짜를 기준으로 반복 일정을 3개로 분할하면 어떨까요?"라는 제안을 하자 AI가 이를 구현했습니다. AI는 기존 패턴에서 크게 벗어나는 해법을 제시하기 어려워 보이며, 창의적 접근은 인간이 주도해야 하는 것 같습니다.

5. 우선순위 판단

Refactoring Engineer가 6가지 개선 사항을 제안하고 "모두 수정하겠습니다"라고 했을 때, 시간 제약을 고려하지 않은 판단이었습니다. "중복 코드 제거만 먼저 진행하세요"라고 우선순위를 지정해주어야 했습니다. AI는 비즈니스 제약(시간, 리소스)을 고려하기 어려워 보입니다.

6. 예외 상황 처리

매월 30일 반복 시 2월을 건너뛰어야 하는데 AI는 2월 30일을 생성하려고 했습니다. 이러한 경계값 케이스는 테스트로 명시해야 AI가 인식할 수 있었습니다. AI는 명시되지 않은 경계값 케이스를 예측하기 어려워 보입니다.

AI를 효과적으로 활용하는 방법

이번 프로젝트를 통해 AI에게 맡겨도 되는 작업과 인간이 직접 해야 하는 작업을 구분할 수 있었습니다.

반복적인 작업(코드 생성, 문서 작성), 패턴이 명확한 작업(테스트 작성, 리팩토링), 단순 로직 구현은 AI가 잘 수행했습니다.

반면 전체 아키텍처 설계, 트레이드오프 판단, 창의적 문제 해결, 우선순위 결정, 비즈니스 요구사항 해석, 복잡한 로직의 Root Cause 분석은 여전히 인간이 주도해야 했습니다.

개인적으로는 인간이 설계와 리뷰, 방향 제시를 담당하고 AI가 구현과 테스트, 문서 작성을 담당하는 역할 분담이 효과적이었다고 생각합니다.

AI에게 전부 맡기는 것이 아니라 협업하는 것이 중요하다

처음에는 AI에게 모든 것을 맡기면 되겠다고 생각했습니다. 하지만 실제로는 구체적인 역할과 명확한 지시가 있을 때 훨씬 좋은 결과를 얻을 수 있었습니다.

초기에는 "반복 일정 기능 완전히 구현해줘"와 같이 큰 단위로 지시했을 때 AI가 방향을 잃는 경우가 많았습니다. 하지만 "Implementation Engineer님, 이 테스트를 통과시켜주세요" "Refactoring Engineer님, 중복 코드를 제거해주세요"와 같이 역할과 작업을 명확히 구분하여 지시하니 훨씬 안정적으로 작업이 진행되었습니다.

AI에게 구체적인 지시를 주면 빠르고 정확하게 수행하지만, 방향을 잃으면 헤매는 경향이 있어 중간 확인과 피드백이 필수적이라고 느꼈습니다.

에이전트 시스템을 통한 역할 분리

PO, QA, 개발자, 리팩토링 엔지니어 등 각 역할을 명확히 분리하고 worklog로 인수인계하며 보고하는 구조를 만들었습니다.

이 구조 덕분에 각 단계의 품질이 보장되었고, 문제 발생 시 어느 단계에서 잘못됐는지 추적할 수 있었으며, 재사용 가능한 프로세스를 구축할 수 있었습니다.

TDD와 AI의 조합

테스트 없이 AI와 개발하면 AI가 코드를 생성하고 버그가 발견되면 수정하다가 다른 곳을 파손하는 무한 루프에 빠지기 쉬웠습니다.

하지만 TDD를 적용하면 RED → GREEN → REFACTOR 사이클을 통해 각 단계가 객관적으로 검증 가능했고, AI가 방향을 잃지 않고 작업할 수 있었습니다. 이번 과제에서 172개 테스트를 모두 통과하면서도 안정적으로 개발할 수 있었던 것이 이 때문이라고 생각합니다.

XML이 AI에게 더 효과적이었다

처음에는 XML이 가독성이 떨어진다고 생각했습니다. 하지만 실험 결과 Markdown보다 XML을 사용했을 때 AI가 더 정확히 이해하는 것을 확인했습니다.

구조화된 정보가 AI에게 중요하다는 것을 알 수 있었습니다. <constraint> 태그를 통해 제약사항을 명확히 인식시키고, <step n="1"> 속성을 통해 순서를 정확히 따르도록 할 수 있었습니다.

Trade-off는 있지만, AI와의 협업에서는 XML이 더 효과적이라고 판단됩니다.

Few-Shot Learning의 중요성

단순히 "이렇게 하면 안 돼"라고 설명하는 것보다 "이건 좋은 예, 이건 나쁜 예"와 같이 구체적인 예시를 제공하는 것이 훨씬 효과적이었습니다.

적절한 작업 범위 설정

"전체 기능 구현해줘"와 같이 범위가 넓으면 실패하는 경우가 많았고, "이 함수만 구현해줘"와 같이 적절한 범위로 제한하면 성공률이 높아졌습니다. 책임 단위로 작업을 쪼개는 것이 중요했습니다.

인간의 역할

AI가 발전해도 개발자의 역할이 사라지지는 않을 것 같습니다. 오히려 전체 아키텍처 설계, 트레이드오프 판단, 창의적 문제 해결, 우선순위 결정, AI가 생성한 코드의 품질 검증 등 인간의 역할이 더 중요해지는 것 같습니다.

이번 과제에서도 AI 혼자서는 복잡한 로직의 Root Cause를 찾지 못했고, 힌트를 제공해야 문제를 해결할 수 있었습니다.

배운 점

이번 과제를 통해 AI와 협업하는 방법을 배웠습니다. 어떻게 지시해야 하는지, 어떤 컨텍스트를 제공해야 하는지, 어느 단계에서 개입해야 하는지, AI가 잘하는 것과 못하는 것은 무엇인지 경험할 수 있었습니다.

특히 에이전트 시스템 설계, TDD와 AI의 조합, XML을 활용한 구조화, 적절한 작업 범위 설정 등의 경험이 앞으로의 개발에 유용할 것 같습니다.

AI 시대의 개발자는 코드를 직접 작성하는 것보다 코드를 설계하고 검증하는 역할이 더 중요해질 것이라고 생각합니다.
