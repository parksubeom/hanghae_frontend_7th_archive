  3. 일관된 디자인 토큰과 컴포넌트 API 구축

Before 패키지 분석 후 After 패키지 개편

  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축

  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

  • Management.tsx 파일 내부에서 상태관리가 밀집되어 있던 점
  • 한 파일 내에서 관리하게 되면 이후 유지보수 시간이 길어질 수 있음

2. UI 컴포넌트 내부에 도메인 로직이 포함되어 있던 점

  • 도메인 별 props를 UI 내부에서 관리한 점
  • 도메인 타입이 추가되는 경우 UI 컴포넌트에 분기를 추가해야하고, 다른 프로젝트에서 재사용 시 도메인 로직을 걷어내야 하는 문제가 발생할 수 있음
interface BadgeProps {
  // 도메인별 props가 UI 컴포넌트에
  status?: 'published' | 'draft' | 'archived' | 'pending' | 'rejected';
  userRole?: 'admin' | 'moderator' | 'user' | 'guest';
  priority?: 'high' | 'medium' | 'low';
  paymentStatus?: 'paid' | 'pending' | 'failed' | 'refunded';
}

// 내부에서 도메인 -> UI 매핑
if (status) {
  switch (status) {
    case 'published':
      actualType = 'success';
      actualContent = '게시됨';
      break;
    // ...
  }
}

4. 상태 공유 메커니즘 부재

// 모든 상태가 ManagementPage에 있어서
// 하위 컴포넌트에 props로 전달해야 함
<Table
  onEdit={handleEdit}
  onDelete={handleDelete}
  onPublish={(id) => handleStatusAction(id, 'publish')}
  onArchive={(id) => handleStatusAction(id, 'archive')}
  onRestore={(id) => handleStatusAction(id, 'restore')}
/>
  • 컴포넌트 간 상태공유가 props drilling으로 진행되고 있음
  • 상위 컴포넌트의 상태가 변경되면, 하위 컴포넌트들까지 다시 리렌더링 되는 상황
  • 필드가 늘어나게 되면 모든 props에 필드를 추가해야하는 번거로움 발생

개편 과정에서 집중한 부분

  1. 디자인 토큰 시스템 구축
styles/tokens/
  color-primitives.css   # 원시 컬러 (blue-600, gray-100 등)
  color-semantics.css    # 의미 컬러 (primary, success 등)
  • 원시컬러와 시멘틱 컬러를 토큰으로 관리함
  • 브랜드컬러나 시멘틱 컬러 변경 시, 토큰 파일만 수정하면 변경 가능
  1. 컴포넌트 아키택쳐 분리
components/
  ui/           # 순수 UI (Badge, Button, DataTable)
  features/     # 도메인 UI (EntityStats, EntityTable)
  forms/        # 폼 컴포넌트 (EntityForm)
  • 재사용 가능한 UI rncnr
  • 도메인 로직 변경시 features 내부에서 관리
  • 새 도메인 추가시, UI 컴포넌트를 수정하지 않고 사용 가능해짐
  1. custom hooks 분리

사용한 기술 스택 경험

일단 tailwindcss, storybook, shadcn등 이미 알고 있는 기술들이었지만, 항상 새로 시작하는 프로젝트에 적용시켜본 경험은 많지만 리팩토링에 해당 스택을 적용시켜보는 경험은 처음이어서 새로웠다. storybook은 실제로 적용시켜 본 경험이 많지 않았는데, 이번 리팩토링 과제에서 사용해보니 shadcn이랑 독립적인 UI를 구현할 떄 확인하기 좋은 도구인 것 같다는 생각이 들었다. 매번 개발하고 UI 확인할 때 번거로웠는데 손에 익혀두면 유용하게 사용할 수 있을 것 같았다.

어려웠던 점과 해결 방법

사실 디자인 시스템 구축 경험은 없어서 어디까지 토큰을 관리해야하는지 고민을 많이 했던 것 같다. 잘된 레퍼런스 코드를 봄면 breakpoint, border, blur,, 등등 모든 스타일 요소를 토큰으로 관리하는 것 같아서 처음에는 모든 스타일 요소를 토큰으로 관리해보려고 했지만, 디자인 가이드에 대한 피그마 파일이 없기도 했고, 리팩토링하면서 모든 요소를 관리하기에는 시간이 많이 소요될 것 같아 컬러 위주로 리팩토링을 진행했다. (디자이너가 있다는 가정을..)

before 패키지와 동시에 띄워두고 UI를 확인했는데, 버튼이나 뱃지의 mx, p가 적용되지 않는 문제가 있었다. 이유를 살펴보니 글로벌 css에 적용되어 있던 전역 스타일링에서 margin, padding을 지우니 해결되었다. 아마 우선순위가 더 높은 글로벌 파일에서 스타일을 주입하고 있어서 발생하고 있었던 문제였던 것 같다.

일단 방대한 Management.tsx파일을 보고 어떤 방식으로 리팩토링 해나가면 좋을지 고민을 해봤는데, features 폴더를 생성하여 도메인 관련 컴포넌트는 해당 폴더에서 관리할 수 있도록 반영하여 순수 UI와 분리하여 컴포넌트를 설계했다.

src/
├── components/
│   ├── ui/           # 순수 UI 컴포넌트 (도메인 무관)
│   │   ├── button.tsx
│   │   ├── badge.tsx
│   │   ├── data-table.tsx
│   │   └── ...
│   │
│   ├── features/     # 도메인 로직이 포함된 컴포넌트
│   │   ├── entity/
        │   ├── EntityStats.tsx        # 통계 표시 (Context 사용)
        │   ├── EntityTable.tsx        # 테이블 + 액션 핸들러
        │   ├── CreateEntityModal.tsx  # 생성 모달 + 폼 로직
        │   ├── EditEntityModal.tsx    # 수정 모달 + 폼 로직
        │   └── index.ts               # 배럴 export
│   │   └── EntityTableCells.tsx
│   │
│   └── forms/        # 폼 컴포넌트
│       └── EntityForm.tsx
│
├── pages/            # 페이지 (조합만 담당)
│   └── ManagementPage.tsx

추후 프로젝트가 커졌을 떄에도 도메인 별로 컴포넌트를 설계하여 관리하면 추가되는 도메인을 관리하기 용이할 것 같았다. 이번 과제를 하면서 UI 컴포넌트는 최대한 도메인을 모르고 순수 UI만 렌더링 할 수 있도록 노력하는 연습을 해볼 수 있었다. 그리고 features에서 순수 UI를 조합하여 도메인 별 컴포넌트를 구성해나가는 작업을 하는 것에 초점을 두었다. 이런 과정을 반복하니 아직 사용해보지 않은 FSD 설계에 대해 조금 더 이해해 볼 수 있는 계기가 된 것 같다.

Q) 현재 리팩토링 한 hooks가 조금 과하게 설계되어 있다는 느낌을 받고 있습니다. (ex .useTableColumns, useEntityStats) 아직 시간이 없어서 개선을 하지는 못했지만, 도메인 관련된 로직이지만 순수 계산함수나, 단수 상수를 반환하는 로직은 유틸로 분리해도 괜찮지 않을까?하는 판단에 대해서 어떻게 생각하시는지 궁금합니다.

A) 일단 hook은 보존해도 무방하고 hook 내부의 로직들을 순수함수와 상수로 분리해서 사용하면 어떨까!? 라는 의견입니다 ㅎㅎ

무엇이 더 좋고 나쁘고가 있다기보단 "일관성"이 필요하다고 생각해요.

저는 개인적으로 컴포넌트 내부의 로직은 거의다 hook으로 분리하는 편이랍니다!

Q) 테일윈드에서는 이미 기본값을 공해주고 있는데 겹치는 값들 또한 토큰으로 관리하는 방식이 좋은 방법일까요?

A) 만약 tailwindcss를 사용하지 않는 결정을 하게 된다면 어떻게 될까요!? 혹은 디자인 시스템을 아예 라이브러리로 만들어서 사용한다거나!? 이럴 때에는 tailwindcss 가 발목을 잡을 수 있답니다! 그래서 tailwindcss에 의존적이여도 잘 발라낼 수 있는 (쉽게 제거할 수 있는) 방법을 고민해보시면 좋다고 생각해요.

관련 논의: https://discord.com/channels/1288769861589270590/1444676819059216514

Q) 현재 EntityContext로 전역 상태를 관리하고 있습니다. AI로 진행하다보니 Context 관리로 진행되었는데 현재 프로젝트에서 관리하는 props 개수를 봣을 때에는 과한 설계라고 생각이 들었습니다. 혹시 멘토님은 기준으로 Context를 관리하시는지 궁금합니다!

A) 특정 페이지 혹은 특정 구역(?)에서 쓰이는 구간을 Context로 관리하면 좋다고 생각해요. 전역 상태를 만든다기보단, scope를 구성해서 사용하는거죠.

어떤 영역에서만 쓰이는 공통 값이 있을 때 context로 묶어주면 편하답니다! 똑같은 변경주기를 가진 값이라고 해야하나..?
