[5팀 김성민] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by devmineee2025.10.24

Medium

과제 체크포인트

기본과제

Medium

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. medium.useEventOperations.spec.tsx > 아래 toastFn과 mock과 이 fn은 무엇을 해줄까요? toastFn 은 외부 로직으로, toastFn 이 정확히 어떻게 동작하는지는 검증 대상이 아니라는 생각은 했습니다. .... 그런데 아직 정확하게는 잘 모르겠습니다.

Q. medium.integration.spec.tsx > 여기서 ChakraProvider로 묶어주는 동작은 의미있을까요? 있다면 어떤 의미일까요? Provider 은 내부 요소들이 필요한 컨텍스트들을 제공해주게 되는데, 테스트 환경에서 이러한 컨텍스트를 제공해줘야 실제 환경과 유사할 것이고, 신뢰성이 높아진다고 볼 수 있을 것 같습니다.

Q. handlersUtils > 아래 여러가지 use 함수는 어떤 역할을 할까요? 어떻게 사용될 수 있을까요? 테스트를 수행했을 때 다음 수행해야 할 테스트에 영향을 미치면 안됩니다. 예를 들면, 이전 테스트에서 POST 요청을 통해 데이터를 변경해버리면 다른 테스트에서 GET 등으로 조회하는 데이터는 바뀔 수 있습니다. 이러한 문제를 방지하기 위해 독립적인 환경을 만들어줘야 합니다. 그러한 점에서 handlers.ts 에 정의된 메서드만으로는 한계가 있습니다. 따라서 handlersUtils 에서 정의한 use 함수를 테스트 때 마다 사용하여 독립적인 환경을 만들어 줄 수 있습니다.

Q. setupTests.ts > 왜 이 시간을 설정해주는 걸까요? '이번 달에 공휴일을 반환한다 ' 등 현재 시간의 정보를 가지고 수행하는 테스트들이 있을 수 있습니다. 이러한 테스트들은 테스트를 실행하는 시점 마다 매번 달라질 것이고, 이에 기대하는 output 또한 매번 달라질 것입니다. 이러한 문제를 없애기 위해 가짜 시각으로 고정해 두고, 해당 시각에서 기대하는 output을 작성하여 안정적인 테스트를 하기 위함입니다.

과제 셀프회고

매일 꾸준히 새벽까지 했음에도 불구하고 만족스러운 진전이 나오지 않아 후반부에 고생했습니다. 매달리다 보면 Hard 과제를 수행할 수 있을 것 같아 Hard를 너무 오랫동안 붙잡은 탓일까요? 시간 분배를 잘 했으면 오히려 리팩토링에 더 힘을 쏟을 수 있었을 것 같다는 생각이 듭니다. 또한, 주어진 테스트 케이스만 작성하고 경계값에 대한 충분한 추가 테스트를 작성하지 못했다는게 찜찜합니다.

기술적 성장

  • CS 개념으로만 알고 있었던 <단위 테스트> 와 <통합 테스트> 의 기본적인 작성 방법에 대해서 알게 되었습니다.
  • 테스트들을 병렬적으로 처리하는 일부 방법에 대해 알게 되었습니다.

코드 품질

테스트 통과에만 급급한 나머지 코드의 품질에 대해서는 하나도 집중하지 못했습니다. 변수명도 같은 성격을 가졌음에도 테스트 마다 제각각인 경우도 있고, 추가 주석이 필요한 곳에도 하나도 쓰지 못했습니다.

한편으로는 명확한 가이드라인이 존재했더라면 처음 작성했을 때부터 조금 더 신경쓰지 않았을까 싶습니다. 무언가 일단 작성하고 나서 리팩토링으로 맞추자! 라는 생각에 차일피일 미루게 되었던 것 같기도 합니다.

학습 효과 분석

실무에서 테스트 코드를 작성하던게 있었는데, 팀원 중 어느 누구도 MSW 를 통한 API 모킹을 알지 못해 데이터를 실제로 업데이트하고, 다시 지우는 과정을 수기로 수행한 적이 있었습니다. 지금 생각해보면 너무나도 어이없는 짓인데.. 이제는 조금 더 제대로 짤 수 있을 것 같습니다!

수고 많았어요 성민! 이번 과제는 테스트를 하는 다양한 방법들을 접하면서 테스트 코드를 작성한다는 그 심리적 장벽을 허물고 친숙해지는데 있었습니다. 회고를 보아하니 힘들었지만 잘 수행해준 것 같아요.

특히 "매일 꾸준히 새벽까지 했음에도 불구하고 만족스러운 진전이 나오지 않아 후반부에 고생했습니다"라고 하셨는데, 그 과정에서도 끝까지 포기하지 않고 115개의 단위 테스트를 모두 작성하고 통과시킨 점이 정말 대단합니다. 이런 꾸준함이 장기적으로 큰 성장을 가져올 거예요.

"Hard 과제를 너무 오랫동안 붙잡은 탓일까요?" ㅠㅠ 언제나 개발자에게는 시간이 부족하기 마련이지요. 우선순위에 대한 고민은 실무에서도 중요하니 시행착오를 통해 성장해봐요. 우리의 과제는 학습을 하기 위함이니 점수를 따기 위한 과제가 아닌 만큼 다양한 방법의 시도와 시행착오 속에서 성장하기를 바래요. 개발자에게 중요한건 마감과 완성인 만큼 지금처럼 Medium 과제를 다하고 할 수 있을 만큼 진행해본 접근 아주 훌륭합니다.

코드 품질에 대해서 "테스트 통과에만 급급한 나머지 코드의 품질에 대해서는 하나도 집중하지 못했다"고 했는데 지금의 경험을 토대로 앞으로 이런 변수에 이런 상황에서는 이 이름을 써야겠다 라고 하는 아주 세세한 생각없이 통일되는 자기 컨벤션이 있으면 좋아요. 코드를 살펴봤는데 딱히 그런 코드가 심각하게 보이는건 없는데 아쉬움 때문이지 않을까 생각합니다. ㅎ

"실무에서 테스트 코드를 작성하던게 있었는데, 팀원 중 어느 누구도 MSW를 통한 API 모킹을 알지 못해 데이터를 실제로 업데이트하고, 다시 지우는 과정을 수기로 수행한 적이 있었습니다" ㅋㅋ 언제나 개발은 필요에 의해 발전하므로 "아.. 이걸 이렇게해야 되나? 더 편한 방법 없나?" 하면 대개 있기 마련입니다ㅎ 이번 기회에 접하게 되었으니 꼭 써먹어 보기를 바래요

테스트 코드 작성의 심리적 장벽을 넘으신 것을 축하드립니다. "테스트 통과에만 급급"했다고 했지만 완성했고 성공했고 잘 했어요. 아쉬운 마음은 다음 과제에서 보여줍시다 다음 주 과제도 화이팅입니다!
