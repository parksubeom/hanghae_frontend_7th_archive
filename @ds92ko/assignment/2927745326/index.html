[2팀 고다솜] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by ds92ko2025.10.19

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

사실 MSW가 처음이라 요구사항을 완전히 이해하는 데 시간이 필요했습니다. 시간에 쫓기다 보니, 우선 “테스트가 병렬로 실행될 때 서로 간섭하지 않도록 격리된 환경을 구성하자”라는 목표를 설정했습니다. 이를 위해 medium.useEventOperations.spec.ts의 테스트를 전역 MSW 환경에서 실행하도록 초기 세팅을 진행했습니다.

이후 각 테스트가 독립적인 상태를 갖도록 하기 위해 MockEventRepository 클래스를 설계했습니다. MSW 자체에 대한 깊은 이해는 부족했지만, 클래스 내부에서 상태(events 배열)를 캡슐화하면 각 테스트 인스턴스가 독립적인 상태를 관리할 수 있지 않을까? 하는 생각으로 구현했습니다.

마지막으로, 이 패턴을 실제 테스트(medium.useEventOperations.spec.ts)에 적용해, 병렬로 구동할 때 테스트 간 상태 간섭이 발생하지 않는지 확인했습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

  • beforeEach/afterEach 훅: 각 테스트 전후로 상태 초기화
  • 독립적 데이터 생성: 각 테스트마다 새로운 MockEventRepository 인스턴스 사용

과제 셀프회고

항해를 시작하기 전에는 테스트 코드에 큰 관심을 두지 않았습니다. 필요성도 크게 느끼지 못하다보니 기술적으로만 익히려 하고 본질을 놓치고 있었던 것 같습니다.

일주일이 좀 안되는 시간동안 난생 처음 접하는 테스트와 친해지려고 열심히는 해봤는데, vitest도 RTL도 심지어 MSW까지 초면인 상황에 하필 또 난이도는 Hard라서 사실 기술적으로 눈에 띄는 발전은 없었던 것 같습니다

그래도 얻어가는 게 있다면 그렇게 계속 외면하고 미뤄왔던 테스트를 찍먹이라도 해봤다, 그래서 아주 아주 조금은 "도대체 어떤걸 어디까지 테스트 하는건데?"에 대한 감이 잡혔다, 그냥 남들이 테스트의 중요성에 대해 목소리를 높이니 꾸역꾸역 하는 게 아닌 나 자신에게 테스트의 필요성을 납득시켰다 는 것에 의의를 두기로 했습니다..!

가장 좋았던 부분은 과제를 진행하면서 "의미 있는 테스트인가?"에 대해 끊임없이 고민할 수 있었던 점 입니다. 그 질문이 나만의 기준을 만들어주고 저를 사고하는 개발자로 한 걸음 성장시켜준 것 같습니다.

기술적 성장

  • Vitest / RTL / MSW의 기초 사용법을 익혔습니다.
  • 테스트 환경 구성이나 mocking, 비동기 테스트 등 실무에서 자주 접할 만한 패턴을 경험했습니다.
  • "테스트가 가능한 코드 구조"의 중요성을 체감했습니다. 테스트를 작성하기 어렵다면, 코드 구조가 잘못된 경우가 많다는 사실을 몸소 느꼈습니다.

코드 품질

  • 디스크립션을 정말 많이 고민했습니다. 주관적이거나 추상적인 단어를 피하고, 객관적이고 직관적인 문장으로 표현하려 노력했습니다. 또한 혼동을 줄이고 테스트의 의도가 좀 더 잘 드러날 수 있도록 정리했습니다.
  • 가독성과 문서화 역할을 염두에 두고 작성했습니다. 비슷한 유형끼리 그룹화 하고, 의미 단위로 개행을 두고, 최대한 단순하고 읽기 쉬운 테스트를 목표로 했습니다.
  • 기존 코드는 최대한 수정하지 않으려 했습니다. 이번 과제의 목적은 TDD가 아닌 이미 완성된 기능을 테스트하는 것이라고 판단했습니다.
  • 단위 테스트를 진행하면서 "이 테스트, 중복 아닌가?"라는 고민이 많았습니다. 하지만 기준을 기능의 중복이 아닌 목적의 중복 여부로 두니 명확해졌습니다. 기능적으로 비슷하더라도 테스트의 의도가 다르다면 충분히 의미있는 테스트라고 판단했습니다.
  • 통합 테스트를 진행하면서는 "얼마나 세세하게, 어디까지 책임져야 하지?"가 큰 고민이었습니다. 핵심 비즈니스 로직이 올바르게 연결되고 실제 사용자 흐름이 보장되는지를 중심으로 테스트 범위를 정했습니다. 너무 세부 구현까지 검증하려다 테스트 유지보수 비용이 커지는 경험을 통해, 테스트의 목적은 안전망 확보이지 모든 코드 검증이 아니라는 점을 배웠습니다.

학습 효과 분석

  • 가장 큰 배움은 테스트는 단순히 “통과 여부”를 확인하는 게 아니라 코드의 의도를 명확히 하는 행위라는 점이었습니다.
  • 실제로 테스트를 작성하면서 “이 기능이 왜 이렇게 동작하지?”라는 부분을 다시 들여다보게 되었고, 코드 이해도가 확실히 높아졌습니다.
  • 앞으로는 신규 기능을 작성할 때 최소한의 단위 테스트를 함께 작성하는 습관을 들이려 합니다.

리뷰 받고 싶은 내용

  • 현재 단위 테스트의 디스크립션이 충분히 명확하고 일관성 있는지 피드백 받고 싶습니다.
  • 단위 테스트에서 불필요하다고 느껴지는 부분에 주석 남겨두었는데, 제가 판단한 기준이 맞는지, 혹시 놓치면 안되는 엣지케이스가 있었는지 리뷰 받고 싶습니다!
  • 통합 테스트 범위에서 생략된 부분(테스트를 해야하는 데 안한 부분)이나 너무 과도하게 불필요한 부분이 섞여있지 않은지도 궁금합니다.
  • describe로 그룹화 하면서 중첩한 부분도 있는데 이렇게 하는게 맞는지 잘 모르겠습니다..! 테스트를 그룹화하는 더 좋은 방법이 있는지 확인 부탁드립니다!
  • 더 확실한 확인을 위해 toEqual을 남발?한 것 같은데 vitest의 matcher를 선택하는 기준을 잘 모르겠습니다 ㅠㅠ

궁금한 점

  • 가능하다면 점진적으로 회사에 테스트 코드를 도입해 보고 싶은데, 구성원 대다수가 테스트에 대한 이해도가 낮고 그로 인해 테스트를 부정적으로 보는 시각이 있는 상황입니다. 어떤 방식으로 테스트를 도입해야 구성원들이 테스트에 대한 경계심을 내려놓고 너그러운 마음으로 받아들일 수 있는지 궁금합니다!
  • 테스트 피라미드 전략에 따르면 단위 70%, 통합 20% 정도의 비율로 배분한다고 알고 있습니다..! 실무에서 단위/통합의 비율도 비슷한 지 궁금합니다!
  • 통합 테스트가 많아질수록 실행 시간이 길어지는데 실제 프로젝트에서 어떻게 관리하고 계신지도 궁금합니다!

테스트는 사실 혼자 개발할 때 보다 팀원들과 함께 개발할 때 의미가 있어진다고 생각해요!! 단순히 코드만 작성한게 아니라 깊이있는 고민을 계속 해주셨군요 ㅎㅎ 좋습니다.

lint를 추가한 의도는 (제가 오프코치님은 아니지만..!?), 일단 lint나 prettier에 대해 아직 잘 모르는 분들고 계시고 자연스럽게 함께 체험해보면 했으면 하는 바람으로 추가했던게 컸을 것 같아요!

현재 단위 테스트의 디스크립션이 충분히 명확하고 일관성 있는지 피드백 받고 싶습니다.

충분히 명확하게 작성되어있네요! 스펙을 묘사하는 것 처럼 해주셔서 좋았습니다.

단위 테스트에서 불필요하다고 느껴지는 부분에 주석 남겨두었는데, 제가 판단한 기준이 맞는지, 혹시 놓치면 안되는 엣지케이스가 있었는지 리뷰 받고 싶습니다!

전체적으로 테스트 작성이 잘 되어 있고, 주석으로 남겨주신 판단 기준이 대부분 타당하다고 생각해요!

<동의하는 부분>

  1. dateUtils.spec.ts
  • fillZero 중복 테스트 (314-328줄): 맞습니다. 이미 충분히 커버되고 있습니다
  • fillZero 소수점 테스트 (333-335줄): 함수의 실제 사용 맥락과 맞지 않아 제거가 적절합니다
  1. eventOverlap.spec.ts
  • 주석이 없지만 테스트가 간결하고 적절합니다
  1. eventUtils.spec.ts
  • 주석이 없지만 필터링 로직을 충분히 커버하고 있습니다

<고려가 필요한 부분>

  1. getDaysInMonth 유효하지 않은 월 처리 (37-41줄): 제거보다는 유지를 권장합니다. JS Date의 자동 보정 동작을 문서화하는 의미가 있고, 향후 함수가 변경될 때 의도치 않은 동작 변경을 감지할 수 있습니다

  2. getWeekDates 경계 날짜 테스트

  • 일부는 유지를 권장합니다
  • 주의 시작/끝 테스트(66-98줄): 중복이므로 제거 가능
  • 연도/월 경계 테스트(103-153줄): 유지 권장. getWeekDates 함수가 date.setDate()를 사용하여 원본 Date 객체를 변경하는 버그가 있을 수 있고, 월/연도 경계에서 Date 객체의 동작이 복잡하므로 이 테스트들은 중요합니다
  1. getEventsForDay 유효하지 않은 날짜 처리: 유지를 권장합니다. 함수가 예상치 못한 입력에도 안전하게 동작함을 보장합니다. 특히 0이나 32 같은 경계값 테스트는 중요합니다

가능하다면 점진적으로 회사에 테스트 코드를 도입해 보고 싶은데, 구성원 대다수가 테스트에 대한 이해도가 낮고 그로 인해 테스트를 부정적으로 보는 시각이 있는 상황입니다. 어떤 방식으로 테스트를 도입해야 구성원들이 테스트에 대한 경계심을 내려놓고 너그러운 마음으로 받아들일 수 있는지 궁금합니다!

아마 무조건 경계를 할 수 밖에 없다고 생각해요. 그렇다면 그냥 테스트를 작성해서 적용한 다음에 "이렇게 동작하고, 이런 효과가 있어요~"를 이야기해보는거죠. 이미 작성되었고, 이미 추가되었고, 이미 돌아가고 있고, 코드를 눈으로 본다면 아마 단순하게 말로 하는 것과는 느낌이 아예 다를 것이라고 생각해요 ㅎㅎ

결론은, 설득보단 용서!

테스트 피라미드 전략에 따르면 단위 70%, 통합 20% 정도의 비율로 배분한다고 알고 있습니다..! 실무에서 단위/통합의 비율도 비슷한 지 궁금합니다!

비율이 그렇게 중요하진 않아요. 내가 기여하고 있는 프로젝트의 성격이 중요합니다 ㅎㅎ 저희 팀은 단위 테스트가 100%이고, 통합테스트는 아예 없어요. 이렇게 된 이유는, 통합 테스트의 난이도가 현재 프로젝트에 적용하기에는 너무 높고, 무엇보다 꼼꼼하게 검증하는게 어려워서...

그래서 단위 테스트로 최대한 넓은 범위를 검증하고자 하는게 목표였답니다. 즉, 서비스 성격마다 다르고 이건 경험적인 측면에 의존할 수 밖에 없다고 생각해요.

그래서 질문을 바꿔야 해요. 다솜님은 어떤 프로젝트를 하고 있고, 프로젝트의 성격은 어떻고, 어떤 기능들이 있나요? 그리고 그 기능들에 대해 어떤 방식으로 검증을 하는게 제일 효괒거일까요?

통합 테스트가 많아질수록 실행 시간이 길어지는데 실제 프로젝트에서 어떻게 관리하고 계신지도 궁금합니다!
