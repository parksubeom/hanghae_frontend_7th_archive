필수 스펙

    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

이번 과제에서는 원본 저장소의 GitHub Secrets에 토큰이 등록되어 있어야 CI 환경에서 Chromatic 프로젝트를 활용한 자동 시각적 회귀 테스트가 가능하지만, 현재 구조상 제약이 있어 부득이하게 포크 저장소 CI 환경에서 Chromatic 프로젝트 토큰을 활용한 시뮬레이션을 진행했습니다. 이에 따라 시뮬레이션 과정에서 발생한 스텝별 화면 캡처를 포함하여 PR에 제출합니다.

시뮬레이션 스텝

  1. 변경 사항을 포함해 PR 생성 → GitHub에서 UI Tests가 Pending 상태로 표시

  2. UI Tests 클릭 후 Chromatic에서 Unreviewed 상태 확인

  3. UI 변경 사항 검토 → 의도된 변경이면 Accept, 아니면 Deny 선택

  4. 필요 시 Discussions에 코멘트 작성

  5. 상태가 Accepted로 변경되면, 해당 UI 변경이 승인되어 이슈가 해결됨을 확인

  6. GitHub PR 페이지로 돌아와 UI Tests가 Successful Checks로 표시됨 확인

과제 셀프회고

테스트와의 첫 만남으로부터 불과 3주밖에 되지 않았는데, 벌써 테스트 전략에 대해 고민하고 있다는 사실이 신기하기도 하고, 짧은 기간이었지만 많은 것을 배우며 성장할 수 있었던 것 같아 뿌듯했습니다.

테스트 전략에 대해 진지하게 고민해보고, 팀원들과 함께 논의하는 시간은 저에게 특히나 흥미로운 시간이었습니다. 항해 노션 멤버 카드의 "동료들과 나누고 싶은 개발 관련 주제" 항목에 "단순히 작동하는 코드가 아닌, 이유있는 코드와 사고 과정을 공유하고 싶습니다"라고 적었었는데, 이번 주차는 그 희망을 실현할 수 있었던 시간이었다고 생각합니다.

심화 과제를 진행하면서 팀원들과 일정 관리 기능에서 D&D로 날짜를 변경하는 기능의 테스트 범위를 논의했습니다.

저는 D&D 기능이 복잡한 사용자 시나리오를 포함하고 있어, E2E로 검증할 경우 테스트 실행 시간이 길어지고 환경 의존성이 높아져 불안정할 수 있고, D&D 로직은 주로 캘린더 상태 업데이트와 일정 데이터 변경 흐름을 검증하는 것이 핵심이므로, 브라우저 렌더링까지 포함하는 전체 흐름보다는 컴포넌트 간 상호작용을 중심으로 테스트하는 것이 효율적이라고 생각해 E2E가 아닌 통합 테스트로 검증해야 한다고 생각했습니다.

반면, 다른 팀원은 E2E 테스트가 더 적절하다는 의견을 제시했습니다. D&D는 단순한 데이터 변경이 아니라 실제 사용자의 마우스 동작과 브라우저 이벤트 흐름이 결합된 기능이기 때문에, 통합 테스트 수준에서는 “정상적으로 드래그가 동작하는지”를 완전히 보장하기 어렵다는 것이었습니다. 즉, 마우스 이벤트가 올바르게 트리거되고, DOM이 실제로 갱신되며, API 요청이 정상적으로 반영되는 전 과정을 검증해야 진짜 사용자 경험을 담보할 수 있다는 근거였습니다.

이 논의를 통해 테스트의 목적에 따라 검증 범위를 명확히 구분하는 것이 중요하다는 점을 배웠습니다. 단순히 “로직이 올바른가”만이 아니라, “실제 사용자 흐름을 어디까지 포함해야 하는가”를 팀 내에서 논의하는 과정이 테스트 전략을 구체화하는 데 중요한 역할을 한다는 걸 깨달았습니다. 이런 과정을 통해 혼자였다면 미처 생각하지 못했을 여러 요소를 확인하고, 더 나은 방법을 찾을 수 있었습니다.

페어 프로그래밍을 직접 경험할 수 있었던 것도 의미 있는 시간이었습니다. 페어 프로그래밍을 제대로 경험해본 적이 없어서 팀원들과 함께하기 전 유튜브에서 10분 남짓한 페어 프로그래밍 가이드를 찾아보고 개념을 익히고 참여했습니다

처음엔 낯설어서 다소 어색하고 긴장됐지만, 점차 자연스럽게 서로의 의견을 주고받으며 협업의 리듬을 만들어갈 수 있었습니다.

드라이버로서 구현에 집중할 때와 네비게이터로서 전체 흐름을 바라볼 때의 시각이 확연히 다름을 느꼈고, 이 경험 덕분에 문제를 더 입체적으로 이해할 수 있었습니다.

같은 문제를 두고 팀원 간 의견이 엇갈리기도 했지만, 오히려 그 과정이 흥미로웠습니다. 서로 다른 시각과 그에 대한 이유를 들으며 조율해 나가는 과정 속에서 더 나은 해결책을 찾아갈 수 있었고, 그 경험 자체가 큰 배움이 되었습니다. 혼자였다면 놓쳤을 관점들을 발견할 수 있어서 더욱 의미 있는 시간이었습니다.

기술적 성장

  • 새로 학습한 개념

    • Playwright를 활용해 E2E 테스트를 처음 작성해보면서, 실제 사용자처럼 페이지를 탐색하고 상호작용하며 기대 동작을 검증하는 방법을 배웠습니다. 처음에는 비동기 처리와 로케이터 사용이 익숙하지 않았지만, 반복적으로 시도하면서 점점 안정적인 테스트를 작성할 수 있었습니다.

    • Storybook을 활용해 시각적 회귀 테스트를 처음 설정하고 작성해보면서, UI 변화가 의도치 않게 발생하지 않도록 확인하는 방법을 익혔습니다. 처음에는 설정과 브라우저 모드 테스트가 복잡하게 느껴졌지만, 직접 구성하고 실패를 해결하며 테스트 환경에 대한 이해가 크게 늘었습니다.

    • 테스트 전략을 처음 수립해보면서, 어떤 기능을 어떤 단위로 검증할지 고민하는 과정을 경험했습니다. 단순히 테스트 코드를 작성하는 것을 넘어, 테스트 목적과 범위를 설계하는 사고 방식을 배울 수 있었습니다.

  • 구현 과정에서의 기술적 도전과 해결

    • dnd-kit을 사용해 D&D 기능을 구현하는 과정에서, onDragEnd 이벤트가 발생할 때 DragOverlay가 기존 날짜 셀 위치로 되돌아가는 문제가 있었습니다. 처음에는 기존 로직을 최대한 유지하려고 useState와 requestAnimationFrame을 활용해 포지션 상태를 제어하려 했지만, 근본적인 해결책은 아니었습니다. 이후 팀원들과의 논의를 통해 낙관적 업데이트 방식으로 전환하여 문제를 깔끔하게 해결했습니다.

      너무 뿌듯한 나머지, 2팀 코어타임에 화면공유로 자랑했습니다...

    • Storybook을 설치하면서 Storybook과 일반 Vitest 테스트 설정이 섞이면서 MSW 초기화가 중복되고 API 모킹이 제대로 작동하지 않아 테스트가 실패하는 문제가 있었습니다. 이를 해결하기 위해 테스트 환경을 완전히 분리해 Storybook 테스트와 일반 테스트를 각각 독립적으로 실행하도록 수정했고, 그 결과 테스트가 안정적으로 실행되었습니다. 이 과정에서 MSW의 server와 worker의 개념을 이해할 수 있었습니다.

코드 품질

  • Playwright 병렬 실행을 위한 워커별 DB 분리: E2E 테스트를 병렬로 실행할 때 각 워커가 독립적인 DB 파일을 사용하도록 구현했습니다. playwright.config.ts에서 extraHTTPHeaders로 x-worker-id 헤더를 전달하고, 서버에서는 해당 헤더를 읽어 e2e.worker-${workerId}.json 파일을 사용하도록 해 여러 워커가 동시에 실행될 때 DB 충돌을 방지하고, 테스트 간 간섭 없이 안정적인 병렬 실행이 가능했습니다.

    // playwright.config.ts

use: {
  extraHTTPHeaders: {
    'x-worker-id': process.env.TEST_PARALLEL_INDEX || '0',
  },
},
fullyParallel: true,
workers: process.env.CI ? 1 : undefined,

    // server.js

const getDbName = (req) => {
  if (process.env.TEST_ENV !== 'e2e') return 'realEvents.json';

  const workerId = req?.headers['x-worker-id'] ?? '0';
  return `e2e.worker-${workerId}.json`;
};

    또한, E2E 테스트용 /api/test/reset API를 만들어 테스트 서버 구동 중에도 테스트를 다시 실행하면 기존 데이터를 삭제하도록 했고, 테스트 종료 시에는 global-teardown.ts를 통해 생성된 DB 파일을 모두 깔끔하게 삭제하도록 처리했습니다.

    // e2e/global-teardown.ts

async function globalTeardown() {
  const mockDir = path.resolve(__dirname, '../src/__mocks__/response');
  const files = fs.readdirSync(mockDir).filter((f) => f.startsWith('e2e.worker-'));

  files.forEach((file) => {
    const filePath = path.join(mockDir, file);
    fs.unlinkSync(filePath);
  });
}

    // e2e/basic-event.e2e.ts

test.beforeEach(async ({ page }) => {
  const res = await page.request.post('/api/test/reset', {
    headers: { 'x-worker-id': process.env.TEST_PARALLEL_INDEX || '0' },
  });
  expect(res.ok()).toBeTruthy();

  await page.goto('/');
});

    로컬 환경에서는 병렬 실행으로 테스트 속도를 최적화했지만, 이로 인해 CI 환경에서는 타임아웃 에러가 발생했습니다. 정확한 원인은 아직 파악하지 못했으며, 안정성을 위해 CI 환경에서는 직렬 실행되도록 분기 처리했습니다.

    // playwright.config.ts

fullyParallel: process.env.CI ? false : true,

  • 동적 날짜 생성 유틸리티 함수: E2E 테스트에서 시간이 지나도 테스트가 실패하지 않고, 다양한 시나리오에서도 항상 올바르게 동작하도록 고정된 날짜를 사용하지 않고, 현재 시점을 기준으로 동적으로 날짜를 생성하는 유틸리티 함수들을 구현했습니다.

    // src/utils/dateUtils.ts

/**
 * 오늘 날짜를 "YYYY-MM-DD" 형식으로 반환합니다.
 * E2E 테스트에서 사용되는 함수입니다.
 * @returns {string} 오늘 날짜
 */
export function getTodayDate() {
  const today = new Date();
  return formatDate(today);
}

/**
 * 오늘이 속한 주에서 오늘이 아닌 다른 날짜를 "YYYY-MM-DD" 형식으로 반환합니다.
 * E2E 테스트에서 사용되는 함수입니다.
 * @returns {string} 오늘이 아닌 이번 주의 날짜
 */
export function getOtherDateInWeek() {
  const today = new Date();
  const weekDates = getWeekDates(new Date(today));
  const targetDate = weekDates.find((date) => date.getDate() !== today.getDate());

  if (!targetDate) {
    throw new Error('오늘이 아닌 날짜를 찾을 수 없습니다.');
  }

  return formatDate(targetDate);
}

/**
 * 이번 달의 첫째주와 셋째주의 같은 요일 날짜를 "YYYY-MM-DD" 형식으로 반환합니다.
 * E2E 테스트에서 사용되는 함수입니다.
 * @returns {[string, string]} [첫째주 날짜, 셋째주 같은 요일]
 */
export function getFirstAndThirdWeekDate(): [string, string] {
  const today = new Date();
  const year = today.getFullYear();
  const month = today.getMonth();

  const firstWeekDate = new Date(year, month, 1);

  const thirdWeekDate = new Date(firstWeekDate);
  thirdWeekDate.setDate(firstWeekDate.getDate() + 14);

  return [formatDate(firstWeekDate), formatDate(thirdWeekDate)];
}

/**
 * 이번 주의 월요일과 금요일을 "YYYY-MM-DD" 형식으로 반환합니다.
 * E2E 테스트에서 사용되는 함수입니다.
 * @returns {[string, string]} [이번 주 월요일, 이번 주 금요일]
 */
export function getMondayAndFriday(): [string, string] {
  const today = new Date();
  const weekDates = getWeekDates(new Date(today));

  const monday = weekDates[1];
  const friday = weekDates[5];

  return [formatDate(monday), formatDate(friday)];
}

/**
 * 이번 달이면서 이번 주가 아닌 날짜를 "YYYY-MM-DD" 형식으로 반환합니다.
 * 다음주가 다음달이면 지난주를, 아니면 다음주를 반환합니다.
 * E2E 테스트에서 사용되는 함수입니다.
 * @returns {string} 이번 달이면서 이번 주가 아닌 날짜
 */
export function getOtherWeekInMonth() {
  const today = new Date();

  const nextWeek = new Date(today);
  nextWeek.setDate(today.getDate() + 7);

  const prevWeek = new Date(today);
  prevWeek.setDate(today.getDate() - 7);

  const isNextMonth = nextWeek.getMonth() !== today.getMonth();
  const targetDate = isNextMonth ? prevWeek : nextWeek;

  return formatDate(targetDate);
}

    E2E 테스트에서는 이 함수들을 사용하여 고정된 날짜 대신 동적으로 생성된 날짜를 사용합니다.

    // e2e/basic-event.e2e.ts

import { getOtherDateInWeek, getTodayDate } from '../src/utils/dateUtils';

test('일정 생성 성공', async ({ page }) => {
  await page.fill('input[placeholder="날짜"]', getTodayDate());
  // ...
});

test('일정 수정: 모든 필드', async ({ page }) => {
  await page.fill('input[placeholder="날짜"]', getOtherDateInWeek());
  // ...
});

학습 효과 분석

  • 테스트 전략의 중요성: 테스트를 단순히 기능 검증용으로 생각하지 않고, 어떤 순서로 어떤 범위를 먼저 검증할지 전략을 세우는 것이 개발 효율과 코드 품질 모두에 큰 영향을 준다는 것을 깨달았습니다. 테스트를 설계하며 우선순위를 정하고, 반복적인 회귀 테스트를 계획하는 과정 자체가 기능 구현보다 더 깊게 시스템을 이해하게 만들었습니다.

  • 사용자 중심 테스트의 가치: 단순히 동작 여부만 확인하는 것이 아니라, 실제 사용자가 어떤 순서로 UI를 조작할지, 어떤 예외 상황이 발생할지까지 고려해 테스트를 작성하니, UX 측면에서 문제를 조기에 발견하고 개선할 수 있었습니다. 이를 통해 테스트가 단순한 QA 도구가 아니라, 사용자 경험을 설계하는 중요한 수단임을 체감했습니다.

  • Storybook의 다양한 활용: Storybook을 통해 UI 컴포넌트를 독립적으로 개발하고, 시각적 회귀 테스트까지 설정하며 컴포넌트 단위로 안정성을 검증할 수 있다는 점을 처음으로 경험했습니다. 단순히 디자인 확인용이 아니라, 팀 내 다른 개발자와 UI 상태를 공유하고 테스트 자동화까지 연결할 수 있어 개발 프로세스가 훨씬 체계적으로 느껴졌습니다.

  • E2E 테스트의 양면성: Playwright로 E2E 테스트를 작성하면서, 전체 사용자 흐름을 검증할 수 있다는 장점과 동시에, 실행 시간이 오래 걸리고, 테스트가 많아질수록 유지보수 비용이 높아질 수 있다는 단점도 동시에 체감했습니다. 처음 해보는 E2E 테스트였지만, 전체 시스템 동작을 이해하고, 테스트 설계에서 발생할 수 있는 문제를 미리 고민하게 되었다는 점이 큰 배움이었습니다.

과제 피드백

테스트를 처음 접하고 바삐 흘러간 3주 동안, 단순히 “테스트 코드를 작성하는 법”을 배우는 것을 넘어 테스트가 왜 필요한지, 그리고 어떻게 설계되어야 하는지를 깊이 고민할 수 있었습니다.

특히 이번 과제는 단순한 구현 과제라기보다, “테스트 전략”이라는 주제 자체를 중심으로 사고해야 했기 때문에 처음엔 막막했지만, 과제를 진행하면서 테스트의 본질적인 역할을 체감할 수 있었습니다.

E2E, 통합, 시각적 회귀 테스트를 실제로 모두 적용해보며 각 테스트의 목적과 한계를 비교할 수 있었고, 이를 통해 다음과 같은 인사이트를 얻었습니다.

  • E2E 테스트는 ‘사용자 경험의 최종 보증선’이다. Playwright를 사용해 전체 흐름을 검증하면서, 단일 기능 단위로는 확인되지 않는 오류들이 실제 브라우저 동작에서 드러나는 경험을 했습니다. 다만, 실행 시간이 길고 환경 의존성이 높기 때문에 모든 케이스를 E2E로 커버하기보다는 핵심 사용자 플로우 중심으로 제한하는 전략적 접근이 필요하다는 걸 배웠습니다.

  • 통합 테스트는 로직의 안정성을 지탱하는 ‘중간 계층’이다. 통합 테스트는 E2E보다 빠르고, 단위 테스트보다 실제 흐름에 가깝게 검증할 수 있어서 복잡한 상태 관리 로직에 특히 유용했습니다. 이번 과제에서 D&D 기능 검증 범위를 두고 팀원들과 논의하며, “로직 검증”과 “사용자 경험 검증”의 경계를 어디까지 둘 것인가가 테스트 전략 수립의 핵심임을 깨달았습니다.

  • 시각적 회귀 테스트는 UI 품질의 안전망이다. Storybook과 Chromatic을 이용해 UI가 의도치 않게 변경되는 상황을 조기에 감지할 수 있다는 점이 인상 깊었습니다. 특히 “문서화 중심의 스토리 작성”과 “회귀 테스트 중심의 스토리 작성”의 차이를 직접 경험하면서, 테스트 목적에 따라 스토리 구조를 달리 설계해야 한다는 걸 배웠습니다.

이번 과제는 단순히 기술을 익히는 과정을 넘어서, 테스트를 통해 코드 품질을 관리하고 개발 의사결정을 돕는 사고방식을 익히는 기회였습니다. 또한, 팀원들과 서로 다른 시각을 교환하며 토론하는 과정이 매우 의미 있었습니다. 하나의 정답을 찾기보다는, 기능의 성격과 프로젝트의 목적에 따라 “왜 이 수준의 테스트가 필요한가”를 스스로 정의하는 힘이 조금은 생긴 것 같습니다.

앞으로는 이번 경험을 바탕으로, 테스트가 프로젝트에 어떤 가치를 더하는지 명확히 설명할 수 있는 개발자가 되고 싶고, 기능 단위의 코드뿐만 아니라 “테스트 단위의 설계”까지 고려하는 습관을 들이고 싶습니다.

이번 과제를 통해 “테스트는 코드를 검증하는 도구가 아니라, 사고를 검증하는 도구”라는 생각이 들었습니다.

이번 챕터를 마무리하면서, 모든 게 새롭고 어려워서 매일 밤새며 과제를 했던 시간들이 떠올랐습니다. 힘들기도 했지만 그만큼 배우는 게 많았고, 시행착오조차 값진 경험이었습니다. 짧은 3주였지만 정말 밀도 높은 시간이었고, 그래서인지 아쉽기도 하고 한편으론 정말 뜻깊은 여정이었습니다.

리뷰 받고 싶은 내용

  1. Playwright 병렬 실행을 위한 워커별 DB 분리 방식 E2E 테스트를 병렬로 실행하기 위해 워커별로 독립적인 DB 파일(e2e.worker-${workerId}.json)을 사용하도록 구현했습니다. 로컬 환경에서는 정상적으로 병렬 실행이 가능했지만, CI 환경에서는 간헐적인 타임아웃 에러가 발생했습니다. 정확한 원인을 아직 파악하지 못했고, 안정성을 위해 CI에서는 직렬 실행으로 분기 처리했습니다.

    • 이런 식으로 워커별 DB 분리를 하는 접근이 일반적인지 궁금합니다.
    • CI 환경에서도 병렬 실행을 안정적으로 유지할 수 있는 방법(예: 테스트 데이터 관리나 병렬 워커 간 동기화 전략)이 있을까요?
    • 또한, Playwright 기반 E2E 테스트를 효율적으로 자동화하는 일반적인 패턴이나 구조가 궁금합니다.

  2. E2E 테스트에서 동적 날짜 생성 유틸리티 사용 방식 테스트가 시간이 지나도 실패하지 않도록, getTodayDate(), getOtherDateInWeek() 같은 함수를 만들어 동적으로 날짜를 생성하는 방식으로 구현했습니다. 다만, 이런 유틸리티 함수가 잘못 동작하면 테스트 자체가 아니라 테스트 유틸리티의 문제로 인해 실패할 수 있어, 원인 분석이 더 어려워질 수 있다고 생각합니다.

    • E2E 테스트에서 이런 동적 유틸리티 사용 방식이 적절한지 궁금합니다.
    • 테스트의 유지보수성과 신뢰성을 동시에 높이려면, 날짜처럼 실행 시점에 따라 달라지는 데이터는 어떻게 다루는 게 좋을까요?
    • 혹시 테스트 코드 내부에서 직접 날짜를 제어하거나 주입(Injection)하는 더 나은 방법이 있을까요?

  3. Drag & Drop 일정 변경 테스트 케이스 설계 페어 코딩 중 D&D 일정 변경 기능의 테스트 케이스를 논의했습니다. 제가 제안한 테스트 케이스는 다음과 같습니다:

    • 다른 날짜로 D&D 시 일정 이동 성공
    • 빈 셀로 D&D 시 일정 이동 실패
    • 반복 일정 D&D 시 기본 일정으로 변경

    다른 팀원분은 월간 뷰와 주간 뷰를 구분해 테스트 케이스를 나누는 것이 더 명확할 것 같다는 의견을 주었습니다. 아직 E2E 테스트에 익숙하지 않다 보니 통합 테스트와 E2E 테스트의 경계, 그리고 테스트 범위를 어디까지 포함해야 하는지 감이 잘 잡히지 않습니다. 이럴 때 테스트 수준을 결정하는 기준이나 실무적인 판단 기준이 궁금합니다.

  4. 시각적 회귀 테스트용 스토리 구성 방식 이전에는 스토리북을 문서화 중심으로 활용해, 가능한 세세한 컴포넌트 단위까지 스토리를 작성하고 인터랙션도 적극적으로 추가했습니다. 하지만 이번에는 시각적 회귀 테스트를 목적으로 스토리를 작성하면서, 제 생각에는 인터랙션이 오히려 스냅샷 결과를 불안정하게 만들 수 있지 않을까 하는 의문이 들었습니다. 혹시 제가 스토리북의 시각적 회귀 테스트 목적에 맞는 올바른 활용 방식을 잘 모르고 있어서 이렇게 느끼는 것일 수도 있어, 이에 대해 여쭙고 싶습니다. 시각적 회귀 테스트를 목적으로 할 때는 인터랙션을 최소화하는 게 맞는지, 아니면 실제 사용자 플로우를 반영하기 위해 어느 정도 인터랙션을 포함하는 게 좋은지 궁금합니다.

과제 피드백

수고했어요 다솜. 지난 테스트 코드를 통해서 빅테크에서 대형 소프트웨어의 규모가 커졌을때 장애나 무결성을 확보하기 위해서 어떤한 노력들이 필요한지를 지난 3주간에 걸쳐서 간접적으로나마 체험할 수 있는 기회였기를 바래요.

이러한 테스트를 만드는데 들어가는 품이 엄청나지만 이러한 것을 감안하고도 도입하고 시도하려는 걸 보면 테스트가 없었을때의 대규모 프로젝트가 얼마나 고통스러울지를 생각해보며 언젠가 커리어상 그런 프로젝트를 하게 될때 테스트가 주는 그 생소함과 접근성등으로 인해 겁먹지 않을 수 있는 예방 주사가 되어주었기를 바랍니다.

무엇보다 이제 이러한 과정들을 거쳤기 때문에 느낄 수 있는 해냈다와 나도 잘 할 수 있는 개발자구나 하는 마음이 생겨났길 바래요.

이제 앞으로 새로운 챕터들을 맞이 할텐데 새로운 챕터도 지금 처럼 화이팅입니다!
