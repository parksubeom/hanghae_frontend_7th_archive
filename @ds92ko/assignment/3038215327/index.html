Before 패키지에서 발견한 문제점

1. Atomic Design Pattern의 잘못된 적용

  • Atoms에 도메인 로직 포함 (Button, Badge)
  • Molecules가 Atoms 미사용 (FormInput 직접 구현)
  • Organisms가 도메인 로직 포함 (Table 렌더링)
  • 계층 간 의존성 규칙 위반

해결: Atomic Design 패턴의 모호함을 피하고 실용성을 우선시하여 기능별 폴더 구조로 변경했습니다. 컴포넌트를 form, data-display, feedback, overlay 등 용도에 따라 분류하여 컴포넌트의 역할을 바로 파악할 수 있도록 개선했습니다.

2. 관심사 분리 위반

  • UI 컴포넌트가 도메인 로직 포함 (FormInput, Button, Table)
  • 비즈니스 로직과 렌더링 혼재

해결: UI 컴포넌트는 순수하게 렌더링만 담당하도록 설계하고, 모든 비즈니스 로직은 hooks로 분리했습니다. 컴포넌트는 props로 받은 데이터와 콜백만 사용하도록 하여 재사용성을 높였습니다.

3. 타입 안전성 부족

  • any 타입 남용 (formData: any, data: any[], entity: any)
  • 수동 validation으로 타입 체크 없음
  • 런타임 에러 가능성 증가, IDE 자동완성 불가능

해결: Zod schema를 도입하여 type-safe validation을 구현했습니다. z.infer를 사용하여 schema에서 타입을 추론하도록 하여 schema와 타입이 항상 동기화되도록 했고, any 타입을 완전히 제거했습니다.

4. 혼재된 스타일링 방식

  • 하드코딩된 색상 값 (#1976d2, #d32f2f 등)
  • 인라인 스타일 과다 사용
  • CSS 파일과 인라인 스타일 혼재
  • 디자인 토큰 없어 일관성 부족, 다크 모드 지원 불가능

해결: CSS 변수 기반의 디자인 토큰 시스템을 구축했습니다. 모든 색상, 간격, 타이포그래피 등을 CSS 변수로 정의하여 중앙에서 관리하도록 했고, 다크 모드도 .dark 클래스로 자동 전환되도록 구현했습니다.

5. 일관성 없는 컴포넌트 API

  • helpText vs help vs description - 같은 의미인데 이름이 다름
  • width vs size - 비슷한 개념인데 다른 prop 이름

해결: React Hook Form과 통합된 일관된 API를 설계했습니다. 모든 Form 컴포넌트는 FormField, FormItem, FormLabel, FormControl, FormMessage 같은 동일한 패턴을 따르도록 하여 학습 곡선을 낮췄습니다.

6. 중복 코드

  • 생성/수정 모달에서 거의 동일한 폼 필드 중복
  • 약 200줄 이상의 중복 코드

해결: 공통 폼 컴포넌트를 추출하고, 커스텀 훅으로 폼 로직을 분리했습니다. User와 Post의 로직이 거의 동일하다는 점을 발견하여 제네릭 hooks를 만들어 재사용성을 극대화했습니다.

7. 접근성 문제

  • 네이티브 confirm 사용 (스크린 리더 인식 어려움)
  • ESC 키로 닫기 기능 없음, 포커스 트랩 없음
  • ARIA 속성 부족

해결: Radix UI 기반 컴포넌트를 사용하여 접근성을 내장했습니다. Dialog 컴포넌트에 ESC 키로 닫기, 포커스 트랩, ARIA 속성 등이 자동으로 적용되도록 구현했습니다.

8. 거대한 단일 파일 (ManagementPage.tsx)

  • 약 650줄의 단일 파일에 모든 상태 관리, 비즈니스 로직, UI 렌더링이 포함됨
  • 단일 책임 원칙 위반, 테스트/유지보수 어려움, 코드 재사용 불가능

해결: 기능별로 hooks를 분리하고, 컴포넌트도 UserTable, PostTable, UserDialog 등으로 분리했습니다. Constants도 constants/user.ts, constants/post.ts 등으로 중앙화하여 약 150줄의 오케스트레이션 파일로 축소했습니다.

9. Alert 노출 위치 문제

  • Alert이 일반 DOM 흐름에 렌더링되어 페이지 내부에 위치함
  • 모달이 열린 상태에서 Alert이 표시되면 모달의 z-index가 더 높아서 Alert이 모달 뒤에 가려짐
  • Portal을 사용하지 않아 레이어 순서를 제어할 수 없어 사용자가 Alert을 확인할 수 없는 문제 발생

해결: Portal을 사용하여 Toast 컴포넌트를 body 최상위에 렌더링하도록 변경했습니다. 이렇게 하면 모달과 독립적인 레이어로 관리되어 항상 최상위에 표시되며, z-index를 적절히 설정하여 모달보다 위에 표시되도록 구현했습니다.

개편 과정에서 집중한 부분

1. UI와 비즈니스 로직의 완전한 분리

Before 패키지를 분석하면서 가장 크게 느낀 문제는 UI 컴포넌트가 도메인 로직을 직접 포함하고 있다는 점이었습니다. Button 컴포넌트가 entityType과 action을 받아서 비즈니스 규칙을 판단하고, FormInput이 fieldType에 따라 도메인별 검증 로직을 수행하는 구조였습니다. 이런 구조에서는 컴포넌트를 다른 도메인에서 재사용하기 어렵고, 비즈니스 규칙이 변경될 때마다 UI 컴포넌트를 수정해야 하는 문제가 있었습니다.

이를 해결하기 위해 UI 컴포넌트는 순수하게 렌더링만 담당하도록 설계했습니다. 모든 비즈니스 로직은 hooks로 분리하여, 컴포넌트는 props로 받은 데이터와 콜백만 사용하도록 했습니다. 특히 User와 Post의 로직이 거의 동일하다는 점을 발견하고, 제네릭 hooks(useEntityCRUD, useEntityData, useEntityDialog)를 만들어 재사용성을 극대화했습니다. 이 과정에서 타입 안전성을 유지하면서도 공통 로직을 추상화하는 것이 쉽지 않았지만, Config 객체를 통해 도메인별 차이점을 처리하는 방식으로 해결할 수 있었습니다.

2. 타입 안전성 강화

Before 패키지에서는 any 타입이 곳곳에 사용되고 있었고, 수동으로 validation을 수행하는 구조였습니다. 이는 런타임 에러 가능성을 높이고, IDE의 자동완성 기능도 제대로 활용할 수 없게 만드는 문제가 있었습니다. 특히 폼 데이터를 다룰 때 타입 체크가 없어 실수로 잘못된 필드에 접근하거나, 필수 필드를 누락하는 등의 문제가 발생할 수 있었습니다.

이를 개선하기 위해 Zod schema를 도입하여 type-safe validation을 구현했습니다. z.infer를 사용하여 schema에서 타입을 추론하도록 하여, schema와 타입이 항상 동기화되도록 했습니다. 이렇게 하면 schema를 수정하면 타입도 자동으로 변경되어 유지보수가 쉬워집니다. 또한 컴파일 타임에 타입 체크가 이루어지므로, 런타임 에러를 사전에 방지할 수 있었습니다. any 타입을 완전히 제거하는 것을 목표로 했고, Zod v4와 react-hook-form의 타입 호환성 문제는 unknown을 거쳐서 타입 단언하는 방식으로 해결했습니다.

3. 디자인 토큰 시스템 구축

Before 패키지에서는 색상 값이 하드코딩되어 있었고, CSS 파일과 인라인 스타일이 혼재되어 있었습니다. 같은 색상이 여러 곳에 분산되어 있어 일관성을 유지하기 어려웠고, 다크 모드를 지원하려면 모든 곳을 수정해야 하는 상황이었습니다. 또한 색상을 변경하려면 여러 파일을 찾아서 수정해야 하는 번거로움이 있었습니다.

이를 해결하기 위해 CSS 변수 기반의 디자인 토큰 시스템을 구축했습니다. 모든 색상, 간격, 타이포그래피 등을 CSS 변수로 정의하여 중앙에서 관리하도록 했습니다. 이렇게 하면 색상을 변경할 때 CSS 변수 값만 수정하면 전체에 반영되므로 유지보수가 훨씬 쉬워집니다. 다크 모드 구현 시에도 Tailwind의 dark: 클래스를 사용하는 대신, CSS 변수만 변경하여 .dark 클래스로 자동 전환되도록 구현했습니다. 이 방식이 디자인 토큰 시스템의 일관성을 유지하면서도 다크 모드를 깔끔하게 구현할 수 있는 방법이라고 판단했습니다.

디자인 토큰을 Base → Semantic → Component로 계층화하는 것도 고려해봤지만, TailwindCSS를 사용하고 있기 때문에 Base 단계를 별도로 정의할 필요가 없었고, 어드민 페이지로 규모가 크지 않아 Component 레벨까지 계층화할 필요가 없다고 판단했습니다. 또한 shadcn/ui 같은 UI 라이브러리를 사용하고 있어서 컴포넌트를 직접 만들기보다는 기존 컴포넌트를 커스터마이징하는 방식으로 작업했기 때문에, Component 레벨의 토큰 계층화는 불필요하다고 느꼈습니다. 따라서 Semantic 토큰만 사용하는 구조로 결정했습니다.

4. 컴포넌트 API 일관성

Before 패키지의 컴포넌트들을 살펴보니, 각 컴포넌트마다 props 이름이 제각각이었습니다. FormInput은 helpText를 사용하고, FormSelect는 help를 사용하며, FormTextarea는 description을 사용하는 식으로, 같은 의미인데 이름이 달라서 개발자가 매번 문서를 확인해야 하는 불편함이 있었습니다. 또한 width와 size처럼 비슷한 개념인데 다른 prop 이름을 사용하는 경우도 있었습니다.

이런 불일치를 해결하기 위해 React Hook Form과 통합된 일관된 API를 설계했습니다. 모든 Form 컴포넌트는 FormField, FormItem, FormLabel, FormControl, FormMessage 같은 동일한 패턴을 따르도록 했습니다. 이렇게 하면 한 번 익히면 모든 폼 컴포넌트에서 동일한 방식으로 사용할 수 있어 학습 곡선이 낮아집니다. 또한 Barrel exports를 사용하여 import 경로도 단순화했습니다. 이 과정에서 컴포넌트의 재사용성과 개발자 경험을 모두 고려해야 했는데, shadcn/ui의 Form 컴포넌트를 사용하여 일관성을 확보할 수 있었습니다.

5. 코드 구조 개선

폴더 구조는 프로젝트 설계 단계에서 항상 고민하게 되는 영역입니다. 명확한 정답이 없어 매번 다른 방식으로 설계하게 되는데, 처음 Atomic Design Pattern을 접했을 때는 드디어 만능 정답을 찾았다고 느꼈습니다. 하지만 실무에 팀 단위로 적용해보니 곧 한계가 드러났습니다. "이건 molecules야? 아니면 organisms야?" 계층의 경계가 모호해 팀원마다 기준이 달라지고, 분류 자체가 고민거리가 되었습니다. 또한 Atomic Design 원칙을 준수하기 위해서는 계층 간 의존성을 엄격히 지켜야 하지만, 이를 지속적으로 유지하기도 쉽지 않았습니다. 편의를 위해 도입했던 패턴에 오히려 발목이 잡혀 구조가 복잡해지고, 분류하고 규칙을 적용하는 데 매몰되어 본질을 잃었던 경험이 있었습니다.

이런 경험을 바탕으로 이번 과제에서는 폴더 구조를 기능 중심으로 재구성했습니다. 공통으로 사용되는 컴포넌트는 form, data-display, feedback, overlay 등 용도에 따라 분류하여, 컴포넌트의 역할을 직관적으로 파악할 수 있도록 했습니다. ManagementPage는 기능별로 hooks를 분리하고, UserTable, PostTable, UserDialog처럼 특정 도메인에서만 사용되는 컴포넌트들을 별도로 분리해 약 150줄의 오케스트레이션 파일로 축소했습니다. Constants도 constants/user.ts, constants/post.ts처럼 도메인 단위로 중앙화하여 관리 효율을 높였습니다.

사용한 기술 스택 경험

1. TailwindCSS 4.x

이전에 실무에서 TailwindCSS v3를 사용해본 경험은 있었지만, 보조적으로만 사용했기 때문에 깊이 있게 다루지는 못했습니다. 개인적으로도 TailwindCSS를 선호하지 않는 편이라 거부감이 있었는데, 문득 "내가 익숙하지 않아서, 잘 몰라서 싫어하는 걸 수도 있겠다"는 생각이 들었습니다. 그래서 이번 과제를 계기로 TailwindCSS와 조금이라도 더 친해져보고자 여러 시도를 해보았습니다.

  • ChatGPT와 토론하기

    ChatGPT가 TailwindCSS 찬성파, 제가 반대파 역할을 맡아 장단점을 두고 토론해봤습니다. 다양한 관점을 들어보려는 시도였지만, 기대했던 것처럼 논리적인 공방이 오가진 않았습니다. 제가 던지는 반박에 대응하기보다 TailwindCSS의 장단점만 나열하는 식으로 흐르면서 토론이 매끄럽게 이어지지 않았고, 오히려 TailwindCSS가 제 작업 스타일과 크게 맞지 않는다는 점이 더 명확해졌습니다.

  • 팀원분들의 의견 듣기

    팀원분들이 이야기하는 장점들도 제 기준에서는 오히려 단점으로 느껴지는 부분이 많았습니다. 예를 들어 "CSS 파일을 분리하지 않아도 된다"는 의견은 실제 프로젝트 상황과는 거리가 있다고 느꼈습니다. 애니메이션이나 복잡한 스타일을 적용하려면 결국 CVA를 사용하거나 스타일 파일을 별도로 분리해야 하기 때문입니다. 관심사 분리를 중요하게 생각하면서 정작 스타일링만 합쳐두는 흐름은 저에게는 다소 모순적으로 느껴졌습니다.

    "시맨틱 네이밍을 고민하지 않아도 된다"는 점도 개인적으로는 장점으로 받아들이기 어려웠습니다. 시맨틱 네이밍을 하지 않으면 클래스가 자연스럽게 길어지고, 이는 결국 가독성을 크게 떨어뜨립니다. JavaScript 개발에서는 네이밍을 그렇게 신경 쓰고, 매직 넘버 하나도 변수로 분리하려고 노력하면서 정작 스타일링에서는 네이밍을 신경 쓰지 않아도 된다는 주장은 이해하기 어려웠습니다. 긴 클래스 조합을 계속 유지하는 것이 오히려 더 많은 정신적 부담을 만든다고 느꼈습니다.

    또 "개발 속도가 빠르다"는 의견은 신규 개발에서는 유효하겠지만, 유지보수 단계에서는 클래스가 길어지거나 컨텍스트가 보이지 않아 수정에 더 많은 시간을 들이게 된다는 점도 고려해야 한다고 판단했습니다.

  • 구글링을 통한 검증

    검색을 통해 TailwindCSS의 장점으로 인정할 수 있는 부분은 두 가지였습니다. 커뮤니티 규모가 크다는 점, 그리고 다른 라이브러리와의 통합성이 매우 뛰어나다는 점입니다. 다만 이 두 가지 장점이 제가 느끼는 여러 단점을 압도할 만큼 결정적이라고 보기는 어려웠습니다.

결론적으로 TailwindCSS와 친해지지는 못했습니다. 여전히 "굳이 Tailwind를 써야 할 만큼의 결정적 장점이 있나? 그럴 바엔 vanilla-extract를 쓰는 게 낫지 않나?"라는 의문을 완전히 지우진 못했습니다. 다만, 이번 과정를 통해 장점을 객관적으로 이해하려 노력했고, 선호 여부와는 별개로 실무에서 불가피하게 사용해야 하는 상황이 있을 수 있기 때문에 익숙하게 사용할 수 있는 수준까지는 꾸준히 학습하자는 생각으로 마무리 지었습니다.

2. CVA (Class Variance Authority)

이번에 처음으로 사용해본 라이브러리입니다. 존재조차 몰랐던 도구였는데, 실제로 사용해보니 TailwindCSS의 단점이라고 느끼던 부분을 어느 정도 보완해주는 역할을 해주었습니다. 조건부 스타일링을 깔끔하게 처리할 수 있고, 타입 안정성까지 확보할 수 있어 실용성이 높았습니다.

특히 컴포넌트 단위로 변동되는 스타일을 관리할 때 구조가 명확해져 유지보수 관점에서 도움이 된다고 느꼈습니다. TailwindCSS 자체를 선호하지 않더라도, Tailwind를 써야 하는 상황에서 "최소한의 통제력"을 갖게 해주는 도구라는 점이 인상적이었습니다. 앞으로 Tailwind를 사용하는 프로젝트에서는 필수적으로 함께 쓰게 될 것 같습니다.

3. shadcn/ui (Radix UI)

이전에 쓸까 말까 고민하다가 다른 UI 라이브러리를 사용했었는데, 이번에 경험해보니 기존 UI 라이브러리들과 비교해 분명한 차별점이 있었습니다.

가장 크게 느낀 장점은 컴포넌트 소스를 직접 수정할 수 있다는 점으로, 커스터마이징이 매우 유연했습니다. 또한 접근성(A11y) 기능이 기본적으로 잘 갖춰져 있고, react-hook-form 연동이 내장되어 있어 개발 생산성이 높게 느껴졌습니다.

다만 TailwindCSS 기반이라 스타일링 방식이 맞지 않는다는 점은 여전히 아쉬웠습니다. 그럼에도 불구하고 왜 많은 개발자들이 shadcn/ui를 선호하는지 충분히 이해할 수 있는 경험이었습니다. "디자인 시스템을 직접 구축하면서도 기본 뼈대는 탄탄한 상태"라는 균형이 잘 잡혀 있었습니다.

4. React Hook Form + Zod

React Hook Form은 실무에서도 자주 사용해 익숙했습니다. 이번 과제에서는 여기에 Zod를 처음 결합해 사용하게 되었는데, 생각보다 훨씬 강력한 도구였습니다. 존재조차 몰랐던 라이브러리였기에 더 충격적이었고, type-safe한 폼 검증을 이렇게 간결하게 구성할 수 있다는 점에서 큰 만족감을 느꼈습니다.

폼 구조가 복잡해질수록 검증 로직 관리가 어려워지는데, Zod는 이 부분을 매우 깔끔하게 해결해줬습니다. 작업 중 react-hook-form과의 타입 호환성 문제가 있었는데 unknown을 거쳐 단언하는 방식으로 해결했습니다.

5. Storybook

Storybook은 실무와 지난 테스트 챕터에서도 사용해 꽤 익숙한 편이지만, 이번 과제에서는 다크 모드를 연동해보며 이전에 다뤄보지 않았던 기능들을 경험할 수 있었습니다. 다크 모드 적용이 예상보다 간단해 꽤 재밌는 경험이었습니다.

다만 익숙하다고 생각했던 것에 비해 실제로는 제가 아는 기능이 제한적이었다는 점도 느끼게 되었고, 앞으로 더 깊게 학습해야 할 필요성을 확실히 깨달았습니다. 컴포넌트 문서화나 UI 검증 과정에서 Storybook을 더 적극적으로 활용할 수 있을 것 같습니다.

어려웠던 점과 해결 방법

1. Zod v4와 react-hook-form의 타입 호환성

  • 문제: Zod v4와 react-hook-form을 함께 사용할 때 타입 에러가 발생했습니다. zodResolver에 Zod schema를 전달하면 타입이 맞지 않아 컴파일 에러가 발생했고, any를 사용하면 타입 안전성을 잃게 되는 딜레마에 빠졌습니다.

  • 해결: unknown을 거쳐서 타입 단언을 사용했습니다. TypeScript에서 타입 단언이 필요한 경우 권장되는 대로 any 대신 unknown을 사용하여 타입 안전성을 최대한 유지하면서도 타입 호환성 문제를 해결할 수 있었습니다.

2. Dialog와 Toast의 이벤트 전파 문제

  • 문제: Toast와 Dialog가 모두 열린 상태에서 Toast 클릭 시 Dialog가 의도치 않게 닫히는 문제가 발생했습니다. Dialog의 onInteractOutside가 외부 클릭을 감지할 때 Toast 영역도 외부로 인식하여, Toast를 클릭해도 Dialog가 닫히는 현상이 있었습니다.

  • 해결: onInteractOutside 핸들러에서 data-slot 속성을 사용하여 Toast 영역을 식별하고, Toast 영역 클릭 시에는 event.preventDefault()로 Dialog가 닫히지 않도록 예외 처리했습니다. 이를 통해 Toast와 Dialog가 동시에 열려 있을 때도 정상적으로 동작하도록 개선했습니다.

3. 제네릭 hooks 설계

  • 문제: User와 Post의 CRUD 로직이 거의 동일했지만, 각각 별도로 구현하면 중복 코드가 발생하고 유지보수가 어려웠습니다. 하지만 타입 안전성을 유지하면서 공통 로직을 추상화하는 것이 쉽지 않았습니다.

  • 해결: 제네릭 hooks(useEntityCRUD, useEntityData, useEntityDialog)를 만들고, Config 객체를 통해 도메인별 차이점을 처리했습니다. 제네릭 타입을 활용하여 타입 안전성을 보장하면서도 재사용성을 극대화할 수 있었고, User와 Post 모두 동일한 hook을 사용하면서도 각 도메인의 특성에 맞는 로직을 구현할 수 있었습니다.

4. Dark Mode 구현

  • 문제: 다크 모드 구현 방식을 선택하는 과정에서 Tailwind의 dark: 클래스를 사용할지, CSS 변수만 사용할지 고민했습니다.

  • 해결: CSS 변수만 사용하여 완전히 토큰 기반으로 전환하고, .dark 클래스로 자동 전환되도록 구현했습니다. 이렇게 하면 각 컴포넌트는 CSS 변수만 참조하면 되고, .dark 클래스 추가/제거만으로 다크 모드가 자동으로 적용되어 디자인 토큰 시스템의 일관성을 유지하면서도 다크 모드를 깔끔하게 구현할 수 있었습니다. 또한 dark: 접두사를 각 클래스에 붙일 필요가 없어 코드가 더 간결해지고, 향후 색상 변경 시에도 CSS 변수만 수정하면 되므로 유지보수가 쉬워졌습니다.

5. oklch 색상과 라이트하우스 접근성 점수 문제

  • 문제: CSS 변수에 oklch 색상 포맷을 사용했더니 라이트하우스에서 접근성 점수가 전혀 표시되지 않는 문제가 발생했습니다. oklch는 최신 브라우저에서만 지원되는 색상 포맷이라, 라이트하우스의 접근성 검사 도구가 이를 인식하지 못하는 것으로 보였습니다.

  • 해결: oklch 대신 기존의 rgb나 hex 색상 포맷을 사용하도록 변경했습니다. 디자인 토큰 시스템의 일관성은 유지하면서도 라이트하우스 접근성 검사가 정상적으로 작동하도록 했고, 접근성 점수를 정확하게 측정할 수 있게 되었습니다.

리뷰받고 싶거나 질문하고 싶은 내용

1. 제네릭 hooks의 타입 설계

User와 Post의 로직이 거의 동일하여 중복을 제거하고자 제네릭 hooks를 설계했습니다. 그 중 useEntityCRUD의 경우 타입 안전성을 유지하면서 재사용성을 확보하기 위해 TEntity, TFormData, TCreateData, TUpdateData 등 4개의 제네릭 타입을 사용했습니다.

다만 제네릭 타입이 많아져 코드를 읽을 때 복잡해 보일 수 있습니다. 혹시 이 정도의 제네릭 타입이 적절한지, 아니면 더 간단하게 설계할 수 있는 방법이 있을지 조언을 구하고 싶습니다. 예를 들어, TCreateData와 TUpdateData를 TFormData에서 파생시키는 방식으로 줄일 수 있을까요?

2. Zod schema와 Form 타입의 관계

타입 안전성을 확보하고 런타임 검증을 동시에 수행하기 위해 Zod schema에서 z.infer로 타입을 추론하는 방식을 선택했습니다. 이렇게 하면 schema와 타입이 항상 동기화되어 유지보수가 쉬워집니다.

다만 FormData 타입을 별도로 정의하는 방식과 비교했을 때, 각각의 장단점이 무엇인지 궁금합니다. 특히 복잡한 폼이나 schema가 자주 변경되는 경우 어떤 방식이 더 적합할까요?

3. 컴포넌트 폴더 구조의 선택

Before 패키지의 Atomic Design 패턴(atoms/molecules/organisms)이 실무에서 컴포넌트를 찾기 어렵고 분류 기준이 모호하다고 판단하여, 기능별 폴더 구조(form, data-display, feedback 등)로 변경했습니다. 이렇게 하면 컴포넌트의 용도를 바로 파악할 수 있어 실용적이라고 생각했습니다. 하지만 기능별 폴더 구조 방식도 특정 컴포넌트는 분류 기준이 모호하다는 것을 작업 도중 느꼈습니다.

혹시 더 나은 폴더 구조나 다른 접근 방식이 있을까요? 코치님이라면 어떤 방식으로 폴더 구조를 설계하셨을지 궁금합니다.

4. Constants 중앙화의 범위

Before 패키지에서 상수들이 여러 곳에 분산되어 있어 찾기 어려웠기 때문에, constants/user.ts, constants/post.ts 등으로 중앙화했습니다. 다만 모든 상수를 중앙화하면 파일이 커질 수 있고, 컴포넌트별로만 사용되는 상수까지 중앙화하는 것이 과한지 고민이 됩니다.

Constants를 어디까지 중앙화하는 것이 적절할까요? 예를 들어, 특정 컴포넌트에서만 사용되는 상수는 해당 컴포넌트 파일 내부에 두는 것이 나을까요, 아니면 일관성을 위해 constants 폴더에 모두 모아두는 것이 나을까요?

5. 타입 단언의 적절한 사용

Zod v4와 react-hook-form의 타입 호환성 문제를 해결하기 위해 unknown을 거쳐서 타입 단언을 사용했습니다. any를 사용하지 않고 unknown을 거치는 방식으로 타입 안전성을 최대한 유지하려고 했습니다.

다만 타입 단언 자체가 타입 시스템을 우회하는 방법이므로, 이 방식이 적절한지, 혹시 더 나은 해결 방법이 있을지 궁금합니다. 예를 들어, 타입 가드를 사용하거나 다른 라이브러리를 사용하는 방법은 어떨까요?

6. TailwindCSS의 실무 활용 경험

개인적으로 TailwindCSS를 선호하지 않는 편입니다. "CSS 파일을 분리하지 않아도 된다"는 장점은 실제로는 CVA나 별도 스타일 파일이 필요해지고, "시맨틱 네이밍을 고민하지 않아도 된다"는 점은 오히려 긴 클래스 조합으로 가독성을 떨어뜨린다고 느꼈습니다. 또한 "개발 속도가 빠르다"는 장점도 신규 개발에서는 유효하지만, 유지보수 단계에서는 클래스가 길어지거나 컨텍스트가 보이지 않아 수정에 더 많은 시간이 들 수 있다고 판단했습니다.

다만 많은 개발자들이 TailwindCSS를 사용하는 이유가 있을 것 같습니다. 실무에서 TailwindCSS의 어떤 장점들이 실제로 체감되시는지, 특히 유지보수 단계나 대규모 프로젝트에서의 경험을 듣고 싶습니다. 혹시 제가 놓치고 있는 부분이나, TailwindCSS를 더 효과적으로 활용하는 방법이 있을지 조언을 구하고 싶습니다.
