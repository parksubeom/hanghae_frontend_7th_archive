과제 회고

이번 과제에서도 가장 나를 시작하기 어렵게 만든 부분은 '뭐부터 어떻게 시작해야 하는가'였다. shadcn, tailwind, 그리고 지난 과제에서 이미 나왔지만 직접 해보지 못했던 storybook까지 항해에 와서 처음 들어보고 써보는 게 자꾸 늘어나면서, 내가 우물 안 개구리라는 건 알고 있었지만 우물 밖이 이렇게 넓은지는 계속해서 체감하고 있다.

AI에게 4일 안에 완성하는 목표로 투두리스트도 받아봤지만, 그마저도 바로 손이 잘 안 가서 결국 'button 컴포넌트 하나만이라도, 필수 구현 사항 찍먹이라도 해보자'는 목표를 잡고 과제를 진행했다.

새로 학습한 개념

Atomic Design Pattern 5단계 요소

  1. 원자 (Atoms)

기본 HTML 태그 버튼, 입력 필드, 레이블, 기본 텍스트 스타일 단일 목적으로 사용되는 UI의 기본 스타일

  1. 분자 (Molecules)

원자가 결합된 최소 기능 단위 검색창, 로그인 인풋 그룹 특정 기능을 수행하는, 재사용 가능한 가장 작은 의미있는 블록

  1. 유기체 (Organisms)

분자와 원자가 결합된 복합 기능 단위 전체 네비게이션 바, 푸터 섹션, 제품 카드 목록 복잡하고 상황에 맞는 역할을 수행하는 독립적인 섹션

  1. 템플릿 (Templates)

페이지 구조 정의 블로그 포스트 템플릿, 사용자 프로필 레이아웃 레이아웃 확인을 위한 더미 데이터나 플레이스 홀더를 사용해 와이어프레임만 정의

  1. 페이지 (Pages)

템플릿에 실제 콘텐츠 삽입 홈페이지.html, 실제 데이터가 채워진 블로그 상세 페이지 사용자가 최종적으로 보게될 화면을 완성, UX 테스트 단계

Before 패키지에서 발견한 문제점

Button.tsx 파일을 보면서 제일 먼저 든 생각은 '버튼 컴포넌트가 뭘 이렇게 많이 알고 있지..?'였다. 도메인 정보도 알고 있고, 비즈니스 규칙까지 판단하는 코드를 보면서 컴포넌트를 분리할 때 보통 고려하는 재사용성과 확장성은 포기하는 대신 이 컴포넌트 하나만 봐도 앞으로 서비스에서 어떻게 가져다 쓸지는 다 알 수 있겠다 싶은.. 그런 코드였다. 이런 방식이라면 서비스가 확장될수록 파일이 점점 무거워지고, 가독성이 떨어지며 관리 포인트도 분산될 수 있겠다는 인상을 받았다.

개편 과정에서 집중한 부분, 들었던 생각

버튼 컴포넌트가 본래의 UI 역할만 수행하도록 정리하는 것을 목표로 했다. 스타일은 달라도 버튼이라는 컴포넌트가 하는 일은 동일해야 한다는 것에 집중했다.

variants로만 스타일 변화를 주다 보니, primary와 tabActive가 모두 기본 색상으로 파란색을 사용한다는 점에서 공통적인 부분을 상수로 정의해 템플릿 리터럴로 재사용하면 좋지 않을까 하는 생각이 들었다. 그런데 막상 템플릿 리터럴로 작성하다 보니, 이게 정말 효율적인지, 오히려 가독성을 떨어뜨리는 건 아닌지 고민이 되었다. 결국에는 속성마다 독립적인 스타일을 가지는 게 tailwind의 취지에도 더 맞다는 결론에 도달했다.

팀 멘토링 노트 작성 때 모달 컴포넌트에 있는 x 버튼을 구현하는 방식에 대한 질문이 있었는데, 직접 과제를 하다보니 이 버튼을 아이콘 형태로만 존재하고 클릭 이벤트만 수행하는 하나의 타입으로 정의해도 충분하겠다는 생각이 들었다. AI에게 물어보니 ghost 버튼 variant로 정의하여 사용할 수 있다고 해서 안심하고 그렇게 구현했다. 실제로 구현해보니, 간단하면서도 관리가 편하다는 느낌을 받았다.

리뷰받고 싶거나 질문하고 싶은 내용

과제에 대한 내용은 아니지만 이직과 관련해 질문이 있습니다!

제 이력을 돌아보면 풀스택 개발자로 일하고 있지만 실제 업무는 보안 회사의 연구 개발 포지션에서 기획·설계·QA 등 개발 외적인 역할까지 포함되어 있어, 프론트엔드 영역을 깊게 파고든 경험보다 여러 분야의 얕은 지식만 여기저기 흩어져 쌓였다는 생각이 듭니다. 내년을 목표로 프론트엔드 경력 이직을 준비하려 했지만, 현재의 업무 특성상 프론트엔드 개발자로서 강하게 어필할 만한 성과가 부족하지 않을까 걱정이 됩니다.

이런 상황에서 제가 내년에 프론트엔드 경력 이직을 현실적으로 노릴 수 있을까요? 가능하다면 어떤 점을 중심으로 준비해야 할지, 어렵다면 어떤 부분이 보완되어야 하는지 조언을 듣고 싶습니다.

멘토링때 관련된 질문을 들었다면 더 자세히 이야기를 해볼 수 있었을 것 같은데요. 우선 잘 할 수 있습니다. 하지만, 당연히 잘 되려면 그만한 수반되는 여러 경험들과 공부가 있어야겠죠. 우선 지금 있는 팀원들이 어떤 경험을 했는지 그리고 다른 팀원분들한테도 물어봐서 어떤것들을 경험했는지 한번 정리해보세요. 그리고 그 부분에 있어 프런트엔드 개발자에게 어떤역량을 의미하는지 비교해보고 커리어적으로 도움이 될 수 있는 경험인지 자체적으로 판단해보세요. 저한테 물어보셔도 좋구요! 그리고 일단 그 부분에 대해 부족한 부분들을 자체적으로 회사에서 채울 수 있는 노력을 하면 가장 좋고, 아니라면 혼자 공부를 해야겠죠.

과거처럼 이직 시장이 따뜻했다면 크게 문제는 없었겠지만, 지금은 너무 추운것 같아요. 안된다는 것은 아니지만 추워진 만큼 그만큼 더 열심히 해야 하는 것은 맞는 것 같아요. 결국 절대적으로 뭔가를 해야지만 합격을 하는게 아니라 옆에있는 경쟁자들보다 잘하면서 그 자리에 내가 더 잘 맞아야 합격하는것이니까요. 경력에 있어서 엄청 특별한 경험을 하는 분들은 이야기 해보면 크게 없을거에요. 경험적인 부분, 하드 스킬 뿐만 아니라 소프트 스킬 부분까지 내가 경쟁자들에 비해 그 자리에 어떤 점이 더 뛰어나고 잘 맞는 사람인지 어필해보는 점을 찾아봐야 합격할수 있을것 같아요.
