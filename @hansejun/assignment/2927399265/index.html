[3팀 한세준] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by hansejun2025.10.19

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

1 서버 인스턴스 초기화

server.use()로 등록한 핸들러는 MSW 서버 인스턴스 전역에 적용됩니다. 따라서 한 테스트에서 등록한 핸들러가 다른 테스트 실행 시에도 남아 있다면, 서로 다른 테스트가 같은 서버 상태를 공유하게 되어 독립적인 테스트 실행이 어려워집니다.

이를 방지하기 위해, 다음과 같이 테스트가 끝날 때마다 핸들러를 초기화했습니다.

afterEach(() => {
  server.resetHandlers(); 
});

server.resetHandlers() 메서드는 런타임 중에 server.use()로 추가된 모든 요청 핸들러를 제거하여 전역 서버를 초기 상태로 되돌립니다. 이미 setupTests.ts에서 각 테스트 종료 시마다 서버 핸들러를 이미 초기화하고 있으므로 별도의 초기화 코드는 생략했습니다.


2. 테스트마다 독립적인 events 배열 생성

handlersUtils.ts는 한 시나리오 안에서 여러 개의 핸들러 동작을 검증하기 위한 테스트 유틸 함수입니다. 이때 events 배열이 함수 외부에 존재한다면, 한 테스트에서 변경된 배열이 다른 테스트에도 그대로 남아 테스트 간 상태가 공유되는 문제가 발생합니다.

이를 방지하기 위해, events 배열을 각 setupMockHandler 내부에 위치시키고, 매 호출마다 복사하여 테스트별로 독립적인 배열 인스턴스를 가지도록 수정했습니다.

다음과 같은 방식으로 테스트 간 상태 오염을 방지하고, 각 테스트가 완전히 독립적으로 동작할 수 있도록 했습니다.

export const setupMockHandlerCreation = (initEvents: Event[] = []) => {
  const events = [...initEvents];
  ...
};

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

// setupTest.ts

...

beforeEach(() => {
  expect.hasAssertions();
  vi.setSystemTime(new Date('2025-10-15'));
});

afterEach(() => {
  server.resetHandlers();
  vi.clearAllMocks();
  vi.clearAllTimers();
});

...

1, 타이머 초기화

대부분의 테스트가 10월 15일을 기준으로 작성되어 있기 때문에 각 테스트마다 vi.setSystemTime()을 통해 시스템 시간을 고정했습니다. 이를 beforeAll()이 아닌 beforeEach()에 작성한 이유는 다른 테스트에서 시스템 시간이 변경되더라도 각 테스트가 서로 영향을 받지 않고 독립적으로 동작하도록 보장하기 위해서 입니다.

2. mock 함수 초기화

vi.clearAllMocks()를 사용하여 각 테스트가 끝날 때마다 테스트 중 생성된 모든 mock 함수의 호출 이력을 초기화했습니다. 이를 통해 다른 테스트에서 동일한 mock 함수의 호출 횟수 등에 대해 영향을 받지 않도록 하여 테스트 간 독립성을 유지했습니다.

3. 모든 타이머 지우기

vi.clearAllTimers()를 사용하여 스케줄링되었지만 아직 실행되지 않은 모든 타이머를 취소하였습니다. 이번 과제에서는 이 메서드가 없어도 케이스가 실패하진 않았지만, useNotifications 훅 내부에서 setInterval()과 같은 타이머 API를 사용하기 때문에 테스트 간 타이머 누수로 인한 오류를 방지하기 위해 추가했습니다.


과제 셀프회고

기술적 성장

이번 과제를 통해 테스트 코드의 작성의 필요성과 역할을 이해하게 되었습니다. 그동안은 "테스트 코드가 오히려 개발 효율을 떨어뜨리는 것 아닌가? "라는 의문을 가지고 있었지만, 이번 과제를 통해 테스트가 하나의 명세로서 코드의 의도를 문서화하고, 변경으로 인한 문제점을 조기에 발견하게 해주는 역할임을 체감했습니다.

1. API 경로 및 상수 관리의 필요성

특히 기술적인 성장이라고 하기에는 하찮지만 /events/:id 엔드포인트를 /event/:id로 잘못 사용하고 있어 2시간을 허비한 경험을 통해 API 경로 및 상수의 체계적인 관리의 필요성을 절실히 느꼈습니다ㅠㅠ 좋은 습관 하나 얻었다고 생각합니다.🥹

코드 품질

이번 과정을 통해 단순히 “테스트를 통과하는 코드”가 아니라, 테스트를 작성하기 쉬운 구조의 코드가 무엇인지 깊이 고민하게 되었습니다. 테스트 효율을 높이기 위해 명세를 명확히 세우는 것

학습 효과 분석

멘토링과 Q&A 시간을 통해 테스트에서 mock 데이터를 체계적으로 관리하는 방법과 평소 고민하던 페이지 단위 컴포넌트를 역할별로 분리해 통합 테스트를 설계하는 방식을 알게 되었습니다. 직접 테스트를 작성하며 의존성이 높은 코드일수록 테스트가 복잡해진다는 점을 경험했고, 이 과정에서 테스트 하기 좋은 코드에 대한 필요성을 알게되었습니다.

과제 피드백

테스트 전반을 이해할 수 있었던 유익한 과제였습니다. 간단한 유틸 함수 테스트부터 상태 검증, 통합 테스트까지 단계적으로 다루며 테스트 방식을 익힐 수 있는 좋은 경험이었습니다. 다음 주차부터는 좀 더 잘 작성할 수 있다는 자신을 얻었습니다 ㅎㅎ🥳

과제를 수행하면서 느낀점

과제 시작 전 생각

  • “테스트를 꼭 해야 하나?”라는 의문

  • 테스트를 한다면 어느 부분을, 얼마나 깊이 해야 하는지 감이 잘 잡히지 않는 부분

  • 모든 컴포넌트를 테스트해야 하는 건지, 혹은 핵심 로직만 테스트하면 되는 건지에 대한 어려움

과제 제출 후 생각

  • 테스트를 통해 디자인이 완성되기 전이라도 명세 기반으로 코드를 작성할 수 있다는 점

  • 테스트 명세를 세우는 과정이 생각보다 훨씬 어렵고, 동시에 코드의 방향을 잡는 데 도움이 된다는 점

  • 아직은 어렵지만, 테스트를 작성하면서 조금씩 감을 잡았습니다?
