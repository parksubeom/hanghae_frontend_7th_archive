[3팀 한세준] Chapter 1-2. AI와 테스트를 활용한 안정적인 기능 개발

by hansejun2025.10.31

과제 체크포인트

필수 스펙

    1. 반복 유형 선택
    • 일정 생성 또는 수정 시 반복 유형을 선택할 수 있다.
    • 반복 유형은 다음과 같다: 매일, 매주, 매월, 매년
      • 31일에 매월을 선택한다면 → 매월 마지막이 아닌, 31일에만 생성하세요.
      • 윤년 29일에 매년을 선택한다면 → 29일에만 생성하세요!
    • 반복일정은 일정 겹침을 고려하지 않는다.
  1. 반복 일정 표시
    • 캘린더 뷰에서 반복 일정을 아이콘을 넣어 구분하여 표시한다.
  2. 반복 종료
    • 반복 종료 조건을 지정할 수 있다.
    • 옵션: 특정 날짜까지
      • 예제 특성상, 2025-12-31까지 최대 일자를 만들어 주세요.
  3. 반복 일정 수정
    1. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      • 반복일정을 수정하면 단일 일정으로 변경됩니다.
      • 반복일정 아이콘도 사라집니다.
    2. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      • 이 경우 반복 일정은 유지됩니다.
      • 반복일정 아이콘도 유지됩니다.
  4. 반복 일정 삭제
    1. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      1. 해당 일정만 삭제합니다.
    2. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      1. 반복 일정의 모든 일정을 삭제할 수 있다.

기본 과제

공통 제출

  • 테스트를 잘 작성할 수 있는 규칙 명세
  • 명세에 있는 기능을 구현하기 위한 테스트를 모두 작성하고 올바르게 구현했는지
  • 명세에 있는 기능을 모두 올바르게 구현하고 잘 동작하는지

기본 과제(Easy)

  • AI 코드를 잘 작성하기 위해 추가로 작성했던 지침
  • 커밋별 올바르게 단계에 대한 작업
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

기본 과제(Hard)

  • Agent 구현 명세 문서 또는 코드
  • 커밋별 올바르게 단계에 대한 작업
  • 결과를 올바로 얻기위한 history 또는 log
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

심화 과제

  • 모든 질문에 대해 꼼꼼하게 정리했는지

과제 셀프회고

기술적 성장

TDD 개념 학습 이번 과제를 통해 TDD에 대해 실무적인 관점에서 다시 이해할 수 있었습니다. 처음에는 테스트를 먼저 작성한다는 방식이 현실적인 개발 속도와 맞지 않는다고 생각했지만, 실제로 적용해보니 명세를 먼저 작성하고 기능을 구현한다는 점에서 지금 제가 하고 있는 개발 방식과 크게 다르지 않았습니다. 오히려 명확한 테스트가 기능의 기준점이 되어, 구현 방향이 흔들리지 않고 더 체계적으로 작업할 수 있었습니다.


좋은 프롬프트 결과 얻기 AI와 함께 작업하면서 좋은 프롬프트가 단순히 명령을 자세히 적는 게 아니라, 맥락을 이해시켜 주는 과정임을 깨달았습니다. 처음에는 구현하고 싶은 기능을 그냥 설명하는 식으로 진행했는데, 이때는 결과물이 엉뚱하거나 맥락을 이해하지 못한 듯한 답변이 많았습니다. 그래서 시도를 바꿨습니다.

1차 시도: 가능한 한 많은 구현 설명을 넣어봤지만 여전히 기능이 빗나가는 경우가 많았습니다.

2차 시도: “모르면 물어봐라”는 원칙을 세워, AI가 질문을 통해 명세를 구체화하도록 유도했습니다.

테스크를 쪼개기 제가 프롬프트를 잘못 작성한 탓도 있었겠지만, 한 번에 큰 범위의 작업을 맡기면 AI가 의도를 정확히 파악하지 못하고 엉뚱한 방향으로 기능을 구현하는 경우가 자주 있었습니다. 그래서 작업 단위를 더 세분화해보기로 했습니다. 유닛 테스트 단위에서 시작해 통합 단계로 점차 확장하는 방식으로, 작은 주기 내에서 TDD 사이클을 반복하도록 변경했습니다. 이렇게 하자 각 단계에서 결과를 명확히 검증할 수 있었고, 문제 발생 시 원인을 빠르게 추적할 수 있었습니다.

코드 품질

코드 품질 자체에는 솔직히 자신이 없습니다. 에이전트에게 필요한 맥락을 최대한 담으려다 보니, 각 프롬프트가 점점 비대해지는 문제가 생겼습니다. 공통적으로 참고해야 하는 부분을 별도의 작은 문서로 나누어 정의하고, 상황에 따라 필요한 문서만 참조하도록 구성했다면 지금보다 훨씬 가볍고 유지보수하기 쉬운 구조가 되었을 것 같습니다.

설계 과정에서도 고민이 많았습니다. 특히 오케스트레이터 하위에 있는 에이전트들은 독립적으로 동작해야 하는가? 하는 점이 핵심이었습니다. 처음에는 모든 에이전트를 동시에 만들어두고 워크플로우 전체를 실행하면서 테스트했는데, 이렇게 하니 각 에이전트가 의도한 대로 작동하는지 빠르게 확인하기가 어려웠습니다.

그래서 방식을 바꿨습니다. 각 TDD 에이전트에서 오케스트레이터에 대한 의존성을 분리하고, 각자의 아웃풋을 별도로 저장해 다음 단계의 에이전트가 그 결과만 받아 동작하도록 설계했습니다. 이렇게 하자 개별 에이전트의 문제점을 빠르게 찾아낼 수 있었지만, 반대로 전체 흐름이 끊겨 TDD 사이클이 유기적으로 이어지는지 한눈에 파악하기 어려워졌습니다.

또 하나의 문제는 GREEN 단계에서 테스트 케이스의 수정이 발생했다는 점입니다. 원래 GREEN에서는 오직 구현 코드만 수정해야 하지만, 테스트 케이스까지 변경되는 경우가 생겨 TDD 원칙이 깨졌습니다. 이를 방지하기 위해선 테스트 수정이 감지되면 RED 단계로 되돌려 검증하거나 첨음부터 잘 작성하게하는 절차가 추가로 필요합니다.

학습 효과 분석

이번 과제를 통해 가장 크게 배운 점은 AI와 함께 개발을 진행하려면 구체적인 커뮤니케이션(?)이 필요하다는 것이었습니다. 단순히 명령을 내리는 수준으로는 원하는 결과를 얻기 어렵고, AI가 이해할 수 있도록 단계별 목표와 맥락을 구체적으로 전달해야 했습니다. 이를 통해 평소에도 제가 작성하는 문서를 좀 더 명확히 표현하려고 시도하고 있습니다.

추가로 더 학습이 필요하다고 느낀 부분은 페르소나 정의, 컨텍스트 이해 방식, 공통 문서 분리 구조입니다. AI가 역할별로 어떻게 사고해야 하는지를 명확히 정의하고, 컨텍스트를 더 자연스럽게 이해시키는 방법에 대한 이해가 필요할 것 같습니다. 또한 공통적으로 참조하는 문서를 어떤 단위로 나누고 연결해야 효율적인지에 대한 기준도 아직 부족하다고 느꼈습니다.

과제 피드백

이번 과제에서 가장 고민이 되었던 부분은 오케스트레이터에게 어느 정도까지 역할을 위임해야 하는가였습니다. 각 TDD 서브 에이전트들을 완전히 독립적으로 동작시키기 위해, 각 에이전트가 이전 단계의 아웃풋을 직접 참조하도록 설계했습니다. 이렇게 하면 개별 단위로 아웃풋을 빠르게 확인할 수 있었지만, 전체적인 흐름을 제어하거나 에이전트 간의 연속성을 보장하기가 어려웠습니다.

그래서 고민이 생겼습니다. 이 아웃풋 전달과 조율의 책임을 오케스트레이터에게 위임해야 하는 게 맞을까?라는 점이었습니다. 만약 오케스트레이터가 모든 단계의 입출력을 관리한다면 구조는 깔끔해질 수 있지만, 반대로 에이전트의 자율성과 독립성이 약해질 수 있다는 우려도 있었습니다.
