심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

내가 생각한 최적의 테스트 전략

제가 생각한 최적의 테스트 전략은 통합 테스트를 강화하는 것이었습니다. 저희 프로젝트는 여러 컴포넌트와 훅이 서로 연동되어 동작하기 때문에, 이 연결 지점을 중심으로 테스트를 수행하는 것이 전체적으로 프로그램의 안정성을 높이는 가장효과적인 방법이라고 판단했습니다.

왜 유닛테스트가 아닌가?

유닛 테스트를 보강해 세부 로직을 검증하는 것도 안정성을 확보하는 방법이라고 생각합니다. 하지만 유닛 테스트 커버리지를 100% 달성하더라도, UI와 실제 로직이 연결되는 과정에서 문제가 전혀 발생하지 않는다고 단정할 수는 없습니다. 이 점에서 저는 단위 수준보다는 연결부를 중심으로 검증하는 통합 테스트가 더 가치가 있다고 생각했습니다.

왜 E2E가 아닌가?

현재 프로젝트에서 가장 쉽게 작성할 수 있는 테스트는 E2E 테스트였습니다. 별도의 모킹 과정 없이 실제 쿼리를 수행하고 UI 결과만 확인하면 되었기 때문입니다. 그러나 E2E 테스트만으로 모든 실패 케이스나 엣지 케이스를 세세하게 검증하기에는 비용이 너무 크다고 판단했습니다. 그래서 E2E 테스트는 주요 사용자 시나리오를 중심으로 전반적인 흐름을 검증하는 용도로 사용하고, 통합 테스트에서는 다양한 엣지 케이스를 포함해 안정성을 강화하는 방식으로 접근했습니다.

팀원들이 합의한 최적의 테스트 전략

저희 팀은 테스트 트로피 개념을 기반으로 통합 테스트를 강화하는 방향으로 전략을 수립했습니다. 단위 테스트는 세부 로직 검증에는 유용하지만, 실제 사용자 흐름을 충분히 보장하기 어렵다는 한계가 있었고, 반대로 E2E 테스트는 작성 및 유지보수 비용이 크다는 부담이 있었습니다. 이러한 이유로 통합 테스트를 중심에 두고 현실적인 테스트 밸런스를 맞추는 방안에 의견이 모였으며, 이를 구체화하기 위해 두 차례의 회의를 통해 세부 전략을 논의했습니다.

첫 번째 방안

현재 비대해진 App.tsx를 컴포넌트의 역할별로 분리하고, 각 컴포넌트 단위에서 통합 테스트를 작성하는 방식으로 진행하기로 했습니다. 이는 각 컴포넌트의 책임과 역할을 명확히 구분하고, 그 안에서 훅, 로직, UI 간의 연결이 올바르게 동작하는지 검증하는 것을 목표로 하였스빈다. 또한 분리된 컴포넌트들을 다시 연동하여 전체적인 흐름을 검증하는 통합 테스트를 작성함으로써, 테스트 실패 시 문제의 원인을 빠르고 명확하게 파악할 수 있도록 하는 것이 목적이었습니다.

그러나 이 방안을 실현하기 위해서는 UI 분리뿐 아니라 데이터 핸들링 로직까지 리팩토링해야 하는 작업이 필요했습니다. 이에 따라 시간적 여유가 부족한 상황에서는 현실적으로 수행하기 어렵다는 판단을 내렸습니다.

두 번째 방안

두 번째 방안은 현재의 컴포넌트 구조를 유지하면서, 기능 단위로 테스트 파일을 분리하여 각 기능에 맞는 엣지 케이스와 누락된 테스트를 보강하는 방식이었습니다. 이 접근은 첫 번째 방안의 취지를 일정 부분 유지하면서도 현실적으로 실행 가능한 대안이었습니다. 각 기능별로 팀원 두 명이 페어 프로그래밍으로 테스트를 작성하고, 이후 코드를 취합하는 방식으로 진행하여 효율성과 일관성을 함께 확보하고자 했습니다.

리뷰 받고 싶은 내용

POM 패턴

  • E2E 테스트 과정에서 Page Object Model(POM) 패턴을 알게되어 페이지 단위로 요소와 동작을 관리
  • 하나의 페이지 내에 많은 기능이 포함되면서 단일 POM이 지나치게 비대해지는 문제가 발생
  • 이에 따라 컴포넌트 단위로 객체를 분리하고 조합하는 구조로 변경
CalendarPageObject/
├── CalendarViewObject    // 캘린더 표시에 대한 선택자와 액션
├── EventFormObject    //  이벤트 폼
├── EventListObject      // 이벤트 목록
├── SeacrhObjectObject      // 일정 검색
├── OverlapDialogObject      // 일정 겹침 모달
├── RecurringDialogObject      // 반복 일정 모달
├── NotificationObject      // 알림

이렇게 컴포넌트 단위로 객체를 생성하니 테스트 규모가 커질수록 객체의 수가 급격히 늘어나기 때문에, 실무 환경에서는 POM을 어느 수준까지 세분화하는 것이 효율적인지, 혹은 아예 다른 접근 방식을 사용하는지에 대한 고민이 남았습니다.
