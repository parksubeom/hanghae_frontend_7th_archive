TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링

shadcn/ui

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

Before 패키지에서 발견한 문제점

컴포넌트마다 다른 props 이름과 타입

  • 'sm' | 'md' | lg , 'small' | 'medium' | 'large' 등으로 컴포넌트마다 props 형식이 제각각
  • hint ,helperText 등 헬퍼 텍스트를 다루는 prop명도 다름

CSS 파일, 인라인 스타일, 하드코딩된 색상 혼재

  • CSS 클래스: .btn, .badge 등 전역 CSS 클래스 사용
  • 인라인 스타일: 인라인 스타일 사용
  • 하드코딩된 색상 값: 하드코딩된 색상값 수십 곳에 산재
  • 스타일 중복: 같은 색상/간격이 여러 곳에 반복 정의
  • 디자인 토큰 부재로 다크모드 전환 어려움

UI 컴포넌트가 비즈니스 로직 포함

  • Button: 엔티티 타입과 액션등을 받아 도메인 규칙을 판단
  • Badge: 도메인별 색상/텍스트 자동 매핑 로직 내장
  • FormInput: 비즈니스 규칙 검증
  • Table: 엔티티 타입별 렌더링 로직 내장

접근성 지원 부족

  • FormSelect: <label>에 htmlFor 속성 없음
  • Modal: ESC 키 처리 미구현, 포커스 트랩 없음
  • Table: 정렬 버튼에 ARIA 속성 없음 (aria-sort 미사용)
  • Button: 비활성화 이유를 스크린 리더가 알 수 없음 (aria-disabled 설명 없음)

아토믹 디자인 폴더 구조 문제

  • UI 중심 분류라 Product 같은 실제 도메인 단위 관리 불가능
  • 컴포넌트 분류 기준 모호 (Molecule인지 Organism인지 구분 어려움)
  • 개발자는 기능/도메인으로 생각하는데 Atomic은 디자인 계층 강요

개편 과정에서 집중한 부분

기능 및 도메인 중심의 폴더 구조

Before 문제

  • Atomic Design 기반으로 UI 컴포넌트가 도메인 규칙을 직접 처리함
  • Badge, Table, FormInput 등이 여러 도메인의 책임을 동시에 가져 복잡도 증가
  • 단일 페이지에 수백 줄의 로직이 몰려 유지보수 어려움

After 개선

  • pages / features / entities / shared 계층별 책임 분리
  • UI는 shared/ui로 이동하여 순수 UI 역할만 수행
  • 도메인 로직은 entities와 features로 이동
  • 공용 컴포넌트는 도메인을 전혀 모르게 하여 재사용성을 높임
  • Zod + React Hook Form으로 검증 로직 완전 분리

디자인 시스템 + Tailwind + CVA 기반 UI 표준화

어려웠던 점과 해결 방법

Tailwind 유틸리티 클래스 생성의 난관

어려웠던 점

  • tailwind.config.js 없이 CSS로 유틸리티 클래스를 만들고 토큰을 매핑하는 방식이 처음에는 낯설었음.
  • 특히 px → rem 변환을 자동화하는 과정(pxToRem 등)에서 이 값을 어떻게 CSS 변수를 통해 Tailwind에서 사용할 수 있게 만들까?가 막혔음
  • pxr 토큰(예: --12pxr: 0.75rem)을 대규모로 생성해야 했고, 이걸 테일윈드의 @utility 규칙과 연결하는 방식도 익숙하지 않았음.

해결 방법

  • px → rem 변환 스크립트로 pxr 토큰을 자동 생성하도록 구성.
  • 생성된 pxr 토큰을 테일윈드 기능(utility,--value())과 연결해 다음과 같은 형태로 커스텀 유틸리티를 안정적으로 선언
@utility p-pxr-* {
  padding: --value(--pxr-*);
}

...

원래는 16pxr 이런식으로 사용하려 했으나 위 방식과 같이 동적 생성할 경우 The dynamic portion marked by-* must appear once at the end.와 같은 에러 발생.


비즈니스 로직 분리 과정의 난관

어려웠던 점

  • 레거시 구조에서는 UI 컴포넌트 내부에서 비즈니스 로직이 뒤섞여 있어 전체 흐름을 파악하는 데 많은 시간이 필요했음.
  • Button/Badge/Table 같은 공용 UI가 도메인 규칙까지 처리해 어디까지가 UI이고 어디부터가 도메인인지 경계가 흐려져 있었음.

해결방법

  • FSD의 원칙을 바탕으로 컴포넌트 역할을 재정의하고 책임을 분리함.
  • 공용 UI 컴포넌트는 도메인을 전혀 모르는 상태로 재구조화
  • 도메인별 상태·검증·비즈니스 규칙은 entities와 features로 이동.
  • Zod를 통한 검증 스키마 도입으로 UI에서 하던 도메인 검증을 완전히 제거하고 단일 소스로 통합.
  • 결과적으로 구조가 안정되고, 기능 추가/수정 시 영향 범위가 크게 줄어듦.

리뷰받고 싶거나 질문하고 싶은 내용

DTO가 여러 도메인을 섞어 담고 있을 때, FSD 기준으로 어떤 Slice에 배치해야 하는가?

FSD를 적용하면서 가장 애매했던 부분은 서버에서 내려오는 DTO가 한 슬라이스의 순수한 도메인 형태가 아닌 경우였습니다.

예를 들어,

GET  /api/followers/post-summary;

이 API는 아래 형태처럼 User + Follower + Post 데이터가 혼합된 상태로 내려오는 경우가 있습니다.

  • 유저 이름, 프로필 이미지
  • 팔로워 / 팔로잉 정보
  • 최근 게시글의 summary 정보

이런 경우 FSD 관점에서 해당 DTO를 어느 entities 슬라이스에 배치해야 하는지 또는 각 엔티티로 분리해야 하는지 등의 결정이 애매했습니다.

코치님은 엔티티 내부 슬라이스 배치 기준은 어떤 원칙을 따르시나요?

  • DTO에 여러 슬라이스의 데이터가 혼합되어 있을 때, 어떤 기준으로 분리 혹은 배치하시는지?
  • 애매한 DTO는 server endpoint 기준으로 묶는지? (예: entities/follower, entities/post 등)

과제 피드백

잘했어요 세준! 이번 과제는 레거시 코드를 현대적인 디자인 시스템으로 개편하면서 컴포넌트 설계 원칙을 체득하는 것이 목표였습니다. FSD 아키텍처를 적용해서 pages/features/entities/shared로 계층별 책임을 분리하고, Primitive/Semantic 2단계 디자인 토큰 시스템까지 구축한 시도가 인상적이었어요. 특히 Zod + React Hook Form으로 검증 로직을 완전히 분리하고, CVA로 variant를 타입 안전하게 관리한 부분도 좋았습니다. Tailwind 커스텀 유틸리티를 @utility 규칙으로 생성하면서 px → rem 변환 스크립트까지 만든 건 정말 수고가 많았을 거예요.

지금 과제를 하는 데에는 힘이 많이 들어갔어요. 그리고 처음에 배울때에는 이렇게 오버엔지니어링이다 싶을 정도로 끝까지 가본다음에 어디까지는 필요하고 어디까지는 필요없구나는 몸소 느껴보는거 너무 좋습니다. 실제 FSD를 막상해보면 조금 과하다는 생각이 들었을 거에요. 지금 폴더명이나 파일명과 내용들이 너무 세세하게 잘게 되는 게 느껴졌을텐데 이렇게 되면 되려 내가 외울게 많아지는 부작용이 있어요. 앞으로 FSD를 계속 해보면서 가급적 외워야 하거나 새로 만들어야 하는게 적어지는 방향으로 만들어 보면 좋을 거에요

Q) DTO가 여러 도메인을 섞어 담고 있을 때, FSD 기준으로 어떤 Slice에 배치해야 하는가?

정말 좋은 질문이에요. 이건 FSD를 적용할 때 가장 자주 만나는 딜레마인데, 사실 명확한 정답은 없습니다. 다만 판단 기준은 있어요.

가장 쉬운 방법은 그냥 endpoint 기준을 따라가는 것입니다. 우리팀의 백엔드가 잘하는 사람이건 그렇지 못한 사람이건 디버깅을 하거나 문제를 확인할때에는 백엔드의 구조를 알아야만 하죠. 그러니 /api/followers/*로 시작하는 건 다 entities/follower/api/에 두는 거죠. 이게 가장 예측하기 쉽고 "외울게 적고 예측하기 쉬운가?"를 만족하니까요.

figma처럼 백엔드는 그저 데이터를 동기화 해주는 도구이고 클라이언트에서 데이터 구조가 복잡한 경우라면 그럴때는 백엔드 API와 독립적으로 그리고 DTO를 받자마자 각 엔티티 형태로 변환하는 어댑터를 만들면 좋습니다. 이렇게 하면 API 레이어에서는 DTO를 받지만, 그 위 레이어에서는 깔끔하게 분리된 도메인 모델을 받게 만드는 거죠. 그러면 DTO의 혼란이 상위로 전파되지 않아요.

근데 대부분의 경우에는 클라이언트 중심의 어플리케이션을 하기 보다는 백엔드 중심으로 이미 설계가 다 끝나다보니 너무 완벽하게 분리하려고 하지 마세요. 실무에서는 서버가 주는 대로 받아야 할 때가 훨씬 더 많답니다. 하하.

저도 여러가지 방법들을 시도하면서 깨달은 바이니 알려준건 참고만 하시고 직접 해보면서 비교해가며 느껴보기를 바래요. 수고하셨습니다!
