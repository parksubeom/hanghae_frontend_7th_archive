과제 셀프회고

과제를 하면서 내가 알게된 점, 좋았던 점은 무엇인가요?

1. 계층 분리

  • 프로젝트 계층을 분리해 각 레이어의 책임이 명확해졌습니다.
  • 흐름이 예측 가능해져 기능 추가·수정 시 고민이 크게 줄었습니다.
// 예시

1. addItemToCart(entities - utils) // 도메인 유틸
2. cartReducer(entities - model) // 도메인 모델
3. useCart(entities - hook) // 도메인 모델 + 상태 연결 관리
4. useCreateCart(features - hook) // 기능 단위 로직 + UI 피드백

2. Adapter(View Mapper)

  • 도메인 모델을 ViewModel로 변환하는 Adapter를 두어 UI와 비즈니스 로직이 분리되었습니다.
  • 컴포넌트는 렌더링에만 집중할 수 있어 코드가 단순해졌습니다.
// Before
<button disabled={product.stock <= 0}>
  {product.stock > 0 ? '장바구니 담기' : '품절'}
</button>

// After
<button disabled={vm.isAddToCartDisabled}>{vm.buttonText}</button>

알게 된 점

  • 의존성을 단방향으로 유지하니 변경 범위가 명확해졌습니다.
  • 유틸 함수를 도메인 유틸과 공용 유틸로 분리해 재사용성과 구조적 일관성이 높아졌습니다.
  • 비즈니스 규칙 변경 발생 시 엔티티 레이어만 수정하면 되었고, 상위 계층은 영향이 없었습니다.
  • 대부분의 로직이 순수 함수로 작성되어 모킹 없이 단위 테스트가 가능했습니다.

이번 과제에서 내가 제일 신경 쓴 부분은 무엇인가요?

이번 과제에서 가장 신경 쓴 부분은 계층 간 책임을 명확히 분리하고, 특히 Entity 계층에서 UI 의존성을 완전히 제거하는 것이었습니다.


1. Entity의 UI 읜존성 제거

  • 엔티티는 순수 비즈니스 로직과 도메인 컴포넌트(순수 컴포넌트)만 담당하도록 설계했습니다.
  • Toast 같은 UI 피드백은 Features 계층에서만 처리되도록 분리해 Entity의 재사용성을 높였습니다.
  • Features 계층에서는 Entity에서 발생한 결과를 받아 UI 피드백을 결정하는 구조로 구성했습니다.
  • 이를 통해 테스트 시 비즈니스 로직을 독립적으로 검증할 수 있게 되었습니다.

2. Feature 내부에서도 로직과 UI 파일 분리

  • Feature 계층에서도 상태와 UI를 분리하였습니다.
  • 이를 통해 UI에 대한 변경사항은 ui를 로직에 대한 변경사항은 hook을 통해 수정함으로써 책임을 분리하였습니다.
Hook
- 상태 관리
- 도메인 호출
- 이벤트 핸들러
- TOAST와 같은 UI 피드백 결정

Component
- 렌더링 책임
- 스타일 관리

3.유틸 함수 계층화 및 재사용성 고려

  • 유틸 함수들을 도메인 유틸과 공용 유틸로 분리하였습니다.
  • 이를 통해 비즈니스 규칙 변경 시 도메인 유틸만 수정하면 되고, 공용 유틸은 여러 기능에서 재사용 가능해졌습니다.
  도메인 유틸
- 재고/가격/할인 계산

공용 유틸

- 문자열 포맷, 타입 체크 등

실무에서는 DTO를 컴포넌트에서 직접 사용하고 포맷팅·UI 변환 로직이 여러 곳에 흩어져 있어 유지보수가 어려웠는데, 이번 경험을 통해 이를 개선할 구체적인 방향을 잡게 되었습니다.

앞으로는 데이터를 DTO → Domain → ViewModel로 단계별로 정리하는 과정을 실무에 적용해, 비즈니스 규칙과 UI 표현을 분리하고 유지보수성을 높이고자 합니다.


DTO -> Domain Model

  • API에서 받은 데이터(DTO)를 바로 쓰지 않고 네이밍, 타입, 날짜, 숫자 단위 등 비즈니스 친화적인 형태로 변환
  • Domain 계층에 비즈니스 규칙을 모아두어 규칙 변경 시 한 곳만 수정하면 전체 반영되도록 만들기

Domain Model → ViewModel 변환(Adapter/Mapper)

  • UI에서 필요한 형태로 완전히 가공된 ViewModel을 생성
  • UI 컴포넌트 인터페이스에 맞게 Mapper 한 곳에서 중앙화

리뷰 받고 싶은 내용이나 궁금한 것에 대한 질문 편하게 남겨주세요 :)

도메인 모델 + 상태(useCart) 내부에서 에러를 throw하는 방식이 적절한지에 대한 고민이 있습니다.

현재 validateAddToCart에서 재고 부족 시 throw new Error()로 검증 실패를 표현하고 있고, 해당 에러는 상위 Feature 레이어에서 catch하여 UI 피드백(토스트 등)으로 처리하고 있습니다.

export const validateAddToCart = (state: CartState, product: ProductWithUI): void => {
  const remainingStock = getRemainingStock(state.items, product);

  if (remainingStock <= 0) {
    throw new Error('재고가 부족합니다!');
  }
};


// useCart.ts
const addToCart = useCallback(
  (product: ProductWithUI) => {
    validateAddToCart(state, product);   // ❓ 여기서 throw가 적절한가?
    dispatch(cartActions.addItem(product));
  },
  [state]
);

하지만 다음과 같은 이유 때문에 이 방식은 올바르지 않은 설계라는 생각이 듭니다.

  • 엔티티/도메인 계층은 순수성과 사이드 이펙트가 없어야한다.
  • 에러를 던지는 행위 자체가 사이드 이펙트다.
  • 그렇다면 검증 자체를 features에서 수행해야하는가?

궁금한 점

  • 검증 로직이 상위 레이어에서 수행하는 것이 적절한지?

Q) 도메인 모델 + 상태(useCart) 내부에서 에러를 throw하는 방식이 적절한지에 대한 고민이 있습니다. 현재 validateAddToCart에서 재고 부족 시 throw new Error()로 검증 실패를 표현하고 있고, 해당 에러는 상위 Feature 레이어에서 catch하여 UI 피드백(토스트 등)으로 처리하고 있습니다.

하지만 다음과 같은 이유 때문에 이 방식은 올바르지 않은 설계라는 생각이 듭니다.

  • 엔티티/도메인 계층은 순수성과 사이드 이펙트가 없어야한다.
  • 에러를 던지는 행위 자체가 사이드 이펙트다.
  • 그렇다면 검증 자체를 features에서 수행해야하는가?

궁금한 점

  • 검증 로직이 상위 레이어에서 수행하는 것이 적절한지?
  • 혹은 Result/Either와 같은 패턴으로 검증 실패를 도메인이 표현하고 상위 레이어에서 UI 피드백을 처리할지?

A) throw를 하는 부분은 크게 문제되지 않는다고 생각해요 ㅎㅎ 사이드 이펙트라고 보기도 어렵지 않나!? 라는 생각입니다. 사이드 이펙트라는건 시간의 흐름에 따라 달라지는 함수인데, throw가 있다고 해서 input/output에 대한 일관성이 깨지는건 아니라서요!

혹은 제안해주신 "혹은 Result/Either와 같은 패턴으로 검증 실패를 도메인이 표현하고 상위 레이어에서 UI 피드백을 처리할지?" 처럼 진행해도 좋다고 생각합니다.

검증하는 구간은 사실 팀 내의 규칙이 제일 중요하다고 생각해요. 가령, 지금은 addToCart 에서 진행하고 있는데, UI에서 버튼을 클릭하는 시점에 혹은 수량을 조절하는 시점에 이미 오류 여부를 알 수 있다고 생각해요. 그런데 꼭 해당 레이어에서 진행할 필요는 없죠.
