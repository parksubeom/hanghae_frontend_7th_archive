기본과제

목표 : 전역상태관리를 이용한 적절한 분리와 계층에 대한 이해를 통한 FSD 폴더 구조 적용하기

  • 전역상태관리를 사용해서 상태를 분리하고 관리하는 방법에 대한 이해
  • Context API, Jotai, Zustand 등 상태관리 라이브러리 사용하기
  • FSD(Feature-Sliced Design)에 대한 이해
  • FSD를 통한 관심사의 분리에 대한 이해
  • 단일책임과 역할이란 무엇인가?
  • 관심사를 하나만 가지고 있는가?
  • 어디에 무엇을 넣어야 하는가?

체크포인트

  • 전역상태관리를 사용해서 상태를 분리하고 관리했나요?
  • Props Drilling을 최소화했나요?
  • shared 공통 컴포넌트를 분리했나요?
  • shared 공통 로직을 분리했나요?
  • entities를 중심으로 type을 정의하고 model을 분리했나요?
  • entities를 중심으로 ui를 분리했나요?
  • entities를 중심으로 api를 분리했나요?
  • feature를 중심으로 사용자행동(이벤트 처리)를 분리했나요?
  • feature를 중심으로 ui를 분리했나요?
  • feature를 중심으로 api를 분리했나요?
  • widget을 중심으로 데이터를 재사용가능한 형태로 분리했나요?

심화과제

목표: 서버상태관리 도구인 TanstackQuery를 이용하여 비동기코드를 선언적인 함수형 프로그래밍으로 작성하기

  • TanstackQuery의 사용법에 대한 이해
  • TanstackQuery를 이용한 비동기 코드 작성에 대한 이해
  • 비동기 코드를 선언적인 함수형 프로그래밍으로 작성하는 방법에 대한 이해

체크포인트

  • 모든 API 호출이 TanStack Query의 useQuery와 useMutation으로 대체되었는가?
  • 쿼리 키가 적절히 설정되었는가?
  • fetch와 useState가 아닌 선언적인 함수형 프로그래밍이 적절히 적용되었는가?
  • 캐싱과 리프레시 전략이 올바르게 구현되었는가?
  • 낙관적인 업데이트가 적용되었는가?
  • 에러 핸들링이 적절히 구현되었는가?
  • 서버 상태와 클라이언트 상태가 명확히 분리되었는가?
  • 코드가 간결하고 유지보수가 용이한 구조로 작성되었는가?
  • TanStack Query의 Devtools가 정상적으로 작동하는가?

최종과제

  • 폴더구조와 나의 멘탈모데일이 일치하나요?
  • 다른 사람이 봐도 이해하기 쉬운 구조인가요?

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

  1. FSD 아키텍처의 실전 적용
  • 계층별 책임이 명확히 구분되면 코드 위치를 고민하는 시간이 줄어든다는 것을 체감했습니다
  • entities는 비즈니스 엔티티 자체, features는 사용자 인터랙션, widgets는 독립적 UI 블록으로 구분하니 자연스럽게 관심사가 분리되었습니다
  1. DTO/Mapper 패턴의 효용
  • API 응답 타입(PostDto)과 도메인 타입(Post)을 분리하면 백엔드 변경에 유연하게 대응할 수 있음을 알게 되었습니다
  1. 쿼리 키 팩토리 패턴
  • 쿼리 키를 계층적으로 관리하니 invalidation이 정말 편해졌습니다
  1. 낙관적 업데이트의 구현 방법
  • useOptimisticMutation 훅을 만들어 재사용하니 모든 mutation에서 일관된 UX를 제공할 수 있었습니다
export function useDeleteComment(postId: number) {
  return useOptimisticMutation<void, Error, number, PaginatedData<Comment>>(
    {
      mutationFn: commentService.deleteComment,
    },
    {
      optimisticKey: commentKeys.listByPost(postId),
      updateFn: (oldData, commentId) => {
        // 삭제할 댓글을 목록에서 제거
        return {
          ...oldData,
          data: oldData.data.filter((comment) => comment.id !== commentId),
          total: oldData.total - 1,
        }
      },
      strategy: "never",
    },
  )
}

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

  1. 계층별 단일 책임 원칙 지키기
  • 처음엔 "이게 features인가 entities인가?" 고민이 많았지만, "사용자 행동"이면 features, "데이터 모델"이면 entities로 구분하니 명확해졌습니다
  1. 서버 상태와 클라이언트 상태의 완전한 분리
  • 모든 API 호출을 TanStack Query로 전환하고, 검색 파라미터만 별도로 전역 상태로 관리했습니다
  1. 재사용 가능한 추상화 만들기
  • useOptimisticMutation, createMutationOptions 등 공통 로직을 shared 계층으로 추출했습니다

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

  1. 크로스 슬라이스 참조의 경계
  • entities/user/@x/post처럼 엔티티 간 관계를 어디까지 허용할지 아직 감이 부족합니다
  • entities를 참조가 발생할 때의 베스트 프랙티스가 더 필요합니다

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

  1. 프로젝트에 계층 분리 개념 적용 FSD에서 책임을 나누는 계층 분리 개념을 프로젝트 성격에 맞게 녹여보고자 합니다. 기능 확장이나 구조 변경 시 리팩토링 비용을 크게 줄일 수 있을 것으로 느꼈습니다.

  2. DTO / Mapper 패턴을 통한 변경에 강한 구조 유지 API 응답 타입과 도메인 모델을 분리하고 Mapper로 변환하는 구조를 기본 전략으로 가져가고 싶습니다. 백엔드 API 변경이나 다른 API로의 전환이 발생하더라도, 도메인 로직과 UI를 최소한의 수정으로 유지할 수 있다는 점에서 실무적으로 큰 장점을 느꼈습니다.

챕터 셀프회고

클린코드와 아키테쳑 챕터 함께 하느라 고생 많으셨습니다! 지난 3주간의 여정을 돌이켜 볼 수 있도록 준비해보았습니다. 아래에 적힌 질문들은 추억(?)을 회상할 수 있도록 도와주려고 만든 질문이며, 꼭 질문에 대한 대답이 아니어도 좋으니 내가 느꼈던 인사이트들을 자유롭게 적어주세요.

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

이번 경험을 통해 앞으로 코드를 작성할 때 "일단 동작하게 만든다"보다 "나중에 이 코드를 다시 보게 될 나 또는 팀원이 얼마나 쉽게 이해할 수 있을까"를 더 먼저 생각하게 될 것 같습니다.

계층별 책임을 명확하게 가져가 파일 위치를 찾거나 어디에 두어야하는지 등에 대한 고민이 줄어들 것 같습니다.

결합도 낮추기: 디자인 패턴, 순수함수, 컴포넌트 분리, 전역상태 관리

이전에는 컴포넌트가 커지는 걸 보면서도 어쩔 수 없는 복잡도라고 넘겼던 부분들이 많았던 것 같습니다. 이번 챕터를 거치며 느낀 건, 대부분의 복잡도는 역할이 섞여 있기 때문에 생긴다는 점이었습니다.

앞으로는 컴포넌트를 작성할 때 UI, 상태, 비즈니스 로직이 한곳에 모이고 있지는 않은지 한 번 더 의식적으로 점검해보게 될 것 같습니다. 순수함수나 커스텀 훅으로 분리했을 때 얻는 안정감과 가독성을 경험해보니, 이 방식이 깔끔해 보이기 위한 리팩토링이 아니라 실제 개발 속도를 높여준다는 것도 알게 되었습니다.

응집도 높이기: 서버상태관리, 폴더 구조

폴더 구조와 관련해서는 이전처럼 “관례적으로 이렇게 하니까”가 아니라 이 코드의 책임이 무엇인지를 먼저 기준으로 삼게 될 것 같습니다.

FSD를 그대로 따라야 한다기보다는, features / entities / shared처럼 역할을 나누는 사고방식 자체가 구조를 잡는 데 큰 도움이 된다는 걸 느꼈습니다. 앞으로도 프로젝트 성격에 맞게 구조는 달라질 수 있겠지만, “이건 사용자 시나리오인가?”, “이건 도메인에 가까운가?”라는 질문은 계속 가져가고 싶습니다.
