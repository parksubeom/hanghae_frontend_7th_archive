과제 체크포인트

기본과제

Medium

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. medium.useEventOperations.spec.tsx > 아래 toastFn과 mock과 이 fn은 무엇을 해줄까요? 1. - toastFn 은 같은 UI를 여러상황에 따라 다른 텍스트를 제어 할 수가 있다. - 함수/텍스트가 호출 되었는지 확인하는 용도로 사용한다. 2. mock은 toastFn에서 여러상황을 만들 수 있는 상태 값을 비교하거나 사용하기 위해 필요한것이다. 즉, 실제로 렌더링 하지 않고 테스트코드의 검증을 할 수 있게 해주는 핵심 요소라고 할수 있다.

Q. medium.integration.spec.tsx > 여기서 ChakraProvider로 묶어주는 동작은 의미있을까요? 있다면 어떤 의미일까요? 1. 앱/웹의 전체설정을 수정 할 수 있게 하기 위해서이다. => 색상테마, 한/영 버전,

Q. handlersUtils > 아래 여러가지 use 함수는 어떤 역할을 할까요? 어떻게 사용될 수 있을까요? 1. 목데이터를 조작하게 해주는 핸들러를 등록하거나 덮어쓰게 해준다. => 글 작성, 글 수정, 삭제

Q. setupTests.ts > 왜 이 시간을 설정해주는 걸까요? 1. 테스트를 하는 동안 시간을 동일하게 해주기 위해서이다. => 시간이 고정적이야 테스트 결과가 동일하게 나오기 때문이다.

심화 과제

  • App 컴포넌트 적절한 단위의 컴포넌트, 훅, 유틸 함수로 분리했는가?
  • 해당 모듈들에 대한 적절한 테스트를 2개 이상 작성했는가?

과제 셀프회고

과제 회고록

  1. 배움과 성취
  • 테스트 코드 작성 과정에서 describe, it, expect, toEqual 사용법을 이해하게 됨.
  • Easy 문제를 반복하며 패턴을 익히고, 함수 동작을 분석해 테스트 코드를 작성하는 경험을 얻음.
  • 통합 테스트 작성 시 사고 과정을 이해함: 화면 렌더링 → 객체 불러오기 → Form 찾기 → 요소 부모 찾기 → 버튼 클릭 → 수정 → 검증
  • 프론트엔드가 수행하는 일을 UI 관점뿐 아니라 기능 단과 전체 흐름까지 이해하는 계기가 됨.

= QnA 참여를 통해 테스트 코드 작성 중 생기는 불안과 의심이 정상임을 깨닫고, 부담 없이 접근할 수 있게 됨.

  1. 어려움과 도전
  • 자바스크립트 이해 부족으로 함수 해석 및 테스트 코드 작성 방향 파악이 어려움.
  • 함수 동작 원리를 이해하지 못해 it 테스트 문장 해석에 어려움 발생.
  • 테스트 범위 설정, 모킹(mocking), 객체 구성, 타입 지정 방법 등에서 고민 발생.
  • 팀원과 멘토의 전문 용어가 익숙하지 않아 학습 과정에서 번거로움.
  1. 문제 해결 및 학습 전략
  • 문제를 바로 질문하기보다, 해결 방법을 스스로 찾아 시도하며 학습.
  • 함수나 개념 이해 시 GPT뿐 아니라 구글링을 통해 직접 학습하고 주석으로 정리 후 검증.
  • 다른 사람 코드와 GPT를 통해 교차 검증하며 학습 불확실성을 해소.
  • 핵심적인 부분만 테스트 코드로 작성하며, 불필요한 복잡함을 피함.
  1. 느낀 점과 깨달음
  • 코드는 정답이 없으며, 더 좋은 코드를 작성하기 위해 항상 고민이 필요함.
  • 반복적인 연습과 문제 해결 경험이 쌓일수록 점점 접근 방식이 명확해짐.
  • 초기 불안과 부담감은 자연스러운 과정이며, 경험을 통해 자신감을 얻음.

기술적 성장

  • 새로 학습한 개념

    1. 테스트 코드 작성법
    2. 단위 테스트
    3. 통합테스트
    4. 기능 함수를 만들때 제작해야하는 순서 활용하는 방법

  • 기존 지식의 재발견/심화

  • 구현 과정에서의 기술적 도전과 해결 문제: 퍼블리싱만 해왔기 때문에 자바스크립트 이해도가 부족했습니다. 이로 인해 테스트 코드를 작성할 때 필요한 요소와 코드 동작 원리를 이해하기 어려웠습니다.

    해결: 포기하지 않고 꾸준히 검색하고 관찰하며 학습했습니다. 단순히 코드를 작성하는 것이 아니라, 메모를 해가며 하나씩 차근차근 원리를 이해하며 작성했습니다.

코드 품질

-내 코드 품질 : 하

  • 특히 만족스러운 구현 Easy 난이도 Unit 테스트 코드 작성이 만족스러웠습니다 => 첫 문제를 푸는 데 시간이 오래 걸렸지만, 같은 패턴으로 하나씩 풀어가다 보니 나중에는 재미도 붙었습니다.

  • 리팩토링이 필요한 부분 mport한 모듈을 온전히 활용하지 못했습니다. => import한 요소를 적극 활용하면 코드 리팩토링에 도움이 될 것 같습니다.

  • 코드 설계 관련 고민과 결정 QnA에서 간단한 기능은 테스트 코드를 작성할 필요가 없다는 조언을 받았습니다. => 따라서 최대한 간단 명료하고 핵심적인 부분만 테스트 코드로 작성하려고 노력했습니다.

학습 효과 분석

  • 가장 큰 배움이 있었던 부분

    1. 테스트 코드라는 것 자체를 처음 알았습니다.
    2. 프론트단의 프로세스를 알게 되었습니다.

  • 추가 학습이 필요한 영역

    1. 자바스크립트
    2. 함수의 활용도
    3. 타입스크립트

  • 실무 적용 가능성 아직까지는 잘 모르겠습니다.

과제 피드백

  • 과제에서 모호하거나 애매했던 부분

    1. 테스트 코드를 처음 작성하다 보니 “이게 맞나?” 하는 생각이 머릿속을 맴돌아 다음 단계로 넘어가기 어려웠습니다

  • 과제에서 좋았던 부분 1.Easy 문제처럼 같은 패턴을 반복하면서 연습할 수 있어 숙달에 도움이 되었습니다.

피드백 받고 싶은 내용을 구체적으로 남겨주세요

  • 현재 자바스크립트와 테스트 디스크립션 이해도가 부족하여, 테스트 코드에서 무엇을 확인해야 하는지 감을 잡기 어렵습니다.
  • 문제를 풀 때 “왜?”라는 질문을 배제하고, 단순히 검증할 때는 expect, 비교할 때는 toEqual을 사용하여 접근했습니다.

요청

  • 자바스크립트 이해도가 부족하여 테스트 코드 작성이 더욱 어렵게 느껴집니다.
  • 자바스크립트 숙련도를 높이기 위한 방법이 궁금합니다. (예: 추천 도서, 학습 방식, 자바스크립트 사고 방식을 구축하는 방법) [src/mocks/handler.ts]
  • 사전 스터디 문서에서 API를 읽는 것은 했지만, 실제로 사용하는 것은 처음이라 async와 await 개념 이해가 어렵습니다.
  • 관련하여 참고할 만한 자료나 책이 있다면 추천 부탁드립니다.

기타

  • 프론트엔드가 하는 일을 막연하게만 알고 있었지만, 이번 과제를 통해 디테일하게 이해할 수 있는 계기가 되었습니다.
  • 처음 국비교육으로 퍼블리싱을 배웠을 때의 느낌이 떠올랐습니다.
  • 코드는 정답이 없다는 점을 다시 한번 깨닫고, 어떻게 하면 더 좋은 코드를 작성할 수 있을지 고민하는 계기가 되었습니다.
  • PR 사용법/ 작성법 작성 해두면 좋은점 등

좋은데요! 사실 뭔가 다양한걸 먼저 경험하기보다는 하나를 깊게 경험한 뒤에 다양한 걸 공부하는게 좋다고 생각하는 편인데요. 저는 자바스크립트 코드 작성 방법 자체에 익숙해지는걸로는 코딩문제풀이를 하는것을 추천드리고, 개념을 공부하는건 책을 읽으면서 정리하는 것을 추천드려요! 익숙하지 않은 개념은 실제로 우리가 시험공부하듯이 외우기도 하구요. 강의를 듣는것도 좋지만, 앞으로 우리가 하는 제일 기반이 되는 지식이기 때문에 깊게 천천히 완전 머리에 남도록 하는게 좋거든요. 저는 책을 하나 잡고 누군가에게 강의해주듯이 하면서 책을 한권 다 외웠었는데 그런식도 매우 오래걸리고 피곤하지만 지금까지도 도움이되는 좋은 내용이였던거같아요. 책은 딥다이브도 좋고 코뿔소 책도 좋은데, 전체 내용을 빠짐없이 다루는 책이면 좋을것 같습니다! async와 await의 개념은 당장 필요한 부분이니 강의같은걸 참고해서 우선적으로 개념을 잡고 사용해보시면 좋을것 같구요.
