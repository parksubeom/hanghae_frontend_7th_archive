기술적 성장

  1. 1주차 때 배운 테스트들을 실제 프로젝트에서 어떻게 사용하면 좋을지 배웠다.
  • 테스트방법과 전략은 잘못 된게 없고, 어떤 프로젝트, 어떤 것을 중심으로 하는지의 따라 알맞은 것을 골라 쓰면 되는 것 같다.
  1. E2E 테스트랑 통합 테스트랑 헷갈렸는데 이번 과제를 하면서 정확하게 알게 되었다.(핵심은 관점)
  • 같은 테스트여도 개발자 관점이면 통합 테스트가 되고 사용자 관점에서는 E2E테스트가 될 수 가 있다.
  1. 10명중 3명만 테스트코드를 작성하지만 그 3명은 꾸준히 작성하는 이유를 알게 되었다.
  • 테스트 코드를 작성하다 보면 함수가 작게 쪼개진다. 왜?? 테스트를 통과하기시키기위해 함수를 작게 쪼개고, 리팩토링을 많이하게 된다.
  • 버그를 미리차단 왜?? 단위 테스트를 하면 어디서 무엇이 잘못 되었는지 한번에 알 수 있다.
  • 리팩토링이나 기능 추가가 겁이 안난다. 왜?? 인수인계 받은 프로젝트에 새로운 기능을 추가하고 이전의 테스트코드를 실행해보면 어디서 오류가 나는 지 알 수 있으니 새로운 기능 추가에 대한 자신감이 생긴다.(스트레스 감소)
  • 협업 효율이 높아진다. 왜?? 자동으로 테스트 코드가 문서화가 되니 새로 조인한 팀원들이 해당 테스트코드를 보고 업무 파악이 수월해진다.
  • 나의 코드의 근거를 만드는 일이다. 왜?? 함수만 봤을 때 함수 왜 만들엇을까?라는 생각이 들면 테스트코드가 그 함수를 제작한 이유가 된다.

학습 효과 분석

스토리북의 필요성

  1. 어디에 공통 클래스를 사용했는지 다 기억 못한다. 그래서 영향이 가면 안되는 부분에 영향이 갈 수도 있다. 이러한 것을 방지하기위해 사용하면 좋다.
  2. 규모가 큰 프로젝트에 유리하다.

테스트 피라미드 전략

  1. 백엔드의 적합적
  2. 한번 작성하면 변경되지 않은 것들이 대부분 단위 테스트들이다.

테스트 트로피 전략

  1. 프론트엔드 즉 UI에 적합하다.
  2. UI는 대부분 통합테스트이다. 왜냐하면 대부분 동작하는 것이 많기 때문이다.
  3. 대부분이 2가지 이상이 연결되어 있다. ex) 버튼 클릭 >> 모달 등장 ex) input 입력 >> 보내기 버튼 클릭

테스트 코드의 대한 나의 총평

  1. 테스트 코드는 정말 좋은 것이다. -> 큰 프로젝트를 장기적으로 유지해 나가는 회사가 없는 것 같다.
  2. 테스트 코드를 작성하면 기업신뢰도가 올라간다. -> 대표가 가장 먼저 거절 할 것 같다.
  3. 프로젝트의 확신이 있는 상태에서 도입후 차분히 작성해가면 좋을 것 같다.
  4. 이미 돌아가고 있는 프로젝트에 도입하려면 어디서부터 작성할지 생각을 많이 해야 할 것 같다.

Q. 스토리북를 셋팅하는데 많은 어려움이 있었습니다. 또 커서,클로드 등등 같은 도구 등들을 이번 항해를 하면서 알게 되었습니다. 프론트엔드 개발자가 알고 있어야 할 배경지식(?) 같은 것을 쉽게 얻거나 볼 수 있는 사이트? 책? 같은 것들이 있을까요?

A. 배경지식이라고하면 FE는 아니지만 기본적인 지식들 말씀이실까요? 네트워크 같은 것 말씀하시는 것인지 ㅎㅎ 근데 그런건 스토리북 세팅하고 상관없는 문제 인것 같은데욥. 스토리북을 세팅하시는데 어려움이 있었다면 그것을 다음부터는 쉽게 하고 싶다면 사실 무엇을 봐야하는지는 저도 잘 모르겠습니다 :) 사실 항상 어려워요. 도구들은 각각의 설치법도 있고 또 한 번 설정했다고해서 끝나는 것이 아니라 매번 업데이트해줘야하고 피곤하죠ㅜㅜ 그런데 이게요 이런 세팅을 자주하다보면 또 노하우가 생겨서 점점 쉽게 되요 ㅎㅎ 이런 것을 정리한 책은 없는 것 같고. 그냥 자주해보시면 될 것 같습니다 :) 너무 조급해하지 마시고요!

그냥 전반적인 FE소식은 저는 뉴스레터들을 통해 얻습니다~ 요즘 잘 못보고 있지만요 ㅎㅎ
