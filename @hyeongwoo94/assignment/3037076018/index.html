개편 목표

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

Before 패키지에서 발견한 문제점

button 문제점

  1. 도메인 특화: user, post 같은 특정 도메인에 종속됨
  2. 비즈니스 로직 포함: UI 컴포넌트가 비즈니스 규칙을 판단함
  3. 재사용 불가능: 다른 프로젝트나 도메인에서 사용 불가능
  4. 타입 안정성 부족: any 타입 사용으로 런타임 에러 가능
  5. 테스트 어려움: UI 테스트가 비즈니스 로직 테스트가 됨
  6. 관심사 분리 위반: UI와 비즈니스 로직이 섞여 있음

header문제점

  1. 스타일 값 하드코딩
  2. 디자인 토큰 시스템 부재
  3. 일관성 유지 없음
  4. 테마 변경 불가능
  5. 반응형 스타일 어려움
  6. aria 속성 부재

modal 문제점

  1. 접근성 부족
  2. 모달 열릴 때 포커스 이동 없음

badge 문제점

  1. 조건부 로직이 복잡성

table 문제점

  1. 테이블 컴포넌트가 CRUD 작업까지 담당
  2. 다른 도메인에서 재사용 불가
  3. 테이블 컴포넌트의 상태관리 모든게 한곳에 들어 있다.

alert 문제점

  1. 사용자 경험 미반영 => 게시글 작성 오류가 모달 뒤에서 나오니 확인 어려움

managementPage 문제점

  1. 뷰로직과 비즈니스 로직 분리가 부족함

shadcn/ui의 핵심 원칙

  1. 순수한 UI 컴포넌트: 도메인 로직 없음
  2. 타입 안정성: 완벽한 TypeScript 지원
  3. Tailwind CSS 기반: 인라인 스타일 최소화
  4. 접근성: Radix UI 기반 자동 지원
  5. 재사용성: 범용적이고 조합 가능
  6. 컴포넌트 조합: 작은 컴포넌트를 조합하여 복잡한 UI 구성

개편 과정에서 집중한 부분

자바스크립트, 리액트 지식 부족으로인한 하드코딩 하기 어렵다는 이슈 발생

  1. AI 최대 활용 => 복습을 위한 작업한 부분 명세서로 제작
  2. 완성 후 어떻게 코드가 변화 되었는지 분석 => cva/ 디자인토큰/ shadcn/storybook 코드 작성/사용방법/규칙 습득
  3. 비즈니스 로직과 뷰로직를 구분(?)하는 눈 습득
  4. 컴포넌트를 한번만 분리해서 사용하는 것이 아닌 컴포넌트를 조합해서 또 컴포넌트로 만들어서 사용 할 수 있다는 것을 깨달음

사용한 기술 스택 경험

redix-ui 이러한 라이브러리(?)들를 모르는 상태로 연관된 shadcn/ui를 사용하려니깐 막막했지만 발제를 읽으면서 과제의 흐름을 파악하고 '왜?'라는 질문으로 과제의 흐름을 이해하면서 하니 얼추 shadcn을 알게 되었다. =>프론트 개발의 생태계에서 살아남기 위해서는 관심과 애정을 갖고 깊게는 아니더라도 여러 라이브러리/트렌드 등을 정보를 갖고 있어야 원만한 개발자가 될 수 있을 것 같다.

기본 shadcn 작성법

// components/ui/button.tsx
import * as React from "react"
import { cva, type VariantProps } from "class-variance-authority"
import { cn } from "@/lib/utils"

const buttonVariants = cva(
  "inline-flex items-center justify-center rounded-md text-sm font-medium transition-colors", // 공통 클래스
  { // 해당 컴포넌트를 재사용할 때 컨트롤 할 클래스들
    variants: {
      variant: {
        default: "bg-primary text-white hover:bg-primary/90",
        ghost: "bg-transparent hover:bg-gray-100",
      },
      size: {
        sm: "px-2 py-1 text-xs",
        md: "px-4 py-2",
        lg: "px-6 py-3 text-base",
      }
    },
    defaultVariants: {
      variant: "default",
      size: "md",
    }
  }
)

export interface ButtonProps // 타입스크립트
  extends React.ButtonHTMLAttributes<HTMLButtonElement>,
    VariantProps<typeof buttonVariants> {}

//1. React.ButtonHTMLAttributes<HTMLButtonElement> 
//→ 일반 <button> 태그에서 쓸 수 있는 모든 props 허용 (예: onClick, disabled, type 등)
//2.VariantProps<typeof buttonVariants> 
//→ cva로 정의한 variant 옵션을 ButtonProps 타입에 포함 (예: variant, size)

export const Button = React.forwardRef<HTMLButtonElement, ButtonProps>( //컴포넌트의 핵심 코드
  ({ className, variant, size, ...props }, ref) => (
    <button
      ref={ref}
      className={cn(buttonVariants({ variant, size }), className)}
      {...props}
    />
  )
)

Button.displayName = "Button" //디버깅과 React DevTools용 이름을 지정



실제 부모에서 해당 컴포넌트를 사용하는 방법

기본 shadcn 핵심 기능

sva의 스타일을 자식 컴포넌트에게 그대로 전달이 가능!!


// components/ui/button.tsx
 import랑 sva,interface는 동일

export const Button = React.forwardRef<HTMLButtonElement, ButtonProps>(
  ({ className, variant, size, asChild = false, ...props }, ref) => {
    const Comp = asChild ? Slot : "button"  // Slot

    return (
      <Comp
        className={cn(buttonVariants({ variant, size }), className)}
        ref={ref}
        {...props}
      />
    )
  }
)

이렇게 Slot을 작성했다면
부모 컴포넌트에서 이렇게 작성해도 같은 스타일을 갖을 수가 있다.
<Button asChild><button>안녕</button></Button>
<Button asChild><div>안녕</div></Button>
<Button asChild><a>안녕</a></Button>

shadcn 가 위의 처럼 작동 할 수 있는 이유

테일윈드가 클래스 기반 스타일을 제공하는 라이브러리 radix가 접급성을 높여 slot과 asChild를 제공하는 라이브러리 이 두개를 합쳐서 만든 라이브러리가 shadcn/ui이다.

어려웠던 점과 해결 방법

작업 완료후 커스텀 된 버튼에는 키보드 탭 눌럿을 경우 focus가 안되는 오류 발견 컴포넌트 button만 수정하니 모든 버튼에 focus가 되는 경험 함 =>편리한데?

컴포넌트를 언제 어떻게 사용가능하게 만드는게 좋을까?

  1. 어떤 요소를 컴포넌트를 다룰지의 생각보다. 어떤 속성들을 다룰지를 생각해야한다.
  • UI 다양한 변형 => size, color, variant
  • 조건부 표시 여부 => show, hide
  • 상태 => checked, active, disaled
  • 동작패턴 => mode
  • 렌더링 데이터 => items, rows, option
  • 행동 이벤트 => click, input, change

나의 스타일

  • 위의 내용을 컨트롤할때 최우선으로는 클래스로 추가/제거
  • 사용하는 라이브러리/프레임워크의 따른 내부 기능을 사용.
