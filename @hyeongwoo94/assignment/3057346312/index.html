발제 요점정리

전제 조건: 프로젝트가 클 경우, 도메인 주도 설계 도입시, 핵심 로직이 존재하는 경우의 프로젝트 구축 시 연습 이유: 이전에는 프론트단의 복잡도가 높지 않았다. 현재 프론트단도 복잡도가 높아져서 디자인 패턴을 사용하면 좋다. 그래서 연습을 한다.

목표: 애플리케이션의 핵심 로직과 데이터 흐름을 정리해보자.

  1. 디자인_패턴을_프론트엔드에_적용할_수_있음 => UI 상태와 비즈니스 로직 실행 책임 분리
  2. 데이터_계산_액션을_구분할_수_있음 => 같은 비즈니스 함수를 순수함수와 조합해서 사용하는 함수를 구분 할 줄 알아야 함.
  3. 도메인_중심으로_코드를_구조화할_수_있음 => 폴더 구조를 ui 기준이 아닌 도메인 기준으로 하여 폴더의 응집도를 높여야 한다.

위 세가지를 하면 좋은 장점

  1. 유지보수성 향상 = ui,api 수정 없이 도메인 로직 모듈만 수정 할 수 있다.
  2. 테스트 용이성 극대화

결론: 꼭 이렇게 해야 한다기 보단 프로젝트의 성격에 따른 알맞은 방식을 정해서 하는게 좋다.

  1. 도메인 로직이란? 해당 프로젝트의 핵심 로직 == 비즈니스 로직

1-1. 도메인 로직 (Domain Logic)

  • 도메인 객체 자체의 내재된 규칙과 행위
  • 도메인 모델 (엔티티, 값 객체, 애그리게이트) 내부에 위치
  • Order.calculateDiscount() (할인 정책) 1-2. 비즈니스 로직 (Business Logic)
  • 도메인 객체들을 조율하고 목적을 이루는 전체적인 워크플로
  • 도메인 서비스 또는 애플리케이션 서비스에서 도메인 모델을 조합하여 구현
  • OrderService.placeOrder() (주문 생성 → 결제 요청 → 재고 감소)

  2. props drilling 발생

  • 왜 발생하는가?
  • 하위 컴포넌트에 필요한 props를 최상위 부모에게 모두 전달해야한다. 해결 방법 jotai를 사용
  1. jotai이란?
- 원자적 상태 관리 라이브러리
- 쉽게 말해 props가 전달되는 것을 따로 파일로 빼서 접근하는 방법
- 비슷한  도구들

1. Jotai: 작은~중간 규모, Props Drilling 해결, 간단함 중시
2. Redux: 대규모, 복잡한 상태 관리, 미들웨어 필요
3. Zustand: 간단한 전역 상태, Provider 없이 사용
5. Recoil: Facebook 생태계, 실험적 기능 필요
8. Context API: 매우 작은 프로젝트, 간단한 전역 상태

- 원리 및 작동 방식
Basic 방식 (Props Drilling) = 우체통
편지를 보내려면:
1. 집 → 우체통 (상위 컴포넌트)
2. 우체통 → 중간 집 (중간 컴포넌트)
3. 중간 집 → 중간 집 (또 다른 중간 컴포넌트)
4. 중간 집 → 최종 목적지 (실제 사용 컴포넌트)

- 편지(Props)를 여러 집을 거쳐 전달해야 함
- 중간 집들은 편지 내용을 모르지만 전달해야 함

Advanced 방식 (Jotai) = 전화
전화를 걸려면:
1. 전화번호(Atom)를 알고 있으면 직접 연결

- 중간 단계 없이 직접 통신
- 필요한 곳에서 직접 Atom에 접근

- useAtom (읽기 + 쓰기) == useState => 값 변경시 리렌더링
- useAtomValue (읽기 전용)  => 값 변경시 리렌더링
- useSetAtom (쓰기전용) =>리렌더링 없음
- 언제 무엇을 써야 하는가?
- 값을 화면에 표시하거나 조건문을 사용 === useAtomValue
- 값만 변경하고 읽지 않는다. === useSetAtom
- 값을 읽고 쓰는 양방향 바인딩 === useAtom

??왜 이러한 방식으로 나뉘어져 있을까??
1. 성능: 필요한 경우에만 리렌더링
2. 명확성: 의도가 코드에 드러남
3. 최적화: 메모이제이션과 함께 사용 시 효과적
4. 유지보수: 컴포넌트 역할이 명확해짐

  1. 퍼블리싱 경험이 구조 설계와 컴포넌트화의 기반이 되었다는 점
  • 퍼블리셔로서 클래스를 만들 때 항상 다른 사람이 해당 부분을 가져가도 깨지지 않도록 규칙을 정해 사용했습니다. 이 과정이 구조 설계와 컴포넌트화와 본질적으로 같다는 점을 깨달았습니다. 복붙 할때 필수로 최상단에 꼭 들어가야 하는 클래스 == 엔티티
  • 데이터도 컴포넌트화가 가능하다는 것을 알게 되었습니다. Jotai를 통해 상태를 독립적인 단위(Atom)로 나누고 재사용할 수 있다는 점이 HTML/CSS 컴포넌트화와 유사하다고 느꼈습니다.
  1. 프론트엔드와 퍼블리싱의 유사성 발견
  • HTML과 React 컴포넌트 HTML에서 섹션을 조각으로 나눠 어디서든 재사용 가능하게 만드는 것 == React에서는 컴포넌트로 구현
  • CSS와 상태 관리 SCSS/CSS에서 common클래스로 기본 스타일 틀 작성해 놓는 것 == React에서는 상태 관리 저장소(예: Jotai)가 그 역할을 수행

= 프론트엔드와 퍼블리싱의 차이는 사용하는 언어일 뿐이며, 핵심 개념은 동일하다는 것을 배웠습니다. = 웹 에이전시에서 html 노동을 한것처럼 react를 사용한 프로젝트를 반복작업하면 프론트 업무도 무리 없이 할 수 있을 것 같다.

폴더 구조 설계 이전 과제에서 팀원들이 폴더 구조로 어려움을 느낀다는 말을 들었지만, 당시에는 공감하기 어려웠습니다. 혼자 작업하며 나만 이해하기 쉬운 구조로 만들었고 남이 이것을 이해할까? 라는 생각을 전혀 해본적이 없었습니다. 이번 과제에서는 제가 퇴사한 후에도 다른 개발자가 코드를 이해하고 유지보수 할 수 있도록 폴더 구조를 설계하는데 집중을 했고 폴더구조를 도메인 중심으로 생각해보는 시간이었던 것 같습니다.

폴더 구조를 고민하며 알게 된 점들:

  • 순수 함수와 비즈니스 로직의 분리 순수 함수(Models)와 이를 조합한 비즈니스 로직(Hooks)을 구분해야 한다는 점을 배웠습니다.
  • 재사용 가능한 컴포넌트 설계의 어려움 Button 컴포넌트를 만들었지만, 디자인이 다른 버튼이 있을 수 있어 재사용성과 일관성 사이의 균형이 필요하다는 점을 깨달았습니다.
  • 과도한 세분화의 역효과 폴더를 너무 세분화하면 오히려 가독성과 이해도가 떨어진다는 점을 경험했습니다.
  • 문서화의 중요성 각 폴더별로 README를 작성해두면 신규 개발자의 온보딩과 이해에 도움이 된다는 점을 배웠습니다.

배운 것을 실무에 가서 레거시한 프로젝트에 어떻게 적용할 수 있을까?

  1. 게릴라 작전"처럼 티 안 나게, 하지만 확실하게 개선
  2. "순수 함수"로 로직 빼내기
  3. 나만의 작은 FSD" 만들기
  4. 신규 기능만 "제대로" 만들기

순수 함수로 빼내기 위해서는 프로젝트의 비즈니스 로직을 이해해야 한다. => 어떻게?

  1. 입력과 출력 찾기
  • 이 덩어리가 무엇을 받아서 무엇을 내뱉는지
  1. 제어문 (if,for)만 골라 보기 (흐름 파악)
  • 비즈니스 로직의 핵심은 대부분 "조건(규칙)"과 "반복(계산)"에 숨어 있습니다.
  1. 부작용과 순수 계산 분리하기
  • 코드를 읽다가 setState나 API 호출이 나오면 "이건 UI/인프라 로직이네" 하고 넘어가고, 데이터를 지지고 볶는 부분만 찾아서 별도 함수(models/xxx.ts)로 옮기면 됩니다
예시
const handleCheckout = () => {
  // 1. 데이터 준비 (Input)
  const total = cart.reduce((a, b) => a + b.price, 0);

  // 2. 비즈니스 로직 (순수 계산) - 추출 대상!
  let discount = 0;
  if (user.grade === 'VIP') {
      discount = total * 0.1;
  }
  if (total > 50000) {
      discount += 2000;
  }
  const finalPrice = total - discount;

  // 3. 부작용 (Output/Effect)
  setPrice(finalPrice); // 상태 변경
  api.postOrder(finalPrice); // API 호출
};

After (파악 후 분리)
1. 가운데 있는 if문 덩어리가 "할인 계산 정책"이구나!
2. 이걸 calculateDiscount(total, userGrade)라는 함수로 밖으로 뺍니다.
3. handleCheckout 안에는 계산 로직 대신 함수 호출 한 줄만 남깁니다.
4. 이 과정을 반복하면 거대한 컴포넌트가 뼈대만 남고 깔끔해집니다.
