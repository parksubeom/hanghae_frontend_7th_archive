배포 링크

기본과제 (Vanilla SSR & SSG)

Express SSR 서버

  • Express 미들웨어 기반 서버 구현
  • 개발/프로덕션 환경 분기 처리
  • HTML 템플릿 치환 (<!--app-html-->, <!--app-head-->)

서버 사이드 렌더링

  • 서버에서 동작하는 Router 구현
  • 서버 데이터 프리페칭 (상품 목록, 상품 상세)
  • 서버 상태관리 초기화

클라이언트 Hydration

  • window.__INITIAL_DATA__ 스크립트 주입
  • 클라이언트 상태 복원
  • 서버-클라이언트 데이터 일치

Static Site Generation

  • 동적 라우트 SSG (상품 상세 페이지들)
  • 빌드 타임 페이지 생성
  • 파일 시스템 기반 배포

심화과제 (React SSR & SSG)

React SSR

  • renderToString 서버 렌더링
  • TypeScript SSR 모듈 빌드
  • Universal React Router (서버/클라이언트 분기)
  • React 상태관리 서버 초기화

React Hydration

  • Hydration 불일치 방지
  • 클라이언트 상태 복원

Static Site Generation

  • 동적 라우트 SSG (상품 상세 페이지들)
  • 빌드 타임 페이지 생성
  • 파일 시스템 기반 배포

요점 정리

  1. SSR vs SSG
  • html를 언제 만드냐? == 요청이 들어 올때(관리자페이지) vs 빌드 시점 html 미리 생성(블로그)
  • 언제 사용하는게 좋은가? == 사용자마다 내용이 다른 경우 vs 내용 변경이 잦지 않을 경우
  1. 프론트 개발자가 알아야하는 최소 인프라 개념
  • 서버란? 요청을 받아서 응답을 주는 컴퓨터
  • HTTP / HTTPS 데이터 암호화 유무
  • DNS란? 도메인을 서버 IP로 변화해주는 시스템
  • Port란? 한서버 안에서 어떤 프로그램이 응답할지 구분
  • 클라이언트렌더링 / 서버 렌더링 ㄱ. CSR: 브라우저에서 JS로 그림 ㄴ. SSR: 서버에서 HTML 완성 ㄷ. SSG: 미리 HTML 생성
  • 배포란? 내 코드가 다른 사람도 접근 가능한 서버에 올라가는 것
  • CDN: 전세꼐에 미리 복사된 서버에서 가장 가까운 곳이 응답
  • 인증개념: 세션(쿠키) / 토큰
  • 프록시: 중간에서 요청을 대신함
  • CORS: 브라우저 보안 정책

전통적 SSR (2000년대)
  ↓ [문제: 페이지 새로고침, 느린 인터랙션]
  
CSR 등장 (2010년대 초) 
  → 빠른 페이지 전환, 부드러운 UX
  ↓ [문제: 느림, SEO, 소셜 공유]
  
SSR 재등장 (2010년대 후반)
  → CSR의 문제 해결
  ↓ [문제: 서버 부하, 응답 시간]
  
SSG 발전 (현재)
  → 더 빠르고, 서버 부하 없음

  • 바닐라 쇼핑몰을 ssr,ssg를 적절히 이용해서 변환
  • 왜?
  • 바닐라 쇼핑몰은 뭐로 되어 있는데?

CSR

  • CSR이 뭔데?(클라이언트 서버 렌더링?)

브라우저에서 html을 만들어 보여주는 방식 (브라우저가 그림)

  • 브라우저에서 만들어주는게 뭐가 문제 인데?

초기 로딩이 느리다, seo 문제, 공유시 미리보기 이미지/글 등이 안보임

  • 왜 이게 문제 인데?

로딩이 느리다 = 자바스크립트 다운로드>실행>API호출>렌더링 seo문제 = seo는 html만 읽는다. 현재 div 하나에 스크립트로 모두 집어 넣고 있다. 공유시 문제 = 페이지 설명은 페이지마다 달라야 한다. 자바스크립트로 변경할 수 있지만 소셜미디어는 자바스크립트를 실행하지 않는다.

  • 그래서 CSR말고 SSR 과 SSG를 사용하라고?
  • SSR과 SSG는 뭔데?

SSR : 서버에서 HTML을 만들어서 브라우저에 보내주는 방식 (서버가 그림) SSG : 빌드할 때 미리 HTML을 만들어서 정적 파일로 저장해두는 방식 (빌드할 때 미리 그림)

  • SSR은 어떻게/ 언제 사용해야 하는 건데?

구현해보기

  1. Express 서버 구성 ↓
  2. 요청이 오면 서버에서 데이터 가져오기 ↓
  3. 서버에서 HTML 렌더링 ↓
  4. 완성된 HTML을 클라이언트에 전달 ↓
  5. 클라이언트에서 하이드레이션 (인터랙티브 기능 활성화)
  • express 서버 구성을 왜해야해?

서버에서 html을 만들고 내려주기 위해서

  • 그래서 SSR의 핵심은?
  1. 서버에서 데이터를 먼저 가져옴 → API 호출이 서버에서 일어남
  2. 서버에서 HTML을 만들어서 보냄 → 클라이언트는 이미 완성된 HTML을 받음
  3. 클라이언트는 하이드레이션만 → 인터랙티브 기능 활성화
// CSR: 브라우저에서 데이터 가져오기
useEffect(() => {
  fetchProducts().then(setProducts); // 클라이언트에서 실행
}, []);

// SSR: 서버에서 데이터 가져오기
const products = await fetchProducts(); // 서버에서 실행
  • SSG는 어떻게/ 언제 사용해야 하는 건데?

구현해보기

  1. 빌드 스크립트 실행 ↓
  2. 모든 페이지에 필요한 데이터 가져오기 ↓
  3. 각 페이지마다 HTML 파일 생성 ↓
  4. 생성된 파일들을 디렉토리에 저장 ↓
  5. 배포 시 정적 파일로 서빙
  • 빌드 할 때란?

모든 페이지를 미리 만들어두는 시점

  • SSG 핵심포인트는?
  1. 빌드 타임에 모든 페이지 생성 → 런타임에 렌더링 안 함
  2. 정적 파일로 저장 → 파일 시스템에 HTML 파일이 존재
  3. 서버는 파일만 전달 → 렌더링 작업 없음, 매우 빠름
  • 결론

CSR

  • ✅ 빠른 페이지 전환
  • ❌ 초기 로딩 느림
  • ❌ SEO 안 됨
  • ❌ 소셜 공유 미리보기 안 됨
  • ✅ 서버 부하 낮음

SSR

  • ✅ 초기 로딩 빠름
  • ✅ SEO 완벽
  • ✅ 소셜 공유 미리보기 완벽
  • ❌ 서버 부하 높음
  • ❌ 응답 시간이 느려질 수 있음

SSG

  • ✅ 극도로 빠른 응답
  • ✅ SEO 완벽
  • ✅ 소셜 공유 미리보기 완벽
  • ✅ 서버 부하 제로
  • ❌ 데이터 업데이트 시 재빌드 필요
  • ❌ 동적 콘텐츠에 부적합
