[6팀 전이진] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by im-binary2025.10.20

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

테스트가 병렬로 돌아도 안정적이게 동작하려면 각 테스트마다 독립적으로 상태, 데이터를 가져야 한다고 생각합니다.

그래서 setUp어쩌구 함수들 내부에서 독립적인 event 배열을 생성되도록 해 함수 레벨에서 데이터가 격리되도록 해보았습니다.


export const setupMockHandlerCreation = (initEvents = [] as Event[]) => {
  const events = [...initEvents];

  server.use(
    http.get('/api/events', () => {
      return HttpResponse.json({ events: [...events] });
    }),
   ...
}

export const setupMockHandlerUpdating = () => {
  const mockEvents: Event[] = [
    {
      id: '1',
      title: '팀 회의 1',
      date: '2025-10-20',
      startTime: '10:00',
      endTime: '11:00',
      description: '주간 팀 미팅',
      location: '회의실 A',
      category: '업무',
      repeat: { type: 'none', interval: 0 },
      notificationTime: 1,
    },
  ];
  
  
  server.use(
    http.get('/api/events', () => {
      return HttpResponse.json({ events: mockEvents });
    }),

    http.put('/api/events/:id', async ({ params, request }) => {
      const { id } = params;

      const updateEvent = (await request.json()) as Event;
      const eventIndex = mockEvents.findIndex((event) => event.id === id);

      if (eventIndex > -1) {
        mockEvents[eventIndex] = { ...mockEvents[eventIndex], ...updateEvent };

        return HttpResponse.json(mockEvents[eventIndex]);
      } else {
        return HttpResponse.json({ message: 'Event not found' }, { status: 404 });
      }
    })
  );
};

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

사실 위에서 답변한 정도로 독립적으로 구동되게 만들었다고 하기 살짝 애매하다고 생각했습니다.

테스트 A가 setupMockHandlerCreation() 을 호출하고 테스트 B가 setupMockHandlerUpdating() 호출하게 된다는 상황을 가정해보면 테스트 B의 handler로 A의 handler가 덮씌될 수도 있지 않았을까 싶습니다. 근데 이미 setupTest.ts 파일에 hanlder를 reset하는 설정이 있어 독립적으로 구동시킬 수 있었던 것 같습니다.

afterEach(() => {
  server.resetHandlers();
  ...
});

그리고 시간에 대해 검증하는 테스트들이 꽤 있어서 반복적으로 vi.useFakeTimers(), vi.setSystemTime(date) 한 후 vi.useRealTimers(); 했는데 한 번씩 vi.useRealTimers(); 를 누락해 이전에 설정한 타이머로 인해 테스트가 깨지기도 했습니다. 그래서 setupTest.ts 를 아래처럼 설정했습니다.

beforeEach(() => {
  vi.useFakeTimers({ shouldAdvanceTime: true });
});

afterEach(() => {
  vi.useRealTimers();
});

모든 테스트가 시작하기 전에 가짜 타이머를 활성화하고, 모든 테스트가 끝나고 진짜 타이머로 되돌리도록 했습니다.

과제 셀프회고

기술적 성장

이전 회사에서 테스트 코드를 작성해야만 코드 리뷰 요청을 할 수 있어서 반강제적으로 테스트 코드를 작성하곤 했었는데요. 사실 테스트 코드에 대한 지식도 전무했는데 시간내어 공부할 수 있지도 않았습니다. 부끄럽지만,,, 팀원들이 이미 잘 작성해놓은 테스트 코드를 훔쳐보며 작성하다보니 어렴풋이 느낌으로만 알고 지나쳤던 것들을 되돌아볼 수 있는 과제였습니다.

  • test, it 차이 test와 it는 기능적 차이는 없고 테스트의 가독성을 위해 쓰이는 키워드라는 새로운 사실을 알게 되었답니당

  • act는 언제 써야 하나? 원래는 대충 테스트 코드 돌리는데 깨져서 에러가 나면 메시지 읽고 act로 별 생각없이 감싸곤... 했습니다. 이번에 작성하면서 변경된 상태를 적용할 때 컴포넌트가 리렌더링되는데 이 과정이 다 끝날 때까지 기다리는 함수라는 것을 알게 되었습니다. Common mistakes with React Testing Library: wrapping things in act unnecessarily에 흔한 실수로 기록되어 있어서 과거의 제가 살짝 부끄러워졌답니다

  • toEqual() toEqual 을 내부적으로 어떻게 판단하는지가 궁금했습니다. 왜 굳이 toStrictEqual 도 있을까? 뭘 커버를 못하는 걸까? 궁금해졌습니다. 실제로 구현체를 보기도 했습니다.

    const arr1 = [1, , 3]; 
const arr2 = [1, undefined, 3]; 

expect(arr1).toEqual(arr2); // 통과

    요런 테스트를 돌려보면서 정확한 타입의 검증은 안 하고 살짝 느슨하게 검증하는구나를 알게 되었습니다.

  • screen 사용하기 아래처럼 컴포넌트를 render한 쪽에서 getByTestId, getByText 를 꺼내와서 상황에 따른 렌더링 테스트를 작성한 적이 있었는데요.

    const { getByTestId, getByText } = render(<WeekView currentDate={currentDate} filteredEvents={[]} notifiedEvents={[]} />);

    Common mistakes with React Testing Library: Not using screen 창시자가 screen을 사용하는 것을 권장한다고 하더라구요. 근데 전 회사에 이렇게 쓴 테스트 코드를 몇 개 남기고 온 것 같은데 ... 고쳤겠지..?

코드 품질

제가 이미 만들어진 것에 대한 유지보수를 하는 사람이라고 생각하고 조금 이해하기 어려웠던 테스트 문들을 고쳐보았습니다.

또 몇가지 테스트는 굳이 테스트할 필요가 없겠다 싶어 메모를 해놓았습니다. 테스트가 필요없다고 느낀 기준점은 아래와 같습니다.

  1. 이미 충분히 내부에서 검증된 API를 그대로 쓰고 있다.
  2. 내부 구현이 복잡하지 않고 변경이 잦지 않을 거라 예상된다.
  3. 비슷한 조건, 비슷한 결과 값이 나오는 테스트가 반복적으로 작성되었다.

유의미한지는 모르겠지만... 그냥 이상하게 텍스트 전문(e.g. 일정이 추가되었습니다) 를 같은지 비교해서 테스트를 통과 시키는 게 찝찝하더라구요. 메시지의 내용을 확인한다기보다 '일정 저장을 성공하면 성공 메시지를 보여준다' 가 좀 더 테스트 의도에 맞지 않나 싶어 상수로 빼서 나중에 문구가 변경되더라도 한 곳에서 수정이 가능하도록 해보았습니다.

const toastMessages = {
  이벤트_로딩_실패: '이벤트 로딩 실패',

  일정_저장_성공: '일정이 추가되었습니다.',
  일정_저장_실패: '일정 저장 실패',

  일정_수정_성공: '일정이 수정되었습니다.',

  일정_삭제_성공: '일정이 삭제되었습니다.',
  일정_삭제_실패: '일정 삭제 실패',
};

학습 효과 분석

컴포넌트의 테스트 코드를 작성하는 것에만 익숙했는데 MSW로 서버를 모킹해 사용자 행동에 따른 응답값을 검증하는 테스트를 작성하는 것에도 조금은 익숙해진 것 같습니다. 또 외부 의존성을 격리하여 통합 테스트를 작성하는 경험도 할 수 있어 유익했습니다. 조금은 더 알고 싶은 건 어떻게 MSW가 서버 응답을 가로채는지 궁금했는데 과제 하는 것에 좀 급급해서 알아보지는 못했습니다..ㅎㅎ service worker 로 한다는데 아직 service worker 의 개념도 잘 몰라 조금 더 공부가 필요할 것 같습니다. 또 실무에 테스트 코드를 적용한다면 어느 수준부터 적용해볼 수 있을까 고민해보는 과제가 되었던 것 같습니다.

과제 피드백

리뷰 받고 싶은 내용

테스트를 독립적으로 구동시키기 위해 작성했던 설정들이 있는데 (배열 복사, server.resetHandlers()...) 이렇게만 하면 독립적인 테스트 구동이 되는 건지... 궁금합니다.

궁금한 점

의미있는 테스트 코드를 작성하기 위해 의미없는 테스트 코드의 기준을 먼저 세워봤는데, 잘못된 기준이 있거나 여기서 더 기준으로 가져가볼 수 있는 게 있을까요?

  1. 이미 충분히 내부에서 검증된 API를 그대로 쓰고 있다.
  2. 내부 구현이 복잡하지 않고 변경이 잦지 않을 거라 예상된다.
  3. 비슷한 조건, 비슷한 결과 값이 나오는 테스트가 반복적으로 작성되었다.

"이전 회사에서 테스트 코드를 작성해야만 코드 리뷰 요청을 할 수 있어서 반강제적으로 테스트 코드를 작성하곤 했었는데요. 사실 테스트 코드에 대한 지식도 전무했는데 시간내어 공부할 수 있지도 않았습니다. 부끄럽지만,,, 팀원들이 이미 잘 작성해놓은 테스트 코드를 훔쳐보며 작성하다보니 어렴풋이 느낌으로만 알고 지나쳤던 것들을 되돌아볼 수 있는 과제였습니다." 라는 회고가 제일 눈에 들어오네요. 사실 실무에서 테스트 경험이 없는 친구들을 중심으로 만든 과제이지만 테스트에 대해서 생각이 더 깊이있게 해볼 수 있게 된 계기가 된것 같아 좋습니다.

사실 많은 개발자들이 실무에서는 정작 마감에 쫓겨서 제대로 이해할 시간 없이 일단 돌아가게만 만들고, 왜 그렇게 하는지는 깊이 고민하지 못하는 경우가 많죠. 이번 과제가 실무에서 접하는 것들을 어떻게 하면 내 것으로 만들 수 있는지를 고민해보는 시간이 되었기를 바랍니다. 앞으로도 남은 9개의 과제들을 통해서도 실무와 연관 지어서 어떻게 확장할 수 있을지 고민해보면 좋겠습니다.

"대충 시키는대로 하면 되지 않나?~!"에서 시작했는데 몰입하면서 "한 번 의문을 가지니까 다른 것들에 다 의문을 가지게 되고"로의 변화가 정말 멋지네요. 실무에서 반강제로 했던 경험이 있었기에 오히려 이번에 더 깊이 파고들 수 있었을 거라 생각하며 실무 경험을 되돌아보고 제대로 이해하는 시간을 가지는 방법을 이번 전체 커리큘럼에서 가져가게 되길 바래요. 다음 과제도 화이팅입니다!
