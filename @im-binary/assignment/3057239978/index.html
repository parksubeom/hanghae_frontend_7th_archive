과제 셀프회고

과제를 하면서 내가 알게된 점, 좋았던 점은 무엇인가요?

Q&A 시간에 테오가 했던 말이 기억에 남았어요. "추상화 계층을 나눈다는 게 더 나아지거나 편리해진다는 개념이 아니다. 계층을 나눈다는 것은 더 좋게 만드려고 하는 게 아니라 격리가 목적이다" 저는 그동안 코드를 읽기 쉽게 만드는 것 정도만 신경 썼던 것 같은데 이 말을 듣고 코드 구조를 바라보는 관점이 조금 넓어진 것 같아요. 평소 저는 어떤 기능을 만들 때 한 바닥에서 전체 흐름이 보이도록 작성하려고 하는 편이었는데 변경의 영향 범위를 줄이기 위해서는 오히려 한바닥에 모아두지 않는 선택도 필요하겠다는 생각도 들었어요. 실제로 models 계층, hooks 계층, ui 계층으로 분리해보면서 비지니스 규칙이 변경되어도 UI에는 변경이 없게, 상태 관리 방식이 바뀌어도 컴포넌트에 영향이 없게, 디자인이 변경이 되어도 로직에는 영향이 없게 작성해보면서 많이 배운 것 같아요.

이번 과제에서 내가 제일 신경 쓴 부분은 무엇인가요?

한 파일에 있었던 함수와 컴포넌트들을 이번에 나누면서 기준을 한 번 세워보았습니다.

(아직 확신은 못하는) 기준:

  1. 재사용되지 않으면 분리하지 않는다.
  2. 한 함수/컴포넌트는 하나의 역할만 가진다.
  3. 테스트하기 어려우면 분리한다.
  4. 100줄 넘으면 분리를 고려한다.

원래는 App 전체를 모든 context가 감싸고 있는 형태였는데, 페이지별로 필요한 Provider만 제공하도록 리팩토링 했습니다.(https://github.com/hanghae-plus/front_7th_chapter3-2/pull/15/commits/77ca4367755ab63441df0870910553e699a7e0dc)


// before
<Provider>  // 모든 Context
  <App />
</Provider>

// after
<NotificationsProvider>
  {isAdmin ? (
    <AdminPageProvider>  // -> Products + Coupons
      <AdminPage />
    </AdminPageProvider>
  ) : (
    <CartPageProvider>  // -> Products + Cart + Coupons
      <CartPage />
    </CartPageProvider>
  )}
</NotificationsProvider>

AdminPage 에서 Cart 관련 context를 사용하지 않고, 각 페이지에서 의존하는 것들이 명확해졌다고 생각했는데 페이지 전환 시 상태가 초기화 되고 구조가 조금 더 복잡해진 것 같은 느낌을 받았습니다. 제가 진행한 리팩토링이 의미가 있는... 리팩토링이었을까요?

또 굳이 관리자가 아니면 페이지 전환이 일어나지 않을 것 같아 상태 초기화 되는 건 괜찮다고 판단했는데, 만약 상태 유지가 더 중요한 경우라면 어떻게 개선하는 게 좋을까요?

더 나아가 context를 읽기/쓰기로 분리하기도 하는 것 같던데 그렇게 세분화하는 것에 대해 어떻게 생각하시는지도 궁금합니다.
