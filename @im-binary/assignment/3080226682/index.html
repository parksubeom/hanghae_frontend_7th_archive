과제 셀프회고

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

FSD에 대해 잘 모를 때 오히려 복잡도를 높이는 것 아닐까, 아무리 싱크를 맞춘다고 한들 팀원들간에 각 폴더에 대한 기준이 다르면 오히려 관리 포인트가 더 생기는 것이 아닐까란 걱정으로 이번 과제를 시작하게 되었는데요, 그런데 이번에 과제를 하면서 복잡도를 높이기 보다 이미 존재하는 복잡도를 관리하는 방법이 아닐까란 생각으로 변하게 된 것 같습니다.

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

TanStack Query를 적용하면 Context API 를 적용했던 부분을 제거했습니다. 처음에는 Context로 게시물 목록, 사용자 정보, 댓글과 같이 데이터를 전역 상태처럼 사용했는데 TanStack Query 적용하면서 굳이 그럴 필요가 없다는 생각이 들더라구요. 서버 상태는 TanStack Query 가 관리하고 , 그에 따른 loading, error도 마찬가지였기 때문에.. 그리고 현재 구조에서 굳이 클라이언트 상태중에 전역적으로 관리해야 한다고 생각이 드는 상태들도 없어서 제거했습니다. (https://github.com/hanghae-plus/front_7th_chapter3-3/pull/17/commits/75f1ec0407db12705124e98874f153dae42d4790)

이번에 과제를 하면서 FSD 설계하려고 노력했던 것 같은데... 한 세 번 정도 갈아엎은 것 같습니다.

처음에는 기존 코드를 그냥 분리(?) 하는 정도로만 그쳤었는데 그러니까 파일만 분산된 상태였습니다.

두 번째 시도에서는 첫 시도를 바탕으로 하다보니 너무나도 잘게 쪼개게 되었습니다. 버튼이나 입력 필드 하나하나 다 쪼갰던 것 같습니다. 그러니까 폴더가 너무 많고... 기능 수정하려면 여기저기 왔다갔다 해야 하는 건 동일했습니다.

세 번째 시도에서는 일단 나만의 명확한 기준을 세워보자, 생각하며 진행했습니다.

entities는 도메인의 핵심만, features는 하나의 사용자 행동으로, widgets는 비즈니스 로직이 있는 조합인 것들로, shared는 진짜 범용적인 것만 이런 식으로 기준을 세우고 나누다보니 제가 가장 이해하기 쉬운 수준으로 폴더링이 된 것 같았습니다.

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

"얼마나 잘게 쪼개야 하는가?"의 정답을 아직 모르겠습니다.

usePostsManagerViewModel이라는 커스텀 훅 하나에 검색, 태그 필터링, 정렬 로직이 모두 들어있습니다. 얘를 usePostsSearch, usePostsFilter, usePostsSort로 나눠야 할까요? 아니면 지금처럼 하나의 ViewModel로 관리하는 것이 맞을까요?

'기능이 독립적으로 재사용 가능하면 분리하라' 고 하는데... 실무에 적용해보자면, 뭔가 얘는 나중에 쓰일 것 같다? 싶어서 쪼개놓으면 안 쓰게 되기도 하고 얘는 안 쓰겠지 싶어서 안 쪼개면 꼭 나중에 쓰이게 되는 일이 발생하더라구요. 그냥 언제든 쪼개기 쉽게 해야 하나... 그냥 나중에 필요하게 되면 그때 진행하는 방식으로 지금까지 해온 것 같은데 어떤 기준으로 쪼개야 하는지 조금 막연해지는 것 같습니다.

저만의 기준을 세우긴 했지만 widgets와 features의 경계도 여전히 애매하게 느껴집니다.

PostDetailModal은 widgets에 있지만, 사실 이것도 하나의 "게시물 상세 보기" 기능입니다. feature로 봐야 하나요? 제 기준은 "여러 entities와 features를 조합하고, 자체적인 상태가 있으면 widget"이었지만, 정말 맞는 기준인지는 확신이 서지 않네용,,

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

실무에서도 FSD를 조금은 기본적인 전략으로 사용하고 싶다는 생각이 들었습니다.

사실 생각해보면 지금도 거의 파일명만 좀 다를 뿐이지 이런 구조로 사용하고 있다고 생각이 드는데 구체적인 규칙과 기준이 없다보니 자주 '얘는 어디에 두면 좋을까요' 를 이야기하게 되는 것 같습니다. 이번 내용을 토대로 폴더명에서 기대하게 되는 것들에 대해 이야기하고 우리만의 기준을 세워 적용해보면 좋을 것 같습니다.

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

  • 더티코드를 접했을 때 어떤 기분이었나요? ^^; 클린코드의 중요성, 읽기 좋은 코드란 무엇인지, 유지보수하기 쉬운 코드란 무엇인지에 대한 생각을 공유해주세요

평소 저는 코드를 작성할 때 한 바닥에서 전체 흐름이 보이도록 작성하려고 하는 편이었습니다. 기존 코드가 어떻게 돌아가는 지 한눈에 파악하라 수 있어야 수정해야 할 부분과 문제가 되는 부분을 빨리 찾을 수 있을 거라고 생각했기 때문인데요. 그런데 오히려 이렇게 작성하는 게 유지보수하기 어려웠을 수도 있었겠다, 라는 생각이 들었습니다.

원래 기존에 있던 부분에서 어느 한 부분이 사라진다고 가정해보면 위 아래로 스크롤해가며 관련 코드를 찾아야 할 테고, 미처 지우지 못한 부분 때문에 예상하지 못했던 에러를 발생시킬 수도 있었겠다 싶었습니다.

이번 챕터를 지나면서 읽기 좋은 코드는 곧 의도가 명확한 코드고, 유지보수하기 쉬운 코드란 변경 영향 범위가 적은 코드라는 결론을 내리게 된 것 같습니다.

응집도 높이기: 서버상태관리, 폴더 구조

  • "이 코드는 대체 어디에 둬야 하지?"라고 고민했던 시간, FSD를 적용해보면서의 느낌, 나만의 구조를 만들어가는 과정, TanStack Query로 서버 상태를 분리하면서 느낀 해방감(?)등을 공유해주세요

코드를 어디에 둘 지 고민했던 시간도 꽤 있었습니다. Post 타입을 정의하려면 models에 해야 하는지 entities 에 포함시켜야 하는지, 검색 기능도 독립된 features로 만들어야 하는지, 그냥 페이지 안에 두어도 되는지.. 기타 등등의 고민이 생겼습니다. 그러다보니 이 폴더 구조가 진짜 맞나? 오히려 복잡해진 것 같다, 폴더가 이렇게 많이 생기는데 다 관리가 되나 싶기도 했습니다.

그런데 어느정도 윤곽이 잡힌 이후에는 오히려 어떻게 분리해야 하나 싶었던 막막함이 조금은 사라졌습니다.

  1. 도메인 중심이라면 -> entities
  2. 사용자 행동이라면 -> features
  3. 어디에서나 쓰이면 -> shared
  4. 한 페이지에서만 쓰이면 그냥 pages에 두기
  5. 여러 entities, features 조합하여 사용하고 자체적인 UI 이면 -> widgets

사실 이 구분이 맞는지는,,, 확실하지는 않지만 어느정도 이렇게 기준을 잡아두니까 확실히 어떤 기능을 확인하고 싶다, 어떤 UI를 확인하고 싶다 할 때 금방금방 찾을 수 있어 좋았습니다.

그래도 한 번씩 이건 어디에 두지 ? 싶은 순간도 오긴 하는데... 게시물 검색 기능은 순수 함수로 만들면 shared 에 두어야 하는지 API 호출이 생기면 entities 에 두어야 하는 건지 hook으로 만들게 되면 features 로 빼야 하는지... 뭐 이 페이지에서만 쓰는데 pages에 두어야 하는 지 등등.. 이런 생각이 드는 건 그냥 pages에 두고 나중에 여러 곳에서 쓰면 features 로 옮기든지 하자 ~ 라는 선택을 했습니다. 너무 구조에 매몰되면 오히려 복잡할 수도 있을 것 같아서요,,ㅎ... 옳은 선택이었는지는 모르겠습니다. 그래도 구조를 파악하기에는 이정도도 나쁘지 않은 것 같았습니다.

TanStack Query로 서버 상태를 분리하면서 느낀 해방감은 서버 상태와 클라이언트 상태를 구분했다는 지점에서 온 것 같습니다. 도입 전에는 아래처럼 상태 관리를 해왔습니다.

const [posts, setPosts] = useState([]); 
const [loading, setLoading] = useState(false);
const [error, setError] = useState();

도입한 후는 아래처럼 로딩, 에러, 캐싱, 리프레시 모두 처리가 되어 좋았습니다.

const { data: posts, isLoading, error } = usePosts();

그리고 낙관적 업데이트를 손쉽게 구현할 수 있다는 것도 좋았습니다.

원래라면 (1) 현재 데이터를 백업하는 상태를 만들고, (2) 화면에 보이는 데이터 상태를 업데이트하고, (3) 서버에 요청을 보낸 후 (4) 성공하면 그대로 유지하고 실패하면 백업해 둔 데이터로 복원하는 방식으로 작업을 했습니다.

이 과정에서 원본 배열, 백업 배열, 로딩 상태, 에러 상태를 각각 관리해야 하고, 특히 사용자가 연속으로 여러 작업을 수행할 때 동시성 이슈까지 고려하면 코드가 상당히 복잡해집니다.

하지만 TanStack Query는 onMutate에서 백업한 데이터를 context로 자동 전달하고, onError에서 자동으로 롤백해주기 때문에 별도의 상태 관리 없이도 안전하게 낙관적 업데이트를 구현할 수 있었습니다. 특히 cancelQueries로 진행 중인 쿼리를 취소하는 등 동시성 이슈도 표준 패턴으로 해결할 수 있다는 점이 좋았습니다.

    • 팀원들을 어떻게 설득하고 온보딩하는 게 좋을까요?

과제 피드백

고생하셨습니다 이진님 ㅎㅎ 시간이 짧았음에도 굉장히 여러번 많은 시도를 하셨네요 :+1

"얼마나 잘게 쪼개야 하는가?"의 정답을 아직 모르겠습니다.

결국 이 부분은 저번주에 공부하셨던 단일 책임과 어느정도의 파일 추상화를 가질것인가에 이어지는 내용인 것 같은데요. 동일하게 설명을 해주셨을 것 같지만, 변경의 빈도와 범위가 비슷한 코드들을 함께 모으고 관리하는 지점을 찾아내는게 핵심인 것 같아요. 저도 늘 어려운데요. 우선은 처음부터 너무 잘게 쪼개면 과한 추상화가 되니 분류를 적절하게 하고 점진적으로 개선해나가는게 좋다고 생각해요.

저만의 기준을 세우긴 했지만 widgets와 features의 경계도 여전히 애매하게 느껴집니다.

위젯이 항상 이런 부분이 존재하는데요. 위젯은 명시적으로 언제 사용한다고 개념을 잡기 보다는 명시적인 케이스들은 분리하고 피처와 피처를 조합해 사용하거나..레이어의 규칙을 위배되지 않는 선에서 페이지나 앱으로 가기 어려운 컴포넌트를 명확하게 관리하는 영역으로 사용을 저는 자주 했던것 같아요. 적어도 위젯에 대해서는 조금 나이브하게 접근해도 괜찮지 않을까..!

현재 FSD 구조에서 개선할 부분이 있을까요?

지금은 대부분 규칙을 잘 지켜주신 것 같아요. 크게 무리되는 부분도 없고 의존을 역전하는 부분도 제가 보기에는 못찾은 것 같아요 :+1 타입같은 곳에서 가끔 엔티티에서 엔티티로 접근하는게 있는것 같은데 이 부분에 대한 고민을 좀 더 해보셔도 좋을것 같구요!

말씀해주신것처럼 PostsManagerViewModel에서는 조금 더 나눌만한 이야기들이 있는것 같아요. 지금 당장은 해당 코드를 여러 군데에서 사용하지 않아서 그대로 운영을 해도 무리는 없지만 주석으로 남겨주신 기준에 맞춰서 변경 빈도에 맞춰서 분리를 하면 좋지 않을까 싶습니다.

실무에서 FSD를 도입할 때 가장 주의해야 할 점은 무엇인가요?

말씀해주신것처럼 상호간에 모두 공감할 수 있는 규칙을 만들고, 이 규칙을 매번 이야기 하지 않도록 명확한 문서화를 하는게 중요하다고 생각해요 ㅎㅎ 리뷰단계에서도 정적 분석을 통해 검증을 명확하고 쉽게 할 수 있도록 하구요. 그리고 운영하다 보면 점점 더 복잡한 케이스들이 있는데 그럴때 열린 토론을 가능하게 하는것들! 그런것들이 가장 중요한것 같아요.
