[2팀 양진성] Chapter 1-1. 프런트엔드 테스트 코드 익숙해지기

by jinseoIT2025.10.19

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

병렬 테스트 환경에서는 동일한 mock 데이터를 공유할 경우, 테스트 간 상태 변경이 서로 영향을 주어 예측 불가능한 결과가 발생할 수 있습니다.

이를 방지하기 위해 각 테스트 실행 시마다 새로운 스코프와 배열을 생성하도록 구성하여, mock 데이터가 테스트 단위로 완전히 격리되도록 설계합니다.

이 방식은 테스트 간 데이터 충돌을 제거하고, fetch 기반 상태 갱신 로직이 항상 초기 상태에서 일관되게 동작하도록 보장합니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

1. 테스트마다 독립적인 events 생성

병렬 테스트 간 데이터 간섭을 막기 위해, 함수 호출 시마다 새로운 스코프와 배열을 생성해 테스트 간 데이터가 격리되도록 구성

export const setupMockHandlerCreation = (initEvents = [] as Event[]) => {
  const events = [...initEvents];
  ...
}

2. setupTests 설정

afterEach(() => {
  server.resetHandlers();
  vi.clearAllMocks();
});
  • server.resetHandlers() : MSW의 핸들러 목록을 초기화하여 it() 내에서 server.use()로 임시로 등록한 mock 핸들러가 다음 테스트에 누적되지 않도록 막습니다.
  • vi.clearAllMocks(): vi.fn()으로 만든 mock 함수들의 호출 기록(.mock.calls)만 초기화하여 같은 mock 함수가 여러 테스트에서 호출되어도 이전 테스트의 호출 이력이 남지 않습니다.

과제 셀프회고

기술적 성장

이번 과제를 통해 MSW와 React Testing Library(RTL) 를 활용한 테스트 작성에 처음 도전했습니다. 초반에는 서버 모킹과 비동기 흐름 제어가 다소 생소했지만, 구조를 이해하고 나니 단위 테스트를 넘어 사용자 관점에서의 통합 테스트까지 확장할 수 있는 가능성을 체감했습니다.

특히, 새롭게 학습한 개념으로는 MSW의 요청 핸들러 구조, 테스트 격리를 위한 서버 상태 관리, 그리고 mock 서버와 실제 fetch 간의 연동 흐름을 들 수 있습니다.

이를 통해 기존에 알고 있던 테스트는 코드 검증의 수단이라는 인식이 확장되어, 테스트가 곧 기능 명세이자 문서가 될 수 있다는 점을 실감했습니다.

코드 품질

타이머 로직은 다루기 어렵게 느껴졌지만, 제공되는 유틸리티 함수를 활용해 구현할 수 있었습니다. 이러한 유틸을 활용하면, 통합 테스트에서도 사용자 시나리오에 더 가깝게 동작을 검증할 수 있을 것이라 생각했습니다.

학습 효과 분석

단위 테스트를 작성하면서, 단일 책임 원칙을 지키지 않은 함수는 테스트가 복잡해진다는 점을 크게 체감했습니다. 이를 계기로 앞으로 코드를 작성할 때는 종속성을 최소화하고 책임 단위로 나눈 함수 구조를 의식하게 될 것 같습니다.

과제 피드백

중간 테스트 설명이 이해하기 어려운 부분이 있어 파악하는데 시간이 좀 걸린 것 같습니다.

리뷰 받고 싶은 내용

아래 궁금한 점과 같은 내용입니다. medium.interation.spec.tsx(통합 테스트) 파일에서는 통합 테스트 전용 mock 데이터로 events.integration.json을 import하여 동일한 데이터를 기반으로 테스트를 진행하고 있습니다. 이와 관련해, 통합 테스트나 E2E 테스트에서도 하나의 공통 mock 데이터를 종속적으로 사용하는 것이 바람직한지 궁금합니다.

궁금한 점

단위 테스트의 경우, 테스트 간 데이터 종속성을 없애기 위해 각 테스트 컨텍스트 안에서 독립적인 mock 데이터를 생성하여 일관된 테스트 결과를 보장하는 것이 맞다고 생각합니다.

그런데 통합 테스트의 경우는 상황이 조금 다르다고 느꼈습니다. 여러 모듈이 실제처럼 상호작용해야 하므로, 테스트 전체가 하나의 큰 mock 데이터(가상 DB 개념) 를 기반으로 상태를 공유하는 편이 더 현실적인 시나리오 검증에 가깝지 않을까 생각했습니다.

즉,

단위 테스트: 독립성 중심 (mock 데이터 각자 정의)

통합 테스트: 현실성 중심 (공통 mock 데이터 기반)

이런 식으로 관리하는 접근이 적절할지 궁금합니다. 멘토님께서는 통합 테스트에서 mock 데이터를 어떻게 관리하시는지, 또 테스트 간 상태 공유를 어느 정도까지 허용하는 편이신지도 듣고 싶습니다.

과제를 수행하면서 느낀점

리팩토링이나 마이그레이션 시, 기존 테스트 코드가 있었다면 시간과 비용을 크게 절약할 수 있었을 것이라는 점을 느꼈습니다.

과제 시작 전 생각

그동안 테스트 코드를 제대로 작성해본 경험이 없어, 처음에는 막연한 불안감을 느꼈습니다.

과제 제출 후 생각

작성 과정을 반복하다 보니 점차 적응되었고, 앞으로 함수나 훅을 만들때 테스트 코드하기 쉬운 로직으로 작성할 것 같습니다. 그리고 직장다니시면 과제하시는분들을 존경하게 되었습니다.

단위 테스트도 공통 mock 데이터 기반으로 관리하면 어떨까요? 물론 각각의 상황에 필요한 데이터는 직접 만들어서 사용해야할 것 같은데, 이 또한 공용 데이터를 랩핑하거나 오버라이딩해서 사용하면 편하답니다 ㅎㅎ

그리고 저희 조직에서는 테스트간 상태 공유는 허용하지 않는 편이랍니다. 대신 테스트 공용 데이터를 항상 복제하여 사용하는거죠.
