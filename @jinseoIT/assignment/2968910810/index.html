과제 셀프회고

이번 과제에서는 Playwright를 활용한 E2E(End-to-End) 테스트를 직접 수행하면서, 프론트엔드 테스트 전략과 테스트 환경 구성에 대해 깊이 고민해볼 수 있었습니다.

처음에는 어떤 테스트 전략을 프로젝트에 적용해야 할지에 대한 고민이 컸습니다. 전통적으로는 테스트 피라미드(Test Pyramid) 전략이 널리 알려져 있지만, 프론트엔드 환경에서는 E2E 테스트의 비중이 상대적으로 높고, 단위 테스트(Unit Test)의 한계가 분명합니다.

이에 따라, 기능별로 어떤 테스트를 선택해야 효율적일지를 고민하며, 다음과 같은 의사결정 기준 도식도를 정리했습니다.

새 기능을 테스트해야 한다면?
    ↓
┌───────────────────────────┐
│ 순수 함수인가?             │ YES → 단위 테스트
│ (입력 → 출력, 부작용 없음) │
└───────────────────────────┘
    ↓ NO
┌───────────────────────────┐
│ 여러 컴포넌트/Hook이       │ YES → 통합 테스트
│ 데이터를 주고받는가?       │
└───────────────────────────┘
    ↓ NO
┌───────────────────────────┐
│ 브라우저 API가 필요한가?   │ YES → E2E 테스트
│ (D&D, 파일 업로드 등)      │
└───────────────────────────┘
    ↓ NO
┌───────────────────────────┐
│ 핵심 사용자 플로우인가?    │ YES → E2E 테스트
│ (일정 생성, 로그인 등)     │
└───────────────────────────┘
    ↓ NO
┌───────────────────────────┐
│ 시각적 확인이 중요한가?    │ YES → 시각적 회귀 테스트
│ (레이아웃, 색상, 간격)     │
└───────────────────────────┘
    ↓ NO
통합 테스트로 시작 → 필요하면 단위로 분리

이 기준은 “테스트의 목적을 명확히 구분하고, 복잡도와 비용을 고려한 선택을 하자”는 방향으로 설정했습니다.

그 과정에서 이 아티클 을 참고하며, 트로피형(Test Trophy) 테스트 전략이 프론트엔드 프로젝트에 보다 적합하다는 결론을 내렸습니다.

트로피 전략은 통합 테스트(Integration Test)와 E2E 테스트(E2E Test)의 비중을 높게 두고, “실제 사용자 시나리오 중심으로 품질을 보장한다”는 점에서 피라미드 전략보다 프론트엔드 개발 현실에 부합했습니다.

또한, 팀원들과 논의한 결과 역시 트로피 전략의 방향성과 유사했습니다. 2팀 논의기록 에서도 “사용자 경험을 보장하는 테스트에 집중하고, 불필요한 단위 테스트는 최소화하자”는 합의에 도달했습니다.

기술적 성장

1. playwright를 활용한 e2e 테스트

  • 사용자 인터랙션(dragTo) 을 포함한 실제 유저 시나리오를 E2E 수준에서 자동화할 수 있었던 점이 큰 경험이었습니다.
  • Playwright의 UI 디버깅 기능을 활용하면 테스트 중 원하는 시점에서 정확히 어떤 상태인지 시각적으로 확인할 수 있었고, 이를 통해 버그를 빠르게 탐지할 수 있었습니다.
  • 또한, codegen 기능을 사용하여 테스트 작성 시 locator를 자동으로 추출할 수 있다는 점을 배워, 테스트 작성 속도와 정확도를 높일 수 있었습니다.

2. stroybook + chromatic

  • storybook을 활용하여 각 컴포넌트를 독립적으로 시각적 테스트할 수 있도록 구성했습니다.
  • Chromatic과 연동함으로써 **시각적 회귀 테스트(Visual Regression Test)**를 수행할 수 있었고, UI가 자주 변경되는 환경에서 디자인 깨짐이나 예상치 못한 스타일 변경을 조기에 감지할 수 있었습니다. 이 접근은 실무적으로 UI 변경이 빈번한 환경에서 리소스 낭비를 줄이고, QA 효율을 높이는 방법이라는 점을 체감했습니다.

3. CI를 활용한 E2E 테스트 및 시각적 회귀 테스트

  • GitHub Actions를 통해 Playwright 기반의 E2E 테스트를 CI 파이프라인에 통합하였습니다. 로컬 환경과 달리 GitHub Action은 기본적으로 2코어 환경에서 실행되므로, 병렬 테스트의 한계와 배포 속도 저하 문제가 발생할 수 있었습니다. 이를 통해 “모든 테스트를 자동화하는 것보다, 필요한 E2E만 적재적소에 배치해야 한다”는 교훈을 얻었습니다.
  • 시각적 회귀 테스트(Chromatic) 도 배포 파이프라인에 포함시켰습니다.배포시 chromatic사이트에 업로드가 되고 Accept을 통하여 변경된 점을 승인할 수 있었습니다. 시각적 회귀 테스트를 자동화함으로써, UI 변경 시 발생할 수 있는 수동 검증 비용을 크게 줄이며, QA 과정에서의 인적 리소스 낭비를 최소화할 수 있겠다는 생각이 들었습다.

과제 제출 상황상 Git Secret Key를 임의로 추가할 수 없어 CI환경을 제 repository main에 PR시 동작시키도록 workflows를 추가하였습니다.

코드 품질

DnD 기능을 구현하기 위해 별도의 useDnd 훅을 작성하여 역할과 책임을 명확히 분리하였습니다. 이 훅은 드래그 시작, 이동, 취소, 종료 등 각 단계의 이벤트를 독립적으로 처리하며, 내부적으로 낙관적 업데이트(Optimistic Update) 를 적용하여 사용자 인터랙션에 즉각적인 피드백을 제공하도록 설계하였습니다.

이러한 구조를 통해 비즈니스 로직과 UI 로직을 분리하고, 상태 관리의 복잡도를 낮추며, 코드의 재사용성과 유지보수성을 향상시켰다고 판단하였습니다.

// useDnd
export function useDnd({ events, saveEvent, setEvents, onOverlap }: UseDndProps) {
  // 👉 state 내부로 이동
  const [activeEvent, setActiveEvent] = useState<Event | null>(null);
  const [overId, setOverId] = useState<string | null>(null);

  const handleDragStart = (event: DragStartEvent) => {
    const draggedEvent = events.find((e) => e.id === String(event.active.id));
    // 반복 일정은 드래그 불가
    if (draggedEvent && draggedEvent.repeat.type !== 'none') {
      return;
    }
    setActiveEvent(draggedEvent || null);
  };
  const handleDragOver = (event: DragOverEvent) => {
    setOverId(event.over ? String(event.over.id) : null);
  };

  const handleDragCancel = () => {
    setActiveEvent(null);
    setOverId(null);
  };

  const handleDragEnd = async (event: DragEndEvent) => {
    const { active, over } = event;

    if (!over || !String(active.id) || !activeEvent) {
      setActiveEvent(null);
      setOverId(null);
      return;
    }

    const newDate = String(over.id).split('T')[0];

    if (activeEvent.date !== newDate) {
      const updatedEvent = {
        ...activeEvent,
        date: newDate,
      };

      // 겹침 체크
      const overlapping = findOverlappingEvents(updatedEvent, events);
      if (overlapping.length > 0 && onOverlap) {
        onOverlap(overlapping, updatedEvent);
        setActiveEvent(null);
        setOverId(null);
        return;
      }
      // 즉시 상태 초기화 (Optimistic Update)
      setEvents((prev) => prev.map((d) => (d.id === activeEvent.id ? { ...d, date: newDate } : d)));

      // 일정 업데이트
      await saveEvent(updatedEvent, true);
    }
    setActiveEvent(null);
    setOverId(null);
  };

  /* dnd */
  const sensors = useSensors(
    useSensor(MouseSensor),
    useSensor(TouchSensor),
    useSensor(KeyboardSensor, {
      coordinateGetter: sortableKeyboardCoordinates,
    })
  );

  return {
    sensors,
    activeEvent,
    overId,
    handleDragStart,
    handleDragOver,
    handleDragCancel,
    handleDragEnd,
  };
}

학습 효과 분석

1. dnd 구현을 위한 라이브러리 선택 및 테스트 코드 위임

일정 관리 기능 내에서 드래그 앤 드롭(DnD)을 구현하기 위해, 외부 라이브러리를 활용하는 방향으로 결정하였습니다. 이를 위해 react-dnd, dnd-kit, react-draggable 세 가지 라이브러리를 후보로 검토하였고, 각 라이브러리의 기능 범위와 테스트 코드 품질을 비교 분석하였습니다.

처음에는 react-draggable을 사용하여 구현을 진행했으나, 라이브러리의 코어 및 테스트 코드를 검토한 결과 테스트 범위가 매우 제한적이고 모듈화 수준이 낮아 유지보수 측면에서 안정성이 부족하다고 판단하였습니다.

이에 따라, 필요한 DnD 기능을 충분히 지원하면서 테스트 코드가 모듈 단위로 잘 작성되어 있는 dnd-kit을 최종적으로 선택하였습니다. 이로써 핵심 기능을 직접 검증하는 부담을 줄이고, 라이브러리 수준에서의 안정적인 테스트 커버리지를 확보할 수 있었습니다.

2. 테스트 격리 및 동시성 문제

E2E 테스트를 진행하면서 **테스트 격리(Isolation)**와 병렬 실행(Concurrency) 문제를 직접 마주했습니다. Playwright의 기본 설정에서는 테스트가 병렬로 실행되기 때문에, 공유 데이터 파일(realEvents.json)을 동시에 접근할 경우 충돌이 발생했습니다.

이를 해결하기 위해 다음과 같은 접근을 시도했습니다:

// 테스트 환경별 데이터 파일 분리
const e2ePath = join(__dirname, '../../src/__mocks__/response/e2e.json');

test.beforeEach(async ({ page }) => {
  // 각 테스트마다 독립적인 데이터 파일 초기화
  fs.writeFileSync(e2ePath, JSON.stringify({ events: [] }, null, 2));
  await page.goto('/');
});

  • TEST_ENV=e2e 환경 변수를 설정해 테스트 전용 데이터 파일(e2e.json)을 사용하도록 분리

  • beforeEach 훅에서 매 테스트 시작 전 데이터를 초기화해 테스트 간 상태 격리 보장

  • test.describe.serial 옵션을 활용해 순차 실행(serial execution) 방식으로 안정성 확보

이를 통해 테스트 간 데이터 간섭 없이 안정적인 E2E 테스트 수행이 가능해졌습니다.

2-1 확장 시 고려사항

현재는 테스트 파일 수가 많지 않아 순차 실행으로도 충분했지만, 만약 E2E 테스트가 50개 이상으로 확장된다면, 테스트 시간을 단축하기 위해 병렬 실행 환경에서의 데이터 격리 전략이 필요할 것입니다.

예를 들어, 테스트 컨텍스트별로 별도의 데이터베이스 파일(db_{workerId}.json)을 생성하고, 실행 시점에 worker 단위로 독립된 환경을 구성하여 동시성 문제를 해결할 수 있을 것입니다.

3. 시간 기반 기능 테스트: 시간 모킹

Playwright Clock API를 활용한 시간 제어

알림 시스템 테스트에서 실제 시간 대기 없이 시간 기반 기능을 검증했습니다.

test.beforeEach(async ({ page }) => {
  await page.clock.install();
  const baseTime = Date.now();
  await page.clock.setFixedTime(baseTime);
  // ...
});

test('알림 표시', async ({ page }) => {
  // 일정을 현재 시간 + 5분으로 설정
  const startTime = new Date(baseTime + 5 * 60 * 1000);
  
  // 시간을 4분 후로 이동 (알림이 표시되어야 하는 시점)
  await page.clock.setFixedTime(baseTime + 4 * 60 * 1000);
  
  // 알림이 즉시 표시되는지 확인
  await expect(page.getByText(/알림/)).toBeVisible();
});
  • page.clock.install(): 브라우저의 Date, setTimeout, setInterval 모킹
  • page.clock.setFixedTime(): 고정 시간 설정
  • 실제 대기 없이 시간 기반 기능 검증 가능

UTC vs 로컬 시간 처리 toISOString() 사용 시 UTC 변환으로 날짜가 하루 전으로 표시되는 문제를 해결했습니다.

// 문제: toISOString()은 UTC를 사용하여 로컬 시간과 차이 발생
const dateStr = startTime.toISOString().split('T')[0]; // ❌ 잘못된 날짜

// 해결: 로컬 시간 기준으로 날짜 추출
const year = startTime.getFullYear();
const month = (startTime.getMonth() + 1).toString().padStart(2, '0');
const day = startTime.getDate().toString().padStart(2, '0');
const dateStr = `${year}-${month}-${day}`; // ✅ 올바른 날짜

타임존 처리의 중요성을 이해하고, 로컬 시간 기준 처리를 적용했습니다.

4. 동적 날짜 처리 및 테스트 유지보수성 향상

하드코딩된 날짜 제거

테스트의 하드코딩된 날짜('2025-11-15')를 동적 생성으로 전환했습니다.

// Before: 하드코딩된 날짜
await page.getByRole('textbox', { name: '날짜' }).fill('2025-11-15');

// After: 현재 날짜 기반 동적 생성
const startDate = getCurrentDate(15); // 현재 달의 15일
const futureDate = getFutureDate(15); // 다음 달의 15일
await page.getByRole('textbox', { name: '날짜' }).fill(startDate);
  • getCurrentDate(day): 현재 달의 지정 일자 반환
  • getFutureDate(day, monthsLater): 미래 날짜 생성
  • 테스트가 항상 현재 달 기준으로 동작하여 유지보수성 향상 이를 통해 날짜 변경 시 테스트 수정 없이 자동으로 최신 날짜를 사용할 수 있게 되었습니다.

과제 피드백

반복일정 수정시 snackbar가 나타나지 않는점, 반복일정 dnd시 전체 변경이 되는지 반복일정은 dnd로 수정불가 등 과제에 명세되어 있었으면 더 좋았을 것 같습니다.

리뷰 받고 싶은 내용

Q. 현재 useDnd, WeekView, MonthView 컴포넌트는 따로 테스트 코드를 작성하지 않았습니다. useDnd에 사용하는 @dnd-kit은 외부라이브러리로 테스트를 위임하였고 동작하는 기능들은 E2E로 이미 검증된다고 판단하여 테스트를 추가하지 않았습니다. WeekView와 MonthView는 Props를 받아 렌더링만 수행하는 컴포넌트로 비즈니스 로직이 거의없다고 판단하였으며 이미 medium.integration.spec.tsx 파일에서 커버된다고 생각했습니다. 혹시 위 세가지 테스트하지 않은 항목중에 테스트를 했어야 하는 항목들이 있는지 코치님의 의견이 궁금합니다.

Q. playwright, cypress 등 e2e를 테스트할 때 로컬에서 병렬로 테스트가 가능하지만 CI 환경에서는 병렬 테스트의 한계가 있는 걸로 알 고 있습니다. 혹시 코치님께선 CI 환경에서도 병렬 테스트를 위해 따로 설정하신 적이 있으실까요?

Q. DnD와 웹 접근성에 대한 고민 드래그 앤 드롭(DnD) 기능을 구현하면서, 시각적 인터랙션 외에도 웹 접근성(Accessibility) 측면에서의 고려가 필요함을 느꼈습니다. 키보드 입력으로도 dnd를 이동해보고싶어 tab으로 cell을 선택하여 space로 dnd keyboardUp까진 진행하였는데 이동과 keyboardDown은 방법을 몰라 테스트하지 못하였습니다 혹시 코치님께선 웹 접근성관련하여 테스트하실때도 e2e툴인 playwright나 cypress를 활용하여 테스트 하시나요?

과제 피드백

수고했어요 진성. 지난 3주간 테스트 챕터를 하면서 추후 대형 소프트웨어에서 품질을 어떻게 지켜내야하는지 직간접적으로 체험하는 시간이 되었기를 바랍니다. 테스트를 만들기 위한 품이 많이 든다는 것. 그렇지만 이러한 테스트를 하지 않는데 더 큰 비용이 되는 순간이 온다는 것. 그리고 언젠가 커리상 이러한 대형 프로젝트를 맡아야 할 때 "테스트 작성할 줄 아나요?" 라고 했을때 생소하지 않고 할 수 있다는 자신감을 가질 수 있는 시간이 되었기를 바랍니다.

=> 좋은 판단이라고 생각합니다. WeekView와 MonthView등에서 렌더링만 수행하는거라면 각각의 세부적인 컴포넌트를 체크할 필요는 없고 e2e에서 특정 데이터가 제대로 노출되는가 정도로도 충분하겠죠. 테스트는 한번에 완성이 아니라 하면서 쌓아가지는 거니 지금 상황에서는 올바른 판단을 했다 생각합니다.

=> 간단한 키보드 동작이라면 테스트하는 경우도 있지만 접근성이 이미 검증이 된 라이브러리들를 다 접근성 형태로 테스트는 하고 있지 않고 있어요. 필요하게 되면 해야겠죠? 제 경험을 물으신다면 저는 테스트를 깊게 다루지는 않아서 이 정도 까지랍니다.
