과제 회고

이번 과제에서 가장 많이 고민했던 지점은, 이미 Tailwind를 쓰고 있는데 굳이 Primitive / Semantic / Component 레벨을 다시 나누고 토큰을 재정의하는 게 맞을까? 하는 부분이었습니다.

처음에는 Tailwind가 기본 팔레트와 spacing, 폰트 크기 스케일을 잘 제공하고 있으니, 여기에 또 다른 Primitive를 정의하는 것이 과한 추상화처럼 느껴지기도 했습니다.

하지만 실제로 디자인 토큰을 정리하면서, 아래와 같이 레이어를 나누는 것이 이 프로젝트에는 더 적합하다고 판단했습니다.

1. Primitive 레벨 – 값(스케일)을 정의하는 층

  /* Blue Scale */
  --blue-50: #e3f2fd;
  --blue-200: #90caf9;
  --blue-300: #80bdff;
  --blue-500: #007bff;
  --blue-900: #0d47a1;
   /* Gray Scale */
  --gray-50: #f9fafb;
  --gray-100: #f8f9fa;
  --gray-200: #f5f5f5;
  --gray-250: #f3f4f6;
  --gray-300: #e9ecef;
...

여기는 값 그 자체에 대한 정의입니다. Tailwind의 기본 스케일과 크게 결을 맞추되, 이 프로젝트에서 실제로 사용할 색상/그레이 스케일/간격/폰트 크기만 선별해서 CSS 커스텀 프로퍼티로 정리했습니다.

또한, 다크 모드용 토큰(--dark-bg-primary, --dark-text-primary 등)을 함께 정의해 두어 테마 전환을 Primitive 레벨에서 제어할 수 있도록 설계했습니다.

2. Semantic 레벨 – 의미를 부여하는 층

그다음 단계가 Semantic 토큰입니다.

--color-primary: var(--blue-500);
--color-secondary: var(--gray-650);
--color-success: var(--green-600);
--color-danger: var(--red-600);
--color-warning: var(--yellow-500);
--color-info: var(--cyan-600);

여기서는 어떤 색을 쓸지가 아니라 어떤 역할을 하는지를 이름으로 드러내는 데 집중했습니다.

--color-primary → 주요 액션, 핵심 강조용

--color-success → 성공/완료 상태

--color-danger → 오류/위험 상태

이 레벨 덕분에 나중에 브랜드 컬러나 톤이 바뀌더라도, --blue-500을 --blue-600 으로 바꾸는 식이 아니라 --color-primary가 가리키는 Primitive만 교체하면 전체 UI의 의미 체계는 유지될 수 있도록 했습니다.

3. Component 레벨 – 실제 UI에 투영되는 층

마지막으로, Button/Badge/Form/Alert/Modal 같은 컴포넌트 단위 토큰을 분리했습니다.

/* Button */
--color-btn-primary: var(--color-primary);
--color-btn-secondary: var(--color-secondary);

/* Badge */
--color-badge-primary: var(--color-primary);
--color-badge-warning: var(--color-warning);

/* Form */
--color-form-border: var(--gray-500);
--color-form-border-focus: var(--blue-300);
--color-form-placeholder: var(--gray-650);

/* Alert */
--color-alert-info-bg: var(--blue-50);
--color-alert-info-border: var(--blue-200);
--color-alert-info-text: var(--blue-900);

이 레벨은 어떤 컴포넌트가, 어떤 상태에서, 어떤 의미의 색을 어떻게 쓸지를 캡슐화하는 층입니다.

--color-primary 같은 Semantic 토큰을 그대로 여기저기 흩뿌리는 대신, --color-btn-primary, --color-badge-primary, --color-alert-info-bg처럼 컴포넌트별로 한 번 더 래핑해 두었고

덕분에 “버튼의 primary 톤”과 “배지의 primary 톤”을 같이 가져갈지, 살짝 다르게 가져갈지를 일관된 인터페이스 안에서 조정할 수 있도록 설계했습니다.

4. 왜 굳이 이렇게까지 나눴는가

Tailwind의 기본 토큰만 그대로 사용해도 UI를 만드는 데는 문제가 없습니다. 하지만 이번 과제에서는 이 프로젝트가 사용하는 토큰 체계를 명확히 하고, 나중에 변경과 확장이 들어와도 견딜 수 있는 구조를 만든다는 관점에서 접근했습니다.

그래서 Tailwind의 스케일을 기반으로 Primitive 값(색/간격/폰트)을 @theme에 정리하고, 이 값들 위에 역할/의도를 담은 Semantic 토큰을 얹고,마지막으로 Button, Badge, Form, Alert, Modal 같은 컴포넌트별 토큰으로 소비하는 구조를 택했습니다.

결론적으로, **Tailwind를 쓰는데 Primitive를 또 정의해도 될까?”*에서 출발했지만

지금은 Tailwind Primitive는 “도구의 스케일”,@theme에 정의한 Design Tokens는 “우리 프로젝트의 언어와 구조를 가진 토큰 레이어” 라고 역할을 분리해 이해하게 되었고, 이 구조가 이후 다크 모드 확장, 브랜드 리뉴얼, 디자인 가이드 추가 등에도 유연하게 대응할 수 있는 기반이 될 것이라고 판단했습니다. 배포된 Storybook

Before 패키지에서 발견한 문제점

1 컴포넌트 API가 제각각인 문제

“크기 조절”이라는 같은 목적을 갖고 있음에도 컴포넌트마다 props 이름과 값이 제각각입니다.

interface FormInputProps {
     width?: 'small' | 'medium' | 'large' | 'full';
     ...
}
interface FormSelectProps {
     size?: 'sm' | 'md' | 'lg';
     ...
}
interface FormTextareaProps {
      rows?: number;
      ...
}

하는 일은 비슷한데 어떤 컴포넌트는 width, 어떤 컴포넌트는 size를 쓰고 값 스케일도 각각 다르다 보니 사용할 때마다 “이 컴포넌트는 뭐였지?”를 다시 확인해야 합니다. 이 상태로는 컴포넌트를 한 묶음으로 다루기가 어렵고, UI 레벨에서 일관된 설계를 했다고 보기 힘듭니다.

2 스타일링 방식이 뒤섞여 있는 문제

한 프로젝트 안에서 최소 세 가지 스타일링 방식이 섞여 있습니다.

  1. 전통적인 CSS 클래스
    • 예: components.css 안의 .btn-primary { background-color: #1976d2; }
  2. 인라인 스타일
    • 예: FormInput.tsx 내 style={{ color: '#d32f2f' }}
  3. 컴포넌트 내부에서 직접 하드코딩된 스타일
    • 예: ManagementPage.tsx 내
      style={{ color: '#666', fontSize: '14px', background: '#1976d2' }}

이로 인해 비슷한 색이 여러 곳에 제각각 하드코딩되어 있고 색상이나 여백을 한 번에 바꾸기 어렵고 어디까지가 기획된 스타일이고 어디부터는 그때그때 붙인 스타일인지가 불투명합니다. 스타일이 한 군데 모여 있지 않다 보니, 변경이나 통일 작업이 필요할 때 부담이 큽니다.

3 타입 안전성이 떨어지는 부분 (any 남용)

여러 핵심 컴포넌트에서 any 타입이 넓게 쓰이고 있습니다.

  • Button.tsx: entity?: any
  • Table.tsx: data?: any[]
  • ManagementPage.tsx: formData: any

이 상태에서는

  • 타입 체크가 사실상 제대로 이루어지지 않고
  • IDE 자동완성/타입 추론을 온전히 활용하기 어렵고
  • 데이터 구조가 바뀌었을 때, 어디가 깨지는지 미리 알기보다
    런타임에서야 에러로 드러날 가능성이 큽니다.

UI와 도메인을 나누는 것도 중요하지만, 최소한의 타입 정의조차 안 되어 있는 부분은 추후 리팩토링 시 리스크로 작용할 수 있습니다.

UI 컴포넌트 안에 도메인/비즈니스 로직이 섞여 있는 문제

여러 컴포넌트에서 UI와 도메인 로직이 강하게 뒤엉켜 있습니다.

  • Button 컴포넌트
    • entityType, action, entity를 받아서
      “admin 유저는 삭제 불가” 같은 비즈니스 규칙을 내부에서 처리합니다.
  • FormInput 컴포넌트
    • fieldType, entityType, checkBusinessRules 등을 props로 받아
      사용자명 예약어 검증 등 도메인 검증 로직을 직접 수행합니다.
  • Table 컴포넌트
    • entityType과 columnKey에 따라 user/post별 렌더링 분기를 갖고 있습니다.

이 구조의 문제는:

  • 컴포넌트가 특정 도메인(user, post)에 강하게 묶여 있어 재사용성이 떨어지고
  • 새로운 엔티티나 비즈니스 규칙이 추가될 때마다 공용 UI 컴포넌트를 수정해야 하며
  • “UI는 UI대로, 도메인은 도메인대로” 나누기 어려운 구조가 된다는 점입니다.

5 접근성(Accessibility) 측면에서 빠진 부분

폼과 모달 컴포넌트 쪽에서 접근성 관련 속성이 빠져 있는 부분이 있습니다.

  • FormSelect, FormTextarea:
    • label에 htmlFor가 설정되어 있지 않아서,
      스크린 리더 기준으로는 연결 정보가 부족합니다.
  • Modal 컴포넌트:
    • role="dialog", aria-modal="true", aria-labelledby 없음
    • 포커스 트랩이 없어 키보드로 탭 이동 시 모달 밖으로 빠져나갈 수 있습니다.

6 Atomic Design 구조가 실제 개발 경험과 맞지 않는 점

현재 폴더 구조는 Atomic Design을 따르고 있습니다.

components/
├── atoms/
├── molecules/
└── organisms/

Card처럼 단순 컨테이너에 가까운 컴포넌트도 organisms에 위치하는 등분류 기준이 애매한 경우가 있습니다. /components/molecules/FormInput처럼 import 경로가 깊어져, 파일 이동·구조 변경이 부담스럽습니다.

새 컴포넌트를 만들 때마다 “이건 atom인가, molecule인가?”를 먼저 고민하게 되어 정작 구현보다 폴더 배치에 더 신경 쓰게 되는 문제가 있습니다.

개편 과정에서 집중한 부분

개편 과정에서 가장 집중한 부분은 폴더와 책임 구조를 어떻게 가져갈지였습니다. 기존에는 Atomic Design 기준으로 컴포넌트를 분류했지만, 실제 작업 흐름과 변경 포인트를 따라가다 보니 기능 단위로 모으는 것이 더 적합하다고 판단했습니다. 기능별로 관련 컴포넌트·훅·서비스·타입을 한 곳에 모을 수 있고, 공용 UI는 shared/ui 레이어로 분리해 재사용성을 높일 수 있다는 점에서 FSD(Feature-Sliced Design)의 장점이 이 규모에 더 잘 맞는다고 보았고, 그래서 기존 Atomic 구조에서 FSD 기반 구조로 변경하는 방향을 선택했습니다.

src/
├── features/          # 도메인/기능 단위 모듈
│   ├── post-management/
│   │   ├── components/  # 기능 전용 UI
│   │   ├── hooks/       # 비즈니스 로직
│   │   ├── lib/         # 기능 유틸리티/컬럼 정의 등
│   │   ├── schemas/     # 폼 검증 스키마
│   │   ├── services/    # API 통신
│   │   ├── types/       # 타입
│   │   └── index.ts     # Public API
│   └── user-management/ …
│
├── shared/            # 공용 레이어
│   ├── ui/            # 도메인 독립 UI (Button, Table, Modal 등)
│   ├── contexts/
│   └── lib/
│
└── pages/             # 라우트 단위 페이지
    └── ManagementPage.tsx

각 레이어의 역할

features : “게시글 관리”, “사용자 관리”처럼 도메인 단위로 모든 로직을 모아둔 레이어 컴포넌트, 훅, 서비스, 타입, 스키마까지 한 기능 폴더 안에서 완결

shared : 어떤 도메인에서도 쓸 수 있는 범용 UI & 공용 유틸 shared/ui/Button, shared/ui/Table 등은 도메인 모른 척하고 순수 UI만 담당

pages : 여러 feature를 조합해서 페이지를 구성하는 레이어 예: ManagementPage에서 PostManagement, UserManagement를 탭으로 전환

개편을 통해서 얻은 이점들로는

1. 먼저 관심사 분리가 훨씬 명확해졌습니다.

이전에는 도메인 검증 로직이나 비즈니스 규칙이 버튼, 인풋, 테이블 같은 UI 컴포넌트 안에 그대로 섞여 있었는데, 이제는 이런 부분을 전부 features/* 레이어로 올려버렸습니다. 예를 들어 “예약된 사용자명인지 체크한다”, “게시글을 발행할 수 있는 상태인지 판단한다” 같은 검증·비즈니스 규칙은 features/post-management, features/user-management 내부에서 처리하고, shared/ui/*에 있는 컴포넌트들은 말 그대로 UI 표현과 레이아웃만 담당하는 레이어가 되도록 분리했습니다.

이렇게 나누고 나니, 기능 단위로 개발·삭제하는 것도 훨씬 단순해졌습니다. 게시글 기능을 예로 들면, 관련된 컴포넌트, 훅, 서비스, 타입, 스키마가 모두 features/post-management/ 아래에 모여 있기 때문에, 이 기능을 통째로 제거하거나 수정하고 싶을 때 사실상 이 폴더 하나만 보면 됩니다. 예전처럼 atoms/molecules/organisms 여기저기를 왔다 갔다 할 필요가 없습니다.

2. 재사용성 측면에서도 이득이 있습니다.

shared/ui/Table, shared/ui/Button처럼 shared/ui에 위치한 컴포넌트들은 이제 도메인에 대한 정보를 전혀 모르도록 설계했습니다. 그래서 게시글, 사용자, 상품 등 어떤 도메인이든 columns와 data만 맞춰 넘겨주면 쓸 수 있고, 극단적으로는 이 레포지토리가 아니라 다른 프로젝트에서도 가져다 쓸 수 있는 수준의 범용 컴포넌트에 가깝게 정리되었습니다.

3. 의존성 방향이 명확해졌다는 점입니다.

구조를 pages → features → shared 단방향으로 잡으면서, 위에서 아래로만 의존하도록 제한했습니다. shared 레이어는 어느 도메인(features)에도 의존하지 않도록 설계해서, 이 레이어의 순수성을 최대한 유지하려고 했습니다. 덕분에 “공용 레이어가 특정 도메인에 물드는” 상황을 미리 차단할 수 있습니다.

4. 코드 자체도 눈에 띄게 가벼워졌습니다.

Button 컴포넌트에서 entityType, action 같은 도메인 관련 props와 조건 분기를 걷어내고 순수 UI로만 남기다 보니 파일 줄 수가 자연스럽게 줄었습니다. Table 컴포넌트도 마찬가지로, “user일 때는 이렇게, post일 때는 저렇게” 같은 도메인 분기를 제거하고, 컬럼 정의와 렌더링 전략을 features 쪽으로 옮기면서 훨씬 단순한 범용 컴포넌트로 정리할 수 있었습니다.

사용한 기술 스택 경험

이번 과제에서는 기존에 익숙했던 단순 React 훅 기반 폼 구현 방식에서 한 단계 나아가, shadcn/ui의 Form 컴포넌트 + React Hook Form + Zod 조합으로 폼 레이어를 설계해 보았습니다. 실무에서 React Hook Form 자체는 사용해 본 경험이 있었지만, shadcn/ui에서 제공하는 Form 래퍼와 Zod 스키마를 함께 사용하는 형태는 이번이 처음이었습니다.

shadcn/ui가 React Hook Form과 Zod 연동을 위한 기본 템플릿(Form, FormField, FormItem, FormMessage 등)을 제공해 주다 보니, 폼 인프라를 처음부터 세팅하는 데 드는 시간이 크게 줄었습니다. 덕분에 “폼 상태를 어떻게 배선할까?”보다는, 어떤 스키마를 기준으로 검증하고 어떤 UX를 줄 것인가에 집중할 수 있었습니다.

또한 검증 로직을 Zod 스키마로 일원화하고, React Hook Form의 resolver를 통해 바로 연결하는 패턴이 꽤 인상적이었습니다.

필드별 에러 메시지, required 처리, 최소/최대 길이 등의 규칙을 컴포넌트 내부가 아니라 스키마에 모아서 정의할 수 있고

shadcn/ui의 FormMessage 컴포넌트와 결합해 에러 표시 로직을 공통화·추상화할 수 있어서, 폼이 늘어나도 동일한 패턴으로 확장 가능한 구조를 만들 수 있었습니다.

렌더링 관점에서도, React Hook Form이 기본적으로 uncontrolled 패턴과 subscription 기반 업데이트를 사용하는 특성 덕분에, 입력값 변경 시 불필요한 리렌더링을 최소화하는 것을 다시 한 번 체감했습니다. 특히 필드 단위로 Controller/useController를 적절히 사용했을 때, 폼 전체가 아닌 필요한 부분만 다시 그려지는 구조라서, 향후 실무에서도 이 패턴을 적극적으로 활용할 수 있겠다고 느꼈습니다.

마지막으로, shadcn/ui는 Tailwind 기반이지만 Headless UI에 가까운 구성(조합 중심, className 기반 커스터마이징)을 제공하기 때문에, 디자인 토큰/테마 레이어와도 잘 맞았습니다. 폼 UI를 라이브러리 스타일에 맞추는 게 아니라, 이번 과제에 디자인 토큰과 Variant(CVA)에 맞춰 자연스럽게 입혀갈 수 있다는 점이 좋았습니다.

어려웠던 점과 해결 방법

이번 과제에서 가장 어려웠던 지점은, 디자인 시스템을 도입하는 리팩토링인데, 어디까지를 **변경되지 않는 계약(Contract)**으로 봐야 할까? 를 초반에 제대로 정의하지 못했다는 점이었습니다.

처음 과제를 시작할 때 저는 “명세상 기능은 동일하고, 리팩토링이면 결과도 같아야 한다” 라고 이해했기 때문에, 기존 동작을 보존하는 것에 초점을 두고 먼저 테스트 코드와 스토리북을 작성했습니다.

테스트 코드: 리팩토링 전과 동일한 결과를 보장하는 용도
스토리북: UI가 “겉보기에는” 변하지 않도록 하는 시각적 기준점

으로 삼았고, “이 둘이 리팩토링의 안전망 역할을 해줄 것”이라고 기대했습니다. 하지만 실제 리팩토링이 구조 개편 + 관심사 분리 + FSD 전환 + 디자인 시스템 도입까지 포함한 수준으로 확장되면서 상황이 달라졌습니다.

UI 컴포넌트에서 도메인 로직을 떼어내고 도메인별 로직을 features로 옮기고 공용 UI를 shared/ui로 분리하다 보니 초기에 작성해 두었던 테스트 코드와 스토리북이 더 이상 현재 구조를 제대로 설명하지 못하는 상태가 되어버렸습니다.

겉으로 보기엔 기능은 비슷하지만, 컴포넌트의 책임이 달라지고 props의 형태도 바뀌고 레이어(페이지/feature/shared) 경계도 새로 생기면서 “리팩토링 전과 완전히 동일한 테스트”를 유지하는 것 자체가 의미가 없어지는 지점에 도달했습니다.

결국 리팩토링을 진행할수록, 기존 테스트는 옛 구조에 묶여 있는 발목이 되고 스토리북도 “지금 설계 기준”이 아니라 “예전 컴포넌트 경계 기준”에 맞춰져 있어서 조금씩 “수정”하는 수준이 아니라, 사실상 새로 쓰는 것과 다르지 않은 상태가 되었습니다.

이 시점에서 결정을 다시 했습니다. “형식적으로 기존 테스트를 억지로 끌고 가기보다는, 구조 개편과 마이그레이션을 먼저 끝내고, 새 구조 기준으로 테스트와 스토리북을 다시 설계하는 것이 더 낫다.”

그래서 최종적으로는 기존 테스트 코드와 스토리북을 과감히 제거하고 마이그레이션이 완료된 FSD + shared/ui 구조를 기준으로 다시 테스트 코드와 스토리북을 작성하는 방향으로 전환했습니다.

리뷰받고 싶거나 질문하고 싶은 내용

사실 저는 그동안 주로 초기 스타트업 환경에서 일을 해서, 테스트 코드가 강하게 요구되거나 체계적으로 관리되는 문화에 노출되지는 못했던 것 같습니다.

이번 마이그레이션을 진행하면서, 초반에는 변경 범위가 비교적 작다 보니 테스트 코드가 주는 안정감이 굉장히 컸습니다. “아, 먼저 테스트를 작성해 두길 잘했다.” 라고 몇 번이나 느꼈고, 작은 리팩토링을 할 때는 분명 큰 도움이 되었습니다.

하지만 점점 구조 개편의 범위가 커지고, 변경되는 파일과 레이어가 많아지면서 상황이 달라졌습니다. 어느 순간부터는 기존 테스트 코드 대부분을 결국 다시 쓰게 되는 지점에 도달했고, “이 정도 규모의 마이그레이션에서는, 처음에 쓴 테스트가 정말로 효율적이었나?”라는 고민이 들었습니다.

  • 코치님도 비슷하게, 초기에 작성한 테스트 코드가 대규모 구조 변경 과정에서 거의 전부 다시 써야 하는 상황을 겪으신 적이 있으실까요?
  • 그리고 이런 상황에서는 어디까지를 ‘반드시 지켜야 할 영역(Contract)’으로 보고 테스트를 가져가야 할지, 혹은 테스트 전략을 어떻게 가져가는 것이 Best Practice에 가까운지 코치님의 경험과 기준이 궁금합니다.
