과제 셀프회고

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

FSD는 큰 구조에만 어울리고 복잡하다고만 생각했습니다.

처음 FSD 구조를 봤을 때 솔직히 "이거 과한 거 아니야?" 싶었습니다. entities, features, widgets, shared... 뭐가 이렇게 많은지. 그냥 components 폴더 하나면 되는 거 아닌가?

그런데 실제로 적용하려니까 매번 "이거 어디에 넣지?"라는 고민이 반복됐습니다. 특히 PostTags 컴포넌트 만들 때가 가장 어려웠습니다.

처음엔 "태그를 클릭하면 필터링되니까 features/post/filter-by-tag에 넣어야겠다!"라고 생각했습니다. 그런데 막상 넣고 보니 어색했습니다. PostTags는 그냥 태그를 보여주기만 하고, 클릭 이벤트는 props로 받는데 이게 feature인가?

그래서 shared/ui로 옮겼습니다. "여러 곳에서 쓰니까 공통 컴포넌트지!" 했는데, 또 이상했습니다. Post에만 쓰이는데 shared라고 하기엔 너무 특화된 컴포넌트 아닌가?

결국 entities/post/ui에 정착했습니다. 3번째 시도만에요. 이 과정에서 깨달은 건, "어디서 쓰이나?"가 아니라 "무엇에 대한 것인가?"를 먼저 생각해야 한다는 것이었습니다.

PostTags는 "Post의 태그들을 표시"하는 컴포넌트입니다. Post 타입만 있으면 동작합니다. 그러니까 Post 엔티티의 일부인 거죠. 이걸 깨닫는데 하루 반나절 걸렸습니다.

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

Layer에 나눠 파일을 구분할때 기준을 세웠습니다. 처음엔 명확한 기준없이 시작할때는 "이거 features/post에 넣을까 entities/post에 넣을까" 등 고민을 많이 하였습니다.

  • "Post에 대한 표현인가? 행동인가?" → 표현이면 entities, 행동이면 features
  • "사용자가 클릭하는 건가? 그냥 보여주기만 하는 건가?" → 클릭하면 features, 보여주기만 하면 entities
  • "어디서든 쓸 수 있나? 특정 도메인 전용인가?" → 어디서든 쓰면 shared, 특정 도메인이면 해당 entities 이런식으로 기준을 세우고 layer와 계층들을 구분하였습니다.

features를 쪼개는 것이 생각보다 어려웠습니다

처음엔 이렇게 하고 싶었습니다:

features/
  post/
    index.tsx  // 여기에 생성, 수정, 삭제, 검색 다 넣자!

"왜 굳이 폴더를 여러 개로 쪼개? 관리하기 더 복잡해지잖아"라고 생각했습니다.

그런데 막상 create-post, edit-post, delete-post를 분리해놓고 나니 개발할 때 머리가 훨씬 편했습니다.

"게시물 생성 기능 고쳐야지" → create-post 폴더만 열면 됨. 다른 건 신경 안 써도 됨. "삭제 버튼에 확인 모달 추가해야지" → delete-post만 수정하면 됨. 생성이나 수정 기능에 영향 없음.

특히 나중에 "게시물 삭제 기능에 버그가 있어"라는 상황이 왔을 때, delete-post 폴더만 봤습니다. 70줄짜리 코드에서 버그를 찾는 건 쉬웠습니다.

만약 PostFeatures.tsx 하나에 모두 모았다면 500줄 넘는 파일에서 "이게 삭제 로직인가? 수정 로직인가?"를 계속 확인했을 겁니다.

서버상태관리 전역상태관리에 대한 고찰 서버에 대한 상태관리는 TanstackQuery로 관리한다면 그외 모달과 searchParam등은 zustand로 관리해야하는가에 대한 고민이 되었습니다. 저는 이번 상태관리에 대해서는 overlay pattern을 참고하여 context-api를 사용하여 상태관리를 하였습니다. zustand나 jotai를 사용하기에 도메인관련 상태를 관리하지 않다고 판단하였기 때문입니다.

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

widgets의 기준이 생각보다 헷갈렸습니다

도메인 기준으로 entities를 정리하다 보니, 서로 다른 entities를 조합해야 하는 순간이 자주 생겼습니다. 이때 **“entities끼리 조합한 결과를 entities에 두는 게 맞나, widgets로 올려야 하나?”**가 가장 고민이었습니다.

특히 조합 로직을 무리하게 entities에 넣다 보면, 화면을 완성하기 위해 features 컴포넌트가 필요해져서 entities가 features를 import하는 레이어 역전이 발생하기도 했습니다.

그래서 레이어 역전을 막기 위해, 아래처럼 조합 규칙을 명시적으로 정리했습니다.

entities + entities → entities (같은 도메인 단위의 결합/가공은 entities에서 해결)

features + features → features (같은 사용자 행동 단위의 조합은 features 내부에서 해결)

entities + features → widgets (데이터(도메인) + 행동(UI/액션)을 한 블록으로 묶는 순간부터는 widgets 책임)

이 규칙을 기준으로, entities는 도메인에만 집중하고 features는 사용자 행동에 집중하게 하여, 상위 레이어에서만 조합이 일어나도록 구조를 정리했습니다.

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

팀 컨벤션으로 제안해보고 싶습니다

FSD를 경험하면서 "팀에 도입하면 좋겠다"는 생각을 했습니다.

이전 팀에서는 PR마다 이런 논의가 반복되었습니다.

  • "이 파일 components에 넣을까요 utils에 넣을까요?"
  • "공통 로직은 어디에 모으는 게 좋을까요?"
  • "이 컴포넌트는 hooks 폴더에 넣는 게 맞나요?"

매번 리뷰어마다 의견이 달라서 결국 "어디든 상관없지 않나요?"로 끝납니다.

그런데 FSD가 있으면

  • "게시물 생성이니까 features/post/create-post입니다"
  • "Post 타입이니까 entities/post/model입니다"

이렇게 명확한 기준으로 대화할 수 있을 것 같습니다.

다만 갑자기 "FSD로 전환합시다!"라고 하면 "왜 갑자기? 지금도 잘 돌아가는데?"라는 반응이 나올 것 같습니다.

그래서 먼저 작은 프로젝트나 새로 시작하는 기능에 FSD를 적용해보고, 실제 효과를 보여주는 접근이 나을 것 같습니다.

(설득은 말보다 결과물로 하는 게 효과적이라 생각합니다.)

상태 관리 결정을 더 빠르게 내릴 수 있게 되고 싶습니다

이번 과제를 하면서 "이 상태는 어디에?" 고민하는 시간이 개발 시간의 상당 부분을 차지했습니다.

이제는 기준이 생겼습니다:

"서버에서 오는 데이터인가?" → TanStack Query "도메인 상태관리를 하는가?" → Zustand or Jotai "Modal, Taost등 UI에 대한 상태관리인가" → Context "이 컴포넌트에서만 쓰나?" → useState

이 질문들을 습관처럼 하다 보면, 나중엔 고민 없이 자동으로 판단할 수 있을 것 같습니다.

챕터 셀프회고

클린코드와 아키테쳑 챕터 함께 하느라 고생 많으셨습니다! 지난 3주간의 여정을 돌이켜 볼 수 있도록 준비해보았습니다. 아래에 적힌 질문들은 추억(?)을 회상할 수 있도록 도와주려고 만든 질문이며, 꼭 질문에 대한 대답이 아니어도 좋으니 내가 느꼈던 인사이트들을 자유롭게 적어주세요.

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

이전 코드를 다시 보는 것이 가장 힘들었습니다

예전 코드를 변경할 일이 있어 작성된 코드를 분석중

"이 state는 누가 업데이트하는 거지?" "이 함수는 어디서 쓰이는 거지?" "왜 이렇게 짰을까?"

파일을 왔다 갔다 하면서 추적했는데, 알고 보니 제가 2주 전에 작성한 코드였습니다.

그 순간 깨달았습니다. "클린코드는 남을 위한 게 아니라 미래의 나를 위한 것이구나."

도메인별로 구분이 되어있지 않아 여기저기 흩어진 파편을 찾아서 다시 계층별 분리를 했던 기억이 있습니다.

그래서 이번엔 기능별로 철저히 분리했습니다.

"게시물 삭제 기능 수정" → delete-post 폴더만 열면 됨. 끝.

이렇듯 도메인위주 그리고 기능별로 만들면 찾는데도 금방 찾을 수 있습니다.

"명확한 코드"와 "짧은 코드"는 다릅니다

예전엔 코드를 짧게 쓰는 게 클린코드인 줄 알았습니다.

const data = useQuery(postQueries.list(params)).data

이렇게 한 줄로 쓰면 간결합니다.

그런데 나중에 보니 인터페이스가 없다보니 "params는 뭐지? 어디서 온 거지?"를 계속 추적해야 했습니다.

만약 아래처럼 변경한다면

// 게시물 목록 데이터 가져오기 (페이지네이션, 정렬 포함)
const { data: postsData } = useQuery(
  postQueries.list({
    limit,
    skip,
    sortBy,
    sortOrder,
  }),
)

코드량은 늘었지만, 한눈에 "아, 페이지네이션과 정렬을 적용해서 게시물을 가져오는구나"를 바로 알 수 있습니다.

명확한 것이 짧은 것보다 낫다고 생각합니다.

결합도 낮추기: 디자인 패턴, 순수함수, 컴포넌트 분리, 전역상태 관리

전역 상태를 쓰는 이유를 제대로 이해하게 되었습니다

이전 과제에서는 계층 간 분리를 목표로 상태 관리를 **Zustand(전역 상태)**로 선택했습니다.

이번에 PostsParamsContext를 만들면서 전역 상태에 대한 기준이 더 명확해졌습니다. PostsParamsContext는 props drilling을 피하려고 만든 것이 아닙니다. 검색어/필터/페이지 번호 같은 목록 파라미터는 단순 UI 상태가 아니라 내비게이션 상태에 가깝고, **URL과 동기화(뒤로가기/새로고침/딥링크/공유)**되어야 하므로 상위 경계에서 일관되게 관리하기 위해 Context로 구성했습니다.

반대로 CreatePostDialog 내부의 newPost는 전역으로 올릴 이유가 없습니다. 이 값은 다이얼로그가 열려 있는 동안만 의미가 있고, 닫히면 폐기되는 수명이 짧은 로컬 UI 상태이기 때문에 useState로 관리하는 게 가장 자연스럽습니다.

결국 전역 상태는 “props 내려주기 번거로움”을 해결하기 위한 도구가 아니라, 공유 범위가 넓고 수명이 길며, 앱의 정책/동작을 일관되게 유지해야 하는 상태에 대해 단일 진실 공급원(SSOT)과 경계(boundary)를 만들기 위한 아키텍처 선택입니다.

API 함수를 분리하니 신뢰할 수 있었습니다

postApi.ts에 fetchPosts, createPost, deletePost 등을 모아뒀는데, 생각보다 편리했습니다.

"게시물 가져오는 API 어디 있지?" → postApi.ts를 열면 있습니다 "이 API 다른 곳에서도 써야 하는데?" → import해서 사용하면 됩니다

게다가 이 함수들은 순수 함수이기 때문에

  • 같은 params를 넣으면 같은 결과를 반환합니다
  • 외부 상태를 변경하지 않습니다
  • 테스트하기 쉽습니다

CreatePostDialog에서도 쓰고, PostTable에서도 쓰는데 각각 fetch를 중복으로 작성하면 나중에 API 주소가 바뀌면 모두 수정해야 합니다.

한 곳에 모아두니 유지보수가 편했습니다.

feature를 분리하니 집중하기 쉬웠습니다

create-post는 "생성"만 담당합니다. 수정도, 삭제도 모릅니다.

처음엔 "이거 너무 과한 거 아닌가? 그냥 PostManager 하나에 다 넣으면 안 될까?"라고 생각했습니다.

그런데 막상 해보니 개발할 때 집중이 잘 되었습니다.

"게시물 생성 기능 만들어야지" → create-post만 보면 됨. 수정, 삭제 코드는 신경 쓰지 않아도 됨.

500줄짜리 파일 하나보다, 70줄짜리 파일 7개가 훨씬 관리하기 편했습니다.

응집도 높이기: 서버상태관리, 폴더 구조

처음엔 FSD 구조가 과해 보였는데, 지금은 아닙니다

entities, features, widgets, shared... 폴더가 4개나 됩니다.

"이거 너무 복잡한 거 아닌가? 그냥 components 하나면 되는데?"

그런데 실제로 사용해보니 파일 찾는 시간이 정말 줄어들었습니다.

"Post 타입 어디 있지?" → entities/post/model "게시물 생성 기능 어디 있지?" → features/post/create-post "공통 Button 컴포넌트 어디 있지?" → shared/ui

생각하지 않아도 바로 찾을 수 있습니다.

반면 이전 코드는 components 폴더에 PostCard, PostForm, PostButton, PostList... 모두 섞여 있었는데, 이게 무엇을 하는 컴포넌트인지 일일이 열어봐야 했습니다.

"기술"이 아니라 "도메인"으로 생각하게 되었습니다

예전엔 이렇게 생각했습니다

  • "타입이니까 types 폴더"
  • "컴포넌트니까 components 폴더"
  • "함수니까 utils 폴더"

FSD로 도메인 관점에서 생각하면

  • "Post에 대한 것이니까 entities/post"
  • "사용자가 게시물 생성하는 것이니까 features/post/create-post"

도메인 중심으로 생각하니 관련된 코드가 한 곳에 모였습니다.

Post 타입도, Post API 호출도, Post 쿼리도 전부 entities/post 안에 있으니까 응집도가 높아진 것입니다.

useState + useEffect 패턴에서 벗어났습니다

이전에는

const [posts, setPosts] = useState([])
const [loading, setLoading] = useState(false)
const [error, setError] = useState(null)

useEffect(() => {
  setLoading(true)
  fetch("/api/posts")
    .then(...)
    .catch(...)
    .finally(() => setLoading(false))
}, [])

이런 코드를 컴포넌트마다 반복했습니다. 캐싱도 직접 구현하고, 새로고침도 직접 관리해야 했습니다.

그런데 TanStack Query를 사용하니:

const { data: posts } = useQuery(postQueries.list({ limit: 10 }))

이것으로 끝입니다. 로딩, 에러, 캐싱, 재검증이 모두 자동으로 처리됩니다.

처음엔 "마법 같다"고 생각했는데, 사용해보니 이것이 "서버 상태 관리"의 표준이었습니다.

서버에서 오는 데이터는 TanStack Query, 클라이언트 UI 상태는 useState/Context. 이 구분이 명확해지니 혼란이 없어졌습니다.

리뷰 받고 싶은 내용이나 궁금한 것에 대한 질문

1. widgets 기준을 잡는 과정에서 많이 헷갈렸습니다.

도메인 기준으로 entities를 정리하다 보니, 개발하다 보면 서로 다른 entities를 조합해야 하는 순간이 자주 생기더라고요. 이때마다 고민이 생겼습니다.

entities끼리 조합한 결과를 계속 entities에 두는 게 맞을까요?

아니면 조합이 시작되는 순간부터 widgets로 올리는 게 더 맞는 걸까요?

실제로 조합 로직을 무리하게 entities 쪽에 넣다 보면, 화면을 완성하기 위해 features 컴포넌트가 필요해져서 entities가 features를 import하는 레이어 역전이 발생하기도 했습니다.

그래서 일단 제 나름대로 레이어 역전을 막기 위해 아래처럼 규칙을 세워봤습니다.

entities + entities → entities (같은 도메인 단위의 결합/가공은 entities에서 해결)

features + features → features (같은 사용자 행동 단위의 조합은 features 내부에서 해결)

entities + features → widgets (데이터 + 행동을 한 블록으로 묶는 순간부터는 widgets 책임)

이 규칙대로 하면 entities는 도메인에 집중하고, features는 사용자 행동에 집중하면서 조합은 상위 레이어에서만 일어나게 만들 수 있었습니다.

코치님께서는 entities 간 조합 / entities+features 조합의 경계를 어떤 기준으로 나누시는지, 그리고 제가 정한 규칙이 과하게 단순화된 건 아닌지 의견을 듣고 싶습니다.

2. staleTime을 어떻게 결정하는 것이 좋을지 궁금합니다

현재 게시물 목록은 30초, 태그별 게시물은 60초로 설정했는데, 명확한 근거 없이 "느낌"으로 정했습니다.

list: (params) =>
  queryOptions({
    queryKey: [...postQueries.lists(), params],
    queryFn: () => postApi.fetchPosts(params),
    staleTime: 30_000, // 왜 30초? 명확한 근거 없음
  }),

실무에서는 어떤 기준으로 결정하시나요?

데이터 특성? (자주 변하는지 아닌지) 사용자 행동? (얼마나 자주 새로고침하는지) 비즈니스 요구사항? (실시간성이 중요한지)

구체적인 판단 기준이 궁금합니다.

Q. widgets 기준을 잡는 과정에서 많이 헷갈렸습니다.

말씀하신 규칙 (entities+entities→entities, entities+features→widgets)은 FSD 원칙에 잘 맞는 방법 이라고 생각합니다. 무엇보다 제일 피해야하는것은 레이어 의존의 역전되는 상황일것 같아요. 사실 각 레이어가 무엇을 뜻하는지가 중요한다기보다는 어떤 레이어가 어떤 레이어를 사용할 수 있는지 즉 그냥 추상화 래밸의 수준으로 이해하시면 될 것 같아요. 그게 그냥 모든것을 다 무시할 수 있는 FSD의 핵심인 것 같습니다. 그래서 말씀하신 규칙대로 사용하셔도 되고요. 중요한 것을 일관성 일것 같아요. 저도 레이어와 각레어의 특성은 어플리케이션마다 커스텀해서 정의합니다. FSD와 전혀 다른 것도 사용합니다.

Q. staleTime을 어떻게 결정하는 것이 좋을지 궁금합니다

아시다시피 자주 변경되면 낮게 거의 변경이 없다면 높게 설정하는 것이 기본적입니다. 이것을 데이터를 기준으로 한다고해서 늘 정확한것 도 아닌 것 같아요. 중요한건 업데이트 주기 같아요. 유저가 업데이트를 할 수 있고 그것을 모두가 볼 수 있는 상황이라면 업데이트 주기를 판단할 수 없으니 낮게 잡는 것이 맞고 우리가 업데이트 하는 경우는 업데이트 주기를 알 수 있으니 그 주기에 맞게 설정하면 될 것 같습니다. 저도 처음엔 페이지 재로드 빈도같은 데이터를 활용해야 하나 했는데 그것도 그냥 평균일 뿐이니까요. 평균에 함정에 빠질 수 있습니다 :) 아 물론 참고는 할 수 있죠.
