질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

기존 @/__mocks__/handlers에 있는 코드는 모든 테스트가 공유 상태를 사용하기에, 병렬 실행 시 데이터 충돌이 발생하는 구조였습니다. 이를 해결하기 위해 createIsolatedTestHandlers 함수를 활용하였고, 각 테스트마다 독립적인 스토어 인스턴스틑 생성하도록 했습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

  • 독립적 실행을 위해 @/__mocks__/store/testEventStore.ts에 메모리 기반 이벤트 스토어인 TestEventStore 클래스를 구현하였습니다. 이를 통해 개별 테스트가 독립된 데이터 공간을 가지게 되는데요. 그렇게 해서 여러 테스트가 동시에 실행되어도 서로의 데이터에는 영향을 주지 않는 구조를 활용해봤습니다.

심화 과제

  • App 컴포넌트 적절한 단위의 컴포넌트, 훅, 유틸 함수로 분리했는가?
  • 해당 모듈들에 대한 적절한 테스트를 2개 이상 작성했는가?

과제 셀프회고

기술적 성장

새로 학습한 개념

  • MSW (Mock Service Worker): 실제 네트워크 요청을 가로채서 mocking하는 새로운 패러다임을 학습했습니다. 기존의 jest.mock과는 다른 브라우저 레벨에서의 모킹을 통해 더 현실적인 테스트 환경을 구축할 수 있었습니다.

  • 테스트 격리(Test Isolation): createIsolatedTestHandlers와 TestEventStore를 통해 각 테스트가 독립적인 데이터 공간을 가지도록 하는 설계를 학습했습니다. 병렬 테스트 실행 시 데이터 충돌을 방지하는 핵심 개념을 실제로 구현해볼 수 있었습니다.

기존 지식의 재발견/심화

  • React Testing Library: 단순한 컴포넌트 테스트를 넘어서 사용자 중심의 테스트 작성법을 깊이 있게 경험했습니다. userEvent를 활용한 실제 사용자 상호작용 시뮬레이션과 waitFor를 통한 비동기 처리 테스트를 진행하며, 유저의 입장에서 다시 한 번 고민하는 계기가 되었습니다.

  • 단위 테스트 vs 통합 테스트: 단위 테스트는 개별 함수의 정확성을, 통합 테스트는 사용자 시나리오의 완전성을 검증한다는 차이점을 몸소 경험했습니다. 특히 통합 테스트에서 addEventForIntegration 헬퍼 함수를 통해 복잡한 UI 플로우를 재사용 가능하게 만드는 패턴을 학습했습니다. 또한, 이번 과제를 통해 이 둘은 상호 보완적인 관계라는 느낌이 들었습니다. 단위 테스트만 해서도 안되고, 통합 테스트만 하는 것도 지양해야 할 부분인 것 같아요.

구현 과정에서의 기술적 도전과 해결

  • 타임아웃 이슈 해결: 통합 테스트에서 발생한 5초 타임아웃 문제를 개별 테스트별로 10초로 조정하여 해결했습니다. 전역 설정 대신 선택적 조정을 통해 유닛 테스트의 빠른 실행 속도는 유지하면서 복잡한 통합 테스트에만 충분한 시간을 제공하는 균형점을 찾았습니다.

  • 의존성 관리 문제: pnpm lockfile 동기화 문제를 겪으며 명세에 없는 의존성 제거와 CI 환경에서의 frozen-lockfile 오류 해결 과정을 통해 패키지 관리의 중요성을 학습했습니다.

코드 품질

특히 만족스러운 부분

  • 모듈화 및 단일 책임 원칙: 모든 함수를 80줄 이하로 제한하고 하나의 기능만 수행하도록 철저히 분리했습니다. 이를 통해 src/utils/, src/hooks/, src/components/ 구조로 명확한 관심사 분리를 달성했습니다.

  • 이슈 기반 개발 워크플로우: Forked Repository에서 GitHub Issue를 사용할 수 없는 제약을 ISSUES/ 폴더로 해결했습니다. 총 24개의 상세한 이슈 문서를 통해 개발 과정을 체계적으로 기록하고 추적할 수 있는 시스템을 구축했습니다.

  • 재사용 가능한 테스트 헬퍼: addEventForIntegration 함수를 통해 중복되는 UI 상호작용 로직을 공통화했습니다. 입력 필드 초기화, 겹침 경고 dialog 처리 등의 복잡한 시나리오를 캡슐화하여 테스트 코드의 가독성과 유지보수성을 크게 향상시켰습니다.

리팩토링이 필요한 부분

  • TypeScript 타입 정의: 모듈화 과정에서 타입 정의가 다소 미흡했습니다. 특히 공통 타입들을 @/types/에 더욱 체계적으로 정리하고, 유틸 함수들의 제네릭 타입 활용을 개선할 필요가 있습니다.

  • 통합 테스트 성능 최적화: 현재 일부 통합 테스트가 타임아웃 이슈로 10초까지 소요됩니다. addEventForIntegration 헬퍼의 불필요한 대기 시간을 줄이고, MSW 응답 속도를 개선하여 테스트 실행 시간을 단축할 계획입니다.

코드 설계 관련 고민과 결정

  • 테스트 격리 전략: 전역 상태 공유로 인한 테스트 간 간섭을 방지하기 위해 각 테스트마다 독립적인 스토어 인스턴스를 생성하는 설계를 선택했습니다. 이는 테스트 안정성을 높였지만, 메모리 사용량 증가라는 트레이드오프가 있었습니다.
  • 이슈 기반 개발: GitHub Issue 대신 마크다운 기반의 이슈 관리 시스템을 도입했습니다. 이는 코드 리뷰어의 이해를 돕고, Squash merge와의 시너지를 통해 깔끔한 커밋 히스토리를 유지할 수 있게 했습니다.

Q. 저는 하나의 util과 hook은 하나의 기능만 수행해야 한다. 그리고, 코드는 80줄을 넘으면 안된다.는 나름의 철칙을 갖고 있습니다. 그러나, 이 경우 파일들의 depth가 깊어지고, 불필요한 리팩토링이 발생할 수 있다는 생각이 들어요. 다만, 저는 이 방식을 고수할 경우 설계단에서 더 깊게 고민할 수 있고, 공통 | 모듈화 작업을 계속 고민하며 코드를 작성하게 된다는 장점이 있다고 생각하기에 이 방법을 앞으로도 고수할 마음을 갖고 있습니다. 코치님께서 생각하시는 좋은 설계 | 구조는 어떤 것일지 궁금합니다,,! 요즘은 아키텍처나 설계단을 꾸준히 고려하는 연습을 하고 있는데, 코치님의 의견을 들을 수 있다면 정말 큰 공부가 될 것 같습니다🥹.

Q. 저는 현재 통합테스트를 describe 단위로 쪼개어 @/tests/integration/에 넣어두었습니다. 제가 나눈 이유는 바로 하나의 describe는 독립적으로 기능해야 한다. 였으며, 또한 가독성을 위해 나눈 것이었습니다. 테스트 코드는 말 그대로 spec이니까요. 그런데, 통합테스트의 의도 자체가 말 그대로 Integration이라면, 이걸 나눠도 되는 것일까? 하는 생각이 동시에 들었습니다. 만약 정말 유저의 모든 액션을 통합해서 보고 싶다면, 명세 시 하나로 합쳐야 하지 않나? 라고 생각해요. 그러나, 역시 의문이 있습니다. 통합 테스트에서 저렇게 여러 개의 단위로 나누는 행동은 지양해야 할까요? 아니면, 저렇게 하여도 무방할까요?

A. 나누는 것이 나은가 안나누는것이 나은가가 중요하다기보다는 나눴다면 왜 나눴는지 명백한 이유가, 안나눴다면 왜 안나눴는지에 대한 명백한 이유가 있으면 됩니다. 그리고 더 중요한건 동일한 이유에 대해 항상 일관성을 유지하는 것입니다. describe를 어떻게 두는것에 대해서는 사실 답은 없을 것 같아요. 테스트를 분리할 수 있는 기준이 있고 테스트가 중복되지 않는다면 분리하는 것도 좋고, 반대로 파일을 여러개 오픈해서 보는 것이 가독성을 떨어트린다고 판단한다면 한 파일에 몰아둬도 될 것 같아요. 뭐든 설명할 수 있는 이유만 있으면 될 것 같아요~ 어떻게 보면 이것은 취향의 영역일 수도 있고 그때그때마다 판단도 다를 수 있어요. 테스트를 이해하고 파악하는데 현재로 더 좋다고 생각되는 것을 이유와 함게 선택하시면 될 것 같습니다.
