기본과제

목표 : 전역상태관리를 이용한 적절한 분리와 계층에 대한 이해를 통한 FSD 폴더 구조 적용하기

  • 전역상태관리를 사용해서 상태를 분리하고 관리하는 방법에 대한 이해
  • Context API, Jotai, Zustand 등 상태관리 라이브러리 사용하기
  • FSD(Feature-Sliced Design)에 대한 이해
  • FSD를 통한 관심사의 분리에 대한 이해
  • 단일책임과 역할이란 무엇인가?
  • 관심사를 하나만 가지고 있는가?
  • 어디에 무엇을 넣어야 하는가?
**전역상태관리를 사용해서 상태를 분리하고 관리하는 방법에 대한 이해**
- 상태를 도메인별로 분리 (post, comment, user, tag, ui)
- 각 store가 자신의 데이터와 액션만 관리
- 컴포넌트는 필요한 store만 구독하여 불필요한 리렌더링 방지

**FSD(Feature-Sliced Design)에 대한 이해**
- 레이어 기반 아키텍처: app → pages → widgets → features → entities → shared
- 상위 레이어는 하위 레이어만 import 가능 (단방향 의존성)
- 각 레이어는 명확한 책임을 가짐

**FSD를 통한 관심사의 분리에 대한 이해**
- **shared**: 비즈니스 로직 없는 재사용 코드 (UI 컴포넌트, 유틸리티)
- **entities**: 비즈니스 엔티티 (데이터 모델, API, 기본 UI)
- **features**: 사용자 액션 (CRUD 다이얼로그, 검색, 필터)
- **widgets**: 독립적 UI 블록 (entities + features 조합)
- **pages**: 라우팅 진입점, URL ↔ Store 동기화

**단일책임과 역할이란 무엇인가?**
- 하나의 모듈은 하나의 이유로만 변경되어야 함
- `PostRow`는 게시물 행 렌더링만, `usePostStore`는 게시물 상태 관리만
- 변경 시 영향 범위가 명확해짐

**관심사를 하나만 가지고 있는가?**
- `SearchInput`: 검색어 입력 UI + 상태 변경만 (fetch는 widgets에서)
- `PostsTable`: 게시물 목록 표시 + fetch orchestration
- `PostsManagerPage`: URL 파라미터 동기화만 (66줄)

**어디에 무엇을 넣어야 하는가?**
| 코드 종류 | 위치 | 예시 |
|----------|------|------|
| 재사용 UI | shared/ui | Button, Input, Dialog |
| 유틸 함수 | shared/lib | highlightText |
| 데이터 타입 | entities/*/types | Post, Comment |
| API 호출 | entities/*/api | postApi.getPosts() |
| 상태 관리 | entities/*/model | usePostStore |
| 엔티티 UI | entities/*/ui | PostRow, CommentItem |
| 사용자 액션 UI | features/*/ui | AddPostDialog |
| 복합 UI 블록 | widgets/*/ui | PostsTable |
| 페이지 | pages/ | PostsManagerPage |

체크포인트

  • 전역상태관리를 사용해서 상태를 분리하고 관리했나요?
    • Zustand로 post, comment, user, tag, ui store 분리
  • Props Drilling을 최소화했나요?
    • store에서 직접 상태 접근, 필요한 props만 전달
  • shared 공통 컴포넌트를 분리했나요?
    • shared/ui/buttons, inputs, selects, dialogs, cards, tables
  • shared 공통 로직을 분리했나요?
    • shared/lib/highlightText.ts, shared/api/client.ts
  • entities를 중심으로 type을 정의하고 model을 분리했나요?
    • entities/*/types.ts, entities/*/model/store.ts
  • entities를 중심으로 ui를 분리했나요?
    • PostRow, CommentItem, UserAvatar, TagBadge
  • entities를 중심으로 api를 분리했나요?
    • entities/*/api.ts
  • feature를 중심으로 사용자행동(이벤트 처리)를 분리했나요?
    • post-crud, comment-crud, search-posts, user-modal
  • feature를 중심으로 ui를 분리했나요?
    • AddPostDialog, EditPostDialog, SearchInput, TagFilter 등
  • feature를 중심으로 api를 분리했나요?
    • store의 async action으로 API 호출 캡슐화
  • widget을 중심으로 데이터를 재사용가능한 형태로 분리했나요?
    • PostsTable, CommentsSection, Pagination

과제 셀프회고

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

FSD 아키텍처에서 각 레이어의 책임이 명확해야 한다는 것을 배웠습니다.

  • features: 상태만 변경, fetch 호출 X
  • widgets: fetch orchestration 담당, entities + features 조합
  • pages: URL ↔ Store 동기화만 담당

처음에는 features에서 fetch를 호출했는데, widgets와 중복되어 요청 충돌이 발생했습니다. 책임을 widgets로 단일화하니 깔끔해졌습니다.

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

  1. fetch 책임 단일화: useEffect 의존성 충돌 해결을 위해 모드별(검색/태그/기본) useEffect 분리
  2. 코드 스타일 통일: import 방식, 확장자, 세미콜론 등 prettier/eslint 규칙 준수
  3. 점진적 리팩토링: 총 8개 이슈로 나눠서 작은 단위로 커밋, PR 리뷰 반영

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

  • TanStack Query 적용 시 Zustand store와의 역할 분리
  • 낙관적 업데이트 구현 패턴
  • 에러 바운더리와 로딩 상태 처리의 best practice

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

새 프로젝트 시작 시 FSD 구조를 기본으로 잡고, 각 레이어의 책임을 명확히 문서화해두는 방식을 활용해보려 합니다. 특히 "어디에 무엇을 넣어야 하는가"에 대한 팀 컨벤션을 초기에 정하는 것이 중요하다고 느꼈습니다.

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

  • 더티코드를 접했을 때 어떤 기분이었나요? ^^; 클린코드의 중요성, 읽기 좋은 코드란 무엇인지, 유지보수하기 쉬운 코드란 무엇인지에 대한 생각을 공유해주세요

700줄짜리 PostsManagerPage.tsx를 처음 봤을 때 "어디서부터 건드려야 하지?"라는 막막함이 있었습니다. 하지만 작은 단위로 쪼개면서 각 컴포넌트가 하나의 역할만 하게 되니, 수정할 때 영향 범위가 명확해져서 자신감이 생겼습니다.

결합도 낮추기: 디자인 패턴, 순수함수, 컴포넌트 분리, 전역상태 관리

  • 거대한 단일 컴포넌트를 봤을때의 느낌! 처음엔 막막했던 상태관리, 디자인 패턴이라는 말이 어렵게만 느껴졌던 시절, 순수함수로 분리하면서 "아하!"했던 순간, 컴포넌트가 독립적이 되어가는 과정에서의 깨달음을 들려주세요

PostRow, CommentItem에 React.memo를 적용하고 필요한 props만 전달하니, "이 컴포넌트는 이것만 알면 된다"는 명확함이 생겼습니다. 전역 상태를 store로 분리하니 props drilling이 사라지고 각 컴포넌트가 독립적으로 동작할 수 있게 되었습니다.

응집도 높이기: 서버상태관리, 폴더 구조

  • "이 코드는 대체 어디에 둬야 하지?"라고 고민했던 시간, FSD를 적용해보면서의 느낌, 나만의 구조를 만들어가는 과정, TanStack Query로 서버 상태를 분리하면서 느낀 해방감(?)등을 공유해주세요

"CommentsSection은 features인가 widgets인가?" 고민이 있었는데, "독립적인 UI 블록이면 widgets"라는 기준으로 정리하니 명확해졌습니다. 폴더를 보면 코드의 역할이 예측되는 구조가 좋은 구조라는 것을 느꼈습니다.

리뷰 받고 싶은 내용이나 궁금한 것에 대한 질문

  1. widgets에서 fetch orchestration을 담당하는 것이 FSD 원칙에 맞는 방식인지 궁금합니다.
  2. TanStack Query 도입 시 기존 Zustand store를 어떻게 함께 사용하면 좋을지 조언 부탁드립니다.
  3. 이번 주 멘토링 참여 못해서 아쉬워요🥹🥹
