기술적 성장

1. Promise.all의 진정한 병렬 처리 이해

기존 코드를 분석하면서 Promise.all 내부에서 await를 사용하면 병렬 처리가 아닌 순차 처리가 된다는 것을 명확히 이해하게 되었습니다.

// ❌ 잘못된 사용 - 순차 실행 (직렬)
const results = await Promise.all([
  await fetchMajors(),      // 1번이 끝나야
  await fetchLiberalArts(), // 2번 시작
]);

// ✅ 올바른 사용 - 병렬 실행
const results = await Promise.all([
  fetchMajors(),      // 1번과 2번이
  fetchLiberalArts(), // 동시에 시작
]);

이 차이가 실제 네트워크 요청에서 어떤 성능 차이를 만드는지 체감할 수 있었습니다. 각 API가 500ms씩 걸린다고 가정하면, 순차 처리는 1000ms, 병렬 처리는 500ms로 2배의 성능 차이가 발생합니다.

2. 클로저를 활용한 캐싱 패턴 학습

createCachedFetch 함수를 구현하면서 클로저의 실용적인 활용법을 익혔습니다:

const createCachedFetch = <T,>(fetchFn: () => Promise<T>) => {
  let cache: T | null = null;
  let pending: Promise<T> | null = null;

  return async (): Promise<T> => {
    if (cache) return cache;           // 캐시가 있으면 즉시 반환
    if (pending) return pending;       // 진행 중인 요청이 있으면 그 Promise 반환
    pending = fetchFn();
    cache = await pending;
    pending = null;
    return cache;
  };
};

3. useMemo와 useCallback의 적절한 사용 시점 파악

모든 값과 함수에 메모이제이션을 적용하는 것이 아니라, 실제로 비용이 큰 연산이나 자식 컴포넌트에 전달되는 참조에만 적용해야 한다는 것을 배웠습니다:

// ✅ 적절한 사용 - 비용이 큰 필터링 연산
const filteredLectures = useMemo(() => {
  return lectures
    .filter(/* 복잡한 조건 1 */)
    .filter(/* 복잡한 조건 2 */)
    .filter(/* 복잡한 조건 3 */);
}, [lectures, searchOptions]);

// ✅ 적절한 사용 - memo된 자식에게 전달되는 콜백
const handleAdd = useCallback((lecture: Lecture) => {
  // ...
}, [dependencies]);

4. React.memo와 컴포넌트 분리 전략

LectureRow 컴포넌트를 분리하고 memo로 감싸면서, 리스트 렌더링 최적화의 핵심을 이해했습니다:

const LectureRow = memo(({ lecture, onAdd }: 
LectureRowProps) => (
  <Tr>
    {/* 행 렌더링 */}
  </Tr>
));

500개 이상의 강의 목록에서 검색 옵션이 변경될 때, 변경되지 않은 행들은 리렌더링되지 않아 성능이 크게 향상되었습니다.

4. Context 구독 최적화 - 선택적 구독 패턴

전체 schedulesMap을 구독하면 어떤 시간표가 변경되어도 모든 시간표가 리렌더링되는 문제가 있었습니다. 이를 해결하기 위해 useSchedule 훅을 만들어 개별 시간표만 구독하도록 했습니다:

export const useSchedule = (tableId: string) => {
  const { schedulesMap, setSchedulesMap } = useScheduleContext();
  // 해당 tableId의 schedules만 메모이제이션
  const schedules = useMemo(() => schedulesMap[tableId] || [], [schedulesMap, tableId]);

  // 업데이트 함수도 메모이제이션
  const updateSchedules = useCallback(
    (newSchedules: Schedule[] | ((prev: Schedule[]) => Schedule[])) => {
      setSchedulesMap(prev => ({
        ...prev,
        [tableId]: typeof newSchedules === 'function'
          ? newSchedules(prev[tableId] || [])
          : newSchedules
      }));
    },
    [tableId, setSchedulesMap]
  );

  return { schedules, updateSchedules };
};

또한 useScheduleIds 훅을 분리하여 시간표 ID 목록만 필요한 컴포넌트는 개별 시간표 데이터 변경에 반응하지 않도록 했습니다.

학습 효과 분석

"무엇이 리렌더를 유발하는가"를 추적하는 능력

React 최적화의 핵심은 불필요한 리렌더링을 찾아내는 것이라 생각합니다. 이번 과제를 통해 다음을 체계적으로 분석하는 방법을 배웠습니다:

  1. State 변경: 어떤 state가 어디서 변경되는가?
  2. Props 변경: 부모가 리렌더링될 때 자식에게 전달되는 props가 변경되는가?
  3. Context 변경: Context 값이 변경될 때 어떤 컴포넌트들이 구독하고 있는가?
  4. 참조 동일성: 객체/배열/함수가 매번 새로 생성되는가?

과제 피드백

  1. 최적화 "완료" 기준의 모호함: "렌더링 최적화"가 어느 수준까지 해야 완료인지 판단하기 어려웠습니다. 예를 들어:
  • 리렌더링 횟수를 몇 회 이하로?
  • 특정 시나리오에서 프레임 드롭이 없으면?
  • React Profiler에서 특정 수치 이하면?

리뷰 받고 싶은 내용

  1. Context 선택적 구독 패턴의 한계 현재 useSchedule 훅은 schedulesMap[tableId]를 useMemo로 감싸서 해당 시간표만 구독하도록 했습니다:
const schedules = useMemo(
  () => schedulesMap[tableId] || [],
  [schedulesMap, tableId]
);

하지만 이 방식은 schedulesMap 객체 자체가 변경되면 useMemo가 재실행됩니다. 다른 시간표가 변경되어도 schedulesMap 참조가 바뀌므로 모든 useSchedule 훅이 재실행되는 것 아닌가요? 실제로는 useMemo의 반환값이 이전과 동일하면 리렌더링이 발생하지 않는 건지, 아니면 Zustand의 useStore(state => state.schedules[tableId])처럼 selector 패턴을 사용해야 진정한 선택적 구독이 되는 건지 궁금합니다!

  1. DraggableSchedule의 memo 효과 DraggableSchedule 컴포넌트에 memo를 적용했는데, useDraggable 훅이 내부에서 useDndContext를 사용하므로 드래그 상태가 변경될 때마다 리렌더링이 발생할 것 같습니다:
const DraggableSchedule = memo(({ id, data, bg, onDeleteButtonClick }) => {
  const { attributes, setNodeRef, listeners, transform } = useDraggable({ id });
  // ...
});

이 경우 memo가 실질적인 효과가 있는 건지, 아니면 useDraggable을 사용하는 이상 어쩔 수 없는 건지 궁금합니다!

과제 피드백

Q. Context 선택적 구독 패턴의 한계

A. 넵 말씀하신대로 useSchedule 훅은 다시 실행됩니다. 다만 리렌더링이 일어날지 말지는 사실 결과물의 참조에요.

schedulesMap의 참조가 바뀌었더라도 tableId를 통해 셀렉트된 schedules의 참조는 안바뀔 수 있기 때문에 꼭 리랜더링을 유발하는 것은 아닙니다. useSchedule이 아무리 많이 실행되도 리턴해주는 값의 참조가 바뀌지 않는다면 해당 값을 사용하는 컴포넌트가 리렌더링되진 않을테니까요. 물론 컴포넌트도 메모이제이셔을 해야겠지만요.

Q. DraggableSchedule의 memo 효과

A. 네 이렇게 자주 업데이트되는 값의 경우 메모이제이션 효과가 제한적일 수밖에 없어요. 근본적으로 자주 바뀌면 자주 바뀔수록 메모효과는 떨어질 수 밖에 없습니다.

그래서 컴포넌트 리랜더 구조를 Context에 의존하기 보다. 비록 조금 선언적이지 않더라도 별도의 이벤트를 받아서 처리하는 것이 더 효율적일 수 있습니다. 드래그발생 -> 리랜더 보다는 드래그발생 -> 드래그 이벤트 발생 -> 선택적 업데이트 이렇게 말이죠. 움직임이 많은 UI는 리액트의 렌더링 플로우와 분리해서 구현하는 것이 좋을 것 같습니다.
