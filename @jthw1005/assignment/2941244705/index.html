과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

각 테스트가 병렬로 돌아도 안정적으로 동작되게끔, 테스트 별 이벤트 데이터들을 각각의 클로저 내부로 격리시켰습니다.

export const setupMockHandlerCreation = (initEvents = [] as Event[]) => {
  const events = [...initEvents];

  server.use(
    http.get('/api/events', () => {
      return HttpResponse.json({ events: [...events] });
    }),

    http.post('/api/events', async ({ request }) => {
      const newEvent = (await request.json()) as Event;
      newEvent.id = String(events.length + 1);
      events.push(newEvent);
      return HttpResponse.json(newEvent, { status: 201 });
    })
  );
};

setupMockHandlerCreation의 내부에 선언되어 있는 events 배열은 msw 핸들러에서 참조되기 때문에 함수 실행이 종료되어도 클로저가 형성되어 계속해서 참조할 수 있고, 이를 활용해 테스트가 병렬로 돌아도 각각의 클로저 구조로 격리가 되게끔 유지했습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

  • 위에서 언급드린 setupMockHandlerCreation을 이용하여 테스트 간 데이터를 격리시켰습니다.
  • beforeEach, afterEach, vi.setSystemTime을 활용하여 테스트에 필요한 초기 설정을 하고 테스트가 종료되면 설정들을 롤백시켰습니다.

심화 과제

  • App 컴포넌트 적절한 단위의 컴포넌트, 훅, 유틸 함수로 분리했는가?
  • 해당 모듈들에 대한 적절한 테스트를 2개 이상 작성했는가?

과제 셀프회고

항해 1주차를 시작하는 주간에 회사 업무가 갑자기 바빠져서 '테스트 코드'에 대한 주제를 잘 소화시키며 과제를 하는 것이 아니라, 과제를 푸는 데에만 급급했습니다ㅠ

그래도 본 교육 시작 전, 사전 스터디를 했던 효과가 있었는지 발제 영상과 자료를 참고하여 과제를 풀어보니 '테스트 코드'에 대해 어느 정도의 이해력이 생겨 뿌듯합니다! 무엇보다 머릿속에 정리가 안 되어 있던 개념들이 과제를 풀면서 하나 둘 씩 이해되는 과정이 재밌었습니다.

평소 테스트 코드에 관심이 있던터라 예전부터 책도 읽어보고 강의도 찾아 봐왔는데, 막상 실제 업무에서 사용하기에는 우선 순위에서 밀리고 도메인 특성 상 어디서부터 시작해야 할지 막막했었습니다. 이번 주차를 통해 테스트 코드를 잘 도입할 수 있을 것 같은 자신감이 들었습니다!

기술적 성장

  • 테스트 코드에 대한 전반적인 이해를 갖는다.
  • 테스트 코드를 어떻게 관리 및 운영하는지에 대해 생각해 볼 수 있다.
  • MSW를 실제 API 모킹이 뿐만 아니라 테스트 코드에서도 활용할 수 있다.
  • 테스트 별로 독립적인 데이터 환경을 구축할 수 있다.
  • Mock Data를 어떻게 관리해야할지에 대한 이해도가 생겼다.
  • RTL을 이용하여 원하는 HTML 요소를 찾을 수 있다.
  • data-testid 속성을 활용할 수 있다.

코드 품질

구현에만 급급하여 코드 품질에 대해 크게 신경쓰지 못했습니다. 아래 사항들을 개선할 수 있을거 같습니다.

  • handlers와 handlersUtils 간 중복되는 코드를 분리한다.
  • 이벤트(일정) 객체를 생성하는 함수를 만들어 가독성을 높힌다.
  • 테스트 코드 description을 더 명확히 작성한다.

과제 피드백

  • '적절히 처리한다.' 라는 구체적이지 않은 부분이 오히려 '테스트 코드'와 '기능 명세서' 등의 관계에 대해 깊이 생각해보게 되는 기회가 되어서 좋았습니다.
  • 저의 경우 유틸 -> 훅 -> 통합 테스트의 순서로 과제를 구현했는데, 처음에 비교적 간단하고 쉬운 테스트 코드를 작성하며 테스트 코드에 익숙해지고 점점 어려운 난이도로 올라가는게 재미있게 느껴졌습니다.
  • package.json에 eslint-plugin-cypress가 없는데 eslint.config.js에서 이를 참조하고 있어 오류가 발생하고 있습니다!

리뷰 받고 싶은 내용

  • HTML 요소에 접근할 수 있는 방법이 다양하다고 느껴졌는데, 적절히 잘 사용했는지 궁금합니다.

궁금한 점

  • server.js에 있는 API 메서드들을 똑같이 참고하여 msw handler를 작성하는 것을 의도하신 걸까요?
    • 예컨대, server.js의 delete 메서드에는 id가 HIT인 경우만 있고 MISS인 경우에 대한 예외 처리는 없는데, msw handler에서도 해당 예외 처리가 없어야 하는건지 궁금합니다.
  • 저는 현재 '인플루언서 마케팅' 도메인에서 일하고 있습니다. 광고 캠페인에 대한 경우의 수가 굉장히 많은 데요,(ex. 아래 데이터에 대한 각각의 케이스만 4 x 3 x 2 = 24가지 케이스가 존재)
플랫폼 = 인스타그램 | 블로그 | 유튜브 | 쇼핑몰리뷰
상품 = 제품형 | 방문형 | 서비스형
광고 진행 방식 = 인플루언서 모집하기 | 인플루언서 추천받기

또한, 각각의 경우에 따라 정책도 조금씩 달라서 대표로 몇 개의 케이스만 골라 테스트 코드를 작성하기 난해한 거 같습니다. 이러한 경우 어떤 방향으로 테스트 코드를 운영하면 좋을지 조언을 얻을 수 있을까요?

  • 테스트 코드를 읽고 이해할 수는 있는데, 내가 직접 짤 수 있을까?

과제 제출 후 생각

  • 간단한 유틸 함수 테스트로 시작해서 커스텀 훅, 컴포넌트로 늘려나가면 되겠다.
  • 테스트 코드를 어느정도 직접 짤 수 있고 잊어버리지 않게 틈틈이 복습해야겠다.

과제 피드백

수고 많았어요 준태! 이번 과제는 테스트를 하는 다양한 방법들을 접하면서 테스트 코드를 작성한다는 그 심리적 장벽을 허물고 친숙해지는데 있었습니다. Hard까지 완주하시고 회고도 정말 깊이 있게 작성해주셨네요. 잘했습니다. 합격입니다!

"항해 1주차를 시작하는 주간에 회사 업무가 갑자기 바빠져서 과제를 푸는 데에만 급급했다"고 하셨는데, 그 바쁜 와중에도 Hard까지 완주한 게 정말 대단합니다. 특히 "머릿속에 정리가 안 되어 있던 개념들이 과제를 풀면서 하나 둘 씩 이해되는 과정이 재밌었다"는 표현이 너무 좋아요. 사전 스터디를 해온 준비성도 훌륭하고요. "간단한 유틸 함수 테스트로 시작해서 커스텀 훅, 컴포넌트로 늘려나가면 되겠다"는 인사이트도 정확합니다. 이게 바로 테스트 코드 도입의 시작점이에요.

"평소 테스트 코드에 관심이 있어서 예전부터 책도 읽어보고 강의도 찾아 봐왔는데, 막상 실제 업무에서 사용하기에는 우선 순위에서 밀리고 도메인 특성 상 어디서부터 시작해야 할지 막막했다"고 하셨는데, 많은 개발자들이 똑같은 고민을 해요. 그런데 이번 주차를 통해 자신감이 들었다니 정말 뿌듯하네요. 이제 실무에 적용해보면서 효용체감을 느껴보길 바랍니다.

질문 주신 부분들 답변드릴게요.

Q) 시간과 관련된 테스트 유틸 함수들을 적절히 잘 사용했는지, 더 좋은 방향이 있는지 궁금합니다! HTML 요소에 접근할 수 있는 방법이 다양하다고 느껴졌는데, 적절히 잘 사용했는지 궁금합니다. 이벤트(일정) 데이터를 관리하기 좋은 디자인 패턴이 있을까요?

=> 네. 코드 확인해 봤는데 시간 관련 유틸 함수들 잘 사용하셨습니다. vi.setSystemTime, beforeEach/afterEach 조합이 정석이에요. 테스트가 많아지면 setupTests.ts에 타이머 설정을 전역으로 두는 것도 방법입니다.

=> HTML 요소 접근은 data-testid 잘 활용하셨네요. 나중에 getByRole, getByLabelText 같은 쿼리들도 살펴보시면 좋아요. 사용자 관점에서 요소를 찾는 방식이라 접근성 테스트에도 도움됩니다.

=> 이벤트 데이터 관리는 클로저로 격리한 방식 훌륭합니다. "이벤트 객체를 생성하는 함수"라고 개선안에 적으신 것처럼 createMockEvent({ title: '회의' }) 형태의 Factory 함수를 만들면 기본값은 다 채워두고 필요한 속성만 덮어쓸 수 있어서 편해요.

=> 리뷰를 원하는 것에 비해서 대답이 피상적인데, 질문이 추상적이다보니 답변의 추상화 수준도 비슷할 수 밖에 없네요ㅠㅠ 다음에 조금 더 구체적으로 물어보면 더 구체적으로 리뷰를 해볼게요.

Q) server.js에 있는 API 메서드들을 똑같이 참고하여 msw handler를 작성하는 것을 의도하신 걸까요? * 예컨대, server.js의 delete 메서드에는 id가 HIT인 경우만 있고 MISS인 경우에 대한 예외 처리는 없는데, msw handler에서도 해당 예외 처리가 없어야 하는건지 궁금합니다.

=> MSW는 실제 서버에서 테스트하기 어려운 동작들을 mocking을 테스트를 하기 위한 용도입니다. 테스트를 위해서 삭제를 반드시 해야만 한다면 테스트하기가 어려울테니 HIT인 상황을 만드는 것이죠. 반대의 경우 삭제 실패의 경우 에러메시지를 테스트해야 하는데 서버에서 실패를 하도록 만들기는 어려운 상황이니 이 테스트를 위해서는 MSW에서 MISS를 만들어야겠죠. 서버의 의존도 없이 테스트를 하기 위한 도구라는 측면에서 예외처리가 아니라 무엇이 필요한지를 생각하면 어떤 핸들러를 만들면 좋을지 도움이 될거레요.

Q5. 인플루언서 마케팅 도메인 테스트 방향

=> 테스트 커버리지가 100%이면 좋겠지만, 그래서 사실 처음부터 테스트를 만들어가면 되겠지만 현실은 대부분 그러지 못하고 (심지어 대기업 현업에서도 그렇게 하는 팀이 소수입니다 ㅠ) 한번에 대청소를 하려고 하면 힘들죠. 일단 테스트가 필요하다고 느껴진다면 앞으로 새로 만드는 것은 항상 테스트를 만든다는 식으로 컨벤션을 가져가야겠지요.

=> 그리고 모든 테스트의 용도 측면에서 우선순위를 조정해볼 수 있습니다. 리팩토링시 장애감지등을 원한다면 최초 랜딩이 제대로 되는지를 먼저 확인해볼 수 있고, 매번 복잡한 시나리오라 항상 자주 버그가 나는 곳이 있다면 해당 구간은 테스트를 만들어 두면 좋습니다. 사람이 재현하기에 복잡한 상황이나 테스트 경로가 깊은 곳의 디버깅이 어렵다면 그 부분을 테스트로 먼저 만들어 보는 것도 좋습니다.

=> 진짜 어렵기는 한테 디버깅을 하다가 막혔을때 테스트를 고민해보면 좋습니다. 물론 이때는 그 생각을 잘 못하게 되고 버그를 고치고 나면 일단 되니까 다음에 하지 뭐 이렇게 되는게 보편적이기는 하나 자주 버그가 나는 곳에는 테스트가 필요하다는 의미죠.

정말 잘했습니다. "간단한 유틸 함수 테스트로 시작해서 커스텀 훅, 컴포넌트로 늘려나가면 되겠다"고 하셨는데, 이제 실무에서도 그렇게 시작해보세요. 인플루언서 마케팅 도메인에서 24가지 케이스를 한 번에 다 하려고 하지 말고, 앞으로 새로 만드는 기능부터 테스트를 작성하거나 자주 버그가 나는 부분 하나부터 시작하면 됩니다. 이번 과제로 자신감이 생겼다고 하셨으니 충분히 할 수 있어요. 다음 과제도 화이팅입니다!
