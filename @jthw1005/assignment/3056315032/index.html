과제에서 꼭 알아가길 바라는 점

  • 엔티티를 다루는 상태와 그렇지 않은 상태 - cart, isCartFull vs isShowPopup
  • 엔티티를 다루는 컴포넌트와 훅 - CartItemView, useCart(), useProduct()
  • 엔티티를 다루지 않는 컴포넌트와 훅 - Button, useRoute, useEvent 등
  • 엔티티를 다루는 함수와 그렇지 않은 함수 - calculateCartTotal(cart) vs capaitalize(str)

과제 셀프회고

배포 링크

Basic Advanced

과제를 받고 나서 모든 코드가 App.tsx에 전부 작성되어 있어서 적잖이 당황했다. 그래도, 현업에서 다루고 있는 도메인이 나름 복잡하기도 하고(뇌피셜^^) 리팩토링에도 자신이 있었어서 금방 하겠지 생각했는데,

예컨대 cart나 coupon과 같이 생각보다 도메인이 서로 얽혀있는 경우가 있어서 처음에 이러한 것들을 특정 컴포넌트의 내부에 구현했다가 완전히 다른 곳에 위치한 컴포넌트에서도 필요한 경우가 생겨 상위의 부모 컴포넌트로 다시 옮기는 등 여러 뻘짓을 했다.

그러다 화요일 평큐를 봤는데, 5팀 채영님께서 피그잼으로 멋있게 다양한 케이스에 대해 설계를 하신 것을 보고 자극을 받아, 나도 막 구현하다가 얻어 걸려서 과제 통과하지 말고 처음부터 설계를 제대로 해보자 싶어서 A4 용지에 현재 과제에서 주어진 웹 서비스에 대해 어떤 데이터가 있고, 그 데이터의 흐름은 어떻게 흘러가며 누가 해당 데이터를 사용하는지 막 적어보았다.

처음엔 바로 전역 상태 라이브러리를 설치해서 구현을 시작하려 했으나, 발제 자료에 있는 과제 설명에서 기본 과제는 전역 상태 라이브러리 없이 온전히 props drilling을 느껴보라고 나와 있어서, Basic과 Advanced 두 가지의 도식화를 그려보았다.

여러 가지 케이스 중에 구현 방향으로 정한 것만 Mermaid로 정리해보았다.

Basic

  1. 전체 컴포넌트 트리 구조

  2. Props Drilling 흐름

Advanced

  1. 전체 컴포넌트 트리 구조

  2. Zustand 스토어 구조

Basic -> Advanced

  1. Props 변화 비교

  2. 데이터 흐름 비교

과제를 하면서 내가 알게된 점, 좋았던 점은 무엇인가요?

  • 도메인을 기준으로 폴더 구조와 컴포넌트의 흐름을 고민해보는 것.
  • 실무였다면 고민 없이 전역 상태 라이브러리를 썼을 텐데, 과제라는 틀 내에서 props drilling을 겪으며 어떠한 문제점이 있고 어떻게 해결할 수 있을지에 대해 고민하게 된 것.

이번 과제에서 내가 제일 신경 쓴 부분은 무엇인가요?

어떤 것들을 props로 내려주고 어떤 것들을 전역 스토어로 구분할 것인지에 대한 고민

  1. 전역 상태로 옮긴 것
  • 여러 컴포넌트에서 공유되는 상태 (cart, products, coupons)
  • 앱 전체에서 접근해야 하는 상태 (isAdmin, notifications)
  • 콜백 함수들 (addToCart, removeFromCart 등)
  1. Props로 유지한 것
  • 부모가 자식에게 전달하는 UI 관련 데이터 (searchTerm → CartPage)
  • 컴포넌트의 렌더링에 직접적으로 필요한 props (Cart 컴포넌트의 cart, totals 등)
  • props를 받았을 때 재사용 가능한 컴포넌트인지 (ProductCard의 product, formatPrice)

Q) Header처럼 앱 내 특정 상태에 대해 접근하는 경우가 빈번한 컴포넌트는 Props 주입과 전역 상태 직접 접근 중 어떤 것이 더 적절한가요? 만약 상황에 따라 다르다면 그 기준이 무엇인지 궁금합니다.

A) 저는 전역상태가 아니라 컨텍스트를 통해 관리하면 어떨까? 라는 생각입니다 ㅎㅎ

"전역상태관리"의 핵심은 "전역"이 아니라 "상태관리" 라고 생각해요. 여기서 이야기하는 상태관리는 "도메인 데이터 자체와 도메인을 다루는 로직"의 뭉치라고 생각합니다. 도메인 데이터는 영역과 상관없이 어디에서든 쓰일 수 있어요. 그렇기 때문에 "전역"으로 관리할 필요가 있는거죠. 그리고 모든 내용이 다 쓰이는게 아니라 일부 내용만 쓰일 수도 있습니다. 그래서 selector 같은걸 이용하여 일부 데이터를 가져오고, 이에 대한 렌더링만 유발되도록 만드는거죠.

Context는 도메인과는 조금 다르다고 생각합니다. 특정 스코프에 필요한 데이터를 다룰 때 유용한 도구인거죠. 꼭 전역일 필요도 없어요. 그래서 "어플리케이션의 설정" 과 같은 것들은 Context를 통해 관리해야 역할에 조금 더 적합하지 않나!? 라는 생각입니다 ㅎㅎ 대체로 UI와 관련된 상태들이 많아요.

Q) 저는 현재 인플루언서 마케팅 도메인의 회사에서 웹 서비스를 개발 및 운영 중입니다. 유저(광고주)가 광고 캠페인을 오픈하려면 다양한 정보의 Form을 작성해야하는데요. 이 때, 특정 form에서 어떠한 값을 선택하는지에 따라 입력 받는 form의 UI와 데이터가 조금씩 달라집니다. 예컨대, platform으로 인스타그램을 선택하는 경우 팔로워수와 좋아요수를 받아야 하고, 유튜브를 선택하는 경우에는 구독자 수를 받아야 합니다. 위와 같은 상황에서 중복 코드를 감안하더라도 InstagramCampaignForm, YoutubeCampaignForm, BlogCampaignForm 등으로 분리하는게 좋을까요? 아니면 하나의 CampaignForm으로 관리하고 캠페인 내부에서 분기 처리를 통해 관리하는게 좋을까요? 처음에는 후자로 구현했다가 기획도 자주 바뀌고 로직도 복잡해져서 전자로 리팩토링했는데, 관리해야하는 컴포넌트가 많아지다보니 다시 전자로 돌아가고 싶은 고민이 듭니다ㅠ

A) Form에서 어떤 데이터를 다루냐가 중요할 것 같은데요, 이럴 때 사용하면 좋은게 Compound Component 라고 생각해요 ㅎㅎ CampaignForm의 공통 부분을 재활용할 수 있는 요소로 분리하고 조금씩 조금씩 다른 부분을 InstagramCampaignForm, YoutubeCampaignForm, BlogCampaignForm 컴포넌트 하위에서 정의하는거죠.

근데 코드 없이 상황을 통해서만 이야기 하는건 사실 큰 효과가 없는 것 같네요.. 복잡도가 높다면 분리하는게 맞고 복잡도함 속에서 추상화를 찾을 수 있다면 공통로직으로 분리해야 하는데 이 추상화 로직이 복잡하다면 사실상 분리하는 것만 못한 일이 발생하니까요.
