과제 셀프회고

딱 하루 일정이 있어서 못 했는데 과제 제출 기간이 생각보다 너무 촉박 했다. 내가 코드를 작성하고 공부하는 부분에 있어서 너무 느린게 아닌가 실감했다. 누구는 출퇴근 하며 해도 미리 끝내는 모습을 보면 스스로와 수준 차이가 느껴졌다. => 그렇다면 두배로 더 공부하는게 맞다.

내가 고민하고 거슬렸던 부분은 남들의 설명과 코드를 보니 삽질 하고 있던 것들이 많았다. 고물상 처럼 잡다한 지식으로 풀려고 하니 이상한 계열로 가버려서 다시 돌아오는 시간이 걸리는거 같다. => 좀더 확실하게 정보를 알고 진행 하는 버릇을 들이자.

그나마 사전 스터디로 어느정도 테스트 코드의 개념과 실습이 있었기에 망정이지 없었으면 하루종일 질문만 하며 다니고 과제도 끝까지 못할뻔 했다. => 준비의 중요함을 깨닫고 토요일 되기 전이라도 강의와 리팩토링 책을 달고 살아야겠다.

그래도 과제 끝까지 해서 다행이다. => 다음에는 과제 Hard 잡고 첫날부터 밤새 하고싶다.

코드 품질

--- 결론 적으론 많이 아쉬운 품질

  • 시간에 쫒기다 보니 리팩토링 안한 부분이 눈에 밟힌다.
  • 리팩토링 하려고 한 부분만 5곳이 넘는데 해보지 못해 아쉽다.
  • 그나마 초반인 unit 폴더 부분은 다른 부분보다 가독성 있게 작성되어 있다.

--- 리팩토링이 필요한 부분

  • events 객체 생성 부분
  • 날짜 계산 helper 함수
  • 테스트용 네트워크 에러 부분
  • 테스트 코드 반복되는 훅 선언 부분
  • App.tsx 컴포넌트 화

학습 효과 분석

  • 테스트 코드는 최적화가 아니라 가시적이게 잘 정리하는 것도 중요하다는 것

  • 전체적인 페이지 이해도

--- 실무 적용 가능성

  • 매우 높음

과제 피드백

  • 객체 이름이 event 라 DOM 이벤트랑 햇갈렸다.

--- 과제에서 좋았던 부분

리뷰 받고 싶은 내용

  • unit 부분에서 테스트 코드가 코치님께서 보시기에 가시적인지 확인 해주시면 좋겠습니다.
  • 최대한 가시적이게 변수 명을 지어봤지만 함수나 비교부분(expect) 에 직접 값을 써 넣는게 더 좋은지, 변수로 지금처럼 빼서 변수명 잘 지어주는게 좋은지 조언 부탁드립니다.
  • 테스트 helper 함수 만들때 함수니까 효율적이지만 복잡하게 만들어도 되는지, 아니면 그것도 확인 해야 하기 때문에 단순화 해야 하는지 알고 싶습니다.

과제 피드백

준모님 고생하셨습니다!

다음에는 과제 Hard 잡고 첫날부터 밤새 하고싶다. 너무 무서운 말인데요 ㅋㅋㅋㅋㅋ 처음부터 너무 무리하지 마세요 ㅎㅎ

우선 이번주에는 테스트 자체에 익숙해지는거에 목표가 있었기 때문에 작성해주신 코드 보니까 충분히 잘해주신것 같아요 ㅎㅎ msw 사용하는데에도 어려움이 없으셨던것 같고, 테스트를 작성하는 논리도 잘 정립하고 계신 것 같습니다 ㅎㅎ 병렬적이게 테스트를 구동하게 할 때에도 잘 고민해주신 것 같구요.

unit 부분에서 테스트 코드가 코치님께서 보시기에 가시적인지 확인 해주시면 좋겠습니다.

말씀해주신게 utils에 대한 테스트를 말씀해주시는거라면, 잘 작성해주셨어요!

최대한 가시적이게 변수 명을 지어봤지만 함수나 비교부분(expect) 에 직접 값을 써 넣는게 더 좋은지, 변수로 지금처럼 빼서 변수명 잘 지어주는게 좋은지 조언 부탁드립니다.

잘 작성해주셨지만 ㅎㅎ 제 개인적인 취향으로는 여러번 선언되지만, 굳이 여러번 선언할 필요가 없는 변수들도 존재하는 것 같아요! 전체를 관통하는 테스트 케이스를 유지하면서, 필요할때만 이벤트를 재선언하는 형태로 하면 더 깔끔할것 같아요. 이벤트도 종종 여러개를 넣어 처리하는게 있는데, 필요한 만큼만 같은 그룹의 값이라면 사용하지 않는 형태로 처리하면 좋지 않을가!

테스트 helper 함수 만들때 함수니까 효율적이지만 복잡하게 만들어도 되는지, 아니면 그것도 확인 해야 하기 때문에 단순화 해야 하는지 알고 싶습니다.

요거도 결국 테스트를 위한 테스트가 필요해질수 있으니 최대한 단순하게 목적에 맞게 가독성을 올려주는 형태로만 운영하는게 저는 좋다고 생각하는 편입니다.

GPT 없이 개발 지식 수급처가 궁금합니다.
