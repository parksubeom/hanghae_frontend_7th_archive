[3팀 김준모] Chapter 1-3. 완성도있는 테스트 전략 수립하기

by jumoooo2025.11.03

과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.

  • 참고 이미지 (PlayWright)

  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.

  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리
  • 참고 이미지 (StroyBook + Chromatic)

심화 과제

  • 테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 팀원과 회의했던 정보들 노션 링크입니다. 논의했던 내용, 서로 작업하였던 코드들이 작성 되어 있습니다. https://www.notion.so/teamsparta/3-2-2a22dc3ef5148036bea9e54ad80bb4e4#2a22dc3ef51480a09870e3bf442a8764
  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

테스트 전략을 세우기 전, 먼저 현재 진행 중인 프로젝트의 구조를 명확히 이해하는 것이 중요하다고 생각했습니다. 테스트 코드는 단순히 효율성을 위한 것이 아니라, 유지보수가 쉽고 장기적으로 안정적인 구조를 만드는 것이 핵심이라고 봤습니다. 이를 기반으로 다음과 같이 분석했습니다.

  • App.tsx 내부에 컴포넌트로 분리될 부분이 많음
  • 컴포넌트 간 상호작용 존재 여부 확인 필요
  • 상호작용이 많을 경우 단위 테스트(Unit Test)만으로는 충분한 신뢰성을 확보하기 어려움
  • 통합 테스트(Integration Test)를 통해 컴포넌트 간 연계를 검증하는 것이 프로젝트 특성상 더 적합
  • 결과적으로 테스트 피라미드(Pyramid) 보다는 테스트 트로피(Trophy) 구조가 프로젝트에 더 어울림

물론 프로젝트 규모가 크지 않기 때문에, 단위 테스트 중심의 피라미드 전략도 고려할 수 있었습니다. 하지만 추후 컴포넌트가 세분화되고 상호작용이 많아질 가능성을 생각했을 때, 단위 테스트만으로는 유지보수에 한계가 있을 것이라 판단했습니다. 따라서 트로피형 테스트 전략을 선택했습니다.

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

팀 내 논의 결과, 트로피형 테스트 전략을 적용하기로 합의했습니다. 하지만 실제로 적용하는 과정에서 “어디까지를 통합 테스트로 볼 것인가?”라는 기준이 명확하지 않았습니다. 워크플로우 단위로 나눌지, 컴포넌트 단위로 나눌지 고민하다가 컴포넌트 단위의 통합 테스트를 1차 기준으로 정했습니다.

이 방식은 App.tsx를 기능별 컴포넌트로 분리하고, 각 컴포넌트를 렌더링해서 테스트하는 형태입니다. 상위 컴포넌트와 하위 API, Mock 데이터를 연동해 테스트 파일을 구성함으로써 상·하위 간의 연계 동작을 함께 검증할 수 있습니다.

하지만 이 접근에는 두 가지 문제점이 있었습니다.

  • 모든 컴포넌트를 나누는 작업에 시간이 걸림
  • 상태 관리 로직을 분리했을 때, 예상 결과(expected value)를 일관성 있게 관리하기 어려움

이러한 이유로 팀원들이 부담을 느낄 것이라는 생각에 늦은 밤까지 남아 있는 사람들끼리 회의를 통해 더 현실적인 대안을 고민했습니다. 최종적으로, 비즈니스 도메인(기능 영역) 단위로 통합 테스트를 작성하는 방식으로 결정했습니다. 이 접근 방식은 다음과 같은 장점이 있습니다.

  • 기능 단위로 테스트 범위를 명확히 정의할 수 있음
  • 유지보수 용이성 향상
  • 컴포넌트를 전부 분리하지 않아도 됨
  • 팀원 간 테스트 분담이 명확

최종적으로 도메인별 테스트 파일을 2인 1조로 나누어 작성하기로 결정했습니다.

  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

프로젝트 내 부족한 통합 테스트 영역을 검토한 결과, 다음과 같은 테스트 항목을 우선순위별로 정리했습니다.

  1. 검색 및 필터링 통합 테스트
  2. 알림 시스템 통합 테스트
  3. 드래그 앤 드롭 통합 테스트
  4. 날짜 클릭 일정 생성 통합 테스트
  5. 검색과 뷰 전환 조합 테스트
  6. 일정 겹침 처리 테스트

각자 담당할 테스트 파일을 나누고, 일부는 추가 작성 후 페어 코딩으로 통합했습니다. 최종적으로 추가된 주요 통합 테스트 항목은 다음과 같습니다.

  • 일반 일정을 다른 날짜로 드래그하면 날짜가 정상적으로 변경된다.
  • 날짜를 클릭하여 일정을 생성하고, 다른 월의 날짜를 클릭하면 일정 날짜가 수정된다.
  • 주간 뷰에서 다음 달로 넘어가는 주의 일정이 올바르게 표시된다.
  • 2025년 12월에서 2026년 1월로 넘어갈 때 일정이 정확히 필터링된다.
  • 네비게이션 버튼 클릭 시 일정 필터링과 헤더 표시가 올바르게 동작한다.

과제 셀프회고

이번 과제에서는 테스트 전략을 수립하고 수정하는 과정에서 팀 내 논의가 가장 많은 시간을 차지했습니다. 약 3일간 코어 타임 동안 하루 4시간 이상을 회의에 투자했는데, 그만큼 테스트 구조와 방향성에 대한 고민이 깊었습니다.

특히 통합 테스트를 처음에는 “여러 기능을 묶어서 검증하는 테스트” 정도로 단순하게 생각했지만, 막상 실제 구현 단계에서 **“어떤 기준으로 파일을 나누고, 어떤 범위를 통합 테스트로 볼 것인가”**를 정하는 일이 예상보다 훨씬 어렵다는 걸 느꼈습니다. 이 과정을 통해 단순한 테스트 작성이 아니라, 테스트 아키텍처를 설계하는 사고방식을 배울 수 있었습니다.

테스트 코드 3주간의 회고

이번 3주간의 테스트 코드 학습은 단순히 “테스트를 배우는 과정”을 넘어, AI와 함께 개발하는 환경을 이해해 가는 과정이었다고 느꼈습니다. 처음에는 테스트 코드 과목이라 해서 검증 절차나 코드 품질 향상에만 초점을 맞출 줄 알았지만, 되돌아보면 AI와 협업하기 위한 개발 기반을 다지는 과정이었던 것 같습니다.

AI Agent를 본격적으로 다룬 것은 2주차부터였지만, 그 이전 1주차에 테스트 코드의 개념과 구조를 익히는 과정이 결과적으로 AI 개발 워크플로우의 매뉴얼을 설계하는 데 중요한 역할을 했습니다. 특히 3주차에는 Storybook과 Chromatic을 연동하고, E2E 및 통합 테스트를 진행하면서 시각적으로 결과물을 확인했는데, 이 경험을 통해 AI가 생성한 결과물을 검증하고 신뢰할 수 있는 체계를 만드는 것이 테스트의 본질이라는 점을 깨달았습니다.

제 생각은 이번 과정을 이렇게도 정리할 수 있을 것 같습니다:

  • 1주차: AI가 코드 작성 시 참고할 수 있는 명확한 규칙과 테스트 구조를 학습
  • 2주차: AI Agent의 작동 원리와 협업 방식 이해, 자동화된 작업 구조 설계
  • 3주차: 위 과정을 기반으로 실제 기능 개발 및 E2E·통합 테스트 수행, Chromatic을 통한 시각화와 결과물 배포

이렇게 생각하면 이번 학습은 AI가 만든 결과물을 사람이 신뢰할 수 있도록 검증하는 과정이었고, 단순히 테스트 코드만 배우는 것이 아니라 AI와 공존하는 개발자로서 어떤 사고방식을 가져야 하는지 고민할 수 있는 시간이었습니다. 저는 “AI가 만든 결과를 사람이 어떻게 신뢰할 수 있을까”에 대한 탐구 과정이었다고 생각합니다.

앞으로는 테스트 코드를 “품질 관리 도구”로만 보지 않고, AI 협업 환경에서 개발 프로세스를 안전하게 유지하는 핵심 도구로 인식하고 더 깊이 있게 공부해 보고 싶습니다.

기술적 성장

이번 과제에서는 가능하면 AI 도구에 의존하지 않고, 실제 테스트 코드 작성 과정 자체를 익히는 데 집중했습니다. 물론 이전에 사용했던 프롬프트나 설정들은 일부 사용했지만, 대부분은 직접 작성하며 문제를 해결하는 방식으로 진행했습니다.

Cursor를 활용해 폴더 구조와 주석 기반의 테스트 설계 틀을 먼저 만들고, 그 위에 수동으로 테스트를 작성했습니다. 필요할 때만 AI의 도움을 받아 Playwright나 Storybook 등 테스트 도구의 동작 원리와 실무 적용 방식을 직접 학습했습니다.

비록 회의로 인해 실습 시간이 부족했지만, 과정 중 배운 테스트 설계 패턴과 작성 노하우를 추후 재사용할 수 있도록 Notion에 복기 및 정리할 계획입니다. 이 경험을 통해 단순히 “코드를 짜는 사람”이 아니라, 테스트 전략을 이해하고 설계할 수 있는 개발자로 한 단계 성장했다고 느꼈습니다.

코드 품질

1주차에 비해 이번 과제에서는 테스트 코드 작성 과정이 훨씬 자연스러워졌습니다. 특히 Mock 데이터 구분, DOM 요소 접근 방식, 그리고 테스트 대상의 상태를 추적하는 로직 등에서 시행착오가 줄었습니다.

또한 이전에 작성한 테스트 코드를 기반으로 공통 로직을 재사용할 수 있는 구조를 잡으면서 생산성이 향상되었습니다. 테스트 간 중복을 줄이고, 유지보수를 고려한 코드 설계의 중요성을 다시 한 번 체감했습니다.

아직 컴포넌트 단위의 테스트 구조가 완벽히 정돈된 것은 아니지만, 테스트 코드가 점점 **‘읽히는 코드’**로 발전하고 있다는 점이 개인적으로 만족스러웠습니다.

학습 효과 분석

  • Playwright와 Storybook 등 브라우저 기반 E2E 테스트 도구의 기본 사용법 및 동작 구조 이해
  • E2E 테스트와 통합 테스트의 차이점과 목적의 명확한 구분
  • 통합 테스트 파일을 어떻게 분리해야 유지보수성과 협업 효율을 높일 수 있는지에 대한 고민
  • 이론적으로 완벽한 테스트 전략보다는, 팀의 현실적인 작업 시간과 리소스에 맞는 전략이 더 효과적일 수 있다는 인식
  • 테스트 작성 과정이 단순히 품질 보증이 아닌, 개발 구조 개선의 일환이라는 시각

리뷰 받고 싶은 내용

E2E 테스트 실행 시, e2e.json 파일 내 테스트 데이터를 병렬 환경에서도 안정적으로 처리할 수 있도록 개선했습니다. 테스트 시작 시마다 기존 데이터를 자동으로 초기화하여, 각 테스트 케이스가 동일한 초기 상태에서 독립적으로 실행되도록 설정했습니다. 이로 인해 병렬 테스트 실행 중 발생할 수 있는 데이터 충돌이나 상태 오염 문제를 방지할 수 있었습니다만... 뭔가 주먹구구 방식인거 같아서 다른 방법이 있나 질문 드립니다...

현재 저희 팀은 기능 단위, 즉 비즈니스 도메인별로 테스트 파일을 분리하여 작성하고 있습니다. 이 방식이 협업 면에서는 명확하지만, 실제 현업에서는 테스트 범위나 케이스를 어떤 기준으로 나누는지가 궁금합니다.

예를 들어, 기능별(도메인 단위)로 테스트를 나누는 것이 일반적인지, 아니면 컴포넌트나 서비스 레이어 중심으로 나누는 방식이 더 실무적으로 효율적인지 알고 싶습니다.

다음에 테스트 설계를 다시 하게 된다면, 실무 환경에서 사용하는 테스트 구조나 케이스 분류 방식을 참고하여 더 체계적인 테스트 설계를 시도해보고 싶습니다.

그리고 현재 Chromatic의 프로젝트 토큰 관리 방식에 대해 고민하고 있습니다. 현재는 로컬 환경 변수(.env)를 통해 토큰을 관리하고 있지만, GitHub Secrets에 올려두면 여러 팀원에게 노출될 가능성이 있어 보안상 다소 우려가 있습니다.

이런 상황에서 팀 협업 환경에서도 안전하게 Chromatic 토큰을 관리할 수 있는 방법이 무엇인지 궁금합니다.

준모님 수고하셨습니다.

Q. 기능별(도메인 단위)로 테스트를 나누는 것이 일반적인지, 아니면 컴포넌트나 서비스 레이어 중심으로 나누는 방식이 더 실무적으로 효율적인지 알고 싶습니다.

A. 사실 이것은 테스트가 먼저 라기보다는 어플리케이션 구조를 어떻게 잡았느냐가 중요한 것 같아요. 테스트 전략에서 제일 중요한 것은 어플리케이션이 어떤 구조를 가지고 있고 어떻게 추상하된 레이어를 가지고 있는지 입니다. 테스트 입장에서는 사실 이게 기능별인지 서비스 레이어인지 알 필요 없이 대상 컴포넌트를 테스트하게되는 것이고요 다만 예를들어 어떤 레이어가 사이드 이펙트가 너무 많아서 테스트를 하기가 힘들다라는 결론이난다면 해당 레이어를 사이드이펙트가 많은 부분과 테스트가 가능한 부분 두개의 레이어로 나눠서 테스트 가능한 부분만 테스트 집중하고 사이드이펙트가 나는 부분은 최대한 얇게 유지하는 형태로 테스트를 아예 안하기로 결정할 수 있겠죠. 이런 경우는 테스트가 구조에 영향을 준 부분이고 실제로 구조도 유연해집니다. :)

Q. 크로마틱 토큰

A. 크로마틱 토큰은 사실 지금 상황이 그런것이지 실무에서는 협업자들이 모두 알아야하는 키를 사용할 것이고 그리고 애초에 공개된 리포가 아닐것이기에 신경쓸게 없을 것같아요. 지금상황에서는 제생각엔 env와 같은 특정 파일을 git에 올리지 않고 로컬에서만 들고있는 방법밖에는 없을 것 같습니다. 사실 과제니까 이렇지 일반적인 상황은 아닌것 같아요. 과제 평가가 끝나면 키를 무효화하거나 변경하는 작업이 필요할 것 같네요.
