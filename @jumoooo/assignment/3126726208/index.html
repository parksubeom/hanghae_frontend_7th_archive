과제 셀프회고

기술적 성장

Promise.all은 Promise 배열(또는 iterable)을 병렬로 실행하는데, 기존 코드를 보니 Promise.all 안에서 await를 사용하고 있었습니다. 이렇게 하면 애초에 병렬 처리하려고 만든 내용 안에서 await를 해버려서 직렬로 처리되고 있었습니다. 그러니 내부 await를 제거하여 병렬로 처리하였습니다.

이미 호출한 API를 다시 호출하지 않도록 하려고 했는데, 처음에는 데이터를 캐싱하려고 했습니다. 그런데 문제가 있었습니다. Promise.all은 지금 병렬 실행으로 해놨기 때문에, 데이터를 비교할 때 함수들이 거의 동시에 실행하기 때문에 시점 차이로 인해 비교가 안되어 결국 중복 요청을 막지 못합니다. 그리고 데이터는 첫 번째 요청의 답이 완료된 후에 캐싱되기 때문에, 첫 요청이 아직 진행 중이면 다른 요청에 대해서는 기준이 아직 생성이 안되어서 비교가 불가능합니다. 그래서 데이터를 캐싱하는 게 아니라 Promise를 캐싱하는 방식으로 변경했습니다.

왜 Promise로 캐싱하면 다를까? 요청을 시작하자마자 즉시 캐싱을 진행하여 비교가 가능합니다. 첫 로딩부터 중복을 제거할 수 있습니다.

그래서 결국 뭐 했나요? 기존 Promise 함수를 감싸서 캐시를 전달하는 함수를 생성했습니다. 사용할 때 명시적 식별을 하기 위해서 문자열을 key로 받게 만들었습니다.

문자열 없이 하면 어떻게 되나요? 참조값이라 같은 걸로 취급 안해줄 수 있습니다. 특히 함수의 매개변수가 서로 다르게 들어있다면 서로 매개변수에 따라 다른 데이터를 호출하는 함수인데 같은 Promise로 오류가 날 가능성이 있습니다. 익명 함수나 고차 함수 사용 시 매번 변할 수 있어서 안정성이 떨어집니다.

const createCacheFetcher = () => {
  const cache = new Map<string, Promise<unknown>>();
  return async <T>(key: string, fetcher: () => Promise<T>): Promise<T> => {
    if (cache.has(key)) {
      return cache.get(key)! as Promise<T>;
    }
    const requestPromise = fetcher().catch((error) => {
      cache.delete(key);
      throw error;
    });
    cache.set(key, requestPromise); // Promise를 즉시 캐싱
    return requestPromise;
  };
};

이렇게 하니 캐시 후 콘솔을 보면 중복 요청이 제거된 것을 확인할 수 있었습니다.

코드 품질

Promise를 캐싱하는 방식으로 중복 API 호출을 방지한 부분이 가장 만족스럽습니다. 데이터를 캐싱하는 것보다 Promise를 캐싱하는 것이 더 효과적이라는 것을 알수 있었습니다.

const createCacheFetcher = () => {
  const cache = new Map<string, Promise<unknown>>();
  return async <T>(key: string, fetcher: () => Promise<T>): Promise<T> => {
    if (cache.has(key)) {
      return cache.get(key)! as Promise<T>;
    }
    const requestPromise = fetcher().catch((error) => {
      cache.delete(key);
      throw error;
    });
    cache.set(key, requestPromise);
    return requestPromise;
  };
};

이 구현으로 첫 로딩부터 중복 요청을 제거할 수 있었습니다.

학습 효과 분석

데이터를 캐싱하는 것만으로는 부족하고, Promise 자체를 캐싱해야 중복 요청을 효과적으로 방지할 수 있다는 것을 배웠습니다. 특히 병렬 처리 환경에서의 캐싱 전략에 대해 깊이 이해할 수 있었습니다.

리뷰 받고 싶은 내용

이러한 최적화를 진행할 때, 보통 어떤 방법이나 라이브러리, 또는 SaaS 도구를 활용해 검토하시는지 궁금합니다. 특히 추후 포트폴리오를 제작하거나, 회사의 자사 프로젝트를 직접 운영·개선해야 하는 상황에서 실무적으로 활용도가 높은 것들이 있다면 추천도 함께 부탁드리고 싶습니다!

과제 피드백

잘했어요 준모! 이번 과제에서 "왜 리렌더링이 발생하는가"를 진단하고 해결책을 적용하는 접근법을 배우기 위함이었습니다. 측정 기반으로 최적화하는 습관은 실무를 할 때 도움이 될 거예요!

Promise.all 내부에서 await를 사용하면 직렬 처리가 된다는 걸 직접 발견하고 수정한 부분이 좋았어요. 그리고 "데이터 캐싱 vs Promise 캐싱"의 차이를 깊이 파고든 점이 인상적입니다. 병렬 실행 환경에서는 요청이 거의 동시에 시작되기 때문에 데이터가 아닌 Promise 자체를 캐싱해야 한다는 통찰은 실무에서도 자주 마주치는 패턴이에요. 에러 시 캐시를 삭제하는 처리까지 넣은 것도 꼼꼼하네요.

Q) 최적화 진행할 때 어떤 방법/라이브러리/도구를 활용하는지, 실무에서 활용도 높은 것들이 궁금합니다.

=> 측정 도구로는 React DevTools Profiler, Chrome Performance 탭, Lighthouse가 기본이에요. 번들 분석은 webpack-bundle-analyzer나 vite의 rollup-plugin-visualizer를 많이 씁니다. 실무에서는 Sentry 같은 모니터링 도구로 실제 사용자 환경의 성능을 추적하기도 해요.

=> 라이브러리 측면에서는 이번에 직접 구현한 Promise 캐싱 같은 패턴이 react-query나 SWR에 이미 내장되어 있어요. 현재 프론트엔드는 거의 대부분 개발이 될만큼 되어서 상용 라이브러리들의 "왜 그렇게 만들어졌는지"를 이번 과제를 통해 체감했다면 무엇을 써야할지 감이 더 잘 잡히게 될거에요. 포트폴리오에서는 직접 구현한 경험을 통해 본질을 설명할 수 있고 라이브러리들이 어떤 가치를 주고자 하는지 경험까지 둘 다 보여주면 좋습니다.
