[2팀 전희재] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by junijaei2025.10.20

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

Mock 핸들러의 데이터 독립성을 확보하기 위해 클로저를 이용한 데이터 스코프 관리 전략을 적용했습니다. 각 테스트마다 외부의 영향을 받지 않는 독립적인 데이터를 조회, 변경해야 했기에 매개변수로 events 객체를 받아 변수로 선언한 뒤, http Mock 함수에서 해당 값을 사용하도록 했습니다. 마지막으로 events 객체를 참조하고 있는 http 함수를 반환해 외부와 독립된 데이터를 설계할 수 있었습니다.

처음에는 대략적인 감만 오고 구체적인 구현이 떠오르지 않아 고민을 많이 했었습니다. Q&A 시간에 오프코치님이 이건 호이스팅과 관련된 것이라고 힌트를 주시기도 했고, 준형학습메이트님께서도 클로저가 힌트라고 말씀해주셔서 도움을 많이 받은 것 같습니다.

구체적인 구현에 대해 고민했던 점이 있는데, 이는 아래에서 더 자세히 설명하겠습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

beforeEach, afterEach, beforeAll, afterAll 등 vitest에서 제공하는 함수를 활용해 매 테스트 코드마다 세팅하고 리셋하는 로직을 추가했습니다. handlersUtils 파일에 작성한 유틸 함수들을 사용해 독립적인 데이터를 구성했습니다.

과제 셀프회고

기술적 성장

  • 효율적인 테스트 코드 작성법
    • 처음에는 테스트 코드는 많을수록 좋다고 생각했지만, 경계값과 극한값을 검증했다면 중간값을 모두 검사할 필요는 없다는 조언을 듣고 새로운 시각을 얻었습니다.
  • 디스크립션과 코드의 가독성
    • 디스크립션으로 테스트 시나리오를 잘 나타내면서도 명확하고 가독성 좋은 서술에 대해 관심이 생겼습니다.
    • 테스트 코드의 목적상 자유롭게 작성하는 경우가 많았지만, AAA나 GWT 패턴에 맞게 개행을 주어 가독성을 높이는 것이 큰 효과를 준다는 점을 깨달았습니다.
  • 웹 접근성과 테스트 코드
    • 요소 접근자를 선택 할 때, 개발자 관점이 아닌 사용자 관점에서 테스트를 작성하려 노력했습니다.
    • 이 과정에서 ARIA 접근성과 시맨틱 태그의 중요성을 다시금 느꼈으며, 접근성이 높은 HTML 구조가 테스트에도 유리하다는 점을 깨달았습니다.
    • 특히 getByLabelText가 aria-label도 검색 대상으로 포함한다는 점을 새롭게 알게 되었습니다.
  • 클로저를 통한 상태 관리
    • 이론적으로만 알고 있던 개념을 실제 코드에 적용해보면서, 클로저가 낯선 개념이 아니라 익숙한 구조라는 것을 실감했습니다.
    • 특히 데이터의 독립성을 보장하기 위해 클로저를 활용하는게 아주 유용하다는 생각이 들었습니다.
    • 다만 아직은 데이터의 독립성 관리를 위해 클로저를 활용해야겠다는 생각이 바로 이어지지 않는 것 같습니다.

코드 품질

만족스러운 부분

  • 공통화, 모듈화
    • fixture 파일을 추가해 정적 데이터와 mock 데이터 생성 함수를 분리
      • 여러 테스트에서 사용하며 효율성을 많이 높였다고 생각해 만들길 잘했다는 생각이 듭니다.
      • 특히 Partial 타입을 활용해 mock 데이터 생성 함수를 유연하게 설계한 점이 적절했다고 느껴져 뿌듯했습니다.
    • 테스트 코드에서 중복 코드를 제거하면서도 가독성을 해치지 않는 선에 대해 고민함
      • 예를 들어, useEventOperations 테스트에서는 렌더링 후 첫 데이터 fetch가 완료될 때까지 대기하는 로직이 반복되었는데, 이를 적절히 함수화하여 재사용성을 높였습니다.
      • 다만 중복 코드라 하더라도 오히려 명시적으로 적는 게 더 적절한 상황들이 있다고 들었는데, 스스로 명확한 기준을 세우지는 못한 것 같습니다.

고민한 부분

  • 디스크립션 작성
    • 아고라에 올라온 얘기들을 보면서, 같은 테스트라도 작성자마다 포인트가 달라질 수 있다는 점을 느꼈습니다.
    • 테스트 코드의 디스크립션만으로 명확하게 테스트의 목적을 드러내는 것에 집중했습니다.
    • 그 중 하나로, 동일한 동작을 검증하는 테스트의 경우 문장을 통일하려 노력했습니다.
      • 예를 들어, 에러 알림을 띄우는 테스트 케이스가 여러 개 있을 경우 동일한 문장으로 작성했습니다. 문장이 조금씩 다르면 동작이 다른 것처럼 느껴질 수 있을거라 생각했습니다.
  • handlersUtils 구현
    • 제 코드는 server.use(...) 호출을 테스트 코드 내에 남겨두었지만, medium 브랜치의 코드는 이를 함수 내부로 포함시켜 더 높은 추상화를 시도했습니다.
    • 저는 server.use를 테스트 케이스에 작성하는 게, 런타임 시점에 외부 handler가 주입된다는 사실을 명확히 보여줘 테스트 맥락을 더 잘 보인다고 판단했습니다.
      • 다만 이 부분에 대해 지금 다시 생각해보니... handlers 파일에 선언된 handler와 handlersUtils 파일에 선언된 handler는 완전히 동일한 동작을 하고, 다른 점은 데이터의 독립성뿐이라 제 의도와는 맞지 않다는 생각이 듭니다...
      • 다른 분들의 의견이 궁금합니다.

리팩토링이 필요한 부분

  • 비동기적 테스트 코드 작성
    • 통합 테스트로 넘어가면서 테스트 코드가 많이 지저분해졌고, 특히 waitFor 함수의 잦은 사용이 신경쓰입니다.
    • 어느 정도는 getBy, queryBy, findBy 메서드를 상황에 맞게 더 적절히 조합했다면 개선될 수 있는 부분이었을거라 생각합니다.
  • 테스트 코드가 추가로 필요한 부분이 있을거란 생각이 듭니다.
  • 테스트 코드의 가독성이 부족한 것 같습니다.

학습 효과 분석

  • 테스트코드에 확실히 더 익숙해진 것 같습니다.
  • react-testing-library를 통한 DOM 검증이 아직 익숙하지 않아 연습이 필요할 것 같습니다.
  • 실무에도 적용해보고 싶으나 현실적으로 쉽지 않을 것 같습니다... 사이드 프로젝트를 하게 된다면 정말 TDD, BDD 기반으로 설계해보고 싶습니다.

과제 피드백

  • 과제의 양이 많아 처음에는 부담스러웠지만, 전반적인 퀄리티는 매우 높았다는 생각이 듭니다.
  • 난이도별로 나뉘어 있어서 스스로 난이도를 선택하고 비교할 수 있었던 점이 특히 좋았습니다. medium과 hard의 차이를 직접 살펴보면서 그 의도를 추론하는 과정이 흥미로웠습니다.

리뷰 받고 싶은 내용

  • mock 데이터의 위치와 변수명에 대한 고민이 많았습니다.
    • 테스트 코드를 작성하다보면 비교가 필수적이라 자연스럽게 많은 양의 변수가 필요하게 되는데, 이 변수명의 가독성과 팁이 궁금합니다.
  • 전체적인 테스트코드의 가독성이 어떤지, 가독성이 떨어지는 부분이 있다면 어느 부분이고 어떻게 개선할 수 있을지 궁금합니다.
  • 테스트 코드의 성능과 효율성에 대해서는 거의 고려하지 못하고 코드를 작성했는데, 실무에서는 이 부분에 대해서도 충분히 고려해야 할 것 같다는 생각이 듭니다.
    • 비효율적인 테스트 코드의 특징이 있다면 어떤 것일지, 제 코드에서 부족한 부분이 있는지 궁금합니다.
  • 약간 포괄적인 질문이지만 제가 미처 생각하지 못한 관점에서 놓친 부분이 있다면 알려주시면 감사하겠습니다.

과제를 수행하면서 느낀점

과제 시작 전 생각

실무에서는 테스트코드를 작성해 본 경험이 없지만, 이전에 진행했던 사이드 프로젝트에서 테스트코드를 사용해본 적이 있어서 조금인 익숙할거라고 생각했었습니다. 다만 당시 테스트코드를 작성하면서도 이게 맞나 싶었던 적이 많았고, 물어보며 할 사람이 없어 더욱 어렵게 느껴졌었기에 이번 기회가 더 소중하게 다가왔습니다.

과제 제출 후 생각

가장 큰 고비는 dateUtils 파일과 integration 파일이었습니다. dateUtils는 첫 시작으로 선택한 파일이었는데 생각보다 양이 엄청 많아서 그럼 앞으로는 얼마나 더 남았다는거지? 싶어져서 두려웠습니다... 이후 작업은 무탈하게 진행되다가 통합 테스트 코드를 작성하면서 요소 특정과 비동기 제어 부분에서 많이 애를 먹었습니다. 테스트 코드에 익숙해지긴 했지만 스스로 깊이가 부족하다고 느껴집니다. 이번 과제 말고도 전반적으로 구현은 빠르게 하나 깊이가 부족하다는 생각이 자주 드는데, 항해를 통해 잘 채워나가고 싶습니다.

과제 피드백

안녕하세요 희재님! 2주차 과제 잘 진행해주셨네요 ㅎㅎ 고생하셨습니다!

mock 데이터의 위치와 변수명에 대한 고민이 많았습니다. 테스트 코드를 작성하다보면 비교가 필수적이라 자연스럽게 많은 양의 변수가 필요하게 되는데, 이 변수명의 가독성과 팁이 궁금합니다.

흠.. 굳이 변수를 선언할 필요가 있을까? 라는 생각도 들어요 ㅎㅎ 선언하지 않아도 되는 구간은 그냥 리터럴로 표현하면 어떨까 싶습니다.

전체적인 테스트코드의 가독성이 어떤지, 가독성이 떨어지는 부분이 있다면 어느 부분이고 어떻게 개선할 수 있을지 궁금합니다.

전체적으로 중복코드를 제거하기 위한 노력을 많이 해주신 것 같아요. 그런데 테스트의 목적이 중복코드 제거가 맞을까요? 저는 "정확한 값을 검증하는 것"과 내가 아닌 다른 사람이 테스트를 읽었을 때 쉽게 이해할 수 있는지가 중요하다고 생각해요. 그래서 테스트를 작성하는 방식과 일반적인 코드를 작성하는 방식이 달라야한다고 생각합니다.

지금은 변수와 다양한 함수로 랩핑되어 "코드가 감춰진" 상태로 값을 검증하는 경우가 많아요. 그니까, 테스트를 통해 값을 해석하는 과정이 필요하다는거죠.

가독성에 대한 견해를 어떤 기준으로 정해야 할지 모르겠으나.. 개인적으로 "깔끔한 코드"는 맞지만 "읽고 이해하기 쉬운 테스트" 와는 다른 결이라고 생각합니다.

테스트 코드의 성능과 효율성에 대해서는 거의 고려하지 못하고 코드를 작성했는데, 실무에서는 이 부분에 대해서도 충분히 고려해야 할 것 같다는 생각이 듭니다. 비효율적인 테스트 코드의 특징이 있다면 어떤 것일지, 제 코드에서 부족한 부분이 있는지 궁금합니다.

아직은 성능에 대한 병목이 없는 상태인데요, 병목이 있다면 보통 비동기 코드 때문에 발생할 확률이 높답니다 ㅎㅎ 혹은 테스트 코드가 아닌, 실행 코드 자체가 오래걸린다거나!?

그리고 어떤 테스트 라이브러리를 사용하냐에 따라 속도가 달라요. 어떤 테스트 도구가 있고, 각각 어떤 차이가 있는지 찾아보시면 좋을 것 같네요!

약간 포괄적인 질문이지만 제가 미처 생각하지 못한 관점에서 놓친 부분이 있다면 알려주시면 감사하겠습니다.

사실 전체적으로 포괄적인 질문인데요, 이럴 경우 좋은 피드백을 드리기가 어려워요.

희재님이 신경쓴 코드와, 그 근거에 대해 설명을 해주셔야 더 정확한 피드백을 드릴 수 있답니다 ㅎㅎ
