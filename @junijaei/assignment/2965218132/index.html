과제 체크포인트

필수 스펙

  1. 반복 유형 선택

    • 일정 생성 또는 수정 시 반복 유형을 선택할 수 있다.
    • 반복 유형은 다음과 같다: 매일, 매주, 매월, 매년
      • 31일에 매월을 선택한다면 → 매월 마지막이 아닌, 31일에만 생성하세요.
      • 윤년 29일에 매년을 선택한다면 → 29일에만 생성하세요!
    • 반복일정은 일정 겹침을 고려하지 않는다.

  2. 반복 일정 표시

    • 캘린더 뷰에서 반복 일정을 아이콘을 넣어 구분하여 표시한다.

  3. 반복 종료

    • 반복 종료 조건을 지정할 수 있다.
    • 옵션: 특정 날짜까지
      • 예제 특성상, 2025-12-31까지 최대 일자를 만들어 주세요.

  4. 반복 일정 수정

    1. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      • 반복일정을 수정하면 단일 일정으로 변경됩니다.
      • 반복일정 아이콘도 사라집니다.
    2. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      • 이 경우 반복 일정은 유지됩니다.
      • 반복일정 아이콘도 유지됩니다.

  5. 반복 일정 삭제

    1. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      1. 해당 일정만 삭제합니다.
    2. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      1. 반복 일정의 모든 일정을 삭제할 수 있다.

기본 과제

공통 제출

  • 테스트를 잘 작성할 수 있는 규칙 명세
  • 명세에 있는 기능을 구현하기 위한 테스트를 모두 작성하고 올바르게 구현했는지
  • 명세에 있는 기능을 모두 올바르게 구현하고 잘 동작하는지

기본 과제(Easy)

  • AI 코드를 잘 작성하기 위해 추가로 작성했던 지침
  • 커밋별 올바르게 단계에 대한 작업
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

기본 과제(Hard)

  • Agent 구현 명세 문서 또는 코드
  • 커밋별 올바르게 단계에 대한 작업
  • 결과를 올바로 얻기위한 history 또는 log
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

심화 과제

# AI와 테스트를 활용한 안정적인 기능 개발 리포트

## 사용하는 도구를 선택한 이유가 있을까요? 각 도구의 특징에 대해 조사해본적이 있나요?
- 범용적인 단순 프롬프트 제작에는 GPT-5를 사용했고, 
   실제 프로젝트 개발 환경에서는 Cursor IDE + Claude Code 조합을 사용했습니다.
- 직접 비교해본 결과, GPT-5는 간결하고 효율적인 프롬프트 작성에 강점이 있었고, 
   Claude Code는 프로젝트 전반의 구조를 파악한 뒤 상세한 문서나 테스트 기반 코드 작성에 탁월한 성능을 보였습니다.
- 즉, GPT-5는 빠른 아이디어 전개나 기초 프롬프트 설계에 적합했고, 
   Claude Code는 실무 수준의 코드 품질과 문서화를 보장하는 데 적합했습니다.

## 테스트를 기반으로 하는 AI를 통한 기능 개발과 없을 때의 기능개발은 차이가 있었나요? 
- AI를 통한 기능 개발은 초기 세팅과 Context 구성에 많은 시간이 소요되지만, 
   이후에는 TDD 사이클을 일정 부분 강제한다는 점에서 명확한 차이가 있었습니다.
- 테스트 기반으로 작업이 진행되다 보니, ‘기준 충족 여부’ 중심으로 개발이 진행되었고, 
   결과적으로 품질 편차가 줄고 회귀 오류가 줄어드는 효과가 있었습니다.

## AI의 응답을 개선하기 위해 추가했던 여러 정보(context)는 무엇인가요? 

### 프로젝트 수준 Context
- `CLAUDE.md (프로젝트 가이드)`
  - 포함 내용:
    - 프로젝트 개요 (React 19, TypeScript, Vite)
    - 개발 명령어 (dev, test, build)
    - 아키텍처 설명 (State Management Pattern, Core Hooks)
    - 테스트 전략 (MSW Setup, 테스트 조직)
    - 타입 시스템 (Event, EventForm, RepeatInfo)
    - 주요 패턴 (반복 일정 추가/수정/삭제)
    - 커스텀 에이전트 목록
  - 효과: AI가 프로젝트 구조와 컨벤션을 즉시 파악

- `.claude/docs/workflows.md (워크플로우 정의)`
  - 포함 내용:
    - 에이전트 역할 및 책임
    - 데이터 흐름 다이어그램
    - TDD 사이클 정의
    - 운영 원칙 (중복 금지, 입출력 명확화, Git 커밋 관리)
    - 산출물 저장 위치
  - 효과: AI가 자신의 역할과 다음 단계를 명확히 이해

- `Git 커밋 컨벤션`
  - 효과: 일관적인 커밋 메세지 작성

### 에이전트 수준 Context
- `체크리스트 (.claude/docs/check-lists/*.md)`
  - 포함 내용: 
    - 각 에이전트가 완료해야만 작업을 종료할 수 있는 체크리스트
  - 효과: 품질 기준을 일관되게 유지

- `코드 품질 개선을 위한 참고 자료 (각 에이전트 문서 내에 위치)`
  - 포함 내용:
    - 켄트백의 TDD 문서
    - Frontend 코드 품질 관련 문서
  - 효과: 코드 품질의 하방선을 올림

### 도메인 수준 Context
- `PRD (Product Requirements Document)`
  - 포함 내용:
    - 프로젝트 목표
    - 범위 (In/Out of Scope)
    - 주요 기능 (Acceptance Criteria 포함)
    - 제약사항
    - 데이터 모델
    - UI/UX 요구사항
  - 효과:
    - Test Designer가 정확한 시나리오 설계
    - Implementer가 요구사항에 맞게 구현

- `테스트 시나리오 (test-scenarios.md)`
  - 포함 내용:
    - Given When Then 패턴에 맞춰 작성된 각 기능 별 테스트 시나리오
  - 효과:
    - Test Writer가 정확한 테스트 코드 작성
    - Implementer가 예상 동작 파악

## 이 context를 잘 활용하게 하기 위해 했던 노력이 있나요?
- 에이전트 문서 하단에 참고해야할 문서 링크를 명시적으로 기입해 실행 때마다 참고할 수 있도록 작성했습니다.
- 또한, 에이전트 산출물의 형식을 일관되게 유지하기 위해 산출물 템플릿을 관리했습니다.
   이렇게 함으로써 하위 에이전트의 결과물이 상위 에이전트의 Context로 자연스럽게 연결되도록 구조화했습니다.

## 생성된 여러 결과는 만족스러웠나요? AI의 응답을 어떤 기준을 갖고 '평가(evaluation)'했나요?
- 각 에이전트별로 체크리스트 문서를 기반으로 자동 평가가 이루어졌습니다.
   모든 항목이 충족되어야만 해당 작업을 종료할 수 있도록 강제했습니다.
- 또한 **오케스트레이터(Orchestrator)**는 각 에이전트의 결과물을 검증한 뒤,
   재시도 여부를 스스로 판단하도록 설계하여, 결과의 신뢰도를 높였습니다.

## AI에게 어떻게 질문하는것이 더 나은 결과를 얻을 수 있었나요? 시도했던 여러 경험을 알려주세요. 
- 프롬프트의 구조화 수준이 답변 품질을 결정한다는 점을 체감했습니다.
- 얻고 싶은 결과물을 명확히 정의하고,
   반드시 포함되어야 할 조건과 제외되어야 하는 조건을 함께 명시했을 때
   가장 일관적이고 만족스러운 결과를 얻을 수 있었습니다.
- 즉, “무엇을 원하느냐”에 더해서 “무엇까지 포함하고, 무엇은 제외해야 하는가”를 명확히 한 프롬프트가 가장 효과적이었습니다.

## AI에게 지시하는 작업의 범위를 어떻게 잡았나요? 범위를 좁게, 넓게 해보고 결과를 적어주세요. 그리고 내가 생각하는 적절한 단위를 말해보세요.
- 제 경험상, 결과물이 명시적으로 정의되어 있다면 요청 범위가 넓어도 품질에 문제가 없었습니다.
   예를 들어, PRD가 충분히 구체적일 경우
   “이 문서 전체를 기반으로 기능 단위별 구현을 진행하라”는 식의 큰 단위 요청도 꽤나 만족스러운 결과를 냈습니다.
   (물론, 단계적으로 나눠서 진행해달라는 프롬프트를 추가하는 게 더 성능이 좋았습니다.)

- 반면, 요구의 목적 자체가 모호한 경우(예: “에이전트를 생성해줘” 같은 요청)에는
   AI가 방향을 잘못 잡는 경우가 많았습니다.

- 따라서 ‘요청 범위의 크기’도 중요하지만 ‘요구사항의 명확성’이 결과 품질을 결정한다고 생각합니다.

## 동기들에게 공유하고 싶은 좋은 참고자료나 문구가 있었나요? 마음껏 자랑해주세요.
- 이미 과제 참고자료로 올라와있는 내용이지만, 프롬프트 엔지니어링 관련 자료들이 많은 도움이 되었습니다.

## AI가 잘하는 것과 못하는 것에 대해 고민한 적이 있나요? 내가 생각하는 지점에 대해 작성해주세요.
- AI가 잘하는 점은 일관성과 대량 작업 처리 능력입니다.
   초기에 명확한 기준을 세워두면, 이후에는 그 기준을 벗어나지 않는 수준 높은 산출물을 지속적으로 생성합니다.

- 반면 AI가 약한 점은 스스로 결과를 평가하는 능력입니다.
   좋은 결과물이 무엇인지를 프롬프트에서 구체적으로 제공하지 않으면,
   기준이 흔들려 방향이 다른 결과를 낼 가능성이 높았습니다.

- 즉, 인간이 세워주는 ‘판단 기준’이 전체 품질을 좌우했습니다.

## 마지막으로 느낀점에 대해 적어주세요!
- 처음에는 단순히 AI에게 시키면 코드가 알아서 높은 퀄리티로 완성되겠지하고 생각했지만,
   실제로 해보니 프롬프트 구조화에 따라 결과물의 품질 차이가 컸습니다.
   AI에게 제공하는 입력의 질이 곧 출력의 질을 결정한다는 사실을 명확히 깨달았습니다.
- AI를 도구로 쓰는 게 아니라, 후임으로 들어온 주니어 개발자처럼 대하라는 말이 무슨 뜻인지 알게 되는 과정이었습니다.

과제 셀프회고

저는 이번 과제를 하기 전까지만해도 AI를 제대로 활용하며 개발을 진행해본 적이 없었고, 심지어는 커서 IDE조차 사용해본 경험이 없었습니다. 따라서 초반에는 좋은 프롬프트 작성법에 대해 고민하기 보다는 낯선 IDE 환경과 AI도구에게 적응하기 위한 시간으로 주로 사용했습니다.

이후 학습 메이트분들께 추천을 받아, 비매드 메서드와 슈퍼 클로드의 디렉토리 구조와 파일들을 분석하며 AI 기반 개발의 워크플로우가 어떤건지에 대한 막연한 감을 잡을 수 있었습니다. 두 프로젝트의 구조를 참고해 제 과제에도 일부 적용했는데, 초기에는 ‘왜 필요한지’보다 ‘저 프로젝트들에도 있으니까 넣자’는 식으로 에이전트를 추가했습니다.

그 결과, 실제 코드 개발을 담당하는 에이전트는 아직 없는데 PM, PO, UX 설계자, 아키텍트, 스크럼 마스터까지 전부 만들어버렸고, 팀원분들과 학습메이트분들에게 “왜 혼자 스타트업을 설립했냐”는 말을 듣기도 했습니다. (...)

물론 많은 에이전트를 두는 구조가 무조건 잘못된 건 아니지만, 이번 과제의 목표는 기존 프로젝트에 새로운 기능을 추가하는 것이었기 때문에, 초기 설계 중심의 에이전트가 다수 존재하는 건 목적이 전도된 상태였습니다.

이에 따라 초기 설계 관련 에이전트들을 과감히 정리하고, TDD 중심의 필수 에이전트만 남기는 방식으로 workflow를 재설계했습니다.

결론적으로 완성된 workflow는 다음과 같습니다.

TDD와 직접적인 관련은 없지만, 가장 만족스러웠던 에이전트는 Doc-Keeper입니다. 이 에이전트는 이름 그대로 모든 문서를 관리하고 갱신하는 역할에 집중했습니다. 문서 또한 코드처럼 ‘유지보수’가 필요한 자산이라는 점에서, 반복적이고 사소한 관리 업무를 AI가 맡는 것은 매우 효율적이라 느꼈습니다.

기술적 성장

프롬프트 작성 스킬이 많이 성장했다고 느껴집니다. 아래 예시를 통해 말씀드리겠습니다.

초반 프롬프트

안녕 너는 서브에이전트를 만들기 위한 프롬프트 작성을 도와주게 될거야.

에이전트들의 구조는 다음과 같아.
PM(나)
   ↓ PRD 생성
   ├─→ Architect (아키텍트)     [PRD → Architecture 문서]
   ├─→ UX Expert (UX 전문가)    [PRD → UX 설계]
   └─→ PO (Product Owner)       [PRD 검증 & Epic/Story 관리]
        ↓
   SM (Scrum Master)             [PRD + Architecture → Story]
        ↓
   Dev (Developer)               [Story → 코드 구현]
        ↓
   QA (Quality Assurance)        [Story + PRD → 테스트]

이 중에서 코드를 직접 수정할 수 있는 권한은 dev밖에 없어.
모든 에이전트들의 대화는 한글로 하도록 하고, `use Context7`을 사용하도록 해줘.
https://github.com/bmad-code-org/BMAD-METHOD/를 참고해서 만들거야.

지금부터 너가 해야하는 과제가 뭔지 자세히 설명해봐.

처음엔 나름 구체적으로 작성했다고 생각했지만,

  1. 프롬프트의 목적이 불분명했고 (에이전트를 생성하려는 건지, 구조를 정의하려는 건지 모호),
  2. 참고자료와 제약사항이 뒤섞여 구조화가 전혀 되어 있지 않았습니다.

결국은 원하는 결과물을 얻어내지 못했습니다.

후반 프롬프트

repeatInterval은 반복 일정의 인터벌을 설정할 수 있는 속성입니다.
아래는 예시입니다.
1. 1월 1일에 매일 반복, 인터벌 2를 설정하면 1월 1일, 1월 3일, 1월 5일 ... 이런식으로 반복됩니다.
5. 1월 1일에 매주 반복, 인터벌 2를 설정하면 1월 1일, 1월 15일, 1월 29일 ... 이런식으로 반복됩니다.
6. 1월 1일에 매월 반복, 인터벌 2를 설정하면 1월 1일, 3월 1일, 5월 1일 ... 이런식으로 반복됩니다.

현재는 해당 기능이 구현되어있지 않습니다.
1. 오케스트레이터 에이전트를 호출해 해당 에이전트가 필요한 에이전트들을 호출해 TDD 방식으로 구현합니다.
2. 적절한 단위로 커밋하며 진행됩니다.
4. 모든 커밋은 한국어로 진행됩니다.
5. 모든 테스트가 끝날때까지 반복 수행합니다.

후반부에는 구체적인 동작 예시를 제시하고, 수행 단계를 명확히 지시하는 방식으로 발전했습니다.

아직 완벽하진 않지만, AI에게 “무엇을 해야 하는지”와 “어떻게 해야 하는지”를 구체화하려 노력한 점이 큰 차이였습니다.

프롬프트 히스토리 문서 작성

이번 과제에서는 제가 작성한 프롬프트와 및 결과물 히스토리 문서를 별도로 만들어 AI와의 상호작용 결과를 분석했습니다. 이 과정을 통해 어떤 문장은 AI가 잘 이해하고, 어떤 문장은 오해하는가를 파악하며 점진적으로 프롬프트를 개선했습니다.

코드 품질

이번 프로젝트에서 만족스러웠던 점 중 하나는 에이전트 구조의 체계화입니다.

오케스트레이터 에이전트가 전체 흐름을 관리하며, 실패 원인에 따라 특정 단계로 회귀해 작업을 반복하도록 설계했습니다.

특히 신경 쓴 부분은 “부분 회귀” 구조입니다. 단순히 전체 TDD 사이클을 반복하면 불필요한 에이전트까지 다시 호출되어 토큰 낭비와 비효율이 발생했습니다. 이를 개선하기 위해 오케스트레이터가 스스로 판단 후 필요한 에이전트만 선별해 호출하도록 변경했더니, 작업 흐름은 훨씬 단순해지고, 불필요한 토큰 소모도 크게 줄었습니다.

또 하나의 만족스러운 점은 일관된 코드 품질 유지 체계입니다.

각 에이전트마다 개별 체크리스트를 만들어, 이를 통과해야 다음 단계로 넘어갈 수 있도록 했습니다. 오케스트레이터 또한 자체 체크리스트를 통해 서브 에이전트들의 산출물을 재검증했습니다. 이런 반복적인 검증 구조 덕분에 코드 간 일관성이 일정 수준 이상으로 유지되었습니다.

학습 효과 분석

단순히 실력 상승 정도보다도, AI를 활용한 개발에 정말 무지했었는데, 이번 과제를 계기로 어느정도 익숙해지게 된 점이 정말 기쁩니다... 그동안 AI를 써야 한다는 막연한 압박감만 있었지, 어디서부터 공부해야 할지 감이 전혀 없었는데, 이번 과제를 통해 실제 요구사항을 기반으로 AI를 도구로 활용하는 경험을 해볼 수 있어서 기뻤습니다.

무엇보다 코치님, 학습메이트분들, 팀원분들이 같이 의견을 나누고 자료를 공유해주셔서 그나마 잘 따라갈 수 있었던 것 같습니다. 혼자였다면 중간에 포기했을 것 같습니다.

과제 피드백

학습 목적에 적절하게 맞는 좋은 과제였다고 생각합니다.

리뷰 받고 싶은 내용

리뷰 받고 싶은 내용보다는 질문에 가까운데,

에이전트의 결과물을 어떤 방식으로 검증하시는지가 궁금합니다.

저는 나름의 규칙으로 체크리스트도 확인하게 하고, 오케스트레이터가 서브에이전트의 결과물을 피드백하게 하기도 하면서 진행했지만,

  1. 잘못된 PRD를 기반으로 모든 검증을 통과하거나, (애초에 통과 기준이 잘못 잡혀있는 경우)
  2. 테스트 코드를 스킵하는 등 ‘형식적 통과’로 목표를 달성하는 상황이 있었습니다. (통과를 위해 잘못된 방법을 택한 경우)

AI 에이전트가 스스로 결과물을 검증하는 구조인데, 이 자기 검증이 오히려 오류를 은폐할 가능성도 있다고 느껴졌습니다. 이런 구조적 허점을 방지하려면 자기 검증 과정에 어떤 로직을 추가하는 게 효과적일까요?

이런 상황들에서 코치님은 어떤 방식으로 검증을 진행하시는지, 노하우나 팁이 궁금합니다!
