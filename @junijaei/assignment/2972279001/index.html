[2팀 전희재] Chapter 1-3. 완성도있는 테스트 전략 수립하기

by junijaei2025.11.03

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

1️⃣ 내가 생각한 테스트 전략

테스트 전략을 수립할 때 저의 가장 주요한 판단 기준은 테스트 대상의 의존성 복잡도였습니다.

순수한 로직이나 유틸리티 함수처럼 외부 의존성이 없는 코드는 단위 테스트로 검증하고, 여러 컴포넌트나 커스텀 훅이 상호작용하는 경우에는 통합 테스트를, 브라우저 API나 실제 사용자 환경이 필요한 경우에는 E2E 테스트를 선택하는 방향을 제안했습니다. 핵심은 이 기능을 테스트하기 위해 어떤 환경과 의존성이 필요한가를 먼저 파악하는 것이었습니다.

또, 가능한 한 가장 작은 단위의 테스트로 커버할 수 있다면 그 방향을 우선시해야 한다고 생각했습니다. 예를 들어 캘린더에 일정을 추가하는 기능을 생각해보면, 날짜와 시간의 유효성 검증 로직, API 호출 여부, 그리고 최종 UI 렌더링까지 여러 레이어의 검증이 필요합니다. 이 모든 것을 하나의 거대한 테스트로만 검증하는 것은 비효율적일 뿐만 아니라 테스트 실패 시 원인을 특정하기도 어렵다고 생각했습니다.

전략을 생각하는 과정에서 테스트 비용 대비 효과에 대해 많이 고민했습니다.

단위 테스트는 빠르고 유지보수가 용이하기 때문에, 비즈니스 로직을 가능한 한 많이 이 단계에서 검증하면 상위 테스트의 부담을 줄일 수 있습니다. 실행 속도가 빠르고 비용이 적게 드는 단위 테스트를 통해 핵심 로직의 정확성을 보장하는 방향이 효율적이라고 생각했습니다.

반면 통합 테스트나 E2E 테스트 같은 상위 레벨에서는 모든 분기를 일일이 테스트하기보다는, 실제 사용자가 시스템을 사용하는 주요 시나리오와 흐름을 재현하는 데 집중하는게 좋다고 생각했습니다.

결과적으로, 테스트의 신뢰성과 효율성을 동시에 확보하는 방향으로 고민해 설계했습니다.

2️⃣ 팀원들과 합의한 테스트 전략

판단 기준

테스트 종류 판단 기준

  • 로직 중심인가? → 단위(Unit) 테스트
  • 컴포넌트 간 협력이 중요한가? → 통합(Integration) 테스트
  • 브라우저 API나 사용자 인터랙션이 핵심인가? → E2E 테스트
  • 시각적 변화가 중점적 테스트 과제인가? → 시각적 회귀(Visual Regression) 테스트

테스트 우선순위 판단 기준

  • 핵심 사용자 시나리오인가? → 우선순위 높음
  • 오류 발생 시 영향 범위가 큰가? → 우선순위 높음
  • 변경이 잦은 코드인가? → 우선순위 높음
  • 시각적 중요도가 낮거나, 영향이 제한적인가? → 우선순위 낮음
  • 다른 테스트에서 이미 간접적으로 검증되는가? → 우선순위 낮음
  • 변경 가능성이 거의 없는가? → 우선순위 낮음

테스트 설계

1. useEventForm

  • 단위 테스트 (vitest)
  • 선정 이유
    • 폼 상태 관리 핵심 훅으로, UI 의존성이 없음
    • 입력/검증/리셋 등 순수 로직 중심
    • 다양한 엣지케이스 검증 필요
    • 빠른 피드백과 높은 커버리지 달성 가능
  • 검증 항목
    • 초기 상태
    • 상태 변경
    • 폼 리셋 동작
    • 특정 일정 수정 동작
    • 반복 일정 처리
  • 테스트 목표
    • 로직의 정확성 (UI와 독립적으로 동작 보장)

2. 캘린더 날짜 클릭 → 폼 자동 완성 기능

  • 통합 테스트 (React Testing Library)
  • 선정 이유
    • DateCell → useDateClick → useEventForm → EventForm 간 상호작용 검증 필요
    • 브라우저 API 의존 없음 → 통합 테스트로 충분
    • 실제 시나리오 기반 빠른 피드백 가능
  • 검증 항목
    • 날짜 클릭 시 폼에 날짜·시간 자동 입력
    • UI에 값이 올바르게 반영되는지 확인
  • 테스트 목표
    • 컴포넌트 간 데이터 흐름과 폼 자동 완성 시나리오의 정확성 검증

3. 드래그 앤 드롭 기능

  • E2E (Playwright)
  • 선정 이유
    • 마우스 이벤트, 좌표, DOM 조작 등 복합적 사용자 인터랙션
    • DnD 라이브러리와의 통합 검증 필요
    • 시각적 피드백과 실제 사용자 경험 중심 기능
  • 검증 항목
    • 일정 이동 시 날짜 변경, 시간 유지
    • 반복 일정 → 단일 일정 변환
    • 겹침 시 경고 표시 및 처리
    • 속성 유지(제목·시간·카테고리 등)
    • 월간/주간 뷰 간 동작 일관성
  • 테스트 목표
    • 실제 브라우저 환경에서 사용자 중심 동작과 시각적 피드백 검증

3️⃣ 전략에 맞춰 추가된 테스트

테스트 항목유형도구이유

과제 셀프회고

기술적 성장

이번 과제에서 가장 많이 얻은 점은, 한 번도 사용해본 적 없던 E2E 테스트와 시각적 회귀 테스트를 직접 다뤄본 경험이었습니다. 실제로 적용해보면서 두 테스트의 개념과 역할을 확실히 이해할 수 있었고, 각각의 장단점도 체감할 수 있었습니다.

사용법 자체는 익혔지만, 아직 실무 환경에서 어떤 기준으로 적용해야 할지 감이 잘 오지 않습니다. 특히 시각적 회귀 테스트는 자동화의 장점이 크지만, 프로젝트 구조나 컴포넌트 분리 기준에 따라 효율이 달라질 수 있다는 점이 어렵게 느껴졌습니다.

여전히 낯설지만, 이번 경험을 통해 앞으로 테스트 전략을 세울 때 고려해야 할 포인트를 조금씩 잡아가고 있는 것 같습니다.

코드 품질

이전 과제들에서는 효율적인 테스트 코드 문법이나 접근성을 고려한 테스트 케이스 작성 등 기술적인 부분에 집중했다면, 이번에는 “무엇을 테스트해야 하는가”에 초점을 맞췄습니다.

특히 사용자의 실제 시나리오를 테스트 코드 안에 그대로 녹여내는 방법에 대해 많이 고민했습니다. 테스트 코드의 비용에 대해서도 현실적으로 생각하게 되었는데, E2E 테스트는 작성과 실행 모두 비용이 큰 만큼 모든 엣지 케이스를 다루기보다는 핵심 동작 검증에 집중했습니다. 예를 들어 일정 관리 기능에서는 CRUD를 각각 나눠서 검증하기보다, 실제 사용자가 일정을 생성하고 수정하고 삭제하는 일련의 흐름을 하나의 시나리오로 테스트했습니다. 대신 세부적인 동작이나 분기 처리는 통합 테스트나 단위 테스트를 활용해 검증하려고 했습니다.

이런 접근이 실제 서비스 환경에서의 테스트 효율성과 유지보수성 측면에서 더 현실적이라는 생각이 들었습니다.

학습 효과 분석

이번 과제를 통해 테스트라는 행위가 단순히 코드의 안정성을 확보하는 것 이상이라는 점을 느꼈습니다. 단위 테스트, 통합 테스트, E2E 테스트, 시각적 회귀 테스트 각각의 목적이 다르고, 그에 따라 작성 방식과 비용도 달라진다는 점을 체감했습니다. 특히 E2E 테스트를 작성하면서, “사용자 경험을 검증한다”는 본질적인 목표에 더 집중하게 되었습니다. 아직은 테스트 구조를 완전히 체계화하기 어렵지만, 이번 경험이 앞으로 테스트 전략을 수립할 때 좋은 밑거름이 될 것 같습니다.

과제 피드백

적절한 테스트 코드 전략을 세울 수 있도록 팀원들끼리 논의하게 설정한 부분이 참 좋았습니다!!

리뷰 받고 싶은 내용

이번 과제와 직접적인 관련은 없지만, 커밋 단위와 커밋 메시지 작성 방식에 대한 피드백을 받고 싶습니다!

제가 느끼기에, 지금은 커밋이 너무 작은 단위로 쪼개져 있어서 전체적인 흐름이 다소 복잡하게 느껴집니다. 또, 특정 기능을 구현, 테스트 코드, 리팩토링을 진행하면서, 해당 과정이 전부 나눠져 있어 더욱 가독성이 떨어진다고 생각이 듭니다.

그동안의 과제들은 브랜치를 세분화해서 관리하지 않았고, 커밋 메시지 대부분을 Claude 코드에게 위임해서 작성했습니다. 회사에서도 커밋 컨벤션이나 브랜치 전략에 크게 신경 쓰지 않는 분위기라, 일반적으로 개발자들이 협업 시 어떤 기준으로 커밋 단위를 나누는지 잘 모르겠습니다.

협업이나 리뷰 관점에서 가독성이 높고 관리하기 좋은 커밋 단위를 어떻게 설정하는 게 좋을지, 그리고 커밋 메시지를 어떤 식으로 정리하면 명확하게 의도를 전달할 수 있을지에 대해 조언을 받고 싶습니다.

과제 피드백

수고했어요 희재! 지난 3주간 테스트를 경험하면서 대규모 프로젝트에서 품질을 어떻게 지켜내는지 직간접적으로 체험하는 시간이 되었기를 바랍니다. 테스트를 만드는데 들어가는 품이 엄청나지만 이를 감안하고도 도입하려는 걸 보면 테스트가 없었을 때가 얼마나 고통스러울지 짐작할 수 있죠. 언젠가 커리어상 대형 프로젝트를 맡게 될 때 "테스트 작성할 줄 아나요?"라고 물어봤을 때 생소하지 않고 할 수 있다는 자신감을 가질 수 있는 예방 주사가 되어주었길 바래요.

이번 과제에서 "무엇을 테스트해야 하는가"에 집중한 점이 정말 좋았어요. 테스트의 본질인 "사용자 경험 검증"과 "비용 대비 효과"를 고민하면서 e2e에서 핵심 동작을 찾아 집중해서 작성하고 세부 분기는 단위/통합 테스트로 커버한다는 전략은 아주 현실적인 접근입니다.

Q) ... 일반적으로 개발자들이 협업 시 어떤 기준으로 커밋 단위를 나누는지 잘 모르겠습니다. 협업이나 리뷰 관점에서 가독성이 높고 관리하기 좋은 커밋 단위를 어떻게 설정하는 게 좋을지, 그리고 커밋 메시지를 어떤 식으로 정리하면 명확하게 의도를 전달할 수 있을지에 대해 조언을 받고 싶습니다.
