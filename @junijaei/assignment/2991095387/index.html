기본과제

상품목록

상품 목록 로딩

  • 페이지 접속 시 로딩 상태가 표시된다
  • 데이터 로드 완료 후 상품 목록이 렌더링된다
  • 로딩 실패 시 에러 상태가 표시된다
  • 에러 발생 시 재시도 버튼이 제공된다

상품 목록 조회

  • 각 상품의 기본 정보(이미지, 상품명, 가격)가 카드 형태로 표시된다

한 페이지에 보여질 상품 수 선택

  • 드롭다운에서 10, 20, 50, 100개 중 선택할 수 있으며 기본 값은 20개 이다.
  • 선택 변경 시 즉시 목록에 반영된다

상품 정렬 기능

  • 상품을 가격순/인기순으로 오름차순/내림차순 정렬을 할 수 있다.
  • 드롭다운을 통해 정렬 기준을 선택할 수 있다
  • 정렬 변경 시 즉시 목록에 반영된다

무한 스크롤 페이지네이션

  • 페이지 하단 근처 도달 시 다음 페이지 데이터가 자동 로드된다
  • 스크롤에 따라 계속해서 새로운 상품들이 목록에 추가된다
  • 새 데이터 로드 중일 때 로딩 인디케이터와 스켈레톤 UI가 표시된다
  • 홈 페이지에서만 무한 스크롤이 활성화된다

상품을 장바구니에 담기

  • 각 상품에 장바구니 추가 버튼이 있다
  • 버튼 클릭 시 해당 상품이 장바구니에 추가된다
  • 추가 완료 시 사용자에게 알림이 표시된다

상품 검색

  • 상품명 기반 검색을 위한 텍스트 입력 필드가 있다
  • 검색 버튼 클릭으로 검색이 수행된다
  • Enter 키로 검색이 수행된다
  • 검색어와 일치하는 상품들만 목록에 표시된다

카테고리 선택

  • 사용 가능한 카테고리들을 선택할 수 있는 UI가 제공된다
  • 선택된 카테고리에 해당하는 상품들만 표시된다
  • 전체 상품 보기로 돌아갈 수 있다
  • 2단계 카테고리 구조를 지원한다 (1depth, 2depth)

카테고리 네비게이션

  • 현재 선택된 카테고리 경로가 브레드크럼으로 표시된다
  • 브레드크럼의 각 단계를 클릭하여 상위 카테고리로 이동할 수 있다
  • "전체" > "1depth 카테고리" > "2depth 카테고리" 형태로 표시된다

현재 상품 수 표시

  • 현재 조건에서 조회된 총 상품 수가 화면에 표시된다
  • 검색이나 필터 적용 시 상품 수가 실시간으로 업데이트된다

장바구니

장바구니 모달

  • 장바구니 아이콘 클릭 시 모달 형태로 장바구니가 열린다
  • X 버튼이나 배경 클릭으로 모달을 닫을 수 있다
  • ESC 키로 모달을 닫을 수 있다
  • 모달에서 장바구니의 모든 기능을 사용할 수 있다

장바구니 수량 조절

  • 각 장바구니 상품의 수량을 증가할 수 있다
  • 각 장바구니 상품의 수량을 감소할 수 있다
  • 수량 변경 시 총 금액이 실시간으로 업데이트된다

장바구니 삭제

  • 각 상품에 삭제 버튼이 배치되어 있다
  • 삭제 버튼 클릭 시 해당 상품이 장바구니에서 제거된다

장바구니 선택 삭제

  • 각 상품에 선택을 위한 체크박스가 제공된다
  • 선택 삭제 버튼이 있다
  • 체크된 상품들만 일괄 삭제된다

장바구니 전체 선택

  • 모든 상품을 한 번에 선택할 수 있는 마스터 체크박스가 있다
  • 전체 선택 시 모든 상품의 체크박스가 선택된다
  • 전체 해제 시 모든 상품의 체크박스가 해제된다

장바구니 비우기

  • 장바구니에 있는 모든 상품을 한 번에 삭제할 수 있다

상품 상세

상품 클릭시 상세 페이지 이동

  • 상품 목록에서 상품 이미지나 상품 정보 클릭 시 상세 페이지로 이동한다
  • URL이 /product/{productId} 형태로 변경된다
  • 상품의 자세한 정보가 전용 페이지에서 표시된다

상품 상세 페이지 기능

  • 상품 이미지, 설명, 가격 등의 상세 정보가 표시된다
  • 전체 화면을 활용한 상세 정보 레이아웃이 제공된다

상품 상세 - 장바구니 담기

  • 상품 상세 페이지에서 해당 상품을 장바구니에 추가할 수 있다
  • 페이지 내에서 수량을 선택하여 장바구니에 추가할 수 있다
  • 수량 증가/감소 버튼이 제공된다

관련 상품 기능

  • 상품 상세 페이지에서 관련 상품들이 표시된다
  • 같은 카테고리(category2)의 다른 상품들이 관련 상품으로 표시된다
  • 관련 상품 클릭 시 해당 상품의 상세 페이지로 이동한다
  • 현재 보고 있는 상품은 관련 상품에서 제외된다

상품 상세 페이지 내 네비게이션

  • 상품 상세에서 상품 목록으로 돌아가는 버튼이 제공된다
  • 브레드크럼을 통해 카테고리별 상품 목록으로 이동할 수 있다
  • SPA 방식으로 페이지 간 이동이 부드럽게 처리된다

사용자 피드백 시스템

상품 목록 - 새로고침 시 상태 유지

  • 새로고침 후 URL 쿼리에서 카테고리가 복원된다
  • 새로고침 후 URL 쿼리에서 옵션 설정이 복원된다
  • 새로고침 후 URL 쿼리에서 정렬 조건이 복원된다

1. 컴포넌트 기초 구조 설계

이번 프로젝트에서 컴포넌트를 설계하면서 가장 먼저 고민한 것은 클래스형으로 갈지, 함수형으로 갈지였다. 기존 클래스형 구현은 준일 코치님의 블로그와 예시를 통해 많이 접해봤고, 현재 대부분의 프레임워크, 특히 React가 함수형 중심으로 발전하고 있다는 점, 클로저를 활용해 상태와 렌더링 로직을 직접 구성해보고 싶다는 이유로 함수형을 선택했다. 또한 클로저를 활용해 상태 관리와 렌더링 로직을 직접 구성해보고 싶었기 때문에 함수형 패턴을 선택했다.

클래스형 구조를 다시 살펴보면, 모든 컴포넌트는 공통적으로 상태(state)를 가지고 있으며, setState가 호출되면 render가 재실행되어 DOM이 갱신된다. 그리고 setup, render, setEvent 같은 생명주기 메서드가 순서대로 호출된다. 컴포넌트마다 달라지는 부분은 상태의 내용, 렌더링 결과, 이벤트 로직뿐이다. 즉, 내부 로직(상태·렌더·생명주기)과 외부 로직(템플릿·이벤트)을 명확히 구분하는 구조가 핵심이었다.

class Component {
  $target;
  state;
  constructor ($target) {
    this.$target = $target;
    this.setup();
    this.render();
  }
  setup () {};
  template () { return ''; }
  render () {
    this.$target.innerHTML = this.template();
    this.setEvent();
  }
  setEvent () {}
  setState (newState) {
    this.state = { ...this.state, ...newState };
    this.render();
  }
}

함수형 패턴으로 전환하면서, 클래스 인스턴스가 하는 역할을 팩토리 함수로 대체했다. 이 방식은 new 없이 함수 호출만으로 독립적인 컴포넌트 인스턴스를 생성하고, 클로저를 활용해 상태를 안전하게 유지할 수 있다. 상태를 직접 노출하지 않고 접근자와 setState를 통해 관리함으로써 불변성과 명시적 업데이트를 보장했다.

export const Component = (component) => {
  return (selector, props = {}) => {
    let state = {};

    const setState = (newState) => {
	    ...
      render();
    };

    const render = () => {
      ...
      if (!isMounted) {
        component.onMounted?.({ ... });
      } else {
        component.onUpdated?.({ ... });
      }
    };

    const destroy = () => {
      component.onDestroy?.();
    };

    render();

    return { state, setState, destroy };
  };
};

이 구조에서는 컴포넌트 내부 상태(state)와 생명주기 로직(onMounted, onUpdated, onDestroy)이 클로저 안에서 안전하게 캡슐화된다. 외부에서는 상태를 직접 조작하지 못하고, setState를 통해서만 상태를 갱신할 수 있다. 예를 들어, 단순한 헤더 컴포넌트는 다음처럼 사용할 수 있다.

const Header = Component({
  template: () => `<p>헤더입니다.</p>`,
});

Header('#header');


<header id="header">
  <p>헤더입니다.</p>
</header>

이번 경험에서 얻은 중요한 교훈은 함수형 컴포넌트 설계 시, 내부 로직과 외부 로직을 명확히 구분하고, 상태를 안전하게 캡슐화하며, 생명주기를 체계적으로 관리해야 한다는 점이다. 특히 팩토리 함수를 사용하면 컴포넌트마다 독립적인 상태와 렌더링 로직을 유지할 수 있어, 클래스 기반과 달리 상속 구조 없이도 재사용과 확장이 가능하다.

또한 이번 설계 과정에서 얻은 또 다른 인사이트는, 공통 로직과 개별 로직을 명확히 분리하고, 상태 접근과 갱신을 함수로 통제해야만 복잡한 SPA에서도 예측 가능한 렌더링과 유지보수가 가능하다는 점이었다. 초기에는 감이 잡히지 않아 일부 로직을 급하게 작성했지만, 클래스를 분석하고 클로저를 활용한 함수형 구조로 변환하면서 전체 흐름을 이해하게 되었다.

자식 컴포넌트 렌더링

SPA 구조를 직접 구현하는 과정에서, 자식 컴포넌트 렌더링 흐름을 설계하면서 여러 문제가 발생했다. 문제의 근본 원인은 프레젠테이션 함수(HTML 문자열 반환 함수)와 컴포넌트 팩토리(컴포넌트 인스턴스를 생성하고 상태·라이프사이클·DOM 조작을 담당하는 함수)의 역할을 명확하게 구분하지 못한 것과, 라이프 사이클에 대한 몰이해에 있었다.

초기에는 문자열 반환 함수로 구상했다가, 이후 컴포넌트 마운트 함수로 전부 수정했는데, 수정하고 난 뒤에도 템플릿 내부에서 자식 컴포넌트를 호출하면 바로 문자열이 반환될 것이라고 생각했다. (ㅜㅜ) 실제로 구현한 컴포넌트는 문자열을 반환하는 단순 함수가 아니라, DOM에 직접 마운트되는 컴포넌트 인스턴스였다. 즉, 템플릿 렌더 단계에서 호출해 문자열처럼 사용할 수 없는 구조였음에도 불구하고, 아무 생각없이 React 방식처럼 “함수 호출 → UI 반환”으로 동작할 것이라 오해한 것이 문제의 시작이었다.

// 초기 작성 코드
template: () => {
	return `
		<div>${ChildComponent()}</div>
	`
}

당연히 원하는대로 동작하지 않았다… 지금 생각해보면 당연한데, 당시에 생각보다 이 사실을 깨닫는 데 오래 걸렸다…

해결을 시도하면서 여러 접근을 적용했다. 부모의 onMounted 훅에서 자식 컴포넌트를 생성하는 방식으로도 접근했으나, 이 방식은 초기 한 번만 실행되는 구조라서 부모 상태가 변해도 자식은 갱신되지 않았다. 이후 onUpdated 훅에서 자식 컴포넌트를 다시 생성하도록 변경했지만, 자식 내부의 상태 변경이 부모의 업데이트를 다시 유발하고, 부모의 업데이트가 다시 자식을 호출하는 무한 재귀 호출이 발생해 브라우저가 멈추는 상황까지 이어졌다. 부모와 자식간의 의존성 설계를 제대로 하지 못한 탓이었다.

이후에는 렌더 파이프라인을 명시적으로 통제하기 위해, 부모 컴포넌트에 자식을 마운트하는 함수인 children 함수를 추가했다. 자식 생성 로직은 mountChildren으로 추상화하여 사용했고, 부모는 매 렌더 마지막에 children 함수를 호출해 자식 컴포넌트를 마운트했다. 로컬 환경에서는 정상적으로 동작하는 것처럼 보였다.

// 부모 컴포넌트 코드
children: (context) => {
	mountChildren(ChildComponent, "#child-root");
}

하지만 실제 테스트에서는 간헐적인 실패가 반복되었다. 10번 중 3번 정도 실패하는 패턴이었고, 실패 시점도 매번 달랐다.

이것저것 찾아보면서 실패 원인 몇가지를 추정해봤다.

  1. 자식 컴포넌트의 라이프사이클 독립성이 없어졌다는 점이 문제다. 현재 구조에서는 부모 render() 호출 시마다 자식이 강제로 렌더링된다. 하지만 자식은 본래 자신의 onMounted와 onUpdated 기준으로 상태와 렌더를 관리해야 하는데, 부모에 종속되면서 라이프사이클이 깨지게 된다. 초기 렌더링에서도 문제가 발생한다. 자식은 처음 마운트될 때 한 번만 실행되어야 하지만, 부모 렌더에 따라 반복적으로 재생성되면서 상태가 초기화되거나 onMounted가 여러 번 호출된다. 현재 구조대로 가게 되면 라이프 사이클이 명시적으로 잡혀있지 않아, 브라우저 렌더 타이밍과 DOM 반영 지연의 영향도 받았으리라 생각된다. 즉, 전체적인 렌더 파이프라인이 구조적으로 불안정한 상태다.

  2. 모든 컴포넌트를 컴포넌트 팩토리 방식으로 통일해 구현했다. 구조 설계에도 문제가 있었다고 생각한다. 단순히 텍스트·HTML만 구성하면 되는 프레젠테이션 단위까지도 모두 인스턴스 기반으로 만들다 보니, 불필요한 상태·라이프사이클·DOM 책임을 갖게 되었다. 이는 자연스럽게 디버깅의 난이도를 높이는 효과를 가지고 왔다. (…) 모든 상황에 같은 로직을 적용할 필요는 당연히 없는건데, 좁은 시야로 코드를 작성한 점이 아쉽게 느껴진다.

이 문제들을 인지한 시점에는 이미 코드베이스가 커진 뒤였고 전체 구조를 다시 설계하기에는 시간이 부족했다. (무려 과제 제출 3시간 전에서야 원인을 제대로 파악했다…) 테스트 실패율도 전체적으로 치명적이지 않아 우선 과제를 마무리하고 이후 리팩토링을 진행하기로 결정했다. 하지만 이번 경험을 통해 컴포넌트를 설계할 때 먼저 렌더링 모델과 라이프사이클 구조를 정의하지 않으면 구조적 문제가 필연적으로 발생한다는 사실을 명확히 깨달았다. 라이프사이클에 대한 얕은 이해 상태에서 안일하게 설계했고 기술적 기반 없이 구현을 시작한 것이 문제의 핵심이었다.

다음에 구조를 다시 설계한다면 아래 항목을 가장 먼저 명확하게 정의할 것이다.

  • 컴포넌트 인스턴스의 라이프 사이클 순서 정의
  • 부모–자식 렌더링 순서 및 업데이트 우선순위 정의
  • 단순 UI 조각은 문자열 렌더러로 구성하고, 상태·라이프사이클은 반드시 필요한 컴포넌트에만 부여

라우터

export const createRouter = () => {
	let routerInstance;
	const initRouter = () => {...}
	const getRouter = () => {...}
	return {}
}

이번 SPA 라우터 구현에서 핵심은 싱글톤 패턴을 이용해 전역 단일 인스턴스로 라우터를 관리하고, History API를 활용해 URL 변경과 라우팅을 처리하는 구조였다.

initRouter로 초기화 후 Router()로 접근하고, 경로 매칭은 정규식을 통해 :param과 * 패턴을 지원하도록 했다. pushState나 replaceState 호출 시 라우팅 함수를 직접 실행하고, popstate 이벤트로 뒤로가기/앞으로가기 동작을 감지하는 방식으로 설계했다. 쿼리 파라미터와 동적 파라미터는 각각 유틸 함수로 분리했고, 실제 DOM 마운트는 라우트 객체의 page 함수에 위임해 단순화했다. 전체적으로 책임이 명확하게 분리되어 있어 현 요구 범위에서는 큰 문제 없이 안정적으로 동작한다.

특히 이번에 새롭게 알게 된 점은 pushState나 replaceState로 URL을 변경해도 popstate 이벤트는 발생하지 않는다는 사실이다. 뒤로가기/앞으로가기만 popstate를 트리거하며, SPA에서는 URL 변경 시점을 감지하려면 항상 라우터가 직접 호출해주는 방식으로 처리해야 한다.

다만 a 태그 클릭 시에도 페이지가 정상적으로 렌더링되었는데, 이 부분이 처음에는 이해되지 않았다. popstate 이벤트나 push 함수 호출이 발생하지 않았는데도 라우터의 로직이 올바르게 동작했기 때문이다. 찾아보니, 실제로는 a 태그 클릭으로 브라우저가 페이지를 전체 리로드했고, 그 과정에서 라우터가 초기화되며 setup 함수가 호출되어 해당 URL을 처리한 것이었다. (아하!!! 생각해보니 당연하군.. 이라고 생각함)

스토어

export const createStore = (initialState) => {
	let state = initialState;
	const getState = () => {};
	const setState = () => {};
	const subscribe = () => {};
	const selector = () => {};
	return {
    getState,
    setState,
    subscribe,
    select,
  };
}

이번 프로젝트에서 구현한 스토어는 기본적으로 옵저버 패턴을 활용해 상태를 구독하고, 선택적 조회(selector)를 지원하는 구조로 설계했다. 초기 상태를 객체나 함수로 받아 처리하고, 상태 변경 시 등록된 리스너를 호출하는 단순한 전역 상태 관리 기능 정도를 제공했다. 옵저버 패턴을 사용한 스토어의 기본 구조 자체는 단순하고 평범했지만, 실제 사용하면서 구조적 문제가 드러났다.

가장 큰 문제는 현재 스토어가 전역 인스턴스로 관리되지 않고, 호출할 때마다 새로운 스토어 인스턴스를 생성하는 방식이라는 점이다. 이로 인해 CartModal 내에서 스토어를 생성하고, 다른 컴포넌트에서 import해 사용하는 구조가 되었는데, 장바구니 모달과 헤더처럼 동일 상태를 사용해야 하는 컴포넌트 간의 상태 공유가 일관되지 않고 의존성이 복잡해졌다. 이 구조는 유지보수와 확장성 측면에서 큰 부담이 된다.

만약 이 방식을 유지하려면, 최소한 스토어 생성 지점이 CartModal이 아니라 상위 컴포넌트(예: main)였어야 한다고 생각한다. Context API처럼 하위 컴포넌트에 내려주는 형태까지는 아니더라도, 논리적으로 특정 하위 컴포넌트 내부에 스토어를 두는 것은 적절하지 않았다.

더 좋은 해결책은 스토어를 전역 단일 인스턴스로 관리하고, useStore와 같은 접근 함수를 통해 동일 키의 상태를 조회하고 구독하도록 설계하는 것이라고 생각한다. 옵저버 패턴과 selector 기능은 그대로 유지하되, 상태 접근과 공유 방식을 일관되게 하는 것이 더 안정적이었을 것이다.

시간이 부족해 급하게 구현한 코드라 퀄리티가 아쉽지만, 이번 경험을 통해 상태 공유와 접근 방식, 스토어의 생성 위치 등을 신중하게 설계하지 않으면 컴포넌트 간 의존성이 복잡해지고 유지보수가 어려워진다는 점을 분명히 깨달았다. 앞으로는 이러한 구조적 고민에 충분한 시간을 쏟아야겠다.

이번 과제를 진행하면서 가장 아쉬웠던 점은 시간 배분과 부족한 자료 조사라고 생각합니다. 평소에도 코드를 작성할 때 일단 완벽하게 알기 전에 시작하고 보자, 하고 행동하는 경향이 있는데, 이번 과제는 초안을 작성한 이후에 다시 살펴볼만한 시간이 나지 않았고, 초반에 기능 구현을 AI를 전혀 사용하지 않고 하려다가 시간 배분이 꼬여버렸습니다... 물론 그 과정에서 스스로 성장한 부분이 많지만, 시간 분배를 더 잘했다면 어땠을까 하는 아쉬움이 남습니다.

기술적 성장

자랑하고 싶은 코드

라우터 구현이 가장 깔끔하게 작성되었다고 생각합니다. 싱글톤 패턴으로 전역 인스턴스를 관리하고, 정규식을 통해 :param과 * 패턴을 지원하도록 설계한 부분도 마음에 듭니다. 특히 pushState/replaceState와 popstate의 역할을 명확히 구분하고, 쿼리 파라미터와 동적 파라미터를 리턴하는 함수를 각각 유틸 함수로 분리한 점에서 책임이 잘 분리되었다고 느껴집니다. 함수형 Component 생성 함수도 만족합니다. 초반의 혼란을 거쳐 최종적으로 정리된 구조는 그래도 깔끔한 편이라고 생각합니다. 클래스에서 했던 것을 함수형으로 재구성하면서 클로저의 장점을 충분히 살릴 수 있었고, onMounted, onUpdated, onDestroy 같은 생명주기 메서드가 직관적으로 구성되어 마음에 듭니다. 비록 자식 컴포넌트 렌더링에서 문제가 있었지만, 팩토리 함수 자체의 설계 원칙은 나쁘지 않았다고 생각합니다.

개선이 필요하다고 생각하는 코드

  • 자식 컴포넌트 렌더링 구조가 가장 개선이 필요하다고 생각합니다. 부모의 onUpdated에서 계속 자식을 재생성하는 방식은 근본적으로 라이프사이클 독립성을 해쳤고, 간헐적인 버그가 나타났습니다. 다음에 다시 설계한다면, 부모와 자식의 렌더링 순서를 명시적으로 정의하고, 자식은 최초 마운트 이후 자신의 상태 변경으로만 업데이트되도록 분리할 것 같습니다.

  • 스토어 구조도 개선이 필요합니다. 현재는 CartModal 내에서 스토어를 생성하는 방식인데, 이로 인해 스토어가 다른 컴포넌트와 공유되지 않거나 의존성이 뒤얽혔습니다. 스토어를 전역 싱글톤으로 관리하고, getStore(key) 같은 접근 함수를 통해 일관되게 상태를 조회하는 방식으로 개선하고 싶습니다. 옵저버 패턴과 selector 개념은 좋은데, 접근 방식에 신경을 못 쓴 점이 아쉽습니다.

  • 모든 UI를 인스턴스 기반 컴포넌트로 만든 점도 아쉽습니다. 단순한 텍스트나 HTML 조각까지도 팩토리 함수로 감싸다 보니 불필요한 상태 관리와 라이프사이클이 추가되었습니다. 프레젠테이션 레이어와 상태 관리 레이어를 구분했다면, 코드베이스가 훨씬 명확하고 디버깅이 쉬웠을 거라고 생각합니다.

학습 효과 분석

가장 큰 배움은 "설계 없이 구현을 시작하면 나중에 치른다"는 사실입니다. 제가 컴포넌트의 라이프사이클을 제대로 정의하지 않고 시작했기 때문에, 부모-자식간의 렌더링 문제가 생겼고, 원인을 찾는 데에도 오랜 시간이 걸렸다고 생각합니다. 만약 처음부터 라이프 사이클을 그림으로라도 그려보고 순서를 정의했다면, 훨씬 빠르고 안정적으로 진행했을 것 같습니다.

또 바닐라 js로 SPA 프로젝트를 구현하면서, 평소에 신경을 못 썼던 자바스크립트 문법에 대해 학습하는 계기가 되어 좋았습니다. 또, 상태 관리와 라이프사이클 같은 부분은 React나 Vue 같은 프레임워크의 원리를 조금이나마 이해하는 계기가 되었습니다.

AI 활용 경험 공유하기

저는 이번 과제 초반에 AI를 거의 사용하지 않으려고 했습니다. 순수하게 스스로 구현해보고 싶은 마음에 코드를 직접 작성하고 디버깅했는데, 결과적으로 시간이 예상보다 훨씬 오래 걸렸습니다... 😅 아쉬웠던 점은 AI를 조금 더 활용했다면 구조를 검토받고 개선할 여유가 있었을 텐데, 기능 구현에만 매달리다가 시간이 남지 않았다는 것입니다. 이번 과제의 핵심은 SPA 프로젝트 개발이었는데, 우선 순위를 잘못 잡았던 것 같아 아쉬움이 남습니다. 다음에 다시 비슷한 과제를 받는다면, 기본 구조 설계에 시간을 더 많이 투자해서 직접 하고, 구체적인 기능 구현은 AI에게 좀 더 많은 비율을 맡길 것 같습니다.

그 외에 느낀 점은, 만고불변의 진리인 AI가 만능이 아니다라는 것입니다. 질문하고 있는 제가 구조와 설계에 대한 이해가 없으니 올바른 질문을 하지 못하게 되고, 이미 제 주관이 들어간 질문을 하게 되어 구조에 대한 올바른 피드백을 받기 힘들었습니다. 다만 단순한 기능 구현과 같은 부분은 매우 잘 해주었습니다. 단순 구현 외에 설계와 관련된 부분은, AI에게 물어보더라도 스스로 참고 자료를 많이 찾아보고, 지식을 쌓은 뒤에 질문해야 되겠다는 생각을 했습니다.

수고했어요 희재! 프레임워크 없이 SPA를 직접 만들어보면서 React가 왜 그렇게 설계되었는지 깊이 체험하는 시간이 되었기를 바랍니다.

코드를 보니 함수형 컴포넌트 패턴을 클로저로 구현하면서 생명주기를 체계적으로 설계하려는 시도가 인상깊었어요. 아마 왜 초창기 프레임워크들이 모두 접근방식을 함수형이 아니라 클래스로 했을지 이해가 되지 않았을까 생각해요. 분명 함수형으로 작성을 하면 훨씬 가독성 측면에서는 뛰어나지만 내부 상태관리나 UI만 관리를 한다는게 참 쉽지 않거든요. 이로써 React는 이러한 과정을 어떻게 해결했을까? 하는 고민을 이번 과제때 제대로 느껴볼 수 있겠네요.

희재의 자기반성과 성찰이 가득한 회고가 너무 좋았어요. 과제를 잘하고 못하고 보다고 과정안에서 내가 어떤 과정을 거쳤고 어떤 시도와 문제 그리고 고민이 있었는지를 잘 볼 수 있어서 아주 좋았습니다. 자신의 맥락을 구체적으로 전달한다는 것은 개발자에게 정말 중요한 스킬이니 지금의 회고는 너무 잘 작성했어요!

Q) 부모-자식 컴포넌트 렌더링 구조 개선안

희재가 겪은 문제의 핵심은 "자식 컴포넌트가 부모 render()에 종속되면서 라이프사이클 독립성을 잃었다"는 거죠. 맞아요. 각자 컴포넌트 렌더링은 스스로에게 있는데 부모 렌더링에 자식을 렌더링 기능을 기술해야 하다보니 문제가 생긴것이죠.

사실상의 정답으로써 React의 방식은 컴포넌트를 함수화 하고 결과값을 렌더링이 아닌 값으로 만들게 해서 부모가 렌더링시 자연스레 자식의 함수호출을 통해 최종적인 값을 생성해서 부모가 렌더링이 되면 자식도 함께 props와 함께 렌더링이 되고 자식이 업데이트가 필요할땐 스스로 업데이트를 하도록 구분한거죠.

희재의 구조에서 만들고 싶었다면 렌더링을 하는 과정을 내부로 추상화해서 숨기고 자식 컴포넌트를 template에서 약속된 방식으로 렌더링을 하거나 교체할 수 있는 방법을 만들어야 합니다. 다만 이 경우 템플릿은 문자열이고 컴포넌트는 객체인데 렌더링을 섞어 넣으려고 하면 지금의 문자열 구조를 DOM으로 바로 그리려고 하면 매우 복잡해 진다는 거에요.

다음 시간에 React가 해결한 부분인 DOM과 문자열로 직접 사용하는 것이 아닌 별개의 데이터 구조를 만들고 DOM 렌더링과 DOM의 구조의 변경이라는 부분을 값으로 분리해서 처리한 방법이 어떤 것인지 잘 이해할 수 있을거에요. 현대의 대부분의 에디터, figjam, notion등의 복잡한 구조들은 DOM을 직접 수정하는것이 아니라 별도의 데이터 구조를 만들어두고 렌더링 엔진을 만드는 방식으로 구성이 된답니다.

Q) 프레젠테이션과 상태 관리 계층 분리 단순한 UI 조각은 문자열 렌더러로, 상태가 필요한 부분만 컴포넌트로 나누는 방식이 맞는지 확인하고 싶습니다. 이렇게 하면 어떤 이점이 생기고, 실제로 어디까지를 프레젠테이션으로 분류해야 할까요?

=> 초기 프레임워크들은 지금의 구조처럼 문자열을 바탕으로 일부를 수정하거나 혹은 DOM의 구조를 그대로 이용한채로 상태의 변경에 따라 DOM을 변경하는 방식으로 구현했습니다. 이렇게 하면 매우 가벼우면서도 효율적인 구조를 가질 수 있으나 DOM구조와 실제 컴포넌트 구조가 달라지는 Portal 기능이나 서버사이드 렌더링과 같이 문자열로 렌더링을 하는 등 DOM 구조에 의존하지 못하는 경우가 생겨나죠. 그래서 그냥 DOM 전체의 구조를 하나의 객체로 관리하는 방식으로 다루는게 훨씬 낫다는 것을 알게 됩니다. 그래서 여러가지로 고민하지 말고 그냥 일관된 형태 그리고 단순한 Node의 결합으로써 데이터를 표현하죠. 희재의 방식은 템플릿이 정해져있고 변화가 많지 않다면 훨씬 가벼운 방식으로 처리할 수 있는데 여러가지 유연성을 담보하기가 어렵습니다. React가 결국은 라이브러리의 비용이 비싸고 더 느림에도 쓰는건 일관성있는 구조로 인해 같은 방식으로 여러 환경에서 상용할 수 있다는 점이죠.
