  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

과제 회고

과제를 진행하면서 느낀 점, 배운 점을 자유롭게 작성해주세요.

처음엔 프로젝트의 규모가 작아 금방 끝날 것이라 생각했으나, 예상보다 훨씬 오래 걸렸다.

구조 설계를 하는 과정에서 고민이 많았고, 명확한 정답이 없는 과제인 만큼 고민할 부분이 많았다. 또한 고민을 하면 할수록 코드가 나아지는 것이 느껴졌기 때문에 오히려 더 욕심이 나 많은 시간을 투자하게 됐다.

진행 과정

Phase 0: Before 기준 테스트 코드 작성 및 코드 분석

  • 리팩토링 프로젝트인 만큼 최종 기준을 잘 잡아두는 것이 중요하다고 생각했다.
  • 비즈니스 로직과 UI 로직이 섞인 것을 신경 쓰지 않고, 당장의 코드 동작 그대로 테스트 코드를 작성해 기준을 잡으려 했다.
  • 이후 Before 프로젝트의 코드를 상세히 분석하며, 디자인 시스템 관련된 글을 많이 읽었다.

Phase 1: 디자인 시스템의 근간이 되는 라이브러리들 세팅

  • TailwindCSS & Lint 설정
  • 기존 UI 컴포넌트 shadcn/ui로 변경

Phase 2: 구조적 리팩토링 (Refactoring)

  • Directory Structure 개선: atoms/molecules/organisms의 복잡한 중첩 구조를 버리고, components/ui, components/composed, components/layout로 평탄화했다.
  • Service Layer 순수 함수화: 기존의 클래스 기반이나 상태를 가진 서비스 로직을 순수 함수로 변경하여 테스트 용이성을 높였다.
  • Custom Hooks 도입: 컴포넌트 내부에 산재된 useEffect와 상태 로직을 useModal, useForm 등의 커스텀 훅으로 추출했다.
  • 이 과정에서 비즈니즈 로직을 조금씩 UI 컴포넌트에서 페이지나 훅으로 옮겼고, 테스트코드도 그에 맞춰 업데이트했다.

Phase 3: 패턴 도입 및 고도화

  • Compound Component Pattern: Header나 UserInfo 같이 연관된 컴포넌트들을 compound 컴포넌트 패턴으로 재구현하여 사용성을 개선했다.
  • Table 컴포넌트 개선: 테이블을 composed 방식에서 ui 컴포넌트 방식으로 변경해, 순수 UI 로직만 남겨 비즈니스 로직을 완전히 제거했다.

Phase 4: 안정성 및 완성도 향상

  • DataTable: 순수 UI만 남긴 Table 컴포넌트를 이용해 DataTable 컴포넌트를 개발했다. 해당 컴포넌트는 컬럼과 데이터, 옵션 등을 받아 주어진대로 테이블을 그려내는 역할을 한다. Pagination 컴포넌트와 훅도 추가했다.
  • React Hook Form + Zod
  • Type Safety: 프로젝트 전반에 걸쳐 any 타입을 제거하고, Post, User 등 구체적인 인터페이스를 정의했다.
  • Design Token System Refinement: 초기에는 단순했던 토큰을 primitives와 semantic으로 분리하여 다크 모드 대응과 테마 변경이 용이하도록 개선했다.

Before 패키지에서 발견한 문제점

1. 의미가 불분명한 변수명과 데이터 흐름

  • actualContent, actualType과 같이 맥락을 알기 어려운 변수명이 사용되고 있었다.
  • Prop으로 전달받은 content를 바로 사용하지 않고, 내부 로직을 거쳐 변환된 값을 다시 변수에 할당하는 방식이어서 데이터 흐름을 추적하기 어려웠다. 코드를 처음부터 끝까지 읽지 않으면 변수의 역할을 파악할 수 없었다.

2. 타입 안정성 부족

  • as 키워드를 통한 무분별한 타입 단언과 any 타입의 잦은 사용으로 인해 타입 안정성이 낮아졌다.

3. 스타일 적용 방식의 비표준화

  • CSS 파일, 인라인 스타일(style={{...}}), 문자열 합성을 통한 클래스 명명(badge-${type}), 조건부 연산자를 통한 클래스 분기처리(error && 'error') 등 다양한 스타일링 방식이 기준없이 혼재되어 있었다.
  • 이러한 비표준화는 코드의 가독성을 떨어뜨리고, 디자인 변경 시의 수정 지점을 파악하기 힘들게 만듦.

4. 비즈니스 로직과 UI 로직의 강한 결합

  • UI 컴포넌트 내부에서 데이터를 가공하거나 특정 상태를 판단하는 로직이 포함되어 있었다.
  • 이로 인해 컴포넌트가 특정 도메인 로직에 종속되어, 다른 곳에서 재사용하기 어려운 구조였다.

5. 도메인 분리의 부재

  • Post와 User는 화면에 그려지는 형태(Card, List 등)가 비슷할 뿐, 본질적으로 다른 도메인이다.
  • 그러나 기존 코드는 '보여지는 모습'이 같다는 이유로 하나의 컴포넌트에서 두 도메인의 로직을 분기 처리(if (type === 'user') ...)하여 다루고 있었다.
  • 이는 단일 책임 원칙에 어긋나고, 한 도메인의 변경이 다른 도메인에 영향을 줄 수 있는 구조였다.
  • 로직 통합의 기준은 '어떻게 보이는가'가 아니라 '무엇을 보여주는가'여야 한다.
  • 중복 코드가 발생하더라도 도메인별로 분리하는 것이 옳다고 판단했다.

개편 과정에서 집중한 부분

변경에 유연한 구조를 만들기 위해 가장 노력했다.

비즈니스 로직과 UI 로직을 분리하는 것에 집중했고, 특정 로직이 어디에 위치해야 하는지, 기준을 어떻게 세우는 게 좋을지 많이 고민했다. 단순한 당장의 프로젝트 구조와 앞으로의 확장성 사이의 중간점을 찾는 것이 어려웠.

처음엔 최대한 현재의 프로젝트 규모에 맞게 설계하고, 앞으로의 확장 가능성을 열어두는 정도로만 작업했다. 일부러 토큰과 디렉토리 구조를 최대한 단순하게 가져갔으나, 현재의 규모에서도 너무 단순한 구조로 갔더니 문제가 생겨 처음의 구조에서 조금씩 다듬어가며 진행했다. (후술할 내용이지만 특히 디자인 토큰이 그랬다.)

사용한 기술 스택 경험

TailwindCSS를 제외하고는 거의 모든 기술 스택을 사용해본 경험이 없었어서 전체적으로 새로운 경험이었다.

TailwindCSS

  • 사이드 프로젝트 및 업무에서 종종 사용해본 경험이 있어 익숙했다.
  • 다만 간단한 설정 외에 디자인 토큰 설계를 제대로 해본 적도, @theme 파일을 여러 개 만들어가며 설정해본 경험이 없었기에, 새로웠다.

shadcn/ui

  • 유명하다는 건 알고 있었지만, 정확히 어떤 이유 때문에 유명한지 제대로 이해하지 못했었다.
  • 발제를 통해 "라이브러리가 아닌 코드"라는 철학 때문에 많은 사랑을 받게 되었다는 걸 알게 되어 매우 흥미로웠다.
  • 필요한 부분만 가져와서 우리 프로젝트에 맞게 수정(커스터마이징)하는 것이 자유로워 좋았다.

React Hook Form

  • 사용해본 적이 없어 정확한 장점을 알지 못했다.
  • 가장 먼저 와닿은 편의성은 상태 관리와 change 로직을 따로 작성할 필요가 없다는 것이었다.
  • 그 외에 에러 핸들링과 성능 최적화 등의 이점이 있다는 걸 확인했다.

Zod

  • 개인적으로 스키마 작성 과정이 순수하게 즐거웠다. 문법이 간결하고 가독성이 좋았다.
  • 다양한 유효성 검사를 매우 간결한 문법으로 전부 처리할 수 있다는 점이 좋았다.

Storybook

  • 스토리북 자체는 이전 과제를 통해 접해본 적이 있어 낯설지 않게 작업했다.
  • Preview를 통해 전역적인 설정 세팅 로직을 처음으로 작업해 보았다.
  • 데코레이터를 통해 리액트 문법으로 전역적인 환경 세팅을 하는 과정이 재미있었다.

어려웠던 점과 해결 방법

1. 토큰 세팅

구조 설정에 가장 많은 시간을 투자한 부분이다. 처음에 디자인 토큰이란 개념 자체가 감이 안 와서 애를 먹었다.

시도 1: index.css에 모든 디자인 토큰 입력

@theme {
  /* Colors - Primary */
  --color-primary: #1976d2;
  --color-primary-hover: #1565c0;
  --color-primary-foreground: #ffffff;
  
  /* Spacing */
	...
  /* Border Radius */
  ...
  /* Font Sizes */
  ...
  /* Font Weights */
  ...
}

나쁘지 않다고 생각함. 다만 디자인 토큰이 늘어나게 될 때를 생각해보면, 한 파일에 너무 많은 정보가 들어갈 것 같았고, 각 범주의 토큰들이 따로 구조화된 형태 없이 주석으로만 영역이 나뉘어 안정성이 부족할 것 같다고 느꼈다.

시도 2: tokens 디렉토리 하위에 ts 파일 생성

tokens
	ㄴ colors.ts
	ㄴ shadow.ts
	ㄴ radius.ts
	ㄴ spacing.ts
	ㄴ ...

export const radius = {
  none: '0px',
  sm: '2px',
  default: '3px',
  md: '4px',
  lg: '10px',
  xl: '12px',
  '2xl': '16px',
  full: '50%',
} as const

토큰을 상수값으로 관리하는 게 더 나을 것 같아 변경해 보았다. ts 파일이나 JSON 파일로 토큰을 관리하면 특정 라이브러리에 종속되지 않고, 문자열을 통해 명시적으로 토큰을 설정할 수 있다. 하지만 이렇게 하면 TailwindCSS를 사용하는 의미가 없어지는 것 같았고, 사용하는 곳에서 한번의 처리과정을 거쳐야 한다는 단점이 있다. | 예를 들어, p-[radius[none]] 이런식? 또, 토큰의 단위가 커질수록 결국은 CSS 모듈 분리와 비슷해질 것 같았다.

시도 3: @theme 사용하여 파일 분리

tokens
	ㄴ colors.css
	ㄴ shadow.css
	ㄴ radius.css
	ㄴ spacing.css
	ㄴ ...

관심사별로 확실히 분리되어 유지보수가 쉬울 것 같았으나, 오버 엔지니어링이라는 생각이 들었다. 어차피 모든 토큰을 한 번에 import 해야 하고, TailwindCSS는 자체적으로 사용되는 클래스만 빌드해야 하기에 파일을 나누는 것에서 큰 이점을 못 얻을 것 같다는 생각이 들었다. 각 파일마다 @theme 블록이 반복되는 단점도 있었는데, 불필요하게 파일을 나눠 보일러플레이트 코드만 늘어난 거 아닌가 싶어져 첫번째 방법으로 돌아가게 됐다.

최종 해결 결국 첫 번째 방법(index.css 통합)으로 돌아간 뒤 큰 문제없이 개발을 진행했다. 그러다 다솜님과 대화하는 과정에서 다솜님이 TailwindCSS의 :dark 수식어도 전부 토큰으로 정의해서 사용했다는 얘기를 듣게 됐다. 이 얘기를 듣고서야 불현듯 아 이렇게 사용하려고 토큰이 필요한거구나를 깨닫게 됐다. 이 대화를 통해 토큰에서의 primitives와 semantic 분리의 유용함을 깨달았다. 이후 내 과제에서도 Primitives에서 색상을 정의하고, Semantic에서 background와 같은 이름을 붙인 뒤, 다크 모드 여부에 따라 색상이 분기 처리되도록 했다. 이를 통해 일일이 dark: 수식어를 붙일 필요가 없어졌고, 클래스명만 보고도 용도를 알 수 있게 되었다. 이전에는 만약 dark 모드의 색상이 변경되는 일이 생겼다면, 모든 코드를 수정했어야 했을테지만, 위와 같이 로직을 변경해 디자인 토큰만 변경하면 자동으로 모든 곳에 적용되게 할 수 있었다.

이 과정을 통해 디자인 토큰이 무엇이고, 어떤 용도를 가지며, 어떤 효용성이 있는지를 몸소 체감하게 되었다.

2. 뱃지 렌더링 로직

초기 코드: renderStatusBadge 함수

const renderStatusBadge = (status: string) => {
    const statusVariant =
      status === 'published'
        ? 'success'
        : status === 'draft'
          ? 'warning'
          : status === 'archived'
            ? 'secondary'
            : status === 'pending'
              ? 'info'
              : 'destructive';
    const statusLabel =
      status === 'published'
        ? '게시됨'
        : status === 'draft'
          ? '임시저장'
          : status === 'archived'
            ? '보관됨'
            : status === 'pending'
              ? '대기중'
              : '거부됨';
    return <Badge variant={statusVariant}>{statusLabel}</Badge>;
  };

AI가 처음 작성해준 코드였다. 삼항 연산자의 중첩으로 가독성이 좋지 않았고, variant와 label을 한 함수에서 동시에 처리하고 있었다.

개선 1: 함수 분리

const renderStatusBadge = (status: PostStatus) => {
	const statusVariant = getVariantByStatus(status);
	const statusLabel = getLabelByStatus(status);
	return <Badge variant={statusVariant}>{statusLabel}</Badge>;
}

const getVariantByStatus = (status: PostStatus) => {
  switch (status) {
    case 'published':
      return 'success';
    case 'draft':
      return 'warning';
    case 'archived':
      return 'secondary';
    case 'pending':
      return 'info';
    default:
      return 'destructive';
  }
};

const getLabelByStatus = (status: PostStatus) => {
	...
}

getVariantByStatus, getLabelByStatus로 함수를 분리하고 switch 문을 사용하여 가독성을 높였다. 하지만 여전히 함수를 통해 JSX를 반환하여 화면 렌더링 코드가 산재되는 느낌을 받았다. 또한 status와 variant, label은 통합되어 사용되는 변수인데 각각의 처리 로직이 분리되어 있어 유지보수성을 해칠 것 같았다.

최종 해결: 상수 파일로 관리

// /constants/badge-mapping.ts
export const POST_STATUS_BADGE: Record<PostStatus, { variant: BadgeVariant; label: string }> = {
  published: { variant: 'success', label: '게시됨' },
  draft: { variant: 'warning', label: '임시저장' },
  archived: { variant: 'secondary', label: '보관됨' },
  pending: { variant: 'info', label: '대기중' },
  rejected: { variant: 'destructive', label: '거부됨' },
};

// PostManagement.tsx
<Badge variant={POST_CATEGORY_BADGE[post.category]}>{post.category}</Badge>

상태에 따른 색상과 라벨 정의는 값 계산이나 UI 위주의 로직이 아닌 비즈니스 로직이라고 판단했다. 따라서 위의 로직은 언제든지 변경될 수 있기에 UI 로직과 확실하게 분리할 필요가 있다고 느껴졌다. 위 매핑 값을 상수로 분리함으로써 UI는 주어진 값을 그리기만 하면 되도록 변경했고, Record 타입을 사용하여 타입 안정성을 높였다.

아쉬운 점

이번 과제에서는 디자인 시스템이 주제인 만큼 별도의 패키지로 분리하는 방식을 시도해보고 싶었으나, 물리적인 시간이 부족해서 시도해보지 못했다. 패키지 분리를 해본 적이 없어 한번 해보고 싶었는데, 이번 과제가 정말 적절한 기회였다고 생각했기에 아쉬움이 남는다. 다음에 여유가 생기면 꼭 시도해보고 싶다. 또, 완전히 까먹고 있었는데... 접근성에 대한 고려를 거의 하지 않았다. (...) shadcn 라이브러리를 사용했으니 어느정도의 접근성은 이미 처리가 되어있겠지, 기능 먼저 구현 한 뒤에 마지막으로 살펴봐야겠다고 생각했었다. 그 과정에서 우선 순위가 나도 모르게 밀려 작업을 하지 않았다.

리뷰받고 싶거나 질문하고 싶은 내용

  • CVA의 복잡도 관리
    • 컴포넌트의 Variant가 많아질수록 CVA 설정 객체가 무거워집니다. Button 컴포넌트의 경우 size, variant, outline, ghost 등 조합이 많아지는데, 이를 쪼개서 관리하는 패턴이 있나요?
  • Tailwind 클래스 가독성
    • TailwindCSS는 확실한 장점과 편리함이 있지만,className이 길어져 가독성이 저하되는 문제도 있다고 생각합니다. 실무에서 이를 해결하기 위해서는 보통 어떤 방식을 사용하시나요? (예: @apply 사용, 변수로 분리, 혹은 그대로 두기)
  • 디렉토리 구성 방식
    • types/ 디렉토리에 Post, User 인터페이스를 정의했습니다. 하지만 컴포넌트에서만 쓰이는 Props 타입(PostCardProps 등)은 컴포넌트 파일 안에 두었습니다.
    • 모든 타입을 types/로 모으는 것과, 사용하는 곳에 두는 것(Colocation) 중 어느 쪽을 선택할지는 보통 어떤 기준으로 결정되나요? 코치님의 기준은 뭔가요?
    • (질문을 적다보니 타입 정의 파일에만 국한되는 질문이 아닌 것 같은데, 테스트나 스토리북 등 다양한 영역에서 디렉토리의 설계 기준이 듣고 싶습니다!)
  • 접근성 처리
    • 과제를 진행하면서 기능 개발 (aka 돌아가는 코드)에 중점을 두다보니 접근성 개선에 대한 처리가 미비했습니다.
    • 현업에서도 리소스 문제로 접근성이 후순위로 밀리는 경우가 많을 것 같은데, 개발 프로세스 단계에서 접근성 준수를 자연스럽게 강제하거나 챙길 수 있는 효율적인 워크플로우나 팁이 있을까요?
