과제 체크포인트

기본과제

목표 : 전역상태관리를 이용한 적절한 분리와 계층에 대한 이해를 통한 FSD 폴더 구조 적용하기

  • 전역상태관리를 사용해서 상태를 분리하고 관리하는 방법에 대한 이해
  • Context API, Jotai, Zustand 등 상태관리 라이브러리 사용하기
  • FSD(Feature-Sliced Design)에 대한 이해
  • FSD를 통한 관심사의 분리에 대한 이해
  • 단일책임과 역할이란 무엇인가?
  • 관심사를 하나만 가지고 있는가?
  • 어디에 무엇을 넣어야 하는가?

체크포인트

  • 전역상태관리를 사용해서 상태를 분리하고 관리했나요?
  • Props Drilling을 최소화했나요?
  • shared 공통 컴포넌트를 분리했나요?
  • shared 공통 로직을 분리했나요?
  • entities를 중심으로 type을 정의하고 model을 분리했나요?
  • entities를 중심으로 ui를 분리했나요?
  • entities를 중심으로 api를 분리했나요?
  • feature를 중심으로 사용자행동(이벤트 처리)를 분리했나요?
  • feature를 중심으로 ui를 분리했나요?
  • feature를 중심으로 api를 분리했나요?
  • widget을 중심으로 데이터를 재사용가능한 형태로 분리했나요?

심화과제

목표: 서버상태관리 도구인 TanstackQuery를 이용하여 비동기코드를 선언적인 함수형 프로그래밍으로 작성하기

  • TanstackQuery의 사용법에 대한 이해
  • TanstackQuery를 이용한 비동기 코드 작성에 대한 이해
  • 비동기 코드를 선언적인 함수형 프로그래밍으로 작성하는 방법에 대한 이해

체크포인트

  • 모든 API 호출이 TanStack Query의 useQuery와 useMutation으로 대체되었는가?
  • 쿼리 키가 적절히 설정되었는가?
  • fetch와 useState가 아닌 선언적인 함수형 프로그래밍이 적절히 적용되었는가?
  • 캐싱과 리프레시 전략이 올바르게 구현되었는가?
  • 낙관적인 업데이트가 적용되었는가?
  • 에러 핸들링이 적절히 구현되었는가?
  • 서버 상태와 클라이언트 상태가 명확히 분리되었는가?
  • 코드가 간결하고 유지보수가 용이한 구조로 작성되었는가?
  • TanStack Query의 Devtools가 정상적으로 작동하는가?

최종과제

  • 폴더구조와 나의 멘탈모데일이 일치하나요?

과제 셀프회고

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

FSD 아키텍처를 적용하면서 "기능"을 어떤 단위로 쪼개야 할지에 대해 깊이 고민하게 되었다. 단순히 눈으로 보이는 UI를 기준으로 나누는 것을 넘어, 도메인과 행위 단위로 entity, feature를 세분화했을 때 컴포넌트 간의 결합도가 낮아지고 각 기능의 독립성이 보장된다는 것을 알게 되었다. 또한, entities > features > widgets > pages 로 이어지는 의존성의 단방향성과 각 entity, feature끼리 서로를 모르는 블랙박스의 영역으로 설계하는 것을 통해 응집도를 높이고 결합도를 낮추는 것을 몸소 느꼈던 점이 좋았다.

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

컴포넌트, 로직별 각자의 책임과 관심사에 따른 코드 분리에 가장 신경을 많이 썼다.

1. Atoms의 적절한 위치 선정

  • 처음에는 단순히 상태의 공유에만 집중해서 shared/model/atoms에 모든 atoms를 두었음
  • shared는 도메인에 종속되지 않는 계층이어야 하는데 계층이 오염되었다고 느껴짐
  • "이 상태가 누구의 관심사인가?"를 기준으로 재배치
    • selectedPostAtom → entities/post (여러 Feature에서 공유하는 도메인 데이터)
    • showAddPostDialogAtom → features/create-post (게시물 작성 Feature만의 관심사)
    • newPostAtom → features/create-post (게시물 작성 Feature만의 관심사)

shared 계층에 범용 유틸리티만 남기고, atom을 사용하는 곳의 응집도를 높이려 함.

코드 변화:

# Before
src/shared/model/atoms/uiAtoms.ts (21 lines 삭제)
  - selectedPostAtom
  - showAddPostDialogAtom
  - newPostAtom
  - ... (모든 atoms)

# After
src/entities/post/model/atoms.ts
  - selectedPostAtom (여러 Feature가 공유)

src/features/create-post/model/atoms.ts
  - showAddPostDialogAtom (자신만의 상태)
  - newPostAtom (자신만의 상태)

2. PostDetailDialog 간소화

  • 리팩토링 초반의 PostDetailDialog는 너무 많은 책임을 가지고 있었음

    • 댓글 추가 로직
    • 댓글 수정 로직
    • 댓글 삭제 로직
    • 댓글 좋아요 로직
    • 다이얼로그 UI 렌더링

  • 각자의 책임별로 역할을 나누려 함

  1. UI 컴포넌트 분리: 댓글 추가/수정 다이얼로그를 각 Feature로 이동
  2. Hook 캡슐화: useEditCommentDialog로 상태 제어 로직 은닉
  3. Modal별 상태 관리: 댓글 추가/수정 다이얼로그가 각각 상태관리를 하도록 하며 PostDetailDialog에서의 불필요한 핸들러들 삭제

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

1. Widget vs Feature 경계

현재 PostsTable을 Widget으로 분류했는데, 검색을 동반한 Post 목록 조회가 Feature가 될 수도 있지 않을까 하는 생각이 듦

  • 현재 기준: URL 관리 + Entity 조합 → Widget
  • 고민: 사용자 행동(검색, 필터링)이 포함되면 Feature인가?

이 부분은 프로젝트 규모와 팀 컨벤션에 따라 달라질 것 같아 더 많은 케이스를 경험해봐야 할 것 같다.

2. Feature 간 데이터 공유 방식

현재 entities에 두었지만, 실제로는 하나의 feature에서만 사용되는 파일들이 많다. 하지만 미래에 다른 feature도 사용할 가능성을 고려하면 entities가 맞는 것 같은데... 오버엔지니어링이지 않을까 고민이 됨.

  • entities에 두면 확장성 좋음, 하지만 현재는 불필요할 수 있음
  • features에 두면 현재 필요한 만큼만 설계해서 사용, 하지만 나중에 리팩토링 필요할 수 있음

현재 선택은 entities지만 YAGNI 원칙과 충돌하지 않는 기준점을 설정하는 게 아직은 헷갈린다.

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

앞으로의 프로젝트에서도 FSD 아키텍처를 기반으로 초기 설계를 진행하고 싶다. 특히 디렉토리 구조에 신경써서 코드를 읽는 사람이 "아, 이 파일은 여기 있겠구나"라고 직관적으로 예상할 수 있는 구조를 만들고 싶다.

또 이번 프로젝트에 적용했던 쿼리 키 팩토리 패턴도 다음 개발에 적용해보고 싶다.

export const postKeys = {
  all: ['posts'] as const,
  lists: () => [...postKeys.all, 'list'] as const,
  list: (filters: PostFilters) => [...postKeys.lists(), filters] as const,
  details: () => [...postKeys.all, 'detail'] as const,
  detail: (id: number) => [...postKeys.details(), id] as const,
};

이 패턴을 적용하면 쿼리 무효화, 캐시 관리가 훨씬 체계적일 것 같아 기대된다.

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

더티코드를 접했을 때 어떤 기분이었나요? ^^; 클린코드의 중요성, 읽기 좋은 코드란 무엇인지, 유지보수하기 쉬운 코드란 무엇인지에 대한 생각을 공유해주세요

  • 처음에는 기능 구현에 급급해 코드가 지저분해지는 것을 느꼈지만, 리팩토링 과정을 통해 코드를 정리하면서 "읽기 좋은 코드"가 협업뿐만 아니라 미래의 나를 위해서도 얼마나 중요한지 깨달았다. 특히 디렉토리 구조 설계에서도 관심사별로 분리하니, 파일 디렉토리 구조만 보고도 프로젝트를 대략적으로 파악할 수 있었던 점이 신기했다.

기존에는 상위 컴포넌트에서 하위 컴포넌트로 Props를 계속 내려주는 Props Drilling 구조였는데, Jotai를 도입하고 feature 단위로 컴포넌트를 쪼개면서 결합도가 많이 낮아진 것 같다. PostDetailDialog에서 댓글 관련 로직을 뜯어내어 별도의 feature로 만들었을 때, 서로 영향을 주지 않고 독립적으로 동작하는 것을 보며 쾌감을 느꼈다.

응집도 높이기: 서버상태관리, 폴더 구조

"이 코드는 대체 어디에 둬야 하지?"라고 고민했던 시간, FSD를 적용해보면서의 느낌, 나만의 구조를 만들어가는 과정, TanStack Query로 서버 상태를 분리하면서 느낀 해방감(?)등을 공유해주세요

관련된 코드(UI, Model, API)를 하나의 폴더(Feature/Entity)에 모아두니, 파일을 찾아 헤매는 시간이 줄어들고 수정이 필요한 부분이 명확해졌다. FSD 구조가 처음에는 복잡해 보였지만, 익숙해지니 "어디에 무엇을 넣어야 할지"에 대한 명확한 가이드를 제공해 주어 고민 시간을 줄여주는게 체감됐다.

리뷰 받고 싶은 내용이나 궁금한 것에 대한 질문

1. 미래의 확장성과 YAGNI 원칙

현재 단 한군데의 feature에서만 사용되는 atom이나 custom hook들이 많이 있습니다.

  • "미래에 쓸 수도 있다"는 이유로 분리한 것들이 오히려 YAGNI 원칙에 위배되는지 궁금합니다.
  • 일단은 사용하는 feature에 두고, 나중에 다른 곳에서 필요해지면 그때 entities로 올리는 게 나을까요?

2. import 접근 제한

현재 index.ts에서 export하는 것만 import하게 하고, 단방향 import도 강제하고 싶지만, 실제로는 다른 파일 어디서든 접근 가능합니다.

  • 내부 모듈 접근을 어느정도 강제하는 방법으로는 ESLint rule 설정과 모노레포 방법이 있다고 들었습니다.
  • 이러한 방법들을 사용해 import를 강제하는 방법이 실제로도 사용되는지, 만약 사용된다면 주로 어떤 방식으로 강제하는지 궁금합니다.
