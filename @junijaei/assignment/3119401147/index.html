1. Promise.all 오류 수정 및 API 호출 최적화

수정 사항

초기 코드 (SearchDialog.tsx)

const fetchAllLectures = async () =>
  await Promise.all([
    (console.log('API Call 1', performance.now()), await fetchMajors()),
    (console.log('API Call 2', performance.now()), await fetchLiberalArts()),
  ]);

수정 후 코드 (src/api/lectures.ts)

let lecturesCache: Lecture[] | null = null;
let fetchPromise: Promise<Lecture[]> | null = null;

export const fetchAllLectures = async (): Promise<Lecture[]> => {
  // 캐시된 데이터가 있으면 반환
  if (lecturesCache) {
    return lecturesCache;
  }

  // 진행 중인 요청이 있으면 해당 Promise 반환 (중복 호출 방지)
  if (fetchPromise) {
    return fetchPromise;
  }

  // 새로운 요청 시작
  fetchPromise = (async () => {
    const results = await Promise.all([
      fetchMajors(),
      fetchLiberalArts(),
      fetchMajors(),
      fetchLiberalArts(),
      fetchMajors(),
      fetchLiberalArts(),
    ]);
    const lectures = results.flatMap((result) => result.data);
    lecturesCache = lectures;
    return lectures;
  })();

  return fetchPromise;
};

수정 이유

  1. Promise.all 오용 수정
    • Promise.all은 인자로 Promise 객체를 받아야 하지만, await 키워드가 들어가 Promise의 결과값을 인자로 받고 있었다.
    • Promise.all이 병렬로 실행되지 않고 직렬로 실행되어 API의 지연이 발생했다.
  2. 중복 API 호출 문제
    • 원래는 SearchDialog가 열릴 때마다 API를 호출했다.
    • 동일한 데이터를 여러 번 가져오게 되어 비효율적인 API 요청이 이루어지고 있었다.
  3. 캐싱 부재
    • 한 번 가져온 강의 데이터를 재사용하지 않고 매번 새로 요청하고 있었다.
    • 강의 데이터는 정적 데이터로, 변경될 가능성이 낮아 캐싱 전략이 필요했다.

결과

  • 병렬 처리: Promise.all을 올바르게 사용하여 fetchMajors()와 fetchLiberalArts()가 병렬로 실행되어 API 응답 속도가 향상되었다.
  • 캐싱 도입: 첫 번째 호출 이후 lecturesCache에 데이터를 저장하여, 이후 호출 시 네트워크 요청 없이 즉시 캐싱된 값을 반환하도록 했다.
  • 중복 호출 방지: fetchPromise를 활용하여, 동시에 여러 컴포넌트에서 호출되어도 이미 호출된 API면 재요청하지 않게끔 했다.
  • 성능 개선: API 호출 횟수가 대폭 감소하고, 모달을 여러 번 열어도 첫 로딩 이후에는 즉각적으로 데이터가 표시되고 있다.

2. useMemo를 활용한 불필요한 계산 방지

수정 사항

초기 코드

const getFilteredLectures = () => {
  const { query = '', credits, grades, days, times, majors } = searchOptions;
  return lectures.filter(...).filter(...).filter(...);  // 매 렌더링마다 재실행
}

const filteredLectures = getFilteredLectures();  // 함수 호출
const lastPage = Math.ceil(filteredLectures.length / PAGE_SIZE);
const visibleLectures = filteredLectures.slice(0, page * PAGE_SIZE);
const allMajors = [...new Set(lectures.map(lecture => lecture.major))];

수정 후 코드

const filteredLectures = useMemo(() => {
  const { query = '', credits, grades, days, times, majors } = searchOptions;
  return lectures.filter(/* ... */).filter(/* ... */);
}, [lectures, searchOptions]);

const visibleLectures = useMemo(() => filteredLectures.slice(0, page * PAGE_SIZE), [filteredLectures, page]);

const allMajors = useMemo(() => [...new Set(lectures.map((lecture) => lecture.major))], [lectures]);

수정 이유

  1. 불필요한 재계산 방지
    • 컴포넌트가 리렌더링될 때마다 getFilteredLectures() 함수가 재실행되었다.
    • 수천 개의 강의 데이터에 대해 매번 필터링 연산을 다시 수행했다.
    • 의존성이 변경되지 않았는데도 불필요하게 재계산되고 있었다.
  2. 렌더링 최적화
    • useMemo는 의존성 배열의 값이 변경될 때만 연산을 다시 수행한다.
    • 검색 조건이 변경되지 않았다면 이전에 계산한 결과를 캐시해서 재사용한다.
    • 비싼 연산(필터링, 중복 제거 등)의 불필요한 반복을 방지해 최적화했다.

결과

  • 메모이제이션: useMemo로 감싸서 의존성(filteredLectures, page, lectures)이 변경될 때만 재계산되도록 했다.

3. React.memo를 활용한 컴포넌트 리렌더링 최적화

  // ...
});

export const TimeControl = memo(({ allTimes, initialValue, onChange }: Props) => {
  // ...
});

수정 이유

  1. 불필요한 리렌더링 방지
    • memo는 컴포넌트의 props를 얕은 비교(shallow comparison)하여 변경이 없으면 리렌더링을 건너뛰게 된다.
    • memo를 사용할 시, 부모 컴포넌트가 리렌더링되어도 props가 동일하면 자식 컴포넌트는 리렌더링되지 않는다.

결과

  • 부모 컴포넌트의 상태 변경이 불필요하게 자식 컴포넌트로 전파되지 않는다.

4. Form 렌더링 최적화 - 독립적인 상태 관리

수정 사항

초기 코드

const SearchDialog = ({ ... }) => {
  const [searchOptions, setSearchOptions] = useState<SearchOption>({
    query: '',
    grades: [],
    days: [],
    times: [],
    majors: [],
  });

  // 모든 폼 상태가 하나의 객체에 있음
  // 하나의 input이 변경되면 전체 SearchDialog가 리렌더링

  return (
    <Input
      value={searchOptions.query}
      onChange={(e) => setSearchOptions({...searchOptions, query: e.target.value})}
    />
    // ... 다른 폼 요소들
  );
};

수정 후 코드

// QueryControl.tsx - 검색어 컨트롤 분리
export const QueryControl = memo(({ onChange }: { onChange: (value: string) => void }) => {
  const [query, setQuery] = useState('');  // 자체 상태 보유

  const handleChange = (value: string) => {
    setQuery(value);
    onChange(value);  // 부모에게 값만 전달
  };

  return (
    <FormControl>
      <FormLabel>검색어</FormLabel>
      <Input
        placeholder="과목명 또는 과목코드"
        value={query}
        onChange={(e) => handleChange(e.target.value)}
      />
    </FormControl>
  );
});

수정 이유

  1. 전역 상태로 인한 리렌더링 폭증
    • 초기 코드에서는 searchOptions라는 하나의 상태 객체가 모든 폼 필드를 관리했다.
    • 그 결과, 하나의 상태만 변경되어도 모든 Form Control이 리렌더링되었다.
  2. memo 효과 무력화
    • memo를 적용해도 props로 전달되는 searchOptions 객체가 매번 새로 생성되어 얕은 비교(shallow comparison)를 통과하지 못해 리렌더링이 반복됐다.

결과

  • 각 폼 컨트롤이 독립적인 상태를 가져, 예를 들어 DayControl만 변경되면 TimeControl은 리렌더링되지 않는다.
  • memo로 감싸져 있어 props가 동일하면 리렌더링을 스킵한다.
  • 부모에게는 onChange 콜백을 통해 값만 전달하고, 상태는 온전히 자식이 소유한다.

5. Popover의 lazy 속성 활용

수정 사항

수정 후 코드

const DraggableSchedule = memo(({ id, data, bg, onDeleteButtonClick }: Props) => {
  return (
    <Popover isLazy>
      {/* isLazy 속성 추가 */}
      <PopoverTrigger>
        <Box>{/* 시간표 블록 */}</Box>
      </PopoverTrigger>
      <PopoverContent>
        <PopoverBody>
          <Text>강의를 삭제하시겠습니까?</Text>
          <Button onClick={() => onDeleteButtonClick(data)}>삭제</Button>
        </PopoverBody>
      </PopoverContent>
    </Popover>
  );
});

수정 이유

  1. 불필요한 DOM 렌더링
    • Chakra UI의 Popover는 Popover가 닫혀 있어도 PopoverContent의 내용을 DOM에 렌더링한다.
    • 시간표 블록이 많을 경우 (예: 10개 강의 × 평균 3개 시간대 = 30개 Popover), 사용되지 않는 30개의 Popover 컨텐츠가 DOM에 렌더링되고 있었다.
  2. 메모리 사용
    • 보이지 않고 사용되지 않는 DOM 노드들이 메모리를 차지해 성능에 영향을 미치고 있었다.

결과

  • isLazy 속성을 추가하여 Popover가 실제로 열릴 때만 내부 컨텐츠를 렌더링하도록 했다.
  • 초기 렌더링 시에는 Popover가 렌더링되지 않는다.
  • 불필요한 DOM 노드가 제거되어 메모리 사용량과 렌더링 비용이 감소했다.

  • 그 결과, 드래그와 전혀 관련 없는 테이블에서도 리렌더링이 발생해 테이블이 많아질수록 성능이 기하급수적으로 하락했다.

결과

  • 각 시간표가 독립적인 DndContext를 가져, 이제는 테이블 A에서 드래그해도 테이블 B와 C는 영향받지 않는다.
  • 드래그 중 업데이트해야 할 컴포넌트 범위가 축소되어 렌더링 비용이 감소했다.

7. ScheduleContext 삭제 및 Jotai Store로 전환

수정 사항

초기 코드

const ScheduleContext = createContext<ScheduleContextType | undefined>(undefined);

export const ScheduleProvider = ({ children }: PropsWithChildren) => {
  const [schedulesMap, setSchedulesMap] = useState<Record<string, Schedule[]>>(dummyScheduleMap);

  // schedulesMap 전체가 하나의 상태
  // 어떤 테이블이든 변경되면 전체 상태가 변경됨

  return (
    <ScheduleContext.Provider value={{ schedulesMap, setSchedulesMap }}>
      {children}
    </ScheduleContext.Provider>
  );
};

수정 후 코드

import { atom, PrimitiveAtom } from 'jotai';

// 전체 테이블 ID 목록 atom
export const tableIdsAtom = atom<string[]>(Object.keys(dummyScheduleMap));

// 테이블별 스케줄 atom 저장소 (Map으로 직접 관리)
const scheduleAtomsMap = new Map<string, PrimitiveAtom<Schedule[]>>();

// 테이블별 스케줄 atom getter (없으면 생성)
export const getScheduleAtom = (tableId: string): PrimitiveAtom<Schedule[]> => {
  let scheduleAtom = scheduleAtomsMap.get(tableId);
  if (!scheduleAtom) {
    scheduleAtom = atom<Schedule[]>([]);
    scheduleAtomsMap.set(tableId, scheduleAtom);
  }
  return scheduleAtom;
};

// 사용 예시
const ScheduleTableItem = ({ tableId }) => {
  const [schedules, setSchedules] = useAtom(getScheduleAtom(tableId));
  // 이 컴포넌트는 해당 tableId의 atom만 구독
  // 다른 테이블의 변경에는 반응하지 않음
};

수정 이유

  1. React Context의 리렌더링 문제
    • Context는 value가 변경되면 해당 Context를 구독하는 모든 컴포넌트가 리렌더링된다.
    • schedulesMap 객체는 모든 테이블의 스케줄을 포함하므로, 하나의 테이블만 변경되어도 schedulesMap 전체가 새 객체로 교체되고 있었다.
    • 결국 아무리 작은 상태라도 스케줄이 변경되는 순간, useScheduleContext()를 사용하는 모든 컴포넌트가 리렌더링되는 문제가 발생하고 있었다.
  2. 세밀한 구독 불가능
    • Context는 value를 전부 구독할 수밖에 없다.
    • 특정 테이블의 스케줄만 필요한 컴포넌트도 불필요하게 전체 schedulesMap을 구독해야 했다.

결과

  • Jotai의 atom 단위 구독 메커니즘을 활용해 각 컴포넌트에서 필요한 스케줄만 구독할 수 있게 됐다.
  • 하나의 스케줄을 변경할 시 해당하는 스케줄이 있는 컴포넌트만 리렌더링되며, 다른 컴포넌트는 리렌더링되지 않는다.
  • 따라서 테이블 수가 많아져도 성능 저하가 발생하지 않는다.
  • 상태관리 라이브러리가 불변성과 같은 많은 부분을 대신 해결해줘서, 코드가 훨씬 간결해졌다. 
    // 이전: 복잡한 중첩 객체 업데이트
setSchedulesMap((prev) => ({
  ...prev,
  [tableId]: prev[tableId].filter(/* ... */),
}));

// 이후: 간단한 배열 업데이트
setSchedules((prev) => prev.filter(/* ... */));

    • 시간표 그리드(교시, 요일 헤더, 빈 셀)는 절대 변경되지 않는 정적 컴포넌트다.
    • 반면 스케줄 블록들은 매우 자주 변경되는 동적 UI기에 리렌더링이 잦을 수밖에 없다.
    • 두 UI 요소가 한 컴포넌트에 있으면 동적 UI가 변경될 때마다 정적 UI까지 함께 리렌더링된다.
  2. 책임 분리
    • 기존 코드에서는 ScheduleTable이 그리드 렌더링도 하고, 스케줄 렌더링에 이벤트 핸들링, 색상 계산 등 너무 많은 책임을 가지고 있었다.
    • 단일 책임 원칙에 따라 컴포넌트를 나눠 응집도를 높이고 결합도를 낮출 필요가 있었다.
  3. 재사용성 부족 / 유지보수성 떨어짐
    • UI 로직과 비즈니스 로직이 한 곳에 혼재되어 있어, 테스트하거나 디버깅하기 어려웠다.

결과

  • ScheduleTableStatic 분리:
    • 정적 그리드만 담당하는 컴포넌트로 분리
    • memo로 감싸서 한 번 렌더링 후 절대 리렌더링되지 않음
    • 시간표에 강의를 추가/삭제해도 그리드는 다시 렌더링되지 않는다.
  • 렌더링 최적화:
    • 스케줄이 추가/삭제될 때: ScheduleTableStatic은 리렌더링 안 됨, DraggableSchedule들만 업데이트
    • 드래그 중: 드래그되는 블록만 transform 업데이트, 나머지는 memo로 보호
  • 코드 가독성 및 유지보수성 향상:
    • 각 컴포넌트가 명확한 단일 책임을 가짐
    • 테스트 및 디버깅이 용이해짐
    • 향후 기능 추가 시 영향 범위가 명확함

9. useCallback을 활용한 함수 메모이제이션

수정 사항

초기 코드

const ScheduleTable = ({ tableId, schedules, onDeleteButtonClick }) => {
  return (
    <>
      {schedules.map((schedule, index) => (
        <DraggableSchedule
          onDeleteButtonClick={() =>
            onDeleteButtonClick?.({
              // 매번 새 함수 생성
              day: schedule.day,
              time: schedule.range[0],
            })
          }
        />
      ))}
    </>
  );
};

수정 후 코드

const ScheduleTable = ({ tableId, schedules, onDeleteButtonClick }) => {
  const handleDeleteButtonClick = useCallback(
    (schedule: Schedule) =>
      onDeleteButtonClick?.({
        day: schedule.day,
        time: schedule.range[0],
      }),
    [onDeleteButtonClick],  // onDeleteButtonClick이 변경될 때만 새 함수 생성
  );

  return (
    <>
      {schedules.map((schedule, index) => (
        <DraggableSchedule
          onDeleteButtonClick={handleDeleteButtonClick}  // 안정된 참조
        />
      ))}
    </>
  );
};

수정 이유

  1. 함수 재생성으로 인한 memo 무력화
    • React 컴포넌트는 리렌더링될 때마다 내부 함수를 새로 생성한다.
    • 자식 컴포넌트에 props로 전달된 함수가 매번 새로 생성되면, 자식 컴포넌트가 memo로 감싸져 있어도 props가 다르다고 판단되어 리렌더링된다.
    • 결과적으로 컴포넌트에 memo를 적용한다 하더라도 불필요한 리렌더링이 발생한다.
  2. 이벤트 핸들러의 불필요한 재생성
    • 버튼 클릭, 입력 변경 등 이벤트 핸들러는 자주 사용되지만 로직이 변하진 않는다.
    • 매 렌더링마다 동일한 로직의 새 함수를 생성하는 것은 여러모로 비효율적이다.

결과

  • 안정적인 함수 참조: useCallback은 의존성이 변경되지 않으면 이전 함수 참조를 재사용하게 된다.
  • memo 효과 보장:
    • 부모 컴포넌트가 리렌더링되어도 콜백 참조가 유지되어, prop의 변경이 없으면 자식의 리렌더링이 일어나지 않는다.
    // 이전: 매번 새 함수 생성 → memo가 있어도 리렌더링
<DayControl onChange={(days) => handleFormChange({ days })} />

// 이후: 함수 참조가 안정적 → memo가 제대로 작동
<DayControl onChange={handleDaysChange} />

기술적 성장

  • 업무를 하며 성능 최적화를 시도해본 적이 한번도 없었기에 디버깅 방법부터 모든게 새로웠다. 성능 최적화의 정의와 그 방법을 알아가는 과정에서 많은 걸 배운 것 같다. 과제 결과물을 켜놓고 profiler로 계속 비교하며 내 결과물과 코치님의 결과물이 다른 이유를 찾아가는 과정에서의 깨달음이 많았다. 또 react의 memoization에 대해 깊게 파볼 수 있었던 점이 좋았다.

코드 품질

  • 성능 최적화를 진행하면서 어느정도의 리팩토링을 같이 진행해보려고 했지만... 역시 많이 신경쓰진 못한 것 같아 아쉽다.
  • 나름 열심히 챙기려고 노력했기에 내가 놓친 성능 최적화가 있다면 어느 부분일지 궁금하다. (가상 스크롤 제외..)

학습 효과 분석

  • React DevTools Profiler를 활용한 렌더링 분석 및 디버깅 능력이 많이 향상되었다.
  • 성능 최적화는 단순히 기법을 적용하는 것이 아니라, 문제의 원인을 정확히 파악하는 것이 중요함을 깨달았다.
  • 최적화 기법들이 서로 연결되어 있음을 이해했다 (예: useCallback 없이는 memo가 무력화됨).

과제 피드백

  • 초기 코드에 의도적으로 심어둔 성능 이슈들이 실무에서 흔히 발생하는 패턴이라 좋았다.
  • 최적화에 대해 파볼 수 있는 기회가 되어 좋았다.

리뷰 받고 싶은 내용

1. 렌더링 최적화와 단일 진실 공급원(Single Source of Truth) 원칙의 균형점

렌더링 성능을 개선하기 위해 각 폼 컨트롤이 서로 독립적인 상태를 가지도록 설계했습니다. 이 방식은 특정 컨트롤의 상태 변경이 다른 컨트롤의 리렌더링으로 전파되지 않기 때문에, 실제로 렌더링 성능 측면에서는 명확한 개선 효과가 있었습니다.

다만 이 구조는 단일 진실 공급원 원칙을 따르지 않는 설계이며, 그 결과 initialValue를 포함한 초기 상태 동기화 로직이 복잡해지는 문제가 발생했습니다. 상태가 여러 곳에 분산되면서, 초기값 설정 시점과 각 컨트롤의 상태 간 관계를 명확히 관리하기 어려워졌기 때문입니다.

이러한 맥락에서 다음과 같은 점이 궁금합니다.

  • 렌더링 최적화를 위해 상태를 분산하는 설계를 선택할 때, 실무적으로 고려해야 할 기준은 무엇인지 궁금합니다.
  • 단일 진실 공급원을 유지하는 구조와 성능 최적화 사이에서의 판단 기준은 어떻게 잡는 것이 적절한지 알고 싶습니다.
  • 두 요구사항을 동시에 만족시키기 위한 패턴(예: 상위 상태 + 파생 상태, 컨트롤 단위 메모이제이션, 외부 상태 관리 도구 활용 등)이 있다면, 어떤 접근이 적절한지도 궁금합니다.

2. 컴포넌트 분리 기준과 과도한 최적화 경계

정적 UI와 동적 UI를 분리하고, 각 폼 컨트롤을 독립 컴포넌트로 분리하며 최적화를 진행했습니다. 하지만 어느 순간부터 "이 정도면 과도한 최적화가 아닐까?"라는 의문이 들었습니다. 컴포넌트를 분리하고 memo를 적용하는 것도 비용이므로, 실무에서는 어떤 기준으로 최적화의 경계를 판단하시는지 궁금합니다. 성능 측정 결과 외에 고려해야 할 요소가 있을까요?

음.. 최적화라는게 섣부르게 시작하면 안된다고 늘 말하는데요. 실제로 구현은 단순하게 하되(리액트의 내부 최적화를 믿고) 실제 문제가 발생하면 해당 문제를 과제하시는데 사용하셨던 것처럼 퍼포먼스 탭이나 프로파일러로 규정하고 해당 문제를 해결하는거죠. 그 문제 해결 방법에 컴포넌트를 과하게 분리하는 것도 있을 것 같은데, 이 문제라는게 비즈니스 적으로 유저가 사용하기에 문제가 발생하는 부분이니 코드상으로 과하더라도 처리를 먼저 진행하고 우아한 구현 방법을 찾는 순서가 맞는것 같아요. 문제는 '느리다' 해결은 속도가 '빠르게한다'지 '구현을 잘 한다'가 아니니까요.
