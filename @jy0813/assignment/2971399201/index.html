기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

우선 테스트 트로피 전략은 모듈이 조합된 상태에서 비즈니스 로직을 잘 수행하는지 검증하는 통합 테스트의 비중을 가장 높이 두고 있습니다. 현재 프로젝트에서는 일정 생성 → API 호출 → 상태 업데이트 → UI 반영 의 흐름을 가장 핵심적으로 생각한다고 분석하였습니다.

  • 선정이유

    • 통합 테스트가 실제 사용자 시나리오에서의 버그 발견이 높음. 단위 테스트는 개별 함수의 동작만 검증하므로, 함수들이 조합되었을 때 발생하는 버그를 놓칠 수 있습니다. 통합 테스트는 Hook + Component + API 호출 + UI 반영 까지 전체 흐름을 검증하므로, 실제로 사용자가 겪을 수 있는 버그를 잡아냅니다.

    • RTL 철학: “사용자처럼 테스트하라”(https://kentcdodds.com/blog/testing-implementation-details) 통합 테스트는 이 철학에 가장 부합합니다. 사용자가 버튼을 클릭하고, 입력창에 값을 입력하고, 화면에 결과가 표시되는 것을 확인합니다.

    • 켄드 벡 아저씨의 말 중 “기억하세요. 중요한 건 투자에 대한 좋은 수익을 얻는 것입니다. "수익"은 "자신감"이고 "투자"는 "시간"입니다. 만약 우리에게 시간이 무제한이라면, 굳이 분류하려고 애쓸 필요도 없고, 그저 끝없이 테스트만 작성하겠죠! 하지만 시간이 무한하지 않으니, 어디에 노력을 기울여야 할지 고민할 때 이 글이 도움이 되기를 바랍니다.” 즉, 여기서 하고싶은 말은 모든 걸 테스트하려 하지 말고, 한정된 시간 안에서 가장 효율적으로 자신감을 확보할 수 있는 테스트를 하라. 시간은 유한하니, ROI(Return on Investment)가 높은 테스트에 집중해야 한다는 뜻 입니다. 이번 프로젝트는 트로피 전략을 사용하여, 실제 사용자 시나리오 → 버그 발견률이 높아 피라미드 전략보다는 높은 ROI 를 가져갈 수 있다고 생각합니다. (https://kentcdodds.com/blog/the-testing-trophy-and-testing-classifications)

    • 단위 테스트는 구현 세부사항에 의존하여 리팩토링에 취약하고, 통합 테스트는 사용자 인터페이스에 의존하여 리팩토링에 강합니다.

    • 어디에 에너지를 더 써야 할까’를 판단하는 기준을 예시로 작성된 코드입니다.

      단위 테스트: fetch 호출만 확인 → UI 미반영 버그를 놓침

      통합 테스트: 실제 화면에 일정이 보이는지 확인 → 버그를 발견

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

5팀 테스트 전략 논의 피그잼

위 피그잼에 팀원들과 처음 모여서 각자 생각해온 테스트 전략에 대해 발표하고 서로 의견을 나누며 논의하는 시간을 가졌습니다. 그 후 초안과 정리본을 기반으로 최종 테스트 전략이 수립되었습니다. 처음 5팀에서 나온 테스트 전략은 현재 Unit Test와 Integration Test에는 꽤나 많은 개발이 진행되었고, 퍼센티지(개발치 → 목표치 차이)로 판단하였을 때, 가장 큰 중요성을 띌 것 같은 Components나 E2E 혹온 시각적 회귀에 대한 개발이 선수 되어야한다고 생각한다는 의견에 가장 많은 투표를 받고 시각적 회귀 => E2E => 부족한 통합 테스트 커버를 하자는 전략으로 정해졌습니다. 그런데 기본과제인 시각적 회귀 테스트를 진행하다가 느낀점이 있어 테스트 전략에 대해서 다시 논의 하자고 디스코드로 다시 제안을 했습니다. 목요일이라 다들 과제하느라 바쁘실텐데 너무 늦게 다시 수정 제안을 해서 죄송했어요 5팀 여러분..

그래서 코어 타임에 다시 모여서 제가 과제를 하면서 느낀점을 토대로 팀원분들께 제안을 드렸습니다.

지금 우리가 시각적 회귀 테스트를 같이 진행하고 있잖아요.
저도 직접 해보면서 느낀 게 있는데,
이 테스트가 ‘무엇을 위해 존재하는가’를 한 번 다시 짚어볼 필요가 있을 것 같아요.

지금은 컴포넌트 기반 개발이 워낙 일반적이라,
Storybook에서 각 컴포넌트를 독립적으로 보고 스타일이나 상태가 잘 변하는지 확인하는 건 자연스러운 흐름이죠.
근데 Chromatic이 들어오면서, 이제는 디자이너나 PM도 변경사항을 한눈에 보고 승인까지 할 수 있게 됐잖아요?

이게 되게 좋아요. QA 리소스를 확실히 줄여주죠.
다만 직접 써보니까, Chromatic은 결국 디자인 일관성을 위한 도구라는 생각이 들어요.
테마 수정, 글로벌 CSS 변경, 컴포넌트 리팩토링 이후
“이게 의도된 변화인가?”를 자동으로 잡아주는 — 일종의 시각적 QA 시스템인 거죠.

근데 그게 어디까지나 디자인 시스템의 안전망이고,
“기능이 정상적으로 작동하느냐”는 검증과는 또 별개더라고요.
그래서 Chromatic은 좀 더 “QA 자동화” 쪽 역할에 가깝지,
사용자 경험의 흐름이나 실제 로직의 신뢰성을 테스트하는 건 아니라고 느꼈어요.

그래서 제가 처음 제안드렸던 트로피 전략이 다시 생각나더라고요.
트로피 전략은
단위 테스트로 로직 안정성을 보고,
통합 테스트로 기능 간 흐름을 확인하고,
E2E는 진짜 중요한 사용자 시나리오만 최소한으로 검증하는 구조죠.

예를 들어, “알림이 일정 시간 후 표시된다”는 걸 E2E로 굳이 테스트하지 않아도,
통합 테스트에서 충분히 검증할 수 있잖아요.
저는 이게 훨씬 현실적이라고 봤어요 — 유지보수나 실행 속도 면에서도요.

그래서 지금 시각적 회귀 테스트를 같이 해보면서
이 도구는 “디자인 일관성 확인용”으로 명확히 포지셔닝하고,
테스트 전략의 중심은 다시 트로피 전략 기반으로 재정비하면 좋겠다는 생각이 들었어요.

물론 이건 제 개인적인 의견이니까,
다른 분들은 시각적 회귀 테스트를 직접 하면서 어떤 느낌 받으셨는지,
그리고 테스트 전략 전체를 놓고 봤을 때 어디에 더 무게를 두면 좋을지 같이 이야기해보면 좋겠어요.

결국 우리가 진짜 지켜야 하는 건,
“UI가 깨졌는가?”보다는 “사용자 경험이 깨졌는가?”잖아요.
그 관점에서 테스트 밸런스를 다시 잡아보면 좋겠다는 생각이에요.

그래도 다들 긍정적으로 받아들여주시고 팀원분들도 과제를 진행하면서 각 테스트의 특징과 장단점을 파악하고 1차에 협의한 전략이 현재 프로젝트가 UI가 동적으로 자주 변경되는 프로젝트가 아니고 시각적 회귀는 캘린더 프로젝트에 필요한 테스트일지에 대한 의문과 기존 제공된 테스트 스펙으로도 충분히 검증 가능하다고 결론이 도달하여, 통합 테스트 중심 => E2E로 검증 가능한 부분만 E2E 테스트 변경하였습니다. 그래도 다행인건 추가 테스트에 대한 케이스를 이미 회의를 통해 많이 추가 해놓은 상태라 전략이 바뀌면서 추가 테스트에 대해 고민하는 시간이 오래걸리지는 않고 다들 빠르게 몇개의 추가 테스트만 일부 추가하였습니다.

  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

통합테스트 10개 추가 완료

  • 일정 추가 시 캘린더에서 날짜 셀을 클릭하면 해당 날짜에 일정이 생성된다.

  • 일정 수정 시 캘린더에서 날짜 셀을 클릭하면 해당 날짜로 일정이 수정된다.

  • 빈 날짜 셀을 클릭하면 해당 날짜가 폼에 입력되지만 일정은 생성되지 않는다.

  • 같은 날짜 셀을 여러 번 클릭해도 날짜가 올바르게 유지된다.

  • 이미 일부 정보를 입력한 상태에서 날짜 셀을 클릭하면 날짜만 변경되고 다른 필드는 유지된다

  • 필수 필드 누락 시 에러 메시지가 표시된다. (필수 필드 검증)

  • 종료 시간이 시작 시간보다 빠르면 에러 메시지가 표시된다. (시간 유효성 검증)

  • 시간만 수정 시 다른 필드는 유지된다. (부분 수정 검증)

  • 반복 일정 설정 검증

    • 반복 간격 설정이 올바르게 동작한다.
    • 반복 종료일 없이 무한 반복 일정 생성이 가능하다.

    dateClickWorkflow.spec.tsx eventValidation.spec.tsx recurringEventWorkflow.spec.tsx

E2E 4개 추가 완료

  • 폼 검증 워크플로우

    • 필수 필드 누락 → 에러 확인 → 수정 → 일정 생성 성공
    • 잘못된 시간 → 에러 확인 → 수정 → 일정 생성 성공

  • 반복 일정 설정 검증 워크플로우

    • 격주 반복 일정 생성 → 목록에서 반복 확인
    • 무한 반복 일정 생성 → 종료일 없음 확인

    e2e/basic-events.e2e.spec.ts e2e/recurring-events.e2e.spec.ts

과제 셀프회고

이번 과제를 마지막으로 챕터 1이 끝났습니다…. 처음 겪어보는 테스트였지만 그래도 무사히 챕터 1을 마무리하여 너무 뿌듯합니다! 테스트 코드의 종류부터 RTL, MSW의 새로운 사용법, 직접 에이전트를 만들어보는 경험까지…. 많은 걸 배운 챕터 1이었습니다.

이번 주차는 유난히 팀원들과의 교류가 활발했던 것 같습니다. 단순히 “과제 제출”만을 위한 시간이 아니라, 진짜 테스트 전략을 세우는 팀 프로젝트를 해보는 느낌이었고, 테스트의 목적이나 전략에 대해 얘기하다 보면 자연스레 코드 전체의 구조나 흐름까지 언급하게 되고, 그러다 보니 저 혼자만의 고민으로 묶어두기보다 자료를 더 찾아보고, 그걸 정리해서 서로에게 설명하고 싶은 동기도 생기고 좋았습니다.

그리고 그냥 공부하고 말한다를 넘어서, 팀원들 앞에서 테스트 전략을 왜 이렇게 설계했는지, 이 방식이 장기적으로 어떤 의미가 있을지 까지 함께 얘기해 볼 수 있었던 게 정말 좋았습니다. 사실 혼자 코드만 짜고 있었다면 대충 넘어갔을 이슈들도, 팀과 논의하다 보면 숨겨진 문제점들이 드러나고, 또 가끔은 서로 다른 경험이 합쳐져서 더 좋은 솔루션이 나온거 같습니다.

어느덧 3주가 지나면서 테스트 코드를 배웠지만 이제 테스트 코드를 마스터 했다 이러한 생각은 하지 않습니다. 하지만 안 해본 경험을 하면서 나중에 제가 테스트 코드를 회사에서나 개인적으로 다루기 시작할 때는 지금의 경험들이 많이 도움이 될 것 같습니다.

물론 이번에 한 모든 시도들이 항상 정답은 아닐 거라 생각하지만 직접 구조를 짜보면서 느끼고, 그걸 해결할 수 있는 선택지를 생각해본 경험은 분명 오래 남을 거고 나중에 회사에서도 어느 날 비슷한 벽을 만나면, 아마 이 경험을 꺼내 잘 정리해서 팀과 공유할 수 있을 거라고 생각합니다.

// e2e/fixtures.ts
import { test as base } from '@playwright/test';

type CustomFixtures = {
  createEvent: (eventData: Partial<Event>) => Promise<void>;
  createRecurringEvent: (eventData: Partial<Event>) => Promise<void>;
};

export const test = base.extend<CustomFixtures>({
  // 1) 자동 설정: 모든 테스트에서 반복 제거
  page: async ({ page }, use) => {
    // 시간 고정 (2025-11-06)
    await page.clock.install({ time: new Date('2025-11-06T00:00:00Z') });

    // DB 초기화
    await page.request.post('http://localhost:3000/api/reset');

    // 앱 페이지 로드
    await page.goto('/');

    await use(page);
  },

  // 2) 헬퍼 함수 주입: 테스트 코드 간결화
  createEvent: async ({ page }, use) => {
    await use(async (eventData) => {
      const response = await page.request.post('http://localhost:3000/api/events', {
        data: {
          title: eventData.title || 'Test Event',
          date: eventData.date || '2025-11-10',
          startTime: eventData.startTime || '10:00',
          endTime: eventData.endTime || '11:00',
          description: eventData.description || '',
          location: eventData.location || '',
          category: eventData.category || '업무',
          repeat: { type: 'none', interval: 1 },
          notificationTime: eventData.notificationTime || 10,
        },
      });

      if (!response.ok()) {
        throw new Error(`Failed to create event: ${response.status()}`);
      }
    });
  },

  createRecurringEvent: async ({ page }, use) => {
    await use(async (eventData) => {
      const response = await page.request.post('http://localhost:3000/api/events-list', {
        data: {
          title: eventData.title || 'Test Recurring Event',
          date: eventData.date || '2025-11-10',
          startTime: eventData.startTime || '10:00',
          endTime: eventData.endTime || '11:00',
          description: eventData.description || '',
          location: eventData.location || '',
          category: eventData.category || '업무',
          repeat: {
            type: eventData.repeatType || 'weekly',
            interval: eventData.repeatInterval || 1,
            endDate: eventData.repeatEndDate,
          },
          notificationTime: eventData.notificationTime || 10,
        },
      });

      if (!response.ok()) {
        throw new Error(`Failed to create recurring event: ${response.status()}`);
      }
    });
  },
});

E2E 테스트 구조 설계의 고민..

  1. src/__tests__/e2e/: 모든 테스트를 한 곳에
  2. 루트 e2e/: E2E를 별도 분리 (Playwright 권장)

고민 포인트

  • 일반적으로 src/__tests__/에 모든 테스트
  • Playwright 공식 문서: 루트 e2e/ 권장
  • Vite 빌드: src/ 내부 파일들은 빌드 대상

루트 e2e/ 폴더 (선택)

  • Vite 빌드 충돌 방지 (src/ 내부는 빌드 대상)
  • Playwright 권장 구조
  • E2E 전용 설정 분리 용이 (playwright.config.ts)
  • 테스트 데이터 격리 (TEST_ENV=e2e)

학습 효과 분석

  • 가장 큰 배움이 있었던 부분

시각적 회귀 테스트를 직접 경험하면서, 테스트 도구의 역할을 명확히 구분하는 중요성 을 배웠습니다. 처음에는 시각적 테스트를 도입하면 테스트 전반이 탄탄해지겠구나 라고 막연히 기대했지만, 실제로 써보고 나니 이 도구가 디자인 일관성 확인에 특화된 QA 자동화 도구라는 것을 알게 되었습니다. 즉, 기능 테스트나 사용자 경험의 흐름을 전부 커버해줄 수는 없고, 어디까지나 디자인이 깨졌는가? 에 집중한다는 것을 깨달았습니다. 이는 단순히 ‘툴을 도입했다’에서 끝나는 것이 아니라, 테스트 전략의 균형을 다시 고민해보게 되는 계기가 되었습니다.

  • 추가 학습이 필요한 영역

    • 테스트 피라미드/트로피 전략 기반으로 테스트 균형 잡기: 단위 테스트 → 통합 테스트 → E2E 테스트에 어떤 비중을 둘지, 구체적인 기준을 더 학습하고 싶습니다. 특히, “이건 E2E 나 통합 테스트에서 다뤄도 될까?” 같은 판단 기준이 모호해지는 상황이 있습니다.
    • 테스트 자동화의 ROI(투자 대비 효과) 분석: 도입만 중요한 게 아니라, 실제로 비용 대비 얼마나 효율을 올려주는가? 를 측정할 수 있는 감각도 필요하다고 느꼈습니다.

  • 실무 적용 가능성

    • 통합 테스트 중심의 현실적인 테스트 도입 전략: E2E만 고집하지 않고, 통합 테스트를 중심으로 구성하는 전략은 실제로 비용과 유지보수 면에서 효과적일 것으로 보입니다. 예시로 “알림이 일정 시간 후 표시된다”처럼 로직 중심 시나리오는 E2E가 아닌 통합 테스트에서 보다 가볍게, 빠르게 테스트 가능하다는 점을 팀에 설득할 수 있다고 느꼈습니다.

리뷰 받고 싶은 내용

  1. E2E 테스트를 작성하다 보면 반복되는 코드 패턴을 마주하였는데, 특히 테스트 데이터를 API로 생성하는 로직은 여러 테스트에서 중복이었습니다. 이러한 반복 코드를 공통화하는 두 가지 접근 방식으로 첫 번째로 고려한 방법은 test-utils.ts 헬퍼 함수로 별도의 유틸리티 파일을 만드는 것이고 Playwright는 fixture 시스템을 통해 테스트 환경을 확장할 수 있으니 이를 활용하면 헬퍼 함수를 fixture로 만들어 자동 주입할 수 있는 방식 이렇게 두 가지를 생각하였습니다. 결국은 헬퍼 함수를 fixture로 만들어 주입하였는데, 오프코치님은 두 가지 방법 중 어떤걸 더 선호하실까요? 아니면 이 두 가지 방법보다 더 좋은 방법이 있을까요?

  2. "알림 시스템 관련 노출 조건에 대해 검증하세요" 라는 기본과제 목록 중 하나인데, 여기서 시나리오를 작성한다면, "실제 시간 경과 후 알림 배너를 표시한다" 라는 시나리오는 통합테스트에서 이미 검증했으니 E2E에서는 필요없다고 생각했습니다. 그 이유는 실제 시간 경과 후 알림 배너를 표시한다. 라는게 이미 검증이 된 상태에서 close 버튼을 누르면 사라진다는 사용자 검증 시나리오 이런게 비용대비 E2E 검증할 필요성이 있을까? 라는 의문이 들어서인데...이러한 경우에는 E2E를 진행하는게 맞을까요?

  3. playwright.config.ts 에서 fullyParallel 를 true로 설정하고, workers를 undefined 로 설정해도, e2e.json 한개로 공유하고 있어서 e2e.json을 각 테스트 전에 리셋 해주는 코드를 추가해줘도 병렬로 실행하면 race condition 문제가 일어나는데, e2e.json 한개를 바라보게 하지않고 여러개를 만들거나, Playwright의 page.route()로 Mock 처리로 진행해야만 병렬 처리가 가능할까요?? QnA 때 이거에 대한 질문과 답변을 본거같은데 제가 재대로 이해를 못했습니다 😢

고생하셨습니다 재윤님 ㅎㅎ 회고도 매우 꼼꼼하게 작성해주시고, 테스트 코드 작성하시면서도 꽤 깊게 많은 고민을 해주셨던것 같아요. 전반적으로 테스트 코드 잘 작성해주셨고 크게 군더더기 없습니다. 특히 주도적으로 전략에 관해 이야기 나눠주시고 체득하신 것 같아서 너무 좋습니다. 제 의도보다 더 과제를 잘 수행해주신 것 같습니다. 감사합니다 ㅎㅎ

질문 주신 부분 답변 호다닥 드려보고 마무리 해볼게요.

결국은 헬퍼 함수를 fixture로 만들어 주입하였는데, 오프코치님은 두 가지 방법 중 어떤걸 더 선호하실까요? 아니면 이 두 가지 방법보다 더 좋은 방법이 있을까요?

저는 둘 다 상관없이 많이 쓰는것 같은데요. 전에 설명드렸던것 처럼 전체를 관통하는 픽스쳐가 있고, 각 상황에 맞게 각 TC에서 수정해서 사용하는 것을 선호하는 것 같습니다 ㅎㅎ

그 이유는 실제 시간 경과 후 알림 배너를 표시한다. 라는게 이미 검증이 된 상태에서 close 버튼을 누르면 사라진다는 사용자 검증 시나리오 이런게 비용대비 E2E 검증할 필요성이 있을까?

음.. 이 부분도 일리있는 내용인데요. 제가 작성을 말씀드렸던 내용은 최대한 많이 테스트를 작성해보도록 경험해보려고 한 목적도 있고 결국 통합테스트에서 해당 부분에 대해 검증할 때 타이머에 대한 부분을 모킹하고 진행하는 부분이 있으니 확인할 필요가 있지 않을까 싶어서였어요. 근데 결국 e2e를 운영하려면 타이머를 어느정도 모킹해야하니 기존과 크게 차이는 없을것 같네요 ㅎㅎ 좋은 지적인 것 같습니다. 질문을 조금 다르게 해서 같은 부분에 대한 검증이라면 시간이 빠르고 비용이 적은 통합테스트로만 운영하는게 더 좋겠죠! 그러면 될 것 같습니다.

playwright.config.ts 에서 fullyParallel 를 true로 설정하고, workers를 undefined 로 설정해도, e2e.json 한개로 공유하고 있어서

넵 맞습니다! 영서님이 관련해서 올려주신 질문 보고 별도로 구성해도 되고, 말씀해주신것처럼 fullyParallel를 쓰지 않고 폴더별로 별도로 설정을 구성해서 병렬로 구동하게 하는것도 방법이 있을 것 같아요. 이 부분은 리뷰세션때 말씀드린것 같아서 확인해보시면 좋을것 같습니다. ㅎㅎ
