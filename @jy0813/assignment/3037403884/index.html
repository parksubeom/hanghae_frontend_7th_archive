Before 패키지에서 발견한 문제점

1. 스타일링 문제점

인라인 스타일 과다사용

<div style={{ minHeight: '100vh', background: '#f0f0f0' }}>
  <div style={{ maxWidth: '1200px', margin: '0 auto', padding: '20px' }}>
    <h1 style={{
      fontSize: '24px',
      fontWeight: 'bold',
      marginBottom: '5px',
      color: '#333'
    }}>

이거는 리팩토링 하면서도 너무 힘들었던 부분입니다.. css 파일뿐만 아니라 인라인 스타일도 모두 살펴봐야했기 때문에.. 우선적으로 재사용이 불가능하고, 유지보수도 어려워집니다.

css 클래스와 인라인 스타일 혼용

컴포넌트스타일 방식
ButtonCSS 클래스 (.btn-primary)
Stats 카드인라인 스타일
Tab 버튼인라인 스타일
TableCSS 클래스 (.table)

일부 컴포넌트는 css 클래스를 사용하고, 일부는 인라인 스타일로만 적용하는 방식으로 일관성이 전혀 없습니다.

하드코딩된 색상 값

background: '#1976d2'  // Tab 버튼
color: '#1976d2'       // Stats 카드
borderColor: '#1565c0' // Button CSS

동일한 색상이 여러 곳에 중복되며 디자인 토큰 미사용 및 css 변수 미사용으로 다크모드에 대응이 불가능했습니다.

모놀리식 css 파일


.btn { }              /* BEM 아님 */
.form-input { }       /* kebab-case */
.checkbox-wrapper { } /* 의미적 네이밍 */
.card-default { }     /* 변형자 */

우선 기본적으로 네이밍이 불가능하고, 컴포넌트 별 분리도 없으며 스타일 충돌 가능성이 많았습니다. 또한 사용하지 않는 스타일에 추적이 어렵습니다.

2. 컴포넌트 설계 문제점

ui 컴포넌트와 비즈니스 로직 결합

interface ButtonProps {
  // 순수 UI props
  variant?: 'primary' | 'secondary' | 'danger' | 'success';
  size?: 'sm' | 'md' | 'lg';

  // 도메인 props (문제)
  entityType?: 'user' | 'post';
  action?: 'create' | 'edit' | 'delete' | 'publish' | 'archive';
  entity?: any;
}

// 컴포넌트 내부에 비즈니스 규칙
if (entityType === 'user' && action === 'delete' && entity.role === 'admin') {
  actualDisabled = true; // 관리자는 삭제 불가
}

Atomic Design의 원자(atoms) 컴포넌트가 도메인 지식을 포함하고있어, 단일책임원칙을 위반하고있습니다.

이렇게 작성되어있으면, 컴포넌트가 도메인 변경에 영향을 받기때문에 문제가 됬습니다.

버튼 컴포넌트 외 에도 FormInput 에서 검증 로직이 내장되어있고, Table 에서도 도메인별 렌더링 분기 , Badge 에서도 도메인 props 문제 등 대부분의 컴포넌트에서 비즈니스 규칙이 포함되어있으며, 다수의 도메인을 바라보고있었습니다.

3. 폼 관리 문제점

타입 안전성 부재

const [formData, setFormData] = useState<any>({});

any 타입을 사용하여 타입 안전성 상실이되고, 컴파일 타임 에러 감지 불가 및 런타임 에러 위험도 있습니다.

그리고 작업을 진행하면서도 IDE 자동완성을 지원하지 않아 DX 도 좋지않습니다.

수동 폼 상태 관리

<FormInput
  value={formData.username || ''}
  onChange={(value) => setFormData({ ...formData, username: value })}
/>

모든 필드에서 반복되는 패턴으로 나오며 username 하나만 바꿔도 전체 formData 객체가 새로 생성됩니다.

검증에 관한 부분이 여러곳으로 분산

검증 위치검증 내용문제
FormInput 내부형식 검증 (이메일, 길이)컴포넌트 재사용 어려움
userService.ts중복 체크서버/클라이언트 불일치
ManagementPage없음통합 검증 부재

alert 상태 관리 복잡성

const [showSuccessAlert, setShowSuccessAlert] = useState(false);
const [alertMessage, setAlertMessage] = useState('');
const [showErrorAlert, setShowErrorAlert] = useState(false);
const [errorMessage, setErrorMessage] = useState('');

setAlertMessage('사용자가 생성되었습니다');
setShowSuccessAlert(true);

4개의 분리된 상태로 관리하고 사용 시 매번 2개 상태를 업데이트 해줘야합니다.

탭, 데이터, 모달, 폼 알림 상태, CRUD 작업 함수, 로딩, 통계 유틸리티, 테이블 컬럼 등 하나의 페이지에서 너무 많은 상태관리와, 전체 렌더링 관련 코드가 다 들어있습니다.

커스텀 훅 및 Context 의 부재

위에서 논하긴 했지만, 탭, 데이터, 모달, 폼 알림 상태, CRUD 작업 함수, 등을 hook으로 추출 가능하지만 모두 ManagementPage 에서 관리하고 Context가 없어 모든 상태가 단일 컴포넌트에 집중되고 상태를 공유하면 컴포넌트 분리에 어려움이 생깁니다.

접근성(A11y) 미흡

컴포넌트누락된 접근성 속성
Modalrole="dialog", aria-modal, aria-labelledby, focus trap
Tablescope, caption, aria-sort
Buttonaria-pressed, aria-expanded
Alertrole="alert", aria-live
FormInputaria-invalid, aria-describedby

모달 열림 시 포커스 트랩 미구현으로 Tab 키로 모달 밖 요소에 접근 가능합니다. aria-modal 미사용으로 모달 뒤 콘텐츠를 스크린 리더가 읽어버리고 알림 발생 시 role="alert" 미사용으로 스크린 리더가 자동 안내하지 않으며, 폼 에러 발생 시 aria-invalid, aria-describedby 미사용으로 어떤 필드에 문제가 있는지 맥락 전달 불가 등 의 사용자 경험 저하가 있습니다.

개편 과정에서 집중한 부분

체계적인 설계와 시스템 구축 을 하며 Before의 패키지에서 After 로 리팩토링을 하면서 6가지에 중점을 두었습니다. 가장 많은 시간을 투자한 디자인 토큰 시스템이 모든 컴포넌트의 기반이 되었고, 이를 통해

  • 일관된 시각적 언어 확립
  • 다크모드 지원
  • 유지보수 용이성 확보
  • 관심사 분리를 통해 648줄의 God Component를 137줄로 줄이고, 각 로직을 독립적인 훅으로 분리하여 재사용 가능한 코드를 만들었습니다.
  1. 디자인 토큰 시스템 + Tailwind + CVA

Before 문제점

  • 하드코딩된 색상값 (#1976d2, rgba(0,0,0,0.87))
  • 인라인 스타일 남용 (style={{ color: '#333' }})
  • 다크모드 미지원
  • 일관성 없는 spacing, font-size

After 해결책으로 아래와 같이 3단계 레이어 토큰을 구성하였습니다. 유틸리티 클래스로 컴포넌트를 설계하며 CVA 로 함께 적용하는데 토큰화를 통한 정리가 되어있으니 확실히 유틸리티 클래스를 사용하여 깔끔하게 정리할수있었습니다.

토큰 계층 구조
├── primitive.css (순수 값)
│   ├── colors (gray, blue, red, green, orange scale)
│   ├── spacing (0-32px)
│   ├── font-size, font-weight, line-height
│   ├── radius, shadow, z-index
│   └── duration, easing
│
├── semantic.css (의미 부여)
│   ├── intent colors (primary, destructive, success, warning)
│   ├── surface colors (background, foreground, border)
│   ├── feedback variants (info, success, error, warning)
│   └── typography utilities (@utility typo-*)
│
└── components.css (컴포넌트별 토큰)
    ├── btn-*, badge-*, alert-*
    ├── modal-*, table-*, tabs-*
    └── header-*, card-*, form-*

:root {
  --raw-background: var(--color-white);
  --raw-foreground: var(--color-black-a87);
}

.dark {
  --raw-background: var(--color-gray-800);
  --raw-foreground: var(--color-white);
}

  1. Atomic Design → 기능별 분류

Before 문제점

// Before (Atomic Design)
components/
├── atoms/      # Button, Badge
├── molecules/  # FormInput, FormSelect
└── organisms/  # Modal, Table, Alert
  • Atomic Design의 atoms/molecules/organisms 구분이 모호함
  • 구분이 모호함만이 있는게 아닌 여러가지 기능들이 내부 로직에 저장되어있었음.

After 해결책으로 기능별 분류로 파일 탐색 및 유지보수에 더 편한 폴더구조로 리팩토링 하고 domains 외 에는 Ui만 담당합니다.

// After (기능별)
components/
├── inputs/       # Button, Input, Textarea, Select, Form
├── feedback/     # Alert, Dialog
├── data-display/ # Table, Badge, StatCard
├── surfaces/     # Card, Header
├── navigation/   # Tabs
└── domains/      # 도메인별 복합 컴포넌트

사용한 기술 스택 경험

이번 과제에서 처음으로 shadcn/ui, CVA 그리고 Tailwind CSS 기반 3계층 토큰 시스템을 도입하여 디자인 시스템을 구축했습니다. CVA 도 진짜 신세계 같은 느낌..? Tailwind 만 사용하면 불편한점들을 컴포넌트의 다양한 스타일 변형을 선언적으로 정의하고, TypeScript를 통해 타입 추론까지 가능하게 하는게 진짜..아 찰떡 궁합이구나 라고 느꼈습니다. 토큰화랑 같이 해놓으니 전혀 위화감 없는 정말 깔끔한 컴포넌트로 구현한거같습니다. 그리고 Variants도 하나의 컴포너트에서 동일 Variant로 다중 CVA 인스턴스를 사용가능한거도 좋았습니다.

import { cva, type VariantProps } from 'class-variance-authority';

const buttonVariants = cva(
  // 기본 스타일 (모든 버튼에 적용)
  "btn-base typo-btn-md focus-ring focus-visible:focus-ring-visible",
  {
    variants: {
      variant: {
        primary: 'btn-primary hover:btn-primary-hover',
        destructive: 'btn-destructive hover:btn-destructive-hover',
        secondary: 'btn-secondary hover:btn-secondary-hover',
        success: 'btn-success hover:btn-success-hover',
        outline: 'btn-outline hover:btn-outline-hover',
        ghost: 'btn-ghost hover:btn-ghost-hover',
        link: 'btn-link hover:btn-link-hover',
      },
      size: {
        sm: 'padding-btn-sm typo-btn-sm',
        md: 'padding-btn-md typo-btn-md',
        lg: 'padding-btn-lg typo-btn-lg',
        icon: 'size-9',
      },
      fullWidth: {
        true: 'btn-fullwidth',
      },
    },
    defaultVariants: {
      variant: 'primary',
      size: 'md',
    },
  },
);

shadcn/ui 는 좋다고만 얘기로 들어봤지 실제로 써본건 처음인데 복사-붙여넣기 방식의 컴포넌트 라이브러리라는거도 꽤 신기하고, Tailwind 클래스 충돌 해결도 유틸리티로 제공하고 Radix UI 프리미티브를 기반이여서 접근성 까지 챙기다니 너무 편리한거같습니다. 사실 원래 처음부터 꼭 컴포넌트를 다 작업하는걸 좋아했는데.. 이렇게 접근성이나 기본적인걸 모두 제공하니 실무에서도 적용해보면 참 좋을거같다는 생각이 들었습니다.

3계층 토큰 시스템은 아래와 같은 형태로 진행하였습니다.

primitive.css (순수 값)
    ↓
semantic.css (의미 부여)
    ↓
components.css (컴포넌트 특화)

Layer 1: Primitive Tokens

역할: 의미 없는 순수 값 정의

@theme {
  /* 색상: 숫자 스케일 */
  --color-blue-500: #1976d2;
  --color-blue-600: #1565c0;

  /* 간격: 픽셀 값 */
  --spacing-8: 8px;
  --spacing-12: 12px;

  /* 폰트 크기 */
  --font-size-14: 14px;
  --font-size-16: 16px;
}
  • 이름에 의미가 없음 (blue-500, spacing-8)
  • 변경되지 않는 값 저장소
  • 다크모드 영향 없음

Layer 2: Semantic Tokens

역할: Primitive 값에 의미(Intent) 부여

@import './primitive.css';

@theme inline {
  /* 의도 기반 색상 */
  --color-primary: var(--color-blue-500);
  --color-primary-hover: var(--color-blue-600);
  --color-destructive: var(--color-red-600);

  /* 의미 기반 간격 */
  --spacing-sm: var(--spacing-4);
  --spacing-md: var(--spacing-8);
  --spacing-lg: var(--spacing-12);

  /* 의미 기반 타이포그래피 */
  --text-base: var(--font-size-14);
  --text-lg: var(--font-size-16);
}
  • 의도/용도를 이름에 반영 (primary, destructive, sm, md)
  • 다크모드 전환 지점 (--raw-* 토큰 활용)
  • Tailwind 유틸리티 클래스 생성 (text-primary, bg-destructive)

Layer 3: Component Tokens

역할: 컴포넌트별 구체화된 스타일

@import './semantic.css';

@theme inline {
  /* 버튼 전용 토큰 */
  --color-btn-primary-bg: var(--color-primary);
  --color-btn-primary-bg-hover: var(--color-primary-hover);
  --radius-btn: var(--radius-base);
}

/* 복합 스타일 유틸리티 */
@utility btn-base {
  display: inline-flex;
  align-items: center;
  border-radius: var(--radius-btn);
  cursor: pointer;
  transition: color, background-color 150ms;
}

@utility btn-primary {
  background-color: var(--color-btn-primary-bg);
  color: var(--color-btn-primary-text);
}
  • 컴포넌트 이름 접두사 (btn-, badge-, card-)
  • @utility 디렉티브로 복합 스타일 정의
  • 컴포넌트에서 유틸리티 클래스로 직접 사용

어려웠던 점과 해결 방법

오히려 분리하고, 뭔가 ui와 비즈니스 로직을 나누는거보다 저는 tailwind가 익숙하지 않음 + 새로운 v4 버전 문제가 더 어려웠던 점 같습니다. tailwind v4 에 대한 아는 정보가 별로 없어서 공식 문서로 확인해야하는데...theme lnline 이나 theme 에 대한 자세한 설명이 없는거같아요..

우선 제가 이해하고 알게된거 theme inline은 중간 변수를 건너뛰어서 스코프 문제를 피한다.

@theme로 var를 참조하면,

  @theme {
    --color-background: var(--raw-background);
  }

아래 처럼 생성되고

  :root {
    --color-background: var(--raw-background);
  }

  .bg-background {
    background: var(--color-background);
  }

@theme inline은

  @theme inline {
    --color-background: var(--raw-background);
  }

:root에 --color-background 생성 안 됨!

  /* 유틸리티만 생성 */
  .bg-background {
    background: var(--raw-background);  /* 직접 참조 *
  }

@theme inline이든 @theme이든 CSS 변수는 모두 생성하고, 차이점은 @theme inline에서는 var() 참조를 사용할 수 있다는 것과 중간 변수를 건너뛰어서 스코프 문제를 피한다. @theme는 var() 참조 사용이 불가능하다.

 /* primitive.css */
  @theme {
    --color-blue: #1976d2;  /* ✅ 순수 값만 가능 */
    --color-primary: var(--color-blue);  /* ❌ 에러! var() 사용 불가 */
  }

  /* semantic.css */
  @theme inline {
    --color-primary: var(--color-blue);  /* ✅ var() 참조 가능 */
  }

@theme은 Tailwind가 빌드 타임에 값을 확정해야 하므로 var() 같은 런타임 참조를 허용하지 않음. @theme inline은 **"이 값은 다른 변수를 참조한다"**고 Tailwind에게 알려주는 것. 참조 체인이 있으면 inline 필수이다!

라고 뭔가 개념 잡는게 어려웠던거같습니다.

과제 피드백

잘했어요 재윤! 이번 과제는 레거시 코드를 현대적인 디자인 시스템으로 개편하면서 컴포넌트 설계 원칙을 체득하는 것이 목표였습니다. Before 패키지 분석이 정말 꼼꼼했어요. 스타일링, 컴포넌트 설계, 폼 관리, 아키텍처, 접근성까지 문제점을 체계적으로 정리했고, 특히 접근성 부분에서 Modal의 focus trap 미구현, aria-modal 부재까지 짚어준 부분이 좋았네요. 접근성은 신경쓰기 어려운 부분인데 잘 작성해주었습니다.

엔티티와 UI간의 관계에 대해서는 과제의 의도를 느꼈기를 바래요. useManagementData, useManagementForm, useManagementActions로 분리를 하다보면 계속해서 entityType에 따라 분기처리가 필요해진다는 것을 알게 됩니다. 이렇게 되면 나중에 Product, Order, Comment 같은 엔티티가 추가될 때 모든 훅과 컴포넌트에 분기를 추가해야 해요. 코드를 쪼갠 건 맞지만, 분기의 복잡성은 여전히 남아있게 됩니다.

그래서 더 나은 방향은 엔티티를 중심으로 분리하는 것입니다. useUserData, usePostData처럼 엔티티별로 훅을 만들고, <UserTable>, <PostTable>처럼 엔티티별 컴포넌트를 두는 거죠. 공통 UI는 <DataTable>, <FormDialog> 같은 범용 컴포넌트로 추출하고요. 그러면 새 엔티티 추가할 때 기존 코드를 건드리지 않고 새 파일만 만들면 됩니다. 다음 과제에서 이 관점의 차이를 경험해보길 바라요.

Q) @theme과 @theme inline 이해가 맞는지?

네, 맞게 이해했어요! @theme은 빌드 타임에 값이 확정되어야 해서 var() 참조가 불가능하고, @theme inline은 var() 참조가 가능하면서 중간 변수를 건너뛴다는 핵심을 잘 짚었습니다. "참조 체인이 있으면 inline 필수"라는 정리도 정확해요.
