2. 핵심 패턴 습득

State/Dispatch Context 분리

// Before: 단일 Context - 모든 구독자가 리렌더링
const ScheduleContext = createContext({ schedulesMap, setSchedulesMap });

// After: 분리 - 필요한 것만 구독

학습 포인트: setSchedulesMap은 React의 setState로 참조가 안정적. Dispatch만 구독하면 State 변경에 리렌더링되지 않음.

DndContext 테이블별 분리

Before: App 레벨 단일 DndContext → 모든 시간표 리렌더링
After: 테이블별 독립 DndContext → 해당 시간표만 영향

학습 포인트: Context 범위를 좁히면 리렌더링 범위도 좁아짐. 하지만 같은 Context 내 useDndContext() 구독자는 여전히 리렌더링됨 (라이브러리 한계).

3. TanStack Query 분석 및 Promise 캐싱 패턴

기존 코드의 문제점 분석

// Before: Promise.all 오용 - 실제로는 순차 실행됨!
async function fetchAllLectures() {
  await Promise.all([
    (await fetchMajors()),      // ← 이미 await 완료
    (await fetchLiberalArts()), // ← 이미 await 완료
    (await fetchMajors()),
    (await fetchLiberalArts()),
    (await fetchMajors()),
    (await fetchLiberalArts()),
  ]);
}

문제 분석

  • Promise.all() 안에서 각 함수 앞에 await가 있어 순차 실행됨
  • 총 6번의 API 호출이 순차적으로 발생

TanStack Query 내부 구조 분석

TanStack Query의 동작 원리를 분석하며 핵심 인사이트를 얻었습니다.

// TanStack Query 내부 (간략화)
class QueryCache {
  private queries = new Map();

  find(queryKey) {
    return this.queries.get(hashKey(queryKey));
  }

  build(queryKey, queryFn) {
    let query = this.find(queryKey);
    if (!query) {
      query = new Query(queryKey, queryFn);
      this.queries.set(hashKey(queryKey), query);
    }
    return query;
  }
}

class Query {
  private promise: Promise | null = null;

  fetch() {
    // 핵심: 이미 진행 중인 Promise가 있으면 재사용!
    if (this.promise) {
      return this.promise;
    }

    this.promise = this.queryFn().finally(() => {
      this.promise = null; // 완료 후 초기화
    });

    return this.promise;
  }
}

핵심 발견

  • 데이터가 아닌 Promise 자체를 캐싱함
  • 같은 요청이 동시에 여러 번 발생해도 하나의 네트워크 요청만 실행
  • 중복 호출 방지(request deduplication) 패턴

Promise 캐싱 패턴 직접 구현

TanStack Query의 원리를 학습하여 직접 구현하였습니다.

// api.ts - Promise 캐싱 구현

type CacheKey = 'majors' | 'liberalArts';

// 핵심: Promise 자체를 캐싱하는 Map
const cache = new Map>();

const fetchers: Record<CacheKey, () => Promise> = {
  majors: () => fetchMajors().then((res) => res.data),
  liberalArts: () => fetchLiberalArts().then((res) => res.data),
};

export function getCachedLectures(key: CacheKey): Promise {
  // 캐시에 Promise가 없을 때만 새로 생성
  if (!cache.has(key)) {
    cache.set(key, fetchers[key]());
  }
  // 같은 Promise 반환 - 결과 공유!
  return cache.get(key)!;
}

// 모든 호출이 병렬로 실행되면서 중복 요청은 자동으로 방지됨
export async function getAllLectures(): Promise {
  const results = await Promise.all([
    getCachedLectures('majors', 1),     // FETCH
    getCachedLectures('liberalArts', 2), // FETCH
    getCachedLectures('majors', 3),     // CACHE HIT!
    getCachedLectures('liberalArts', 4), // CACHE HIT!
    getCachedLectures('majors', 5),     // CACHE HIT!
    getCachedLectures('liberalArts', 6), // CACHE HIT!
  ]);

  return results.flat();
}

2. 개선된 패턴

memo 커스텀 비교 함수

// 불변 데이터 → id만 비교
const SearchTableRow = memo(Component, (prev, next) => prev.id === next.id);

// 무한스크롤 패턴 → 길이 + 첫/마지막 비교
const SearchTable = memo(Component, (prev, next) => {
  if (prev.items.length !== next.items.length) return false;
  return prev.items[0]?.id === next.items[0]?.id;
});

// Context 구독 분리 → 모든 관련 props 비교
const ScheduleTablePresenter = memo(Component, (prev, next) => {
  if (prev.tableId !== next.tableId) return false;
  if (prev.isDragging !== next.isDragging) return false; // 드래그 상태!
  if (prev.schedules === next.schedules && prev.colorMap === next.colorMap) {
    return true;
  }
  return prev.schedules.every((s, i) => s === next.schedules[i]);
});

학습 효과 분석

1. 학습 방법론

Bottom-up 학습

문제 발견 → 해결책 구현 → "라이브러리는 어떻게?" → 소스코드 분석 → 원리 이해

예시:
API 중복 호출 문제
→ Promise 캐싱으로 해결
→ "TanStack Query는 어떻게?"
→ 소스코드 분석 (QueryCache, Query 클래스)
→ "데이터가 아닌 Promise를 캐싱한다!" 인사이트 획득

실험적 학습

가설 설정 → React DevTools Profiler로 측정 → 검증 → 개선

예시:
"memo를 적용하면 리렌더링이 줄어들 것이다"
→ Profiler로 측정 → 여전히 리렌더링 발생
→ "Context 직접 구독 시 memo 무시됨" 발견
→ Container/Presenter 패턴으로 해결

2. 핵심 깨달음

memo의 한계 이해

memo는 "부모 리렌더링 시" props를 비교함.
컴포넌트가 직접 Context를 구독하면 memo 비교 함수가 호출되지 않고 바로 리렌더링됨.

// 잘못된 예: memo가 작동하지 않음
const ScheduleTableBody = memo(({ tableId }) => {
  const schedules = useSchedulesByTableId(tableId);  // Context 직접 구독!
  // schedulesMap 변경 시 → 리렌더링 (memo 무시)
});

// 올바른 예: memo 정상 작동
const ScheduleTable = ({ tableId }) => {
  const schedulesMap = useSchedulesMap();  // Container에서 구독
  return <ScheduleTablePresenter schedules={schedulesMap[tableId]} />;
};

Context API의 근본적 한계

Context는 "세분화된 구독(selective subscription)"을 지원하지 않음.
Context 값이 변경되면 모든 useContext 호출 컴포넌트가 리렌더링됨.

해결책:
1. Context 분리 (State/Dispatch)
2. Context 범위 좁히기 (테이블별 DndContext)
3. 구독 위치 최상위로 올리기 (Container 패턴)

@dnd-kit 라이브러리 한계

// @dnd-kit 내부 구조
function useDraggable({ id }) {
  const context = useDndContext();  // 내부적으로 전체 Context 구독!
  // → 같은 DndContext 내 모든 드래그 가능 컴포넌트가 리렌더링
}

참조 비교의 중요성

// 복제 시 상태 업데이트
setSchedulesMap((prev) => ({
  ...prev,  // 기존 배열들은 참조 유지!
  [`schedule-${Date.now()}`]: [...prev[targetId]]  // 새 배열만 생성
}));

// 결과:
prev['schedule-1'] === newMap['schedule-1']  // true! 참조 동일
prev['schedule-2'] === newMap['schedule-2']  // true! 참조 동일

// → 기존 시간표의 memo 비교에서 true 반환 → 리렌더링 스킵!

Promise 캐싱 인사이트 (TanStack Query 분석에서)

일반적인 캐싱: 결과 데이터를 저장
TanStack Query: Promise 자체를 저장

차이점:
- 데이터 캐싱: 완료 전 중복 요청 시 여러 네트워크 호출
- Promise 캐싱: 완료 전 중복 요청도 같은 Promise 재사용

핵심 코드:
if (!cache.has(key)) {
  cache.set(key, fetchers[key]());  // Promise 저장
}
return cache.get(key)!;  // 같은 Promise 반환

측정 없는 최적화는 추측

React DevTools Profiler로 실제 렌더링 확인 필수.
"이렇게 하면 빨라질 것 같다" → 측정으로 증명해야 함.

최종적으로, 성능 최적화 과제이긴 하지만, 오히려 성능 최적화를 하기위해 시도한 학습 방법, 라이브러리의 작동 방식 등에 대해서 직접 code를 확인하고 어떤식으로 동작하는지 파악하는 등, 다시 한번 좋은 경험을 통해 배워갔습니다!

실제 프로덕션에서 마주칠 수 있는 성능 문제들을 직접 해결본거같아 좋았습니다. Context 전역 구독으로 인한 리렌더링 문제나, 무한 스크롤 성능, React DevTools Profiler 사용법, 퍼포먼스, 라이트 하우스 탭 등 을 이번에 활용을 아주 많이 해볼수있어서 좋았습니다.

그리고, 개인적으로 복제할때 렌더링이 일어나지 않아야하는 부분이 가장 시간이 많이 들고 힘들었던 작업같습니다.여기서 React DevTools Profiler 로 가장 많이 분석해서 근본적인 원인 등을 찾아서 해결할수있었습니다.

리뷰 받고 싶은 내용

  1. 과제에서는 무한스크롤로 구현했지만, 실무에서 무한스크롤 → 가상스크롤로 전환하는 기준이 있을까요?

  2. 이번 과제에서 대부분의 컴포넌트에 memo를 적용했습니다. 모든 컴포넌트에 memo를 붙이는 것은 오버헤드인지, 아니면 기본으로 적용하는 편인지 궁금합니다.

  3. 이번 과제에서 Context의 전체 구독으로 인한 리렌더링를 경험하고 State/Dispatch 분리로 해결했습니다. 실무에서는 보통 전역 상태관리로 zustand를 사용하기도하고, 사실 이렇게 React DevTools Profiler 로 렌더링 부분에 대해서 자세히 살펴본적이 없어서 몰랐는데, 렌더링 문제로 인해서 Zustand 같은 상태관리 라이브러리를 도입하기도하나요?

과제 피드백

Q. 과제에서는 무한스크롤로 구현했지만, 실무에서 무한스크롤 → 가상스크롤로 전환하는 기준이 있을까요?

A. 무한 스크롤과 가상스크롤은 목적이 다른 것 같아요~ 무한 스크롤은 성능보다는 사용자가 끝에 다다르면 자동으로 데이터 로드하는 즉 페이지네이션류의 UI종류 라고 볼 수 있는 반면, 가상스크롤은 많은 데이터를 그릴때 성능을 고려하는 방법이지요..

Q. 이번 과제에서 대부분의 컴포넌트에 memo를 적용했습니다. 모든 컴포넌트에 memo를 붙이는 것은 오버헤드인지, 아니면 기본으로 적용하는 편인지 궁금합니다.

A. 그냥 무조건 적용한다기보다는 말씀하신대로 필요할때 적용합니다. 하지만 대부분 필요한 경우가 많긴 하죠 ㅎㅎ 다만 부모가 렌더링되면 조건없이 다시 렌더링해야 하는 상황이면 성능 때문이 아니라 안씀으로 "의도"를 표현할 수 있을 것 같아요. 생각보다 메모가 비용이 별로 안들어서 필요할때 부담없이 사용하셔도 됩니다.

Q. 렌더링 문제로 인해서 Zustand 같은 상태관리 라이브러리를 도입하기도하나요?

A. 컨텍스트는 말씀하신대로 세밀한 렌더링 조율이 힘들기 때문에 상태가 자주 변경되지 않는 말그대로 "공유"자체가 목적일때 적합할 것 같아요. 상태가 자주 변경되고 상태가 많은 경우라면 주스탠드같은 상태관리를 사용하는 것이 나은 것이죠. 렌더링 뿐만 아니라 상태관리자체에도 여러가지 기능을 포함하고 있어서 상태관리도구를 사용합니다. 반대로 탄스택쿼리와 같은 캐싱 기반 패치도구로 인해 공유해야할 상태를 캐시를 통해하게 되서 상태관리도구의 커버범위가 조금 줄어들기도 했습니다.
