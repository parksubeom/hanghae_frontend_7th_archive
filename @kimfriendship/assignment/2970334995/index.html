과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리

크로마틱 라이브러리 프로젝트 토큰 이슈로 워크플로우가 개인 레포에서만 실행되어 라이브러리 링크와 스크린샷을 첨부했습니다.

  - 추가로 필요한 테스트가 생길 수 있지만, 작성은 자유롭게 선택해서 진행해주세요.

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요. 거시적으로 보았을 때, 우리 앱에는 트로피 테스트 전략이 더 적합하다고 생각했습니다. 피라미드 전략은 금융 서비스처럼 안전성과 안정성이 중요한 서비스에 어울리고, 트로피 전략은 사용자 인터랙션이 많고 비즈니스 로직이 복잡한 서비스에 적합하다고 합니다. 이러한 관점에서 우리 앱은 트로피 구조가 더 맞다고 판단했습니다.

    다만, 현재 테스트 비율을 보면 실제로는 피라미드 형태에 가깝게 구성되어 있으며, 이 또한 나쁘지 않다고 생각합니다. 따라서 부족한 부분을 중심으로 보완해 나가는 방향이 좋다고 판단했습니다.

    테스트가 누락된 부분이 많지만, 이번에는 새롭게 추가된 드래그 앤 드롭(DnD) 기능과 캘린더 셀 클릭 시 폼에 날짜가 자동 입력되는 기능에 대한 테스트를 우선적으로 추가하고자 했습니다. 또한 useEventForm 훅이 핵심적인 기능임에도 테스트가 빠져 있어, 이 부분도 함께 보완하고 싶었습니다. 시간상의 제약으로 모든 테스트를 작성하지는 못했지만, DnD 기능에 대해서는 통합 테스트와 E2E 테스트를 추가했습니다.

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요. 저희 팀이 의논한 흔적들은 노션에 남겨두었습니다. 페어코딩을 하지는 못했지만 전반적인 의견은 아래와 같이 추려졌습니다.

    • DND 기능을 e2e로 테스트하기
    • 날짜 클릭시 자동입력 기능을 통합테스트로 테스트하기
    • useEventForm을 단위 테스트하기

  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요. 7ce4c3a, e74eee9

    실제로 추가해본 테스트는 DND 통합/e2e 테스트입니다. e2e는 비용이 많이 들고, e2e에서 검증하려는 것은 실제 drag and drop이 가능한지 여부이기 때문에 하나만 있어도 충분할 거라고 생각했습니다. 나머지 비즈니스 로직은 통합테스트에서 확인했습니다.

과제 셀프회고

기술적 성장

  1. 스토리북 작성하기
    • 기능추가하고 테스트 코드 짜다 보면 리팩토링 되어있는 상태가 작업하기 수월할 거라고 생각해서 컴포넌트들로 나누는겸 스토리북을 작성했습니다.
    • 컴포넌트들은 과제로 주신 시각적 회귀 테스트 대상에 맞게 나누었고, 각각의 큰 컴포넌트들에서 최소단위로 쪼개어 컴포넌트를 만들어 스토리를 추가했습니다.
    • 초반에는 직접 리팩토링 및 스토리 작성을 하였는데, 꽤나 시간 소요가 되어서 AI에게 맡겨보았습니다. 생각보다 컴포넌트를 잘 나눠주고 스토리들도 잘 작성해주었습니다.
    • 각 셀 텍스트 길이에 따른 처리를 어떻게 해야할까 고민하다가, CalendarCell의 스토리로 셀의 너비보다 긴 텍스트를 입력했을 때의 UI를 추가했습니다.
  2. 기능 구현하기
    • 지피티에게 가장 직관적이고 현대적인 DND 라이브러리로 dnd-kit을 추천받아 사용하였습니다.
    • 시간이 없어서 기능 구현도 최대한 AI에게 맡겨보았습니다.
    • 기존에 만들어진 test-design, test-code 에이전트들을 활용하여 명세를 문서로 만들어달라고 했습니다. 생각보다 엄청 구체적으로 정리해주고, 기존 코드 파악도 잘 된 것 같았습니다. 예를 들어 드래그오버 할 때 시각적으로 확인할 수 있게 배경색 바꾸기, 일정 이동 후 성공/실패 스낵바 노출하기 등 꼼꼼히 작성해주었습니다.
    • 그리고 놀라웠던 건 그렇게 디테일한 아웃풋을 위해 명령을 명확화하는 질문들을 한다는 것이었습니다. 에이전트들이 일을 너무 잘해서 에이전트 문서들도 다시 복습해보고 싶습니다.
    • 명세를 만든 후에는, 작은 단위로 개발하고 매번 확인 검증하면서 커밋을 남기기 위해 투두리스트를 만들어달라고 했습니다. 투두리스트도 구체적인 단위로 잘 만들어줬는데, 사람이 짜는 순서랑 조금 다른 것 같았습니다. 보통 사람이 개발을 할 때에는 동작을 확인하면서 작업을 하다보니 눈으로 확인할 수 있는 흐름대로 코드를 짜게 되는 것 같은데, AI는 이미 머리속에 계산이 다 되어있고 컴포넌트 단위로 수정하는 느낌이었습니다. 그래서 흐름을 따라가기 어려웠고 검증하기도 쉽지 않았습니다. 직접 테스트하면서 따라갈 수밖에 없었고, 버그를 찾더라도 아직 구현이 안되어서 그런지, 버그가 생긴건지 구분이 잘 안됐습니다.
      챗 히스토리
  3. e2e 테스트 작성해보기
    • 먼저 e2e 테스트 케이스를 작성할 때 고려할 점에 대해 살펴보았습니다. 제가 처음 가지고 있던 e2e에 대한 인식은 하나의 시나리오에서 정말 유저 플로우처럼 여러 기능을 사용하며 검증하는 형태여서 호흡이 긴 테스트라고 생각했습니다. 그러나 테스트가 실패했을 때 디버깅하기 어렵다는 점과 시간이 너무 많이 걸리는 이유로 e2e 또한 하나의 케이스에서는 하나만 검증해야하는 걸 알았습니다. 또 케이스를 작성하다보니, 통합테스트와 똑같은 테스트를 중복해서 확인하는 느낌이 많이 들었습니다. 하지만 e2e에서는 단순 기능 검증이 아닌 서비스가 실제로 비즈니스 로직대로 동작하는가를 확인해야한다는 것에 초점을 두려고 노력했습니다.
    • UI를 확인하면서 테스트를 할 수 있으니까 너무 좋았습니다. 통합테스트 할 때보다 훨씬 수월하게 테스트 코드를 작성할 수 있었습니다.
    • 알림 기능 검증할 때, e2e에서는 실제 시간만을 사용해야하나 고민했는데, 통합테스트에서 fakeTimer를 사용했던 것처럼 playwright에서도 시간 설정하는 기능을 제공하고 있어서 적용하였습니다.
    • e2e를 병렬로 테스트 가능하게 하려고 했는데 쉽지 않았습니다. 테스트용 db인 e2e.json을 따로 두고 beforeEach에서 매번 db를 리셋시켜주긴 했지만 테스트가 서로 영향을 받았습니다. 그래서 serial하게 실행하는 방향으로 수정했습니다.
  4. chromatic 연동하여 시각적 회귀테스트 하기
    • 생각보다 chromatic 설치하고 워크플로우 만드는 건 어렵지 않았던 것 같습니다.
    • 회사에서 스토리북을 사용하고 있고, 점진적으로 컴포넌트들을 추가하고 있는데 크로마틱을 도입해서 활용해보면 도움이 많이 될 것 같습니다.
    • 지난 주에 팀원들이랑 UI가 많이 바뀌지 않는 어플리케이션을 만들 때 오히려 UI 변화가 생기진 않았는지 검증하기 위해 시각적 회귀 테스트가 필요할 것 같다고 이야기 했었는데요. 막상 직접 해보니, UI 변경사항이 없었으면 아예 비교대상이 없으니 크로마틱이 돌지 않게 되더라구요. 그래서 디자인이 자주 바뀌지 않는다면 아예 테스트를 돌려볼 일조차 없는 것 같다고 생각이 바뀌었습니다. 변경사항이 생겼을 때 테스트를 통해 의도된 변경사항은 옳게 변경 되었는지, 그 외의 사이드이펙트로 변경된 부분은 없는지 확인하는 시나리오가 더 바람직한 게 아닐까 싶습니다.

코드 품질

코드 품질은 아주 낮은 것 같습니다.. 사실 AI가 짜준 코드를 검증해야 하는 단계에서, 코드를 꼼꼼히 확인하지 못한 것도 있고 리팩토링을 더 할 여유가 없었네요ㅜㅜ 이렇게 AI한테 의존하다보니 과제를 하는 의미가 있나 싶기도 한데, 그래도 AI가 벌어주는 시간만큼 단기간에 많은 걸 체험할 수 있어서 좋기도 했습니다.

학습 효과 분석

3주간의 테스트 챕터를 하면서, 단위테스트부터 통합, e2e, 시각적회귀 테스트까지 다양하게 맛볼 수 있어서 좋았습니다. 각각의 테스트 코드를 짜보면서 직접 그 특성들을 배우고 느끼고, 그것들을 바탕으로 어떤 상황에서 어떤 테스트 전략을 짜야할지 고민할 수 있게 된 것 같습니다.

리뷰 받고 싶은 내용

  • 과제에서는 테스트용 db를 따로 구축했는데, 실무에서는 어떻게 해야할지 잘 그려지지 않는 것 같습니다. 백엔드와의 소통과 협업이 필요하다고 하셨는데 구체적으로 어떤 커뮤니케이션을 하고 어떤 요청을 드려야할지 설명해주시면 감사하겠습니다!
  • 과제하면서 테스트 케이스를 해피케이스에 대해서만 검증하는 것 같다는 느낌이 들었습니다. 당연히 기능들이 정상적인 작동을 하는지 확인하는 게 가장 중요하지만, 어떠한 이유로 기능이 깨졌을 때 그거에 대한 적절한 대응이 되고 있는가에 대해서는 테스트하지 않아도 괜찮은지 궁금합니다.

과제 피드백

수고했어요 우정. 지난 3주간 테스트 라는 챕터를 잘 진행해주었네요. 본인이 직접하다가 또 AI를 통해서도 해보다가 이를 직접적으로 비교해가면서 차이점을 익혀하는 과정이 너무 좋아요. AI에게 시켰을때 잘 하고 있는지 아닌지도 명확히 구분을 하듯이 구분이 되어 간다는 점에서 AI를 잘 활용하고 실력이 올라갔다 생각합니다.

UI가 보여서 e2e 테스트 과정은 좋았군요. e2e와 통합테스트는 분명 비슷하지만 e2e는 로직을 생각하지 않고 그냥 사용자 봐야 하는거 -> 눌러야 하는 거 -> 변해야 하는 거를 측정하는 거라면 통합 테스트는 로직을 중심으로 체크를 한다는 면에서 관점이 다르게 적용이 되더라구요. 저도 e2e를 작성하는게 좀 더 수월하다고 생각해요.

Q) 과제에서는 테스트용 db를 따로 구축했는데, 실무에서는 어떻게 해야할지 잘 그려지지 않는 것 같습니다. 백엔드와의 소통과 협업이 필요하다고 하셨는데 구체적으로 어떤 커뮤니케이션을 하고 어떤 요청을 드려야할지 설명해주시면 감사하겠습니다!

=> 일단 인프라적 측면에서 백엔드의 도움을 받고 실제로 DB의 값을 조회하고 읽고 쓰고 삭제해야 하므로 테스트용 서버가 별도로 필요하겠죠. 특히나 테스트의 일관성 측면을 위해서는 DB의 초기화도 필요하고 이메일 발송, 인증, 결제 등등은 테스트용 mocking도 필요하겠죠.

=> 우선은 테스트를 한다면 스스로 테스트 환경을 다 갖춰본다면 쓰기가 필요한상황이 되었을때 먼저 설명해보세요. 백엔드 개발자분이 알아듣는다면 함께 현 상황에서 논의해볼 수 있고, 그럴 수 없는 상황이라면 읽기 전용으로만 시도하거나 아쉽지만 내부 MSW 정도만 가지고 진행한다거나 지금처럼 별도의 테스트 DB를 따로 구축하는 형태로 시도해보면 될거에요.

=> 데이터를 격리할 수 있는 방법, 데이터를 되돌릴 수 이는 방법, DB가 아닌 실제 기능을 mocking하는 방법등 테스트를 했다고 기존 상태에서 달라지지 않도록 만들 수 있는 방법을 중심으로 함께 논의해보길 바래요.
