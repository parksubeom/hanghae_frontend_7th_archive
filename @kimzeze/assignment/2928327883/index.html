[4팀 김도현] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by kimzeze2025.10.20

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

  • handlers.ts는 전역 변수 mockEvents를 모든 테스트가 공유합니다. 만약 테스트가 병렬로 실행된다면 아래와 같은 결과가 나올거로 예상했습니다.
  1. 테스트 A가 이벤트 추가 → 전역 배열 수정
  2. 테스트 B가 동시 실행 → 테스트 A의 데이터까지 조회
  3. 결과: 테스트 B가 예상과 다른 데이터로 실패

제가 생각한 해결 방법은 클로저 개념을 사용하여 데이터를 분리시키는 것입니다.

export const setupMockHandlerCreation = (initEvents: Event[] = []) => {
  const events = [...initEvents]; // 함수 호출마다 새로운 배열 생성
  
  return [
    http.get('/api/events', () => {
      return HttpResponse.json({ events: [...events] });
    }),
    http.post('/api/events', async ({ request }) => {
      const newEvent = await request.json();
      events.push(newEvent); // 이 테스트만 배열에 추가
      return HttpResponse.json(newEvent, { status: 201 });
    }),
  ];
};

각 테스트에서 함수를 호출할 때마다 새로운 events 배열이 클로저에 생성되고, 반환된 핸들러들은 각자의 독립적인 배열만 참조하여 테스트 간 데이터 격리가 보장된다고 생각했습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

  1. Event Factory 패턴
export const createMockEvent = (overrides: Partial<Event> = {}): Event => {
  return {
    id: '1',
    title: '기본 제목',
    date: '2025-10-01',
    // ... 기본값들
    ...overrides, // 필요한 것만 덮어쓰기
  };
};

매번 새로운 객체를 생성하여 테스트 간 참조 공유를 방지하고, Event 타입이 변경되어도 factory만 수정하면 되도록 했습니다.

  1. Fake Timer 설정
beforeEach(() => {
  vi.useFakeTimers();
  vi.setSystemTime('2025-10-01');
});

afterEach(() => {
  vi.useRealTimers();
});

날짜/시간 의존적인 테스트(알림, 캘린더)에서 시간을 고정해서 일관된 결과를 얻도록 했습니다.

과제 셀프회고

기술적 성장

테스트 코드를 처음 작성해보다

이번 과제를 통해 처음으로 테스트 코드를 작성해봤습니다. AAA 패턴(Arrange-Act-Assert), GWT 패턴(Given-When-Then) 같은 테스트 작성 패턴을 학습하고 테스트를 작성하면서 자연스럽게 "준비 → 실행 → 검증" 순서로 생각하게 되었습니다.

독립적인 테스트

말로만 들었던 "테스트는 독립적이어야 한다"를 직접 구현하며 왜 중요한지 깨달았습니다. 처음에는 테스트 순서를 바꿨을 때 결과가 달라져서 당황했는데, 클로저를 사용해 데이터를 격리하니 테스트 실행 순서를 바꿔도 결과가 동일한 것을 확인했을 때 뿌듯했습니다.

MSW를 처음 사용해보다

실제 API를 호출하지 않고도 테스트할 수 있다는 게 신기했습니다. http.get, http.post 같은 핸들러로 가짜 응답을 만들 수 있다는 것도 처음 알았습니다.

클로저 개념의 실전 활용

클로저는 이론으로만 알고 있었는데, 실제로 문제를 해결하는 데 사용해보니 이해가 확실히 되었습니다. 함수 안에서 선언한 변수를 반환된 함수들이 "기억"한다는 게 이런 거구나 싶었습니다.

코드 품질

만족스러운 부분

Factory 패턴 도입: createMockEvent 함수를 만들어서 테스트 코드가 훨씬 간결해졌습니다. 필요한 속성만 오버라이드하면 돼서 편했습니다.

아쉬운 부분

describe을 나누지 못한 것: 작성하면서 나눠서 구분하면 더 깔끔하게 정리되겠다고 생각은 했으나, 시간 여유가 부족하여 최대한 이미 적힌 방식에 대해서 테스트 통과를 먼저 한뒤 개선해야겠다고 생각했습니다.

  • 구현하는데 급급해서 좀 더 리팩토링을 하지 못한 것.... 완전히 내 것으로 만든 느낌이 들지 않아 너무 아쉽습니다

학습 효과 분석

테스트 코드를 작성해본 경험 자체가 저에게 있어서 가장 뜻깊었습니다. 테스트 코드를 작성해야 된다고 머릿속으로만 항상 생각하고 추상적인 개념으로만 있었습니다. 직접 이렇게 일주일간 테스트 코드를 작성해보면서 console.log로 확인하던 게 이런 걸로 확인할 수 있구나 싶었던 부분도 있었습니다. 특히 expect로 값을 비교하니까 훨씬 명확하고 자동화할 수 있다는 게 좋았습니다. 경계값 테스트를 하면서도 실무에서 코드를 작성하면서 안일하게 작성했던 부분이 많았던 과거를 되돌아보게 되었습니다. "13월을 입력하면?", "0일을 입력하면?" 이런 걸 제대로 고려하지 않았던 것 같습니다. 정답이 있는 듯 하면서 없는 듯한 이 테스트 코드를 작성하는 과정에서 함수 작성에 대해서도 많이 고민하게 되는 시간이었습니다. "이 함수는 무엇을 책임져야 하는가?", "어떤 입력에 대해 어떤 출력을 해야 하는가?"를 더 명확하게 생각하게 되었습니다.

과제 피드백

처음 과제 설명을 읽었을 때, 테스트 코드를 처음 보는 입장에서는 조금 막막했습니다. MSW또한 처음 사용해보는 상황이라 양에 압도되는 느낌이 들어 초반에 코드를 작성하는 시간보다는 알아보는 시간이 오래걸렸습니다.

리뷰 받고 싶은 내용

  1. 클로저를 사용한 데이터 격리 방식이 적절한가요?: 클로저로 데이터를 격리했는데, 이게 실무에서도 통용되는 좋은 방법인지 궁금합니다. 혹시 더 나은 패턴이 있을까요?

  2. Factory 패턴 사용의 적절성 createMockEvent로 테스트 데이터를 만들었는데, 이게 좋은 접근인지 아니면 오히려 복잡도만 높이는 건지 확신이 서지 않습니다. 실무에서는 어느 정도 수준까지 추상화하는 게 적절한가요?

  3. 테스트 구조 개선 현재 describe를 나누지 못한 부분이 많은데, 어떤 기준으로 describe를 나누면 가독성이 좋아질까요? 예를 들어 "정상 케이스 / 에러 케이스"로 나누는 게 좋을지, 아니면 "기능별"로 나누는 게 좋을지 궁금합니다.

궁금한 점

이번 과제를 하면서 실무에서도 적용하고 싶다고 계속 생각이 들었는데, 기획의 큰 방향이 자주 바뀌거나 예상치못한 것들이 추가되는 스타트업에서는 어떤 방식으로 접근해야 좋을지 궁금합니다!

Q. 클로저를 사용한 데이터 격리 방식이 적절한가요?: 클로저로 데이터를 격리했는데, 이게 실무에서도 통용되는 좋은 방법인지 궁금합니다. 혹시 더 나은 패턴이 있을까요.

A. 실제 코드를 주셨으면 더 좋았을 것 같아요. 자바스크립트에서 특정 데이터를 물리적으로 격리시키지만 특정 함수나 메서드에서는 접근이 가능하게 만들고 싶을 때 즉 캡슐화와 은닉화를 위해서는 궁극적으로 클로저가 제일 명확한 답이라고 생각합니다. 게터 세터로 프로퍼티 필드로 만들 수 있겠지만 이건 변수보다는 값을 연산해내는 도구로 더 활용이 되는 것으로 봐야할 것 같고욥. 자바스크립트에서는 데이터를 격리하고 캡슐화하는 방법이 몇가지 없어요. 클로저는 아주 좋은 답입니다.

Q. Factory 패턴 사용의 적절성 createMockEvent로 테스트 데이터를 만들었는데, 이게 좋은 접근인지 아니면 오히려 복잡도만 높이는 건지 확신이 서지 않습니다. 실무에서는 어느 정도 수준까지 추상화하는 게 적절한가요?

A. 목데이터를 생성하는 함수는 자주 사용됩니다. 다만 로직을 많이 단순하게 유지해야겠죠. 지금처럼 선언적으로 만든 함수라면 괜찮을 것 같아요. 저는 어쨌거나 함수는 함수니까 테스트를 위해 사용되는 데이터는 가급적 스태틱하게 생성하는 것을 선호하는데요. 이렇게 데이터의 양이 많고 중복적이라면 의도적은 필드의 가시적인 변경을 위해 비슷한 함수를 만들어 사용하고 있습니다.

다만 가급적 테스트를 위해 사용하는 추가 함수(로직)는 지양하는 편입니다.(그 함수는 누가 테스트하나욥..ㅎㅎ)

Q. 테스트 구조 개선 현재 describe를 나누지 못한 부분이 많은데, 어떤 기준으로 describe를 나누면 가독성이 좋아질까요? 예를 들어 "정상 케이스 / 에러 케이스"로 나누는 게 좋을지, 아니면 "기능별"로 나누는 게 좋을지 궁금합니다.

A. 특정 모듈이 무엇을 할 수 있는지를 기준으로 TC를 작성하는 것이 좋습니다. 이말은 기능별에 가깝겠죠. 궁극적으로는 describe-it 이런 뎊스를 가진 구조가 제일 이해하기 쉬워요. 다만 조건이 복잡하거나 조건을 재사용하기 위해서 조건을 공유할 수 있게 describe로 중간 뎊스를 둘 수도 있을 것 같고요. 여기서의 중간 뎊스는 필요에 의해서 필요할때 추가해도 될 것 같습니다. 예를들어 로그인 모듈이라면 로그인 모듈이란 describe안에 기능적으로 분리가능한 "비밀번호 찾기", "아이디 찾기", "로그인", "로그인 실패" 이런 중간 뎊스로 describe를 둘 수 있겠죠. 이건 그냥 의도했기 때문에 구분하는 것이고 이것에 대한 정답은 없을 것 같아요. 현재로 제일 적합하다고 판단되는 의도가 있고 그 의도대로 작성하면될 것 같습니다.

말씀하신 정상케이스와 에러케이스로 나누었을때 어떤 장점이 있을까요? 장점이 있다고 판단되면 그렇게 나누셔도 좋지만, 저라면 특정 기능에서 정상 케이스 it와 에러케이스 it를 묶어서 보고 싶을 것 같아요.
