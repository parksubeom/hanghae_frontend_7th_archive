과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리

- 기존에 작성한 테스트는 깨지지 않도록 해주세요.
  - 이 과정에서 테스트가 분리될 수도 작성법이 수정될 수도 있습니다.
  - 정말 불필요한 케이스가 아니면, 테스트 케이스를 삭제하는건 최소화 해주세요.
  - 추가로 필요한 테스트가 생길 수 있지만, 작성은 자유롭게 선택해서 진행해주세요.

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

과제 셀프회고

기술적 성장

E2E와 시각적 회귀 테스트라는 개념도 처음 알게되었고 playwright, chromatic도 초면이었습니다. storybook은 공부하면서 정말 간단하게 구성만 해봤었습니다. 처음에는 조금 막막했는데 팀원들과 그리고 discord 채널에 올라오는 글들도 확인하며 하나씩 사용해봤을 때, 도입하는 과정자체는 AI의 도움을 받아 어렵지는 않았습니다! 무엇보다도 항상 개발할 때 일단 한번 뭐가되든 한번 해보고, 아웃풋을 만들어 내보고 나면 다음에 다시 시작할 때 좀 더 빠르게 시작할 수 있고, 부족했던 부분들을 보면서 개인의 방향성을 빠르게 찾을 수 있는 것 같다고 생각하기에 좋은 경험이었습니다!

코드 품질

AI를 많이 활용하였고, 스스로 공부하면서 AI가 만든 코드를 내가 컨트롤하면서 따라갔다고 할 수 없는 것 같아 코드 품질에는 신경을 못쓴것 같아 너무 아쉽습니다.

학습 효과 분석

과제 피드백

테스트코드 과정 1~3주차를 겪으면서, 테스트코드에서 unit, 통합, e2e테스트라는게 어떤 느낌인지 짐작할 수 있었던 것 같아서 좋은 경험이었습니다. 특히 툴들을 한번씩 경험해보고, 어찌됐던 한번 코드를 만들어본 것 자체가 저에게는 의미있는 시간이었다고 느껴집니다. 1주차에는 사실 개념도 모른채 그냥 시작해본 느낌이었는데 3주동안 하면서 나름 테스트 코드를 만들어 나갈 때 이런부분은 통합이구나, 이런부분은 유닛으로 검증하는게 좋겠다. 같은 생각까지 갈 수 있었던게 좋았습니다!

리뷰 받고 싶은 내용

storybook-chromatic을 세팅하고 적용해보면서, 실무에서는 시안도 변경되면서 각 push, PR마다 변경사항을 확인하고 데이터로 만드는 과정에서 많은 도움이 되겠다고 생각습니다. 지금은 역량이 부족하여 snapshot을 가지고 업무적으로 어떤 대화를 나눌수 있지? 에 대한 생각이 짐작이 가지 않습니다.

Storybook-Chromatic 실무 활용 방안에 관하여

Storybook-Chromatic을 세팅하고 적용해보면서, 실무에서는 시안이 변경되면서 각 push, PR마다 변경사항을 확인하고 데이터로 만드는 과정에서 많은 도움이 되겠다고 생각합니다.

하지만 지금은 역량이 부족하여 snapshot을 가지고 업무적으로 어떤 대화를 나눌 수 있을지에 대한 생각이 짐작이 가지 않습니다. (디자이너가 있다면 피그마에 버전 기록이 남지 않나에 대한 생각도 듭니다 ㅠㅠ... 프론트엔드는 주어진 시안을 똑같이 만든다는 가정하에...)

예를 들어:

  • PM이나 기획자가 "이전 버전과 뭐가 바뀌었는지 보고 싶다"고 할 때, Chromatic 리포트를 전달하는 방식으로 사용하나요?!
  • 또는 개발자 간 코드 리뷰에서 "이 변경사항이 UI에 어떤 영향을 미치는지"를 논의할 때, snapshot을 보면서 이야기하기도 하는지 궁금합니다

코치님께서 이런 시각적 회귀 테스트를 활용하여 효과적으로 소통했던 경험이 있다면 도움이 많이 될 것 같습니다!

과제 피드백

수고했어요 도현! 지난 3주간 테스트를 경험하면서 대규모 프로젝트에서 품질을 어떻게 지켜내는지 간접적으로나마 체험하는 시간이 되었기를 바랍니다. 도현은 E2E, 시각적 회귀 테스트, Playwright, Chromatic, Storybook 모두 처음 접했다고 했는데, 그럼에도 불구하고 일단 한번 뭐가 되든 해보고 아웃풋을 만들어냈다는 점이 정말 좋네요!

AI를 많이 활용하는 과정에서 코드품질 컨트롤을 못한게 아쉽다고 했는데 다행히 테스트 쪽 코드는 있는 그대로 작성하는 편이라 AI가 만들어도 코드 품질이 그렇게 나빠지지 않으니 테스트는 AI의 도움을 받으면서 점진적으로 이해도를 높이는 방향으로 진행해보세요.

Q) Storybook-Chromatic을 실무에서 어떻게 활용하는지? snapshot을 가지고 업무적으로 어떤 대화를 나누는지?

=> 아쉽네요. 제가 커리어를 하면서 Storybook-Chromatic를 실무에서 써본 경험은 전무해서 제 경험을 바탕으로 설명을 드리기는 어렵네요. 함께 협업하는 팀에서 PR마다 snapshot이 올라간 것을 본 적이 있는데 든든할 것 같다는 생각이 듭니다. 종종 리팩토룅 과정에서 CSS 단순 누락으로 인해 핫픽스를 할때면 이런거 알려주면 참 좋겠다 싶었어요. 프로젝트가 너무 커서 그걸 하나하나 눈으로 다 확인하기에는 비용이 너무나 많이 들거든요.

나중에 다른 분들의 경험을 알게 되면 또 공유 남겨볼게요. 수고많았습니다. 이제 새로운 챕터에서 함께 해요!
