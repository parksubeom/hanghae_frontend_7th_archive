과제 회고

Before 패키지에서 발견한 문제점

1. UI와 비즈니스 로직의 혼재

Before 패키지에서 가장 먼저 눈에 띈 문제는 UI 컴포넌트가 도메인 지식을 직접 알고 있다는 점이었습니다.

  • Button 컴포넌트가 entityType, action 같은 비즈니스 개념을 props로 받음
  • FormInput이 "예약어 체크", "회사 이메일만 허용" 같은 비즈니스 규칙을 직접 검증
  • Table이 User/Post 도메인별 렌더링 로직을 내부에 포함

이런 구조는 컴포넌트 재사용성을 크게 떨어뜨립니다. 비즈니스 규칙이 변경되면 UI 컴포넌트까지 수정해야 하고, 다른 프로젝트에서는 전혀 사용할 수 없는 컴포넌트가 됩니다.

2. 일관성 없는 컴포넌트 API

같은 의미의 props가 컴포넌트마다 다른 이름을 사용하고 있었습니다.

  • FormInput의 helpText, FormSelect의 help, FormTextarea의 description
  • 크기 조절: 어떤 컴포넌트는 width, 어떤 컴포넌트는 size

개발자가 매번 각 컴포넌트의 API를 따로 외워야 하고, 팀원마다 다르게 사용할 가능성이 높습니다.

3. 혼재된 스타일링 방식과 디자인 토큰 부재

인라인 스타일, Tailwind 클래스, 하드코딩된 색상 값이 섞여 있었습니다.

// 인라인 스타일
style={{ padding: '10px', backgroundColor: '#007bff' }}

// 하드코딩된 색상
const primaryColor = '#007bff';
const dangerColor = '#d32f2f';

디자인 토큰이 없어서 색상, 간격, 타이포그래피가 컴포넌트마다 제각각이고, 일관성 있는 디자인을 유지하기 어려웠습니다.

4. 타입 안전성 부족

Before 패키지에서 Stats 인터페이스를 보면 stat1, stat2, stat3, stat4 같은 의미없는 이름을 사용하고 있었습니다. 이런 구조는 어떤 통계가 무엇을 의미하는지 코드만 봐서는 알 수 없고, 실수로 잘못된 통계를 사용해도 타입 체크에서 걸러지지 않습니다.

과제를 진행하며 집중하고 고민한 부분

1. CVA를 어디까지 적용해야 할까?

과제에서 "CVA를 활용한 variants 패턴 적용"이 요구사항이었는데, shadcn/ui 컴포넌트들이 이미 내부적으로 CVA를 사용하고 있었습니다.

처음에는 shadcn/ui 컴포넌트를 그대로 사용하면 CVA를 "활용"한 게 아닌가 싶어서 고민했습니다. 모든 커스텀 컴포넌트에 CVA를 적용해야 하나, 아니면 대표적인 컴포넌트 하나만 시연하면 되나 싶었습니다.

결국 다음과 같이 결정했습니다:

  • PageHeader 컴포넌트를 새로 만들어 CVA variants 패턴을 직접 구현
  • size (sm, md, lg), align (left, center, right) variants 추가
  • ManagementStats 컴포넌트에서도 CVA로 statCard와 statText variants 구현
  • shadcn/ui의 Button, Badge, Alert 등은 이미 CVA를 잘 활용하고 있으므로 참고 사례로 활용

2. shadcn/ui 컴포넌트를 직접 수정할까, 감싸서 사용할까?

shadcn/ui는 복사-붙여넣기 방식이라 직접 수정이 가능합니다. 하지만 프로젝트 요구사항에 맞게 커스터마이징할 때 고민이 있었습니다.

Button에 cursor-pointer를 추가하고 싶은데 직접 수정해야 할까, 아니면 wrapper 컴포넌트를 만들어야 할까? 나중에 shadcn/ui 업데이트를 받으려면 어떻게 해야 할까?

일단 기본적인 스타일 수정(cursor-pointer 등)은 직접 수정하기로 했습니다. shadcn/ui의 철학이 "복사해서 자기 것으로 만들기"이므로 직접 수정이 맞다고 판단했습니다. 하지만 도메인별 로직이 필요한 경우는 별도 컴포넌트로 분리했습니다 (예: UserFormFields, PostFormFields).

근데 실무에서는 어떻게 하는게 좋을지 궁금합니다. 원본을 수정하면 나중에 shadcn/ui가 업데이트됐을 때 변경사항을 놓칠 수 있을 것 같은데, 그렇다고 모든 커스터마이징을 wrapper로 만들면 너무 복잡해질 것 같습니다.

3. Atomic Design을 폴더 구조에 반영해야 할까?

과제에서 "Atomic Design의 개념을 이해하되, 폴더 구조는 실용적으로"라는 방향을 제시했습니다. shadcn/ui처럼 components/ui로 단순하게 가져갈까 고민했습니다. 컴포넌트 분류 기준을 어떻게 세우는게 좋을지도 애매했습니다.

결국 이렇게 정리했습니다:

components/
├── ui/           # shadcn/ui 기반 재사용 컴포넌트
├── layout/       # 레이아웃 컴포넌트 (AppHeader)
├── shared/       # 프로젝트 공통 컴포넌트 (PageHeader, AlertContainer, ThemeToggle)
└── domain/       # 도메인별 컴포넌트
    ├── user/     # UserFormFields
    ├── post/     # PostFormFields
    └── management/ # ManagementStats, ManagementTabs

Atomic Design의 "조합과 재사용" 개념은 이해하되, 폴더는 용도별로 구분해서 import 경로를 단순하게 유지하고 컴포넌트를 찾기 쉽게 했습니다.

4. Design Token을 어디까지 확장할까?

TailwindCSS 4.0의 @theme inline으로 토큰을 정의할 수 있었는데, 범위를 어디까지 할지 고민했습니다.

모든 색상을 토큰화해야 할까? 타이포그래피 스케일을 더 세분화해야 할까? spacing, border-radius, shadow도 모두 토큰으로 만들까?

일단 현재 프로젝트에서 실제로 사용하는 값들 위주로 정의하고, spacing, typography, colors, shadows, border-radius 기본 구조만 구축했습니다. 필요에 따라 나중에 확장 가능하도록 구조만 잡아뒀습니다.

근데 spacing 같은 경우 space-1부터 space-20까지 미리 다 정의하는게 좋을지 궁금합니다. 디테일하게 만들어두면 나중에 편할 것 같긴 한데, 실제로 안 쓰는 값들까지 정의하면 오히려 복잡해질 것 같기도 하고...

5. 컴포넌트 분리 기준: UI vs 도메인

UI/비즈니스 로직을 분리하려고 했는데, 경계가 애매한 경우가 많았습니다.

UserFormFields는 UI 컴포넌트일까, 도메인 컴포넌트일까? Form 필드 배치는 UI인데, 필드 종류는 도메인 지식 아닌가? 이런 컴포넌트를 어떻게 분류하고 관리해야 할까?

고민 끝에 이렇게 정리했습니다:

  • 순수 UI 컴포넌트 (components/ui): 도메인을 전혀 모르는 재사용 컴포넌트 - Button, Input, Card, Badge 등
  • 공통 컴포넌트 (components/shared): 프로젝트 공통이지만 특정 도메인에 속하지 않음 - PageHeader, AlertContainer 등
  • 레이아웃 컴포넌트 (components/layout): 페이지 레이아웃 관련 - AppHeader 등
  • 도메인 컴포넌트 (components/domain): 특정 도메인 지식을 포함하지만 UI 위주 - UserFormFields, PostFormFields, ManagementStats 등
  • 비즈니스 로직: 커스텀 훅으로 완전히 분리 - useManagementData, useEntityStats

6. Stats 인터페이스 리팩토링

Before 패키지의 stat1, stat2, stat3, stat4를 보면서 이것도 바꿔야겠다고 생각이 들었습니다. 코드만 봐서는 무슨 통계인지 전혀 알 수 없고, 나중에 유지보수할 때 헷갈릴 것 같았습니다.

타입을 어떻게 분리할지 고민했는데, 사용자 통계와 게시글 통계는 완전히 다른 데이터 구조니까 UserStats와 PostStats로 아예 분리하기로 했습니다.

interface UserStats {
  total: number;
  active: Stat; // stat1 → 의미있는 이름으로
  inactive: Stat; // stat2
  suspended: Stat; // stat3
  admin: Stat; // stat4
}

interface PostStats {
  total: number;
  published: Stat; // stat1
  draft: Stat; // stat2
  archived: Stat; // stat3
  totalViews: Stat; // stat4
}

타입 가드 함수도 추가해서 타입 안전성을 강화했습니다. 이제 코드만 봐도 어떤 통계가 무엇을 의미하는지 명확하게 알 수 있습니다.

사용한 기술 스택 경험

shadcn/ui의 설계

"복사-붙여넣기" 방식이 처음에는 이상했는데, 사용해보니 장점이 많았습니다.

  • npm 패키지가 아니라서 버전 관리 걱정 없음
  • 내 프로젝트에 코드가 들어오니 완전히 커스터마이징 가능
  • Radix UI 기반이라 접근성이 기본으로 잘 갖춰져 있음
  • 내부적으로 CVA를 사용해서 variants 패턴을 배우기 좋았습니다

근데 커스터마이징을 어디까지 해야 할지는 여전히 고민입니다. 스타일을 오버라이드하는 방식이 좋을지, 아니면 기본 스타일을 싹 걷어내고 새로 작성하는게 좋을지... 그리고 shadcn/ui에서 제공하는 props나 기능 중에 안 쓰는 것들은 미리 지워두는게 좋을지, 아니면 나중에 쓸 수도 있으니까 원본을 유지하는게 좋을지도 궁금합니다.

CVA (Class Variance Authority)

타입 안전한 variants 패턴을 구현할 수 있다는 점이 가장 좋았습니다.

const pageHeaderVariants = cva("mb-6", {
  variants: {
    size: {
      sm: "[&_h1]:text-xl",
      md: "[&_h1]:text-2xl",
      lg: "[&_h1]:text-3xl",
    },
  },
});

// TypeScript가 자동으로 타입 추론
type PageHeaderProps = VariantProps<typeof pageHeaderVariants>;

근데 CVA를 모든 컴포넌트에 적용하는게 좋을지는 잘 모르겠습니다. PageHeader나 ManagementStats처럼 variants가 필요한 컴포넌트에만 쓰는게 좋을지, 아니면 일관성을 위해 모든 컴포넌트에 적용하는게 좋을지 궁금합니다.

Storybook

컴포넌트를 독립적으로 개발하고 문서화할 수 있어서 좋았습니다.

  • Dark Mode 같은 다양한 상태를 빠르게 확인 가능
  • 페이지를 렌더링하지 않고도 컴포넌트만 테스트 가능
  • autodocs로 자동 문서 생성

ManagementStats, ManagementTabs, AlertContainer 스토리도 추가했는데, 다양한 상태(EmptyStats, LargeNumbers 등)를 쉽게 테스트할 수 있어서 유용했습니다.

아쉬운 점과 개선하고 싶은 부분

1. CVA 적용 범위

PageHeader와 ManagementStats에 CVA를 적용했는데, AlertContainer나 다른 커스텀 컴포넌트에도 적용해볼 수 있었을 것 같습니다. 근데 variants가 필요 없는 컴포넌트에까지 억지로 적용하는 건 오버엔지니어링일 수도 있고...

2. Design Token 확장

현재는 기본적인 토큰만 정의했는데, 실무에서는 더 세밀한 토큰 시스템이 필요할 것 같습니다.

  • 더 다양한 색상 팔레트 (50-900 scale)
  • 타이포그래피 line-height, letter-spacing
  • 애니메이션 duration, easing

3. Storybook Interaction Tests

현재는 Stories만 작성했는데, Interaction Tests를 추가하면 컴포넌트 동작을 자동으로 검증할 수 있을 것 같습니다.

4. shadcn/ui 커스터마이징 가이드

shadcn/ui를 프로젝트에 맞게 커스터마이징하는 명확한 기준을 세우지 못했습니다. 어떤 건 직접 수정하고, 어떤 건 wrapper로 만들고... 일관성이 없는 것 같아서 아쉽습니다.

리뷰받고 싶거나 질문하고 싶은 내용

1. CVA 적용 범위에 대해서

CVA를 모든 컴포넌트에 적용하는게 좋을까요? 지금은 PageHeader와 ManagementStats 같이 variants가 명확히 필요한 컴포넌트에만 적용했는데, 일관성을 위해 모든 커스텀 컴포넌트에 적용하는게 나을까요?

제 생각엔 variants가 2개 이상 필요한 컴포넌트에만 적용하는게 맞는 것 같긴 한데, 실무에서는 어떻게 하시는지 궁금합니다.

2. shadcn/ui 커스터마이징 전략

shadcn/ui 컴포넌트를 수정할 때 두 가지 방식이 가능한데 어떤게 좋을까요?

방법 1: 원본 직접 수정

// components/ui/button.tsx 직접 수정
export const Button = ({ ... }) => {
  // cursor-pointer 추가, 색상 변경 등
}

방법 2: Wrapper 컴포넌트 생성

// components/shared/CustomButton.tsx
import { Button } from "@/components/ui/button"
export const CustomButton = ({ ... }) => {
  return <Button className="cursor-pointer" {...props} />
}

알아본 바로는 shadcn/ui의 철학이 "복사해서 자기 것으로 만들기"라서 직접 수정이 맞는 것 같긴 한데, 나중에 업데이트를 받거나 변경사항을 추적하기 어려울 것 같아서 고민입니다. (미리 안쓰는 기능이나 스타일링을 다 지워둔다면 문제 없을 것 같긴합니다)

3. shadcn/ui 불필요한 기능 제거

shadcn/ui 컴포넌트들을 보면 다양한 props와 기능이 기본으로 제공되는데, 프로젝트에서 안 쓰는 기능들은 미리 지워두는게 좋을까요?

예를 들어 Button 컴포넌트에 asChild prop이 있는데 현재 프로젝트에서 안 쓴다면:

  • 원본 유지: 나중에 쓸 수도 있으니까 그대로 두기
  • 미리 제거: 사용하지 않는 코드는 제거해서 단순하게 유지

원본을 유지한다면, 추후에 어떤 개발에 들어가서든 빠르게 확장성있게 개발한다는 장점이 있을 것 같지만, 당장 불필요한 기능이 들어가있어 유지보수나 가독성에서 아쉬울 것 같습니다!

4. 스타일링 오버라이드 vs 재작성

shadcn/ui 컴포넌트의 기본 스타일을 변경할 때:

방법 1: className으로 오버라이드

<Button className="bg-custom-color hover:bg-custom-color-dark">

방법 2: CVA variants에서 완전히 재정의

// button.tsx의 buttonVariants 자체를 수정
const buttonVariants = cva("...", {
  variants: {
    variant: {
      default: "bg-custom-color hover:bg-custom-color-dark",
    },
  },
});

프로젝트 디자인 시스템이 shadcn/ui 기본 스타일과 많이 다르다면 아예 variants를 재정의하는게 나을까요, 아니면 필요한 부분만 오버라이드하는게 좋을까요?

5. Design Token 범위

이번 과제를 하면서 Design Token으로 spacing 같은 경우에는 sm, md같은걸 미리 정의해두지 않고, space-y-1 같은 방식으로 사용했는데, 이걸 정의한다면 아마도 마진, 패딩, gap 등 모든 곳에서 정의를 해놔야되지 않을까 라고 생각이 드는데 디자인 토큰을 사용하면 숫자로 사용하는걸 지양하는게 좋은지 (통일성), 아니면 문서화를 통해 숫자로 여백을 지정하는 걸 쓰되, 가이드라인을 따라서 쓰도록 하는지 실무에서는 어떤 방식을 더 많이 사용하는지 궁금합니다!
