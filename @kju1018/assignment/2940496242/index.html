기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. medium.useEventOperations.spec.tsx > 아래 toastFn과 mock과 이 fn은 무엇을 해줄까요?

vi.mock은 코드상 'notistack' 라이브러리의 특정 부분을 가짜(mock)로 대체해주는 역할을 합니다. 또한 const enqueueSnackbarFn = vi.fn(); 코드를 통해 enqueueSnackbarFn을 mock 함수로 생성하여 테스트할 수 있게 해줍니다.

아래 코드는 실제 useSnackbar 훅의 enqueueSnackbar 함수를 가짜함수로 대체해주는 역할을 합니다.

useSnackbar: () => ({
  enqueueSnackbar: enqueueSnackbarFn,
}),

Q. medium.integration.spec.tsx > 여기서 ChakraProvider로 묶어주는 동작은 의미있을까요? 있다면 어떤 의미일까요?

Provider의 import 경로가 '@mui/material/styles' 인것으로보아 Provider로 묶어줘야 일관된 스타일로 컴포넌트을 사용할 수 있다고 생각됩니다!

Q. handlersUtils > 아래 여러가지 use 함수는 어떤 역할을 할까요? 어떻게 사용될 수 있을까요?

use 함수는 기존에 handler에 정의된 get, put, post 함수등을 use안에 정의한 내용으로 덮어쓰는 기능이라고 이해했습니다. 해당 기능은 특정 테스트에서는 해당 조건으로 구현된 서버를 사용해서 API 응답을 바꾸고싶을때 사용할 수있습니다.

Q. setupTests.ts > 왜 이 시간을 설정해주는 걸까요?

각 테스트 파일이 실행되기전에 setupTests.ts를 실행하게되는데 beforeEach함수에 시간을 설정해주면 모든 테스트에 원하는 시간으로 설정해서 테스트할 수 있습니다.

과제 셀프회고

이번 주에는 야근이 많아서 테스트 코드를 작성할 때 깊이 있게 고민하지 못했던 것 같습니다. 조금만 더 시간이 있었다면, 좀 더 여유롭게 생각하며 작성할 수 있지 않았을까 하는 아쉬움이 있습니다. 그래도 이번에는 머릿속으로만 생각하던 테스트 코드를 직접 작성해봤다는 점에 의미를 두고 싶고, 앞으로 회사나 개인 프로젝트에서도 적극적으로 도입해보고 싶다는 생각이 들었습니다. 또 과제를 하면서 정말 열심히 하고 실력이 뛰어난 분들이 많다는 것을 느꼈고, 그 모습에 큰 자극을 받았습니다. 지금까지는 공부에 집중하지 못했지만, 이 경험을 계기로 지금부터라도 꾸준히 배우고 성장해야겠다는 다짐을 하게 되었습니다.

기술적 성장

  • 테스트 코드를 이렇게 집중적으로 작성해본 건 이번이 처음이었습니다. 초반에는 단순히 함수를 통과시키는 데에만 집중하며 테스트를 작성했지만, 이 방식에는 한계가 있다는 걸 느꼈습니다. 특히 커스텀 훅이나 복잡한 기능을 테스트하려면 우선 본 기능의 동작을 충분히 이해한 뒤 테스트를 작성해야 한다는 점을 배울 수 있었습니다. 아직은 쉽지 않지만, API 통신처럼 비동기 로직이 포함된 부분은 더욱 주의 깊게 테스트해야 한다는 것도 느꼈습니다.그래도 이번 경험을 통해 간단한 테스트의 경우에는 어떤 방식으로 접근하면 좋을지 감이 생길 정도로는 성장했다고 생각합니다.
  • handler나 handlersUtils를 직접 작성해보지는 못했지만, 미리 구현되어 있던 코드들과 7기 분들이 디스코드에 공유한 글들을 참고하면서 MSW가 어떤 역할을 하는지, 그리고 use 함수가 언제 활용되면 좋은지에 대해 이해할 수 있었습니다.

코드 품질

코드품질은 사실 테스트를 작성하는데 급급해서 코드품질은 일관성도 없고 매우 떨어진다고 생각합니다..

학습 효과 분석

초반에는 테스트를 단순히 통과시키기 위한 코드만 작성했던 것 같아요. 사실 그때는 “이게 정말 큰 도움이 될까?” 하는 생각도 많이 들었습니다. 그런데 테스트 코드를 작성하면서, 이게 단순한 확인용 코드가 아니라 하나의 명세서 역할도 할 수 있다는 점을 알게 되었습니다. 또 테스트가 미리 작성되어 있으면, UI 요소나 함수를 수정할 때 사람이 놓치기 쉬운 실수를 미리 잡아줄 수 있다는 장점도 느꼈습니다. 이번 경험을 통해 테스트 코드의 필요성과 역할을 확실히 배울 수 있었고, 앞으로는 기능을 구현하기 전에 테스트를 먼저 생각하는 습관을 가져야겠다는 생각이 들었습니다.
