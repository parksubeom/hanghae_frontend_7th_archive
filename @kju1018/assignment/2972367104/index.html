과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요. 시각적 회귀 테스트 링크입니다
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리

당장은 적절한 스토리를 작성하는데 더 집중해주세요.
적절한 추상화, 응집도를 높이고 결합도를 낮추는
우아한 설계는 이후 주차때 좀 더 집중해봐요.

- 물론, AI를 활용해도 좋습니다.
- 기존에 작성한 테스트는 깨지지 않도록 해주세요.
  - 이 과정에서 테스트가 분리될 수도 작성법이 수정될 수도 있습니다.
  - 정말 불필요한 케이스가 아니면, 테스트 케이스를 삭제하는건 최소화 해주세요.
  - 추가로 필요한 테스트가 생길 수 있지만, 작성은 자유롭게 선택해서 진행해주세요.

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

팀원들과 회의 한 노션 링크입니다: 1팀 테스트전략 회의

  1. 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유
E2E 테스트는 드래그 앤 드롭(D&D), 일정 추가 등 실제 브라우저 환경에서 모킹(mocking) 없이 테스트를 수행하여 가장 높은 신뢰도를 확보할 수 있기 때문에 1순위로 선정하였습니다.

2순위는 통합 테스트로,
Hooks와 API 통신이 결합된 부분, 컴포넌트 간 상호작용이 필요한 영역, 그리고 날짜 클릭 시 날짜가 바인딩되는 동작처럼 여러 요소가 함께 작동하는 구간에 적용하였습니다.
이 단계에서는 fetch API와의 통합 검증이 필요하므로 다음과 같은 특징이 있습니다.

MSW를 활용하여 실제와 유사한 API 응답을 시뮬레이션합니다.

서버 통신, 상태 업데이트, 알림 노출 등 복합적인 동작 흐름을 검증합니다.

실제 사용자 시나리오를 반영하여 여러 컴포넌트 간의 상호작용을 테스트합니다.

이러한 이유로 통합 테스트를 2순위로 설정하였습니다.

마지막으로, **3순위는 커스텀 훅(일부)과 Utils 함수에 대한 단위 테스트(Unit Test)**입니다.
사실 단위테스트는 이미 해당주차 프로젝트를 시작할때 이미 많이 작성을 했고, 많이 작성이 되어있어서 3순위로 생각했었습니다.
  1. 팀원들이 합의한 최적의 테스트
저희 팀원들은 테스트 전략의 최우선 순위를 E2E 테스트에 두기로 합의하였습니다.
특히 드래그 앤 드롭(Drag & Drop) 기능은 실제 사용자 환경과 가장 유사한 브라우저 상호작용이 핵심이기 때문에, 반드시 E2E 테스트로 검증하기로 결정했습니다.

이미 통합 테스트와 단위 테스트는 충분히 구축되어 있는 상태이므로, 새로운 단위 테스트를 추가하는 일의 우선순위는 비교적 낮게 설정하였습니다.
대신 사용자 관점에서 실제 시나리오를 검증하는 테스트를 중심으로,
전체적인 서비스 흐름이 기대한 대로 동작하는지 확인하는 데 집중하기로 하였습니다.
  1. 제가 추가한 테스트
  1. D&D(드래그 앤 드롭) - e2e테스트(event-drag-and-drop.spec.ts)
  2. 날짜 클릭 시 왼쪽 폼에 날짜 반영 - e2e테스트, 통합테스트(date-binding-on-empty-cell.e2e.spec.ts, dateBindingWorkflow.spec.tsx)
  3. dragUtils - 단위테스트(dragUtils.spec.ts)
  4. useEventForm - 커스텀 훅 단위테스트(useEventForm.spec.ts)

과제 셀프회고

처음 과제 내용을 들었을 때는 조금 걱정 됐었습니다. Storybook, Chromatic, Playwright 모두 제대로 사용해본 적이 없거나, 한 번도 다뤄본 적이 없었기 때문입니다. 그래서 과제를 시작하기 전에 우선 세 도구의 개념과 사용 목적을 먼저 공부하고, 바로 적용하기보다는 예제를 통해 동작 방식을 이해한 뒤 과제를 진행하였습니다.

또한 기능 구현을 할 때 AI를 활용할지, 직접 구현할지에 대해서도 많은 고민을 했습니다. AI를 사용하면 확실히 구현 속도는 빨랐지만, 예상치 못한 곳에서 오류가 발생하거나 기존에 정상적으로 동작하던 부분이 달라지는 경우가 있어서 여러 번 롤백을 해야 하는 상황이 있었습니다. 테스트가 갑자기 전부 실패하기도 했습니다. 아직 제가 AI 도구 사용에 익숙하지 않다는 것을 느꼈고, 이번 과제에서는 직접 구현을 우선으로 하되, 실패하거나 막히는 부분만 AI의 도움을 받는 방식으로 접근했습니다. 지금은 이 방식이 저에게 가장 잘 맞는 방법이라고 생각합니다. 그 덕분에 과제를 무사히 마무리할 수 있었습니다.

기술적 성장

E2E 테스트를 진행하면서 “사용자의 행동 흐름을 검증한다”는 관점을 배울 수 있었습니다. 또한 시각적 회귀 테스트를 통해 제공되는 데이터를 기반으로 컴포넌트의 UI 변화가 어떤 영향을 미치는지 직접 확인할 수 있었고, 이를 통해 UI 변화를 보다 직관적으로 파악하는 경험을 얻을 수 있었습니다. 그리고 E2E 테스트 과제를 진행하기 전에는 솔직히 통합 테스트와 E2E 테스트가 어떻게 다른지 잘 이해하지 못했지만, 직접 테스트를 작성하고 실행해보면서 아직 완벽히 구분되지는 않더라도 두 테스트의 차이를 어느 정도 체감할 수 있었습니다.

저는 'dnd-kit'을 활용해 드래그 앤 드롭을 구현했는데.. 조금 민망하지만 HTML에도 Drag and Drop API가 내장되어 있다는 사실을 처음 알게 되었습니다..

과제 피드백

리뷰 받고 싶은 내용

  1. 이번에 E2E 테스트를 진행하면서, 테스트를 실행할 때 이 프로젝트에서 임시 DB로 사용하는 JSON 파일의 데이터가 실제로 변경된다는 점을 확인했습니다. 그 결과, 첫 번째 테스트에서 생성한 일정 데이터가 이후 테스트 문에도 영향을 미치는 상황이 발생했는데요. 실무에서는 이렇게 이전 테스트에서 생성된 데이터를 기반으로 다음 테스트가 이어지는 방식으로 E2E 테스트를 진행하는지 궁금합니다.

  2. 위의 방식과는 반대로, 저는 첫 번째 테스트의 결과가 두 번째 테스트에 영향을 미치게 되어, 뒤에 있는 테스트 코드들까지 연쇄적으로 실패할 가능성이 있다고 생각했습니다. 그래서 각 테스트는 서로 독립적으로 실행되어야 한다고 판단했고, 이를 위해 e2e.json 파일과 seed 역할을 하는 e2e-template.json 파일을 별도로 만들어 관리했습니다. beforeEach 문에서 테스트를 시작하기 전에 e2e-template.json의 내용을 e2e.json에 덮어씌우도록 설정하여, 모든 테스트가 항상 같은 초기 상태의 e2e.json 파일을 참조하도록 구성했습니다. 이 방식이 제가 생각한 가장 안정적인 방법이었는데, 실무에서는 이러한 데이터 초기화를 어떤 방식으로 처리하는지 궁금합니다.

A1) 보통 테스트 시작전에 json을 항상 초기화 해주는 방식으로 진행한답니다! 그래서 테스트 간의 사이드 이펙트를 줄이는 방식으로 관리하는 편이에요 ㅎㅎ 아니면 의도적으로 테스트 데이터를 보존한 다음에 순차적으로 실행하는 방식도 생각해볼 수 있어요. 중요한건 "의도"를 했는지 안 했는지의 차이겠네요.

그리고 검증하고자 하는 구간을 테스트 스펙으로 구분하기보단 아예 하나의 테스트에서 주석이나 혹은 다른 장치를 토대로 구분할 수도 있을 것 같아요!
