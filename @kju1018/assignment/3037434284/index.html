과제 목표

레거시 코드베이스를 현대적인 디자인 시스템으로 개편하는 실무 경험

  1. 정리되지 않은 레거시 코드의 문제점 식별 및 분석
  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

Before 패키지 분석 후 After 패키지 개편

개편 목표

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

3. 컴포넌트 개편

  • UI와 비즈니스 로직 분리

과제 회고

과제를 진행하면서 느낀 점, 배운 점을 자유롭게 작성해주세요.

Before 패키지에서 발견한 문제점

  1. UI 컴포넌트가 도메인/비즈니스 로직을 직접 가짐
  • entityType, action, entity 같은 props로 user/post 도메인을 직접 알고 있음.
  • “관리자는 삭제 못 함”, “이미 게시된 글은 publish 비활성화” 같은 비즈니스 규칙을 버튼 안에서 판단.
  • 버튼 레이블도 action === 'create'면 “사용자/게시글 만들기”처럼 버튼이 스스로 결정함.
  • fieldType, entityType, checkBusinessRules 등의 prop으로 username 길이/문자 검증
  1. 한 컴포넌트에 역할이 너무 많음
  • Table.tsx 한 파일 안에 검색(search), 정렬(sort), 페이지네이션(pagination)등 을 모두 처리함.
  1. 디자인 시스템/일관성 부족
  • before 쪽 컴포넌트들은 각자 독자적인 CSS 클래스(btn-, alert-, modal-, table- 등)를 사용하고 Variant 시스템이나 공통 토큰(색상, 간격 등)이 정리되어 있지 않음.
  • Alert/Modal/Card/Table 등이 각자 다른 스타일/패턴을 쓰기 때문에: 테마 변경, 스타일 가이드 변경에 취약하고 컴포넌트 재사용 시 UI 일관성을 유지하기 어려움.

개편 과정에서 집중한 부분

UI와 비즈니스 로직 분리

  • before에서는 Button, FormInput, Table이 entityType(user, post)랑 규칙(관리자 삭제 불가, 금칙어 등)을 전부 내부에서 처리했는데, after에서는 폼 검증/도메인 규칙은 바깥으로 빼고, 컴포넌트는 “표현(렌더링)”과 “상태 제어”에만 집중하게 만들었습니ㄷ다.

  • 기존 Table.tsx는 검색, 정렬, 페이지네이션, 렌더링, 도메인별 액션까지 한 컴포넌트에 모두 들어있었는데, 이를 DataTable.tsx, hook 등으로 최대한 분리시켰습니다.

재사용 가능한 ui 정리

  • ui/button, ui/card, ui/alert, ui/table, form/input, form/select, Modal, Alert, DataTable 등으로 역할별 컴포넌트 레이어를 정리하고, cva variant, 공통 스타일, FormInput, FormSelect, DataTable 등으로 구조를 만들었습니다.

사용한 기술 스택 경험

tailwind - 예전에 Bootstrap만 사용해보고 tailwind는 처음 사용해보는데 Bootstrap을 사용했던것처럼 비슷하게 사용했습니다.

shadcn - shadcn/ui는 처음 들었을 때는 MUI처럼 설치해서 쓰는 라이브러리라고 생각했는데, 막상 사용해보니 컴포넌트 코드를 프로젝트로 직접 가져와서 사용하는 방식이라 신기했습니다. 라이브러리 의존성이 아니라 실제 코드가 내 프로젝트 안에 들어와서 커스터마이징이 자유롭다는 점이 인상적이었습니다.

cva – 기존에는 스타일이나 클래스를 다르게 적용할 때 컴포넌트 안에서 분기 처리(if/조건문)로 클래스를 변경해 줬습니다. cva를 사용하면서 이런 분기 로직을 한 곳에 모아 정의할 수 있게 되었고, variant를 기준으로 클래스를 관리하기 훨씬 편해졌습니다.

React Hook Form + Zod - 사실 이런게 있는줄은 처음 알았고, 사용하는법이 익숙치 않아서 AI에 도움을 많이 받았습니다.

어려웠던 점과 해결 방법

  1. Atomic 디자인 폴더 구조를 어떻게 바꿔야 할지 고민이 많았습니다.
  • 그래서 우선은 역할/관심사 기반으로 폴더 구조를 재구성했는데, 이게 충분히 적절하게 나뉘었는지는 아직 확신이 없습니다.
  1. UI와 비즈니스 로직 분리 기준 잡기
  • 비즈니스 로직은 ManagementPage에서 처리하고, 검증과 데이터 가공 로직은 hook과 Zod 스키마로 분리해 UI 컴포넌트와 역할을 나누었습니다.

리뷰받고 싶거나 질문하고 싶은 내용

  1. 기존 before폴더구조를 제 나름대로 역할/관심사 기반으로 다시 수정을 했는데 코치님은 어떤식으로 폴더구조를 잡았을지 궁금합니다.
