기본과제

Medium

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. medium.useEventOperations.spec.tsx > 아래 toastFn과 mock과 이 fn은 무엇을 해줄까요?

Q. medium.integration.spec.tsx > 여기서 ChakraProvider로 묶어주는 동작은 의미있을까요? 있다면 어떤 의미일까요?

  • DOM 구조와 테스트 환경에서 동일한 구조로 세팅되어 작성된 코드에 맞게 테스트 시에 동일한 element를 찾기 위해 provider로 묶어줘야합니다.

Q. handlersUtils > 아래 여러가지 use 함수는 어떤 역할을 할까요? 어떻게 사용될 수 있을까요?

  • 테스트 환경에서 로직이 제대로 실행될 수 있도록 도와주는 역할을 한다고 봅니다. 코드에 바인딩된 함수들이 실행될 때 API 요청 등 유저 액션을 통해 실행되야하는 로직들이 있는데 이 테스트가 진행될 수 있도록 환경을 구성해두는데 쓰였다고 생각합니다.

Q. setupTests.ts > 왜 이 시간을 설정해주는 걸까요?

  • 테스트 환경에서 원하는 시점에 테스트가 진행될 수 있도록 가상 시간을 설정해주어야하기 때문

심화 과제

  • App 컴포넌트 적절한 단위의 컴포넌트, 훅, 유틸 함수로 분리했는가?
  • 해당 모듈들에 대한 적절한 테스트를 2개 이상 작성했는가?

과제 셀프회고

  • 과제 양이 얼마 안될 줄 알았는데 너무 많아서 1차 당황 ( + 이 여파로 급하게 과제하느라 자세하게 코드를 살펴볼 여력이 없었음 .. 매우 아쉽 ..)
  • 쉽지않는 난이도에 2차 당황
  • 매일 새벽까지 과제하는 팀원들 보고 1차 각성 ..
  • 케이스 하나에도 여러 의문을 갖는 팀원들 보고 2차 각성 .. (찐회고)
  • 테스트 코드가 처음이다보니 단순 unit테스트에서 input, output을 예상하는 과정부터 쉽지않았다. 항상 비어있는 input에 output을 예상하고 코드를 짜기만 했을 뿐, 예상값과 결과를 예측해서 테스트를 한다는 자체가 낯설었다. 그러다 보니 테스트 조건을 이상하게 해석하고, 괜히 불필요하게 값을 변수에 할당해서 테스트 조건 외에 상황에도 모두 대응하려고 애쓰는 ,, 이상한 포인트에서 시간을 낭비하기도 했다.

기술적 성장

  • 테스트 라이브러리 찍먹 (queries, matcher 를 조합해서 테스트하는 방법)
