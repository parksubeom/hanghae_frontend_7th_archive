[6팀 김현우] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

과제 체크포인트

기본 과제

Easy

  • 총 8개의 파일, 90개의 단위 테스트를 무사히 작성하고 통과시킨다.

심화 과제

  • App 컴포넌트 적절한 단위의 컴포넌트, 훅, 유틸 함수로 분리했는가?
  • 해당 모듈들에 대한 적절한 테스트를 2개 이상 작성했는가?

과제 셀프회고

테스트 코드를 직접 코드로 구현해보는 건 이번이 처음이여서 그런지, 기본과제를 하는데도 여러 고민들이 들었고, 그로 인해 과제 속도도 나지 않았다. medium, hard레벨까지 풀지 못해 아쉽기도 하지만, 지금까지 풀은 과제를 통해서 테스트 코드에 대한 여러 내용들을 학습한 것 같다.

기술적 성장

  • 테스트 코드 작성 과제를 진행하면서 기초적인 테스트 코드의 환경 구성(setupTests.ts, beforeAll, beforEach, afterEach, afterAll), 작성 내용(환경 구성, 쿼링, 실제 검증 프로세스 순), 기본적인 RTL API에 대해 학습하였다.
  • 한 테스트 케이스에 긍정, 부정 검증(assertion)을 섞으면 의도가 흐려질 수 있기에 별도의 케이스로 빼는게 더 낫다
  • useCalendarView.spect.ts는 훅 테스트이기에, 컴포넌트에 무엇이 렌더링 된다는 테스트는 할 필요 없음.
  • useCalendarView.spect.ts 의 삼일절 확인 문제
    • 처음엔 fetchHolidays 함수를 왜 테스트에서 호출해야 하는지 의아스러웠다. useCalendarView 훅 내부에서 currentDate가 변경되면 holidays를 알아서 업데이트 해줄 것이고, 나는 그 결과 내부에 삼일절이 있는지만 확인하면 되는 것이라 생각했었다.
    • 그런데 fetchHolidays를 테스트에서 호출하지 않는 상황에서 holidays가 정상적으로 업데이트 되지 않는 오류가 발생했다면 이것이 useEffect의 문제인지, useState 문제인지 혹은, fetchHolidays 함수의 문제인지를 일일히 찾아야 하는 상황을 마주하게 된다.
    • 결론적으로 해당 함수를 호출하여 정상적인 기능을 제공하는지를 테스트하게 된다면 에러 파악이 더 용이해지게 되고 해당 훅에 대한 더 자세한 명세 제공과 기능의 정상작동을 확인할 수 있는 친절한 테스트 코드가 되지 않을까 싶다.

코드 품질

학습 효과 분석

  • 테스트 코드를 작성하면서 어떤 것들을 테스트 해야하는지 학습할 수 있었다. 이를 통해 코드의 기본적인 기능과 코드의 명세서 역할을 한다는 것이 무엇인지 체감하였다. 막연한 주석보다 명확한 의도가 드러난 테스트 코드를 작성하는 것이 코드를 이해하는데 도움이 될 것 같다.

과제 피드백

수고 많았어요 현우님! 이번 과제는 테스트를 하는 다양한 방법들을 접하면서 테스트 코드를 작성한다는 그 심리적 장벽을 허물고 친숙해지는데 있었습니다.

"...테스트 코드를 직접 코드로 구현해보는 건 이번이 처음이여서 그런지, 기본과제를 하는데도 여러 고민들이 들었고, 그로 인해 과제 속도도 나지 않았다. medium, hard레벨까지 풀지 못해 아쉽기도 하지만, 지금까지 풀은 과제를 통해서 테스트 코드에 대한 여러 내용들을 학습한 것 같다." 라고 했는데 우선 테스트 코드라는 것에 익숙해지기만 해도 충분히 가치가 있어요. (그렇다고 찍먹만 하는건 아니고 꼭 다 마스터 해야겠죠?)

특히나 "책으로 테스트 코드에 대해 익혀 두었으니 hard까지 시간 쏟으면 풀 수 있지 않을까" → "생각보다 생소하고 공부해야 할 것들이 많다" 처럼 책으로 아는 것과 직접 코드로 부딪혀 보는 건 완전히 다른 경험이에요. 저희가 발제는 세상에 이런게 있다라고 시야를 틔여주는거라면 실습은 직접 부딫히면서 이런 것들이 필요하고 중요하고 알게 해주는 시간이랍니다.

fetchHolidays 함수에 대한 고민과 접근 아주 좋아요. 테스트는 단순히 결과만 확인하는 게 아니라 "문제가 생겼을 때 어디서 잘못됐는지 빠르게 파악"할 수 있게 하는 도구예요. 각 함수를 개별적으로 테스트하면 에러의 원인을 정확히 특정할 수 있죠. 이런 고민을 스스로 해내신 게 정말 멋집니다. 과제를 비록 다 하지 못하는 순간이 오더라고 지금처럼 회고에서 고민의 깊이는 계속 가져주길 바래요!

리뷰 받고 싶으신 내용인 description 작성에 대해 답변드리자면, 테스트 description은 "이 테스트가 무엇을 검증하는가"를 누가 봐도 명확히 알 수 있게 작성하는 게 목적입니다. "getDaysInMonth는 2024년 2월에 29일을 반환한다"처럼 구체적이고, 실행 가능한 문장으로 작성하면 좋아요. 지금 작성하신 description들 보면 이미 잘하고 계실 것 같은데 구체적으로 어떤 부분이 고민되셨는지 알려주시면 더 자세히 피드백 드릴 수 있을 것 같아요. 질문이 피상적이면 저도 답변히 같은 수준에서 밖에 답을 못드리겠네요.

다음번 회고는 보다 더 깊은 고민과 구체적인 질문으로 가득차기를 기대합니다. 다음 과제도 화이팅입니다!
