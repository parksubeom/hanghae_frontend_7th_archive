[6팀 김현우] Chapter 1-3. 완성도있는 테스트 전략 수립하기

by lecto172025.11.03

과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리

당장은 적절한 스토리를 작성하는데 더 집중해주세요.
적절한 추상화, 응집도를 높이고 결합도를 낮추는
우아한 설계는 이후 주차때 좀 더 집중해봐요.

- 물론, AI를 활용해도 좋습니다.
- 기존에 작성한 테스트는 깨지지 않도록 해주세요.
  - 이 과정에서 테스트가 분리될 수도 작성법이 수정될 수도 있습니다.
  - 정말 불필요한 케이스가 아니면, 테스트 케이스를 삭제하는건 최소화 해주세요.
  - 추가로 필요한 테스트가 생길 수 있지만, 작성은 자유롭게 선택해서 진행해주세요.

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

    • E2E 테스트

      • 절대 깨져서는 안되는 기능에 대해서는 E2E 테스트로 다양한 환경을 가정해서 기능의 안정성을 높여가면 좋을 것 같습니다.
      • 예) 열악한 network 환경에서의 로그인 / 결제 작업
      • 예) 크로스 브라우저나 다양한 기기에서 layout 깨지지 않는지 확인 

    • 통합 테스트

      • e2e 대비 빠른 구동 환경, 낮은 작성 난이도인 통합 테스트를 많이 활용해서 서비스 품질을 높일 수 있기에 통합 테스트를 많이 작성하면 좋을 것 같습니다.
      • e2e 대비 실패 원인 추적도 명확하며, 여러 모듈을 일괄적으로 검증할 수 있기에 통합 테스트를 작성하게 되면 다른 테스트 대비 더 큰 효과를 얻을 수 있지 않을까 싶습니다.

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

    • E2E
      • 사람이 직접 테스트하는 것이 제일 좋은 방법이지만, 그럴만한 환경이 안되니 그와 제일 근접한 e2e 테스트로 검증하는 것이 좋은 방법. 그런데 e2e는 테스트 하기 까지 셋팅할 환경이 너무 크고, 여러 변수들로 인해 테스트가 깨질 수 있는 취약점이 있다. 그렇기에 서비스의 핵심 기능, 깨져서는 안되는 기능, UI에 대해서는 E2E를 진행하자고 논의하였습니다.
      • 정상 케이스를 중심으로 작성하되 무궁무진한 예외 케이스는 통합 테스트로 관리하는 것이 좋다고 논의하였습니다.
    • 통합 테스트
      • E2E 대비 빠른 구동 환경에서 진행되고, 여러 모듈들을 통합해서 테스트 하기에 기능의 안정성을 확보할 수 있기에 제일 많이 가져가면 좋을 테스트.

  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

    • drag-and-drop.spec.ts - 브라우저 환경에서 제공되는 API(drag start, end)를 검증하는 테스트. 달력 형식의 일정 관리 어플리케이션에서 dnd로 일정을 수정하는 것은 핵심 기능 중 하나라고 판단하여 E2E 테스트를 진행하였습니다.
    • date-click.spec.ts - 달력 형식의 일정 관리 어플리케이션에서 일정을 추가할 때 달력의 날짜를 클릭하여 새로운 일정을 추가할 수 있는 기능은 필요한 기능이라 판단하여 해당 기능은 E2E 테스트를 진행하였습니다.

과제 셀프회고

기술적 성장

  • storybook으로 컴포넌트를 테스트하는 것과 chromatic에서 story로 만든 컴포넌트를 테스트하는 것의 차이
    • Storybook은 컴포넌트를 보는 곳
    • Chromatic은 그 보이는 게 ‘달라졌는지 테스트하고 승인’하는 곳
  • 시각적 회귀 테스트를 진행할 떄, 각각의 story 단위로 진행하는 이유
    • 변경 감지 정확도 향상
      • Chromatic은 스토리 단위 로 스냅샷을 찍어요. 즉, 한 스토리 = 한 시각 상태로 비교됩니다.
      • 여러 상태를 한 스토리에 몰아 넣으면, 하나만 바뀌어도 전체 diff로 인식돼서 실제 변경과 상관없는 영역까지 잡힐 수 있습니다.
    • 디버깅 & 승인 속도 향상
      • 특정 UI만 바뀐 경우, 해당 스토리만 diff로 표시돼 리뷰어가 빠르게 이 변경이 의도된 건지 판단할 수 있습니다.
    • 테스트 독립성 보장
      • “타입별 캘린더 뷰 렌더링”, “다이얼로그 상태”처럼 독립된 케이스는 각각 렌더링 환경(상태, props, theme)이 다르므로 독립 스토리로 만들어야 회귀 테스트가 의미를 가집니다.
    • PR 코멘트 및 승인 단위 명확
      • Chromatic이 PR에 “변경된 스토리 목록”을 자동 코멘트로 달 때, 각 항목이 개별적으로 표시돼 협업 효율이 높습니다.
  • 테스트 코드를 작성할 때 명확한 의도가 담긴 테스트 코드 작성을 해야하는데, AI로 테스트 코드를 작성하는 경우, 불필요하거나 다른 테스트 코드로 커버될 수 있는 케이스들을 여럿 작성해주었다. 테스트 코드 작성하는 중간 과정은 도움을 받되, 전체적인 부분은 사람이 체크하면서 바른 가이드를 안내해야하는 것 같다. AI에게 모든 작업의 주도권을 맡기는 것이 아닌, 명확한 방향 아래에서 AI를 활용하게 빠르게 작업을 완성해가는 선상에서 AI를 사용해야함을 다시금 생각해보게 됬다.
  • e2e 테스트 진행 시, 이전의 테스트 실행 결과가 이후 실행하는 테스트에 영향을 미치는 경우가 있었다. 이럴 떄마다 db로 사용되고 있는 e2e.json을 직접 초기화 해주는 작업을 해주었는데, 이를 다른 방식(setup/tear-down 등)으로 대체할 수 있다는 걸 알았다. setup 방식으로 e2e.json을 초기화해주었는데, 다른 개발자들은 어떤 방식을 활용하는지, 어떤 패턴들이 있는지, 각각의 UC, 장, 단점은 무엇인지에 대해 학습하였다.
  • E2E 테스트가 비용적으로 비싼 작업이기에 병렬적으로 실행하는 것이 좋다는 생각이 들었다. 이미 playwright 측에서도 기본 환경 설정으로 병렬로 돌아가게 설정해 놓은 것을 볼 수 있었다(fullyParallel: true, workers: process.env.CI ? 1 : undefined). 그런데 문득 workers의 갯수를 CI 환경에서는 1개로 지정하여 병렬적으로 돌아가지 않도록 지정한 것이 좀 의외라고 생각이 들었다. 로컬에서는 빠르게 테스트를 실행하도록 하되, CI 환경에서는 순차적으로 테스트하여 속도보다 안정성을 도모하는 것이 나은 선택이라고 하는 것 같다.
  • 마주한 E2E Error

    • Playwright의 strict mode violation 에러가 발생하였다. getByText('E2E 테스트 일정') 함수가 페이지에서 2개의 요소를 찾았기 때문.

      • month-view 안의 캘린더 셀에 있는 일정 제목
      • event-list 안의 일정 목록에 있는 일정 제목

      -> Playwright는 기본적으로 strict mode로 동작하여, 하나의 locator가 여러 요소를 매칭하면 에러를 발생시킨다.

    • 개별 테스트 케이스를 검증했을 때는 문제 없이 실행되는데, describe 블록 단위로 테스트 케이스를 검증했을 때는 불특정 테스트가 깨지는 상황

궁금증

  • 테스트 코드를 작성하는데 방식이 처음부터 끝까지 일일히 지정해주어야 해서 명령형에 가깝다고 느껴졌는데, 선언형 스타일에 가깝게 테스트 코드를 작성할 수 있을지 궁금합니다. 이렇게 되면 테스트 코드도 더 가독성이 올라갈 수 있을 것 같다 생각했는데 이러한 관점이 테스트 코드 작성하는데 있어선 불필요한 관점일까요? 애초에 테스트 코드라는게 구체적인 환경에서 일련의 시나리오 대로 수행되어야 하는 것이기에 명령형 스타일로 작성되는 것이 일반적인 건가 싶어 질문 드립니다.

요런느낌?

describe("areAnagrams()", () => {
  it.each([
    ["cat", "bike", false],
    ["car", "arc", true],
    ["cat", "dog", false],
    ["cat", "act", true],
  ])("areAnagrams(%s, %s) returns %s", (first, second, expected) => {
    expect(areAnagrams(first, second)).toBe(expected);
  });
});

결국 "선언형" 이라는건 무언가를 추상화해서 사용하는거라고 생각해요. 테스트의 동작을 추상화해서 "선언"하기만 해도 동작하도록 만드는거죠 ㅎㅎ 그렇다면 테스트를 위한 함수를 많이 만들어놔야 하는데, 그렇게 할 필요가... 대체로 없는 편이죠.
