Before 패키지에서 발견한 문제점

스타일링
  • 인라인 형식과 css module 방식을 혼용하여 스타일이 작성되어, 특정한 엘리먼트의 스타일을 수정, 확인하기 위해서는 해당 엘리먼트의 스타일이 적혀 있는 곳이 css module 인지 혹은 인라인으로 적용하고 있는지 확인해야한다.
  • 엘리먼트들 마다 스타일이 하드 코딩 형식으로 작성되어 있어 버튼 색상, padding 간격, border 넓이, 색상 등 디자인 규칙이 변경될 경우 하나 하나 대응해야한다.
  • semantic 태그를 사용하지 않아 UI 내 어떤 영역에 해당하는 부분인지 바로 유추하기가 어렵다.
  • 하나의 css 파일(components.css)에 모든 컴포넌트들에 대한 스타일들(eg, components.css 내 button, form, card, badge 등)과 심지어 글로벌하게 적용하는 스타일(eg, * { margin:0; padding:0; })도 포함되어 있어 관심사가 분리되지 않아 유지보수하기에 매우 어려운 상황.
컴포넌트 설계
  • 하나의 컴포넌트 내에서 여러 책임과 관심사가 뭉쳐져 있는 상황. 이로 인해
    • 개발자가 코드를 읽고 유지 보수하기가 어렵다. 하나의 컴포넌트 내에 user에 대한 코드와 post에 대한 코드가 합쳐져 있기에.
    • 가독성 저하
    • 다른 관심사의 state가 변화할 때 마다 ManagementPage 컴포넌트 중복 렌더링이 발생하고 이로 인한 성능 이슈
  • 컴포넌트명과 props명은 다른 개발자들이 선언적으로 이해할 수 있는 인터페이스인데 일관되지 않은 props naming(eg, helpText, hint) 으로 인지 부담을 늘린다.
폼 관리
  • 폼에 대한 검증 로직이 UI 컴포넌트 내에 같이 작성되어 있어 관심사와 책임이 한 곳에 몰려있는 상황.
  • 실시간 검증이 필요한 폼이 아닌데, form이 바뀔 때마다 검증하도록 되어 있어 리렌더링과 validation에 많은 비용 발생

개편 과정에서 집중한 부분

과제를 할 때 가정해본 상황

  1. 리팩토링하는 업무를 받음
    • 간격, 폰트, 색상, border, 그림자 등에 대한 토큰화 작업과 이를 활용한 UI를 출력해야한다. 하지만 리팩토링 된 프로젝트의 UI도 레거시와 동일한 UI를 출력해야한다. 급격한 UI 변화는 사용자에게 거부감을 줄 수 있으니, 디자인 시스템을 구축해서 시스템을 운영하도록 한다.
    • 동일한 UI 상태를 보장하기 위한 테스트들이 있다는 것이 리팩토링 하는데에 큰 안정감을 준다는 말을 다시금 느끼게 되었다.
  2. 개편을 설득해야하는 상황이라는 가정
    • 레거시 코드를 리팩토링 해야하는 이유를 설득해야하는 상황을 가정. 다른 사람들을 납득시킬 근거들을 찾아 리팩토링 전, 후로 체감할 수 있는 성능 지표를 가져가는 것이 가장 설득력 있을 것 같다. 그리고 최소 수정 대비 최대 아웃풋을 낼 수 있는 부분, 가장 시급하게 리팩토링해야할 부분이 무엇일지 고민했다.
  • UI/비즈니스 로직 분리
    • 기존 PostManagement 안에 섞여 있던 Post/User에 대한 관심사를 각각의 컴포넌트로 분리하고 UI 컴포넌트는 props만 받아 화면을 그리도록 정리
  • 폼 구조 정리
    • uncontrolled + 매 입력마다 검증 → shdcn form(controll 기반) + Zod 기반 onSubmit 중심 구조로 변경
    • 불필요한 리렌더링을 줄이고, 검증 로직을 한 곳에서 관리할 수 있도록 했다.

사용한 기술 스택 경험

디자인 시스템

  • 디자인 시스템을 구현하기 위해 색상(color), 타이포 그래픽(font size, weight), 스페이싱(margin/padding), 그림자, breakpoints와 같은 요소에 대해 디자인 토큰화 작업을 진행했다.
  • 이 과정을 거치면서 디자이너들이 디자인 토큰화를 왜 그렇게 했는지에 대해 조금이나마 생각해볼 수 있는 시간이였다. 가령 타이포 그래피에 관한 토큰을 작성하는 과정에서 size, weight, line-height, font-family 같은 것들을 한 번에 녹여 토큰을 만들지 묻는 AI를 보며 위의 종류들을 하나의 토큰에 여러 primitive한 값들을 합쳐 놓는 것이 일반적인 작업임을 느끼게 되었다.

어려웠던 점과 해결 방법

  • before 레포에서 사용된 색상, 폰트, spacing, border, 그림자 등을 기반으로 토큰화 작업을 진행하고 이것을 after에 적용하는 과정 중 토큰화 되지 않은 상수값들은 여전히 하드코딩으로 작성되고 있는 상황을 발견했다. 이런 상황을 보고 '어떻게 토큰화가 안된 값이 있을 수 있을까?' 고민하다보니 before 레포에서 뽑은 일관된 디자인 시스템과 맞지 않는 value들이 이미 많이 사용되고 있기 때문에 그럴 수 있겠다 싶었다. 잘 갖춰진 디자인 시스템이 있다면 많은 디자이너들이 작업하더라도 규격화된 디자인을 만들어 낼 수 있다는 것, 디자이너와 개발자간 싱크가 맞추지 못한 채 화면 작업을 하는 것 등의 문제를 해결할 수 있겠다는 생각이 들었다.

안녕하세요 현우님! 6주차 과제 잘 진행해주셨네요 ㅎㅎ 고생하셨습니다!! 다만 스토리북 작성이 안 되어있네요 ㅠㅠ 심화과제를 불합격으로 남겨놓도록 하겠습니다.

Q) 사실 리팩토링되는 코드도 처음엔 다른 개발자가 심혈(?)을 기울여 작성했던 코드였을 것 같은데, 그러한 코드들도 리팩토링 되는 걸 보면 좀 안타깝다는 생각도 듭니다. 리팩토링 되지 않을 코드를 작성하려면 갖아야할 관점 같은 것이 있을지 궁금합니다. 그리고 코치님 개인적으로나 팀에서 리팩토링을 결정하기까지의 근거들도 궁금합니다.

A) 결국 코드는 제품을 위해서 존재하는 것이고, 제품 개발에 방해된다면 당연히 없애거나 개선해야 한다고 생각해요 ㅎㅎ 어쩔 수 없는거죠.

리팩토링 되지 않을 코드를 작성하려면, 다양한 요구사항에 대해 생각해보면 좋겠죠!? 우리가 리팩토링을 하는 이유는 "요구사항의 변경에 대응하기 용이한 코드"를 만들어가기 위함이라고 생각해요. 그러니까 다양한 요구사항에 대한 "시뮬레이션"을 돌려보면서 (꼭 직접 작성하지 않더라도..?) 어떤 상황에 대응할 수 있는 코드인지, 코드의 수정 동선은 괜찮은지 등 다양한 관점으로 바라보는거죠.

의사결정의 근거는 대체로 회고를 통해서 산출이 되어요. "요즘 개발할 때 불편한점 혹은 문제되는 점"을 다같이 이야기 해보고, 이 때 나오는 주제를 선정해서 리팩토링을 진행하는거죠.

Q) 컴포넌트를 구현할 때 설계를 먼저하고 구현을 하듯, 리팩토링 과정에서도 설계하는 작업이 필수적이라고 생각합니다. 이때 어떠한 것들을 고민하시고 계획하시나요? 그리고 실질적으로 리팩토링할 때 꿀팁 같은게 있을까요? 가령 선 테스트 코드 작성 후 리팩토링 같은 게 있을 것 같은데, 이외에 다른 것이 있을까요?

A) 위에서 이야기한 것과 동일한데요, "어떤 요구사항에 대응할 수 있는가"를 볼 수 있어요. 가령, 지금처럼 디자인 시스템을 만든다고 가정했을 때 일단 만들어서 배포해보고 다양한 프로젝트에 설치하여 잘 적응 되는지 등을 토대로 판단해보는거죠.

그런데 사실 그렇게 하기가 쉽지 않아요. 시간제약이 무엇보다 크달까..

여튼 저는 리팩토링을 할 때에는 가끔 "극단적"인 생각이 필요하다고 생각해요. 디자인 시스템을 만든다면, 지금은 리액트로 되어있지만 vue 기반의 프로젝트에서 불러올때는 어떻게 해야 좋은지 고민해보는거죠 ㅎㅎ

코드 자체에 대해 생각하기보단, 코드의 쓰임에 대해 생각해보는 습관을 가지시면 좋답니다!
