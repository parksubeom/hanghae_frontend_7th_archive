[2팀 정나리] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

  • 테스트 간 상태 공유로 인한 충돌을 방지하기 위해, 각 CRUD 함수에서 initEvents를 파라미터로 전달받고, 이를 내부에서 initEvents를 shallow copy하여 별도의 events 배열을 사용하도록 변경했습니다. 해당 수정을 통해 테스트가 병렬로 실행되어도 각 함수가 독립적인 클로저 스코프를 유지해 안정적으로 동작합니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

  • Mock handler 설정 각 테스트마다 고유한 mockhandler 를 설정할 수 있도록 setupMockHandler~ 함수를 사용했습니다. 이를 통해 병렬 테스트 환경에서도 각 테스트가 독립적인 mock 서버 상태를 유지합니다.

  • 테스트 전/후 비동기 로직 처리 beforeEach, afterEach를 활용해 테스트 시작 전후에 타이머 및 비동기 로직을 초기화/정리했습니다. 이를 통해 잔여 비동기 동작이 다음 테스트에 영향을 주지 않도록 방지했습니다.

심화 과제

과제 셀프회고

테스트코드의 ㅌ도 실무에서 작성해보지 않았었는데, 이번 기회를 통해 학습하고 익숙해질 수 있었습니다. 초반에 디스크립션에 대한 너무 세세한 고민을 하다 보니 전체 코드 작성이 다소 늦어졌는데, 다음 과제에서는 PR에서 코치님이 생각해보라고 주신 내용들을 위주로 전체적인 고민을 먼저 해보면 좋을 것 같습니다.

조금 더 리팩토링을 했으면 좋았을 테지만 그래도 끝까지 포기하지 않고 완료한 저에게 칭찬 한마디~ 해주겠습니다. 또, 과제 시작 전 과제 수행 계획을 세우고, “당장 과제를 수행하자”보다, “어떤 내용을 배워가면 좋을지”를 기준으로 생각했더니 배운 점이 많았습니다. 평일에 생각보다 시간이 부족해서 다음 과제는 일요일부터 시작할 예정입니다….

기술적 성장

  • 단위 테스트(Unit Test) vs 통합 테스트(Integration Test)

    • 단위 테스트: 개별 모듈이나 클래스의 기능을 독립적으로 검증하는 데 초점을 맞춘다.
      (예: 함수의 반환값이 예상과 일치하는지 테스트)
    • 통합 테스트: 여러 모듈이 통합되어 상호작용할 때 발생할 수 있는 문제를 검증한다.
      (예: 컴포넌트 간 데이터 흐름, API 호출 후 UI 업데이트 등)

  • Vitest

    • Vite에 직접 통합된 테스트 러너(Test Runner)로, vite.config.js 설정을 그대로 활용할 수 있다.
    • Jest와 동일한 API를 제공하여 대부분의 Jest 테스트 코드를 그대로 재사용할 수 있다.
    • 내부적으로 Assertion Library인 Chai 위에서 동작한다.
      (즉, expect 문법은 Jest 스타일이지만 실제 비교는 Chai가 처리한다.)

  • React Testing Library (RTL)

    • 단위 테스트보다는 통합 테스트에 초점을 맞춘 라이브러리이다.
    • 컴포넌트의 내부 구현이 아닌, 사용자 관점에서의 실제 동작(UI, 상호작용, 상태 변화)을 검증한다.
      (예: 버튼 클릭 후 토스트가 뜨는지 확인)

  • act()의 필요성

    • vi.advanceTimersByTimeAsync(1000)을 단순히 호출하면 Vitest의 가상 타이머는 시간을 진행시키지만 React의 렌더링 사이클이 아직 완료되지 않는다.
    • await vi.advanceTimersByTimeAsync(1000) 또는 act(() => { vi.advanceTimersByTimeAsync(1000) })로 감싸면
      React가 해당 시간 변화로 인한 useEffect, setState 등의 비동기 업데이트를 모두 처리한 뒤 안정된 렌더링 결과를 반영한다.

  • toContain vs toContainEqual

    • toContain: 참조 동일성 비교 (객체의 동일 참조 여부를 확인)
    • toContainEqual: 값 동일성 비교 (객체나 배열의 값이 동일한지 확인)

  • handler vs setup

    • handler.ts: 전역으로 공유되는 핸들러로, events 배열을 직접 import하여 사용한다.
      이 경우 테스트 간 데이터가 공유되어 상태 오염(test pollution)이 발생할 수 있다.
    • handlerUtils.ts: 팩토리 함수 형태로 각 테스트마다 독립적인 events 배열을 생성한다.
      병렬 테스트 환경에서도 테스트 간 데이터 간섭이 발생하지 않는다.

  • getBy vs findBy

    • getBy: 렌더링 직후 이미 존재하는 요소를 즉시 찾는다.
      (예: 화면에 처음부터 보이는 텍스트나 버튼)
    • findBy: 렌더링 직후에는 없지만, 비동기 로직(state 변화, fetch 등) 이후 나타나는 요소를 기다렸다가 찾는다.
      (예: 저장 버튼 클릭 후 뜨는 토스트)
    • 예시: 
      expect(await screen.findByText("저장되었습니다")).toBeInTheDocument();
      → 이는 await waitFor(() => expect(getByText(...))) 와 같은 의미이다.

코드 품질

  • 주로 description에 대한 고민과 어떻게 테스트 코드를 구성하는 것이 좋을까에 대해 고민했습니다.
  • 통합 테스트에서는 최대한 사용자가 경험하는 프로세스 그대로 로직을 구현하려고 노력했습니다.
  • 오프 코치님이 평일 Q&A 세션에서 말씀하셨던 것처럼 관리 포인트를 줄이기 위해 전역 이벤트 객체를 만들어서 사용할 수 있도록 수정하면 좋을 것 같습니다.
  • 또한 계속 반복되는 이벤트 생성/수정/삭제 테스트 로직을 리팩토링 할 수 있을 것 같습니다.

학습 효과 분석

  • 오프 코치님이 만들어주신 과제를 보며 대략 어떤 방식으로 테스트 코드와 환경을 구성해야 할지 배웠습니다.
  • 추가적으로 다양한 메서드들의 동작 원리와 테스트를 어떻게 하면 더 효율적으로 구성할 수 있을지 학습이 필요합니다.

과제 피드백

  • hint 가 있어서 어느 부분에 집중해서 과제를 진행해야 하는지 알 수 있어서 좋았습니다.
  • 과제의 난이도가 학습하기 적절하게 좋았습니다. 테스트코드를 제대로 처음 학습했지만 노력해서 찾아본다면 그래도 과제를 포기하지 않고 수행할 수 있을 정도였다고 느껴졌습니다.
  • handler.ts와 handlerUtils.ts가 각각 어떤 역할로 사용되며, 어떻게 구성해야하는지가 어려웠습니다.

리뷰 받고 싶은 내용

  • 테스트 디스크립션 작성과 테스트 케이스 분리가 적절하게 된 것인지 궁금합니다.

궁금한 점

  • events 객체를 각 describe 함수 스코프 내에서만 동일하게 사용하도록 정의한 경우가 많은데, expect(result).toBe()로 검증할 때 events.title 이렇게 참조로 사용하는 것보다 ‘일정 타이틀’ 이라는 원시값 그대로 사용하는게 나을까요? 원시값으로 사용하면 너무 코드가 길어지고, 참조로 사용하면 데이터가 오염되었을 수 있어서 고민되었습니다.

안녕하세요 나리님! 1주차 과제 잘 진행해주셨네요 ㅎㅎ 고생하셨습니다!

테스트 디스크립션 작성과 테스트 케이스 분리가 적절하게 된 것인지 궁금합니다.

어떤 부분을 신경쓰면서 작성했고, 어떤 기준으로 분리했는지를 말씀해주셔야 더 정확한 피드백을 드릴 수 있답니다!

가령 이런 질문에 대해 답변드릴 수 있는건.. "적절하게 되었네요" 인데요 왜 적절한지가 궁금한거겠죠?

마찬가지로 나리님이 가지고 계신 적절함의 기준에 대해 이야기 해주시면 좋아요.

어쨌든 답변 드리자면,

<1. 명확한 테스트 구조>

  • describe로 함수별 그룹화가 잘 되어 있음
  • 각 it 블록이 하나의 명확한 시나리오만 테스트

<2. 구체적인 테스트 설명>

  • "1월은 31일 수를 반환한다" - 입력과 예상 결과가 명확
  • "윤년의 2월에 대해 29일을 반환한다" - 특수 케이스 명시
  • "유효하지 않은 월에 대해 '유효하지 않은 월입니다...'라는 에러를 반환한다" - 에러 메시지까지 명확히 기술

<3. 경계값 테스트>

  • 월초/월말, 연초/연말, 윤년 등 엣지 케이스 잘 포함
  • 유효/무효 입력 케이스 구분

events 객체를 각 describe 함수 스코프 내에서만 동일하게 사용하도록 정의한 경우가 많은데, expect(result).toBe()로 검증할 때 events.title 이렇게 참조로 사용하는 것보다 ‘일정 타이틀’ 이라는 원시값 그대로 사용하는게 나을까요? 원시값으로 사용하면 너무 코드가 길어지고, 참조로 사용하면 데이터가 오염되었을 수 있어서 고민되었습니다.

변수로 검증할경우 어떤 값이 들어있는지 다른 사람이 파악하기가 어렵기 때문에, 테스트 목적에 적합하지 않다고 생각해요! 그래서 리터럴(원시값)을 사용하는 방식을 추천드립니다!

코드가 길어지는게 사실 그렇게 문제일까? 라고 생각해보면 큰 문제는 아니라고 생각해요 ㅎㅎ

코드를 에디터에서 접어서 사용할수도 있으니까요!
