캘린더 DnD(Drag and Drop) 기능 테스트 관련 논의 요약

E2E 테스트 관점
  • 실제 사용자 플로우를 재현하며, DnD 동작이 UI 상에서 올바르게 수행되는지
    (예: 마우스 다운 → 이동 → 업 이벤트) 를 검증할 수 있습니다.
  • 하지만 실행 비용이 높고, 디버깅이 어렵기 때문에
    **핵심 기능(예: 로그인, 결제 등)**처럼 절대 깨지면 안 되는 부분에 우선 적용하는 것이 효율적이라는 의견이 있었습니다.
통합 테스트 관점
  • DnD 자체의 내부 동작(좌표 계산, 드롭 이벤트 등)은 라이브러리에 위임할 수 있으므로 직접 테스트하지 않아도 됩니다.
  • 대신, “DnD 결과로 일정 데이터가 올바르게 업데이트되는지”,
    즉 드래그 후 PUT 요청이 정상적으로 보내지고, UI가 옮겨진 날짜로 반영되는지를 검증하는 것이 더 실질적이라는 의견이 많았습니다.
  • 이를 통해 유지보수 비용을 줄이고, 기능의 비즈니스 로직에 초점을 맞출 수 있습니다.

결론

  • 비용과 신뢰도를 종합적으로 고려하여 DnD는 E2E 테스트로 진행하기로 결정했습니다.
  • 과제의 전체 개발 시간과 리소스 제약을 감안했을 때, 모킹이 적고 실제 사용자 환경에 가까운 테스트가 효율적이라고 판단했습니다.
  • 특히 DnD는 DOM API 기반의 상호작용 기능이므로, Node.js 환경의 통합 테스트보다 실제 브라우저 환경에서 수행되는 E2E 테스트가 신뢰도가 높다는 점에 팀이 합의했습니다.

1. useEventForm 테스트 (단위 테스트 / vitest)

  • 선정 이유
    • 폼 상태 관리 핵심 훅으로, UI 의존성이 없음
    • 입력/검증/리셋 등 순수 로직 중심
    • 다양한 엣지케이스 검증 필요
    • 빠른 피드백과 높은 커버리지 달성 가능
  • 검증 항목
    • 초기화 로직
    • 상태 변경(setters)
    • 시간 유효성 검사
    • 폼 리셋 및 편집 로직
    • 반복 일정 처리
  • 테스트 목표
    • 로직의 정확성과 회귀 방지 (UI와 독립적으로 동작 보장)

2. 캘린더 날짜 클릭 → 폼 자동 완성 (통합 테스트 / React Testing Library)

  • 선정 이유
    • DateCell → useDateClick → useEventForm → EventForm 간 상호작용 검증 필요
    • 브라우저 API 의존 없음 → 통합 테스트로 충분
    • 실제 시나리오 기반 빠른 피드백 가능
  • 검증 항목
    • 날짜 클릭 시 폼에 날짜·시간 자동 입력
    • UI에 값이 올바르게 반영되는지 확인
  • 테스트 목표
    • 컴포넌트 간 데이터 흐름과 폼 자동 완성 시나리오의 정확성 검증

3. 드래그 앤 드롭 (D&D) 테스트 (E2E / Playwright)

  • 선정 이유
    • 마우스 이벤트, 좌표, DOM 조작 등 복합적 사용자 인터랙션
    • DnD 라이브러리와의 통합 검증 필요
    • 시각적 피드백과 실제 사용자 경험 중심 기능
  • 검증 항목
    • 일정 이동 시 날짜 변경, 시간 유지
    • 반복 일정 → 단일 일정 변환
    • 겹침 시 경고 표시 및 처리
    • 속성 유지(제목·시간·카테고리 등)
    • 월간/주간 뷰 간 동작 일관성
  • 테스트 목표
    • 실제 브라우저 환경에서 사용자 중심 동작과 시각적 피드백 검증

과제 셀프회고

기술적 성장

  • 지난 1주차 과제에서 통합 테스트를 진행하며 알림 select가 카테고리 select와 다르게 선택되지 않는 이슈가 있었습니다. 해당 부분 코드를 확인해보니 aria-label 속성 관련 문제였고, 이 내용은 1주차 팀 리뷰 때도 언급된 바 있어, 원인을 정리해 팀원들과 공유했습니다. → 링크

  • 이번 과제를 통해 가장 크게 성장한 부분은 E2E 테스트 도입에 대한 자신감이었습니다. Playwright의 다양한 기능(codegen, --ui, debug test)을 직접 활용하면서 테스트 코드 작성이 생각보다 훨씬 실용적이고 생산적인 과정이라는 것을 느꼈습니다. 테스트가 자동으로 기다려주는 locator의 자동 대기 기능, 병렬 실행 구조가 Playwright만의 장점인 부분을 배웠고, 이 장점을 직접 체감했습니다. 이전에는 E2E 테스트가 어렵고 비용이 많이 든다고 생각했지만, 직접 익숙해지고 도구를 이해하니 충분히 실무에 도입 가능한 생산적인 방법이라고 느꼈습니다.

  • 또한 Playwright의 Clock API(page.clock.setFixedTime)을 사용해 테스트가 시간에 의존하지 않도록 고정함으로써, 날짜 기반 테스트의 안정성을 확보했습니다. 캘린더처럼 시간이 중요한 서비스에서는 중요하다고 생각합니다.

  • Testing Trophy와 Pyramid 개념도 새롭게 배웠습니다. 예전에는 단순히 “테스트는 E2E로 다 하면 되는 거 아닌가?”라고 생각했는데, 이번에는 단위·통합·시각적 회귀 테스트를 균형 있게 설계하는 것이 훨씬 현실적이라는 것을 알게 되었습니다. 특히 DnD 기능 테스트를 논의하면서 “테스트가 실행되는 환경이 Node인지 Browser인지”를 고려해야 한다는 점을 팀원들과 이야기하며 크게 느꼈습니다. 실제로 DOM API를 직접 다루는 기능은 브라우저 환경에서 테스트해야 더 신뢰할 수 있다는 것도 깨달았습니다.

  • 시각적 회귀 테스트는 이번 주차를 진행하면서 처음 알게 되었습니다. UI가 안정적으로 유지되는지를 확인하는 테스트가 있다는 점이 인상적이었습니다.

코드 품질

  • 이번 과제에서는 유지보수성과 테스트 용이성을 고려한 구조 설계에 중점을 두었습니다. 먼저 E2E 테스트 환경에서 테스트 데이터를 관리하기 위해 JSON 파일을 별도로 분리하여 테스트 실행 간 데이터 충돌을 최소화했습니다. UI 측면에서는 Storybook에서 모킹이 최소화되도록 비즈니스 로직이 배제된 순수 UI 컴포넌트 단위로 구조를 정리했습니다. Form 요소들도 공통 UI로 추출하여 재사용성과 시각적 일관성을 높였고, 스냅샷이 안정적으로 유지되도록 개선했습니다.

  • 비즈니스 로직 측면에서는 Optimistic Update 로직을 생성 로직과 분리하여 유지보수성을 높였습니다. 실제 서버 데이터를 갱신하는 saveEvent와, 드래그 앤 드롭 시 클라이언트 상태만 낙관적으로 반영하는 moveEvent를 구분했습니다. DnD 기능에서는 UX를 고려해 드래그 중인 요소를 숨기고, 미리보기(Overlay)를 별도의 UI 컴포넌트로 재활용했습니다. EventItem은 순수 UI를 담당하고, DnD를 감싸는 컨테이너 컴포넌트는 상호작용만 담당하도록 구조를 분리했습니다. 또한 DnD 관련 핸들러(onDragStart, onDragEnd)를 별도의 hook으로 분리해 코드의 명확성과 테스트 용이성을 높였습니다.

  • Playwright 테스트는 각 CRUD 플로우를 독립적으로 구성하여 테스트 간 의존성을 최소화했습니다. 또한 setup과 teardown 구조를 명확히 하여 테스트 재실행 시 안정성을 확보했습니다. 접근성 규칙(aria-label, aria-labelledby)을 적용해 테스트 신뢰도와 유지보수성을 함께 개선했습니다. 다만 병렬 실행 시 JSON 데이터 충돌 문제를 완전히 해결하지 못한 점은 아쉬웠습니다. 다른 팀원들이 실행마다 JSON을 분리해 환경을 완전히 격리한 점을 참고하여, 향후 구조 개선 포인트로 삼을 계획입니다.

학습 효과 분석

  • 이번 챕터를 통해 단순히 테스트 코드를 작성하는 법이 아니라, 테스트를 설계하고 전략적으로 도입하는 사고방식을 배우게 되었습니다. Playwright를 사용하며 --ui와 debug test 모드 덕분에 오류 원인을 빠르게 시각적으로 확인할 수 있었습니다. 이러한 툴 제공이 통합 테스트나 단위 테스트보다 훨씬 빠르게 테스트 코드를 작성하고 활용할 수 있게 해준다고 느꼈습니다.

  • 또한 팀원들과 변하지 않는 것에 대한 테스트를 해야하는지를 논의했습니다. 예를들어 UI가 잘 변경되지 않는 제품의 경우 오히려 시각적 회귀 테스트를 만들어두면 의도하지 않은 UI 변경이 발생했을 때 잘 발견할 수 있지 않을까? 생각했습니다.

  • 이번 과제를 계기로 다양한 테스트가 있지만, 가장 장벽이 낮고 효율성은 높은 Playwright를 활용한 E2E 테스트를 실무에 도입해보고 싶다는 목표가 생겼습니다.

과제 피드백

이번 과제는 단순히 테스트를 작성하는 것을 넘어 테스트 전략 수립 -> 테스트 종류별 분류 -> 실제 코드 적용까지 전 과정을 경험할 수 있었다는 점에서 매우 의미 있었습니다. 팀원들과 E2E, 통합, 시각적 회귀 테스트의 역할과 경계를 직접 비교하고 토론할 수 있었던 점이 큰 도움이 되었습니다. 개인적으로 항해를 신청한 주요 이유 중 하나가 테스트에 대한 경험이었는데, 3주간 많이 익숙해졌고 다양한 내용을 배울 수 있어서 좋았습니다. 그리고 특히 이번 주 내용은 어떤걸 해야 하는지 과제가 명확해서 과제 진행이 용이했던 것 같습니다. 다만 Storybook도 처음 써보고, E2E도 처음 써봐서 코드를 다 작성하기에는 시간이 부족해서 그 부분이 아쉽습니다. 추가로 발제 자료에 CRX나 codegen, playwright UI에 관한 내용도 있었으면 더 잘 활용해서 개발 할 수 있었을 것 같습니다. 감사합니다!

리뷰 받고 싶은 내용

  1. StoryBook을 활용한 시각적 회귀 테스트 단위 질문

    • EventItem 내부 title이 길어졌을 경우에 ellipsis가 되어야하는 요구사항이 있었습니다. 이때, 이 UI에 대한 시각적 회귀 테스트를 CalendarView, EventItem, Cell 중 어느 단위에서 수행하는 것이 효율적이며 유지보수성이 높은지에 대해 피드백을 받고 싶습니다.

  2. E2E 테스트 구조 및 데이터 독립성 관련 피드백

    • 현재 E2E 테스트는 e2e.json을 따로 만들어서 관리하고, setup에서 clearEvents를 해주도록 했습니다. 그런데 실무에서 실제 DB를 기준으로 E2E 테스트를 작성하게 되면 어떻게 관리 하는지 궁금합니다.

  3. E2E 테스트 케이스의 최소화 전략에 대한 피드백

    • 현재 E2E에서는 테스트 비용을 줄이기 위해 기본 CRUD 플로우만 검증하고 있습니다.
    • 예를 들어, 1. 일반 일정 생성 2. 일정 겹침 시 다이얼로그 예/아니오 케이스 (2개). 이런 식으로 최소한의 핵심 케이스만 작성했는데, 실무에서는 엣지 케이스를 통합 테스트로 보강하는 것이 좋은지, 아니면 유지보수성을 고려해 E2E를 최소화하는 전략이 더 나은지에 대해 피드백을 받고 싶습니다.

  4. UI가 잘 변경되지 않는 제품의 경우 오히려 시각적 회귀 테스트를 만들어두면 의도하지 않은 UI 변경이 발생했을 때 잘 발견할 수 있지 않을까? 에 대해 어떻게 생각하시는지 궁금합니다. 저는 처음에 UI가 잘 변경되지 않는 제품이면 시각적 회귀 테스트를 만드는 것이 더 비용이라고 생각했는데, 오히려 변경 되었을 때 오류 잡기가 편하지 않을까 하는 의견을 들었습니다.

수고했어요 나리! 지난 3주간 테스트 챕터를 진행하면서 "어떤 테스트를 어디에 써야 하는가"를 깊이 고민한 흔적이 보여요. 특히 이번 과제를 통해 E2E 테스트에 대한 인식이 완전히 바뀐 것 같아서 정말 좋네요.

"이번 과제를 통해 가장 크게 성장한 부분은 E2E 테스트 도입에 대한 자신감이었습니다." 이라고 느낀 변화가 큰 성과라 생각합니다. 대부분 개발자들이 E2E를 막연히 어렵다고만 생각하고 시도조차 못하고 있었죠. 이번 기회에 Playwright와 같은 도구들을 직접 써보면서 그렇게 어렵지 않구나. 실용적이구나 하고 깨달음을 얻었다는게 중요합니다.

이런 경험이 있어야 나중에 실무에서 "우리 팀에 E2E 테스트 도입해봅시다"라고 자신있게 제안할 수 도 있고 "e2e 할 줄 알아요?" 라고 물어봐도 "할 줄은 안다. 해봤다" 라고 얘기할 수 있게 될거에요. 특히나 깊이 있게 알지 못해도 이제 막 해야 하는 상황이라면 지금 수준 이상의 것들이 아니니 충분히 자신감을 가져도 좋아요

Q1) StoryBook 시각적 회귀 테스트 단위 - EventItem 내부 title이 길어졌을 경우 ellipsis 요구사항을 CalendarView, EventItem, Cell 중 어느 단위에서 테스트하는 것이 효율적인지? => EventItem 단위에서 테스트하는 게 가장 효율적입니다. ellipsis는 EventItem의 책임이기 때문이죠. CalendarView는 너무 범위가 넓어서 노이즈가 많고, Cell은 EventItem을 포함하는 컨테이너라 역할이 다릅니다. 시각적 회귀 테스트는 "변경의 영향 범위"와 "책임 소재"를 기준으로 단위를 정하세요. EventItem에서 title prop에 긴 문자열을 넣은 스토리를 만들고 그 스냅샷을 관리하면, 나중에 EventItem의 스타일이 변경될 때만 해당 스냅샷이 깨지니 유지보수가 쉽습니다.

Q2) 실무에서 실제 DB를 기준으로 E2E 테스트를 작성하게 되면 어떻게 관리하는지? => 테스트 전용 DB 환경을 별도로 구축합니다(dev, staging, prod처럼 test 환경). 그리고 수정과 삭제시 테스트 일관성 유지를 위해 트랜잭션 롤백을 추가합니다. 시작할 때 강제로 초기화를 시킬 수 있는 방안이 있어야 동일 테스트에 동일한 결과를 줄 테니까요. 여차하면 쓰기는 포기하고 읽기전용만 e2e를 하는 경우도 있습니다. 안되면 지금처럼 mocking을 하거나 e2e.json등으로 활용하겠죠. 핵심은 어떤 방식으로는 "테스트 실행 후 상태가 변하지 않는 환경"을 만드는 게 핵심입니다.

Q3) E2E 최소화 전략 - 핵심 케이스만 E2E로 작성하고 엣지 케이스는 통합 테스트로 보강하는 게 좋은지? => 맞아요. E2E는 "핵심 사용자 플로우"만 검증하고, 엣지 케이스는 통합/단위 테스트로 커버하는 게 맞아요. E2E는 실행 비용이 높고 디버깅이 어려우니 이거 안되면 장애! 이런 필수 시나리오를 중심으로 검증합니다. 복잡한 로직에 대한 검증은 통합 테스트로 검증하구요.

Q4) UI가 잘 변경되지 않는 제품의 경우 시각적 회귀 테스트를 만들어두면 의도하지 않은 UI 변경을 잘 발견할 수 있지 않을까? => 실무의 아이러니인것 같아요 ㅠ 비용을 고려하다보니 잘 안변하는데 왜 해? 했다가 한번쯤은 그걸로 고생하고 막상 하려니 잘 안변할텐데 하면서 안하고... 그래서 그냥 그런 생각없이 하면서 함께 만들어두는게 제일 좋긴 한데 또 그렇게 하지 못하는 거죠. 잘 되어있는 곳에 가면 잘 되어 있는 걸 배우고 없다면 처음 입사했을때 의욕을 가지고 한번 도전해보는거 어때요? ㅎ
