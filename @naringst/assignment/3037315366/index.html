사용할 도구

TailwindCSS 4.x

shadcn/ui

  • Radix UI 기반, 접근성 내장

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

Before 패키지에서 발견한 문제점

  1. 공통적인 문제들

  • Props라는 인터페이스를 통해 해당 컴포넌트가 어떤 역할을 하는 컴포넌트인지 알 수 있어야 한다. 그런데 모든 컴포넌트가 UI요소와 비즈니스 요소가 props에 뒤섞여 있다.

  • 따라서 이 컴포넌트가 "뭘 하는 컴포넌트인지" 컴포넌트를 불러서 렌더링 하는 부분에서 그 책임을 알기가 어렵다.

  • Modal.tsx

useEffect(() => {
    if (isOpen) {
      document.body.style.overflow = 'hidden';
    } else {
      document.body.style.overflow = 'unset';
    }
    return () => {
      document.body.style.overflow = 'unset';
    };
  }, [isOpen]);
  1. atoms/Badge
  • 기능은 하나도같지 않고 도메인도 같지 않고 피쳐도 같지 않은 요소들이 UI가 같다는 이유 하나만으로 하나의 뱃지로 사용됨.
  • 이렇게 되면 userRole을 나타내는 내용만 다른 변경사항이 들어왔을때, 연관없는 priority 부분도 수정이 필요한 상황이 발생함.
  • 이렇게 되면 하나의 기능이 추가될 때마다 수정해야 하는 코드가 많아지고, 이렇게 되면 유지보수가 어려워짐.
  • 따라서 badge는 statusBadge, userRoleBadge, priorityBadge를 받게 하거나, UI 컴포넌트로 만들고 싶었다면 서비스 로직을 해당 컴포넌트 안에 넣어두면 안됨. 따라서 UI만 보여주고 priority, status, userRole등은 아예 인터페이스에 등장하지 않아야 함.
  • 결국 UI요소로서 Badge를 사용한다면 BadgeProps에 비즈니스 적인 props들이 담긴 것 자체가 문제고, 인터페이스를 우선적으로 바꿔야 함.
//before
interface BadgeProps {
  children?: React.ReactNode;
  type?: 'primary' | 'secondary' | 'success' | 'danger' | 'warning' | 'info';
  size?: 'small' | 'medium' | 'large';
  pill?: boolean;
  status?: 'published' | 'draft' | 'archived' | 'pending' | 'rejected';
  userRole?: 'admin' | 'moderator' | 'user' | 'guest';
  priority?: 'high' | 'medium' | 'low';
  paymentStatus?: 'paid' | 'pending' | 'failed' | 'refunded';
  showIcon?: boolean;
}


// after
interface BadgeProps {
  label: string;
  type?: 'primary' | 'secondary' | 'success' | 'danger' | 'warning' | 'info';
  size?: 'small' | 'medium' | 'large';
  pill?: boolean;
  showIcon?: boolean;
}
  • 그리고 여기에 비즈니스 로직을 입히고 싶다면 그 로직을 서비스 함수로 빼서 그 함수에서 계산한 결과를 Badge에 넘겨주거나, 그걸 다루는 wrapper로 싼 컴포넌트를 만들어야 한다.
  • 예를들어 getPaymentStatusColor 이런식으로 함수를 만들고 → Paid 를 넣으면 ⇒ danger 이런식으로 뱉어서 아래와 같이 type이나 아니면 bg, color 등등의 "UI 연관 프로퍼티"를 받도록 해야 한다.
<Badge type={getPaymentStatusColor(paid)} />..
  1. Form
  • 버튼과 마찬가지로 FormInput, FormSelect, FormTextarea는 UI와 도메인 로직이 섞여 있다.
  • 그리고 form 데이터 관리를 Page 단위에서 하고 있는데, 그렇게 되면 폼 상태가 변경될 때마다 전체 페이지가 리렌더 된다. 따라서 폼 컴포넌트를 따로 분리하고, 해당 컴포넌트 내에서 form 데이터 상태 관리를 하도록 한다.
  • 또한 shadCN에서는 자체적으로 react-hook-form을 사용해서 폼 상태관리 최적화가 된다.

컴포넌트 설계, 폼 관리 측면에서 문제점 파악

개편 과정에서 집중한 부분

  • 레거시가 있고 이걸 새로 디자인 시스템을 적용하여 수정한다고 가정하고 개발했습니다.

  • 또한 현업에서 여러 사람의 작업이 이루어진다고 했을때, 어디서부터 어떻게 작업하면 좋을지를 고민했습니다.

  • 따라서 기존에 atoms 로 나뉘어 있던 요소들부터 작업을 진행했는데, 하면서는 UI 컴포넌트는 UI의 역할만, 나머지 비즈니스 로직은 함수로 분리하거나 UI 컴포넌트 밖에서 처리하도록 했습니다. 가장 많이 신경 쓴 부분은 UI와 도메인 로직이 섞이지 않도록 하고, 도메인 로직 중에서도 서비스 함수로 분리 가능한 부분들은 따로 service함수로 분리했습니다. 그렇게되면 비즈니스 로직이 바뀌었을 때, 해당 함수만 찾아서 변경하면 되기 때문에 유지보수 측면에서 유리하다고 판단했습니다.

  • 그렇게 작은 단위부터 점점 큰 단위로 이동하면서 마이그레이션을 진행했고, 우선 UI 컴포넌트 분리에만 집중했습니다. 따라서 아직 다 고치지는 못해서 추후에 리팩토링을 추가로 해야 할 것 같습니다.

  • 또한 폴더 구조는 FSD 구조를 사용했습니다. 이는 응집도를 높이기 위함인데 예를들어 user 관련 로직에서만 필요한 타입이나 함수들이 가까이에 있으면 찾기가 용이하고, 기능 수정이 될 때에도 해당 feature 폴더 안에서만 찾아서 수정하면 됩니다. 또한 해당 feature가 사라질 경우에는 그 폴더만 삭제하면 되기 때문에 유지보수성을 위해 FSD구조를 사용했습니다.

사용한 기술 스택 경험

[폴더 구조] FSD 구조를 활용해보았습니다. 현재 구현해야 하는 로직이 사용자 관련 로직, 게시글 관련 로직 두개로 나뉘는데, 둘의 로직이 UI 컴포넌트에 섞여있었습니다. 따라서 이를 명확히 하기 위해 각 기능별 폴더로 분리하고, 공통으로 사용하는 것만 따로 common으로 두었습니다.

[디자인 시스템]

  • primary token, semantic token을 원티드 디자인 시스템을 참고하여 만들었습니다.
  • 이후 cva에서 semantic token을 사용하도록 지정했습니다.
  • 컬러와 타이포그래피, 다크/라이트모드 까지만 설정했습니다.
