[FSD란 무엇인가] (펼치기)

1. 등장 이유

전통적인 components/, pages/, utils/ 중심 기술 기반 구조는
기능이 여러 폴더에 흩어져 규모가 커지면 유지보수가 어려워진다.
FSD는 이를 해결하기 위해 비즈니스 도메인 중심 구조를 제안한다.

기존 구조의 문제

  • 기능 단위 코드가 여러 폴더에 흩어짐 → 추적·수정 비용 증가
  • utils/hooks가 점점 커지고 아무 곳에서나 import됨 → 의존성 혼란
  • 변경 시 어디가 깨질지 예측하기 어려움
  • 확장성·재사용성 부족

2. FSD가 해결하려는 핵심

  • 코드를 기술이 아닌 기능·도메인 중심으로 묶기
  • 높은 응집도, 낮은 결합도 유지
  • 기능을 기반으로 역할을 명확히 나누어 재사용성과 유지보수성을 높임

3. FSD의 3가지 차원

Layer (수직 계층)

상위 레이어만 하위 레이어를 의존할 수 있음.

계층:
app → pages → widgets → features → entities → shared

Slice (수평 도메인 분리)

레이어 내에서 비즈니스 도메인 기준으로 작성.
같은 레이어의 다른 슬라이스는 참조 불가.

Segment (기술적 목적 분리)

도메인 내부에서 목적에 따라 분리.

예: ui, model, api, lib, config

4. Layer별 책임

app

  • 앱 전체 설정
  • Router, Provider, 글로벌 스타일, 전역 상태

pages

  • 라우팅에 직접 대응되는 페이지
  • 여러 widget + feature 조합으로 페이지 구성

widgets

  • 페이지에서 사용되는 완성된 UI 블록
  • feature/entity를 조합하지만 도메인 로직은 직접 갖지 않음

features

  • 사용자에게 가치를 주는 기능(동사적 개념)
  • entity를 사용해 특정 기능을 구현
  • UI + model + api 포함 가능

entities

  • 비즈니스 핵심 데이터(명사적 개념)
  • 타입, api, 상태, 도메인 UI
  • entity 간 import 불가

shared

비즈니스와 무관한 전역 공통 자원:

  • shared/ui: 버튼, input 등 범용 UI
  • shared/api: axios instance, http client
  • shared/lib: 날짜 utils, debounce 등 순수 로직
  • shared/routes: 전역 라우트 상수

[과제 페이지 FSD 적용 내용]

shared에 들어가야 하는 것

  • ui: Button, Input, Select, Dialog, Card, Table, Textarea 등 범용 UI
  • lib: useQueryParams, highlightText 등 순수 로직 유틸
  • config: API_BASE_URL 등 환경 설정

app 책임

  • QueryClientProvider (TanStack Query)
  • RouterProvider (React Router)
  • ReactQueryDevtools
  • 전역 레이아웃 (Layout 컴포넌트 → widgets/layout으로 분리)

pages 책임

  • URL 단위 페이지 구성
  • 여러 widget/feature 조합
  • 예: PostsManagerPage

widgets 책임

  • 페이지에서 여러 번 재사용 가능한 UI 조각
  • Smart Widget 패턴: hooks를 직접 호출하여 props drilling 최소화
  • 예: PostListWidget, Layout

features 책임

  • 특정 기능 실행 담당 (사용자 액션 / mutation)
  • 예:
    • post-add
    • post-edit
    • post-detail
    • post-filters
    • comment-list
    • user-detail

entities 책임

  • 도메인 데이터 정의 및 조회(query)
  • 예:
    • post
    • comment
    • user
    • tag

[FSD에서 entities와 features의 경계]

  • entities: 조회(query) 담당, 재사용 가능한 순수 데이터 로직
  • features: 사용자 액션(mutation) 담당, 특정 기능에 종속된 로직
  • entities는 features를 import할 수 없다는 레이어 규칙의 중요성

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

[테스트 코드 추가]

  • 기존의 기능들이 리팩토링 하면서 동작하지 않게 되는 이슈를 막기 위해 통합테스트를 일부 추가하여 리팩토링 하면서 확인했습니다.

[App 진입점 routes, query provider 분리]

  • main.tsx → app/App.tsx → app 하위 providers, routes 설정
  • 설정들을 각각 segment로 분리하여, 추후 라우트가 확장되거나 다른 provider가 추가될 때에도 한 폴더에서 관리할 수 있도록 했습니다.
  • <Routes/> 태그를 <RouterProvider router={router} />로 수정하고, router에서 분기 처리를 확인할 수 있도록 했습니다.
  • 내부 Header와 Footer를 Layout 컴포넌트(widgets/layout)로 분리하여 보다 선언적인 코드로 구조를 파악할 수 있도록 했습니다.
// app/routes/index.tsx
export const router = createBrowserRouter([
  {
    element: <Layout />,
    children: [{ path: "/", element: <PostsManagerPage /> }],
  },
])

[Smart Widget 패턴 적용]

  • 처음에는 widget이 UI만 가지고 있어야 한다고 생각해서, page에서 hook을 호출해서 내려주었더니 불필요한 props drilling이 많아졌습니다 (14개의 props).
  • FSD에서 widget은 features/entities의 hooks를 직접 호출할 수 있다는 것을 알게 되었고, widget에서 훅을 호출하도록 변경하여 props를 4개로 줄였습니다.
// Before: page에서 모든 상태를 내려줌 (14 props)
<PostListWidget
  posts={posts}
  total={total}
  isLoading={isLoading}
  search={search}
  tag={tag}
  // ... 14개의 props
/>

// After: Smart Widget (4 props - callback만 전달)
<PostListWidget
  onPostDetail={handlePostDetail}
  onPostEdit={handlePostEdit}
  onPostDelete={handlePostDelete}
  onUserClick={handleUserClick}
/>

[search params 관리하기]

search params로 관리되는 limit, sortOrder, tag, sortBy, search를 어떻게 관리할 것인가?

  • 이 모든 값들은 post와 관련된 기능이므로 features/post-filters로 분리했습니다.
  • Zod를 사용해서 URL로부터 오는 값의 타입을 검증해서 사용할 수 있도록 했습니다.
  • 검색은 onChange가 아닌 Enter 키를 눌렀을 때만 URL에 반영되도록 구현했습니다.
// features/post-filters/lib/schema.ts
export const postFiltersSchema = z.object({
  search: z.string().optional(),
  sortBy: z.enum(["none", "id", "title", "reactions"]).default("none"),
  sortOrder: z.enum(["asc", "desc"]).default("asc"),
  skip: z.coerce.number().default(0),
  limit: z.coerce.number().default(10),
  tag: z.string().optional(),
})

[게시물 정렬 기능 구현]

  • sortBy(id, title, reactions)와 sortOrder(asc, desc)로 클라이언트 사이드 정렬 구현
  • 정렬 로직은 비즈니스 로직이므로 entities/post/lib/sortPosts.ts로 분리
  • 얼리 리턴 패턴으로 가독성 향상
// entities/post/lib/sortPosts.ts
export const sortPosts = (posts: Post[], sortBy: SortBy, sortOrder: SortOrder): Post[] => {
  if (sortBy === "none") return posts

  const sorted = [...posts].sort((a, b) => getComparison(a, b, sortBy))

  if (sortOrder === "desc") return sorted.reverse()
  return sorted
}

[TanStack Query 낙관적 업데이트]

  • 게시물 추가/수정/삭제 시 낙관적 업데이트 적용
  • dummyjson API가 실제로 데이터를 저장하지 않으므로, invalidateQueries 대신 캐시만 업데이트
  • 새 게시물의 임시 ID(Date.now())가 너무 길어서 "NEW"로 표시
// features/post-add/api/postAddMutation.ts
onMutate: async (newPost) => {
  await queryClient.cancelQueries({ queryKey: postKeys.all })

  queryClient.setQueriesData<PostsData>({ queryKey: postKeys.all }, (old) => {
    const optimisticPost: Post = {
      id: Date.now(), // 임시 ID
      title: newPost.title,
      body: newPost.body,
      author: { id: newPost.userId, username: "나", image: "" },
      // ...
    }
    return { posts: [optimisticPost, ...old.posts], total: old.total + 1 }
  })
}

[태그/검색 필터에서 페이지네이션 수정]

  • 태그나 검색어가 있을 때 페이지네이션이 동작하지 않는 버그 발견
  • postsByTagQueryOptions와 searchPostsQueryOptions에 limit/skip 파라미터가 누락되어 있었음
  • query key에도 filters를 포함시켜 페이지별로 캐시가 분리되도록 수정
// entities/post/api/postKeys.ts
export const postKeys = {
  all: ["posts"] as const,
  list: (filters) => [...postKeys.lists(), filters] as const,
  byTag: (tag, filters) => [...postKeys.all, "tag", tag, filters] as const, // filters 추가
  search: (q, filters) => [...postKeys.all, "search", q, filters] as const, // filters 추가
}

[헷갈렸던 부분 - comments]

comment에서 api, lib, model, ui 이렇게 분리했는데, UI에서 필요한 onClick 메서드나 이런 것들은 어디서 어떻게 참조해야하지?

  • entity: comment 조회(query) 관리, 재사용성 높은 로직, UI 상태를 포함하지 않음
  • features: 특정 기능을 나타냄(mutation), 재사용성 낮음, UI 상태를 포함

=> 따라서 comment 조회 로직은 entity/comment에 정의하고, 실제로 댓글을 추가/수정/삭제하는 mutation은 features/comment-list/api에서 관리한다!

[헷갈렸던 부분 - post]

post entity 하위 ui에 어떤 요소를 넣어야 하지?

=> PostReactions, TagBadge, UserInfo 같이 여러 곳에서 재사용될 수 있는 작은 단위의 UI 컴포넌트만 entity/ui에 넣었다.

  • 조회(query)는 entities, 변경(mutation)은 features로 분리했다.
  • entities/post/model/usePost.ts: useQuery로 posts 조회만 담당
  • features/post-add/api/postAddMutation.ts: 게시물 추가 mutation
  • features/post-edit/api/postEditMutations.ts: 게시물 수정/삭제 mutation

[widgets]

feature, entity를 쪼개다 보니 그것보다 상위의 영역으로 구별이 필요하다고 생각되는 단위가 있었고, 그 단위를 widget으로 분류.

  • PostListWidget: 필터바 + 테이블 + 페이지네이션을 조합한 위젯 (Smart Widget)
  • PostTable: 게시물 목록을 테이블 형태로 보여주는 위젯
  • Layout: Header + Footer를 포함한 레이아웃 위젯

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

  • Smart Widget vs Dumb Widget 선택 기준: 어떤 경우에 props로 받고, 어떤 경우에 직접 hook을 호출해야 하는지 명확한 기준이 더 필요함
  • URL 상태와 컴포넌트 상태의 동기화: 검색어처럼 입력 중인 값과 실제 필터링에 사용되는 값을 분리하는 패턴에 대한 고민
  • 낙관적 업데이트 시 롤백 처리: 현재는 에러 시 롤백만 하지만, 사용자에게 피드백을 주는 방법 고민 필요

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

  • Zod를 활용한 URL 파라미터 검증: 런타임에 타입 안전성을 확보하여 잘못된 URL 값으로 인한 버그 방지
  • FSD 레이어 규칙 준수: entities는 조회만, features는 mutation만 담당하는 명확한 역할 분리

[Widget에서 Hook을 호출해도 되는가?]

FSD 문서에서는 widget이 "완성된 UI 블록"이라고 설명하는데, 이게 순수 UI만 담아야 한다는 의미인지 헷갈렸습니다.

현재 구현에서는 PostListWidget 내부에서 usePostFilters(), usePost() 같은 hooks를 직접 호출하고 있습니다:

// widgets/post-list/ui/PostListWidget.tsx
export const PostListWidget = ({ onPostDetail, onPostEdit, ... }) => {
  const { search, skip, limit, ... } = usePostFilters()  // features hook
  const { posts, total, isLoading } = usePost({ ... })   // entities hook
  // ...
}

이렇게 하면 props가 14개 → 4개로 줄어들어 코드가 깔끔해지지만, widget이 특정 도메인에 결합됩니다.

질문:

  1. Widget 내부에서 features/entities의 hooks를 직접 호출해도 FSD 규칙에 맞는 건가요?
  2. 아니면 page에서 모든 데이터를 props로 내려주는 게 더 권장되는 패턴인가요?
  3. 상황에 따라 선택해야 한다면, 어떤 기준으로 판단하면 좋을까요?

[Widget의 적절한 단위는?]

현재 프로젝트에서 widget으로 분류한 컴포넌트들:

widgets/
├── layout/          # Header + Footer + Outlet 조합
├── post-list/
│   ├── PostListWidget.tsx   # 필터바 + 테이블 + 페이지네이션 (큼)
│   └── PostTable.tsx        # 테이블만 (작음)

고민되는 점:

  • PostListWidget은 필터바 + 테이블 + 페이지네이션을 모두 포함하는데, 이게 너무 큰 단위인가요?
  • PostTable은 테이블 하나인데, 이 정도면 widget이 아니라 entities/post/ui에 있어야 하나요?
  • Layout(Header + Footer)은 widget이 맞나요, 아니면 app 레이어에 있어야 하나요?

질문:

  1. Widget의 적절한 크기/단위를 판단하는 기준이 있나요?
  2. "여러 feature/entity를 조합"해야 widget인가요? 단일 entity의 UI도 widget이 될 수 있나요?
  3. 페이지에서 한 번만 사용되는 큰 UI 블록도 widget으로 분류해야 하나요?

Q. Widget에서 Hook을 호출해도 되는가?

Widget에서 features/entities hooks를 직접 호출하는 현재 방식이 FSD 규칙에 어긋나지 않느 것 같아욥~ FSD에서 widget은 “완성된 UI 블록”으로서 데이터 로직을 내부에 캡슐화해야할 것 같아요. 어찌보면 그냥 prop이 없을 수록 좀 더 이상적인 위젯인것 같습니다. 언제 어디서나 데이터 신경 안쓰고 그냥 툭 붙일 수 있게 말이죠 (예를들면 추천 목록등등)

A. Widget의 적절한 단위는?

PostListWidget은 나쁘지 않은 것 같아요 위젯이라는 레이어가 레이어 중에 래밸이 높기때 괜찮습니다~ 사실 위젯을 판단하는 기준은 레이어의 방향성을 유지하는 수준으로 판단하는 수밖에 없을 것 같아요. 크기나 단위보다도 말이죠. 저는 기능적으로는 독립적으로 사용될 수 있어서 데이터 신경쓰지 않고 어디든 붙일 수 있는 것으로 이해했지만 사실 이것도 잘 모르겠어요. FSD의 모호한 점이라고 생각해요 그러니 각 레이어별 특성은 스스로 정의하기 나름인 것 같고 그저 중요한건 레이어간 의존의 방향성 인 것 같습니다. 그 수준을 맞추다보면 위젯에도 가고 피처로도 가는거죠.

저라면 post 테이블은 entity와 widget 사이의 어딘가의 레이어로 들어갈 것 같아요. entity에 table이라는 슬라이스를 만들어 재사용하거나 그렇게 재사용될 테이블이 아니라면 그냥 entity에 넣을 것 같습니다. 저는 레이어의 특성보다는 의존구조만 생각합니다 :)
