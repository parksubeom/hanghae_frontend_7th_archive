1. 렌더(Render) 단계

렌더 단계에서는 DOM 등에 어떤 변경이 필요한지 결정합니다.

이 단계에서 React는 render를 호출하고, 그 결과를 이전 렌더 결과와 비교합니다.

2. 커밋(Commit) 단계

커밋 단계는 실제로 변경 사항을 적용하는 단계입니다.

(React DOM의 경우 DOM 노드를 삽입, 수정, 제거하는 시점입니다.)

또한 이 단계에서 componentDidMount, componentDidUpdate 같은 라이프사이클 메서드도 호출됩니다.

DevTools의 Profiler는 성능 정보를 커밋 단위로 그룹화합니다.

Profiler 상단에는 커밋을 나타내는 막대 그래프가 표시됩니다.

커밋 막대 그래프

  • 각 막대는 하나의 커밋을 의미합니다.
  • 현재 선택된 커밋은 검은색으로 표시됩니다.
  • 막대를 클릭하거나 좌/우 화살표 버튼으로 다른 커밋을 선택할 수 있습니다.
  • 막대의 색과 높이는 해당 커밋의 렌더링 소요 시간을 나타냅니다.
    • 키가 크고 노란색일수록 오래 걸린 커밋
    • 키가 작고 파란색일수록 빠른 커밋

input이 한글자 바뀔 때마다 전체 SerachDialog가 리렌더되어 불필요한 리렌더링이 발생했습니다. 따라서 SearchInput 컴포넌트를 분리하고, 그 내부에 지역 상태를 두어 유저에게 input 입력 시 바로 변경되어 보이는 값은 해당 상태를 사용하도록 했습니다. 그리고 디바운싱을 300ms 주어 실제 onSearch 함수에 변화가 적용되는 것은 0.3초마다 반영되도록 최적화했습니다.

const SearchInput = memo(
  ({ onSearch }: { onSearch: (query: string) => void }) => {
    const [query, setQuery] = useState<string>('');

    useEffect(() => {
      const timer = setTimeout(() => {
        onSearch(query);
      }, 300);

      return () => clearTimeout(timer);
    }, [query, onSearch]);

    return (
      <FormControl>
        <FormLabel>검색어</FormLabel>
        <Input
          placeholder="과목명 또는 과목코드"
          value={query}
          onChange={(e) => setQuery(e.target.value)}
        />
      </FormControl>
    );
  },
);

// 변경 전

 const getFilteredLectures = useCallback(
    ({ query = '', credits, grades, days, times, majors }: SearchOption) => {
      return lectures
        .filter(
          (lecture) =>
            lecture.title.toLowerCase().includes(query.toLowerCase()) ||
            lecture.id.toLowerCase().includes(query.toLowerCase()),
        )
        .filter(
          (lecture) => grades.length === 0 || grades.includes(lecture.grade),
        )
        .filter(
          (lecture) => majors.length === 0 || majors.includes(lecture.major),
        )
        .filter(
          (lecture) => !credits || lecture.credits.startsWith(String(credits)),
        )
        .filter((lecture) => {
          if (days.length === 0) {
            return true;
          }
          const schedules = lecture.schedule
            ? parseSchedule(lecture.schedule)
            : [];
          return schedules.some((s) => days.includes(s.day));
        })
        .filter((lecture) => {
          if (times.length === 0) {
            return true;
          }
          const schedules = lecture.schedule
            ? parseSchedule(lecture.schedule)
            : [];
          return schedules.some((s) =>
            s.range.some((time) => times.includes(time)),
          );
        });
    },
    [lectures],
  );


// 변경 후
export const filterLectures = (
  lectures: Lecture[],
  { query, credits, grades, days, times, majors }: SearchOption,
): Lecture[] => {
  const combinedFilter = composeFilters<Lecture>(
    filterByQuery(query),
    filterByGrades(grades),
    filterByMajors(majors),
    filterByCredits(credits),
    filterByDays(days),
    filterByTimes(times),
  );

  return lectures.filter(combinedFilter);
};

[Schedule Tables Context API를 제거하고 전역 상태관리 라이브러리 Zustand 사용]

  • 그렇게 선택한 이유 Context API를 scheduleMap 상태와 setSecheduleMap을 위해 사용하고 있었습니다. 그런데 setScheduleMap은 해당 영역 전체에서 필요하다기보다, 데이터를 렌더링 하는 부분에서만 scheduleMap 상태를 , 스케쥴을 수정하는 부분에서만 setSecheduleMap이 필요했습니다. 따라서 이 두 경우가 아닌 부분에서는 해당 상태나 setState의 영향을 받지 않아도 되기 때문에, 리렌더에 영향을 주는 범위를 줄이기 위해 전역상태를 선택했습니다.

[Drag and Drop Context 축소] Drag and Drop 되는 Context가 전체 ScheduleTables 에 포함되어 있었습니다. 하지만 실제로 드래그 앤 드랍의 유효 범위는 각 테이블 내부입니다. 또한 테이블 내에서도 최상단과 좌측 메뉴는 드래그 앤 드랍 범위에 포함되지 않습니다. 따라서 정적인 부분들은 따로 컴포넌트로 분리하여 메모이제이션을 진행하고, 실제 드래그 범위에 해당하는 부분만 Drag and Drop 컨텍스트로 감싸서 사용했습니다.

[Drop 최적화]

  • handleDragEnd함수는 moveSchedule 함수를 호출합니다. 따라서 moveSchedule 함수를 Zustand selector로 함수만 구독하도록 했습니다. 또한 상단에서 설명했던 Drag and Drop Context 축소도 드랍으로 인한 리렌더링을 줄일 수 있습니다.

학습 효과 분석

회사 업무에서 비슷하게 최적화 작업을 한 경험이 있습니다. 컴포넌트 메모이제이션과 useCallback, useMemo 등을 사용해서 최적화를 진행했습니다. 그런데 진행하면서 과연 이렇게 하는게 맞는 건지, 어떻게 더 최적화를 할 수 있는지 잘 모르겠다는 생각이 들었습니다. 그런데 이번 과제를 진행하면서 컴포넌트 메모이제이션이나 가상화 작업을 했던 부분이 아주 틀린 선택이 아니었다는 생각이 들어서 나름 뿌듯했습니다. 다만 그 때도 Profiler를 사용했었는데, profiler에서 막대 그래프 하나하나가 실제 어떤 걸 의미하는지 제대로 모르고 렌더되는 모양만 보고 사용했었습니다. 이번 기회를 통해 Profiler 지표가 무엇을 뜻하는지 더 정확하게 학습할 수 있었습니다.
