Q&A 세션에서 받은 충격

"계층을 나눈다 = 격리가 목적"

처음 리팩토링을 시작할 때 formatPrice라는 함수를 만들었다.

// 내가 만든 첫 번째 버전
export const formatPrice = (price: number, isAdmin: boolean, isSoldOut: boolean) => {
  if (isSoldOut) return 'SOLD OUT';
  if (isAdmin) return `${price.toLocaleString()}원`;
  return `₩${price.toLocaleString()}`;
};

"오, 깔끔하게 한 함수로 처리했네!" 싶었는데, Q&A 때 코치님 얘기를 듣고 머리를 한 대 맞은 것 같았다.

"계층을 나눈다 = 더 좋게 만들려는 게 아니라 '격리'가 목적" "Utils는 비즈니스 로직을 전혀 몰라야 함" "다른 회사 프로젝트에 가져가도 당장 쓸 수 있는 코드"

이 말을 듣고 다시 생각해봤다. isAdmin은 우리 도메인의 비즈니스 규칙이다. 가격을 포맷팅하는 유틸 함수가 "관리자 모드"라는 개념을 알아야 하나? 만약 이 함수를 다른 프로젝트에 가져간다면? "관리자"라는 개념이 없는 프로젝트라면 이 파라미터는 뭐지?

결국 완전히 다시 짰다.

// After: 순수한 포맷팅 함수들
export const formatCurrency = (price: number): string => {
  return `₩${price.toLocaleString()}`;
};

export const formatCurrencyWon = (price: number): string => {
  return `${price.toLocaleString()}원`;
};

이제 이 함수들은 정말 "숫자를 특정 형식의 문자열로 바꾸는" 일만 한다. 비즈니스 로직은 컴포넌트에서 처리한다.

// 컴포넌트에서
{isSoldOut ? 'SOLD OUT' : formatCurrency(price)}
{isAdmin ? formatCurrencyWon(price) : formatCurrency(price)}

이렇게 바꾸고 나니 테스트가 Basic 21/21, Advanced 21/21 모두 통과했다. 더 중요한 건, 코드가 훨씬 명확해졌다는 거다. 함수를 보면 정확히 뭘 하는지 바로 알 수 있다.

1124줄의 App.tsx를 146줄로 만들기

처음의 막막함

App.tsx를 열었을 때 정말 압도당했다. 상품 목록 관리, 장바구니 로직, 쿠폰 할인 계산, 알림 표시... 모든 게 하나의 파일에 들어가 있었다. "이걸 어디서부터 어떻게 나눠야 하지?" 하는 생각만 맴돌았다.

일단 useState가 몇 개나 있는지 세어봤다. products, cart, coupons, selectedCoupon, isAdmin, notifications... 10개가 넘었다. 그리고 각 상태를 다루는 핸들러 함수들이 그 밑에 주르륵 나열되어 있었다.

Custom Hook으로 나누기 시작

먼저 가장 명확해 보이는 장바구니 로직부터 빼내기로 했다. useCart라는 Hook을 만들고, 장바구니 관련 상태와 함수들을 옮겼다.

// hooks/useCart.ts
export const useCart = () => {
  const [cart, setCart] = useState<CartItem[]>([]);

  const addToCart = (product: ProductWithUI) => {
    // 장바구니 추가 로직
  };

  const updateQuantity = (productId: string, quantity: number) => {
    // 수량 업데이트 로직
  };

  return { cart, addToCart, updateQuantity, removeFromCart, clearCart };
};

이걸 만들고 App.tsx로 돌아와서 보니, 벌써 100줄 정도가 사라진 것 같았다. "오, 이거 되네?" 하면서 신이 나서 useProduct, useCoupon, useNotification을 차례로 만들었다.

최종적으로 App.tsx는 1124줄에서 146줄로 줄어들었다. 스크롤을 한 번만 해도 전체 구조가 한눈에 들어왔다. 이게 바로 "관심사의 분리"구나 싶었다.

순수함수의 발견

Hook을 만들다 보니, 계산 로직들이 눈에 띄기 시작했다. 할인율 계산, 장바구니 총액 계산... 이런 것들이 Hook 안에 섞여 있는데, 뭔가 어색했다.

Q&A에서 배운 대로, "계산"과 "상태 변경"을 분리하기로 했다. cartCalculator.ts를 만들어서 순수 계산 함수들만 모았다.

// utils/cartCalculator.ts
export const calculateItemTotal = (
  item: CartItem,
  cart: CartItem[]
): number => {
  const { product, quantity } = item;
  const discount = getMaxApplicableDiscount(item, cart);
  return product.price * quantity * (1 - discount);
};

export const calculateCartTotal = (
  cart: CartItem[],
  selectedCoupon: Coupon | null
) => {
  // 순수 계산만
};

이렇게 빼놓으니 정말 좋았다. 함수만 보면 뭘 하는지 명확하고, 테스트하기도 쉬웠다. 무엇보다 Hook이 훨씬 간결해졌다. Hook은 "상태를 관리"하는 데 집중하고, 계산은 순수함수에게 맡기니까 각자의 역할이 명확해졌다.

순수함수의 힘을 체감하다

calculateItemTotal 같은 함수를 만들면서 순수함수의 가치를 정말 체감했다. 외부 상태에 의존하지 않으니까, 함수만 따로 떼어서 테스트할 수 있었다.

// 테스트하기 쉬움
expect(calculateItemTotal(mockItem, mockCart)).toBe(9000);

그리고 디버깅할 때도 편했다. 계산 결과가 이상하면 그 함수만 보면 됐다. 외부 상태가 언제 어떻게 바뀌는지 추적할 필요가 없었다. "같은 입력 → 같은 출력"이 보장되니까, 머릿속으로 흐름을 따라가기도 훨씬 쉬웠다.

계층 구조가 보이기 시작했다

코드를 정리하다 보니, 자연스럽게 계층이 생겼다.

utils (순수 함수, 엔티티 계산)
  ↓
hooks (상태 관리, 액션)
  ↓
components (UI 렌더링)
  ↓
App (조립)

utils는 hooks를 모르고, hooks는 components를 모른다. 위에서 아래로만 의존성이 흐른다. Q&A에서 배운 "단방향 의존성"이 이거구나 싶었다. 덕분에 utils를 수정해도 components는 건드릴 필요가 없었다.

Jotai를 만나다 (심화 과제)

Props Drilling의 지옥

Basic 버전을 만들면서 정말 답답했던 게, props를 계속 물려줘야 한다는 거였다.

// App.tsx에서
<CartPage
  products={products}
  cart={cart}
  coupons={coupons}
  selectedCoupon={selectedCoupon}
  onAddToCart={handleAddToCart}
  onUpdateQuantity={handleUpdateQuantity}
  onRemoveFromCart={handleRemoveFromCart}
  onApplyCoupon={handleApplyCoupon}
  onClearCoupon={handleClearCoupon}
  onCompleteOrder={handleCompleteOrder}
  // ... 계속...
/>

props가 10개가 넘어가니까 타이핑하는 것만으로도 피곤했다. 그리고 CartPage에서 또 ProductCard에게 props를 물려주고... "이게 맞나?" 싶었다.

Jotai로의 전환

Advanced 버전에서 Jotai를 도입했다. 처음엔 "상태 관리 라이브러리 하나 더 배워야 하나..." 싶었는데, 막상 써보니 정말 간단했다.

// atoms.ts에서 선언하고
export const cartAtom = atomWithStorage('cart', []);
export const productsAtom = atomWithStorage('products', initialProducts);

// 컴포넌트에서 바로 쓰면 됨
const cart = useAtomValue(cartAtom);
const { addToCart } = useCartActions();

App.tsx가 정말 깔끔해졌다.

// 이게 전부!
export const App = () => {
  return (
    <div>
      <CartPage />
    </div>
  );
};

props를 물려줄 필요가 없었다. 각 컴포넌트가 필요한 상태를 atoms에서 직접 가져왔다. 이게 바로 "Props Drilling 제거"구나 싶었다.

테스트에서 만난 함정

그런데 테스트를 돌렸더니 이상한 현상이 생겼다. 한 테스트가 성공하면 다음 테스트가 실패하고, 순서를 바꾸면 또 다른 테스트가 실패했다. 알고 보니 Jotai atoms가 테스트 간에 공유되고 있었다. 첫 번째 테스트에서 장바구니에 상품을 추가하면, 그 상태가 다음 테스트에도 남아있는 거였다.

해결책은 Provider로 store를 격리하는 것이었다.

// 각 테스트마다 새로운 Provider
const { getByText } = render(
  <Provider>
    <App />
  </Provider>
);

이 경험을 통해 배운 건, 상태 관리 라이브러리를 쓸 때는 테스트 환경까지 고려해야 한다는 거다.

과제를 하면서 내가 알게된 점, 좋았던 점은 무엇인가요?

1. 계층 분리의 핵심은 "좋아 보이기"가 아니라 "격리"

  • 코드 구조를 나누는 목적은 변경 영향을 최소화하는 것임을 깨달았다.
  • 비즈니스 로직과 공통 함수가 분리되니 유지보수가 훨씬 쉬워졌다.

2. 좋은 Utils는 프로젝트 독립적이어야 한다

  • Utils 함수는 특정 서비스 규칙이나 문맥에 의존하면 안 된다.
  • 재사용 가능한 유틸리티로 분리하면서 함수의 역할과 범위를 더 명확히 이해하게 됐다.

3. 순수함수는 테스트·디버깅·이해를 쉽게 만든다

  • 외부 상태 없이 입력→출력만 의존하는 구조가 안정성과 예측 가능성을 높여줌을 경험했다.

4. Props Drilling의 불편함을 체감

  • Jotai 같은 상태 관리 도구를 사용하면서 props를 계속 전달하는 방식의 한계를 명확하게 느꼈다.
  • 컴포넌트 구조가 더 단순해지고 재사용성이 높아졌다.

5. 테스트 환경에서도 상태 격리가 필요

  • 전역 상태 공유로 테스트가 간섭하는 문제를 겪으며, 상태 관리 라이브러리를 사용할 때 테스트 설계까지 고려해야 한다는 점을 배웠다.

이번 과제에서 내가 제일 신경 쓴 부분은 무엇인가요?

1. Utils의 순수성 유지 Q&A를 보고 깨달았는데, formatPrice에 isAdmin 파라미터를 넣은 게 잘못된 설계였다. 처음엔 "한 함수에서 처리하면 편하지 않을까?"라고 생각했는데, 그게 아니라 "이 함수가 비즈니스 로직을 알아야 하는가?"를 먼저 물어야 했다. 수정하면서 느낀 건, formatCurrency(price)는 정말 "숫자를 문자열로 변환"만 하니까 어디든 가져다 쓸 수 있다는 것.

2. Hook의 책임 분리 useCart에 모든 장바구니 로직을 때려박고 싶은 유혹이 있었는데, 순수한 계산 로직은 무조건 utils로 빼려고 노력했다. 덕분에 useCart는 상태 관리만, cartCalculator는 계산만 하게 됐다.

3. Entity vs UI 분리 ProductCard가 Product 엔티티를 다루는 컴포넌트인지, 아니면 순수 UI 컴포넌트인지 계속 고민했다. 결국 엔티티를 직접 다루는 쪽으로 결정했는데, 이 기준이 명확하니까 다른 컴포넌트들도 쉽게 분류할 수 있었다.

4. Jotai atoms 설계 어떤 상태를 atom으로 만들고, 어떤 상태를 로컬로 유지할지 고민을 많이 했다. searchTerm은 전역으로, debouncedSearchTerm은 컴포넌트 로컬로 두는 식으로 기준을 세웠다.

// 현재: utils에 두 함수
formatCurrency(price)  // ₩1,000
formatCurrencyWon(price)  // 1,000원

// 대안 1: helper로
const useFormatProduct = () => {
  return (price, isAdmin) => isAdmin ? formatCurrencyWon(price) : formatCurrency(price)
}

// 대안 2: 컴포넌트에서 직접
{isAdmin ? formatCurrencyWon(price) : formatCurrency(price)}
