[5팀 박수범] Chapter 1-3. 완성도있는 테스트 전략 수립하기

by parksubeom2025.11.01

과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리
♨️
이 과정에서 컴포넌트 리팩토링이 필요할 수 있습니다!
여유가 있다면 컴포넌트를 아름답게 분리하는 고민을 해주시면 좋지만,
당장은 적절한 스토리를 작성하는데 더 집중해주세요.
적절한 추상화, 응집도를 높이고 결합도를 낮추는
우아한 설계는 이후 주차때 좀 더 집중해봐요.

- 물론, AI를 활용해도 좋습니다.
- 기존에 작성한 테스트는 깨지지 않도록 해주세요.
  - 이 과정에서 테스트가 분리될 수도 작성법이 수정될 수도 있습니다.
  - 정말 불필요한 케이스가 아니면, 테스트 케이스를 삭제하는건 최소화 해주세요.
  - 추가로 필요한 테스트가 생길 수 있지만, 작성은 자유롭게 선택해서 진행해주세요.

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

테스트 전략 논의 피그잼

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요. 제가 제안한 초기 전략은 리스크 기반의 계층형 테스트였습니다. 이미 150개의 단위/통합 테스트가 존재하는 프로젝트에 신규 기능을 추가하는 상황이었기 때문에, 모든 것을 유닛 테스트부터 다시 작성하는 정석적인 TDD 방식보다는 이것이 ROI가 높다고 판단했습니다. 전략의 핵심은 "기존 기능이 깨지지 않았는지(회귀)를 먼저 확인하고, 기존 테스트가 커버하지 못하는 영역인 E2E 테스트와 시각적 회귀 테스트(VRT)를 추가하여 신뢰성을 높이는 것"이었습니다.

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요. 팀원들과의 합의를 통해, 저희는 E2E/VRT의 비중을 줄이고 통합 테스트(Integration Test)에 집중하는 '테스팅 트로피(Testing Trophy)' 모델을 최종 전략으로 채택했습니다.

팀원들은 E2E 테스트를 "API 모킹(Mocking) 없이 실제 백엔드와 연동하는 테스트"로 정의했습니다. 하지만 현재 저희 프론트엔드 팀이 구현 가능한 유일한 API 검증 방식은 MSW(Mock Service Worker)를 사용하는 것이었습니다.

저희는 MSW를 사용하는 방식은 '리얼 API' 기준의 E2E 테스트가 아닌 **'강화된 통합 테스트'**의 영역에 속한다고 보았습니다.

따라서 Mock API(MSW)에 의존해야 하는 현 상황에서 E2E 테스트의 비중을 높이는 것은 부적합하며, 대신 MSW를 활용한 기존 150개의 통합 테스트를 중심으로 신규 비즈니스 로직 전반을 검증하는 것이 더 현실적이고 효율적인 전략이라고 결론 내렸습니다.

  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

  • it('월별 뷰에서 빈 셀(날짜가 없는 셀)을 클릭하면 날짜가 자동 입력되지 않는다'):

    • Given: 현재 날짜가 2025-10-01이고 월별 뷰가 표시된 상태. 폼의 날짜 필드는 '2025-10-15'로 채워져 있음.
    • When: 캘린더의 이전 달 날짜로 인한 '빈 셀'(예: 9월 29일)을 클릭함.
    • Then: 폼의 날짜 필드 값은 '2025-10-15'로 유지되어야 하며, 다른 폼 필드 값도 변경되지 않아야 함.

  • it('기존 일정 수정 중에 날짜 셀을 클릭하면 수정 폼의 날짜가 변경된다'):

    • Given: 사용자가 특정 일정의 '수정 아이콘'을 클릭하여 폼이 열리고, 수정 모드가 활성화됨.
    • When: 폼이 열린 상태에서 캘린더의 다른 날짜 셀(예: 20일)을 클릭함 (Click-to-Create 동작).
    • Then: 폼의 날짜 입력 필드 값이 '2025-10-20'으로 즉시 변경되어야 함 (수정 대상이 아닌 새 일정 생성 모드로 전환).
    • And: '일정 수정'이 아닌 '일정 추가' 버튼을 클릭 시 해당 날짜로 새 일정이 생성되고 스낵바가 노출되어야 함.

과제 셀프회고

날짜: 2025년 11월 2일 (일)

작업 내용

오늘은 App.tsx 파일의 수동 리팩토링을 진행했습니다.

App.tsx에 인라인으로 구현된 모든 UI 로직(폼, 캘린더 뷰, 이벤트 목록, 모달)을 역할별 컴포넌트로 분리하는 작업을 수행했습니다.

  • App.tsx (기존) $\rightarrow$
    1. src/components/EventFormPanel.tsx (폼 UI 분리)
    2. src/components/CalendarView.tsx (월간/주간 뷰 렌더링 로직 분리)
    3. src/components/EventListPanel.tsx (우측 이벤트 목록 분리)
    4. src/components/AppModals.tsx (모든 Dialog/Modal 로직 분리)
    5. src/App.tsx (이 컴포넌트들을 조립하고 훅을 호출하는 최상위 컨테이너로 변경)

작업한 이유

다음 단계인 E2E (End-to-End) 테스트와 시각적 회귀 테스트를 스토리북과 플레이라이트로 진행하기 위한 필수 선행 작업이었습니다.

스토리북의 핵심은 각 UI 컴포넌트(EventFormModal 등)를 독립적/고립된 환경에서 렌더링하고 테스트하는 것입니다. 하지만 이전 App.tsx 파일처럼 모든 로직이 하나의 파일에 인라인으로 결합되어 있으면, 컴포넌트별 분리가 불가능하여 E2E 테스트가 극도로 어려워집니다.

AI 에이전트가 E2E 테스트를 원활하게 수행할 수 있도록,먼저 테스트 가능한 환경을 구축하는게 처음 과제라고 생각했습니다. 또한 컴포넌트를 분리해가면서, 기존의 테스트가 망가지지 않았는지 수시로 확인 후 작업을 진행했는데 다행히 기존 테스트는 수정없이 잘 분리할 수 있었습니다.

느낀점

에이전트를 사용하지 않고 직접 코드를 분리하니, 오히려 AI가 작업할 환경(스토리북)을 제가 정리해주는 기분이었습니다. 결국 AI는 도구일 뿐,아키텍처를 설계하고 결정하는 것은 개발자의 몫임을 다시 한번 느꼈습니다.

날짜: 2025년 11월 3일 (월)

작업 내용

  1. ESLint 설정 오류 수정: TDD 파이프라인 실행 전, src/__mocks__/handlers.ts 파일에서 발생하던 import/no-unresolved (모듈 경로 해석 불가) 오류를 수정했습니다.
  2. 테스트 전략 수립: 어제(11/2) App.tsx를 리팩토링하여 컴포넌트를 분리한 것을 기반으로, 새로 구현할 D&D 및 Click-to-Create 기능에 대한 상세한 테스트 전략을 4순위(회귀 $\rightarrow$ 핵심 E2E $\rightarrow$ 엣지 케이스 E2E $\rightarrow$ VRT)로 나누어 팀원들과 공유했습니다.

작업한 이유

  • ESLint 오류는 eslint-plugin-import가 .ts 확장자를 인식하지 못해 발생한 문제로, TDD 파이프라인의 3단계(테스트 코드 작성) 이상 진행을 방해했기 때문에 최우선으로 해결했습니다.
  • 새로운 D&D 기능, 특히 '반복 일정 D&D'는 여러 훅(useEventOperations, useCalendarView)과 모달 로직이 복잡하게 얽혀있습니다. 이미 제공받은 프로젝트 환경에서는 이전 과제까지의 스펙을 모두 충족하고 테스트도 모두 통과하는 상태이기 때문에 이 상태를 유지하면서 새로운 기능을 붙이는게 먼저라고 생각했습니다. 회귀테스트를 통해 되던것이 안되는지 검증하는것이 최우선이라고 판단했습니다.

느낀점

  • ESLint 설정 같은 개발 환경(Environment)까지 신경쓰지 못했었는데 항상 Lint 오류를 보고 red 에러가 아니라서 넘어갔던게 많이 아쉬웠습니다. 간결한 코드, 군더더기 없는 코드를 위해 신경써서 작업해야겠습니다.
  • 테스트 전략을 1~4순위로 정리하면서, 프로젝트에 따라, 구현하는 기능에 따라 테스트의 우선순위는 유연하게 바뀐다는 걸 느꼈습니다.
  • 좋은 테스트는 여러 테스트를 다 적용하는게 아닌 필요한 테스트를 적절한 비용으로 하는것이 좋은 테스트 전략이라고 생각하게 됐습니다.

날짜: 2025년 11월 5일 (수)

작업 내용

  • 신규 기능(D&D, Click-to-Create) 테스트 전략 수립: E2E 테스트와 시각적 회귀 테스트(VRT)를 도입하기 위한 구체적인 테스트 우선순위를 정의하고 문서화했습니다.
  • 테스트 전략 순서 확정:
    1. 1순위: 회귀 테스트 (기존 기능 안정성 확보)
    2. 2순위: 시각적 회귀 테스트 (VRT - UI 피드백 검증)
    3. 3순위: E2E 테스트 (핵심 요구사항 검증)
    4. 4순위: E2E 테스트 (복합 엣지 케이스 검증 - 예: 반복 일정 D&D)
    5. 5순위: 부족한 통합테스트 보강

작업한 이유

  • D&D 기능은 Update, 날짜 클릭 생성 기능은 Create 로직에 직접적인 변화를 줍니다. 이로 인해 기존 CRUD 기능이 망가질 수 있으므로, 리스크 기반 관점에서 "기존 핵심 기능이 깨지지 않는 것을 1순위로 검증해야 했습니다.
  • 또한, D&D 기능은 반복 일정이나 겹침 알림과 엮일 때 복잡한 엣지 케이스가 발생합니다. 사용자 경험(UX) 신뢰성을 확보하기 위해, 이 복합 시나리오(3순위)를 E2E 테스트로 명확히 정의할 필요가 있었습니다.

느낀점

  • "리스크 기반 우선순위" (회귀)를 먼저 정의하면서, AI에게 구현을 맡기기 전에 제가 어떤 안전망을 먼저 구축해야 하는지에 대한 고민을 하게됐습니다.

날짜: 2025년 11월 6일 (목)

작업 내용

  • E2E 테스트 5대 핵심 워크플로우 코드 작성:
      1. 기본 일정 관리 (CRUD) 전반 검증
      1. 반복 일정 관리 전반 검증
      1. 일정 겹침 처리 방식 검증
      1. 알림 시스템 노출 조건 검증
      1. 검색 및 필터링 전반 검증
  • 테스트 전략 재조정 및 합의: 팀 토론을 통해, 시각적 회귀 테스트(VRT)의 범위를 축소하고 대신 통합 테스트(Integration Test)의 커버리지를 늘리기로 결정.

작업한 이유

  • E2E 테스트 5종은 프로젝트의 핵심 기능(기본 과제) 전반이 사용자의 실제 시나리오에서 올바르게 동작하는지 확인하기 위한 필수 안전망이었습니다.
  • VRT(시각적 회귀 테스트)의 유지보수 비용 대비 효용성을 팀원들과 재평가했습니다. 현 단계에서는 UI의 시각적 일관성보다, 데이터 흐름과 컴포N 넌트 간 상호작용을 검증하는 통합 테스트가 프로젝트 안정성에 더 중요하다고 판단했습니다.

느낀점

  • E2E 테스트는 생각보다 훨씬 어려웠습니다. 머릿속으로 그린 시나리오대로 테스트 코드가 한 번에 동작해주지 않았고, 수많은 수정과 반복이 필요했습니다.
  • 특히 로케이터(Selector) 문제가 가장 컸습니다. 동적으로 렌더링되는 목록이나 모달창 때문에 Strict Mode 위반 에러를 자주 만났는데, 이는 켄트 벡이 말한 **'취약한 테스트(Fragile Tests)'**가 왜 좋지 않은지 직접 체감하는 계기가 되었습니다.
  • '사용자 관점'으로 테스트를 짜는 것과, 실제 자동화된 테스트가 그 관점을 안정적으로 흉내 내게(예: 로케이터 체이닝) 만드는 것 사이에는 기술적인 갭이 크다는 것을 느꼈습니다.

기술적 성장

이번 과제를 통해 TDD 에이전트를 구축하고 E2E 테스트 전략을 수립하면서 다음과 같은 성장을 이룰 수 있었습니다.

  1. E2E 테스트의 필요성 이해:
    • 기존 150개의 통합 테스트가 통과하더라도, D&D 기능은 실제 브라우저 환경에서 CSS z-index 충돌이나 DndProvider 초기화 누락 같은 문제로 실패할 수 있음을 파악했습니다. jsdom 환경의 통합 테스트가 아닌, Playwright를 통한 E2E 테스트가 왜 필수적인지 명확히 이해하게 되었습니다.
  2. 안정적인 테스트 코드 작성:
    • nth(3)나 filter({ hasText: ... }) 같은 불안정한 선택자가 E2E 테스트를 얼마나 쉽게 깨뜨리는지(Fragile) 경험했습니다.
    • 이를 해결하기 위해 **data-testid**를 도입하고, **getByRole (접근성 기반)**을 우선 사용하는 것이 테스트 안정성을 극대화한다는 것을 배웠습니다.

코드 품질

  1. 관심사 분리 (SoC) 리팩토링:
    • 문제점: App.tsx 파일 하나에 모든 UI 로직(폼, 모달, 캘린더 뷰, 목록)을 인라인으로 구현되어 있었으나, 이는 Storybook 및 E2E 테스트를 매우 힘들게 만들었습니다.
    • 개선: E2E 테스트 준비 단계에서 App.tsx를 수동으로 리팩토링하여 4개의 독립적인 컴포넌트(EventFormPanel, CalendarView, EventListPanel, AppModals)로 분리했습니다.
    • 결과: App.tsx는 훅을 호출하고 상태를 전달하는 순수 컨테이너 역할만 맡게 되어, 코드의 가독성과 테스트 용이성이 크게 향상되었습니다.

학습 효과 분석

오늘 팀원들과 처음으로 테스트 전략에 대해 깊게 머리를 맞대고 토론하는 시간을 가졌습니다.

각자 마주한 상황과 스펙에 따라, 어떤 테스트가 더 효율적이고 비용 대비 강력한 효과를 얻을 수 있는지 판단해야 했습니다. 이 과정에서 각자의 경험을 바탕으로 서로 다른 기준을 세우고 의견을 나누는 것 자체가 매우 의미 있는 시간이었습니다.

특히 저에게 가장 큰 고민은 **"이 기능은 통합 테스트로 충분할까? 아니면 E2E 테스트가 반드시 필요한가?"**라는 경계를 판단하는 것이었습니다.

D&D(드래그 앤 드롭)와 같이 복잡한 상호작용이 얽힌 기능을 보며, 저는 이 토론을 통해 다음과 같은 저만의 명확한 기준을 세울 수 있었습니다.

1. 유닛/통합 테스트의 책임 (Logic Verification)

  • 언제 사용하는가? $\rightarrow$ 새로운 훅(Hook)이나 유틸리티 함수, 비즈니스 로직을 처음 개발할 때 사용합니다.
  • 목적: TDD의 1단계 성숙도에 해당하며, renderHook이나 fireEvent를 통해 해당 모듈이 명세서대로 '정확한 데이터를 반환하는지', '올바른 API를 호출하는지' 등 로직의 계약(Contract) 자체를 검증하는 데 집중합니다.
  • 예시: useEventOperations 훅이 '단일 수정' 시 POST와 PUT API를 순차적으로 호출하는지 검증하는 것.

2. E2E 테스트의 책임 (Workflow Verification)

  • 언제 사용하는가? $\rightarrow$ 위 1단계의 통합 테스트가 이미 존재하여 개별 훅과 유틸리티의 기능이 검증되었을 때 사용합니다.
  • 목적: TDD의 2단계 성숙도에 해당하며, 이미 검증된 훅들이 **실제 UI(예: App.tsx)와 올바르게 '연결(Wiring)'**되어, **사용자의 실제 행동(Action)**에 반응하는지 검증합니다.
  • 예시: useEventOperations 훅이 검증됐다는 것을 알아도, "사용자가 캘린더의 이벤트를 드래그하여 CalendarView의 다른 셀에 드롭했을 때, 정말로 화면에서 이동되는가?**"를 검증합니다.

결론

결론적으로, E2E 테스트는 "개별 로직의 합이 사용자의 워크플로우가 되는가"를 검증하는 테스트입니다.

새로운 훅을 만든다면 유닛/통합 테스트를 먼저 추가하겠지만, 이번 D&D 기능처럼 기존에 검증된훅 등)을 조합하여 **새로운 '동작'**을 만드는 경우에는, E2E 테스트를 도입하는 것이 중복을 피하고 사용자 경험의 신뢰성을 확보하는 가장 좋은 방법이라는 확신을 얻었습니다.

과제 피드백

이번 과제에서 가장 좋았던 점은 **스토리북(Storybook)**을 실제로 도입하고 사용해 볼 기회가 생겼다는 것입니다. 프론트엔드 개발을 하면서 컴포넌트 기반 개발(CBD)과 시각적 테스트의 중요성에 대해 많이 들어왔는데, CalendarDayCell.tsx나 EventFormModal.tsx 같은 컴포넌트를 고립된 환경에서 렌더링하고 VRT(시각적 회귀 테스트)의 기반을 마련하는 과정이 매우 흥미로웠습니다.

또한, E2E(End-to-End) 테스트 시나리오를 작성하는 것도 재미있는 경험이었습니다.

단순히 renderHook으로 훅의 반환 값만 검증하는 통합 테스트와 달리, E2E 테스트는 **"사용자가 날짜 셀을 클릭 $\rightarrow$ 폼 모달이 열림 $\rightarrow$ 날짜가 채워짐 $\rightarrow$ 저장 버튼 클릭 $\rightarrow$ 폼이 리셋됨"**과 같이 **실제 사용자의 전체 워크플로우(User Flow)**를 코드로 재현해 볼 수 있었습니다.

특히 D&D 기능처럼 복잡한 상호작용은 E2E 테스트가 왜 필수적인지 체감할 수 있었습니다.

리뷰 받고 싶은 내용

테스트 코드를 처음 접했을 때는, 솔직히 어디서부터 시작해야 할지 막막하고 어렵게만 느껴졌습니다. 3주라는 시간 동안 이 과제와 계속 부대껴보니, 제가 테스트 코드의 모든 것을 100% 안다고 자신 있게 말하기는 어렵습니다.

하지만 적어도 "테스트를 어떻게 도입해야 하는가"에 대한 저만의 관점과 기준이 생긴 것 같아, 이번 3주는 저에게 매우 유의미한 시간이었습니다.

TDD가 매우 훌륭하고 강력한 개발론이라는 것을 체감하면서도, 동시에 이 세상에 '실버 불릿(Silver Bullet)'은 없다는 것도 다시 한번 깨달았습니다. 특히 D&D 기능처럼 UI 상호작용이 강한 기능을 구현할 때는, 유닛 테스트만 고집하는 것이 오히려 비효율적일 수 있음을 배웠습니다.

이번 챕터를 통해 테스트 코드 문법 자체는 아직 익숙해지는 과정에 있지만, 그보다 더 중요한 것은 '명확한 명세서'를 기반으로 개발을 지향하는 태도라는 것을 확실히 깨달았습니다.

3주 동안 TDD 파이프라인 구축과 기능 구현 사이에서 고군분투하며, 결과적으로는 (UI 통합 실패 등) 반쪽짜리 과제를 제출하게 되어 아쉬움이 많이 남습니다. 하지만 이 실패의 경험을 통해 AI 에이전트의 한계와 가능성, 그리고 견고한 테스트 전략의 중요성까지 정말 많은 것을 배웠습니다. 감사합니다.
