HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

// ! 이벤트는 생성, 수정 되면 fetch를 다시 해 상태를 업데이트 합니다. 이를 위한 제어가 필요할 것 같은데요. 어떻게 작성해야 테스트가 병렬로 돌아도 안정적이게 동작할까요?//

테스트 케이스가 병렬로 실행될 때, 전역 setupServer 인스턴스에 server.use()된 핸들러들이 동일한 이벤트 데이터 소스를 참조하면 데이터 경합이나 상태 누수가 발생할 수 있습니다.

다양한 방법이 있겠지만, 저는 병렬 실행 자체를 제한하기 보단, 테스트 케이스 간에 참조하는 데이터 컨텍스트를 완전히 분리하는 방향으로 해결해야 한다고 생각했습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

각 테스트가 독립적으로 사용할 수 있는 인메모리 데이터베이스와 전용 핸들러 셋을 반환하는 팩토리 함수 createMockEventApiInstance를 구현했습니다.

이 구조는,

  • 각 테스트마다 독립된 데이터 컨텍스트를 생성하여, 이벤트 생성/수정 시 상태가 전역적으로 섞이지 않음
  • POST, PUT, DELETE 요청이 인메모리 DB에 직접 반영되고, GET 요청은 항상 최신 상태를 반환하도록 구성
  • 데이터를 변형하는 테스트 / seed 데이터가 다른 테스트 / 전역 handler를 공유하는 테스트 모두 일관된 패턴으로 관리 가능 이들을 보장합니다.

추가로, 전역 afterEach에서 server.resetHandlers()를 호출하여, 테스트 간 등록된 핸들러가 겹치는 병목 현상을 방지합니다.

🫢 다만, 현재 구조에서는 테스트 실패 시 afterEach가 실행되지 않으면 resetHandlers가 호출되지 않는 경우가 있을 수 있습니다. 이 경우, 특정 테스트가 등록한 핸들러가 이후 테스트에 잔존해 예상치 못한 응답을 반환할 가능성이 있습니다.

이를 통해, 테스트 케이스 간에는 전역 서버를 공유하되, 각 테스트는 자신만의 인메모리 데이터베이스와 핸들러는 가지는 구조를 통해 병렬 실행 환경에서도 데이터 경합이나 상태 오염 없이 안정적으로 동작하도록 설계했습니다.

// 구현
function createMockEventApiInstance("초기 시드 데이터": Event[] = []) {
  const "인메모리 디비" = { events: [..."초기 시드 데이터"] };

  const "핸들러 셋" = [ "인메모리 DB를 참조하는 핸들러들" ];

  return {
    db: "인메모리 디비",
    handlers: "핸들러 셋",
    ...
  };
}

// 사용 패턴

// 1. 전역 SetupTest

const { handlers } = createMockEventApiInstance("디폴트 Mock Events");

export const server = setupServer(...handlers);
...
afterEach(() => {
  server.resetHandlers();
  ...
});
...

// 2. 테스트 케이스
it('생성 후 목록에 반영된다', async () => {
  const { handlers } = createMockEventApiInstance(SEED_DATA);
  server.use(...handlers);

  // ③ render → 사용자 액션(POST/PUT/DELETE) → 내부적으로 fetch(GET)
  // ④ expect: 최신 DB 상태가 반영된 UI/상태 확인
});

"+ 초기에 고민했던 부분" ‘테스트가 병렬로 돌아가는 상황’을 어떻게 해석할지 고민했습니다. 물리적으로 여러 테스트가 동시에 실행되는 상황도 있겠지만, 저는 테스트가 순차적으로 실행되더라도 동일한 데이터나 핸들러 셋을 공유하는 논리적 병렬 상황에 집중했습니다.

실제 병렬 실행까지 고려하면 setupServer 인스턴스를 테스트마다 분리해야 해서 구조적으로 오버엔지니어링에 가까워졌습니다. 그래서 저는 테스트 실행 단위별로 인메모리 데이터베이스와 개별 핸들러 셋을 생성하는 방식으로 격리 환경을 구현했습니다.

기술적 성장

새로 학습한 개념

테스트 코드에 대한 지식이 거의 전무 했습니다. 방법론과 이론 공부는 야금야금 해왔지만, 밀도있게 작성해본 경험은 없었습니다. 이번에 밀도있게 테스트 코드를 작성해보며, 실제로 테스트 코드 구현 과정에서 겪을 수 있는 경험들을 했습니다. 테스트 코드 운영환경에서 생기는 다양한 이슈들과 그것들을 해결하기 위한 구조화 경험, 그리고 테스트 코드가 돌아가는 원리와 환경에 대한 개념을 학습할 수 있었습니다. 그리고 테스트 코드 관점에서 웹 접근성을 고려해야 하는 이유도 확실히 느꼈습니다. 단순히 UI검증이 아니라 사용자 경험 전체를 검증하는 과정이라고 느꼈습니다.

기존 지식의 재발견/심화

이론으로만 접하던 테스트 코드 작성의 효용들을 직접 체감할 수 있었습니다. 현재 실무에서도 MSW를 세팅해서 사용하고는 있지만, 테스트 코드보다는 데이터 모킹을 위해서 사용하고 있었습니다. 이번에 내부 구조와 원리를 파헤치면서 "왜 이렇게 설계되어 있고, 어떤 순서로 동작하는지"를 이해하게 되었스빈다. 단순히 동작하게 하는 거싱 아니라, 그 안에서 디버깅하고, 원리를 잡아가는 경험이었습니다.

구현 과정에서의 기술적 도전과 해결

handlerUtils 쪽 문제를 해결하면서 구조적/기술적으로 고민할 지점이 많았습니다. 각 테스트 케이스가 어떤 환경을 공유하는지, 데이터가 어디까지 독립적으로 관리되어야 하는지, 전역 인스턴스가 병목을 만들지 않으려면 어떻게 해야하는지 등의 고민을 했습니다. 이를 해결하며 문제 해결을 위해 구조와 원리에 대한 이해가 중요하다는 것을 느꼈던 것 같습니다.

  • 테스트 코드나 자동화된 E2E 시나리오를 작성할 때, 내부 구조를 알 수 없는 외부 라이브러리(MUI 등)의 컴포넌트를 Query하기 위해 data-testid를 사용하는 것은 정당할까요?
  • 반대로, 단순히 특정 UI 섹션(Form, Calendar, EventList 등)을 쉽게 Query하기 위해 data-testid를 부여하는 것은 남용일까요?
  • 즉, 테스트 편의성을 위해 data-testid를 구조적으로 붙이는 것이 어디까지 허용될까요?
  1. 테스트 코드(TC)와 서비스 코드 간의 관계에서, TC가 서비스 코드 구조에 영향을 받는 것은 자연스럽지만, 반대로 서비스 코드가 TC를 위해 data-testid나 특정 aria-* 속성을 추가해야 하거나, <Select/> 의 옵션에 대한 식별을 위해 option-${value} 같은 형태의 label/testid 를 강제해야한다면, 이는 테스트 주도(TC → 코드)가 지나치게 코드 구조에 간섭하는 것이 아닐까요? 다시 말해, “접근성 속성(aria-label 등)”이 테스트 용도로 강제되거나, “테스트 전용 식별자(data-testid)”가 실제 서비스 코드의 구조에 영향을 주는 상황은 어디까지 허용될 수 있을까요?
