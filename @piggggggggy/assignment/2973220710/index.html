1. [제출] 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

실제로 현업에서 제품 혹은 팀 내 테스트 코드의 운용에 대해 고민을 많이 했었습니다. 그렇다보니, 이번 과제에서도 현재 직면한 상황을 바탕으로 최적의 테스트 전략은 무엇일지? 고민해보게 되었습니다.

저는 "캘린더 앱을 실제 상용 제품으로 런칭한다."는 상황을 가정했습니다.

제품의 기본적인 MVP는 완성이 되어있고, 최근 스프린트에서 유저 경험 개선 작업을 진행했습니다. 구체적으로

  1. D&D 기능,
  2. 캘린더 셀 클릭 시 폼 날짜 자동 입력 기능 이 도입되었고, 이제 제품의 실제 런칭이 임박한 상황이었습니다.

기존 개발 과정에서는 TDD 방식으로 Unit Test와 Integration Test를 작성하며 진행했기에, 코드 레벨의 품질은 어느 정도 보장된 상태였습니다.

그러나 최근 추가된 UX 기능들은 사용자 관점의 전체 플로우 검증이 필요했고, 런칭 직전 시점에서 치명적인 사용자 경험 결함을 사전에 발견하는 것이 가장 중요하다고 생각했습니다.

특히 이 캘린더 앱은 구조적 특성을 고려할 필요가 있었습니다. 폼 / 캘린더 / 일정목록이라는 세 가지 주요 화면이 하나의 페이지에 공존하며, 사용자 상호작용 시 실시간으로 동기화되는 구조입니다. 단일 기능 중심의 앱보다 사용자 경험 측면에서 복잡도가 높고, 화면 간 상태 동기화 오류가 발생할 여지가 많은 제품이라고 생각했습니다.

이러한 맥락에서 ROI 관점으로 볼 때, E2E 테스트 보강을 통한 사용자 시나리오 검증이 가장 비용 효율적이고 제품 품질에 직접적인 영향을 줄 수 있는 전략이라고 판단했습니다.

따라서, 다음과 같은 우선순위로 테스트 전략을 수립했습니다.

  1. E2E 테스트 우선 보강 - 핵심 사용자 시나리오 및 동기화 검증
  2. Integration Test 및 Unit Test 보완 - 코드 레벨의 안정성 강화

2. [제출] 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

팀원들과 함께 각자 준비한 프로젝트의 현황과 특성을 정리해 분석한 후, 최적의 테스트 전략에 대해 논의했습니다.

먼저 우리 프로젝트의 특성을 명확히 정의했습니다.

  • 기능 복잡도: 낮음 (CRUD 중심의 할일 관리)
  • UI 복잡도: 높음 (Form + Calendar + List가 한 화면에서 동기화)
  • 현재 단계: 주요 기능 완성, 배포 준비 단계
  • 기존 테스트: 통합/단위 테스트 상당량 작성됨

이러한 특성을 바탕으로, 팀은 다음과 같은 우선순위를 도출했습니다.

1순위: Drag & Drop 기능 최근 스프린트에서 추가된 핵심 UX 기능으로, 브라우저의 실제 이벤트 처리가 필요하고 사용자 경험에 직접적인 영향을 미치는 부분입니다. 특히 Calendar 내 날짜 간 드래그, 일정 겹침 처리 등 다양한 엣지 케이스가 존재합니다.

2순위: 세 영역(Form ↔ Calendar ↔ List) 간 동기화 한 페이지 내에서 세 개의 주요 화면이 실시간으로 동기화되는 구조이기 때문에, 상태 동기화 오류 발생 가능성이 높습니다. 이는 사용자가 직접 체감하는 품질 문제로 이어질 수 있습니다.

3순위: 실제 사용자 플로우 일정 추가 → 이동 → 완료까지의 전체 플로우가 실제 환경에서 정상 동작하는지 검증이 필요합니다.

테스트 유형별 전략 합의

팀원들은 각 테스트 유형의 목적과 트레이드오프를 논의한 결과, 다음과 같이 합의했습니다.

E2E 테스트: 최우선 보강 (만장일치)

E2E 테스트를 최우선으로 보강하기로 결정했습니다. 이유는:

  • Drag & Drop 같은 브라우저 실제 이벤트는 E2E로만 제대로 검증 가능
  • 세 영역의 실시간 동기화는 실제 사용자 관점에서 검증이 필수
  • 배포 직전 단계에서 치명적인 UX 결함을 사전 발견하는 것이 가장 중요
  • 제품의 실제 신뢰도에 가장 가까운 테스트

팀은 7개의 핵심 E2E 시나리오를 선정했습니다:

  • D&D 일정 수정
  • 반복일정 D&D 시 단일 일정 변환
  • 일정 겹침 시 경고 및 처리
  • 캘린더 셀 클릭 → 폼 반영 (Integration으로도 충분히 검증 가능하다는 의견이 주였으나, 런칭을 앞둔 제품 관점에서 E2E/Integration 이중 테스트를 하자는 합의)

통합 테스트: 보완 수준

이미 상당량 작성되어 있어 추가 작성은 최소화하되, 다음 부분만 보완하기로 했습니다:

  • 캘린더 날짜 클릭 시 폼 날짜 반영 (E2E와 중복 작성)
  • D&D 커스텀 훅 (작성한 팀원에 한해)
  • D&D 취소 및 API 실패 상황

단위 테스트: 추가 작성 안 함 (전원 합의)

단위 테스트는 추가로 작성하지 않기로 결정했습니다. 이유는:

  • 기존에 이미 작성된 유틸 함수 및 핵심 로직 테스트가 충분함
  • UI 컴포넌트 단위 테스트는 통합 테스트로 커버 가능
  • ROI 관점에서 우선순위가 낮음
  • 배포 직전 단계에서 리소스를 E2E에 집중하는 것이 더 효율적

결론

팀은 "E2E 테스트 우선 보강 → 통합 테스트 부분 보완" 전략을 선택했습니다.

이는 프로젝트의 현재 단계(배포 직전), 높은 UI 복잡도(세 영역 동기화), 그리고 최근 추가된 핵심 UX 기능(D&D)을 고려했을 때, 실제 사용자 관점에서의 품질 검증이 가장 중요하다는 판단이었습니다. 기존에 작성된 통합/단위 테스트로 코드 레벨 안정성은 확보되어 있으므로, 남은 리소스를 실제 사용자 경험 검증에 집중하는 것이 ROI 관점에서 가장 효율적이라는 결론을 내렸습니다!

다만,, 개인적으로 기술적인/코드적인 디테일은 떨어지는 결과를 낸 것 같습니다. 시간 부족이라는 핑계로 많은 부분을 Agent driven 으로.. 진행하기도 했습니다.

학습 효과 분석

단순히 테스트 코드 도입이라는 관점에서 생각하는 것보단, 제품의 상황에 적절한/최적의 테스트 전략이라는 관점에서 고민했을 때, 나오는 테스트 코드 로드맵이 달랐습니다. 단순 개발자로서 기술적인 고민보다는, 제품의 KPI를 기술과 어떻게 연결할 수 있을까? 제품의 개선을 위해 어떤 엔지니어링을 할까? 의 고민들을 하게 되는 경험이었습니다.

  • ROI 중심 사고
  • 제품 단계별 테스트 전략
  • 엔지니어링과 비즈니스의 연결 : 제품의 KPI(UX, 제품 출시)를 기술적 의사결정(어떤 테스트 전략?)과 어떻게 연결?

추가로 Playwright에서 제안하는 Page Object Models(POM)를 활용해보며, 기존에 현업에서 E2E 도입을 실패했을 때 효율적인 유지보수를 위해 이런 방식을 사용해봤으면 참 좋았을 텐데 라는 아쉬움을 갖기도 했습니다.

아쉬운 점

  • 역시나 과제에서 가장 아쉬운 점은, 코드 품질과 디테일을 챙기지 못한 부분입니다. 또 중간에 DnD kit을 쓰지 않고, 직접 구현이라는 악수를 둔 점도 아쉽습니다. 하지만 이 경험이 Playwright에 대한 이해도를 더 넓혀 주긴.. 했습니다.

  • 처음 계획은 Playwright MCP를 활용하여, 자연어로 구성된 E2E Test 코드 관리를 계획했는데, 이것을 시도해보지 못한 점이 아쉽습니다. 멋드러지게 팀원들에게 공유도 해보고 싶었는데 말이죠...

  • 테스트별 CI 전략을 많이 고민해보지 못한 점입니다. 테스트 유형과 역할에 맞는 트리거를 설정하고 이 것을 운용하는 경험도 해보면 좋지 않았을까 하는 아쉬움이 있습니다.

  • 지난주에도 아쉬웠던 점인데, 내 과제하기 급급해서 팀원들과 과제에 대한 대화를 많이 나누지 못한 점이 아쉽습니다. 이번주는 기필코 경험을 나누고 도울 것은 도와가며 과제를 진행하고 싶었는데 이번주도 실패했읍니다. 다음 주차에 이 부분을 우선적으로 바꿔보고 싶습니다.
