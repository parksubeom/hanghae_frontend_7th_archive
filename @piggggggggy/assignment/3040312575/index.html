배포 링크

  • /app : after application
  • /storybook/after : Composed Component 스토리북
  • /storybook/bento : Design System (bento-ui) 스토리북

과제 목표

레거시 코드베이스를 현대적인 디자인 시스템으로 개편하는 실무 경험

  1. 정리되지 않은 레거시 코드의 문제점 식별 및 분석
  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

Before 패키지 분석 후 After 패키지 개편

개편 목표

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

필수 과제

1. 디자인 시스템 구축

  • TailwindCSS 설정 및 디자인 토큰 정의
  • shadcn/ui 컴포넌트 설치 (Button, Input, Select, Card, Table 등)
  • CVA를 활용한 variants 패턴 적용
  • 일관된 스타일 시스템 구축

2. Before 패키지 분석

  • Before 패키지 실행 및 전체 코드 탐색
  • 스타일링, 컴포넌트 설계, 폼 관리 측면에서 문제점 파악
  • 개선이 필요한 부분과 그 이유 정리

3. 컴포넌트 개편

  • UI와 비즈니스 로직 분리
  • 순수한 UI 컴포넌트로 재구성
  • 일관된 컴포넌트 API 설계
  • 적절한 컴포넌트 구조 설계

심화 과제

  • Dark Mode 완전 지원 (CSS Variables + Tailwind)
  • Design Token 시스템 고도화 (색상 팔레트, 타이포그래피 스케일)
  • 뷰와 비즈니스로직이 분리되도록

디자인시스템을 모노레포로 분리하여, Basic Test CI가 실패합니다ㅠ

정도은(대표) : "...8월 알죠..?" 박준형(CFO) : "8월 초에 급여일인데요.. 그 전에 최소 2억은 필요해요." 황태영(영업팀장) : "지금 파이프라인에 있는 건 9월에나 들어와요." 정도은 : "그럼 방법이 뭔데?" .... (침묵) 황태영 : "... 하나 있긴 한데.."

모두의 시선이 집중됐다.

"G그룹 프로젝트요. 어드민 5개 시스템 통합 구축. 계약금만 3억, 총 8억짜리. 근데.."

정도은 : "빨리 말해" "경쟁 입찰이에요. 제안서 마감이 8월 15일. 우리 말고 2개 업체 더 들어가요. 그리고 G그룹이 원하는 납기가..."

황태영이 말음 멈췄다.

정도은 : "그래서 못한다고..?"

다시 침묵. 그때, 구석에 앉아 있던 박용태가 손을 든다.

3. 군밤보다 뜨거운 심장

프론트엔드 리드, 박용태가 말한다.

"방법이 있습니다..."

모든 시선이 박용태에게 모인다.

"지금 우리 문제가 뭔지 아세요? 매번 바퀴를 새로 만들고 있어요. 테이블 컴포넌트, 프로젝트마다 새로 만들어요. 폼, 모달, 페이지네이션... 전부요. 어드민 프로젝트의 70%는 같은 UI예요. 근데 우리는 매번 100%를 새로 만들어요."

차현빈 : "그건 알아. 근데 규격이 없잖아"

"그래서 만들자는 겁니다. 디자인 시스템. 우리가 반복해서 만드는 모든 컴포넌트를 한 번만 제대로 만들어두면..."

윤영서 : "잠깐, 그거 만드는 데 얼마나 걸리는데?"

"8주 요."

정도은 : "8주? 우리 지금 8주 여유 없어요. 저 탈출합니다?"

"알아요. 그래서 제안이 있습니다. 지금 진행 중인 H사 프로젝트 있잖아요. 그거 하면서 동시에 디자인 시스템을 만들어요. H사 프로젝트를 첫 번째 적용 사례로 쓰는 거예요."

차현빈 : "그건 리스크가.."

"리스크 있어요. 근데 지금 우리한테 선택지가 있나요? 8주 후면 7월 말이에요. G그룹 제안서 마감 2주 전. 그때 디자인 시스템이 있으면, 우리는 **'4개월에 5개 어드민'**을 자신 있게 제안할 수 있어요. 없으면? 16주 × 5개 = 80주 제안서 내야 해요. 그러면 우리는 탈락이고, 8월에 급여 못 줘요."

회의실이 조용해졌다.

(누군가) : "... 성공 확률이 얼마나 돼요?"

박용태가 대답한다.

"모르겠습니다. 해본 적 없어요. 근데 확실한 건, 지금 방식으로는 100% 실패예요."

긴 침묵 후, 정도은이 말한다.

"해보자. 근데 지금 배고프니까 여기서 회의 끝내자"

"발표 자료 대신, 이걸 먼저 보여드리겠습니다."

Storybook을 열었다. 실제로 작동하는 컴포넌트들을 시연한다.

"이게 저희가 쓰는 디자인 시스템입니다. 어드민에 필요한 모든 컴포넌트가 여기 있습니다. 다른 업체들은 '이렇게 만들겠다'고 말씀드릴 거예요. 저희는 '이미 이렇게 있다'고 말씀드립니다. 그래서 4개월에 5개 어드민이 가능합니다."

그리고 After App 데모를 열었다.

"이건 실제로 돌아가는 관리자 페이지입니다. G그룹 요구사항 중 '사용자 관리', '게시글 관리'... 이미 템플릿이 있습니다. 저희는 처음부터 만드는 게 아닙니다. 조합하고 커스터마이징하는 겁니다."

발표가 끝나고, G그룹 담당 임원이 물었다.

"근데 이 시스템... 다른 프로젝트에도 쓰신 거예요?" 황태영 : "네. 바로 직전에 완료한 H사 프로젝트가 이 시스템으로 만든 첫 번째 사례입니다. 원래 12주 예상이었는데, 8주에 끝났습니다."

임원의 눈이 커진다.

6. 결과

9월 5일 : 전화 한 통

영업팀장 황태영에게서 전화.

"....." "뭐에요? 술먹었어요?" "..됐어..." "... 뭐가요?" "G그룹. 우리한테 줬어. 계약금 3억, 다음 주에 들어와."

그날 저녁, 회사 전체 회식. 대표 정도은이 말했다.

"솔직히... 8월 넘기기 힘들 줄 알았다. 나 정도은 너네 못 믿은 거 맞다. 탈출 생각한 거 맞다. 근데 너네가 해냈다. 특히..."

정도은이 박용태를 쳐다본다.

"나 정도은, 디자인 시스템 만들자고 했을 때, 솔직히 반신반의했다. '지금 당장 급한데 그런 거 할 시간이 있나? 그냥 쨀까?' 근데 그게 결국 우리를 살렸다. 당장 급하다고 당장 급한 것만 하면, 영원히 급한 상태에서 못 벗어나더라고."

개편 과정에서 집중한 부분

1. '실제 어떤 상황에서 이 작업이 필요할까?'를 가정하고 진행했습니다.

2. 첫번째로 작업의 흐름을 어떻게 가져갈지 고민했습니다.

  • 목표 설정
    • 레거시 코드 구조를 개선한다.
    • 지속가능하고 확장하기 쉬운 구조를 위해 Design System을 도입한다.
  • Phase 1 : 레거시 코드 구조를 개선한다. a. 컴포넌트에서 도메인 관심사 분리 (UI 책임만) b. tailwind 도입 (primitive 토큰, semantic 토큰) c. Primtive 컴포넌트 구성 (shadcn 컴포넌트) d. cva을 활용한 컴포넌트 Variant 구조 도입 e. Composed 컴포넌트 구성 f. ManagementPage에 Component 적용 및 개선
  • Phase 2 : Design System을 도입한다. a. App의 Primitive 컴포넌트를 Design System (Bento) 패키지로 분리한다. b. App에서 Bento를 사용한다. c. App의 Composed 컴포넌트와 Bento의 Primitive 컴포넌트에 각각 Storybook을 구성한다.
  • 추가 작업
    • App에 다크 모드 도입
    • 폴더 구조 개선 (FSD-흉내-lite)

3. 컴포넌트 분리

  • 처음에는 단순히 컴포넌트 내부에 복잡하게 얽힌 관심사를 분리하는데 집중했습니다.
  • 기본적으로 도메인 관심사 분리
  • 그리고 컴포넌트의 역할 정의
    • Table 컴포넌트는 데이터를 보여주는 역할만?
    • Table 컴포넌트는 데이터에 대한 계산(필터)까지?
  • Primitive 컴포넌트를 구성/관리할 것 인지?
    • 순수 Input만? Label을 포함한 Input만? (FieldInput)

4. Shadcn 으로 컴포넌트를 만드는 기준은?

  • 계획대로 Input, Select, Button, Dialog... 등은 Shadcn을 활용해 Primitive 생성
  • 근데,, Card는..? 굳이 Shadcn을? 그런데 그러면 다른 Primitive는 왜 Shadcn을 사용했지? 나는 왜 Shadcn을 사용하고 있지?
  • Radix wrapper로 접근성이 필요할 때 사용해보자.
  • 고로 Card는 직접 구현

5. Design System 구축

  • Primitive 컴포넌트들을 Design System으로 분리
  • 재사용 가능한 구조를 위해 Monorepo의 새로운 workspace로 구축
  • Foundation도 DS로 가야할까..? (어차피 몇줄짜리 지만,,)
  • 결국 cn util과 foundation(token)도 DS로 분리

어려웠던 점과 해결 방법

1. 폴더 구조를 어떻게 가져가야할까?

  • Primitive 컴포넌트 분리 좋다 이거야.
  • Composed 컴포넌트 분리 좋다 이거야.
    • Primitive 컴포넌트를 Wrapping해 사용성을 올린 컴포넌트 (여전히 순수 UI 관심사만)
  • 근데, DataTable이나 FormModal 같은 Composed 컴포넌트를 페이지에서 직접 이용하니, 코드가 너무 길어지네... 결국엔 도메인이나 데이터 관심사를 갖는 컴포넌트가 필요하다.
  • Header는 여기서 어떤 맥락의 컴포넌트지...?
  • 근데 그렇다면 폴더 구조를 어떻게 구성해야할까?
결국 features 구조의 도입!
``` ... ├── components/ # Composed 컴포넌트 │ ├── DataTable.tsx │ ├── FormModal.tsx │ ├── ... │ ├── ... │ ├── ... │ └── ... │ ├── features/ # 도메인별 컴포넌트 │ ├── users/ │ │ ├── UserFormModal.tsx │ │ └── UserTable.tsx │ └── posts/ │ ├── PostFormModal.tsx │ └── PostTable.tsx │ ├── layouts/ # 레이아웃 │ └── Header.tsx │ ```

2. DS으로 App의 스타일 토큰을 옮기는게 맞나?

  • App도 기본적인 스타일 Foundation이 필요할텐데, 그것을 외부 DS에 전부 의존하는게 맞나?

  • DS에 있어야 할 것: primitive 토큰 (color, spacing, radius)

  • App에 남겨도 되는 것: App 고유의 레이아웃 상수, 특정 페이지 전용 스타일

    → DS는 "공유 가능한 기반"을 제공하고, App은 그 위에 확장 → Foundation이 몇 줄이라도 DS에 있는 게 맞음 (일관성 + 재사용성)

문제 상황 : Tailwind 스타일 토큰을 bento로 옮긴 후 두 가지 이슈 발생

  1. App에서 bento 토큰 사용 불가
  • bento 내부에 정의된 토큰이 App에서 인식 안 됨
  • bg-primary 같은 클래스가 App에서 작동하지 않음
  1. 토큰 가시성 부재
  • App 개발자가 bento에 어떤 토큰이 있는지 알 수 없음
  • 자동완성도 안 됨

해결: Arbitrary Values 방식 채택

/* bento - CSS 변수로 토큰 정의 */
  :root {
    --color-primary: #1976d2;
    --color-background: #ffffff;
  }

  /* bento 컴포넌트 - arbitrary values 사용 */
  className="bg-[var(--color-primary)]"

이 방식의 장점

항목설명
App에서 사용 가능CSS 변수는 import만 하면 전역 적용
오버라이딩 용이App에서 동일 변수명으로 재정의하면 덮어씀
다크모드 대응.dark { --color-primary: #90caf9 } 한 줄로전환
// App에서 오버라이딩
// index.css
:root {
  --color-primary: #ff5722;  /* bento 토큰 덮어쓰기 */
}

트레이드오프

  • 클래스가 길어짐 (bg-primary → bg-[var(--color-primary)])
  • Tailwind 자동완성 혜택 감소 → 패키지 간 토큰 공유를 위해 감수한 선택
