[5팀 김채영] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by rlacodud2025.10.19

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

테스트의 독립성과 병렬 실행 안정성을 위해 setupCrudHandlers 함수 내부에 클로저를 활용했습니다.

let mockEvents 변수는 함수가 호출될 때마다 독립적인 복사본으로 초기화되어 각 테스트 케이스는 분리된 이벤트 배열을 CRUD 작업에 사용하게 되고 이를 통해 테스트 간 데이터 중복을 방지할 수 있습니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

매 테스트가 시작되기 전, msw 서버에 새로운 Mock 핸들러 세트를 등록하여 initialEvent로 초기화된 데이터 환경을 보장합니다.

  beforeEach(() => {
    // 서버에 Mock 핸들러 등록
    server.use(...setupCrudHandlers(initialEvent));
  });

vi.useFakeTimers와 vi.setSystemTime을 사용하여 시간 관련 로직을 제어하고 vi.useRealTimers를 afterEach 또는 테스트 끝에서 호출하여 시간 환경을 원래대로 복원함으로써 다른 테스트에 시간적 영향을 주지 않도록 합니다.

과제 셀프회고

기술적 성장

- 새로 학습한 개념

  • 테스트코드: 코드가 의도한대로 잘 동작하는지 자동으로 확인하기 위한 코드로, 코드 명세서의 역할을 합니다.
  • Mock: 테스트 환경에서 진짜 객체나 함수 대신 사용할 가짜 객체로, 실제 서버나 외부 의존성 없이 테스트를 실행할 수 있도록 만들어줍니다.

- 기존 지식의 재발견/심화

  • 단위 테스트 VS 통합 테스트: 단위 테스트는 다른 모듈에 대한 의존성이 없는 독립적인 역할을 수행하는 모듈을 검증하는 테스트이고 통합 테스트는 **이 코드들이 서로 연결되었을 때 잘 동작하는가?**를 검증할 때 사용합니다.

코드 품질

- 특히 만족스러운 구현

useEventOperations Hook 테스트에서 헬퍼 함수를 작성한 부분입니다. 다른 분들에게는 일반적인 헬퍼 함수로 보일 수도 있지만 복잡하고 반복적인 스크립트를 작성해볼 기회가 적은 저로서는 반복되는 로직을 최소화하고자 헬퍼 함수로 구현해두는 생각을 할 수 있음에 만족스러웠습니다.

/*
  훅을 렌더링하고, 초기 이벤트 로딩이 완료될 때까지 기다린 후,
  로딩 성공 스낵바 호출을 검증하는 헬퍼 함수
*/
const renderHookAndWaitForLoad = async (editing: boolean) => {
  const renderResult = renderHook(() => useEventOperations(editing));

  // 1. 초기 로딩 완료 대기
  await waitFor(() => {
    // initialEvent가 로드되었는지 체크합니다.
    expect(renderResult.result.current.events).toEqual(initialEvent);
  });

  // 2. Mock 함수의 toHaveBeenCalledWith의 인수가 올바르게 들어왔는지 체크합니다.
  expect(enqueueSnackbarFn).toHaveBeenCalledWith('일정 로딩 완료!', { variant: 'info' });

  return renderResult;
};

- 리팩토링이 필요한 부분

통합테스트에서 "월별 뷰에 일정이 정확히 표시되는지 확인한다"를 테스트하는 케이스입니다.

제일 의문이 드는 테스트 케이스로, "월별 뷰"에 대해서만 확인하는 것과 "일정이 정확히 표시되는지"에 대한 기준이 이해가 가지 않았습니다.

지금 PR 템플릿을 작성하며 드는 생각은 "월별 뷰"에 대한 조건을 정한 이유가 달력의 view를 확인하는 게 아닐까 싶습니다.

현재가 10월이라면, 10월에 해당하는 일정들이 달력에 정확히 표시되는지 등에 대한 테스트를 정의한 것이라고 생각합니다.

이 부분에 대해서는 리팩토링하며 추가 테스트를 진행해보겠습니다!

    // 의도하는 바가 무엇일지 고민됩니다.
    // 일정이 정확히 표시된다는 게 모든 항목이 노출된다는 것인지
    // 현재 월에 해당하는 일정만 노출되는 것을 확인하는 것인지 모르겠습니다.
    it('월별 뷰에 일정이 정확히 표시되는지 확인한다', async () => {
      renderApp();

      const eventListEl = await screen.findByTestId('event-list');
      expect(await within(eventListEl).findByText('생일 파티')).toBeInTheDocument();
      expect(await within(eventListEl).findByText('2025-10-28')).toBeInTheDocument();
      expect(await within(eventListEl).findByText('친구 생일 축하')).toBeInTheDocument();
      expect(await within(eventListEl).findByText('친구 집')).toBeInTheDocument();
    });

- 코드 설계 관련 고민과 결정

통합테스트의 알림 테스트 영역입니다.

"notificationTime을 10으로 하면 지정 시간 10분 전 알람 텍스트가 노출된다" 처음에는 기존에 있는 이벤트 수정 => 알림 설정값 변경 => 타이머 테스트를 통해 진행하는 테스트라고 생각했습니다.

그러나 "수정"에 대한 조건이 없다는 걸 깨닫고 시작시간 10분 전으로 환경을 세팅해두고 알람이 노출되는지만 체크하면 되겠다는 생각이 들었고 전자의 생각을 구현하던 중 "알림 설정" 요소를 찾는 데에서 어려움이 있었습니다.

describe('알림 테스트', () => {
    /*
      1. 이 테스트는 사실 생각보다 맞는 결과값을 받기 어려웠습니다.
      처음에는 기존에 있는 이벤트에서 수정 => 알림 설정값을 변경하고 타이머를 테스트하며 진행한다고 생각했습니다.

      2. 그러나 수정이라는 조건이 없었기에 시작시간 10분 전 상황을 만들면 되지 않을까 싶어 이와 같이 작성했습니다.
      1에 작성한 내용으로 테스트하려고 했을 때 일단 "알림 설정" 요소를 찾는 데에서 어려움이 있었습니다.
      주간 뷰 셀렉트박스와 구조가 비슷하다고 생각해 주간 뷰를 변경하는 방식과 같이 진행하면 될 거라고 생각했지만
      실제로는 같은 방식으로는 combobox까지 접근이 안됐습니다.
      
      그 이유와 통합 테스트에서 비동기적 접근과 타이밍이 중요한 것 같아 고민해보려고 합니다.
    */
    it('notificationTime을 10으로 하면 지정 시간 10분 전 알람 텍스트가 노출된다', async () => {
      vi.useFakeTimers({ shouldAdvanceTime: true });
      vi.setSystemTime(new Date('2025-10-20T09:50:00'));
    
      try {
        renderApp();
    
        await screen.findByTestId('event-list');
    
        // 시간 진행
        await act(async () => {
          await vi.advanceTimersByTimeAsync(1000);
        });
    
        // 10분 전 알람 텍스트가 노출되는지 체크합니다.
        await waitFor(() => {
          expect(screen.getByText('10분 후 팀 회의 일정이 시작됩니다.')).toBeInTheDocument();
        });
    
      } finally {
        vi.useRealTimers();
      }
    }, 10000);
  });

최종적으로는 비동기로 처리하여 알람이 노출되는지 체크할 수 있었지만 다른 방법으로는 또 어떻게 처리할 수 있을지 궁금합니다.

학습 효과 분석

- 가장 큰 배움이 있었던 부분

테스트 코드는 단순히 동작 검증이 아니라, 코드의 명세서 역할을 한다는 점을 깨달았습니다. 테스트를 작성하려면 각 모듈, 훅, 시스템의 내부 동작을 정확히 이해해야 하므로, 테스트 코드가 프로젝트 전체의 이해도를 높이는 도구가 될 수 있음을 느꼈습니다.

- 비동기 렌더링 및 쿼리

통합 테스트에서 가장 어려웠던 부분은 요소 탐색 시점과 비동기 렌더링 처리였습니다. 렌더링이 완료된 뒤 탐색이 이뤄져야 하는데, 그 타이밍을 정확히 맞추는 것이 쉽지 않았습니다. 비동기 렌더링과 관련된 로직은 추가로 학습을 진행할 예정입니다.

과제 피드백

- 난이도 설정

테스트코드를 처음 접하는 사람도 차근차근 진행하며 이해해나갈 수 있도록 단계별 난이도 설정이 좋았습니다.

리뷰 받고 싶은 내용

궁금한 점

- describe 블록의 기준

테스트의 범위 혹은 전제조건이 같거나 큰 기능면에 묶을 수 있겠다 싶은 것을 기준으로 describe 블록을 추가 구성한 부분들이 있습니다. 이 부분들에 대해 올바른 판단이었을지, 더 개선된다면 어떻게 나눌 수 있을지 궁금합니다.

- beforeEach, afterEach의 올바른 사용 기준

어느순간부터 beforeEach, afterEach 사용에 익숙해지더니 테스트코드 내에서 반복되어 설정하는 내용이 있으면 두 키워드를 활용하여 반복 처리될 수 있도록 했습니다. 그러나 문득 이렇게 반복시키다보면 성능 저하를 일으킬 수도 있지 않을까 하는 걱정이 생기며 지금 제가 잘 사용하고 있는지에 대해 궁금해졌습니다.

과제를 수행하면서 느낀점

과제 시작 전 생각

  • 테스트코드의 중요성에 대해 막연하게 생각하고 있었습니다.
  • 제가 이 과제를 제대로 완료할 수 있을까는 고사하고 테스트코드에 대한 이해가 가능할까 싶었습니다.
  • 발제를 듣자마자 모르는 내용투성이라 막막했지만 동시에 모르는 걸 배울 생각에 기대가 됐습니다.

과제 제출 후 생각

  • 테스트코드의 중요성에 대해 깨달은 게 제일 좋은 결과인 것 같습니다.
  • 과제 수행이라고 너무 순차적으로 코드를 인지하고 진행한 것 같다는 생각이 들었습니다. 미리 어떤 테스트에 어떤 테스트 데이터가 필요할지 생각했다면 테스트 데이터를 두번 쓰는 일은 없었을 것 같다는 생각이 들었습니다.
  • 어디까지 테스트를 해야하고, 이 테스트가 과연 어디까지 보장해줄 수 있을까에 의문이 들었습니다. 이번에 과제를 진행하면서 하드코딩식의 더미데이터와 단순히 텍스트가 화면에 존재하는지 등으로 테스트를 하는 것으로 괜찮을지에 대해 생각이 들었고 실무 코드 또한 이런 식으로 진행되는지, 그렇다면 어떻게 보장될 수 있을지 궁금해졌습니다.

과제 피드백

수고 많았어요! 이번 과제는 테스트를 하는 다양한 방법들을 접하면서 테스트 코드를 작성한다는 그 심리적 장벽을 허물고 친숙해지는데 있었습니다. Hard까지 완주하시고 회고도 정말 상세하게 작성해주셨네요!

"테스트코드의 중요성에 대해 막연하게 생각하고 있었는데 과제를 제출 후 중요성에 대해 깨달은 게 제일 좋은 결과"라고 하셨는데 너무 뿌듯하네요. 정확합니다! 테스트 코드가 단순히 동작 검증이 아니라 코드의 명세서 역할을 한다는 걸 깨달으셨다니 이번 과제의 목표를 정확히 달성하셨네요. 특히 handlersUtils에서 클로저를 활용해 테스트 독립성을 확보한 부분과 beforeEach에서 서버 핸들러를 등록하는 설정들을 정확히 이해하고 계셔서 좋습니다.

헬퍼 함수 renderHookAndWaitForLoad 작성한 부분이 만족스러웠다고 하셨는데 ㅎㅎ 좋네요. 개발자들은 반복을 참지 못하죠. 아주 좋은 접근입니다. "반복되는 로직을 최소화"하려는 생각이야말로 좋은 개발자의 시작이에요. 실무에서도 테스트 코드에서 반복되는 패턴을 헬퍼 함수로 추출하는 건 필수적인 작업이죠. 이름은 살짝 아쉽네요. "renderHookAndWaitForLoad" 라면 뭔가 대단히 보편적인 행동을 할 것 같은 이름인데 하는 행동은 꽤 구체적이라 수준에 맞는 이름을 지었다면 더 좋았겠어요.

"어디까지 테스트를 해야하고, 이 테스트가 과연 어디까지 보장해줄 수 있을까" 하는 의문이 들었다고 하셨는데 그것이 테스트 코드의 진짜 실무의 능력이죠. 지금은 영어 회화를 배우는 거라면 실전 영어는 완전히 다른게 지금 상황에 필요한 말을 할 수 있는가잖아요? 그리고 이건 어떻게든 말로 설명을 해 줄 수 있는게 아니니 "효용체감을 느낄 수 있는 테스트 코드 작성"이라는 최종 미션은 꼭 실무에서 경험해보게 되기를 바랍니다.

통합테스트에서 비동기 렌더링과 타이밍 처리가 어려웠다고 하셨는데 프론트엔드에게 비동기는 누구나 어렵게 만드는 녀석이죠. findBy는 자동으로 기다려주지만 getBy는 즉시 찾으려 하고, waitFor는 조건이 충족될 때까지 기다려주는 등 유사해보이지만 왜 각각 만들어 두었을까를 고민해보면서 각각의 특성을 생각해보면 왜 이렇게 만들었을까? 를 생각해보면 더 머리속에 잘 남겨둘 수 있습니다.

describe 블록 기준과 beforeEach/afterEach 사용에 대해 궁금해하셨는데, 지금처럼 "같은 전제조건"을 가진 테스트들을 묶는 게 맞습니다. 대개 같은 목적을 가지고 것들을 기준으로 한데 묶어준다거나 그렇기에 지금처럼 유사한 조건에서 시작하는 경우등을 묶어내는데 사용할 수 있죠. 공통점이 있다면 자연스러게 유사한 곳에 둬야 하는 구조적 사고들도 테스트 코드에서 발휘해야 합니다.

beforeEach/afterEach에 대한 성능 걱정은 하지 않아도 됩니다. 오히려 테스트의 최적화(?) 보다는 각 테스트가 독립적인 환경에서 실행되는 게 훨씬 중요합니다. 테스트 A가 테스트 B에 영향을 주면 그게 더 큰 문제가 됩니다.
