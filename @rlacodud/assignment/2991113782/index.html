기본과제

상품목록

상품 목록 로딩

  • 페이지 접속 시 로딩 상태가 표시된다
  • 데이터 로드 완료 후 상품 목록이 렌더링된다
  • 로딩 실패 시 에러 상태가 표시된다
  • 에러 발생 시 재시도 버튼이 제공된다

상품 목록 조회

  • 각 상품의 기본 정보(이미지, 상품명, 가격)가 카드 형태로 표시된다

한 페이지에 보여질 상품 수 선택

  • 드롭다운에서 10, 20, 50, 100개 중 선택할 수 있으며 기본 값은 20개 이다.
  • 선택 변경 시 즉시 목록에 반영된다

상품 정렬 기능

  • 상품을 가격순/인기순으로 오름차순/내림차순 정렬을 할 수 있다.
  • 드롭다운을 통해 정렬 기준을 선택할 수 있다
  • 정렬 변경 시 즉시 목록에 반영된다

무한 스크롤 페이지네이션

  • 페이지 하단 근처 도달 시 다음 페이지 데이터가 자동 로드된다
  • 스크롤에 따라 계속해서 새로운 상품들이 목록에 추가된다
  • 새 데이터 로드 중일 때 로딩 인디케이터와 스켈레톤 UI가 표시된다
  • 홈 페이지에서만 무한 스크롤이 활성화된다

상품을 장바구니에 담기

  • 각 상품에 장바구니 추가 버튼이 있다
  • 버튼 클릭 시 해당 상품이 장바구니에 추가된다
  • 추가 완료 시 사용자에게 알림이 표시된다

상품 검색

  • 상품명 기반 검색을 위한 텍스트 입력 필드가 있다
  • 검색 버튼 클릭으로 검색이 수행된다
  • Enter 키로 검색이 수행된다
  • 검색어와 일치하는 상품들만 목록에 표시된다

카테고리 선택

  • 사용 가능한 카테고리들을 선택할 수 있는 UI가 제공된다
  • 선택된 카테고리에 해당하는 상품들만 표시된다
  • 전체 상품 보기로 돌아갈 수 있다
  • 2단계 카테고리 구조를 지원한다 (1depth, 2depth)

카테고리 네비게이션

  • 현재 선택된 카테고리 경로가 브레드크럼으로 표시된다
  • 브레드크럼의 각 단계를 클릭하여 상위 카테고리로 이동할 수 있다
  • "전체" > "1depth 카테고리" > "2depth 카테고리" 형태로 표시된다

현재 상품 수 표시

  • 현재 조건에서 조회된 총 상품 수가 화면에 표시된다
  • 검색이나 필터 적용 시 상품 수가 실시간으로 업데이트된다

장바구니

장바구니 모달

  • 장바구니 아이콘 클릭 시 모달 형태로 장바구니가 열린다
  • X 버튼이나 배경 클릭으로 모달을 닫을 수 있다
  • ESC 키로 모달을 닫을 수 있다
  • 모달에서 장바구니의 모든 기능을 사용할 수 있다

장바구니 수량 조절

  • 각 장바구니 상품의 수량을 증가할 수 있다
  • 각 장바구니 상품의 수량을 감소할 수 있다
  • 수량 변경 시 총 금액이 실시간으로 업데이트된다

장바구니 삭제

  • 각 상품에 삭제 버튼이 배치되어 있다
  • 삭제 버튼 클릭 시 해당 상품이 장바구니에서 제거된다

장바구니 선택 삭제

  • 각 상품에 선택을 위한 체크박스가 제공된다
  • 선택 삭제 버튼이 있다
  • 체크된 상품들만 일괄 삭제된다

장바구니 전체 선택

  • 모든 상품을 한 번에 선택할 수 있는 마스터 체크박스가 있다
  • 전체 선택 시 모든 상품의 체크박스가 선택된다
  • 전체 해제 시 모든 상품의 체크박스가 해제된다

장바구니 비우기

  • 장바구니에 있는 모든 상품을 한 번에 삭제할 수 있다

상품 상세

상품 클릭시 상세 페이지 이동

  • 상품 목록에서 상품 이미지나 상품 정보 클릭 시 상세 페이지로 이동한다
  • URL이 /product/{productId} 형태로 변경된다
  • 상품의 자세한 정보가 전용 페이지에서 표시된다

상품 상세 페이지 기능

  • 상품 이미지, 설명, 가격 등의 상세 정보가 표시된다
  • 전체 화면을 활용한 상세 정보 레이아웃이 제공된다

상품 상세 - 장바구니 담기

  • 상품 상세 페이지에서 해당 상품을 장바구니에 추가할 수 있다
  • 페이지 내에서 수량을 선택하여 장바구니에 추가할 수 있다
  • 수량 증가/감소 버튼이 제공된다

관련 상품 기능

  • 상품 상세 페이지에서 관련 상품들이 표시된다
  • 같은 카테고리(category2)의 다른 상품들이 관련 상품으로 표시된다
  • 관련 상품 클릭 시 해당 상품의 상세 페이지로 이동한다
  • 현재 보고 있는 상품은 관련 상품에서 제외된다

상품 상세 페이지 내 네비게이션

  • 상품 상세에서 상품 목록으로 돌아가는 버튼이 제공된다
  • 브레드크럼을 통해 카테고리별 상품 목록으로 이동할 수 있다
  • SPA 방식으로 페이지 간 이동이 부드럽게 처리된다

사용자 피드백 시스템

토스트 메시지

  • 장바구니 추가 시 성공 메시지가 토스트로 표시된다
  • 장바구니 삭제, 선택 삭제, 전체 삭제 시 알림 메시지가 표시된다
  • 토스트는 3초 후 자동으로 사라진다
  • 토스트에 닫기 버튼이 제공된다
  • 토스트 타입별로 다른 스타일이 적용된다 (success, info, error)

심화과제

SPA 네비게이션 및 URL 관리

페이지 이동

  • 어플리케이션 내의 모든 페이지 이동(뒤로가기/앞으로가기를 포함)은 하여 새로고침이 발생하지 않아야 한다.

상품 목록 - URL 쿼리 반영

  • 검색어가 URL 쿼리 파라미터에 저장된다
  • 카테고리 선택이 URL 쿼리 파라미터에 저장된다
  • 상품 옵션이 URL 쿼리 파라미터에 저장된다
  • 정렬 조건이 URL 쿼리 파라미터에 저장된다
  • 조건 변경 시 URL이 자동으로 업데이트된다
  • URL을 통해 현재 검색/필터 상태를 공유할 수 있다

상품 목록 - 새로고침 시 상태 유지

  • 새로고침 후 URL 쿼리에서 검색어가 복원된다
  • 새로고침 후 URL 쿼리에서 카테고리가 복원된다
  • 새로고침 후 URL 쿼리에서 옵션 설정이 복원된다
  • 새로고침 후 URL 쿼리에서 정렬 조건이 복원된다
  • 복원된 조건에 맞는 상품 데이터가 다시 로드된다

장바구니 - 새로고침 시 데이터 유지

  • 장바구니 내용이 브라우저에 저장된다
  • 새로고침 후에도 이전 장바구니 내용이 유지된다
  • 장바구니의 선택 상태도 함께 유지된다

상품 상세 - URL에 ID 반영

  • 상품 상세 페이지 이동 시 상품 ID가 URL 경로에 포함된다 (/product/{productId})
  • URL로 직접 접근 시 해당 상품의 상세 페이지가 자동으로 로드된다

상품 상세 - 새로고침시 유지

  • 새로고침 후에도 URL의 상품 ID를 읽어서 해당 상품 상세 페이지가 유지된다

404 페이지

  • 존재하지 않는 경로 접근 시 404 에러 페이지가 표시된다
  • 홈으로 돌아가기 버튼이 제공된다

(0) 배포 설정

과제 제출 조건에 GitHub Pages로 배포하는 것이 포함되어 있었기 때문에 초기 설정 단계에서 배포 환경을 먼저 구축하는 것이 전체 개발 효율에 유리하다고 판단했습니다.

  • SPA는 라우팅 설정에 따라 GitHub Pages에서 경로 문제가 발생할 수 있다. (예: /product/1 접근 시 404 문제가 생기는 일반적인 GitHub Pages 이슈)
  • 개발 막바지에 배포를 시도하면 이 문제 해결이 더 큰 리스크가 된다.
  • 초기에 배포를 구성해두면 개발 중에도 실시간으로 배포 환경에서 테스트할 수 있다. ㄴ그러나 후일담으로 목요일에 배포 설정이 꼬이면서 다시 설정하는 이슈가 있었습니다..🥲

(1) 상태 관리

SPA 구현에서 가장 중요한 두 가지는 상태 관리와 렌더링 라이프사이클이라고 생각합니다.

그중 라이프사이클은 이후 리팩토링으로 개선 가능하지만, 상태 관리 체계는 초기 설계가 전체 코드 구조를 결정한다고 생각하기에 상태 관리부터 설계했습니다.

다음을 고려하여 store를 구축했습니다.

  • 여러 페이지와 컴포넌트에서 공유하는 데이터 존재 (예: 장바구니, 카테고리 조건, 검색 조건 등)
  • URL 상태와 UI 상태를 동기화해야 하는 기능 존재
  • 상태를 기준으로 DOM을 렌더링해야 하는 구조

(1-1) Observer Pattern

중앙 저장소에 상태를 만들었으면 이 상태가 변경될 때마다 처리해줘야 하는 동작이 있을텐데 예를 들어 카테고리 조건이 변경되면 => 그에 맞춰 다시 상품 목록을 불러오고 => 상품 목록 리스트를 렌더해야 합니다. 이런 식으로 객체의 상태 변화가 있을 때마다 각 옵저버에게 통지하도록 하는 디자인 패턴을 Observer pattern이라고 합니다.

1️⃣ store store 객체가 상태를 관리하고 변경을 통지하는 주체 역할을 합니다.

export const store = {
  state: {...},
  listeners: new Set(),
  ...

state는 관찰되는 데이터 목록이고 listeners는 관찰하는 관찰자의 목록입니다.

2️⃣ subscribe 컴포넌트나 UI 렌더 함수를 fn으로 받아서 listeners에 넣어두고 필요할 때 호출합니다.

  subscribe(fn) {
    this.listeners.add(fn);
    fn(this.state); // 초기 호출
    return () => this.listeners.delete(fn);
  },

3️⃣ setState setState를 호출하여 상태를 변경하고 모든 observer(listeners)에게 알립니다.

  setState(newState) {
    this.state = { ...this.state, ...newState };
	...
    this.listeners.forEach((fn) => fn(this.state));
  },

(2) 컴포넌트 분리

main.js에 집중되어 있던 UI 로직을 기능 단위로 세분화했고 이를 통해 SPA 구조 내에서 UI 독립성과 재사용성을 확보했습니다.

  • ProductList, CartModal, Toast, Breadcrumbs 등으로 분리

(3) 이벤트 처리

이벤트는 모두 이벤트 위임(Event Delegation) 방식을 적용했습니다.

document.body.addEventListener("click", (e) => {
  if (e.target.closest(".product-card")) {
   ...
  }
});

(4) UX 개선사항

명세에는 없지만 UX적으로 보완이 필요한 부분이 발견될 때마다 개선을 진행했고 각 개선 사항은 사용자 경험, 맥락 유지, 데이터 신뢰성, 오류 방지를 위한 근거에 기반했습니다.

(4-1) 수량 input 제어 상품 상세에서 구매 수량을 제어할 때 input에 재고 수량을 초과하게 입력하는 경우에 대한 처리가 되어있지 않았습니다. 이 부분은 input 이벤트로 value가 재고를 초과할 경우를 감지해서 초과하면 재고 값으로 설정되도록 처리했습니다.

(4-2) 상세 페이지 스크롤 초기화 메인 > 상품을 클릭하거나 상세 > 관련 상품을 클릭하여 상세 페이지로 이동할 경우 스크롤이 초기화되어 최상단으로 가는 게 사용자 경험측에서 좋을 것이라고 생각했습니다.

SPA의 특징은 실제 페이지가 전환되지 않기 때문에 사용자는 현재 페이지가 바뀌었는지 직관적으로 인식하기 어렵습니다. 따라서 스크롤을 초기화하면 자연스럽게 "새로운 페이지에 진입했다"고 인지하게 됩니다.

(4-3) 메인 > 상품 카드 내부 정렬 각 상품 카드의 정보 중 brand 유무에 따라 카드 내부 정렬이 다르게 보이는 문제가 있었고 사실 이 부분은 예시 페이지 처음 봤을 때부터 개선하고 싶던 부분이어서 버튼 정렬이 동일하게 보이도록 수정했습니다.

(5) 라이프 사이클

SPA에서는 페이지 이동 시 실제 DOM이 사라지지 않으므로 컴포넌트의 mount/unmount 시점을 명확히 관리해야 했습니다.

  • mount: 이벤트 등록, 초기 렌더링, 데이터 패칭
  • unmount: 리스너 해제, 상태 정리, 스크롤 관리

이 과정을 통해 중복 이벤트 등록 및 메모리 누수를 방지했습니다.

🐛 버그 수정: 검색어 초기화 시 상품 목록이 업데이트되지 않는 문제

1. 문제 원인

E2E 테스트를 실행하던 중 발견한 버그로, 다음과 같은 시나리오에서 문제가 발생했습니다🥲

(1) 카테고리 필터 적용 후 검색어 초기화

  • 사용자가 "전체" 브레드크럼을 클릭하여 카테고리 필터 초기화
  • 검색어 입력창을 비우고 Enter 키를 눌러 검색어를 초기화
  • 예상: 모든 상품 목록(340개)이 표시되어야 함
  • 실제: 이전 검색 결과(12개)가 그대로 유지됨

2. 근본 원인 분석

(1) Race Condition 문제

  • "전체" 브레드크럼 클릭 시 loadProducts가 시작되어 isFetchingProducts = true 상태
  • 검색어를 비우고 Enter를 누르면 store.subscribe에서 loadProducts를 호출하려고 시도
  • 하지만 이전 요청이 진행 중이어서 isFetchingProducts가 true인 상태
  • loadProducts 함수 내부의 조기 종료 조건(if (this.isFetchingProducts) return;)에 걸려 실제 API 호출이 이루어지지 않음

(2) 상태 관리 문제

  • pendingLoadProducts 플래그를 통해 재시도 로직을 구현했지만
  • 재시도된 loadProducts가 isLoaded: true 상태 때문에 또 다른 조기 종료 조건(if (!append && store.state.isLoaded && !store.state.error) return;)에 걸림
  • 이전 요청이 완료되면서 isLoaded: true로 설정되었기 때문에, 새로운 검색어로 상품을 다시 로드해야 함에도 불구하고 조기 종료됨

(3) PopState 이벤트와의 충돌

  • 브라우저 뒤로 가기/앞으로 가기 시 popstate 이벤트가 발생
  • popstate 핸들러에서 store.setState를 호출하면 store.subscribe가 트리거됨
  • popstate 핸들러에서 이미 loadProducts를 직접 호출하는데, store.subscribe에서도 파라미터 변경을 감지하여 중복 호출 시도
  • 이로 인해 예상치 못한 동작이 발생할 수 있음

3. 해결 방법

(1) Race Condition 해결: pendingLoadProducts 플래그 활용

// store.subscribe에서 파라미터 변경 감지
if (shouldLoad) {
  if (this.isFetchingProducts) {
    // 이미 fetching 중이면 pendingLoadProducts 플래그만 설정
    // loadProducts 완료 시 재시도됨
    store.setState({ isLoaded: false });
    this.pendingLoadProducts = true;
  } else {
    // 즉시 호출 가능한 경우
    this.pendingLoadProducts = true;
    setTimeout(() => {
      store.setState({ isLoaded: false });
      this.loadProducts(1, false);
    }, 0);
  }
}

// loadProducts의 finally 블록에서 재시도
finally {
  this.isFetchingProducts = false;
  
  if (this.pendingLoadProducts) {
    this.pendingLoadProducts = false;
    store.setState({ isLoaded: false }); // 재시도 시에도 isLoaded 초기화
    setTimeout(() => {
      this.loadProducts(1, false);
    }, 0);
  }
}

(2) 상태 관리 개선: 파라미터 변경 시 isLoaded 명시적 초기화 파라미터 변경으로 인한 재로드 시에는 항상 isLoaded: false를 설정하여 조기 종료를 방지

// 파라미터 변경으로 인한 재로드이므로 isLoaded를 false로 설정
store.setState({ isLoaded: false });
this.loadProducts(1, false);

(3) PopState 이벤트 처리 개선: isPopState 플래그 활용 popstate 이벤트로 인한 변경은 popstate 핸들러에서 직접 처리하므로, store.subscribe에서 파라미터 변경 감지를 건너뛰도록 수정

// store.subscribe에서
if (state.path === "/" && !this.isFetchingProducts && !this.pendingLoadProducts && !this.isPopState) {
  // 파라미터 변경 감지 및 loadProducts 호출
}

// popstate 핸들러에서
this.isPopState = true;
// ... 상태 업데이트 및 loadProducts 호출
queueMicrotask(() => {
  this.isPopState = false; // store.subscribe 실행 후 플래그 리셋
});

4. 테스트 코드의 중요성에 대한 깨달음

이번 버그 수정 과정에서 테스트 코드의 중요성을 다시 한 번 깊이 느꼈습니다.

처음에는 로컬 환경에서 수동으로 테스트했을 때 문제가 없어 보였습니다!

  • "전체" 브레드크럼 클릭 → 정상 동작
  • 검색어 입력 후 Enter → 정상 동작
  • 검색어 비우고 Enter → 정상 동작

하지만 특정 타이밍과 순서에서만 발생하는 Race Condition이었기 때문에, 수동 테스트로는 발견하기 어려웠고 E2E 테스트(e2e.basic.spec.js)를 실행했을 때만 문제가 명확히 드러났습니다.

  • 테스트는 정확한 순서와 타이밍으로 사용자 시나리오를 재현
  • "전체" 브레드크럼 클릭 직후 검색어를 비우는 특정 타이밍에서만 발생하는 버그를 정확히 포착
  • 콘솔 로그를 통해 loadProducts가 조기 종료되는 지점을 명확히 확인 가능

테스트 코드의 가치

  1. 재현 가능한 버그 발견: 수동 테스트로는 발견하기 어려운 타이밍 이슈를 정확히 재현
  2. 디버깅 효율성: 콘솔 로그와 함께 테스트를 실행하면 문제의 정확한 원인을 빠르게 파악 가능
  3. 회귀 방지: 수정 후 테스트를 다시 실행하여 동일한 문제가 재발하지 않음을 보장
  4. 문서화: 테스트 코드 자체가 예상되는 사용자 시나리오를 문서화하는 역할

이번 과제는 프레임워크 없이 SPA 구현이었지만 E2E 버그 경험을 통해 테스트 코드의 가치에 대해 명확히 깨달았습니다. 챕터1에서 테스트 코드를 진행한만큼 해당 명세를 더 깊이있게 이해하고 과제를 진행했다면 좋았을 것 같다는 아쉬움도 남았습니다.

관련 이슈

  • E2E 테스트 실패: e2e.basic.spec.js:112 - 검색어 초기화 후 상품 목록 업데이트 실패
  • E2E 테스트 실패: e2e.advanced.spec.js:366 - 브라우저 뒤로가기 후 상품 목록 렌더링 실패

개선이 필요하다고 생각하는 코드

1. 이중 상태 관리(URL + Store)

현재 코드는 URL 파라미터와 Store 상태를 모두 관리하고 있어 복잡도가 높고 이로 인한 문제를 테스트 코드에서 경험했습니다.

// 현재 흐름
performSearch() 
  → navigate() 
    → history.pushState() 
    → store.setState() 
      → store.subscribe() 
        → 파라미터 변경 감지 
          → loadProducts()

문제점

  • URL과 Store 상태가 동기화되어야 함
  • navigate() 함수에서 URL을 업데이트하고, 다시 파싱해서 Store에 반영
  • store.subscribe에서 파라미터 변경을 감지하기 위해 previousSearch, previousSort 등을 추적

2. 불필요한 상태 변경 감지

store.subscribe가 모든 상태 변경을 감지하고 있습니다

store.subscribe(async (state) => {
  // 장바구니 모달 상태 변경도 감지
  if (state.isCartModalOpen) { ... }
  
  // 검색/필터 파라미터 변경 감지
  if (state.path === "/" && !this.isFetchingProducts && !this.pendingLoadProducts && !this.isPopState) {
    const searchChanged = this.previousSearch !== state.search;
    // ... 복잡한 변경 감지 로직
  }
  
  // UI 렌더링
  await this.handleStateChange(state);
});

문제점

  • 장바구니 모달 열기/닫기 같은 UI 상태 변경도 store.subscribe를 트리거
  • 파라미터 변경 감지를 위해 previousSearch, previousSort 등을 인스턴스 변수로 추적
  • 복잡한 조건문 (!this.isFetchingProducts && !this.pendingLoadProducts && !this.isPopState)

3. 복잡한 플래그 관리

여러 플래그로 상태를 관리하고 있습니다

this.isPopState = false;        // popstate 이벤트인지 구분
this.pendingLoadProducts = false; // 재시도가 필요한지 구분
this.isFetchingProducts = false;  // 현재 로딩 중인지 구분
this.previousSearch = null;      // 이전 검색어 추적
this.previousSort = null;        // 이전 정렬 추적
// ... 등등

문제점

  • 플래그 간의 상호작용이 복잡함
  • Race condition 해결을 위해 pendingLoadProducts 같은 플래그 추가
  • 디버깅이 어려움

4. 간접적인 호출 체인

이벤트 핸들러에서 loadProducts를 직접 호출하지 않고, store.subscribe를 통해 간접적으로 호출

// performSearch에서
this.navigate(url);  // URL 업데이트
store.setState({ isLoaded: false });  // 상태 업데이트
// loadProducts는 store.subscribe에서 호출됨

// handleCategoryFilter에서
this.navigate(url);
this.updateCategoryState();
// loadProducts는 store.subscribe에서 호출됨

문제점

  • 호출 흐름이 명확하지 않음
  • 디버깅 시 어디서 loadProducts가 호출되는지 추적하기 어려움
  • 테스트 작성이 복잡함

학습 효과 분석

✅이벤트 위임에 대한 이해

그동안은 Vanilla Javascript에서 이벤트를 작업해야 할 때 각 요소를 찾아서 이벤트 리스너를 붙이곤 했습니다.

그러나 코치님은 라이브 코딩에서 body에 이벤트 리스너를 등록하고 실제 클릭된 요소를 e.target으로 확인 후 처리했습니다. 이런 식으로 공통된 부모 요소에 이벤트 리스너를 등록하여 처리하는 방식을 이벤트 위임이라고 합니다.

그렇다면 왜 이벤트 위임을 사용할까요?

(1) 동적 DOM 고려 첫번째로 동적으로 생성되는 DOM에 자동으로 이벤트가 적용되기 때문입니다. SPA의 특성 상 렌더링이 자주 일어나기 때문에 각 요소에 직접 이벤트를 달면, 렌더링이 바뀔 때마다 코드를 다시 실행해야 작동합니다.

(2) 편한 구조 관리 두번째로, 상위 컨테이너에 이벤트 캐쳐가 있기 때문에 구조 관리가 편합니다. 만약 이벤트를 요소마다 관리하면 특정 이벤트가 어디서 등록되었는지 찾기 어렵고 컴포넌트 제거 시 리스너 제거까지 신경써야 하게 됩니다.

(3) 이벤트 리스너 메모리 측면 지금까지 product-card에 forEach로 같은 함수를 등록했지만 그럴 경우, addEventListener 자체가 product-card의 수만큼 발생하는 것이므로 product-card가 200개면 200개의 리스너가 생성돼서 등록돼버립니다.

document.querySelectorAll(".product-card").forEach(card => {
  card.addEventListener("click", handleCardClick);
});

이 개념을 이해하면서 리스너를 같은 함수로 등록한다고 해서 1개만 등록되는 게 아니라는 걸 처음 알았습니다. 함수가 같아도 등록 행위(addEventListener) 자체가 새로운 리스너를 만들고, 이는 메모리 누수로 이어지게 됩니다.

이를 이벤트 위임 방식으로 바꾸면, 리스너는 1개만 존재하기 때문에 200개의 상품 카드가 있어도 1개의 리스너만 생성돼서 등록됩니다.

document.body.addEventListener("click", (e) => {
  if (e.target.closest(".product-card")) {
    handleCardClick();
  }
});

이벤트 위임의 흐름은 아래 순서로 일어납니다.

  1. 유저가 상품 카드를 클릭한다.
  2. 이벤트 버블링으로 body까지 도달한다.
  3. body의 리스너가 e.target을 보고 분기 처리를 한다.

과제 피드백

1️⃣ 상태 관리의 이해 Observer Pattern을 통해 React의 useState와 Context 개념을 직접 체화했습니다. 불변성 기반의 상태 관리가 왜 필요한지, 그리고 어떻게 렌더링과 연결되는지 스스로 구현하며 배웠습니다.

2️⃣ 비동기 처리 & Race Condition 대응 isFetchingProducts, pendingLoadProducts 플래그를 통해 비동기 로딩 중복을 제어하고 상태 동기화를 유지했습니다.

3️⃣ 테스트 코드의 가치 E2E 테스트에서 발생한 버그를 통해 "사용자 시나리오 단위 테스트"의 중요성을 깨달았습니다. 테스트가 단순한 검증이 아니라 기능 명세의 문서화 역할을 한다는 점을 체감했습니다.

AI 활용 경험 공유하기

이번 과제에서는 개념 정리나 오류 설명뿐 아니라, 구현 단계에서도 AI를 적극적으로 활용하며 개발을 진행했습니다. 특히 “프레임워크 없이 SPA를 직접 구현한다”는 요구사항은 상태 관리·라우팅·렌더링 흐름이 서로 얽혀 있기 때문에 설계 단계에서 많은 고민이 있었고, 이때 AI가 좋은 참고 역할을 해줬습니다.

(1) 기능 설계 및 초기 구조 수립 Observer 패턴 기반의 store 구조를 만들 때, AI가 제안한 기본 설계(중앙 store, subscribe, setState 구조)가 지금 프로젝트의 상태 관리 뼈대가 되었습니다.

이를 참고해 저만의 방식으로 확장했고 subscribe-first 렌더링 흐름을 비교적 빠르게 안정적으로 잡을 수 있었습니다.

(2) Race Condition 디버깅 E2E 환경에서만 발생하던 검색어 초기화 후 상품 목록이 갱신되지 않는 문제를 해결할 때, AI와 함께 로그를 정리하며 원인을 단계적으로 좁히는 과정에서 Race Condition이라는 본질적인 문제를 파악할 수 있었습니다.

플래그 충돌, popstate 중복 호출, isLoaded 조기 종료 등 여러 요소가 얽혀 있었기 때문에 AI의 시각을 빌려 흐름을 다시 정리한 것이 큰 도움이 됐습니다.

(3) AI와 나의 방식 비교 같은 요구사항을 기준으로 AI가 구현했을 접근 방식과 비교해보며 배운 점도 많았습니다.

  • 저는 store.subscribe에서 상태 변화 감지 + 데이터 로딩 로직을 함께 처리하는 방식을 사용했고, 그 과정에서 여러 플래그가 필요해졌습니다.
  • 반면 AI는 라우터(router)와 store 역할을 명확히 분리하고 라우팅에서 직접 action을 호출하는 구조를 제안했습니다.

AI가 제안한 방식은 더 확장성 있고 테스트하기 쉬운 설계였다는 점에서 리팩토링 방향에 대한 인사이트를 얻을 수 있었습니다!

리뷰 받고 싶은 내용

  • 현재 옵저버 기반 상태 관리 로직이 React의 리액티브 구조에 비추어 충분히 추상화된 구조인지 궁금합니다.
