🧐코드 최적화, 왜 해야할까?

최적화를 하는 것의 본질적인 목적은 UX를 개선하기 위해서라고 생각합니다.

결국 웹이든 앱이든 사용자가 있기 때문에 존재하고, 사용자의 존재 유무가 우리가 개발을 하는 이유라고 생각한다면, 타겟인 사용자의 경험을 최적화해주고 개선하는 것이 중요하다고 생각합니다.

결론적으로 최적화는 측정값이 아니라 병목을 줄이는 행위입니다. 수치상으로는 개선되어보일 수 있지만 그 결과가 UX 개선에 유의미한 영향을 주지 않았다면 불필요한 추상화, 마이크로테스크일 수 있습니다.

(3) API 호출 최적화(Promise.all 이해)

(3-1) 기존 문제점

Promise.all이 현재 잘못 사용되고 있습니다.

const fetchAllLectures = async () => await Promise.all([
  (console.log('API Call 1', performance.now()), await fetchMajors()),
  (console.log('API Call 2', performance.now()), await fetchLiberalArts()),
  (console.log('API Call 3', performance.now()), await fetchMajors()),
  (console.log('API Call 4', performance.now()), await fetchLiberalArts()),
  (console.log('API Call 5', performance.now()), await fetchMajors()),
  (console.log('API Call 6', performance.now()), await fetchLiberalArts()),
]);

performance.now는 페이지가 로드되고 나서 해당 호출이 실행되었을 때의 시간을 찍어주는 용도로 실행하는건데 병렬 실행이 보장되었다면 완전히 동일할 필요는 없지만, 직렬 실행에서처럼 누적되는 시간 차이는 발생하지 않아야 합니다.

하지만 콘솔을 보면 차이가 꽤 나는 걸 알 수 있습니다.

❗️코드에서의 문제점 1️⃣ 중복 호출 호출해야 하는 API는 /schedules-majors.json, /schedules-liberal-arts.json 두개인데 불필요하게 3번씩 중복으로 실행하고 있습니다.

const fetchAllLectures = async () => await Promise.all([
  (console.log('API Call 1', performance.now()), await fetchMajors()),
  (console.log('API Call 2', performance.now()), await fetchLiberalArts()),
  (console.log('API Call 3', performance.now()), await fetchMajors()),
  (console.log('API Call 4', performance.now()), await fetchLiberalArts()),
  (console.log('API Call 5', performance.now()), await fetchMajors()),
  (console.log('API Call 6', performance.now()), await fetchLiberalArts()),
]);

2️⃣ 취지에 맞지 않는 사용방식 게다가 Promise.all의 사용 목적에 부합하지 않는 사용 방식입니다.

Promise.all은 객체 안에 있는 코드의 내용을 동시에 실행하고 준비가 되었을 때 그에 대한 결과가 실행되고 넘어가게 하기 위해 사용하는 것인데, Promise.all 안에 await을 사용하게 되면서 개별적으로 비동기 처리를 진행하고 그에 대한 결과는 Promise.all의 내용으로 넘기면서 의미없는 Promise.all이 됩니다.

(3-2) 해결 방법

일단 불필요한 중복 호출을 없애고 Promise.all의 취지에 맞도록 await을 제외해서 코드를 개선했습니다.

const fetchAllLectures = () =>
  Promise.all([
    (console.log("API Call 1", performance.now()), fetchMajors()),
    (console.log("API Call 2", performance.now()), fetchLiberalArts()),
  ]);

개선 효과는 호출 시간을 비교해봤을 때 체감할 수 있습니다.

이전에는 호출당 50ms정도의 차이가 있었다면, 개선 후에는 0.2ms정도의 차이만 있고 총 API 호출에 걸린 시간도 131 => 59로 절반 이상 성능이 향상됐습니다!

(4) SearchDialog 불필요한 연산 최적화

(4-1) 기존 문제점

SearchDialog가 열릴 때마다 내부에 있는 함수들이 실행되는데, 그 때 내부에 있는 연산에 대한 결과값이 변경되지 않았음에도 매번 호출하게 되어있어 불필요한 메모리 낭비를 하고 있습니다.

1️⃣ filteredLectures 팝업이 열릴 때마다 filteredLectures에 getFilteredLectures를 호출하여 엄청난 체이닝 필터 함수를 실행합니다.

2️⃣ allMajors 큰 변동이 일어나지 않을 전공 데이터에 대해서도 매번 new Set으로 재정의하고 있습니다.

const allMajors = [...new Set(lectures.map((lecture) => lecture.major))];

3️⃣ 엄청난 filter 체이닝 불필요하게 엄청난 filter 체이닝을 진행하고 있습니다.

  const getFilteredLectures = () => {
    const { query = "", credits, grades, days, times, majors } = searchOptions;
    return lectures
      .filter(
        (lecture) =>
          lecture.title.toLowerCase().includes(query.toLowerCase()) ||
          lecture.id.toLowerCase().includes(query.toLowerCase())
      )
      .filter(
        (lecture) => grades.length === 0 || grades.includes(lecture.grade)
      )
      .filter(
        (lecture) => majors.length === 0 || majors.includes(lecture.major)
      )
      .filter(
        (lecture) => !credits || lecture.credits.startsWith(String(credits))
      )
      .filter((lecture) => {
        if (days.length === 0) {
          return true;
        }
        const schedules = lecture.schedule
          ? parseSchedule(lecture.schedule)
          : [];
        return schedules.some((s) => days.includes(s.day));
      })
      .filter((lecture) => {
        if (times.length === 0) {
          return true;
        }
        const schedules = lecture.schedule
          ? parseSchedule(lecture.schedule)
          : [];
        return schedules.some((s) =>
          s.range.some((time) => times.includes(time))
        );
      });
  };

(5-2) 해결 방법

1️⃣ useMemo와 useCallback을 활용한 연산 최적화 비용이 큰 연산(색상 맵 생성, 고유 ID 추출)을 useMemo로 메모이제이션하고 함수는 useCallback으로 메모이제이션하여 참조 안정성을 확보합니다.

export const useScheduleTable = ({
  schedules,
  onScheduleTimeClick,
  onDeleteButtonClick,
}: UseScheduleTableProps) => {
  // schedules가 변경될 때만 재계산
  const uniqueLectureIds = useMemo(
    () => extractUniqueLectureIds(schedules),
    [schedules]
  );

  // uniqueLectureIds가 변경될 때만 재생성
  const colorMap = useMemo(
    () => createColorMap(uniqueLectureIds),
    [uniqueLectureIds]
  );

  // 참조는 항상 같지만, 최신 값을 사용할 수 있도록 함
  const onScheduleTimeClickRef = useRef(onScheduleTimeClick);
  useEffect(() => {
    onScheduleTimeClickRef.current = onScheduleTimeClick;
  }, [onScheduleTimeClick]);

  const onDeleteButtonClickRef = useRef(onDeleteButtonClick);
  useEffect(() => {
    onDeleteButtonClickRef.current = onDeleteButtonClick;
  }, [onDeleteButtonClick]);

  // 의존성이 없으므로 항상 같은 참조 유지
  const handleCellClick = useCallback(
    (day: string, time: number) => {
      onScheduleTimeClickRef.current?.({ day, time });
    },
    []  // 빈 의존성 배열로 항상 같은 함수 참조 유지
  );

  // schedules가 변경될 때만 재생성
  const deleteHandlers = useMemo(() => {
    return schedules.map((schedule) => {
      return () =>
        onDeleteButtonClickRef.current?.({
          day: schedule.day,
          time: schedule.range[0],
        });
    });
  }, [schedules]);

  // schedules나 colorMap이 변경될 때만 재계산
  const scheduleBgs = useMemo(() => {
    return schedules.map((schedule) =>
      getScheduleColor(schedule.lecture.id, colorMap)
    );
  }, [schedules, colorMap]);

  return {
    scheduleBgs,
    deleteHandlers,
    handleCellClick,
  };
};

2️⃣ useRef를 활용한 핸들러 참조 안정화 핸들러 함수를 useRef로 저장하여 참조는 항상 같게 유지하고 useEffect로 핸들러 함수를 업데이트합니다.

const onScheduleTimeClickRef = useRef(onScheduleTimeClick);
useEffect(() => {
  onScheduleTimeClickRef.current = onScheduleTimeClick;
}, [onScheduleTimeClick]);

// 항상 같은 참조를 반환하지만, 최신 함수를 호출
const handleCellClick = useCallback(
  (day: string, time: number) => {
    onScheduleTimeClickRef.current?.({ day, time });
  },
  []  // 의존성이 없어서 항상 같은 참조
);

3️⃣ React.memo를 활용한 컴포넌트 메모이제이션 ScheduleTable 컴포넌트를 React.memo로 감싸서 props가 변경되지 않으면 리렌더링을 방지하고 커스텀 비교 함수를 제공하여 핸들러 참조 변경은 무시하고 실제 데이터 변경만 감지하도록 합니다.

export default memo(ScheduleTable, (prevProps, nextProps) => {
  // tableId와 schedules 배열 참조가 변경되지 않았으면 리렌더링 방지
  // 핸들러 함수는 참조가 변경될 수 있지만, schedules가 같으면 리렌더링할 필요 없음
  if (
    prevProps.tableId === nextProps.tableId &&
    prevProps.schedules === nextProps.schedules
  ) {
    return true; // props가 같으면 리렌더링 안 함
  }
  return false; // props가 다르면 리렌더링
});

  • 문제: 드롭 후 데이터 변경으로 인한 리렌더링
  • 원인: schedulesMap 객체 전체 업데이트 → Context 구독 컴포넌트 전체 리렌더링
  • 해결: 세밀한 Context 업데이트 + React.memo + 핸들러 안정화 (여러 단계 필요)

(6-1) 기존 문제점

시간표 블록을 드래그하는동안 다른 시간표 컴포넌트도 리렌더링되고 다른 시간대로 이동하여 드롭했을 때도 모든 시간표가 함께 리렌더링되는 현상이 발생했습니다.

게다가 dev tools로 보여지는 것뿐만 아니라 사용자가 직접 조작하는 인터랙션 구간에서 끊김이 느껴진다는 점이 가장 큰 문제였습니다.

1️⃣ 전역 DnD Context 공유 모든 시간표가 하나의 DnD Context를 공유하고 있었기 때문에 한 시간표에서 드래그 이벤트가 발생하면 모든 시간표가 useDndContext를 구독한 상태로 함께 리렌더링되고 있었습니다.

<ScheduleDndProvider>
  <ScheduleTables />
</ScheduleDndProvider>

즉, 한 곳의 인터랙션이 전체로 전파되는 구조였습니다.

2️⃣ schedulesMap 객체 전체 업데이트 특정 시간표만 변경되었음에도 불구하고 schedulesMap 객체 전체가 새로 생성되면서 => 이를 구독하는 모든 컴포넌트가 리렌더링되는 문제가 있었습니다.

setSchedulesMap((prev) => ({
  ...prev,
  [tableId]: newSchedules,
}));

(6-2) 해결 방법

최종 보스같은 느낌이 들어서 차근차근 하나씩 적용해보면서 개선시켜나갔습니다.

1️⃣단계: 시간표별 독립적인 DnD Context 제공 일단 제일 문제라고 생각했던 모든 시간표가 같은 context를 공유한다는 부분을 개선하기 위해 각 시간표별 독립적인 context를 가질 수 있도록 했습니다.

{tableIds.map((tableId) => (
  <ScheduleDndProvider tableId={tableId}>
    <ScheduleTable ... />
  </ScheduleDndProvider>
))}

결과

  • 드래그 중에는 해당 시간표만 리렌더링됨
  • 다른 시간표는 DnD 이벤트의 영향을 받지 않음 하지만 드롭 이후에는 여전히 다른 시간표까지 리렌더링되는 문제가 남아 있었습니다.

2️⃣단계: Context 구조 개선 - 시간표 단위 상태 관리 각 시간표의 schedules를 독립적으로 업데이트할 수 있도록 Context 구조를 개선했고 변경되지 않은 시간표의 schedules 배열은 참조를 유지하도록 했습니다.

const setSchedules = useCallback((tableId, schedules) => {
  setSchedulesMap((prev) => {
    if (prev[tableId] === schedules) return prev;
    return {
      ...prev,
      [tableId]: schedules,
    };
  });
}, []);

결과

  • 변경된 시간표의 schedules만 갱신됨
  • 불필요한 schedules 배열 생성 감소 하지만 여전히 상위 컴포넌트에서 schedulesMap 객체 자체가 변경되면서 렌더링 체인이 완전히 끊어지지는 않았습니다.

3️⃣단계: 시간표 컴포넌트 단위 분리 각 시간표를 독립적인 컴포넌트로 분리하고 schedules 배열 참조만을 기준으로 메모이제이션했습니다.

const ScheduleTableItem = memo(({ tableId, schedules }) => {
  return (
    <ScheduleDndProvider tableId={tableId}>
      <ScheduleTable schedules={schedules} />
    </ScheduleDndProvider>
  );
}, (prev, next) =>
  prev.tableId === next.tableId &&
  prev.schedules === next.schedules
);

결과

  • 변경되지 않은 시간표는 리렌더링되지 않음
  • 드롭 시에도 영향을 받는 범위가 명확해짐

Q. 실무에서의 성능 최적화 진행 방식에 대해 궁금합니다.

A. 말씀하신 내용이 모두 맞다고 할 수 있을 것 같아요. 애초에 고려해야 할 성능은 당연히 고려해야하고, 성능에 민감할 수 있는 코드는 조금 더 성능을 고려해서 개발을 해요. 그렇게하고 배포 이후 성능 이슈가 발견되면 성능 개선 업무로 별도로 진행을 합니다 :) 어느 수준까지라고 한다면, 일단 할 수 있는 것까진 하고 실제 테스트시 성능에 문제가 없고 더이상 할 것도 안보이면 멈추는 것이죠 다만 테스트시 성능에 문제가 아직도 있다면 해결할때까지 파야죵.. 문제가 없다고 판단하는 기준은 유저가 사용했을때 문제가 없는지 쾌적한지가 기준인 것 같아요. 만약 그기준을 룰로 만들고 싶다면, 룰로 정해도 좋을 것 같습니다. 실제로 예를들어 LCP는 어느정도까지 나와야한다 이런 기준이 있는 회사들이 많습니다.

Q. 성능 최적화를 위한 CI 설계에 대해 궁금합니다.

A. CI에서하는 경우도 있지만 사실 CI중에 라이트하우스를 통해 진행하는 성능 확인으로는 모든 것을 커버하진 못하는 것 같아요. 개발자 혹은 협업자들에게서 혹은 유저에게 오는 VOC를 발견되는 경우도 많아요. 그럴때 다시 과제로 만들어 해결될때까지 개발합니다. 조금 더 정확히 확인하려면 테스트케이스가 성능을 측정하고 성능에 의해 테스트가 깨질 수 있게 만들면 될 것 같습니다. 성능에 민감한 코드일수록 도움이 될 것 같습니다.
