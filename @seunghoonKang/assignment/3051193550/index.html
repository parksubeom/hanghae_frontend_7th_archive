1. 공통화돼있지 않음

2. UI 컴포넌트와 로직이 같이 있음


  // 🚨 Bad Practice: Table 컴포넌트가 도메인별 렌더링 로직을 알고 있음
  const renderCell = (row: any, columnKey: string) => {
    const value = row[columnKey];

    // 도메인별 특수 렌더링
    if (entityType === 'user') {
      if (columnKey === 'role') {
        return <Badge userRole={value} showIcon />;
      }
      if (columnKey === 'status') {
        // User status를 Badge status로 변환
        const badgeStatus =
          value === 'active' ? 'published' :
          value === 'inactive' ? 'draft' : 'rejected';
        return <Badge status={badgeStatus} showIcon />;
      }
      if (columnKey === 'lastLogin') {
        return value || '-';
      }
      if (columnKey === 'actions') {
        return (
          <div style={{ display: 'flex', gap: '8px' }}>
            <Button size="sm" variant="primary" onClick={() => onEdit?.(row)}>
              수정
            </Button>
            <Button size="sm" variant="danger" onClick={() => onDelete?.(row.id)}>
              삭제
            </Button>
          </div>
        );
      }
    }
  • 앞서 주석으로 Bad Practice라고 적혀있듯, 위의 테이블 예시에서 user일 때 어떤 badge와 어떤 button 을 그리라고 적혀 있는걸 볼 수 있습니다. 이는 추후 확장성면에서 매우 곤란한 일입니다.
  • 가령, user, post가 아닌 다른 내용으로 Table을 구성한다면, 그 안에 또 다시 조건문을 넣고, 조건에 따라 렌더링하게 될텐데 늘어나면 늘어날수록 더 재사용성 어렵고 유지보수하기 힘든 컴포넌트가 된다 생각합니다.

개편 과정에서 집중한 부분

1. 디자인 시스템에 집중하기

저는 토큰화와 토큰화를 통한 디자인 시스템을 구축하는 것에 조금 더 신경썼습니다.

  • 처음 3계층(3tier-model)을 보고, 토큰을 계층화하는 것이 왜 중요할까? 를 많이 고민했습니다.

  • 다른 여러 동료들과의 대화에서도, 굳이 semantic 넘어 component까지 계층화를 할 이유가 있을까? 이건 규모에 따라 스스로 정할 수 있는게 아닐까? 라는 이야기도 주고 받았습니다.

  • 저도 처음 제대로 구현해보는 입장에서, 어떤 방법이 더 좋을지에 대해 생각하다 일단 3계층 모델을 작성해볼까? 생각하며 작성했습니다.

왜 3계층 모델?

  1. 명확성 / 일관성
  • 가령 색상이 더 많아졌을 때, 30~40개, 혹은 그 이상을 정의하려한다면, button-primary와 같이 단순한 이름으로 명명한다면, 그 외 어떤 이름으로 정확히 이름을 짜서 지을 것인가? 에 대한 고민이 들었고, 그렇기에 더 세부적인 내용으로 작성하였습니다.
  • 그렇다면 그 색상이, 그 크기가 정확히 어디에 쓰이는것인지 이름만 보고도 알 수 있다 생각합니다.

2. 유지보수

  • 또 이렇게 --color-button-primary-bg 로 지은 부분이 왜 bg-primary 보다 나은것인가 생각했습니다.
  • 그 이유로는, 추후 색상 변경에 대한 유지보수가 원활히 된다는 점입니다.
    • 가령 카드에도 bg-primary를, 버튼에도 bg-primary를 적용했을 때, 몇 개월이 지나고 primary 색상을 gray-500으로 합시다. 라고 한다면 카드와 버튼 모두에게 bg-primary가 변경될것입니다. 이렇게 두 개가 아니라 수 십, 수 백개가 동일한 bg-primary를 지니고 있다면, 전부 다시 다르게 변경해야 한다는 점입니다.
    • 그렇지만 버튼에는 --color-button-primary-bg 로, 카드에는 --color-card-primary-bg: var(--color-blue-100); 로정의한 값을 사용한다면, 나중에 변경됐을 때 버튼의 정의된 값만 변경해도 버튼만 변할 수 있습니다 !

라고 생각했는데, 이유가 충분히 타당하지 못하다고 생각하여 취소선을 그었습니다. 🫠

사용한 기술 스택 경험

  • tailwind, shadcn

어려웠던 점

1. tailwind의 namespace 충돌

  • 토큰으로 정의할 때, tailwind는 namespace를 제공하며, namespace에 새 테마 변수를 정의하면 프로젝트에서 해당 유틸리티와 변형을 사용할 수 있게 됩니다. 
// semantic/colors.css
  --color-button-primary-text-color: var(--color-white);

// semantic/typography.css
  --text-size-button-sm: var(--text-sm);

// button.tsx
 variants: {
      variant: {
        primary:
          "bg-button-primary-bg text-button-primary-text-color hover:bg-button-primary-bg-hover",

      size: {
              default: "h-9 px-4 py-2 has-[>svg]:px-3",
              sm: "h-8 gap-1.5 px-3 has-[>svg]:px-2 text-size-button-sm",

      defaultVariants: {
          variant: "primary",
          size: "sm",
        },

버튼에 적용된 cva 부분입니다. 보이는 부분 중 primary의 text-button-primary-text-color 와 size sm의 text-size-button-sm으로 작성 후 확인해보면,

이렇게 semantic으로 작성한 두 개의 text 네임스페이스가 충돌하게되어 마지막으로 작성된 내용만 반영됨을 볼 수 있었습니다. 그래서 결국 사이즈 부분은 그림의 마지막과 같이 semantic 작성한 내용이 아닌, tailwind가 기본 제공하는 text-sm을 적용하였습니다.

2. 자유도 높은 프로젝트

  • 처음에는 before의 형태 전체를 동일하게 디자인 시스템으로 작성해야하는 줄 알았습니다. 일요일, 월요일동안 어떻게하면 before 형태에서 디자인을 추출해서 사용할까를 많이 고민했습니다.
  • Q&A 시간에 똑같이 따라할 필요 없고, 다른 곳에서 어떻게 쓰는지 베껴와도 된다! 라는 말과, 기존 디자인 똑같이 따라하려고 이상하게 짜여진 사이즈 같은경우를 토큰화 하려는데, 그걸 굳이 토큰화한다는건 오히려 좋지 않은 것! 디자인 시스템 컨벤션에 어울리지 않는것이라 할 수 있다. 라는 말을 듣고 다시 토큰화에 대해 고민을 하게 됐습니다..
  • 이런 부분에 시간을 많이 쏟게 되며, 결국 모든 컴포넌트를 shadcn으로 만드는 걸 할 수 없었습니다.
  • 그렇지만 디자인 시스템이라는, 기준이 되는 것을 만들어야한다는 것에 대해 다시 한 번 생각해볼 수 있는 계기가 되었습니다..

tailwind의 namespace 충돌을 어떻게 해결할 수 있을까요?

이건 tailwind의 규칙을 따르는게 최선이라고 생각해요. 아니면 이렇게 시도해볼 수 있을 것 같아요.

// css
@theme {
  /* Base Colors */
  --brand-500: #3b82f6;
  --brand-600: #2563eb;

  /* Semantic Tokens */
  --color-primary: var(--brand-500);
  --color-primary-hover: var(--brand-600);

  --color-text-default: #111827;
  --color-text-muted: #6b7280;

  --color-bg-default: #ffffff;
  --color-bg-muted: #f9fafb;
}

// tsx
<button className="px-4 py-2 rounded bg-[var(--color-primary)] text-white hover:bg-[--color-primary-hover]" >
  Save
</button>

혹은 tailwindcss 에 종속적이지 않은 별개의 테마 시스템을 만들어서 사용하는거죠 ㅎㅎ emotion같은걸 이용해서 시도해볼 수 있답니다!

3계층 토큰화, 정말 필요할까요?

일단 1~2계층은 "전사"를 기반으로 사용하고, 3계층은 "프로덕트" 기반으로 정의해서 사용하는거죠.
