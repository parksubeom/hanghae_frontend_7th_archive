[5팀 손승현] Chapter 1-2. AI와 테스트를 활용한 안정적인 기능 개발

by sonsonsh11252025.10.31

과제 체크포인트

필수 스펙

  1. 반복 유형 선택

    • 일정 생성 또는 수정 시 반복 유형을 선택할 수 있다.
    • 반복 유형은 다음과 같다: 매일, 매주, 매월, 매년
    • 31일에 매월을 선택한다면 → 매월 마지막이 아닌, 31일에만 생성하세요.
    • 윤년 29일에 매년을 선택한다면 → 29일에만 생성하세요!
    • 반복일정은 일정 겹침을 고려하지 않는다.

  2. 반복 일정 표시

    • 캘린더 뷰에서 반복 일정을 아이콘을 넣어 구분하여 표시한다.

  3. 반복 종료

    • 반복 종료 조건을 지정할 수 있다.
    • 옵션: 특정 날짜까지
      • 예제 특성상, 2025-12-31까지 최대 일자를 만들어 주세요.

  4. 반복 일정 수정

    1. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      • 반복일정을 수정하면 단일 일정으로 변경됩니다.
      • 반복일정 아이콘도 사라집니다.
    2. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      • 이 경우 반복 일정은 유지됩니다.
      • 반복일정 아이콘도 유지됩니다.

  5. 반복 일정 삭제

    1. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      1. 해당 일정만 삭제합니다.
    2. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      1. 반복 일정의 모든 일정을 삭제할 수 있다.

기본 과제

공통 제출

  • 테스트를 잘 작성할 수 있는 규칙 명세
  • 명세에 있는 기능을 구현하기 위한 테스트를 모두 작성하고 올바르게 구현했는지
  • 명세에 있는 기능을 모두 올바르게 구현하고 잘 동작하는지

기본 과제(Easy)

  • AI 코드를 잘 작성하기 위해 추가로 작성했던 지침
  • 커밋별 올바르게 단계에 대한 작업
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

심화 과제

  • 모든 질문에 대해 꼼꼼하게 정리했는지
  1. 사용하는 도구를 선택한 이유가 있을까요? 각 도구의 특징에 대해 조사해본적이 있나요?

  • 기존에는 ChatGPT를 사용하였으나 맥락 인식은 좋으나(애매한 질문에도 말을 제대로 알아먹고 대답) 깊은 로직에 대해 이해력이 부족하다고 생각하였습니다. 기존에 진행한 사전스터디 팀원분께서 claude를 잘 활용하시는걸 보았으나 CLI 보다는 IDE 전체를 적용할 수 있는 Cursor를 선택하게 되었습니다. 여러 모델을 선택해서 사용가능하고 전체적인 파일 구조를 읽는데 효과적이라 판단하였습니다.

  • 이에 관련하여 개인 벨로그에 글을 작성하였습니다. (https://velog.io/@tmdgus8911/%ED%85%8C%EC%8A%A4%ED%8A%B8-%EC%BD%94%EB%93%9C-%EC%9E%91%EC%84%B1%EC%9D%84-%EC%9C%84%ED%95%9C-%EB%82%B4-%ED%94%BD%EC%9D%98-AI-%EC%B0%BE%EA%B8%B0%EB%B6%80%EC%A0%9C-AI-%ED%94%84%EB%A1%9C%EB%93%80%EC%8A%A4-101)

  1. 테스트를 기반으로 하는 AI를 통한 기능 개발과 없을 때의 기능개발은 차이가 있었나요?
  • AI를 사용하기 전에는 직접 함수들을 확인하며 테스트를 작성해서 속도면에서는 늦었지만 코드에 대한 체화는 잘 되었습니다. 반면 AI를 통해서 작업시에는 속도면에서는 효과적이었으나 내부 동작을 깊게 이해하지 않고 넘어간 부분도 있기에 코드에 대한 학습 효과가 적었습니다. 이로인해 디버깅시 확인하는 작업에서 시간소요가 많이 되었습니다. 또한 AI를 통해서 작업시 의도치 않게 코드의 복잡도가 과잉된 부분도 존재했습니다.
  1. AI의 응답을 개선하기 위해 추가했던 여러 정보(context)는 무엇인가요?
  • TDD의 작성방법 및 각 사이클에 대한 검증에 대한 설명을 추가하였고 각 사이클에서 어떻게 작업을 해야하며 피해야할 오류 검증들에 대해서 정보를 추가하였습니다. 또한 현재 작업 환경에 대해 프레임 워크 및 라이브러리에 대한 설명을 추가하였습니다. 이러한 부분을 cursorrules에 작성하긴 했는데 요구사항이 제대로 적용됐는지 확인하기 어려웠습니다.
  1. 이 context를 잘 활용하게 하기 위해 했던 노력이 있나요?
  • 프레임워크 특화 도구를 사용하게 하였으며 테스트 구조에 대해 설명하면서(AAA/GWT) 구조 기반으로 테스트 코드를 작성하도록 지시하였습니다.
  1. 생성된 여러 결과는 만족스러웠나요? AI의 응답을 어떤 기준을 갖고 '평가(evaluation)'했나요?
  • 가장 기본적으로는 테스트가 통과하는지를 기준을 삼았습니다. 추가적으로 코드의 정확성을 기본적으로 체크하였으며 지시된 명세서 대로 작동을 하는지 또는 테스트 구조를 지키며 작성했는지 평가하였습니다. 또한 변수명과 함수명에 대한 가독성 및 터무니없이 리소스 사용이 비효율적인지에 대한 판단도 하였습니다.
  1. AI에게 어떻게 질문하는것이 더 나은 결과를 얻을 수 있었나요? 시도했던 여러 경험을 알려주세요.
  • persona에 대한 정확한 설명 및 부정형 설명보다는 해야하는 것에 대해서 정확히 지시하였습니다. 또한 마지막까지 원하는 수준까지 도달하지 않으면 실행 조건을 정확히 명시하였습니다.
  1. AI에게 지시하는 작업의 범위를 어떻게 잡았나요? 범위를 좁게, 넓게 해보고 결과를 적어주세요. 그리고 내가 생각하는 적절한 단위를 말해보세요.
  • 범위를 넓게 하였을때는 불필요한 라이브러리 도입을 하거나 디버깅시 오류 부분을 찾아내기 힘들었고, 반면 좁게 하였을때는 구체적인 코드를 작성하였으나 전체 흐름이 생략된 경우가 존재 하였습니다. 적절한 단위는 하나의 기능 단위로 작업 범위를 하는것이 흐름판단에 있어서 적절하다고 생각하였습니다.
  1. 동기들에게 공유하고 싶은 좋은 참고자료나 문구가 있었나요? 마음껏 자랑해주세요.
  • 발제에 있는 awesome-cursorrules가 많이 도움이 되었습니다. persona 설정 및 프레임워크별 설정을 참고 할 수 있었습니다.
  1. AI가 잘하는 것과 못하는 것에 대해 고민한 적이 있나요? 내가 생각하는 지점에 대해 작성해주세요.
  • 설명을 동반한 코드 작성 및 패턴화된 반복 코드 작업은 퀄리티가 좋으나 코드의 일반화가 심하고 예외 케이스에 대한 커버리지가 좁은것 같습니다.
  1. 마지막으로 느낀점에 대해 적어주세요!
  • easy 과제로 선택하여 cursor의 도움으로 정말 금방 끝낼줄 알았던 과제인데 에이전트 작업 제외한 나머지 부분을 AI 응답으로 처리하는 것도 제가 인간 agent가 된 기분이었습니다. 또한 원하는 답변이 안나오거나 통과하지 못하는 테스트를 답변했을때 이걸 수정하거나 추가하는 것이 이 과제의 의도가 맞을까 라는 생각이 들었습니다.

TDD는 테스트 기술이 아니라 코드 기술이다.

  • 테스트 코드 관련 글 중에 인상 깊은 문구가 있었습니다. TDD는 결국 부수적인 효과가 테스트 기술이며 근간은 코드 기술이라는 의미로 받아들였는데 각 코드 구조에 대한 이해가 기본이며 효율적인 함수 및 코드구성은 결국 성공적인 테스트를 이끄는것 같습니다.

과제 셀프회고

기술적 성장

TDD에 대한 전반적인 이해가 가능해졌습니다. 과제 이전에는 테스트 코드와 TDD는 동일하다는 것이라 생각하였으나 구분하여 이해하게 되었고 또한 cursor rules를 활용하여 agent 작성이 가능하다는 것도 처음알게 된 개념 이었습니다.

코드 품질

AI를 통한 코드 품질이 제가 생각한 기대만큼 품질이 좋지 못했습니다. 특히 유닛 테스트를 통과하는 것은 어렵지 않았으나 통합 테스트에서 해당 명세에 대한 설명에 AI를 통해 제대로 작성되지 않는듯한 느낌이 들었습니다. Cursor를 선택한 이유는 코드를 확인할때 단편적인 부분 말고 전체적인 맥락을 이해하기를 기대하였으나 App.tsx를 참조해서 수정해야하는 것을 전혀 갈피를 찾지 못한다는 생각이 들었습니다.

학습 효과 분석

오히려 이번주차에서 더 많이 학습한건 TDD인것 같습니다. 각 단계에서 어떠한 것을 수행하는지 애매한 개념만 있었는데 해당 과제를 통해서 각 단계별에서 해야할 작업들에 대해 제대로 익혔습니다. 특히 Red Cycle 이후에는 테스트 코드는 절대 건드리지 않고 함수 구현만 수정해야 한다는 기본 개념도 체득했습니다.(저에게는 이게 좀 애매한 개념이었습니다...)

과제 피드백

lint 에러같은 경우는 리팩토링 단계에서 수행해야하는지 Green 단계에서 수행해야 하는지 모호한 부분이 있었습니다. 또한 TDD에서는 GWT 보다는 AAA가 더 유용하다고 생각하는데 이부분도 혹시 어떻게 생각하시는 궁금합니다.

만약 Green cycle에서 작업 중 Red Cycle에서 작성한 테스트 코드 자체에 오류가 있다는 생각이 든다면 계속 무한 회귀해야 하나요? 이 굴레에서 몇번을 헤맨것 같습니다.
