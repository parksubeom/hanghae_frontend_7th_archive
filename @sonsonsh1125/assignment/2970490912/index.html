[5팀 손승현] Chapter 1-3. 완성도있는 테스트 전략 수립하기

by sonsonsh11252025.11.02

과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리

당장은 적절한 스토리를 작성하는데 더 집중해주세요.

우아한 설계는 이후 주차때 좀 더 집중해봐요.

- 물론, AI를 활용해도 좋습니다.
- 기존에 작성한 테스트는 깨지지 않도록 해주세요.
  - 이 과정에서 테스트가 분리될 수도 작성법이 수정될 수도 있습니다.
  - 정말 불필요한 케이스가 아니면, 테스트 케이스를 삭제하는건 최소화 해주세요.
  - 추가로 필요한 테스트가 생길 수 있지만, 작성은 자유롭게 선택해서 진행해주세요.

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.
  1. 단위테스트 목적: 훅(useEventForm, useEventOperations)의 상태 전이·검증 로직 신뢰 확보 전략: pure logic만 테스트, 분기별 로직(시간 검증, 반복 토글, 비동기 처리) 커버 효과: 빠른 피드백 제공, E2E 복잡도 감소
  2. 통합테스트 목적: UI 이벤트와 비즈니스 로직의 연결 검증 전략: React Testing Library + MSW로 사용자 상호작용(DnD, 모달 흐름 등) 중심 테스트 효과: 실제 유저 흐름 기반 신뢰 확보
  3. E2E Test 목적: 브라우저 환경에서 전체 사용자 시나리오 검증 전략: Playwright로 일정 추가 → 수정 → 이동(DnD) → 겹침 처리 등 주요 흐름 검증 효과: 실제 사용자의 “기능적 신뢰” 확보
  4. Visual Regression Test 목적: UI 변경으로 인한 시각적 회귀 방지 전략: Playwright toHaveScreenshot() 기반 캡처 비교, 안정된 시점(후반)에 적용 효과: “시각적 신뢰” 확보 및 QA 효율 향상
  5. 비중 Static (10%) → 단위 테스트 (20%) → 통합테스트 (45%) → E2E (20%) → VRT (5%)
  6. 각 테스트별 상호 보완

  • **단위(Unit)**는 모든 것의 가정(assumption)이다. 훅·함수의 계약을 보장하면 상위 레이어에서 “로직이 틀려서” 깨질 가능성을 크게 줄여준다.

  • **통합(Integration)**은 경계(컴포넌트 ↔ 훅 ↔ API) 를 검증한다. 단위가 신뢰된다는 전제 하에 UI 이벤트와 네트워크 흐름이 결합될 때 의도대로 동작하는지 확인한다.

  • E2E는 종단간(사용자 관점) 신뢰를 검증한다. 통합에서 검증한 흐름들이 실제 브라우저 환경에서 합쳐질 때 깨지지 않는지 확인한다.

  • VRT는 시각적 신뢰의 최종 방패다. 기능이 맞더라도 레이아웃·스타일이 깨질 수 있으니 별도로 잡아둔다.

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요. https://www.figma.com/board/SwcN2Sgc2ksMtjCcPzCImn/%ED%95%AD%ED%95%B4?node-id=0-1&p=f&t=yECx9JBVRyGY6YHl-0

  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

  1. src/tests/hooks/useEventForm.spec.ts: useEventForm 훅이 시간 검증을 즉시 반영하고, resetForm과 editEvent로 상태를 확실히 재설정/주입하는지 단위 레벨에서 보장. 이렇게 훅 내부 계약이 굳어지면 상위 UI 테스트 부담이 줄어든다.
  2. src/tests/hooks/medium.useEventOperations.spec.ts : useEventOperations 훅의 반복 일정 생성 로직(createRepeatEvent)과 성공 토스트 옵션 분기, 실패 시 에러 토스트 방출까지 검증하기 위해 테스트 케이스 3개를 확장. API 응답을 MSW로 통제하면서 훅이 이벤트 리스트를 재로딩하고 메시지를 올바르게 내보내는지 확인.
  3. src/tests/integration/recurringEventWorkflow.spec.tsx : 실제 UI에서 드래그 앤 드롭으로 일정 이동, 반복 일정 모달에서 ‘단일 이동’ 선택 흐름을 검증하려고 두 개의 통합 테스트를 추가. App 전체를 렌더해 사용자 액션→훅 호출→토스트/화면 업데이트까지 이어지는 체인을 검토하는 목적.

과제 셀프회고

기술적 성장

이번 과제를 진행하면서 정말 새로운 개념들을 많이 접했습니다. 특히 E2E 테스트라는 개념 자체가 생소했고, 이를 구현하기 위한 Playwright, Chromatic 등의 도구를 처음 사용해보면서 많은 시행착오를 겪었습니다. 단순히 코드를 작성하는 수준이 아니라, “E2E 테스트가 왜 필요한가”, “Playwright가 다른 테스트 도구와 비교했을 때 어떤 장점이 있는가”, “어떤 구조로 작성해야 가장 효율적인 테스트가 될까” 등을 고민하며 공식 문서를 반복해서 읽었습니다. 덕분에 단순히 기술을 ‘사용’하는 데서 그치지 않고, 테스트의 본질적인 목적과 구조적 사고를 익히는 데 집중할 수 있었습니다.

코드 품질

E2E 테스트를 작성하면서 가장 고민이 많았던 부분은 유닛 테스트와의 역할 구분이었습니다. ‘이건 유닛 테스트로 해야 할까, 아니면 진짜 사용자의 흐름으로 봐야 할까?’라는 경계가 생각보다 명확하지 않아서, 테스트의 목적을 계속 스스로 점검해야 했습니다. 결국 사용자 입장에서의 시나리오를 기준으로 테스트를 설계하려고 노력했고, “실제 유저라면 이 버튼을 눌렀을 때 이런 행동을 기대하겠지?” 하는 흐름으로 작성했습니다. 다만 코드가 길어지고 셀렉터가 중복되는 부분은 조금 더 리팩토링이 필요할 것 같습니다. 앞으로는 테스트 내에서의 핸들러 추상화나 공통 유틸화를 통해 코드 품질을 높이고 싶습니다.

학습 효과 분석

E2E 테스트를 작성하다 보니, 실제 회사의 QA 프로세스와의 연계나 자동화된 품질 관리에 대한 감이 조금씩 잡혔습니다. 솔직히 처음엔 ‘이 정도면 QA팀 없어도 되지 않나?’ 싶었을 정도로 테스트 자동화의 가능성이 흥미로웠습니다. 다만 아직은 셀렉터 선택이나 비동기 처리 관련 에러 핸들링 등, 좀 더 실무적인 부분의 숙련도가 필요하다고 느꼈습니다.

과제 피드백

테스트를 여러 번 실행하면서, 테스트 실행 순서에 따라 결과가 달라지는 현상이 있었습니다. 처음엔 코드 문제라고 생각했지만, 알고 보니 상태가 공유되거나 초기화되지 않는 부분이 원인이었다고 생각합니다. 또한 VSCode Playwright 플러그인을 통해 테스트를 개별로 실행할 때와 전체를 한 번에 돌릴 때 결과가 달라서 꽤 혼란스러웠습니다. 테스트 환경 자체의 불안정함도 있었고, 브라우저 캐시나 세션 관리가 영향을 미치는 부분도 있어서, 실제 CI/CD 환경에서의 테스트 안정성을 더 고민하게 됐습니다.

시각적 회귀 테스트(Visual Regression Test) 도 흥미로운 개념이었지만, 학습 시간 대비 이해해야 할 내용이 많아서 놓친 부분이 많았습니다. 단순히 스냅샷 비교로 끝나는 게 아니라, 어떤 기준으로 “변화”를 판단할지 명확히 정의하는 과정이 필요하다는 걸 느꼈습니다.

리뷰 받고 싶은 내용

E2E 테스트를 작성하면서 가장 어려웠던 부분은 셀렉터 선택과 핸들링 로직의 정확성입니다. 특히 getByRole, getByTestId, locator 등을 어떻게 선택해야 가장 안정적인 테스트가 될지 확신이 서지 않았습니다. 테스트가 자주 깨지는 이유 중 상당수가 셀렉터의 모호함이었는데, 어떤 기준으로 셀렉터를 설계해야 하는지, 그리고 테스트의 내구성을 높이기 위해 어떤 접근이 좋은지에 대한 피드백을 받고 싶습니다. 또한, 테스트를 모듈 단위로 나누는 방식이 현재 구조적으로 적절했는지도 검토받고 싶습니다.

과제 피드백

승현님 고생하셨습니다! 아쉽게 e2e 테스트가 정상적으로 통과하지 않네요 ㅠㅠ 스토리북은 잘 작성해주셨습니다. 질문 주신 관점에서 조금 더 덧붙여 보면 playwright도 다른 테스트와 동일하게 유저 시나리오 관점에서 구현에 따라 수정이 될 수 있는 테스트가 아닌 명세 기반으로 잘 동작하도록 선택자를 선택하는게 중요하다고 생각해요. 그리고 css나 DOM에 직접 접근하기 위한 선택자를 사용하게 될 수도 있는데 그런 경우보다는 test id를 사용하는게 더 나은 상황이 있을 수도 있습니다. 이 부분은 오늘 리뷰 시간 때 자세히 설명 드린것 같아서 참고해보시면 좋을것 같아요 ㅎㅎ 그리고 선택자는 종종 테스트에 wait을 걸거나 catch를 걸게 되면 발견할 수 있는 에러를 발견할 수 없기 되거나, 시간이 매우 늘어나게 되는 상황이 만들어 질 수 있어요. 정말 필요한 케이스들도 있어서 절대 사용하지 마세요! 라고는 말씀은 못드리겠지만, 이걸 절대 남발해서는 안됩니당.
