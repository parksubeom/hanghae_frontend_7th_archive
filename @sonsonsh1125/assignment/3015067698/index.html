[5팀 손승현] Chapter 2-2. 나만의 React 만들기

과제 체크포인트

배포 링크

https://sonsonsh1125.github.io/front_7th_chapter2-2/

기본과제

가상돔을 기반으로 렌더링하기

  • createVNode 함수를 이용하여 vNode를 만든다.
  • normalizeVNode 함수를 이용하여 vNode를 정규화한다.
  • createElement 함수를 이용하여 vNode를 실제 DOM으로 만든다.
  • 결과적으로, JSX를 실제 DOM으로 변환할 수 있도록 만들었다.

이벤트 위임

  • 노드를 생성할 때 이벤트를 직접 등록하는게 아니라 이벤트 위임 방식으로 등록해야 한다
  • 동적으로 추가된 요소에도 이벤트가 정상적으로 작동해야 한다
  • 이벤트 핸들러가 제거되면 더 이상 호출되지 않아야 한다

심화 과제

Diff 알고리즘 구현

  • 초기 렌더링이 올바르게 수행되어야 한다
  • diff 알고리즘을 통해 변경된 부분만 업데이트해야 한다
  • 새로운 요소를 추가하고 불필요한 요소를 제거해야 한다
  • 요소의 속성만 변경되었을 때 요소를 재사용해야 한다
  • 요소의 타입이 변경되었을 때 새로운 요소를 생성해야 한다

과제 셀프회고

아하! 모먼트 (A-ha! Moment)

  • 과제 작업 후 E2E 테스트 작동시 오류가 나온 상황에서 디버깅중 다시한번 테스트코드의 필요성에 대해 알게 되었습니다. 역으로 함수를 작성할때 테스트 코드를 참고 했어야 했는데 일단 작성해보자 라는 마음이 급급해 테스트 코드를 무시한채 작성하였고 이로인해 타입별 처리에 대한 부분이 누락이 되었는데 다시 한번 테스트 코드를 보고 수정하였습니다.

  • renderElement에서 “DOM에 노드가 있는데 이전 vNode가 없으면 왜 다시 전부 갈아엎지?”라는 의문에서 출발해, “Diff는 DOM이 아니라 이전 vNode에 의존한다”는 사실을 깨닫고 조건을 oldVNode 하나로 단순화했습니다. 이걸 이해한 순간, 정렬 이후 화면이 리셋되던 이유가 명확하게 풀렸습니다.

기술적 성장

  • Virtual DOM 업데이트 흐름을 순서대로 추적하면서 “정규화 → 렌더 → diff → 이벤트 유지” 전체 라이프사이클을 머릿속에 그릴 수 있게 됐습니다.
  • updateElement 케이스 분기를 정리하면서, 텍스트·타입 변경·속성 업데이트·자식 업데이트 등 각각을 독립적으로 다루는 패턴이 몸에 익었습니다.

코드 품질

  • 중복 정규화 제거와 조건 단순화 덕분에 렌더 루프가 한결 안정적이고 예측 가능해졌습니다. 상태 변화마다 전체 DOM을 재생성하지 않으니 이벤트 위임, 무한 스크롤, 정렬 등 UI 상호작용도 자연스럽게 이어집니다.
  • containerVNodeMap에 현재 vNode를 꼭 저장해 두는 흐름으로 바꾸면서 다음 렌더 때 diff 기준이 항상 준비되도록 만들었습니다.

학습 효과 분석

  • E2E 실패 시나리오(정렬 옵션) → router.query 변경 → render → diff → API 재호출 흐름을 손으로 따라가며 디버깅하는 과정을 통해 SPA에서 상태·라우팅·렌더링이 얽히는 방식을 체감했습니다.

과제 피드백

  • Virtual DOM 과제의 핵심은 “조금 나쁜 조건 하나”가 전체 UI 흐름을 깨뜨릴 수 있다는 점을 직접 경험하게 해 줬습니다. 덕분에 상태를 저장하는 위치, diff의 입력, 이벤트 위임 유지 등 세부 구현이 왜 중요한지 체감할 수 있었습니다.

리뷰 받고 싶은 내용

  • 이벤트 위임 구조(setupEventListeners, addEvent, removeEvent)가 실제 대규모 UI에서도 문제없이 유지될 수 있을지, 혹시 놓친 메모리/성능 리스크가 현재 구상한 함수에서 있을지 궁금합니다.

데일리 회고

createVNode의 역할을 모르겠다(251116)

createVNode의 역할은 가상 DOM 노드(Virtual DOM Node)를 만들어주는 함수며 HTML 태그나 컴포넌트 구조를 자바스크립트 객체로 표현 해준다. 즉, createVNode를 호출하면 **"이런 구조의 화면을 만들고 싶다"**를 JS 객체로 남겨놓는다. 단지 구조를 데이터(객체)로 저장하는 역할만 하고 "렌더링"이나 "실제 DOM 생성"은 나중에 다른 함수가 한다! 우리가 해야 할 건 createVNode 가 “어떤 vNode 객체를 리턴할지” 정하는 것

type: 태그 이름 또는 함수 컴포넌트 props: 속성 객체 (없으면 null 또는 {}) children: 나머지 인자들 (텍스트, 다른 vNode 등)

updateAttributes(251117)

다른 함수들은 그래도 노션에 정리되어있는 부분으로 작성할수 있지만 updateAttributes 이부분은 AI 도움을 피할수 없었다. 애초에 updateAttributes 이게 뭔 역할인가 싶었으나 jsx로 되어있는 부분들을 실제 HTML 적용하게 하는 함수 였다!

제 E2E가 무슨 문제라도(251119)

renderElement.js의 핵심 개선

  1. DOM이 있지만 vNode가 없으면 초기 렌더링으로 처리되어 DOM이 초기화됨
  2. 정렬 변경 시 productStore 업데이트 → render() 호출 → renderElement 호출
  3. 이때 DOM은 있지만 vNode가 없으면 초기 렌더링으로 잘못 처리될 수 있음

updateElement.js의 핵심 개선

  1. renderElement에서 이미 정규화했는데 다시 정규화
  2. 중복 정규화로 인한 불필요한 연산과 잠재적 버그
  3. 정규화 과정에서 객체 참조가 바뀌어 diff 비교가 부정확할 수 있음

DOM 초기화 문제: existingNode 체크로 인해 잘못된 초기 렌더링 발생 상품 목록 DOM이 초기화되어 이벤트 리스너 손실 diff 비교 부정확: 중복 정규화로 객체 참조가 달라져 diff가 제대로 동작하지 않음 상품 목록 업데이트가 누락되거나 오작동 vNode 저장 누락: 일부 경로에서 vNode 저장이 누락되어 다음 렌더링에서 diff 실패

과제 피드백

수고했어요 승현! 기본 과제랑 심화 과제 모두 잘 해냈네요.

데일리 회고를 보니까 매일 학습 과정을 기록하면서 진행했군요. "createVNode의 역할을 모르겠다"에서 시작해서 "Diff는 DOM이 아니라 이전 vNode에 의존한다"까지 도달한 과정이 잘 보여요. 특히 정렬 이후 화면이 리셋되던 문제를 디버깅하면서 이 핵심을 깨달은 게 좋았습니다. 이런 삽질이 진짜 학습이에요.

"일단 작성해보자는 마음에 테스트 코드를 무시했다가 타입별 처리가 누락됐다"는 회고도 좋았어요. 테스트 코드는 "검증 도구"이기도 하지만 "명세서"이기도 하거든요. 다음부터는 테스트부터 읽고 시작하는 습관을 들이면 훨씬 수월할 거예요.

리뷰 요청한 이벤트 위임 구조에 대해 답변드리면, 현재 방식은 대규모 UI에서도 충분히 동작합니다. 다만 주의할 점은 removeEvent 호출이 누락되면 핸들러 맵에 참조가 남아서 메모리 누수가 생길 수 있어요. 컴포넌트 언마운트 시 정리가 확실히 되는지만 신경 쓰면 됩니다.

이번 과제의 핵심은 내가 평소에 쓰던 것들의 동작 원리를 직접 구현해보면서 이해하는 거였어요. 깊이를 내려가고 싶다면 그 영역의 핵심 컨셉만 한번 직접 구현해보면 돼요. 상태관리, CSS Framework, SSR, 번들러 등 전부 구현할 필요 없이 핵심만 만들어보면 훨씬 쉽게 이해할 수 있다는 방법을 이번에 배웠기를 바랍니다.
