1. 순수함수의 중요성

순수함수

순수함수는 동일한 입력에 대해 항상 동일한 출력을 반환하고, 부수 효과(side effect)가 없는 함수입니다. 이번 과제에서 순수함수의 중요성을 여러 측면에서 경험했습니다.

Before: 컴포넌트 내부의 계산 로직

처음에는 컴포넌트 내부에 계산 로직이 섞여 있었습니다:

// ProductCard.tsx (Before)
{
  product.discounts.length > 0 && (
    <span>~{Math.max(...product.discounts.map((d) => d.rate)) * 100}%</span>
  );
}

이 방식의 문제점: 테스트가 어렵고 재사용이 불가능 하며 로직의 의도 파악이 어렵습니다.

After: 순수함수로 분리

계산 로직을 순수함수로 분리

// models/discount.ts
export function getMaxProductDiscountRate(
  discounts: Array<{ quantity: number; rate: number }>
): number {
  if (discounts.length === 0) return 0;
  const maxRate = Math.max(...discounts.map((d) => d.rate));
  return Math.round(maxRate * 100);
}

// ProductCard.tsx (After)
import { getMaxProductDiscountRate } from "../../models/discount";

{
  product.discounts.length > 0 && (
    <span>~{getMaxProductDiscountRate(product.discounts)}%</span>
  );
}

개선된 점: 재사용성 및 가독성이 향상 되었습니다. 이로인해 유지보수도 더 용이해졌습니다.

순수함수로 분리한 함수들

  • getMaxProductDiscountRate: 상품 최대 할인율 계산
  • calculateCartTotal: 장바구니 총액 및 할인 금액 계산
  • calculateItemTotal: 개별 아이템 총액 계산
  • calculateItemDiscountRate: 개별 아이템 할인율 계산
  • getRemainingStock: 남은 재고 계산
  • filterProducts: 상품 필터링

2. 계층 구조의 명확성

3계층 아키텍처

이번 과제에서는 컴포넌트 → Hook → Model의 3계층 구조를 구분했습니다.

계층별 책임과 역할

1. Components Layer (UI Layer)

  • 책임: 사용자 인터페이스 렌더링
  • 역할:
    • Props를 받아 UI를 표시
    • 사용자 이벤트를 상위 컴포넌트로 전달
    • 계산 로직을 직접 수행하지 않음
// CartPage.tsx - 컴포넌트 계층
export const CartPage: React.FC<CartPageProps> = ({ searchTerm }) => {
  // Hook에서 상태와 로직을 가져옴
  const { cart, totals, addToCart } = useCart(products, addNotification);
  const { products, filteredProducts } = useProducts(searchTerm, ...);

  // UI만 렌더링
  return (
    <div>
      <ProductList
        products={products}
        onAddToCart={addToCart}  // Hook에서 받은 함수 전달
      />
      <CartSidebar
        cart={cart}              // Hook에서 받은 상태 전달
        totals={totals}          // Hook에서 계산된 결과 전달
      />
    </div>
  );
};

2. Hooks Layer (State & Logic Layer)

  • 책임: 상태 관리와 비즈니스 로직의 조합
  • 역할:
    • 컴포넌트의 상태를 관리 (useState, useLocalStorage)
    • 여러 Model 함수를 조합하여 복잡한 로직 구현
    • 컴포넌트에 필요한 데이터와 함수 제공
// useCart.ts - Hook 계층
export function useCart(products: ProductWithUI[], addNotification?: NotifyFn) {
  const [cart, setCart] = useLocalStorage<CartItem[]>("cart", []);

  // Model 함수들을 조합하여 비즈니스 로직 구현
  const addToCart = useCallback((product: Product) => {
    const result = cartModel.addItemToCart(cart, product);  // Model 함수 호출
    if (result.success) {
      setCart(result.cart);
    }
    addNotification?.(result.message, result.success ? "success" : "error");
  }, [cart, setCart, addNotification]);

  // 계산 함수는 Model 함수를 래핑
  const calculateTotal = useCallback(() => {
    return calculateCartTotal(cart, selectedCoupon);  // Model 함수 호출
  }, [cart, selectedCoupon]);

  return { cart, addToCart, calculateTotal, ... };
}

3. Models Layer (Pure Logic Layer)

  • 책임: 순수한 계산 로직과 데이터 변환
  • 역할:
    • 부수 효과 없는 순수함수로 구성
    • 테스트하기 쉬운 독립적인 함수들
    • 재사용 가능한 비즈니스 로직
// models/cart.ts - Model 계층
export function calculateCartTotal(
  cart: CartItem[],
  selectedCoupon: Coupon | null
): {
  totalBeforeDiscount: number;
  totalAfterDiscount: number;
  discountAmount: number;
} {
  // 순수함수: 입력만으로 결과 계산
  let totalBeforeDiscount = 0;
  let totalAfterDiscount = 0;

  cart.forEach((item) => {
    totalBeforeDiscount += item.product.price * item.quantity;
    totalAfterDiscount += calculateItemTotal(item, cart);
  });

  // 쿠폰 할인 적용
  if (selectedCoupon) {
    // ... 할인 계산 로직
  }

  return {
    totalBeforeDiscount: Math.round(totalBeforeDiscount),
    totalAfterDiscount: Math.round(totalAfterDiscount),
    discountAmount: totalBeforeDiscount - totalAfterDiscount,
  };
}

계층 간 의존성 규칙

단방향 의존성 원칙:

  • Components는 Hooks에만 의존
  • Hooks는 Models에만 의존
  • Models는 다른 계층에 의존하지 않음

이를 통해:

  • 의존성 역전: 상위 계층이 하위 계층의 구현에 의존하지 않음
  • 테스트 용이성: 각 계층을 독립적으로 테스트 가능
  • 유지보수성: 한 계층의 변경이 다른 계층에 최소한의 영향만 미침

계층 분리의 실질적 이점

  1. 관심사의 분리 (Separation of Concerns)

    • UI 변경이 비즈니스 로직에 영향 없음
    • 비즈니스 로직 변경이 UI에 영향 없음

  2. 재사용성

    • Model 함수는 어떤 Hook에서도 사용 가능
    • Hook은 어떤 컴포넌트에서도 사용 가능

  3. 테스트 전략

    • Model: 단위 테스트 (순수함수)
    • Hook: 통합 테스트 (상태 관리)
    • Component: E2E 테스트 (사용자 시나리오)

  4. 코드 탐색 용이성

    • 계산 로직이 필요하면 → Models 폴더
    • 상태 관리가 필요하면 → Hooks 폴더
    • UI 컴포넌트가 필요하면 → Components 폴더

이번 과제에서 내가 제일 신경 쓴 부분은 무엇인가요?

1. 계산 로직의 순수함수화

  • 컴포넌트에 남아있던 계산 로직을 모두 순수함수로 분리했습니다.
  • getMaxProductDiscountRate: 상품 최대 할인율 계산
  • calculateCartTotal의 discountAmount 필드 추가: 할인 금액 계산을 함수 내부로 이동
  • 이를 통해 컴포넌트는 순수하게 UI만 담당하게 되었습니다.

2. Entity별 모델 분리

  • models/cart.ts, models/product.ts, models/coupon.ts, models/discount.ts로 Entity별로 명확히 분리했습니다.
  • 각 모델이 특정 Entity만 다루도록 해 단일 책임 원칙을 지켰습니다.

3. Hook의 책임 분리

  • useCart, useCoupons, useProducts, useNotification 각 Hook이 명확한 책임을 가지도록 구성했습니다.
  • Hook 간 의존성을 최소화하고, 필요한 경우에만 의존하도록 설계했습니다.

// ❌ Before: 컴포넌트가 모든 걸 다 함
function CartItem({ item }) {
  // 가격 계산 (비즈니스 로직이 컴포넌트 안에!)
  const originalPrice = item.product.price * item.quantity;
  const discountRate = Math.round((1 - itemTotal / originalPrice) * 100);
  
  return <div>{discountRate}% 할인!</div>
}

// ✅ After: 컴포넌트는 보여주기만
function CartItem({ item, discountRate }) {
  // 계산은 hook에서 이미 됨, 여기선 보여주기만!
  return <div>{discountRate}% 할인!</div>
}

비즈니스 로직에서 계산 분리하기

🎮 Hook (useCart)     = 게임 컨트롤러 (상태 관리 + 조작)
🧮 Model (cart.ts)    = 게임 엔진 (순수 계산)

// 📁 models/cart.ts - 순수 계산만! (입력 → 출력)
export function calculateTotal(cart, coupon) {
  let total = 0;
  for (const item of cart) {
    total += item.price * item.quantity;
  }
  return total;
}

// 📁 hooks/useCart.ts - 상태 관리 + 계산 함수 호출
export function useCart() {
  const [cart, setCart] = useState([]);
  
  // model의 계산 함수를 가져다 쓰기만 함
  const totals = useMemo(
    () => calculateTotal(cart, coupon),  // ← model 함수 호출
    [cart, coupon]
  );
  
  return { cart, totals };
}

Entities → Features → UI 계층

🍽️ 서빙 (UI)          = 예쁜 접시에 담아서 손님에게 전달
