[5팀 손승현] Chapter3-3. 기능 중심 아키텍처와 프로젝트 폴더구조

배포링크

과제 체크포인트

기본과제

목표 : 전역상태관리를 이용한 적절한 분리와 계층에 대한 이해를 통한 FSD 폴더 구조 적용하기

  • 전역상태관리를 사용해서 상태를 분리하고 관리하는 방법에 대한 이해
  • Context API, Jotai, Zustand 등 상태관리 라이브러리 사용하기
  • FSD(Feature-Sliced Design)에 대한 이해
  • FSD를 통한 관심사의 분리에 대한 이해
  • 단일책임과 역할이란 무엇인가?
  • 관심사를 하나만 가지고 있는가?
  • 어디에 무엇을 넣어야 하는가?

체크포인트

  • 전역상태관리를 사용해서 상태를 분리하고 관리했나요?
  • Props Drilling을 최소화했나요?
  • shared 공통 컴포넌트를 분리했나요?
  • shared 공통 로직을 분리했나요?
  • entities를 중심으로 type을 정의하고 model을 분리했나요?
  • entities를 중심으로 ui를 분리했나요?
  • entities를 중심으로 api를 분리했나요?
  • feature를 중심으로 사용자행동(이벤트 처리)를 분리했나요?
  • feature를 중심으로 ui를 분리했나요?
  • feature를 중심으로 api를 분리했나요?
  • widget을 중심으로 데이터를 재사용가능한 형태로 분리했나요?

심화과제

목표: 서버상태관리 도구인 TanstackQuery를 이용하여 비동기코드를 선언적인 함수형 프로그래밍으로 작성하기

  • TanstackQuery의 사용법에 대한 이해
  • TanstackQuery를 이용한 비동기 코드 작성에 대한 이해
  • 비동기 코드를 선언적인 함수형 프로그래밍으로 작성하는 방법에 대한 이해

체크포인트

  • 모든 API 호출이 TanStack Query의 useQuery와 useMutation으로 대체되었는가?
  • 쿼리 키가 적절히 설정되었는가?
  • fetch와 useState가 아닌 선언적인 함수형 프로그래밍이 적절히 적용되었는가?
  • 캐싱과 리프레시 전략이 올바르게 구현되었는가?
  • 낙관적인 업데이트가 적용되었는가?
  • 에러 핸들링이 적절히 구현되었는가?
  • 서버 상태와 클라이언트 상태가 명확히 분리되었는가?
  • 코드가 간결하고 유지보수가 용이한 구조로 작성되었는가?
  • TanStack Query의 Devtools가 정상적으로 작동하는가?

최종과제

  • 폴더구조와 나의 멘탈모데일이 일치하나요?
  • 다른 사람이 봐도 이해하기 쉬운 구조인가요?

과제 셀프회고

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

  • FSD를 이번 과제를 통해 처음 접해보았습니다. 각 레이어별 역할(entities → features → widgets → pages → app → shared)을 실제로 나누면서 수직적 구조와 도메인 분리의 감을 익혔습니다.
  • TanStack Query로 서버 상태를 선언적으로 관리하며, 기존 fetch/useState 패턴을 쿼리 키·캐시·무효화 중심으로 대체하는 방법을 배웠다.
  • FSD 관점에서는 작은 유틸도 레이어 규칙을 지켜야 한다는 것을 알게 되었습니다. 공통 텍스트 처리 로직인 highlightText를 shared/lib로 분리하고, 정규식 특수문자 이스케이프를 넣어 어디서든 안전하게 재사용하도록 만들면서 “작은 함수도 올바른 레이어에 두고 견고하게 만들어야 한다”는 점을 체감하였습니다.

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

  • FSD 구조를 처음 접해 수직적 폴더링·도메인 분리에 대한 감이 없어서, 단계마다 질문하며 레이어 경계를 잡았습니다. 페이지에서 거대한 컴포넌트를 쪼개 위젯/피처/엔티티로 역할을 명확히 분리했고,
  • 서버 상태는 전부 TanStack Query로 치환하면서 클라이언트 UI 상태(Zustand)와 구분하는 흐름을 반복 학습했습니다.
  • URL 파라미터 기반 필터/페이지네이션, 공통 API 레이어 분리, 에러 알림 중앙화, 그리고 배포용 API base URL을 환경변수로 빼는 작업까지 FSD 원칙에 맞춰 재정렬한 점이 가장 큰 집중 포인트였습니다.

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

  • 레이어 경계: feature vs widget vs entity에 무엇을 둘지 기준이 아직 헷갈립니다(예: 태그/검색 필터 로직의 위치).
  • 상태 분리: 서버 상태(React Query)와 클라이언트 UI 상태(Zustand·URL 파라미터) 역할 차이에 대해서 더 명확하게 알아야 할 것 같습니다.

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

  • TanStack Query 기본 패턴(쿼리 키 일관성, staleTime/gcTime 설정, onError+토스트, 낙관적 업데이트)에 대해서 적용해 보고싶습니다.
  • 레거시 components 폴더를 공통 유틸/컴포넌트는 shared로 일원화하는 것에 도전

챕터 셀프회고

클린코드와 아키테쳑 챕터 함께 하느라 고생 많으셨습니다! 지난 3주간의 여정을 돌이켜 볼 수 있도록 준비해보았습니다. 아래에 적힌 질문들은 추억(?)을 회상할 수 있도록 도와주려고 만든 질문이며, 꼭 질문에 대한 대답이 아니어도 좋으니 내가 느꼈던 인사이트들을 자유롭게 적어주세요.

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

  • 더티코드를 접했을 때 어떤 기분이었나요? ^^; 클린코드의 중요성, 읽기 좋은 코드란 무엇인지, 유지보수하기 쉬운 코드란 무엇인지에 대한 생각을 공유해주세요

처음 레거시 코드들을 마주하였을때 어디서 부터 손봐야할지 막막 했습니다.. 레거시를 리팩토링해본 경험이 실무에서 거의 없었기 때문에 관심사의 분리도 어떤 기준으로 해야할지 기준조차 없었습니다. 3주동안 작업해보니 가장크게 깨달은 점은 깨끗한 코드가 테스트도 쉽게 만든다는 걸 체감했습니다. 입력·출력이 명확한 순수 함수와 작게 나뉜 컴포넌트는 바로 단위 테스트를 걸 수 있고, 의존성이 분리돼 있어 작성도 간단하였습니다.

결합도 낮추기: 디자인 패턴, 순수함수, 컴포넌트 분리, 전역상태 관리

  • 거대한 단일 컴포넌트를 봤을때의 느낌! 처음엔 막막했던 상태관리, 디자인 패턴이라는 말이 어렵게만 느껴졌던 시절, 순수함수로 분리하면서 "아하!"했던 순간, 컴포넌트가 독립적이 되어가는 과정에서의 깨달음을 들려주세요

거대한 단일 컴포넌트를 보면 의존성이 얽혀 있어 어디서부터 분리해야 할지 막막하였습니다. 디자인 패턴을 적용해 관심사를 나누고, 순수 함수로 로직을 분리하며, 컴포넌트를 작게 쪼개고, 전역 상태로 props drilling을 줄이는 과정을 단계적으로 진행하였습니다. 순수 함수로 분리하면서 "아하!" 순간이 왔었는데, 입력과 출력이 명확하고 사이드 이펙트가 없어 테스트와 재사용이 쉬웠습니다. 예를 들어 highlightText 같은 유틸 함수는 정규식 이스케이프를 추가해 안전성을 높였고, 어디서든 독립적으로 사용할 수 있었다.

응집도 높이기: 서버상태관리, 폴더 구조

  • "이 코드는 대체 어디에 둬야 하지?"라고 고민했던 시간, FSD를 적용해보면서의 느낌, 나만의 구조를 만들어가는 과정, TanStack Query로 서버 상태를 분리하면서 느낀 해방감(?)등을 공유해주세요

FSD 구조를 적용하면서 위의 개념들(클린코드 및 결합도 낮추기)이 자연스럽게 연결되었습니다. 순수 함수는 shared/lib에, 컴포넌트는 레이어별로 분리했고, 전역 상태는 entities/model에서 관리합니다. 디자인 패턴(예: Container/Presenter)이 FSD의 레이어 경계와 맞아떨어지면서, 결합도가 낮아지고 응집도가 높아지는 구조가 완성되었습니다.

또한 코드 정리 및 분리에 대한 것들은 FSD의 레이어 규칙을 적용하면서 기준이 명확해졌습니다. entities는 비즈니스 도메인(Post, User, Comment, Tag)을 중심으로 타입, 모델, API, UI를 묶었고, features는 사용자 행동(생성, 수정, 삭제, 좋아요)을 중심으로 분리했습니다. widgets는 여러 피처와 엔티티를 조합한 재사용 가능한 블록으로 만들었고, shared는 프로젝트 전반에서 사용하는 공통 로직과 컴포넌트를 담았습니다.

TanStack Query로 서버 상태를 분리하면서 해방감을 느꼈습니다. 기존에는 useState와 useEffect로 복잡하게 관리하던 로딩, 에러, 캐싱 상태를 선언적으로 처리할 수 있게 되었습니다. useQuery와 useMutation을 사용하면서 비동기 로직이 함수형 프로그래밍의 원칙에 맞게 작성되었고, queryKey를 통해 캐싱과 무효화 전략을 명확하게 관리할 수 있었습니다. 특히 invalidateQueries를 통해 데이터 일관성을 자동으로 유지할 수 있어, 수동으로 상태를 동기화하던 번거로움이 사라졌습니다.

리뷰 받고 싶은 내용이나 궁금한 것에 대한 질문

과제 피드백

잘했어요 승현! FSD를 처음 접했는데도 각 레이어의 역할을 실제로 나눠보면서 감을 잡아간 게 느껴집니다. 특히 "작은 함수도 올바른 레이어에 두고 견고하게 만들어야 한다"는 깨달음이 좋았어요. highlightText처럼 작은 유틸 하나도 제자리에 두는 습관이 쌓이면 나중에 큰 프로젝트에서 빛을 발합니다.

"깨끗한 코드가 테스트도 쉽게 만든다"는 것도 정확히 맞아요. 입력과 출력이 명확한 순수 함수는 테스트하기 쉽고, 의존성이 분리돼 있으면 수정할 때도 영향 범위가 명확해지거든요. 이걸 3주 만에 체감한 건 큰 수확입니다.

승현이가 아직 막연하다고 한 부분들에 대해 얘기하자면, 레이어 경계가 헷갈리는 건 자연스러운 거예요. 근데 중요한 건 경계를 고민하는 게 아니라, 규칙을 하나 정해서 그 규칙대로만 하면 분리가 다 되느냐 아니냐를 발견하는 겁니다. 인간은 거대한 프로젝트의 세부적인 것들을 다 외우지 못하지만 규칙은 잘 외우거든요.

저는 이렇게 단순화해서 쓰고 있어요. entities는 서버 상태 관리, features는 클라이언트 상태 관리, widgets는 화면 단위. 이렇게 나누니까 응집도가 높아지더라고요. 태그/검색 필터 로직 같은 것도 "이게 서버에서 오는 데이터인가, 사용자가 UI에서 조작하는 상태인가?"로 판단하면 돼요. 필터 옵션 목록은 entities, 현재 선택된 필터 값은 features에 두면 됩니다.

서버 상태와 클라이언트 UI 상태의 구분은 "새로고침하면 날아가도 되는가?"로 판단하세요. 모달 열림/닫힘, 검색어 입력 중인 값은 날아가도 되니까 클라이언트 상태. 게시글 목록, 유저 정보는 서버에서 다시 받아야 하니까 서버 상태예요.

FSD를 잘해야겠다보다는, 이 철학을 바탕으로 내 폴더 구조에 적용하면서 규칙을 설명할 수 있는 구조를 만드는 게 더 중요합니다. 그리고 "엔티티가 드러나는 폴더 구조가 좋다"는 감각을 가져가세요. 폴더 이름만 봐도 이 프로젝트가 뭘 하는지 알 수 있는 구조가 좋은 구조예요.

3챕터 동안 수고했어요. 처음 접한 개념인데도 끝까지 따라온 것, 그게 가장 중요합니다!
