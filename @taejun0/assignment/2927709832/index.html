[5팀 오태준] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by taejun02025.10.19

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

| 이벤트는 생성/수정 시 fetch로 다시 로드되어 상태를 갱신합니다. 이때 테스트가 병렬로 돌아도 안정적이게 동작하려면?

해당 질문의 가장 중요한 포인트는 상태를 완전히 분리해 내는 것이다. 테스트가 서로 영향을 주게 된다면, failure를 만들어 낼 수 있다. 특히, 테스트 간에 단 하나의 배열만을 공유하게 된다면 실행 순서나 타이밍으로 인해 실패를 만들 수 있는 것이다.

로컬에서 통과하더라도 CI 과정처럼 워커가 병렬로 도는 환경에서는 특히나 더욱 민감하다.

즉, 테스트라는 것은 다른 요인에 영향을 받지않고, 독립적이어야만 하며, 같은 상황에서 같은 값을 뱉는 단일 실행이 필수적이어야한다. (결정성)

이를 막기 위해 handlersUtils.ts는 다음과 같은 규칙을 갖는다.

  1. 전역 상태 방지를 위한 독립 저장 공간 부여

  • 문제 상황 만약, 전역 배열이 공유되고 있다면 다음과 같은 상황이 발생할 수 있다. A 테스트와 B 테스트가 있다고 가정하자. A 테스트는 POST 한번, B 테스트는 GET 호출을 원하고 있다. | A 테스트가 POST로 이벤트를 추가 -> fetchEvents()로 GET 재로딩 -> 그 후 B 테스트 GET 호출 즉, 기존에 B 테스트가 원하지 않던 A 테스트의 POST 값이 추가되어 B 테스트는 정상적임에도 실패가 일어난다.

  • 해결 createStore(initialEvents)가 매 호출마다 새로운 배열을 반환하도록 하여, 각 테스트는 자신만의 스토어가 발생한다. 특히, 해당 스토어를 핸들러가 클로저로 가져감으로서 핸들러가 다뤄야할 데이터 경계는 테스트 단위에서 완벽하게 분리될 수 있었다.

function createStore(initialEvents: Event[] = []) {
  return [...initialEvents];
}
// 스토어 생성 예시

export const setupMockHandlerCreation = (initEvents = [] as Event[]) => {
  // 테스트별 저장소 초기화 및 GET & POST 관련 핸들러를 정의한다.
  const store = createStore(initEvents);
  server.use(createGetHandler(store), createPostHandler(store));
};
// 핸들러 예시
  1. 요청 응답 경로도 테스트 별로 독립화 한다

  • 문제 상황 한번 등록된 핸들러가 다른 테스트에서도 계속 살아있다면, 추후 테스트에서 원하지 않는 스토어나 핸들러를 참조할 수 있다.

  • 해결 각 테스트 시작부에 setupMockHandler를 명시적으로 호출하게 한다. -> 해당 함수들이 server.use(...)을 통해 테스트 전용 핸들러 세트를 주입한다.

afterEach(() => {
  server.resetHandlers();
  vi.clearAllMocks();
});

동시에, 이미 구현되어있는 사항이었으나, setupTest.ts에서 누수를 방지했다. 이를 통해, 이전 테스트가 등록한 핸들러는 더이상 다음 테스트에 남지 않으며, 테스트, 핸들러, 스토어는 각각 일대일 대응함을 보장한다.

  1. ID 생성도 테스트 별로 독립화 한다

  • 문제 상황 동일한 ID 생성기를 만약 여러 테스트가 공유한다면, ID가 예측 불가능하게 증가하여 테스트에 영향을 끼칠 것이다.

  • 해결

export function createGetHandler(store: Event[]) {
  return http.get('/api/events', () => {
    return HttpResponse.json({ events: store }, { status: 200 });
  });
}

makeIdGenerator()를 각 테스트 핸들러의 생성마다 새로 호출하였다. 간단하게 테스트가 달라지면 sequnce가 1부터 다시 시작하는 구조였다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요. createGetHandler, createPostHandler, createPutHandler, createDeleteHandler, 총 4개의 핸들러가 클로저로 store를 관리한다. 또한, 매 테스트마다 setupMockHandlerCreation, setupMockhandlerUpdating, setupMockDeletion 등이 호출되어 독립 store를 유지하며, 해당 CRUD만을 가진 핸들러를 server.use(...)로 가질 수 있었다.

다음으로는 테스트를 초기화 및 정리하였는데, afterEach(() => server.resetHandlers()) 코드를 통해 누수가 되지 않도록 초기화 하였고, vi.clearAllMocks()를 활용하여, DOM/모킹 상태를 초기화하여 누수를 방지하였다.

마지막으로는 알림 테스트에서 vi.useFaketimers(), vi.setSystemTime(...), vi.advanceTimersByTime(...) 을 통해 타이머 및 현재 시간을 고정함으로서 문제를 해결하였다.

과제 셀프회고

기술적 성장

  • 과제 전형에 도입하기 위해 테스트코드를 짜본 경험은 있으나, 명확하게 왜 짜야하는지 의문만 남았었다. 그러나, 해당 경험을 통해서 테스트 코드는 단순히 Mock을 통해서 잘 돌아가는지 확인하는 것이 아닌, 나의 코드의 가장 불안정한 경계부분을 return함으로 완성된 구조를 만들어나가는 과정임을 깨달았다.

  • 특히, MSW를 이용한 상태 분리 과정이 가장 재밌었던 것 같다. 처음에는 hint를 보아도 이걸 왜 분리해야하는 것인지, 그 목적은 무엇인지 알기 어려웠다. 그러나 테스트 코드를 하나씩 짜면서 상태 분리의 중요성을 깨달았고 방법을 모색하였다. 그떄, MSW의 server.use가 함수 실행 시점에서 클로저로 데이터를 캡처한다는 점을 활용하여 테스트별 상태 분리를 구현할 수 있었다. 그 결과, 간섭이 없는 CRUD 로직을 구현할 수 있었다.

  • 또한 Vitest의 vi 객체를 공부한 것도 소중한 경험이 되었다. vi를 공부하며, 실제로 내가 작성해온 테스트 코드가 얼마나 엉망이었는지도 깨달았다. 기존 jest를 통해서 구현했지만 이해하지 못한 테스트 코드들이 여럿 있었다. 특히, "공부하지 않았기에" 가지고 있었던 두려움이 있었던 것 같다. 앞으로의 테스트 코드에 있어서도 이러한 친숙함을 통해 적용할 수 있으니 보다 큰 성장을 얻었다고 생각한다.

코드 품질

만족스러운 부분은, 잘 모르지만 AI의 도움을 받지 않은 채로 구현에 성공했다는 점이다. 특히나, 가장 자신 없었던 부분을 내 힘으로 해내는 경험은 참 좋았다. 그러나, 코드 품질이 좋았을까?라는 의문은 해결되지 않는다.

특히, 통합 테스트 부분에서는 전혀 코드 품질이 좋다고 느껴지진 않았다.

간단한 구현에서는 AI 없이 구현하는 것이 굉장히 빠르게 끝났고 쉽다고 느껴졌으나, 해당 부분에서는 그저 화면상에 이루어지는 값들을 찾아서 이들이 있는지 없는지만을 구현하였으니, 보다 나은 방법이 충분히 있지 않을까? 하는 아쉬움이 남았다.

예를 들어, searching에 대한 결과값에 대해 찾아야한다면. 결과 리스트에 내가 원하는 결과 값이 있는지 or 검색 결과 없음이 있는지 만을 구현해 넣었다..

또한, 주석을 보다 면밀하고 자세히 적는다면 더 좋은 코드가 되지 않았을까 하는 아쉬움만 남았다. 현재 주석의 방식은 쓰는 사람 기준으로 작성된 것 같은데 실제로는 읽는 사람을 기준으로 한번에 그 구조를 알아보기 쉽게 작성하는 것이 더 좋았을 것 같다.

참 아쉬웠다.

학습 효과 분석

  • 먼저 테스트의 중요성을 깨달았다는 점에서 굉장히 긍정적이었다. 수학을 공부하지 않고는 그 위대함을 모르듯, 테스트 코드를 제대로 공부해본 경험이 없으니, 그 중요성 또한 파악하기 어려웠던 것 가탇.
  • 앞으로 MSW를 활용한 API 모킹과 독립 스토어 구조를 그대로 적용할 수도 있을 것 같다.

과제 피드백

  • 처음에는 설명 주석의 모호한 부분이 너무 추상적이라 어색함을 느꼈다. 그러나 과제를 수행하면서,, 흑백 요리사에 나왔던 안성재 쉐프의 말이 떠올랐다. 음식에 대한 어떠한 설명을 주지 않은 요리사에 대해, 설명이 없었기에 상상하는 모든 것이 정답이 되었다 라는 말을 돌려주었다. 이번 과제도 동일하였다. 테스트 코드에 대해 어려움을 가지고 있는 신입 개발자에게 끊임없이 상상하고 생각할 수 있는 시간을 제공받은 기분이었다.

  • git으로 과제를 관리하는 만큼 Rule set을 통해 upstream branch에는 승인없이 merge하지 못하도록 막는 것은 어떨까? 하는 생각이 들었습니다. 사실,, 아무도 그런일을 벋이진 않았으나, 초록 색의 Merge pull request를 보고 있으면,, 한번쯤 눌러보고 싶다는 생각이 들지 않을까 해서요!

리뷰 받고 싶은 내용

  • 현재 handlersUtils 구조가 충분하게 모듈화가 되어있는지, 보다 나은 패턴이 존재한지가 궁금합니다. 또한, handler setup 함수의 중복되는 부분이 꽤 있어 추후에는 setupMockHander(type: 'create' | 'update' | 'delete')의 형태로 합쳐볼까하는 생각이 들었는데 이 생각이 올바른 생각인지도 궁금합니다.
