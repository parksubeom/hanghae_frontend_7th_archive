[1팀 천진아] Chapter 1-3. 완성도있는 테스트 전략 수립하기

by totter152025.11.02

과제 3

필수 스펙

  • 드래그 앤 드롭(D&D) 기능 개발
    • 캘린더의 일정을 마우스로 끌어 다른 날짜나 시간으로 옮기는 기능을 구현합니다.
  • 날짜 클릭으로 일정 생성 기능 개발
    • 캘린더의 비어있는 날짜 셀을 클릭하면, 해당 날짜가 자동으로 폼에 채워지도록 하세요.

기본 과제

필수 스펙 개발과 E2E, 시각적 회귀 테스트를 모두 작성해주세요.

기본과제 제출

  • 아래 작성된 E2E 테스트 작성은 필수입니다. 추가로 작성하고 싶다면 작성해주세요.
  • 여기서 말하는 전반은 Create, Read, Update, Delete 모두에 해당합니다.
  1. 기본 일정 관리 워크플로우 전반을 검증하세요.
  2. 반복 일정 관리 워크플로우 전반을 검증하세요.
  3. 일정 겹침 처리 방식에 대해 검증하세요.
  4. 알림 시스템 관련 노출 조건에 대해 검증하세요.
  5. 검색 및 필터링 전반에 대해 검증하세요.
  • 아래 시각적 회귀 테스트는 필수 입니다. 추가로 작성하고 싶다면 작성해주세요.
  1. 타입에 따른 캘린더 뷰 렌더링
  2. 일정 상태별 시각적 표현
  3. 다이얼로그 및 모달
  4. 폼 컨트롤 상태
  5. 각 셀 텍스트 길이에 따른 처리

당장은 적절한 스토리를 작성하는데 더 집중해주세요.
적절한 추상화, 응집도를 높이고 결합도를 낮추는
우아한 설계는 이후 주차때 좀 더 집중해봐요.

- 물론, AI를 활용해도 좋습니다.
- 기존에 작성한 테스트는 깨지지 않도록 해주세요.
  - 이 과정에서 테스트가 분리될 수도 작성법이 수정될 수도 있습니다.
  - 정말 불필요한 케이스가 아니면, 테스트 케이스를 삭제하는건 최소화 해주세요.
  - 추가로 필요한 테스트가 생길 수 있지만, 작성은 자유롭게 선택해서 진행해주세요.

심화 과제

테스트 전략 작성해보기의 내용을 참고해서 지금까지 작성했던 프로젝트에 대해 테스트 전략을 구상해보세요.

심화 과제 제출

  • 내가 생각한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

현재 저희 프로젝트는 일정 CRUD의 스펙을 가지고 있기에 e2e로 메인 시나리오를 검증하고 이후 통합테스트, 단위 테스트순으로 추가를 하는게 최적의 테스트 전략이라고 생각합니다. 작은 프로젝트에 e2e는 필요없다고 생각할 수 있지만, 오히려 프로젝트가 간단하기에 주요 시나리오 몇개를 e2e로 작성한다면 테스트에 적은 노력을 들이고도 프로젝트에 신뢰도를 줄 수 있다 생각했습니다.

지금은 과제진행을 위해서 거의 모든 e2e프로세스와 통합, 단위 테스트가 작성되어있지만 테스트코드가 없는 상태에서 해당 프로젝트에 테스트코드를 추가한다면, e2e -> 통합 -> 단위로 진행을 할 것 같습니다.

  • 팀원들이 합의한 최적의 테스트 전략은 무엇인지, 그리고 그 이유를 작성해주세요.

각자의 전략을 소개하고나니, 결론적으로 모두 동일하게

E2E > 통합 > 단위

라는 결론이 나왔습니다.

결론은 모두 통일 됐지만 그 안에서도 공통점과 차이점을 보이는 부분이 있었습니다.

공통점

✅ E2E 테스트가 최우선
✅ Drag & Drop은 반드시 E2E로 검증
✅ 통합/단위 테스트는 이미 많이 작성됨
✅ 단위 테스트 추가 우선순위는 낮음
✅ **사용자 관점**의 시나리오 검증 중요

차이점

🔴 E2E 범위
- 용태: 주요 기능 모두 E2E로 (주요 시나리오 추리기)
- 민석/승훈: 통합으로 불가능한 것만
- 재현/진아/희정: 테스트 커버리지 분석 후 결정

🟡 통합 vs E2E 경계
- "일정 추가 플로우" → E2E 필요? 통합 충분?
- "백엔드 API 연동" → 어디까지 E2E?

1. 현황 공유 및 정렬

먼저, 프로젝트의 특성과 현황에 대해 공유하였습니다.

1-1. 우리 프로젝트 특성은?

✓ 기능 복잡도: 낮음 (CRUD 중심의 할일 관리)
✓ UI 복잡도: 높음 (Form + Calendar + List가 한 화면에서 동기화)
✓ 주요 UX: 세 영역 간 실시간 동기화
✓ 현재 단계: 주요 기능 완성, 배포 준비 단계
✓ 기존 테스트: 통합/단위 테스트 상당량 작성됨

1-2. 설계해야 할 가장 중요한 기능/영역은?

1순위: Drag & Drop (캘린더 내, List → Calendar)
2순위: 세 영역 동기화 (Form ↔ Calendar ↔ List)
3순위: 실제 사용자 플로우 (추가 → 이동 → 완료)

프로젝트의 큰 흐름에 대한 논의 이후, 이제는 테스트 유형별로 어떻게 작업하면 좋을지를 고민하였습니다.

2. 테스트 유형별 논의

앞서 논의한 차이점을 바탕으로 한번 더 단위, 통합 , e2e, 시각적 회귀 테스트 각각의 사용 대상 범위를 점검하고, 테스트 전략에 대해 논의하였습니다.

✓ 모든 함수 테스트? → NO, 핵심 로직만
✓ 컴포넌트 단위 테스트? → NO, 통합으로 충분
✓ 우선순위: 낮음
✓ 액션: 추가 작성 안 함 (기존 것만 유지)

3. 테스트가 필요한 기능/단위 시나리오 작성하기

합의된 내용을 바탕으로 테스트가 필요한 기능/단위에 대해 시나리오를 작성하여, 심화과제에 대한 테스트 목록을 준비하였습니다.

3-1. 시나리오 작성

필수 시나리오: D&D

  • 유저는 캘린더의 셀을 드래그해서 일정을 수정하고 완료 스낵바가 뜬다. - e2e
    • 일정이 겹치면 다이얼로그 출력 (예/아니오)
    • 드래그앤 드롭 취소시 취소 스낵바 노출, 일정은 수정되지 않는다.
    • 월간 / 주간 뷰 드래그앤드롭
  • 유저가 반복일정인 셀을 드래그해서 일정을 수정할 시, 단독 일정으로 수정된다.
    • 단독일정으로 빠져나온 일정이 겹치면 다이얼로그

메인 시나리오

  • 유저는 폼을 입력해 일정을 생성한다.
    • Base : 생성한 일정이 목록에 표시된다
    • Case - 1 : 반복으로 생성된 일정은 목록에 복수로 표시될 수 있다.
  • 유저는 목록에서 생성된 일정들을 조회할 수 있다.
    • Base : 캘린더에 있는 일정이 모두 표시된다.
    • Case - 1 : 검색창에 검색어를 입력하면 검색어에 해당하는 일정만 표시된다.

서브 시나리오

  • 유저가 캘린더의 셀을 드래그 해 일정이 겹칠 경우 경고창이 나온다.
  • 캘린더 폼에 내용이 없으면 일정에 추가가 안되도록 한다.
  • 생성한 일정이 임박하면, 일정 알림이 뜬다.

의미있는 테스트를 뽑아내기 위해 시나리오를 작성하였고, 이를 통해 최종적으로 각 테스트 별 전략이 수립되었습니다.

  • 그 전략에 맞춰 추가한 테스트는 무엇인지 나타내주세요.

저희는 현재 작성되어있는 테스트에 추가로 e2e와 통합 테스트를 작성하기로 했습니다. 추가한 테스트는 아래와 같습니다.

E2E 테스트

D&D (e2e)

  1. 월간뷰에서 D&D로 일정을 옮기면, 성공 스낵바가 뜨고, 일정이 수정된다.(e2e)
  2. 주간뷰에서 D&D로 일정을 옮기면, 성공 스낵바가 뜨고, 일정이 수정된다.(e2e)
  3. D&D로 반복일정을 옮기면, 성공 스낵바가 뜨고 단일 일정으로 수정된다.(e2e)
  4. D&D로 겹침일정일 경우 알림이 뜬다.(e2e) 1. 계속 진행 ⇒ 일정 바뀜 2. 취소 ⇒ 원상복구

캘린더 클릭으로 날짜추가 기능 (e2e)

  1. 월간뷰에서 캘린더 날짜 클릭 시 왼쪽 폼에 날짜 반영해서 추가된다까지 (통합&e2e)
  2. 주간뷰에서 캘린더 날짜 클릭 시 왼쪽 폼에 날짜 반영해서 추가된다까지 (통합&e2e)

4-2. 통합 테스트

  1. 월간뷰에서 캘린더 날짜 클릭 시 왼쪽 폼에 날짜 반영해서 추가된다까지 (통합&e2e)
  2. 주간뷰에서 캘린더 날짜 클릭 시 왼쪽 폼에 날짜 반영해서 추가된다까지 (통합&e2e)
  3. D&D hook
    • D&D 훅을 만든 팀원에 한 해 작성

## 🧵 워크플로우(승인 게이트 포함)

1. 요구사항 입력

- 입력: 사용자가 기능/변경 요구를 서술
- 결과: PM 단계 실행

2. PM 단계(@pm.md 활용 → Issue 생성)

- 실행: `*create-issue [feature]`
  - [ ] @pm.md가 issue 파일을 업데이트 했다.
- 검토(2-1): `/.cursor/checklists/pm-checklist.md` 기준으로 자체 점검 요약을 생성하여 사용자에게 공유
  - [ ] 검토(2-1) 과정을 실행했다.
- 요약 갱신(2-1-1): Issue의 "🧾 요약 (Summary)"에 상태를 `기획`으로, 마지막 수정 에이전트를 `Issue Writer (PM)`로, 주요 변경 요약을 업데이트
- 승인(2-2): 사용자에게 다음 단계 진행 여부를 묻고 명시적 허락을 기다림. 허락할 경우 테스트 설계 단계 실행.

3. 테스트 설계 단계(@test-designer.md 활용)

- 실행: `*design-tests [issue-path]`
  - [ ] @test-designer.md가 issue 파일을 업데이트 했다.
- 검토(3-1): `/.cursor/checklists/test-plan-checklist.md` 기준 요약을 생성하여 사용자에게 공유
  - [ ] 검토(3-1) 과정을 실행했다.
- 요약 갱신(3-1-1): Issue의 "🧾 요약 (Summary)"에 상태를 `테스트 설계`로, 마지막 수정 에이전트를 `테스트 설계 에이전트`로, 주요 변경 요약을 업데이트
- 승인(3-2): 사용자에게 다음 단계 진행 여부를 묻고 허락을 기다림. 허용할 경우 테스트 코드 단계 실행.

4. 테스트 코드 단계(@test-code-developer.md 활용 → RED)

- 실행: `*scaffold-tests [issue-path]`
  - [ ] @test-code-developer.md가 issue 파일을 업데이트 했다.
- 검토(4-1): `/.cursor/checklists/test-code-checklist.md` 기준 점검 요약 후 사용자에게 공유
  - [ ] 검토(4-1) 과정을 실행했다.
- 요약 갱신(4-1-1): Issue의 "🧾 요약 (Summary)"에 상태를 `테스트 코드 작성(RED)`로, 마지막 수정 에이전트를 `테스트 코드 에이전트`로, 추가/수정된 테스트 파일 요약을 기록
- 사용자 검토 및 승인 요청(4-2):
  - [ ] 테스트 코드 작성 완료 후 사용자에게 "테스트 코드 작성이 완료되었습니다. 작성된 테스트 파일을 검토해주세요." 메시지와 함께 작성된 테스트 파일 경로 및 주요 내용 요약을 제공
  - [ ] 사용자 검토 후 승인 대기
- 커밋 승인 요청(4-3):
  - [ ] **Lint 오류 확인**: `read_lints` 도구로 테스트 파일 및 관련 파일의 lint 오류 확인
  - [ ] **Lint 오류 해결**: 모든 lint 오류를 수정하고 재확인
  - [ ] 사용자가 테스트 코드 검토를 승인하면 "커밋 진행해도 될까요?"를 질문하여 명시적 커밋 승인 요청

UI테스트 작성 방법

chromatic 링크 Storybook 링크

사실 사내에서 공통 컴포넌트를 만들고나서 Storybook을 통해 문서화를 했는데도 UI테스트까지는 뭔가 복잡해보이기도 하고 그래서 미뤄뒀던 기억이 납니다. 이번 과제에서 UI테스트를 진행해보니 Storybook만 잘 작성되어 있다면 테스트 자체는 쉽다는걸 알게되었습니다. 예전에는 UI테스트가 어디에 필요할까? 란 생각을 했었는데, 몇년 된 서비스 코드에 요구사항을 추가하다보니 공통 컴포넌트들에 UI변경이 있을때 어디까지 영향이 갈지를 파악하기 힘들때 UI 테스트의 필요성을 느끼게 되는것 같습니다.

시간적인 여유가 없어 모든 컴포넌트에 대한 리팩토링은 진행하지 못하고 시각적 회귀 테스트에서 사용할 컴포넌트들만 분리를 진행했습니다. Storybook을 위한 컴포넌트 분리를 하면서 각 컴포넌트가 UI만 관심사로 가지고 다른 기능들은 외부에서 주입하는 방식으로 분리를 진행했습니다. 스토리별로 최대한 여러 케이스가 확인 가능하도록 작성을 하려 노력했습니다.

e2e 테스트 작정 방법

e2e로 많이 사용하는게 Playwright와 Cypress가 있다고 알고 있습니다. 이번 과제에서는 (Playwright Crx를 이용하면) 좀더 작성하기 쉬운 Playwright를 선택해서 e2e테스트를 작성했습니다. 사실 작성하면서 이미 통합테스트에서 메인 시나리오(일정추가, 일정수정, 일정 삭제등)를 검증하고 있어서 e2e와 통합테스트는 어떤기준으로 차이를 둬야하나에 대한 고민을 좀 했던 것 같습니다.

실무에서 e2e는 비용이 많이드는 테스트인 만큼 꼭 필요한 테스트로 작성이 되면 좋을 것 같은데, 여기서 calendar view가 week인지 month 인지 여부도 확인을 하는게 좋은지? -> 일단 두개의 케이스를 작성하지는 않고 월간뷰에서 만든 이벤트가 주간뷰에서도 조회가 되는지를 테스트 했습니다. -> 수정, 삭제도 동일하게 하나의 액션으로 영향을 받는 세영역(주간뷰, 월간뷰, 이벤트 리스트)을 검증하는 방식으로 테스트를 진행했습니다.

구현 과정에서의 기술적 도전과 해결

독립적인 환경 구성

Playwright 테스트시 직렬로 하게 되면 한 5분 정도의 시간이 걸리는 것 같습니다. e2e 테스트의 단점인 시간이 오래걸리는 문제를 해결하려면 병렬로 테스트가 독립적인 환경에서 동작해야 하는데 하나의 DB를 공유하니 동시에 테스트가 실행되면서 독립적인 테스트가 진행되지 않았습니다.

이걸 해결하기 위한 방법을 찾다가 playwright에서 worker를 여러개 두고, 각 worker에 대한 db를 생성하는 방법을 알아냈습니다. 테스트 실행시 server의 header에 worker의 index값을 전달하면 해당 index값을 가진 e2e-${INDEX}.json 파일을 생성해서 독립적이게 실행하는 방법인데 이걸 목요일 새벽까지 구성해보려했지만 어디서 문제인건지 결국 실패를 했습니다...결국엔 worker를 1로 설정해서 직열로만 테스트가 가능하게 하고 매번 테스트 시작시엔 만들어둔 reset api를 호출해서 db를 비우는 방식을 사용해서 테스트를 독립적으로 실행될 수 있도록 만들었습니다.

테스트를 병렬적이게 하되 서로 영향을 받지 않게 만드는 방식으로, 최대한 겹치지 않는 event를 생성하게 해서 병렬적이게 테스트를 진행하는 방법도 생각해봤지만 추후 유지보수 관점에서 모든 이벤트 데이터를 세심하게 관리해줘야 한다는 점에서 직열로 하는 방식이 최선이라 생각되어 해당 방식을 선택했습니다.

vitest를 이용했을 때와 같이 저희 프로젝트에서는 시간이 중요하기에 이를 위해 fixtures.ts를 구성해서 각 테스트가 시작할때 cleanUp함수를 호출하고 시스템 시간을 테스트용 날짜로 설정해 주었습니다.

리뷰 받고 싶은 내용
