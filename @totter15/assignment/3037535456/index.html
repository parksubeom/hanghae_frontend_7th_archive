Before 패키지에서 발견한 문제점

1. UI 컴포넌트에 비지니스 로직이 섞여 있습니다.

interface BadgeProps {
  children?: React.ReactNode;
  type?: 'primary' | 'secondary' | 'success' | 'danger' | 'warning' | 'info';
  size?: 'small' | 'medium' | 'large';
  pill?: boolean;
  showIcon?: boolean;

// 아래는 비지니스의 요구사항에 따라 추가된 props
  status?: 'published' | 'draft' | 'archived' | 'pending' | 'rejected';
  userRole?: 'admin' | 'moderator' | 'user' | 'guest';
  priority?: 'high' | 'medium' | 'low';
  paymentStatus?: 'paid' | 'pending' | 'failed' | 'refunded';
}

Badge 컴포넌트를 보면 type외에 status, userRole, priority등의 상태값을 받아와 실제로 적용할 type값을 계산하는 로직이 UI컴포넌트 내부에 있습니다. 비지니스 로직은 요구사항의 변경이 일어나기 쉽습니다. 이를 변경이 자주 발생되지 않을 공통 UI 컴포넌트 내부에 작성하게 되면 UI 컴포넌트의 재사용성이 떨어지며, 도메인의 변경사항이 UI컴포넌트를 계속 수정하게 만듭니다.

2. 스타일을 정의하는 방식이 하나로 통일되지 않고 여러 방식으로 작성되어 있습니다.

어떤건 inline style로 어떤건 css파일을 통해서등 이럴경우 프로젝트 전반에서 스타일 일관성이 무너지고 유지보수가 어려워지는 문제가 발생합니다.

3. 다른 도메인에 대해서 성급한 추상화로 인해 컴포넌트가 복잡해졌습니다.

Table 컴포넌트의 경우 user와 post 도메인에서 공통으로 사용되고 있습니다. 두 도메인에서의 UI는 비슷하지만 요구사항이 다릅니다. 이를 하나의 컴포넌트에서 관리하게 되면서 컴포넌트 내부에서 도메인별 예외처리가 추가되고 있고, 이를 위한 props들이 필요하게 되면서 컴포넌트를 복잡하게 만들고 있습니다.

개편 과정에서 집중한 부분

UI와 로직분리 UI는 view에 대한것만 관심사로 가지고, action 함수들은 hook으로 빼주면서 분리를 했습니다.

Alert, Badge같은 공통 컴포넌트의 경우

폴더구조를 어떻게 잡을지 저는 폴더 구조가 곧 컴포넌트를 나누는 기준에도 영향을 준다고 생각했습니다. 예를 들어 atomic 디자인 구조라면 컴포넌트를 atom/molecule/organism 관점으로 나누게 되고, FSD 구조라면 도메인 중심으로 바라보며 UI 컴포넌트의 역할과 위치를 다르게 판단하게 됩니다.

즉, 어떤 폴더 구조를 선택하느냐가 컴포넌트를 어떻게 설계하고 분류할지에 영향을 준다고 생각했습니다. 그런 관점에서 어떻게 폴더구조를 나눌지, 컴포넌트를 추상화할지에 대해 집중한것 같습니다.

어려웠던 점과 해결 방법

1. 디자인 시스템 먼저할지 리팩토링 먼저할지

디자인 시스템을 먼저 만들고 이걸 이용해서 리팩토링을 해야지! -> 🥹그런데 디자인 token을 어느정도로 잡고 시작할지 감이 잘 안옴... -> 😏그럼 리팩토링먼저 하고 이후 공통된부분들을 token으로 만들어서 추후에 추가해야지! -> 🥹근데 컴포넌트를 분리하다보니 variant 부분 사용할 색상종류가 기본적으로 정의된것보다 가짓수가 많음... -> 😏다시 디자인 시스템을 간단하게 만들고 리팩토링 해야지! -> 😇컴포넌트 만들다가 다시 디자인 시스템 만들기... => 프로젝트에 대해 좀더 분석을 많이 하고 들어갔으면 이렇게 작업을 왔다리갔다리 안했지 않았을까 싶습니다...

2.폴더 구조에 대한 고민

Table 컴포넌트를 리팩토링하면서 가장 많이 고민한 부분은 도메인마다 디자인이나 요구사항이 조금씩 달라지는 Table을 어디까지 추상화하고 어떤 기준으로 컴포넌트를 분리할지였습니다.

2-1. shadcn Table을 그대로 둘지, 합성한 Table을 ui에 둘지에 대한 고민

처음에는 shadcn이 생성한 Table 컴포넌트를 그대로 유지한 상태에서, 이를 조합해 만든 완성된 Table 컴포넌트를 ui/Table에서 하나만 export하는 방식으로 진행했습니다.

하지만 이렇게 하다 보니 고민이 생겼습니다: “완성된 Table만 export 할 거라면, 내부에서 사용하지 않을 합성 컴포넌트들을 굳이 둘 필요가 있을까?” “그렇다고 shadcn이 분리해준 컴포넌트들을 다시 합쳐서 컴파운드패턴이 아닌 Table을 만들기에는 shadcn을 사용하는 의미가 줄어드는 건 아닐까?” “shadcn 파일은 그대로 둔다면, 그걸 조합한 새로운 Table 컴포넌트는 또 어디에 두는 게 맞을까?”

이걸 해결하기 위해서 fsd 방식을 빌린다면 shadcn으로 만들어진 파일들은 shared에 넣고 이걸로 만든 컴포넌트들은 widget/ui에 넣으면 되지 않을까? + Table이 도메인별로 만들어질것 같은데 이걸 관리할 수 있지 않을까? 란 생각에 fsd 폴더구조로 현재 폴더를 변경해보았습니다.

3. 디자인 시스템의 정리

tailwind를 제대로 써본적이 없다보니 디자인 시스템을 어떻게 정의하면 좋을지 고민이 있었습니다. 일단 컬러와 텍스트를 먼저 primitive token으로 정의하고 이걸 조합해서 의미를 담은 sementic token으로 사용하고자 했습니다.

4. 컴포넌트의 추상화를 어디계층에서 하면 좋을지

현재 프로젝트는 유저 / 게시글처럼 탭 단위로 도메인이 나뉘어 있고, 각 도메인은 **비슷한 UI 패턴(테이블, 모달, 통계 박스 등)**을 공유합니다. 그래서 이 공통 UI들을 어떤 계층에서 추상화하는 것이 좋은지 고민이 있었습니다.

공통 컴포넌트 계층 먼저 하위 단위부터 분리해 공통 테이블의 기반을 설계했습니다. DataTable => shadcn Table 컴포넌트를 조합하여 만든 공통 UI 컴포넌트 useTable => table 데이터조회시 페이지별로 불러와지는게 아니라 모든 리스트가 조회됩니다. 이를 페이지 단위로 잘라내주는 hook 이 두 요소를 통해 프로젝트 전반에서 재사용 가능한 테이블 기반을 만들었습니다.

도메인 컴포넌트 계층 그 다음, 공통 테이블 기반 위에 도메인별 요구사항을 반영했습니다. 각 테이블에는 도메인과 관련된 columns와 도메인에 특화된 badge, action button들이 들어갑니다.

페이지 계층 마지막으로 실제 페이지에서 사용하는 데이터 로직을 분리했습니다. useUserTableData => user table을 위한 get / create / edit / delete 로직 usePostTableData => post table을 위한 get / create / edit / delete / ... 로직 Page 단에서는 UserTable / PostTable 컴포넌트와 각 도메인 데이터 훅을 조합하여 UI를 구성합니다.

위처럼 3가지 계층에서 추상화를 진행해 주었습니다.

(사실이 정도 단계로 분리하고 나니 ManagementPage를 이해가기 쉬워서 이단계까지만 추상화를 하고 선택한 domain에 따라 if문을 통해 보여줄 table등은 분기처리를 했는데 지금와서 생각해보니 통계박스와 테이블등을 묶어서 컴포넌트로 만드는 것도 괜찮지 않을까란 생각이 들긴합니다.)

리뷰받고 싶거나 질문하고 싶은 내용

  1. 이런 프로젝트에서 폴더구조는 어떻게 하는게 좋을까요? 요즘 실무에서는 fsd를 많이 사용할까요?

  2. shadcn-ui를 사용하면서 컴포넌트 구조와 Storybook 기준을 어떻게 가져가는 게 좋을지 고민이 있습니다.

  • Alert, Badge 같은 일부 shadcn 컴포넌트는 shadcn이 생성한 원본 파일을 직접 수정해서 사용하고 있고, Storybook도 이 원본(ui/Alert 등)을 기준으로 작성하고 있습니다.
  • 반면 Table 컴포넌트는 shadcn 원본은 거의 그대로 두고, 비즈니스 요구사항을 반영한 DataTable이라는 합성 컴포넌트를 따로 만들어서 사용하고 있으며, Storybook도 DataTable만 작성했습니다.

이렇게 하다 보니 어떤 컴포넌트는 shadcn 원본을 Storybook에 넣고, 어떤 컴포넌트는 합성 컴포넌트만 Storybook에 넣는 등 기준이 일관적이지 않게 되었습니다. shadcn 컴포넌트는 원본 코드는 그대로 두고 한 번 더 wrapping한 컴포넌트를 따로 만들어 Storybook은 이 wrapping 컴포넌트 기준으로 모두 작성하는 방식이 더 좋은 방식일까요?

그런데 shadcn에서 수정이 별로 없는 컴포넌트는 따로 wrapping에서 컴포넌트를 만드는게 좋은 방법인지를 잘 모르겠습니다... 실무에서는 공통 컴포넌트를 만들때 shadcn을 어떻게 사용하고 계실까요?

  1. 컴포넌트의 추상화에 대해 혹시 코치님의 기준이 있을까요? 예를 들어 모달의 경우 유저정보를 생성하는 모달, 유저정보를 수정하는 모달, 게시글을 생성하는 모달, 게시글을 수정하는 모달이 있을 수 있습니다. 저는 이걸 UserDialogContent(유저정보 생성, 수정), postDialogContent(게시글 생성, 수정)으로 분리를 했는데 이걸 누군가는 종류별로 4가지 컴포넌트를 만들수도 있을것 같습니다. 어느정도 기준에서 컴포넌트를 묶는게 좋을지 판단하는 좋은 기준이 혹시 있을까요?

Q) shadcn-ui를 사용하면서 컴포넌트 구조와 Storybook 기준을 어떻게 가져가는 게 좋을지 고민이 있습니다.

A) 제가 아직 실무에서 깊이 있게 shadcn을 사용해본 경험은 많지 않아서 어떤 경험이 좋다더라 라고 이야기 하기는 힘든 감이 없지않아 있네요.

다만 확실한건 shadcn은 쉬운 "수정"을 추구하기 때문에 설치하는 방식이 아니라 내재화 하는 방식으로 제공한다고 인지하고 있는 상태입니다.

그래서 원본 컴포넌트를 랩핑하기보단, 그냥 막 수정해보는거죠 ㅋㅋ 대신 확장성있는 인터페이스를 유지하는 방식으로!
