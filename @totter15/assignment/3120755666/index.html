과제 셀프회고

이번 과제를 진행하면서 Context를 어떻게 사용하는 것이 좋은지에 대해 한 단계 더 이해할 수 있었습니다. 실무에서는 Context를 적극적으로 사용하지 않거나, 사용하더라도 전역 범위에서만 적용했던 경우가 많았는데, 이번 과제에서 성능 최적화를 진행하며 필요한 최소한의 범위에 Context를 적용하는 방식이 더 적절하다는 점을 체감할 수 있었습니다. 또한 Context의 변경이 리렌더링에 미치는 영향과, 이를 제어하기 위한 memo의 동작 방식에 대해서도 보다 명확하게 이해하는 계기가 되었습니다.

Promise.all 수정

현재 코드에서는 각 비동기 함수 호출에 await가 사용되고 있어, Promise가 settled될 때까지 다음 코드가 실행되지 않습니다. 따라서 fetchMajors와 fetchLiberalArts는 순차적으로 실행되며, 각 함수의 반환값이 실행 순서에 따라 Promise.all의 결과 배열에 저장됩니다. 이로 인해 전체 소요 시간은 각 API 호출이 병렬로 처리될 때가 아닌, 모든 호출이 순차적으로 완료되는 데 걸리는 시간이 됩니다.

(동일 API를 중복으로 호출하는건 일부러 리스트를 길게 만들고자 하는걸로 이해해서 그대로 뒀습니다...!)

시간표 블록 드롭시 렌더링 최적화

문제1 시간표 블록 드롭시 해당되는 수정한 시간표뿐만 아니라 전체 시간표가 리렌더링이 됩니다.

원인 useContext의 context를 참조하는 컴포넌트는 memo가 되어도 context값이 변경되면 리렌더링되는게 원인이었습니다.

  1. 시간표 A에서 드래그 종료
  2. 시간표 A의 `ScheduleDndProvider`의 `handleDragEnd` 실행
  3. `setSchedulesMap` 호출 → 전역 `schedulesMap` 업데이트
  4. 모든 `ScheduleDndProvider`가 전역 `schedulesMap`을 구독하고 있으므로:
    - 시간표 B의 `ScheduleDndProvider`도 재렌더링
    - 시간표 C의 `ScheduleDndProvider`도 재렌더링
  5. 각 `ScheduleDndProvider` 재렌더링 → 하위 `DraggableSchedule` 모두 리렌더링

해결

ScheduleContext에서 scheduleValue와 setScheduleValue 로 context를 분리해주었습니다.

export const ScheduleProvider = ({ children }: PropsWithChildren) => {
  const [schedulesMap, setSchedulesMap] = useState<Record<string, Schedule[]>>(dummyScheduleMap);

  return (
    <SetScheduleContext.Provider value={setSchedulesMap}>
      <ScheduleContext.Provider value={schedulesMap}>{children}</ScheduleContext.Provider>
    </SetScheduleContext.Provider>
  );
};
