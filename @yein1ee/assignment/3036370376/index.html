Chapter3-1. UI 컴포넌트 모듈화와 디자인 시스템

과제 목표

레거시 코드베이스를 현대적인 디자인 시스템으로 개편하는 실무 경험

  1. 정리되지 않은 레거시 코드의 문제점 식별 및 분석
  2. TailwindCSS, shadcn/ui, CVA 등의 현대 도구 활용
  3. 일관된 디자인 토큰과 컴포넌트 API 구축
  4. UI와 비즈니스 로직이 적절한 분리된 리팩토링

Before 패키지 분석 후 After 패키지 개편

개편 목표

디자인 시스템

  • TailwindCSS 기반 일관된 디자인 토큰 정의
  • 하드코딩 제거, 재사용 가능한 스타일 시스템 구축
  • dark mode, 반응형 등 확장 가능한 구조

컴포넌트 아키텍처

  • UI 컴포넌트는 순수하게 UI만 담당
  • 도메인 로직은 적절히 분리
  • 일관된 컴포넌트 API 설계

사용할 도구

TailwindCSS 4.x

  • 디자인 토큰 기반 스타일링
  • 유틸리티 클래스 활용
  • dark mode, 반응형 내장 지원

shadcn/ui

  • Radix UI 기반, 접근성 내장
  • 복사 가능한 컴포넌트 (라이브러리가 아닌 소스코드)
  • 자유로운 커스터마이징

CVA (Class Variance Authority)

  • 선언적 variants 패턴
  • 타입 안전한 스타일 조합
  • 조건부 스타일링 처리

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

필수 과제

1. 디자인 시스템 구축

  • TailwindCSS 설정 및 디자인 토큰 정의
  • shadcn/ui 컴포넌트 설치 (Button, Input, Select, Card, Table 등)
  • CVA를 활용한 variants 패턴 적용
  • 일관된 스타일 시스템 구축

2. Before 패키지 분석

  • Before 패키지 실행 및 전체 코드 탐색
  • 스타일링, 컴포넌트 설계, 폼 관리 측면에서 문제점 파악
  • 개선이 필요한 부분과 그 이유 정리

3. 컴포넌트 개편

  • UI와 비즈니스 로직 분리
  • 순수한 UI 컴포넌트로 재구성
  • 일관된 컴포넌트 API 설계
  • 적절한 컴포넌트 구조 설계

심화 과제

  • Dark Mode 완전 지원 (CSS Variables + Tailwind)
  • Design Token 시스템 고도화 (색상 팔레트, 타이포그래피 스케일)
  • 뷰와 비즈니스로직이 분리되도록

과제 회고

이번 과제의 코드를 처음 봤을 땐 현업 프로젝트의 코드를 보는 것 같았습니다. 현재 참여하고 있는 프로젝트에서 겪고 있는 문제(하드코딩된 스타일, 개발자마다 제각각인 스타일, 인라인 스타일의 과도한 사용, 많은 역할을 가진 컴포넌트 등..)들이 고스란히 투영되었기 때문입니다. 현업에서 디자인 시스템을 도입하자고는 했지만 디자인 시스템에 대한 이해가 깊지 않다는 것을 깨달았습니다. 기존에는 디자인 시스템을 단순히 스타일을 통일하는 도구 정도로만 인식했는데, 이번 과제를 통해서 디자인 시스템의 핵심은 의미 기반으로 UI를 구조화하는 데 있다는 것을 명확히 이해하게 되었습니다. 그리고 디자인 시스템과 컴포넌트 아키텍처는 분리해서 생각할 수 없는 개념이라는 생각이 들었습니다. 디자인 시스템을 효과적으로 활용하기 위해서는 먼저 컴포넌트가 명확한 역할을 갖도록 설계되어 있어야 한다고 생각했습니다.

Before 패키지에서 발견한 문제점

  1. 디자인 토큰 부재, 스타일 하드 코딩 inline style과 전역 CSS에 의존하고 있음
<div style={{ minHeight: '100vh', background: '#f0f0f0' }}>
  <div style={{ maxWidth: '1200px', margin: '0 auto', padding: '20px' }}>
    ...
  </div>
</div>

컴포넌트의 스타일이 components.css 안에 hex/px 값으로 하드코딩 되어 있음

.btn-primary {
  background-color: #1976d2;
  color: #fff;
  border-color: #1565c0;
}

컴포넌트가 atoms, molecules, organisms로 나뉘어 있지만, 스타일은 components.css의 전역 클래스에 묶여 있음

<button className={`btn btn-${variant} btn-${size}`} />
  1. Management.tsx에서 모든 것을 처리하고 있음
 - 엔티티 상태 관리 (`entityType: 'user' | 'post')
 - API 호출 (`userService`, `postService`)
 - 테이블 컬럼 정의
 - 생성/수정/삭제/게시/보관/복원 로직
 - 폼 상태(`formData`) 및 모달 상태(`isCreateModalOpen`, `isEditModelOpen`)
 - 성공/에러 Alert 관리
...
  • 단일 책임 원칙 위반
  • 확장성과 재사용성이 떨어짐
  • 테스트/유지보수 난이도 증가
  1. 폼 상태 관리 방식
  • formData를 useState<any>로 관리하여 타입 안정성 부족
  • 검증 로직이 중복되거나 분산됨
  1. entityType 분기 사용 if (entityType === 'user') ... else ... 패턴이 ManagementPage 전체에 퍼져 있어 유지보수가 어렵고 한 엔티티의 동작을 이해하기 어려움

개편 과정에서 집중한 부분

사용한 기술 스택 경험

  1. tailwindCSS 현업에 도입하는 tailwind 버전을 v4로 준비중인데, 브라우저 호환성 때문에 v3을 선호하신다는 분들의 의견을 듣고 한 대 맞은 느낌이었습니다. 기술의 신규성보다 브라우저 호환성과 실제 사용자 환경을 최우선으로 고려해야 한다는 중요한 원칙을 놓치고 있었기 때문입니다.
  2. shadcn/ui 현업에서 사용하던 MUI 대비 커스텀이 굉장히 용이했습니다. 컴포넌트의 소스 코드를 직접 관리할 수 있어서 스타일 오버라이드의 복잡성 없이 디자인 시스템을 구축할 수 있다는 것이 큰 장점인 것 같습니다. 이전에는 단순히 남들이 많이 쓰는 조합이라 채택했지만, 이번 경험을 통해 왜 많이 쓰이는지에 대한 기술적 확신을 얻었습니다. 기술 채택의 이유를 스스로 명확히 하는 것이 프로젝트의 방향성을 설정하고 능동적으로 문제를 해결하는 데 중요하다는 것을 한 번 더 깨달았습니다.
  3. cva 컴포넌트의 state나 variant에 따른 클래스 변화를 선언적이고 재사용 가능하게 관리할 수 있어서 좋았습니다. 복잡한 조건부 클래스 조합을 방지하고 코드의 가독성을 크게 높여주는 좋은 방법인 것 같습니다.
  4. storybook 테스트 챕터 과제에서 시간 부족으로 다뤄보지 못했던 Storybook을 어설프게나마 찍먹 해봤습니다. 컴포넌트를 격리된 환경에서 확인할 수 있어서 좋았고, 팀원들과 효과적으로 컴포넌트를 공유하고 문서화하기 위해 더 공부해야할 필요성을 느꼈습니다.

사용하지 못한 기술 스택에 대한 아쉬움

  • react hook form & zod 꼭 사용해보고 싶었는데 시간 부족으로 그러지 못해서 아쉽습니다. 현업에서는 useState만 사용해서 폼 상태와 유효성 검증 로직을 관리하고 있어서 코드가 복잡하고 유지보수가 어렵기 때문에 현업에 꼭 도입하고 싶은 부분 중 하나입니다.

어려웠던 점과 해결 방법

[어려웠던 점] 항해를 통해서 코드의 문제점과 개선 방향을 어렴풋이 인지하는 수준까지는 도달했지만 여전히 실제 코드를 마주했을 때 어디서부터 고쳐나가야 할지 막막함을 느끼는 상태입니다. 무턱대고 코드를 수정했다가는 기존 기능이 깨질 것 같다는 불안감이 컸고, 테스트의 부재가 더욱 크게 다가왔습니다..ㅎ [해결 방법] 아직 해결 못했습니다. AI의 도움을 받아서 과제는 진행했지만, 도움 없이 혼자서 처음부터 끝까지 개선하는 것은 어려울 것 같습니다.
