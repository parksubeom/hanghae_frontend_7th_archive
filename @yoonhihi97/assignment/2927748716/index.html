[3팀 이윤지] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

by yoonhihi972025.10.19

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요.

문제 상황

여러 테스트가 동시에 실행되면 같은 events 배열을 공유하게 되어, 한 테스트에서 추가/수정한 이벤트가 다른 테스트에 영향을 줄 수 있습니다.

해결 방법

1. 독립적인 배열 생성

export const setupMockHandlerCreation = (initEvents = [] as Event[]) => {
  let events = [...initEvents]; // 함수 호출할 때마다 새 배열 생성
  // ...
}

각 setup 함수를 호출할 때마다 새로운 events 배열을 만들어서, 각 테스트가 독립적인 상태를 가지도록 했습니다.

2. MSW 핸들러 우선순위

server.use()로 추가한 핸들러는 기본 핸들러보다 먼저 매칭됩니다. 그래서 각 테스트가 자기만의 핸들러를 사용하게 되고, 그 핸들러는 클로저로 독립적인 events 배열을 참조합니다.

3. 자동 정리

setupTests.ts의 afterEach에서 server.resetHandlers()를 실행하기 때문에, 각 테스트가 끝나면 추가했던 핸들러가 자동으로 제거됩니다.

결과

  • 테스트 A: events_A 배열로 동작
  • 테스트 B: events_B 배열로 동작 (병렬 실행 가능)
  • 테스트 종료 시 핸들러 자동 정리

이렇게 해서 병렬로 테스트를 돌려도 서로 간섭하지 않고 안정적으로 동작합니다.

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

1. setupTests.ts - 각 테스트 전후로 정리 작업

afterEach(() => {
  server.resetHandlers(); // server.use()로 추가한 핸들러 제거
  vi.clearAllMocks();     // mock 호출 기록 초기화
});

역할: 테스트가 끝날 때마다 추가한 핸들러와 mock을 깨끗하게 정리해서, 다음 테스트에 영향을 주지 않습니다.

2. handlersUtils.ts - 테스트별 독립적인 데이터 저장소

export const setupMockHandlerCreation = (initEvents = []) => {
  let events = [...initEvents]; // 독립적인 배열
  server.use(/* 이 배열만 사용하는 핸들러 */);
};

역할: 각 테스트마다 자기만의 이벤트 배열을 만들어서 사용합니다.

3. 알림 테스트 - 시간 제어

describe('알림 기능', () => {
  beforeEach(() => {
    vi.useFakeTimers({ shouldAdvanceTime: true }); // 가짜 시간 사용
    vi.setSystemTime(new Date('2025-10-23T13:50:00')); // 시간 고정
  });

  afterEach(() => {
    vi.useRealTimers(); // 실제 시간으로 복원
  });

  it('알림 테스트', async () => {
    await vi.advanceTimersByTimeAsync(1000); // 1초 진행
    // 알림 확인...
  });
});

역할:

  • vi.useFakeTimers() - 시간을 제어할 수 있게 만듦
  • vi.advanceTimersByTimeAsync(1000) - 1초를 진행시켜서 setInterval 실행
  • vi.useRealTimers() - 테스트 후 복원

핵심 정리

독립성을 보장하는 3가지:

  1. 데이터 독립: let events = [...initEvents] - 각 테스트마다 새 배열
  2. 핸들러 정리: server.resetHandlers() - 테스트 후 자동 정리
  3. 시간 격리: vi.useFakeTimers() + vi.useRealTimers() - 시간 제어 후 복원

과제 셀프회고

기술적 성장

테스트 코드와의 첫 만남

테스트 코드를 처음 접했을 때는 "어떤 구조로 작성해야 하는지", "어떤 도구를 사용해야 하는지" 전혀 몰라서 막막했습니다. 하지만 describe로 작성된 요구사항과 훅, 유틸 함수들을 하나씩 파악하며 점차 감을 잡아갔습니다.

유닛 테스트 작성 과정:

  • 요구사항을 읽고 "이 테스트를 하려면 어떤 것을 사용해야 할까?"를 고민
  • 필요한 도구를 찾아가며 테스트 코드 작성
  • 반복하면서 자연스럽게 테스트 코드와 친해짐

통합 테스트 진행:

  • 유닛 테스트에서 배운 경험을 바탕으로 더 복잡한 시나리오 테스트
  • 여러 컴포넌트와 기능이 함께 동작하는 상황을 검증하며 실전 감각 익힘

구현 과정에서의 기술적 도전과 해결

1. MUI 컴포넌트 테스트의 벽

처음에는 "요소 찾는다 → 누른다 → 옵션 선택한다" 같은 단순한 방식으로 접근했는데, MUI 컴포넌트의 요소를 찾지 못해서 계속 막혔습니다.

해결 과정:

  • @mui/material 패키지 속 테스트 코드를 직접 확인
  • MUI 컴포넌트의 내부 구조를 이해하고, role이나 특정 attribute로 접근하는 방법 학습
  • 예: Select 컴포넌트는 combobox role을 가진 요소를 클릭한 후, aria-label로 옵션을 선택해야 함
const viewSelect = screen.getByLabelText('뷰 타입 선택');
const dropdown = within(viewSelect).getByRole('combobox');
await user.click(dropdown);

const weekOption = screen.getByLabelText('week-option');
await user.click(weekOption);

배운 점: 라이브러리 공식 테스트 코드를 참고하는 것이 가장 정확한 방법이라는 걸 깨달았습니다.

2. 시간 기반 알림 테스트

알림이 특정 시간에 뜨는지 확인하는 테스트에서 "시간 흘렀는데 왜 안 떠?!"라며 타임아웃 에러를 계속 만났습니다.

문제 원인: 실제 시간이 흐르길 기다리는 게 아니라, 테스트 환경에서는 시간을 직접 제어해야 했습니다.

해결 방법:

// 1. 가짜 시간 모드 활성화
vi.useFakeTimers({ shouldAdvanceTime: true });

// 2. 테스트 시작 시간 고정 (오후 1시 50분)
vi.setSystemTime(new Date('2025-10-23T13:50:00'));

// 3. 시간을 1초 진행 → setInterval 콜백 실행
await vi.advanceTimersByTimeAsync(1000);

동작 원리: 실제 시간이 흐르는 걸 기다리지 않고, advanceTimersByTimeAsync로 시간을 "강제로 진행"시켜서 타이머 기반 로직(알림 체크 등)을 즉시 테스트할 수 있습니다.

배운 점: 테스트에서 비동기나 시간 기반 로직은 vi.useFakeTimers()로 제어해야 안정적으로 동작한다는 걸 배웠습니다.

결과적으로: 막막했던 테스트 코드가 이제는 자연스럽게 작성할 수 있게 되었고, 문제를 만났을 때 공식 문서나 라이브러리의 테스트 코드를 참고하는 습관을 갖게 되었습니다.

코드 품질

이번 주차는 코드 품질 측면에서 많이 아쉬웠던 것 같습니다. 테스트 코드를 처음 접해봐서 "뭐가 더 나은지" 갈팡질팡하며 여러 번 수정하고 되돌리다 보니 일관성이 떨어지는 코드가 완성되었습니다.

그래도 테스트 코드에 익숙해지는 데 최대한 집중했고, 이제는 어느 정도 자신감을 가지고 작성할 수 있게 되었습니다.

리팩토링이 필요한 부분

1. Mock Data 관리의 일관성 부족

Q&A 과제 세션에서 조언받은 대로 "한 테스트를 봤을 때 어떤 걸 하는지 한눈에 파악할 수 있도록" Mock Data를 스태틱하게 넣으려고 노력했습니다. 하지만 마음이 조급해지다 보니 Mock Data가 여기저기 분산되어 있습니다.

현재 상태:

  • 어떤 테스트는 파일 상단에 mockEvents 정의
  • 어떤 테스트는 테스트 케이스 내부에 직접 정의
  • 어떤 테스트는 setupMockHandler의 파라미터로 전달

개선 방향: 명확한 기준을 세워서 일관성 있게 처리하고 싶습니다. 예를 들어:

  • 여러 테스트에서 공통으로 사용하는 데이터 → 파일 상단에 정의, 혹은 변수명을 확실하게 지어서 파일로 따로 빼기
  • 특정 테스트에만 필요한 데이터 → 테스트 케이스 내부에 정의

2. 해결하지 못한 고민들

코드를 작성하면서 중간중간 고민되는 부분은 주석으로 남겨뒀는데, 시간 분배 실패로 대부분의 고민에 대한 해답을 내놓지 못했습니다.

이런 고민들에 대해 팀원들과 논의하거나 더 찾아보면서 명확한 기준을 세우고 싶습니다.

3. 테스트 구조의 일관성 이게 나을까? 저게 나을까? 하면서 많이 수정하다보니 일관성 없는 코드가 완성되었습니다. 개선 방향을 고민해보고 리팩토링이 필요할 것 같습니다.

개선 방향:

  • Given-When-Then 구조를 더 명확하게 적용
  • 테스트 케이스 내 주석을 일관된 방식으로 작성
  • 중복되는 setup 로직을 helper 함수로 분리

학습 효과 분석

가장 큰 배움이 있었던 부분

테스트 코드의 가치 체감

이전까지는 솔직히 "테스트 코드가 정말 필요한가?"라는 생각이 들었습니다. 구현하기도 바쁜데 테스트까지 작성하는 게 비효율적으로 느껴졌거든요.

하지만 직접 작성해보면서 생각이 완전히 바뀌었습니다. 멘토링과 Q&A 세션에서 강조하셨던 "테스트 코드의 가장 큰 효과는 안정감"이라는 말을 이제 이해하게 되었습니다.

테스트 코드를 작성하면서 느낀 점:

  1. 사전 오류 방지: 구현된 코드에 대해 확실하게 검증하고, 배포 전에 미리 알 수 있는 오류들을 잡아낼 수 있었습니다.
  2. 명세서 역할: 항상 프로젝트를 팀원에게 공유할 때 "어떻게 설명해야 할지" 막막했는데, 테스트 코드가 살아있는 명세서가 될 수 있다는 걸 깨달았습니다.
  3. 리팩토링 자신감: 코드를 수정할 때 "혹시 다른 기능이 깨지지 않을까?" 걱정했었는데, 테스트가 있으니 안심하고 수정할 수 있었습니다.

실무 적용 가능성

이러한 경험을 바탕으로 회사에서도 테스트 코드를 도입해보고 싶다는 생각이 들었습니다.

점진적 적용 계획:

  • 회사 내 도메인에 중복적으로 사용되는 기능(공통 컴포넌트, 유틸 함수 등)부터 시작
  • 작은 단위의 유닛 테스트부터 작성하며 팀원들과 함께 익숙해지기
  • 효과를 체감한 후 점차 범위를 확장

기대 효과:

  • 배포 시 불안감 감소
  • 새로운 팀원 온보딩 시 테스트 코드를 통한 빠른 이해
  • 레거시 코드 리팩토링 시 안정성 보장

추가 학습이 필요한 영역

1. 테스트 작성 기준과 범위

아직 "이 기능에는 어떤 테스트 코드가 적절한지" 확실하게 판단하기 어렵습니다.

고민되는 부분:

  • 어느 정도까지 테스트해야 할까? (100% 커버리지가 정답일까?)
  • 통합 테스트와 유닛 테스트의 비율은 어떻게 가져가야 할까?
  • 테스트하기 어려운 코드는 어떻게 처리해야 할까? (외부 API, 복잡한 UI 인터랙션 등)

2. 테스트 코드의 유지보수

실제 프로덕션 코드가 자주 변경되는 환경에서 테스트 코드를 어떻게 효율적으로 관리할 수 있을지도 학습이 필요합니다.

다음 단계: 테스트 코드 작성에 더 익숙해지고, 실무에서 적용 가능한 전략을 수립하기 위해 지속적으로 학습하겠습니다. 특히 "좋은 테스트 코드"와 "나쁜 테스트 코드"의 차이를 명확히 이해하고 싶습니다.

과제 피드백

과제에서 모호하거나 애매했던 부분

"적절히", "적당히" 같은 추상적인 표현

테스트 코드를 아예 처음 접해보는 입장에서 가장 혼란스러웠던 건 요구사항에 나온 "적절히", "적당히" 같은 표현이었습니다.

  • "적절한" 처리가 뭘까? 에러 메시지? alert? 빨간 글씨?
  • 얼마나 구체적으로 테스트해야 할까?
  • 이 정도면 "적절한" 걸까?

처음에는 정말 막막했습니다. 처음에는 아예 감이 안 잡혀서 "예시가 하나라도 있었으면 좋겠다"는 생각도 들었어요. 하지만 지금 돌이켜보면, 예시가 있었다면 너무 그대로 따라 하려고만 했을 것 같기도 합니다. 스스로 고민하고 "이 정도면 적절한 처리일까?"를 판단하며 배운 게 더 값졌던 것 같아요.

명확했으면 좋았을 부분:

  • 추상적인 표현 대신 구체적인 기준 제시 ("에러 메시지가 화면에 표시되어야 합니다" 같은)
  • 테스트 커버리지에 대한 가이드 (어느 정도를 목표로 해야 할까?)
  • MUI 같은 라이브러리 컴포넌트 테스트에 대한 힌트

과제에서 좋았던 부분

1. 실전과 유사한 시나리오

단순히 "버튼 클릭 테스트"가 아니라 일정 관리라는 실제 프로덕트와 유사한 도메인을 다루면서, 실무에서 마주칠 법한 상황들을 경험할 수 있었습니다.

  • 사용자 인터랙션의 흐름 (생성 → 수정 → 삭제)
  • 시간 기반 로직 (알림 기능)
  • 외부 API 모킹 (MSW)

2. 점진적 난이도 상승

유닛 테스트부터 시작해서 통합 테스트로 넘어가는 구조가 학습에 도움이 되었습니다. 처음부터 복잡한 통합 테스트를 작성했다면 완전히 길을 잃었을 것 같아요.

총평

너무너무너무너무 어려웠고, 아직도 제대로 한 건지 잘 모르겠지만... 테스트와 친해지기에는 정말 좋은 과제였던 것 같아요.

(아마도) 다음번에는 좀 더 자신감 있게 테스트 코드를 작성할 수 있을 것 같습니다!

리뷰 받고 싶은 내용

1. 테스트 독립성 검증

독립적으로 만들려고 노력했지만, 결과적으로 정말 독립적인 테스트 코드가 된 건지 확신이 서지 않습니다.

구체적으로 확인받고 싶은 부분:

  • 현재 구조에서 병렬 실행 시 테스트 간 상태 공유 가능성이 있나요?
  • server.use()로 핸들러를 추가하는 방식이 테스트 독립성 측면에서 적절한가요?
  • afterEach의 server.resetHandlers()만으로 충분히 정리가 되는 건가요?
  • 더 안전하고 명확하게 독립성을 보장할 수 있는 패턴이 있을까요?

1. 목 데이터 관리 방식

목 데이터를 파일 상단에 정의하거나, 테스트 내부에 정의하거나, 빈 배열로 시작하는 등 여러 방식을 혼용했습니다. 일관성 없이 작성한 것 같은데, 잘한 부분과 개선이 필요한 부분을 구체적으로 짚어주시면 감사하겠습니다.

2. 테스트 검증 범위

이벤트 추가/수정 테스트에서 입력한 항목(제목, 날짜, 시간, 설명, 위치, 카테고리 등) 중 어디까지 확인해야 적절한지 판단이 어려웠습니다. 현재는 제목, 날짜, 시간만 확인하거나 일부만 검증하는 식으로 일관성 없이 작성했는데, 이 부분이 적절한지 피드백 부탁드립니다.

궁금한 점

1. 테스트 코드 작성 시작점

이번 과제에서는 테스트 케이스가 이미 작성되어 있고 내용을 채우는 식으로 진행했는데, 실무에서 처음부터 테스트를 작성할 때는 어떻게 시작하시나요?

기획서나 기능 요구사항을 보고 "이런 테스트가 필요하겠구나!" 하고 테스트 케이스를 추가하시는 건지, 아니면 다른 방식이 있는지 궁금합니다.

2. 적절한 테스트 방법 찾기

테스트 코드를 처음 접해봤는데 어떻게 공부해야 할지 막막했습니다. 학습 자료와 링크들을 계속 찾아봤지만, "이 테스트에 어떤 방법이 제일 적절할지" 판단하기가 어려웠습니다.

예를 들어 getByRole, getByLabelText, getByTestId 중 어떤 걸 써야 할지, waitFor를 써야 할지 말아야 할지 등... 이런 판단은 어떻게 하시나요?

과제를 수행하면서 느낀점

과제 시작 전 생각

과제 시작 전에는 솔직히 "테스트 코드를 써야 하나?"라는 생각이 들었습니다. 지금 바쁜 일정에 프로젝트 하나 완성시키기도 바쁜데, 그 시간에 차라리 코드를 좀 더 잘 짜는 게 낫지 않나 싶었거든요. 그래서 아예 배워볼 생각을 안 했었고, 이번에 처음 접해보니 어떻게 해야 할지 막막했습니다.

과제 제출 후 생각

하지만 직접 작성해보면서 생각이 완전히 바뀌었습니다. 멘토링과 Q&A 세션에서 강조하셨던 "테스트 코드의 가장 큰 효과는 안정감"이라는 말을 이제 이해하게 되었어요.

테스트 코드를 작성하면서 느낀 점:

  • 구현한 코드를 확실하게 검증하고, 배포 전에 미리 오류를 잡아낼 수 있었습니다
  • 항상 프로젝트를 팀원에게 공유할 때 막막했는데, 테스트 코드가 살아있는 명세서 역할을 할 수 있다는 걸 깨달았습니다

아직도 제대로 작성한 건지는 잘 모르겠지만, 막막했던 테스트 코드가 이제는 조금씩 자연스럽게 느껴지고 있습니다. 회사 업무에서도 공통 컴포넌트나 유틸 함수부터 점진적으로 적용해보고 싶다는 생각이 들었습니다.
