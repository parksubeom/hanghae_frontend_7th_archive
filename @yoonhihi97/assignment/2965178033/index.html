[3팀 이윤지] Chapter 1-2. AI와 테스트를 활용한 안정적인 기능 개발

by yoonhihi972025.10.31

과제 체크포인트

필수 스펙

  1. 반복 유형 선택

    • 일정 생성 또는 수정 시 반복 유형을 선택할 수 있다.
    • 반복 유형은 다음과 같다: 매일, 매주, 매월, 매년
      • 31일에 매월을 선택한다면 → 매월 마지막이 아닌, 31일에만 생성하세요.
      • 윤년 29일에 매년을 선택한다면 → 29일에만 생성하세요!
    • 반복일정은 일정 겹침을 고려하지 않는다.

  2. 반복 일정 표시

    • 캘린더 뷰에서 반복 일정을 아이콘을 넣어 구분하여 표시한다.

  3. 반복 종료

    • 반복 종료 조건을 지정할 수 있다.
    • 옵션: 특정 날짜까지
      • 예제 특성상, 2025-12-31까지 최대 일자를 만들어 주세요.

  4. 반복 일정 수정

    1. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      • 반복일정을 수정하면 단일 일정으로 변경됩니다.
      • 반복일정 아이콘도 사라집니다.
    2. ‘해당 일정만 수정하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      • 이 경우 반복 일정은 유지됩니다.
      • 반복일정 아이콘도 유지됩니다.

  5. 반복 일정 삭제

    1. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘예’라고 누르는 경우 단일 수정
      1. 해당 일정만 삭제합니다.
    2. ‘해당 일정만 삭제하시겠어요?’ 라는 텍스트에서 ‘아니오’라고 누르는 경우 전체 수정
      1. 반복 일정의 모든 일정을 삭제할 수 있다.

기본 과제

공통 제출

  • 테스트를 잘 작성할 수 있는 규칙 명세
  • 명세에 있는 기능을 구현하기 위한 테스트를 모두 작성하고 올바르게 구현했는지
  • 명세에 있는 기능을 모두 올바르게 구현하고 잘 동작하는지

기본 과제(Easy)

  • AI 코드를 잘 작성하기 위해 추가로 작성했던 지침
  • 커밋별 올바르게 단계에 대한 작업
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

기본 과제(Hard)

  • Agent 구현 명세 문서 또는 코드
  • 커밋별 올바르게 단계에 대한 작업
  • 결과를 올바로 얻기위한 history 또는 log
  • AI 도구 활용을 개선하기 위해 노력한 점 PR에 작성

심화 과제

  • 모든 질문에 대해 꼼꼼하게 정리했는지

과제 셀프회고

기술적 성장

1. 페르소나 주입 방식의 개선

문제 상황:
처음에는 "당신의 역할이 이거다!"라고만 지시했더니, 테스트 설계자 에이전트가 자기 역할을 제대로 인지하지 못하고 테스트 코드를 작성하는 등 다른 에이전트의 역할을 침범하는 문제가 발생했습니다.

해결 과정:
단계별로 명확한 페르소나를 부여 해봤습니다.:

  • "QA 리드이자 테스트 전략 컨설턴트"
  • "TDD를 철저히 지키는 시니어 프론트엔드 엔지니어"
  • "실무 경험이 풍부한 시니어 소프트웨어 엔지니어"
  • "클린코드와 유지보수성을 중시하는 리드 엔지니어"

결과:
각 에이전트가 자신의 책임 범위를 명확히 인지하여 역할 충돌 문제가 해결되었습니다.

2. 테스트 설계 프로세스 재구성

문제 상황:
초기에는 테스트 설계 단계에서 바로 describe('반복 유형 선택', () => { it('RS-1: 매일(daily) 선택', () => {}) }) 같은 테스트 케이스 틀을 생성하게 했습니다. 하지만 이렇게 하니 프로젝트 구조나 App.tsx 같은 실제 구현 파일을 이해하지 않은 채 기능 요구사항만 보고 나열하는 식의 설계가 나오게 되었습니다.

해결 과정:
고민 끝에 페르소나를 "QA 리드이자 테스트 전략 컨설턴트"로 설정하고, 틀을 잡게 하는 대신 프로젝트를 먼저 분석한 후 다음을 포함한 전략 문서를 작성하도록 변경했습니다:

  • 테스트 범위
  • 테스트 대상
  • 테스트하지 않는 것
  • 테스트 케이스 분류

이후 이 전략 문서를 바탕으로 테스트 작성자가 코드를 작성하도록 했습니다.

결과:
훨씬 구조화되고 현실적인 테스트가 작성되었습니다.

3. 프롬프트 길이와 효과성의 트레이드오프

문제 상황:
BMAD Code 참고하라, 좋은 테스트 코드 문서 확인하라, 이것도 하지 마 저것도 하지 마, 이건 잘해야 해 등 많은 지침을 프롬프트에 주입하다 보니 너무나.. 길어졌습니다. 이렇게 하니,

  • AI가 오히려 지켜야 할 것을 안 지키기 시작했고
  • 토큰 소모량도 많아져 작업 중간에 5시간 기다려야 하는 상황이 계속 해서 발생해, 과제를 진행하기 어려웠습니다.

해결 과정:
과제 Q&A 세션에서 "너무 길어도 안 좋다"는 조언을 듣고, GPT, 클로드, 커서를 왔다갔다 하며 프롬프트에 대한 피드백을 받아보았습니다.

결론은 핵심 원칙만 요구사항에 녹여내고, 세부 가이드 문서는 참조하라는 방식으로 변경하는 것이었습니다.

예를 들어:

  • ❌ "이 문서를 읽고 진행해!"
  • ✅ "이 문서를 읽어보고 요구사항에 중요한 부분만 넣어줘"

결과:
지침 준수율이 향상되었고 토큰 사용량을 기록하지는 않아 수치로 표현할 수는 없지만 만료 시간이 이전보다는 느리게 찾아오는 것을 확 체감할 수 있었습니다.

4. 작업 범위 명확화 및 검증 체계 구축

문제 상황 1: 범위를 벗어난 작업
"반복 유형 선택" 기능을 만들어달라고 요청했더니 AI가 "아, 이거 하려면 종료일 선택도 만들어야 하고, 수정도 만들어야 하고, 삭제도 만들어야겠네요!"라며 모든 것을 한 번에 처리하려고 시도하는 문제가 발생했습니다.

해결 1:
작업 시작 전에 나에게 질문을 해서 작업 범위를 확인하도록 요청하는 프로세스를 도입했습니다. 그러니 AI가 질문을 하며 "아 그러면 이건 다음번에 하면 되겠네요"라고 인식한 뒤 맡은 업무만 수행하게 되었습니다.

문제 상황 2: 테스트 실패 무시
초기에는 "이 기능 추가해줘"하면 "짜잔 완성"하는 완전 자동화를 꿈꿨지만, 실제로는 테스트 실패를 무시하고 **"15개 중에 3개는 실패했는데 12개는 성공했으니 다음 단계로 넘어갈게요"**라고 하며 넘어가는 문제가 발생했습니다..

해결 2:
오케스트레이션 에이전트에 단계별로 컨펌을 받고, 컨펌을 받지 않으면 다음 단계로 못 넘어가게 하는 로직을 추가했습니다. 컨펌 요청이 들어왔을 때 마음에 들지 않으면 이유를 설명하며 재수정을 요청하도록 했습니다.

결과:
더 나은 결과물을 얻을 수 있었고, "완전 자동화"보다는 **"검증 가능한 자동화"**가 중요하다는 것을 깨달았습니다.

코드 품질

리팩토링이 필요한 부분

오케스트레이션 에이전트의 순서 위반 문제:
가끔씩 오케스트레이션 에이전트가 순서에 맞게 에이전트를 호출해서 처리해야 하는데, 호출하지 않고 혼자 처리를 다 해버리는 경우가 발생했습니다.

개선 방향: 오케스트레이션 에이전트의 로그를 더 자세히 남기고, 각 단계별로 어떤 에이전트가 호출되어야 하는지 명시적으로 체크하는 검증 로직을 추가하려고 합니다. 또한 에이전트 호출 전에 현재 단계와 다음에 실행해야 할 에이전트를 명확히 출력하도록 개선할 예정입니다.

테스트 코드 품질 판단 기준 부족:
테스트 코드 작성 단계에서 코드 작성 단계로 넘어갈 때, 테스트 코드를 이번에 처음 배워보고 TDD에 대해서도 잘 몰라서 그런 것일 수도 있지만, 테스트 코드 작성한 것을 보고 "음 상황에 맞게 기능에 맞게 잘 짜진 것 같다" 하고 승인을 했는데, 막상 코드 작성 단계에서 다시 생각해보면 "테스트 코드가 잘못 작성됐구나..." 싶은 경우들이 있었습니다.

개선 방향:

  • TDD 관련해서 좀더 알아보고 현재 작성해놓은 테스트를 잘 작성할 수 있는 규칙 명세(.claude/docs/test-writing-guide.md) 문서를 보완해서 에이전트가 자동으로 체크할 수 있는 체크리스트 형태로 개선하려고 합니다

학습 효과 분석

  • 가장 큰 배움이 있었던 부분:

    • AI 도구 활용에 대한 새로운 관점: 처음에는 AI가 모든 것을 완벽하게 해결해줄 것이라고 생각했지만, 실제로는 예상치 못한 동작들이 많이 발생했습니다. 이를 통해 AI는 도구일 뿐이고, 사람이 명확한 방향을 제시하고 검증해야 한다는 것을 깨달았습니다.

    • 프롬프트 엔지니어링의 복잡성: 단순히 지시사항을 많이 주는 것이 아니라, 핵심 원칙을 명확히 하고 참조 문서를 활용하는 것이 더 효과적이라는 것을 경험을 통해 배웠습니다.

    • 에이전트 기반 개발의 가능성: 다중 에이전트 시스템을 구축하면서 각 에이전트가 명확한 역할을 가질 때 더 나은 결과를 만들어낸다는 것을 배웠습니다.

  • 추가 학습이 필요한 영역:

    • 에이전트에게 더 나은 결과를 얻기 위한 요청 방법, 효과적인 프롬프트 작성법, AI가 지시사항을 잘 따르도록 하는 방법 등에 대한 의문이 많이 남아있습니다. 이를 해결하기 위해 지속적으로 학습하고 실험하며, 현재보다 훨씬 나은 결과를 만들어낼 수 있는 나만의 에이전트 시스템을 구축해보고 싶습니다.

  • 실무 적용 가능성:

    • 과제를 진행하면서 "이거 실무에서도 활용할 수 있겠다!"라는 생각이 자주 들었습니다. 현재 회사에서는 상품이 안정화 단계에 접어들어 새로운 프로젝트를 시작하지 않는 상황이지만, 기존 프로젝트의 공통 기능들에 에이전트를 활용한 테스트 코드를 점진적으로 도입해볼 계획입니다.
    • 특히 QA 과정에서 자주 발생하는 "사용자가 특정 행동을 했을 때 예상한 결과가 나오지 않는" 문제들을 사전에 방지할 수 있을 것 같습니다. 대부분 조건문 누락이나 엣지 케이스 미처리로 인한 문제인데, 에이전트가 이런 시나리오들을 미리 검증해준다면 버그를 크게 줄일 수 있을 것 같아서 꼭 적용해보고 싶습니다.

과제 피드백

  • 과제에서 좋았던 부분:

    • 에이전트를 만들어보고 사용해본 경험이 제일 좋았던 것 같아요. 지금까지는 AI를 활용할 때 "나 지금 앱 빌드 안되는데 왜그런지 모르겠어 해결해줘", "어떤 사용자한테는 이게 이렇게 보인다는데 왜그런지 모르겠어 디버깅해줘" 이런 식으로 막연한 요청사항으로 "해줘" 하는 식으로 많이 사용했었는데, 이번 과제를 하면서 새롭게 알게되는 부분들을 보며 내가 얼마나 AI를 활용을 못했는지, 이래서 남들이 다들 "클로드 짱!"을 외칠 때 내 클로드는 바보야... 라고 했는지 알 것 같았습니다. 이번주에 배운 내용을 토대로 좀 더 공부하고 실무에 꼭 적용 해보고싶어요.

  • 과제에서 아쉬웠던 부분:

    • 모호하거나 애매했다기보단 처음에 이해하는데 시간이 많이 걸렸던 것 같아요. "에이전트를 왜 만들지? 얘네를 만들어서 뭐 어떻게 해야하지? 그냥 이름 부르면서 기능 추가해달라고 하면되나..?" 하면서 많이 헤맸고, 이래저래 많이 만져봐야 좋은 결과물을 낼 수 있을 것 같은데 토큰 만료 이슈로 이것저것 실험해보지 못한 채로 과제를 제출해야한다는 사실이 많이 아쉽습니다. 클로드 Max upgrade에 마우스를 올렸다내렸다.. 토큰 다썼다 5시간 기다려라 하는 빨간 문장을 만나면 얼마나 절망적이던지.. 만들어보고 싶고 해보고 싶은건 많은데 토큰 만료로 막혀서 이것저것 많이 시도해보지 못한 점이 아쉬웠어요.

리뷰 받고 싶은 내용

1. 에이전트 워크플로우에서 중간 파일 생성 방식의 효율성

각 단계마다 specs, test-designs 폴더에 파일을 추가하게 해서 이 파일들을 참조하며 다음 단계를 진행하도록 설계했습니다. 이 방식이 토큰 효율성 측면에서 적절한지, 그리고 더 나은 대안이 있는지 궁금합니다.

2. 프롬프트 최적화를 위한 체계적인 접근법

에이전트에게 요구사항을 너무 상세히 적으면 업무 수행률이 떨어지고(제가 잘못 적은 걸수도 있겠지만요..), 간결하게 하고 진행하다보면 음.. 이 내용도 적어야하나? 근데 또 놓치면 어떡하지? 싶은 고민이 있습니다. "답변을 여러 번 얻어보며 공통 문제를 정리해서 추가하는" 방식으로 접근해보려고 하는데 괜찮을까요? 프롬프트 품질을 체계적으로 개선할 수 있는 방법론이나 검증 프로세스가 있을까요?

3. TDD 사이클에서 테스트 코드 재작성의 적절성

현재 순차적 진행 중에 코드 작성 단계에서 "테스트 코드부터 잘못됐다"는 상황이 자주 발생합니다. 이때 테스트 설계나 테스트 작성 단계로 다시 돌려보내는 것이 괜찮을지, 아니면 다른 방식으로 접근하는 것이 더 나은지 궁금합니다.
