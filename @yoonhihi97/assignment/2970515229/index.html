심화 과제 제출

1. 내가 생각한 최적의 테스트 전략

단위 테스트 중심 전략

초기에는 단위 테스트 중심의 접근을 생각했습니다.

  • 핵심 판단 기준

    • 변경에 덜 민감하고 리팩토링에 안정적
    • 실패 시 리스크가 큰 비즈니스 로직 우선 테스트
    • 유지보수 용이성 (개별 함수/컴포넌트 단위 테스트)

  • 한계 발견

    • 현재 App.tsx에 로직이 집중되어 있어 컴포넌트 미분리 상태
    • 단위 테스트를 위해서는 과도한 mock 필요
    • 개별 단위는 통과해도 통합 시 실패 가능성 존재

→ 통합 테스트가 현재 프로젝트에 더 적합하다고 판단

기능 도메인 중심 통합 테스트 전략

  • 현재 구조에서 즉시 작성 가능

배운 점

  • 테스트 파일 분리 기준: 컴포넌트 구조보다 사용자 기능 단위가 더 직관적

  • 점진적 개선: 테스트 먼저 작성하고 이후 안심하고 컴포넌트 리팩토링 진행

과제 셀프회고

기술적 성장

1. E2E 테스트를 통한 사용자 관점의 이해

Playwright를 활용해 실제 사용자 흐름을 테스트하면서 단순 기능 검증을 넘어 사용자 경험 중심의 테스트를 학습했습니다.

단순 API 테스트를 넘어선 통합 검증

  • 반복 일정 생성 → 수정(이 일정만 / 모두) → 삭제까지의 실제 워크플로우를 자동화
  • 일정 겹침 감지 시 다이얼로그가 뜨고, 사용자가 "계속 저장" 또는 "취소"를 선택하는 시나리오 검증
  • 알림 시스템이 지정된 시간에 정확히 표시되는지 확인

테스트 환경 설계의 중요성

  • TEST_ENV 환경 변수를 통해 개발 환경(realEvents.json)과 테스트 환경(e2e.json)을 분리
  • page.clock.install()로 브라우저 시간을 고정하여 특정 시점의 테스트 재현
  • 테스트 시작 전 initializeTestData를 실행해 항상 깨끗한 상태에서 테스트 시작

엣지 케이스 발견

  • "14:00 ~ 16:00 일정이 있을 때 14:30 ~ 15:30 일정 추가" 같은 겹침 시나리오 사전 검증
  • 반복 일정의 "이 일정만" 수정 후 다음 반복에 영향이 없는지 확인
  • 시간 역순 입력, 반복 설정 누락 등 사용자가 실제로 겪을 법한 오류를 조기 발견

테스트로 얻은 확신 테스트 전에는 "코드가 동작한다"는 수준이었다면, 테스트 후에는 "사용자가 기대한 대로 동작한다"는 확신을 가질 수 있었습니다.

2. 시각적 회귀 테스트를 통한 UI 안정성 확보

Storybook과 Chromatic을 활용해 UI의 상태를 시각적으로 관리하고, 변경에 민감한 부분을 자동 검증하는 프로세스를 구축했습니다.

상태별 UI 문서화

  • 폼의 에러, 성공, 입력 중 등 상태별 UI를 Story로 정의
  • 월/주 단위 달력 뷰의 다양한 렌더링 패턴을 시각화
  • 텍스트 길이, 이벤트 길이 등 실제 운영 중 발생할 수 있는 다양한 케이스를 반영

자동화된 변경 감지

  • CSS 수정이나 라이브러리 업그레이드 후 발생할 수 있는 시각적 변화를 Chromatic이 픽셀 단위로 비교
  • 의도하지 않은 변경을 PR 단계에서 사전에 확인해 UI 품질 유지

현실적인 한계와 배운 점 Storybook 작성 초기에 App.tsx의 UI 코드를 복사하여 stories를 구성했는데, 이 방식은 App.tsx 변경 시 stories를 매번 수동 수정해야 하는 문제가 있었습니다. 이 경험을 통해 **"컴포넌트 분리는 선택이 아니라 필수"**라는 점을 실감했습니다.

코드 품질

리팩토링이 필요한 부분

향후 과제에서의 컴포넌트 분리

Storybook을 작성하는 과정에서 컴포넌트를 분리하지 않고 App.tsx의 UI 코드를 복사/붙여넣기 방식으로 stories를 작성했습니다. 과제의 요구사항에 따라 "적절한 스토리 작성"에 집중했으나, 이 방식은 향후 다음과 같은 한계가 있습니다:

  • App.tsx의 UI 수정 시 stories.tsx도 동일하게 수정해야 함
  • 변경 사항을 놓치기 쉬워 시각적 회귀 테스트의 신뢰도 저하
  • 코드 중복으로 인한 유지보수 비용 증가

향후 개선 방향 (이후 주차에서): 주요 UI를 컴포넌트로 분리

  • CalendarHeader, CalendarView, EventForm 등으로 분리
  • 각 컴포넌트를 Storybook에서 직접 import하여 관리
  • 수정 시 자동으로 stories에 반영되어 일관성 유지

코드 설계 관련 고민과 결정

[반복 일정 드래그앤드롭 처리]

반복 일정을 드래그앤드롭했을 때 어떤 동작이 가장 자연스러울지 고민했습니다.

문제 상황: 사용자 입장에서는 월급날(10일)을 15일로 옮기면 모든 월급 일정이 함께 이동되길 기대할 수 있지만, 실제로는 월말(31일이 없는 달), 윤년 등 다양한 엣지 케이스로 인해 구현이 복잡해집니다.

참고 사례: 네이버 캘린더를 살펴보니, 반복 일정을 드래그앤드롭하면 그 일정만 반복 그룹에서 분리되어 '단일 일정'으로 변경되더라고요. 반복 규칙 전체 수정은 입력 폼에서만 가능했습니다.

고민과 결정: 이 방식을 참고하면서도, 저는 이 동작이 사용자가 기대하는 결과는 아닐 수 있다고 판단했습니다. 사용자는 드래그 시 전체 반복이 함께 이동되길 기대할 가능성이 높기 때문입니다. 그래서 이번 과제에서는 혼란을 줄이기 위해 반복 일정의 드래그앤드롭을 제한하는 방식을 선택했습니다.

이 결정의 이유:

  • 드래그 동작의 결과를 명확히 예측할 수 있음
  • 반복 규칙 변경은 입력 폼에서만 허용하여 일관성 유지
  • 엣지 케이스(윤년, 월말 등)에서의 오류 가능성 제거

향후 개선 계획: 향후에는 드래그앤드롭 시에도 반복 일정이 사용자의 예상과 일치하는 방식으로 수정되도록 개선해보고 싶습니다. 예를 들어, 드래그 시 "이 일정만 이동 / 전체 반복 이동"을 선택할 수 있는 다이얼로그를 제공하는 방향을 고려하고 있습니다.

[Chromatic ProjectToken 관리]

포크 프로젝트이기 때문에 원본 레포의 secrets를 사용할 수 없었습니다.

현재 상황: Chromatic 워크플로우에 projectToken을 하드코딩한 상태

향후 개선 계획:

  • 자신의 Chromatic 계정에서 새로운 프로젝트 생성
  • GitHub Actions Secrets로 CHROMATIC_PROJECT_TOKEN 추가
  • CI/CD 실행 시에만 환경변수로 주입되도록 처리
  • 보안 관점: 토큰은 읽기 권한만 필요하므로, Chromatic에서 제한된 토큰 생성 가능

학습 효과 분석

가장 큰 배움이 있었던 부분

의견을 낼 때는 "왜"를 먼저 생각하기

드래그앤드롭 기능 구현 시, 반복 일정의 엣지 케이스 처리에 대해 팀에게 물었습니다. 초반에는 "모든 반복 일정을 함께 이동시키는 게 맞지 않을까?"라는 의견을 제시했는데, 이 과정에서 배웠습니다:

  • 단순히 "이렇게 하면 어떨까?"라고 물어보는 것이 아니라, "왜 그렇게 생각하는지"를 먼저 정리해야 한다는 것
  • 사용자 관점에서의 기대감, 기술적 복잡성, 엣지 케이스 등을 구체적으로 설명하니 팀원들도 더 깊이 있게 함께 고민할 수 있었습니다

이를 통해 네이버 캘린더의 사례를 함께 살펴보고, 결국 현재 과제에서는 혼란을 줄이기 위해 반복 일정의 드래그앤드롭을 제한하기로 결정했습니다.

테스트 주도 설계의 실제 가치

E2E 테스트를 작성하면서 깨달았습니다:

  • 테스트는 단순히 "버그 찾기"가 아니라 **"요구사항을 명확히 하는 도구"**입니다
  • "14:0016:00 일정이 있을 때 14:3015:30 일정을 추가하면?"이라는 구체적인 시나리오를 테스트로 만들면서, 실제로 사용자가 직면할 상황들을 미리 대비할 수 있었습니다
  • Arrange-Act-Assert 패턴으로 테스트를 작성하니 코드의 의도가 명확해지고, 버그가 생겼을 때도 어느 부분이 문제인지 빠르게 파악할 수 있습니다

시각적 회귀 테스트는 신뢰도를 높이는 도구

Storybook으로 UI 상태들을 문서화하고, Chromatic으로 자동 검증하면서:

  • 개발 중에 "혹시 이 변경이 다른 곳에 영향을 줄까?"하는 불안감이 사라집니다
  • CSS 수정 후 PR을 올리면 Chromatic이 변경사항을 시각적으로 보여주니, 코드 리뷰할 때도 훨씬 명확합니다
  • 상태별 UI를 정의하는 과정 자체가 UI 설계의 일관성을 높입니다

추가 학습이 필요한 영역

복잡한 엣지 케이스를 다루는 전략

반복 일정 드래그앤드롭에서 마주친 문제:

  • 월말(31일이 없는 달), 윤년 같은 케이스들을 예측하기 어렵습니다
  • 이런 문제에 마주쳤을 때 "직접 구현해보기"보다는 **"검증된 라이브러리가 있는지 먼저 탐색"**하는 습관이 필요합니다
  • 향후 프로젝트에서는 date-fns, day.js 같은 라이브러리를 활용하는 것을 먼저 고려해야 합니다

대규모 E2E 테스트 관리

현재 E2E 테스트가 완벽하지만:

  • 각 테스트 케이스의 독립성으로 인해 초기화 비용이 반복되고 있습니다
  • 테스트 실행 시간과 신뢰도 사이의 트레이드오프를 더 깊이 있게 이해해야 합니다
  • 팀 규모가 커지면서 수백 개의 E2E 테스트를 관리하는 방법론을 배워야 합니다

AI 에이전트 활용 경험

  • 이전 주차에서 배운 AI 에이전트를 실제 프로젝트에 적용해보는 시도를 했습니다.
  • 처음에는 E2E 테스트 에이전트 하나만 생성하여 설계와 작성을 모두 맡겼으나, 한 에이전트가 여러 단계를 처리하다 보니 결과물의 품질이 기대에 미치지 못했습니다.
  • 이를 개선하기 위해 책임을 분리하여 '설계하는 에이전트'와 '작성하는 에이전트'로 나눠서 처리했고, 각 에이전트가 명확한 역할에 집중하면서 더 나은 결과물을 얻을 수 있었습니다.
  • 각 단계마다 에이전트가 생성한 코드를 직접 확인하고, 필요한 부분은 수정 지침을 제공하여 점진적으로 개선해나갔습니다.
