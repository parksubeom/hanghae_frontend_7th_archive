기본과제

상품목록

상품 목록 로딩

  • 페이지 접속 시 로딩 상태가 표시된다
  • 데이터 로드 완료 후 상품 목록이 렌더링된다
  • 로딩 실패 시 에러 상태가 표시된다
  • 에러 발생 시 재시도 버튼이 제공된다

상품 목록 조회

  • 각 상품의 기본 정보(이미지, 상품명, 가격)가 카드 형태로 표시된다

한 페이지에 보여질 상품 수 선택

  • 드롭다운에서 10, 20, 50, 100개 중 선택할 수 있으며 기본 값은 20개 이다.
  • 선택 변경 시 즉시 목록에 반영된다

상품 정렬 기능

  • 상품을 가격순/인기순으로 오름차순/내림차순 정렬을 할 수 있다.
  • 드롭다운을 통해 정렬 기준을 선택할 수 있다
  • 정렬 변경 시 즉시 목록에 반영된다

무한 스크롤 페이지네이션

  • 페이지 하단 근처 도달 시 다음 페이지 데이터가 자동 로드된다
  • 스크롤에 따라 계속해서 새로운 상품들이 목록에 추가된다
  • 새 데이터 로드 중일 때 로딩 인디케이터와 스켈레톤 UI가 표시된다
  • 홈 페이지에서만 무한 스크롤이 활성화된다

상품을 장바구니에 담기

  • 각 상품에 장바구니 추가 버튼이 있다
  • 버튼 클릭 시 해당 상품이 장바구니에 추가된다
  • 추가 완료 시 사용자에게 알림이 표시된다

상품 검색

  • 상품명 기반 검색을 위한 텍스트 입력 필드가 있다
  • Enter 키로 검색이 수행된다
  • 검색어와 일치하는 상품들만 목록에 표시된다

카테고리 선택

  • 사용 가능한 카테고리들을 선택할 수 있는 UI가 제공된다
  • 선택된 카테고리에 해당하는 상품들만 표시된다
  • 전체 상품 보기로 돌아갈 수 있다
  • 2단계 카테고리 구조를 지원한다 (1depth, 2depth)

카테고리 네비게이션

  • 현재 선택된 카테고리 경로가 브레드크럼으로 표시된다
  • 브레드크럼의 각 단계를 클릭하여 상위 카테고리로 이동할 수 있다
  • "전체" > "1depth 카테고리" > "2depth 카테고리" 형태로 표시된다

현재 상품 수 표시

  • 현재 조건에서 조회된 총 상품 수가 화면에 표시된다
  • 검색이나 필터 적용 시 상품 수가 실시간으로 업데이트된다

장바구니

장바구니 모달

  • 장바구니 아이콘 클릭 시 모달 형태로 장바구니가 열린다
  • X 버튼이나 배경 클릭으로 모달을 닫을 수 있다
  • ESC 키로 모달을 닫을 수 있다
  • 모달에서 장바구니의 모든 기능을 사용할 수 있다

장바구니 수량 조절

  • 각 장바구니 상품의 수량을 증가할 수 있다
  • 각 장바구니 상품의 수량을 감소할 수 있다
  • 수량 변경 시 총 금액이 실시간으로 업데이트된다

장바구니 삭제

  • 각 상품에 삭제 버튼이 배치되어 있다
  • 삭제 버튼 클릭 시 해당 상품이 장바구니에서 제거된다

장바구니 선택 삭제

  • 각 상품에 선택을 위한 체크박스가 제공된다
  • 선택 삭제 버튼이 있다
  • 체크된 상품들만 일괄 삭제된다

장바구니 전체 선택

  • 모든 상품을 한 번에 선택할 수 있는 마스터 체크박스가 있다
  • 전체 선택 시 모든 상품의 체크박스가 선택된다
  • 전체 해제 시 모든 상품의 체크박스가 해제된다

장바구니 비우기

  • 장바구니에 있는 모든 상품을 한 번에 삭제할 수 있다

상품 상세

상품 클릭시 상세 페이지 이동

  • 상품 목록에서 상품 이미지나 상품 정보 클릭 시 상세 페이지로 이동한다
  • URL이 /product/{productId} 형태로 변경된다
  • 상품의 자세한 정보가 전용 페이지에서 표시된다

상품 상세 페이지 기능

  • 상품 이미지, 설명, 가격 등의 상세 정보가 표시된다

상품 상세 - 장바구니 담기

  • 상품 상세 페이지에서 해당 상품을 장바구니에 추가할 수 있다
  • 페이지 내에서 수량을 선택하여 장바구니에 추가할 수 있다
  • 수량 증가/감소 버튼이 제공된다

관련 상품 기능

  • 상품 상세 페이지에서 관련 상품들이 표시된다
  • 같은 카테고리(category2)의 다른 상품들이 관련 상품으로 표시된다
  • 관련 상품 클릭 시 해당 상품의 상세 페이지로 이동한다
  • 현재 보고 있는 상품은 관련 상품에서 제외된다

상품 상세 페이지 내 네비게이션

  • 상품 상세에서 상품 목록으로 돌아가는 버튼이 제공된다
  • 브레드크럼을 통해 카테고리별 상품 목록으로 이동할 수 있다
  • SPA 방식으로 페이지 간 이동이 부드럽게 처리된다

사용자 피드백 시스템

토스트 메시지

  • 장바구니 추가 시 성공 메시지가 토스트로 표시된다
  • 장바구니 삭제, 선택 삭제, 전체 삭제 시 알림 메시지가 표시된다
  • 토스트는 3초 후 자동으로 사라진다
  • 토스트에 닫기 버튼이 제공된다
  • 토스트 타입별로 다른 스타일이 적용된다 (success, info, error)

심화과제

SPA 네비게이션 및 URL 관리

페이지 이동

  • 어플리케이션 내의 모든 페이지 이동(뒤로가기/앞으로가기를 포함)은 하여 새로고침이 발생하지 않아야 한다.

상품 목록 - URL 쿼리 반영

  • 검색어가 URL 쿼리 파라미터에 저장된다
  • 카테고리 선택이 URL 쿼리 파라미터에 저장된다
  • 상품 옵션이 URL 쿼리 파라미터에 저장된다
  • 정렬 조건이 URL 쿼리 파라미터에 저장된다
  • 조건 변경 시 URL이 자동으로 업데이트된다
  • URL을 통해 현재 검색/필터 상태를 공유할 수 있다

상품 목록 - 새로고침 시 상태 유지

  • 새로고침 후 URL 쿼리에서 검색어가 복원된다
  • 새로고침 후 URL 쿼리에서 카테고리가 복원된다
  • 새로고침 후 URL 쿼리에서 옵션 설정이 복원된다
  • 새로고침 후 URL 쿼리에서 정렬 조건이 복원된다
  • 복원된 조건에 맞는 상품 데이터가 다시 로드된다

장바구니 - 새로고침 시 데이터 유지

  • 장바구니 내용이 브라우저에 저장된다
  • 새로고침 후에도 이전 장바구니 내용이 유지된다
  • 장바구니의 선택 상태도 함께 유지된다

상품 상세 - URL에 ID 반영

  • 상품 상세 페이지 이동 시 상품 ID가 URL 경로에 포함된다 (/product/{productId})
  • URL로 직접 접근 시 해당 상품의 상세 페이지가 자동으로 로드된다

상품 상세 - 새로고침시 유지

  • 새로고침 후에도 URL의 상품 ID를 읽어서 해당 상품 상세 페이지가 유지된다

404 페이지

  • 존재하지 않는 경로 접근 시 404 에러 페이지가 표시된다
  • 홈으로 돌아가기 버튼이 제공된다

AI로 한 번 더 구현하기

  • 기존에 구현한 기능을 AI로 다시 구현한다.
  • 이 과정에서 직접 가공하는 것은 최대한 지양한다.

과제 셀프회고

기술적 성장

새로 학습한 개념

라이프사이클 관리의 중요성

  • Q&A 세션을 듣기 전까지는 "일단 돌아가게만" 만들었는데, 라이프사이클 개념을 적용하면서 각 단계의 책임을 명확히 분리하는 법을 배웠습니다.
  • beforeMount (로딩) → mount (API 호출) → render (UI) → mounted (이벤트) 흐름을 구현하면서, React같은 프레임워크가 왜 이런 구조를 가지는지 이해하게 되었습니다.

이벤트 기반 아키텍처

  • 직접 렌더링 호출 방식과 이벤트 기반 방식을 혼용하다가, 모든 렌더링을 popstate 이벤트로 통일하면서 일관된 흐름의 중요성을 깨달았습니다.

Router 구현을 통한 프레임워크 이해

  • React Router를 사용만 했지, 내부 동작을 몰랐는데, 직접 구현하면서 history.pushState, 동적 라우팅, 패턴 매칭 등의 원리를 조금이나마 이해하게 되었습니다.

기존 지식의 재발견/심화

Single Responsibility Principle (단일 책임 원칙)

  • Component가 렌더링과 API 호출을 모두 담당하던 코드를, .mount() 패턴으로 분리하면서 SRP의 실제 가치를 느꼈습니다.
  • 각 함수가 하나의 책임만 가지면, 테스트도 쉽고 나중에 수정할 때도 어디를 고쳐야 하는지 명확했습니다.

DOM 최적화

  • 처음에는 장바구니 모달을 미리 렌더링하고 CSS로 숨겼는데, "필요할 때만 생성/제거"하는 방식으로 바꾸면서 불필요한 DOM 조작을 줄이는 법을 배웠습니다.

구현 과정에서의 기술적 도전과 해결

도전 1: 무한 스크롤 상태 관리

  • 현재 페이지, 로딩 상태, 필터 조건 등을 모듈 변수로 관리하다 보니 여러 곳에서 접근하고 수정하게 되어 복잡했습니다.
  • 상태를 객체로 묶고, URL의 current 파라미터와 동기화하여 새로고침 시에도 복원 가능하게 개선했습니다.

도전 2: 이벤트 위임과 동적 컨텐츠

  • 무한 스크롤로 추가되는 상품 카드에 이벤트를 어떻게 붙일지 고민했는데, 이벤트 위임 패턴을 사용하여 #root에 한 번만 등록하고 동적 컨텐츠도 처리할 수 있게 했습니다.

도전 3: 라이프사이클 분리 후 데이터 흐름 이해

  • .mount()에서 반환한 data가 renderer.js를 거쳐 component에 전달되는 흐름이 처음엔 복잡해 보였지만, 직접 구현하면서 데이터 흐름을 명확히 이해하게 되었습니다.

자랑하고 싶은 코드

1. Router의 모듈화 (제대로 한지는 모르겠지만..)

각 기능을 별도 파일로 분리하였습니다:

src/router/
├── index.js           # 초기화
├── renderer.js        # 렌더링
├── navigation.js      # 이동
├── clickHandler.js    # 링크 클릭 처리
├── historyHandler.js  # 브라우저 히스토리
├── routeMatcher.js    # URL 매칭
└── queryParser.js     # 쿼리스트링

개선이 필요하다고 생각하는 코드

1. 무한 스크롤 상태 관리

현재 코드 (homeHandlers.js):

// 모듈 변수로 상태 관리
let currentPage = 1;
let isLoading = false;
let hasMore = true;
let currentFilters = {};
let observer = null;

개선 방향:

  • 상태를 객체로 묶거나, 별도의 Store 패턴으로 관리하면 더 깔끔할 것 같습니다.

2. 에러 처리 일관성

  • HomePage는 try-catch로 에러를 잡아서 { error } 객체를 반환하지만, DetailPage는 에러 처리가 없습니다.
  • 모든 .mount() 메서드에 일관된 에러 처리 패턴을 적용하면 좋겠습니다.

학습 효과 분석

가장 큰 배움이 있었던 부분

"작동하는 코드 ≠ 좋은 코드"

처음에는 요구사항만 충족하면 된다고 생각했는데, Q&A 세션 후 리팩토링을 하면서:

  • 라이프사이클을 명확히 하면 → 테스트하기 쉽고
  • 이벤트 기반으로 통일하면 → 디버깅하기 쉽고
  • Subscribe 패턴을 쓰면 → 확장하기 쉽다

는 것을 직접 체감했습니다. 단순히 "더 복잡한 코드"가 아니라 "더 유지보수하기 쉬운 코드"를 만드는 과정이었습니다.

추가 학습이 필요한 영역

  1. 중앙 상태 관리 패턴: Vuex나 Redux 같은 Store 패턴을 직접 구현해보고 싶습니다.
  2. 가상 DOM: 문자열 기반 렌더링의 한계를 느꼈고, 가상 DOM의 필요성을 이해했습니다.
  3. 테스트 코드: 리팩토링 후 "이제 테스트하기 쉬워졌다"고 느꼈는데, 실제로 테스트 코드를 작성해보고 싶습니다.

실무 적용 가능성

  • 라이프사이클 분리, 이벤트 기반 설계, Subscribe 패턴은 어떤 프로젝트에서도 적용 가능한 범용적인 패턴입니다.
  • 특히 "관심사 분리"와 "단일 책임 원칙"은 실무에서 가장 중요한 원칙이라고 생각합니다.

  2. 요구사항의 디테일: 무한 스크롤, 장바구니 모달, SPA 네비게이션 등 실무에서 자주 마주치는 기능들을 직접 구현해보면서 많이 배웠습니다.
  3. 바닐라 JS 강제: 프레임워크 없이 구현하면서, 프레임워크가 해결하는 문제들을 직접 마주쳐보니 프레임워크의 가치를 더 잘 이해하게 되었습니다. (React야 고마워..)

AI 활용 경험 공유하기

주도적 학습을 위한 AI 활용 전략

1. 방향 설정은 내가 직접

Q&A 세션에서 라이프사이클, 라우터 추상화, 구독 패턴 개념을 들었지만 감이 잡히지 않았습니다. 그래서 먼저 AI에게 이렇게 요청했습니다:

"과제 Q&A 세션에서 알려주신 라이프사이클 개념을 현재 내 코드에 적용하려고 합니다.
어떤 순서로 리팩토링하면 좋을지 단계별 계획을 세워주세요."

→ AI가 Phase 1(라이프사이클), Phase 2(라우터 이벤트화), Phase 3(구독 패턴)로 나눠주었고, 저는 각 단계의 목적을 이해한 후 진행했습니다.

2. 개념 이해를 위한 질문

모르는 개념이 나올 때마다 검색하고 질문했습니다:

  • "왜 mount와 render를 분리해야 하나요?"
  • "popstate 이벤트로 통일하면 어떤 이점이 있나요?"
  • "구독 패턴이 직접 호출보다 나은 이유는?"

→ 각 패턴의 의도를 이해하고 코드 작성을 시작했습니다.

3. 코드는 단계적으로 이해하며 작성

전체 코드를 통으로 달라고 하지 않고, 이렇게 요청했습니다:

"HomePage.mount() 메서드 구조를 주석으로 설명해주세요.
그 다음 제가 직접 채워볼게요."

→ 주석을 보고 제가 먼저 작성 시도 → 막히는 부분만 AI에게 힌트 요청 → 완성 후 리뷰 요청 방식으로 진행했습니다.

4. 검증은 필수

작성한 코드를 AI에게 다시 검토 요청:

"이렇게 구현했는데 제대로 한 게 맞나요? 개선할 점이 있을까요?"

→ 패턴 적용이 맞는지, 놓친 엣지 케이스는 없는지 확인했습니다.

내가 작성한 코드 vs AI가 도운 부분

내가 직접 작성:

  • 초기 기능 구현 (홈, 상세, 장바구니 모달, 무한 스크롤 등)
  • 리팩토링 방향 결정 및 단계별 계획 수립
  • 각 컴포넌트의 데이터 흐름 설계
  • 버그 발견 및 수정

AI가 도운 부분:

  • 라이프사이클 패턴의 구체적인 구현 방법 제시
  • 정규표현식을 활용한 동적 라우팅 로직
  • 구독 패턴의 unsubscribe 함수 반환 패턴
  • 코드 리뷰 및 개선점 제안

깨달은 점:

  • AI에게 "해줘"가 아니라 "어떻게 하면 좋을지 계획 세워줘" → "이 개념 설명해줘" → "내가 작성한 코드 리뷰해줘" 순으로 요청하니 훨씬 많이 배울 수 있었습니다.

리뷰 받고 싶은 내용

1. 라이프사이클 패턴 구현의 적절성

현재 renderer.js에서 구현한 라이프사이클 흐름이 적절한지 검토받고 싶습니다:

unmount (cleanup) → beforeMount (.loading) → mount (.mount) → render (component) → mounted (setupHandlers)

구체적인 질문:

  • .mount()에서 데이터를 반환하고 component에 전달하는 방식이 괜찮은지?
  • 아니면 .mount() 자체가 렌더링까지 담당하는 게 나을지?
  • 라이프사이클 단계를 더 세분화하거나 줄일 필요는 없는지?

2. Subscribe 패턴의 확장성

cartStorage.js에 구현한 Subscribe 패턴을 다른 상태(예: 필터, 검색어)에도 적용할 계획인데:

고민 사항:

  • 각 상태마다 개별 subscribe를 만드는 게 나을지?
  • 아니면 중앙 Store를 만들어서 통합 관리하는 게 나을지?
  • Store 패턴을 바닐라 JS로 구현한다면 어떤 구조가 좋을지?

3. 모듈 분리 전략

현재 src/router/ 아래 7개 파일로 분리했는데:

  • routeMatcher.js와 pathMatcher.js가 역할이 비슷해 보이는데, 통합하는 게 나을지?
  • queryParser.js를 별도 파일로 두는 게 맞는지, 아니면 navigation.js에 포함시키는 게 나을지?

기준이 궁금합니다:

  • 파일을 분리할 때 "이 정도면 분리해야 한다"는 기준이 있는지?
  • 관심사 분리와 파일 개수 간 균형을 어떻게 맞추는지?

4. 이벤트 위임의 범위

현재 homeHandlers.js에서 모든 이벤트를 #root에 위임하고 있는데:

// homeHandlers.js:149-232
container.addEventListener("click", handleEvent);
container.addEventListener("keydown", handleEvent);
container.addEventListener("change", handleEvent);

성능 관점에서:

  • 모든 이벤트를 하나의 핸들러에서 분기 처리하는 게 좋은지?
  • 아니면 기능별로(검색, 필터, 장바구니) 핸들러를 분리하는 게 나을지?

궁금합니다!

고생하셨습니다 윤지님! 테스트 잘 통과하셨고 필요한 기능 잘 구현하셨는데요! 무한스크롤로 하단 상품의 상세보기를 클릭한 뒤에 다시 뒤로 돌아오면 리스트가 초기화 되는 현상이 있더라구요!

  • 어플리케이션 내의 모든 페이지 이동(뒤로가기/앞으로가기를 포함)은 하여 새로고침이 발생하지 않아야 한다. 요게 제대로 구현이 안된 것 같아서 관련해서 꼭 체크해보시면 좋을것 같아요 ㅎㅎ
  1. 라이프사이클 패턴 구현의 적절성

현재 구현한 방식이 크게 나쁘지는 않다고 생각하는데요. .mount()에서 데이터만 반환하고 컴포넌트가 순수하게 렌더링만 담당하는 구조는 적절하게 관심사를 분리하는 것 같아요! 라이프 사이클 같은 경우에도 더 세분화를 하기 보다는, 실제 리액트의 내부 라이프 사이클이 어떻게 되었었는지 왜 그렇게 되었었는지 고민을 더 해보면 좋을것 같습니다.

  1. Subscribe 패턴의 확장성

앱의 규모가 작기 때문에 중복 구현도 괜찮을것 같아요! 라고 말씀을 드릴 수도 있는데, 학습에 목적이 있고 실제로 충분히 모듈화 해서 여러곳에서 사용할 수 있도록 모듈을 구성하는데 어려움이 없을거에요. 각 도메인별로 분리해서 상태를 관리하는 방식은 좋은 것 같고, 학습에 목적이 있으니 좀 더 범용적이게 사용할 수 있는 방식을 만들어보는 고민을 하는 것도 좋을것 같습니다.

  1. 모듈 분리 전략

pathMatcher.js는 저수준 매칭 로직(정규식 변환), routeMatcher.js는 라우트 배열을 순회하는 고수준 로직이니 분리를 하는게 나쁘지는 않아보여요. queryParser.js는 재사용 가능한 유틸 함수니 별도 파일로 두는 것이 좋아보임다. 파일 분리 기준은 "단일 책임 원칙"과 "재사용 가능성"으로 보면 좋을 것 같은데 이 부분은 앞으로 고민을 많이 하게 될 부분이니 더 깊게 공부해보면 좋을 것 같습니다!

  1. 이벤트 위임의 범위

위임 방식은 나쁘지 않은데요. 너무 중복되는 내용이 많고 번잡스럽게 코드의 양이 늘어난다고 생각된다면 페이지별로 이벤트를 각각 분리하고 그걸 상위로 다시 전달해서 등록하는 방식으로 구현한다면 좀 더 깔끔하지 않을까 싶습니다 ㅎㅎ
