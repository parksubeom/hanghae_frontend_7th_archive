기본과제

가상돔을 기반으로 렌더링하기

  • createVNode 함수를 이용하여 vNode를 만든다.
  • normalizeVNode 함수를 이용하여 vNode를 정규화한다.
  • createElement 함수를 이용하여 vNode를 실제 DOM으로 만든다.
  • 결과적으로, JSX를 실제 DOM으로 변환할 수 있도록 만들었다.

이벤트 위임

  • 노드를 생성할 때 이벤트를 직접 등록하는게 아니라 이벤트 위임 방식으로 등록해야 한다
  • 동적으로 추가된 요소에도 이벤트가 정상적으로 작동해야 한다
  • 이벤트 핸들러가 제거되면 더 이상 호출되지 않아야 한다

심화 과제

Diff 알고리즘 구현

  • 초기 렌더링이 올바르게 수행되어야 한다
  • diff 알고리즘을 통해 변경된 부분만 업데이트해야 한다
  • 새로운 요소를 추가하고 불필요한 요소를 제거해야 한다
  • 요소의 속성만 변경되었을 때 요소를 재사용해야 한다
  • 요소의 타입이 변경되었을 때 새로운 요소를 생성해야 한다

과제 셀프회고

아하! 모먼트 (A-ha! Moment)

Easy를 하더라도 제대로 이해하면서

과제 패스가 중요하지 않다는 걸 알고 있으면서도 계속 패스에 집착하게 되었습니다... 그러다보니 항상 원리를 이해하고 시작하기보다 언제나 그랬듯 돌아가게만 만들었던 것 같아요. 그래서 이번에는 Easy를 하더라도 제대로 이해하면서 해보자고 마음먹었습니다.

처음에는 감 잡기가 너무 어려워서 AI랑 같이 만들어보고, 다 지우고, 다시 작성하는 걸 반복했습니다. 코치님이 말씀하셨던 것처럼 "처음부터 잘하려 하기보다 계속 반복하며 만들어보는 게 중요하다"는 걸 실천해보려고 했습니다.

결과적으로 Hard까지는 하지 못했지만, 계속 반복하며 만들어서 완성해보고 싶다는 생각이 들었어요.

"React는 왜 key를 강조하는 걸까?"

처음에는 index 기반으로 children을 비교하는 방식으로 구현했습니다. 근데 문득 "React는 key가 index가 아니였던 것 같은데.. 그리고 key 값을 index로 하면 안되지않나?"라는 생각이 들었습니다.

고민하다가 key 기반으로 바꿔보기로 했는데... 생각보다 훨씬 복잡했습니다.

Key 기반 구현 중 만난 문제들

  1. hasKeys 검증 실수 (updateChildren.js:13-20)

    • "생활/건강" 필터 클릭하면 테스트 타임아웃이 걸렸어요
    • 원인을 찾아보니 some()을 써서 일부 children만 key가 있어도 true를 반환하고 있었어요
    • every()로 바꿔서 모든 객체 노드가 key를 가진 경우에만 key 기반 알고리즘을 사용하도록 수정했습니다
    • 일부만 key가 있거나 전부 없으면 인덱스 기반으로 fallback합니다

  2. DOM 인덱스 변화 문제 (updateChildren.js:62-72)

    • "페이지당 상품 수 변경" 테스트에서 또 타임아웃...
    • insertBefore로 DOM을 조작하는 동안 childNodes 인덱스가 실시간으로 변하는 거였어요
    • DOM 조작 전에 Map에 미리 저장해두는 방식으로 해결했습니다

이 과정에서 "단순히 React가 key를 쓰니까 나도 써야지"가 아니라, key를 사용하면 리스트가 재정렬될 때 기존 DOM 노드를 재사용할 수 있다는 걸 조금은(?) 체감했습니다.

동적 이벤트 타입, 왜 하드코딩하면 안 되는지

검색창에서 Enter를 눌렀는데 작동이 안됐습니다.. 알고 보니 const eventTypes = ["click", "input", "change", "submit"]로 하드코딩해놔서 keydown 이벤트가 빠져있었습니다. 나름.. 한눈에 파악하고자 추가했었는데 생각해보니 그럼 새로운 이벤트가 추가되면? 그 때마다 추가할꺼 아니잖아 하면서 어떻게 할까 고민하다가 Set을 써서 addEvent가 호출될 때마다 이벤트 타입을 자동으로 수집하도록 바꿨습니다

Children 삭제는 왜 역순으로? (updateChildren.js:48-53)

Uncaught TypeError: Failed to execute 'removeChild' 에러가 났어요. children을 삭제할 때 정순으로 하면 index가 틀어지고, 역순으로 업데이트하면 렌더링 순서가 뒤집혔습니다.

해결책은 작업을 3단계로 분리하는 거였어요:

  1. 공통 children 업데이트 (정순)
  2. 새로운 children 추가 (정순)
  3. 남은 children 제거 (역순!)

기술적 성장

구현 과정에서의 기술적 도전

  1. Key 기반 Diffing 구현

    • 리스트가 재정렬될 때마다 모든 노드가 재생성되는 문제가 있었어요
    • key prop으로 Map을 만들어서 기존 DOM 노드를 찾아 재사용하도록 했습니다

  2. 파일 구조 정리

    • updateElement.js가 함수를 채우고 나니 너무 길어져서 3개 파일로 분리했습니다:
      • updateElement.js: 노드 추가/삭제/교체만
      • updateChildren.js: children 배열 diffing
      • updateAttributes.js: props 비교 및 업데이트
    • 지난주 피드백에 단일 책임 원칙을 생각해보면 좋을 것 같다 하셔서 생각해보며 나눠봤는데 맞게한지..는 모르겠지만 읽기 수월해진 느낌을 받았습니다

  3. Props가 없는 경우 처리 (updateElement.js:44)

    • Cannot convert undefined or null to object 에러가 났는데
    • props를 기본값 {}으로 처리해서 해결했습니다

  4. DOM 접근 최적화 (updateElement.js:13)

    • parentElement.childNodes[index]를 반복 호출하던 것을
    • const targetNode = parentElement.childNodes[index]로 한 번만 접근하도록 캐싱했어요
    • JavaScript ↔ Browser 브릿지 호출을 최소화했습니다

  5. 이벤트 핸들러 중복 방지 (eventManager.js:41-44)

    • addEvent에 includes() 중복 체크 추가
    • 동일 핸들러 재등록 방지로 메모리 누수를 차단했어요

  6. 속성 비교 최적화 (updateAttributes.js:8-10)

    • 빈 props 객체에 대한 early return 추가
    • 불필요한 forEach 순회를 제거했습니다

코드 품질

리팩토링이 필요한 부분

  • 현재는 모든 객체 노드가 key를 가진 경우에만 key 기반 알고리즘을 쓰고, 아니면 인덱스 기반으로 fallback하는데, 더 유연한 방식이 있을지 고민이에요

학습 효과 분석

가장 큰 배움

Virtual DOM이 단순히 "가상의 DOM"이 아니라, DOM 조작을 최소화하기 위한 변경 감지 레이어라는 걸 이해했어요. 그리고 코드를 작성하는 것보다 "왜 이렇게 작동하는지"를 이해하는 게 훨씬 중요하다는 것도 알았습니다
