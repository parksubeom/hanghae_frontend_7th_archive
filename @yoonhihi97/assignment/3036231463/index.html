TailwindCSS 4.x

shadcn/ui

CVA (Class Variance Authority)

  • 선언적 variants 패턴

React Hook Form + Zod

  • 선언적 폼 검증
  • 타입 안전한 스키마
  • 최소 리렌더링 최적화

Before 패키지에서 발견한 문제점

1. Atomic Design 폴더 구조의 복잡성

  • atoms/, molecules/, organisms/ 분류로 인한 복잡한 import 경로
  • 컴포넌트 분류에 대한 모호함 (이 컴포넌트는 atom인가 molecule인가?)
  • 리팩토링 시 import 경로 수정 부담

2. 하드코딩된 스타일

  • Header: #007bff, #ffffff, #e3f2fd 등 직접 색상 코드 사용
  • FormInput: #d32f2f 같은 에러 색상 하드코딩
  • components.css: 버튼, 뱃지, 알림 등에 수십 개의 하드코딩된 색상

3. 혼합된 스타일링 방식

  • CSS 클래스 (components.css)
  • 인라인 스타일 (Header.tsx, FormInput.tsx)
  • 일관성 없는 스타일링 전략

4. 일관성 없는 컴포넌트 API

  • FormInput은 helpText, FormCheckbox는 hint 사용
  • FormInput은 width, FormSelect는 size 사용
  • 같은 개념에 다른 prop 이름 사용

5. UI 컴포넌트에 비즈니스 로직 혼재

  • Button에 entityType, action, entity 같은 도메인 prop 존재
  • Badge에 userRole, paymentStatus 등 도메인 로직 포함
  • FormInput에서 비즈니스 규칙 검증 수행

개편 과정에서 집중한 부분

1. 디자인 토큰 시스템 구축

  • CSS Variables로 색상 팔레트 정의 (Neutral, Blue, Green, Red 등)
  • Semantic 토큰 (--color-text-primary, --color-bg-primary 등)
  • Component 토큰 (--button-primary-bg, --card-bg 등)

2. 컴포넌트 구조 개선

  • Atomic Design → Flat components/ui/ 구조 (shadcn/ui 패턴)
  • 비즈니스 로직을 components/management/로 분리
  • Custom hooks로 로직 추출 (useEntityData, useEntityForm, useEntityStats)

3. 일관된 컴포넌트 API

  • CVA를 활용한 타입 안전한 variant 시스템
  • 모든 컴포넌트에 동일한 패턴 적용 (variant, size props)
  • Radix UI 기반 접근성 지원

사용한 기술 스택 경험

기술경험/느낀 점
TailwindCSS v4@theme 문법으로 디자인 토큰 정의가 간편, utility-first가 생산성 향상에 도움
CVAvariant 조합이 타입 안전하게 관리됨, 조건부 스타일링이 깔끔해짐
shadcn/ui복사 가능한 컴포넌트 철학이 커스터마이징에 유리, Radix UI 기반 접근성 내장
Storybook컴포넌트 문서화와 시각적 테스트에 효과적, autodocs가 편리

어려웠던 점과 해결 방법

  1. TailwindCSS 스타일이 적용 안 되는 문제 TailwindCSS가 우선순위가 높을 줄 알았는데 지정한 스타일이 계속 무시당했습니다. 분명 bg-(--button-primary-bg) 이렇게 지정했는데 브라우저에서 확인해보면 다른 스타일이 적용되어 있었어요. 한참 헤매다가 개발자 도구로 확인해보니 기존 components.css의 .btn-primary 같은 클래스 선택자가 TailwindCSS의 유틸리티 클래스보다 specificity가 높아서 덮어쓰고 있었습니다. 결국 레거시 CSS 파일을 제거하고 TailwindCSS로 완전히 마이그레이션하고 나서야 해결됐어요. 이 과정에서 "새로운 스타일 시스템을 도입할 때는 기존 CSS와의 충돌을 먼저 정리해야 한다"는 걸 체감했습니다. 그냥 TailwindCSS 쓰면 되겠지~ 했다가 우선순위 문제로 시간을 많이 쓴거 같습니다.

  2. 반복되는 메시지 처리 before 패키지를 보면 CRUD 작업마다 성공/에러 메시지가 하드코딩되어 있었습니다.

// before - 이런 코드가 여기저기 흩어져 있었어요
setAlertMessage(`${entityType === 'user' ? '사용자' : '게시글'}가 생성되었습니다`);
setAlertMessage(`${entityType === 'user' ? '사용자' : '게시글'}가 수정되었습니다`);

entityType만 다르고 패턴은 똑같은 코드가 계속 반복되길래, 이걸 어떻게 정리할까 고민하다가 messageFactory 유틸리티를 만들었습니다.

// after - 팩토리 패턴으로 한 곳에서 관리
toast.success(messageFactory.success('create', entityType));

새로운 entity 타입이 추가되더라도 messageFactory.ts만 수정하면 되니까 유지보수 측면에서 좋지않을까? 생각했습니다.

  1. ManagementPage 조각조각 before의 ManagementPage가 진짜 읽기 힘들었습니다. useState만 10개가 넘고, 데이터 로딩, 폼 처리, API 호출, 렌더링이 전부 한 파일에 몰려 있었어요. 인라인 스타일까지 포함해서 648줄이었는데, 어디서부터 손대야 할지 막막했습니다. 고민하다가 관심사별로 분리해보기로 했어요: useEntityData: 데이터 fetching, CRUD 로직 useEntityForm: 폼 상태, 모달 열기/닫기 관리 useEntityStats: 통계 계산 로직 EntityTable, EntityFormModal, StatsCards: UI 컴포넌트 분리하고 나니까 ManagementPage는 125줄로 줄어들었고, 각 파일이 뭘 담당하는지 이름만 봐도 알 수 있게 됐습니다. "이 로직 어디 있지?" 할 때 찾기도 조금은 쉬워진 것 같았습니다.

  2. Badge 컴포넌트의 도메인 로직 분리 before의 Badge 컴포넌트가 좀 이상했습니다. userRole, paymentStatus, priority 같은 도메인 prop을 직접 받아서 내부에서 variant를 결정하고 있었어요.

// before - UI 컴포넌트가 도메인을 알고 있음
<Badge userRole="admin" />
<Badge paymentStatus="paid" />

이러면 Badge를 다른 프로젝트에서 재사용할 수가 없잖아... 싶었습니다. 결제 시스템이 없는 프로젝트에서는 paymentStatus prop이 의미가 없으니까요. 그래서 매핑 로직을 badgeMappings.ts로 분리하고, Badge는 순수하게 variant만 받도록 바꿨습니다.

// after - UI는 UI만, 매핑은 따로
<Badge variant={USER_ROLE_BADGE_MAP[user.role].variant}>
  {USER_ROLE_BADGE_MAP[user.role].label}
</Badge>

이렇게 하니까 Badge 컴포넌트는 어떤 프로젝트에서든 재사용할 수 있게 됐어요. UI 컴포넌트는 도메인을 모르는 게 맞다는 걸 배웠습니다.

리뷰받고 싶거나 질문하고 싶은 내용

Button variant 설계에 대한 고민

지금은 primary, secondary, destructive 같은 시맨틱한 variant로 설계했는데, 실무에서는 이걸 벗어나는 디자인이 들어올 때가 많았어요. 예를 들어 primary가 파란색인데 갑자기 검정색 버튼이 필요하다거나... 이런 경우에 어떻게 해야 할지 항상 고민이었습니다:

  1. 그냥 className으로 스타일 덮어씌우기
  2. black 같은 새 variant 추가하기
  3. 아예 fill, outline, ghost 같은 형태 기반 variant + 색상을 prop으로 받는 구조로 바꾸기

2번으로 하면 variant가 점점 늘어나서 관리가 어려워지고, 1번은 디자인 시스템을 깨는 느낌이라 찝찝하고... 3번이 더 유연할 것 같긴 한데 그러면 variant의 의미가 퇴색되는 것 같기도 해요. 실무에서는 보통 어떤 방식으로 접근하는지 궁금합니다.

과제 피드백

잘했어요 윤지! 이번 과제는 레거시 코드를 현대적인 디자인 시스템으로 개편하면서 컴포넌트 설계 원칙을 체득하는 것이 목표였습니다. ManagementPage를 여러 로직과 컴포넌트로 분리를 잘 해주었네요. 수고많았습니다.

과제에서 진행하면서 윤지도 느꼈을 수도 있도 (아닐 수도 있겠지만) EntityFormModal이나 EntityTable등 UI가 비즈니스 로직에 대한 분기를 가지면 나중에 분리를 해도 계속해서 entityType에 따라 분기 처리를 해야 한다는 것을 알 수 있습니다. 그러다보면 분리를 했지만 복잡성이 더 커지는 부작용을 맞이하죠.

더군더나 <EntityFormModal entityType={entityType}>처럼 UI 컴포넌트가 도메인을 알고 있으면, 나중에 Product, Order, Comment 같은 엔티티가 추가된다면 그 모든 컴포넌트들 전부에 분기를 추가해야 하고 Props도 점점 복잡해지게 됩니다. 이번 기회로 UI가 다양한 엔티티를 직접 다루기 시작하면 확장이 어려워진다는 걸 느꼈기를 바래요.

일반적인 방향은 <UserFormModal>, <PostFormModal>처럼 도메인별로 컴포넌트를 분리하고, 공통 UI는 <FormModal>, <Table> 같은 범용 컴포넌트로 추출하는 거예요. 그러면 새 엔티티 추가할 때 기존 코드를 안 건드리고 새 컴포넌트만 만들면 되니까요. 지금 하고 있는 다음 과제에서 엔티티를 중심으로 분리하는 관점의 차이를 느껴보기를 바랍니다.

Q) Button variant 설계 - 시맨틱 variant vs 형태 기반 variant + 색상 prop?

좋은 질문입니다. 그리고 이런 류는 보통 정해진 정답은 없고 원칙만 있죠. "외울게 적고 예측하기 쉬운가?" 그래서 팀에서 정하기 나름입니다. 자주 쓰이고 모두가 암묵적으로 알고 있는 사실상의 표준(?)들은 variant로 작성합니다. 뜬금없이 예외같은 거라면 className을 쓰면 되죠. 그 뜬금없음이 너무 반복이 되어 다수가 된다면 variant로 승급이 되겠죠.

중요한 건 "팀이 합의한 규칙"이 있어야 한다는 겁니다. variant가 늘어나는 기준, className 덮어쓰기 허용 범위 같은 거요. 그게 없으면 각자 마음대로 하다가 디자인 시스템이 무너지거든요. 변수에 대한 대응을 하면서도 또 디자인팀과 긴밀하게 대화하면서 표준화 시도도 계속 해봐야겠죠. 그래서 실무가 참 복잡한 거 같아요.
