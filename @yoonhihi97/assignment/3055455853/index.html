Props Drilling의 고통을 직접 경험하다..........

basic 과제에서 거대한 컴포넌트를 분리하면서 Props Drilling의 고통을 직접 경험했습니다.

App.tsx
  └─ AdminPage (21개 props 전달)
       └─ ProductManagement (11개 props 전달)
            └─ ProductTable (7개 props 전달)
                 └─ 버튼 하나 눌러서 상품 삭제하려면...
                    onDeleteProduct → onDeleteProduct → onDeleteProduct → deleteProduct

addNotification이 어디서 왔는지 추적하려면 5단계를 거슬러 올라가야 했고, deleteProduct에 파라미터 하나 추가하면 App.tsx부터 ProductTable까지 전부 수정해야 했습니다. 컴포넌트를 분리하면서 오히려 코드가 더 복잡해지는 아이러니를 경험했죠...

왜 Zustand를 선택했나?

평소에는 Jotai를 주로 사용했는데, 이번에 평소 궁금했던 Zustand를 써보면서 두 라이브러리의 차이점을 느꼈습니다.

Jotai vs Zustand 비교:

JotaiZustand
철학Bottom-up (atom 단위)Top-down (store 단위)
상태 정의작은 atom들을 조합하나의 store에 관련 상태 모음
액션atom 외부에서 정의store 내부에 함께 정의
적합한 경우독립적인 작은 상태들연관된 상태 + 액션 묶음

이번 과제에서는 cart, products, coupons 같은 엔티티 데이터와 그에 대한 액션(추가/수정/삭제)이 함께 있는 구조라서 Zustand가 더 적합하다고 판단했습니다. 관련 상태와 액션을 하나의 store에 모아두니 **"이 데이터를 어떻게 조작하지?"**라는 질문에 바로 답할 수 있었어요.

심화과제에서 해결한 방법

Before (Props Drilling):

// App.tsx → AdminPage → ProductManagement → ProductTable
<ProductTable onDeleteProduct={onDeleteProduct} ... />

After (Zustand Store 직접 사용):

// ProductTable.tsx - 필요한 곳에서 바로 사용
const deleteProduct = useProductStore((state) => state.deleteProduct);

Props를 남길지 Store로 옮길지, 어떻게 판단했나?

이번 과제에서 가장 고민한 부분입니다. 모든 걸 전역 상태로 만들면 Props Drilling은 사라지지만, Props의 명시성이라는 장점도 함께 사라지는 것 같다 생각했습니다.

내가 세운 기준:

판단 기준Props 유지Store로 이동
사용 범위부모-자식 간 직접 전달여러 컴포넌트에서 공유
데이터 성격뷰 전환, UI 상태엔티티 데이터 (cart, products)
컴포넌트 역할리스트 아이템 (map 렌더링)독립적으로 데이터 접근 필요

실제 적용 예시:

  • Header의 isAdmin, onToggleAdmin → Props 유지 (App에서 관리하는 뷰 전환 로직)
  • ProductCard의 product → Props 유지 (리스트에서 map으로 렌더링되는 컴포넌트라 부모에서 받는 게 자연스러움)
  • cart, products, coupons → Store로 (여러 컴포넌트에서 공유하고 수정)
  • addNotification → Store로 (Props Drilling 유발하던 대표적인 케이스)

순수함수 분리: 파생 데이터는 상태가 아닌 계산으로

CartSummary에서 총 금액을 표시할 때, 처음에는 store에서 cart와 selectedCoupon을 가져와서 컴포넌트 내에서 calculateCartTotal()을 호출했습니다.

// Before: 의도가 불명확
const cart = useCartStore((state) => state.cart);
const selectedCoupon = useCartStore((state) => state.selectedCoupon);
const { totalBeforeDiscount, totalAfterDiscount } = calculateCartTotal(cart, selectedCoupon);

근데 이게 좀 어색하게 느껴져서 useCartTotals() 훅을 만들었습니다.

// After: 의도가 명확
const totals = useCartTotals();
const discountAmount = totals.totalBeforeDiscount - totals.totalAfterDiscount;

이 컴포넌트는 장바구니 총액이 필요하다는 의도가 훨씬 명확해진 것..같기도 합니다.. ㅎ

헤맸던 부분: 테스트 간 상태 공유

Zustand 스토어가 테스트 간에 상태를 공유해서 테스트가 실패했습니다... 각 스토어에 resetStore() 함수를 추가하고 beforeEach에서 초기화하는 패턴으로 해결했는데, 전역 상태 관리 라이브러리 쓸 때 테스트 격리가 중요하다는 걸 배웠습니다.

이번 과제를 통해 배운 것

  1. Props Drilling을 직접 겪어봐야 전역 상태 관리의 가치를 안다

    • 이론으로만 알던 문제를 직접 경험하니까 해결책의 필요성이 확 와닿았습니다

  2. 모든 걸 전역 상태로 만들면 안 된다

    • Props의 명시성 vs 전역 상태의 편의성, 균형이 중요하다는 걸 느꼈습니다

  3. 계산 가능한 건 상태로 만들지 않는다

    • 파생 데이터는 순수함수로 계산하고, 불필요한 상태 동기화는 피하기

앞으로 해보고 싶은 것

코치님만의 커스텀 훅 분리 기준이 있으신가요?!

컴포넌트가 "UI"와 관련된 상태를 들고 있는게 아니라면, 대체로 커스텀 훅으로 분리하는 편이랍니다 ㅎㅎ

저는 컴포넌트의 관심사는 "데이터의 조작"이 아니라고 생각해요. 컴포넌트에게는 "만들어진 데이터"를 보여주는 것이 제일 중요합니다.

데이터를 조작하는 모든 과정은 사실 컴포넌트 입장에서는 "알빠?" 인거죠 ㅋㅋ

그래서 컴포넌트 내부의 다양한 로직을 아예 통째로 Custom Hook 으로 분리하는 편입니다. 그리고 Custom Hook 에 대해 테스트를 작성하는거죠.
