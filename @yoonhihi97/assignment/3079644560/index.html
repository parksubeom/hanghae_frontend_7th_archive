과제 셀프회고

이번 과제를 통해 이전에 비해 새롭게 알게 된 점이 있다면 적어주세요.

handler props

처음에는 "FSD 계층 위반을 피하려면 handler를 props로 받아야 한다"고 생각했습니다. entities가 features를 import할 수 없으니까, handler를 상위에서 주입받으면 되지 않을까..?!

근데 과제 팁 문서를 다시 읽어보니까 Bad 예시더라구요..

// ❌ Bad - handler를 props로 받음
const ProductCard = ({ product, onAddToCart, onWishlist }) => { ... }

// ✅ Good - 도메인 데이터만 받음
const ProductCard = ({ product }) => { ... }

왜 Bad인지 생각해보니까, handler를 props로 받으면 "이 컴포넌트가 뭘 하는지"가 사용하는 쪽에서 결정됩니다. 같은 ProductCard인데 어디서는 장바구니 추가, 어디서는 위시리스트 추가... 컴포넌트의 책임이 흐려지는구나! 라는 생각이 들었습니다.

handler가 있는 컴포넌트는 애초에 entities가 아니라 features에 있어야 맞지 않을까..? features는 "사용자 행동"을 담당하니까, 자기 책임을 직접 처리하는 게 맞다 생각해서 features로 옮겼습니다.

본인이 과제를 하면서 가장 애쓰려고 노력했던 부분은 무엇인가요?

일관된 기준 세우기

"PostRow는 features로 옮겼는데, CommentItem은 왜 아직 entities에 있지?" 이런 불일관함을 발견하고 CommentItem도 features로 옮겼습니다.

내가 세운 기준:

위치기준예시
entities/ui순수하게 데이터만 표시ReactionCount, UserInfo
features/ui사용자 액션이 있는 컴포넌트PostRow, CommentItem

이 기준을 전체 코드에 일관되게 적용하려고 노력했습니다.

아직은 막연하다거나 더 고민이 필요한 부분을 적어주세요.

1. onTagClick, onEditClick은 왜 props로 남겼는가?

PostRow의 onTagClick과 CommentItem의 onEditClick은 아직 props로 받고 있습니다.

  • onTagClick: URL 파라미터 변경은 라우팅 관련이라 상위(widgets)에서 처리하는 게 맞다고 생각
  • onEditClick: Dialog 상태가 PostDetailDialog(widgets)에 있어서 상위에서 처리

근데 이것도 atom으로 만들면 내부 처리가 가능합니다. "모든 handler를 내부 처리해야 하는가?" vs "라우팅/Dialog 조합은 상위 책임인가?" 아직 명확한 답을 못 찾았습니다...

2. 같은 레이어 간 의존성

features/post/ui/PostRow가 features/user/lib/useUserModal을 import합니다. FSD 원칙상 같은 레이어 간 import는 회색 영역인데, 이게 괜찮은 건지 확신이 없습니다.

이번에 배운 내용 중을 통해 앞으로 개발에 어떻게 적용해보고 싶은지 적어주세요.

  1. "어디에 둬야 하지?"보다 "이건 어떤 책임이지?"를 먼저 생각하기

    • 폴더 위치는 책임이 결정되면 자연스럽게 따라옴

  2. handler props = 책임 위임이라는 관점

    • 내가 처리하면 내 책임, props로 받으면 위임
    • features 컴포넌트는 자기 책임을 직접 처리해야 함

  3. FSD는 규칙보다 원칙이 중요

    • 무지개색이 7개인 것처럼, 경계를 나누는 기준이 중요
    • 프로젝트마다 상황이 다르니까 원칙을 이해하고 상황에 맞게 적용하기

클린코드: 읽기 좋고 유지보수하기 좋은 코드 만들기

처음 더티코드 봤을 때 솔직히 어이쿠..싶었습니다. 기나긴 컴포넌트에 모든 게 다 들어있고, 어디서 뭘 하는지 파악하려면 전체를 다 읽어야 했어요.

근데 생각해보면... 나도 급하게 만들다 보면 이렇게 만들었던 적이 있는 것 같습니다. 그때는 "일단 동작하면 됐지" 싶었는데, 나중에 다시 보면 나도 못 읽겠더라구요..

리팩토링하면서 느낀 건, 클린코드는 "예쁜 코드"가 아니라 **"찾기 쉬운 코드"**라는 것입니다. 버그가 생겼을 때 "어디를 봐야 하지?"가 바로 떠오르는 구조, 다른 사람이 봤을 때 흐름이 이해가 가는 구조가 좋은 구조인 것 같습니다.

과제 피드백

안녕하세요 윤지님! 8주차 과제 잘 진행해주셨군요 ㅎㅎ 고생하셨습니다!!

Q1) onTagClick을 props로 받은 결정이 맞나요? URL 파라미터 변경은 라우팅 관련이라 상위에서 처리해야 한다고 생각했는데, 이것도 atom으로 만들어서 PostRow 내부에서 처리하는 게 더 좋을까요?

A1) 재활용 관점에서 봤을 대 좋은 선택이라고 생각합니다! 다만 현재 코드의 위치가 feature에 있기 때문에 PostRow 내부에서 관리해도 무방하다고 생각해요 ㅎㅎ

근데 이제 보니 어차피 내부에서 Atom에 접근중이라, 일관성있게 다같이 store에 접근해도 좋지 않을까 싶네요!

Q1) 코치님은 FSD가 좋다고 느끼실 때가 있으신가요?

사실 이번에 하면서 "FSD 좋은가...?" 싶었습니다. "이건 어디 들어가지?", "hooks 폴더명을 안 쓰는 게 좋다고..?" 하면서 계속 낯설게 느껴졌어요. entities/features 구분이 책임 분리에 도움되는 것 같으면서도, 막상 코드 짜다 보면 "이건 어디 넣지?" 하면서 헷갈리기도 했습니다. 원칙은 이해했는데 아직까지도 어색한 느낌이에요.. 코치님은 "FSD 이럴 때 진짜 좋다"고 느끼시는 순간이 있으신가요?

A1) 제가 오늘(13일 토요일) 발제를 할 때 이야기 하긴 했는데, 저는 FSD를 실무에서 사용해본 경험이 없어요. 다만 FSD가 다루는 "원칙"은 중요하게 생각합니다.

응집도를 높이고, 의존성을 단방향으로 만들고, 관심사 단위로 분리하는 등의 과정은 필요하다고 생각합니다. 다만 이에 대한 구현이 FSD에 종속될 필요는 없어요!
