[4팀 박지영] Chapter 1-1. 프론트엔드 테스트 코드 익숙해지기

HARD

과제 체크포인트

기본과제

  • 총 11개의 파일, 115개의 단위 테스트를 무사히 작성하고 통과시킨다.

질문

Q. handlersUtils에 남긴 질문에 답변해주세요. events를 전역변수로 사용하다보니 각 테스트에서 이 배열을 직접 조작하면 병렬로 실행하게 될 때 테스트 결과값에 영향을 미치게 되므로, 테스트 시작 시 동일한 초기상태를 유지 할 수 있도록 하는 설정이 필요했습니다.

  1. 초기 이벤트를 깊은 복사로 저장하여 원본 데이터 보존

  2. setupTests 에서 매 테스트 실행 시마다 초기 값 유지

    beforeEach(() => {
  vi.setSystemTime(mockDate);
  expect.hasAssertions();
  resetEvents(); // 각 테스트 전에 초기 상태로 리셋
});

  3. handlersUtils의 함수들을 통해서 evenvts 제어

    • resetEvents(), setupMockHandlerCreation(), clearEvents() 제공
    • 배열 자체를 새로 할당하지 않고 참조를 유지하여 handlers.ts의 참조가 유효하도록 함

Q. 테스트를 독립적으로 구동시키기 위해 작성했던 설정들을 소개해주세요.

  1. setupTests.ts 설정

    • beforeEach에서 resetEvents() 호출하여 매 테스트마다 초기 상태로 리셋
    • vi.useFakeTimers()로 시간 제어, vi.setSystemTime()으로 일관된 테스트 환경 제공
    • MSW 서버 설정으로 API 호출 모킹

  2. handlersUtils.ts

테스트 파일에서 직접 접근하지 않고 handlerUtils를 통해 접근하게 하고, events 가 전역으로 수정되지 않도록 함

  • resetEvents(): 초기 상태로 복원
  • setupMockHandlerCreation(): 특정 이벤트들로 초기화
  • clearEvents(): 이벤트 저장소 비우기

과제 셀프회고

기술적 성장

새로 학습한 개념:

  • MSW(Mock Service Worker)를 활용한 API 모킹방식은 그리 어렵게 구현되지 않는데도 쉽게 API 대체가 가능해 다양한 상황에서 사용이 가능해 보였습니다.

  • Vitest의 타이머 제어(vi.useFakeTimers, vi.advanceTimersByTime). 테스트는 가만히 두면 시간이 흐르지 않고, 이것도 타이머 제어가 필요한 부분이 조금 신선했습니다. 언제는 되고 언제는 안되서 시간을 많이 썼던 부분이기도 합니다.

  • React Testing Library의 비동기 테스트(waitFor, findBy*) 함수들에 대해 배웠습니다. 단순 테스트이지만 항목이 많다보니 다양하게 사용해볼 수 있었습니다.

기존 지식의 재발견/심화:

html element관련해서 다시한번 돌아보고 부족함을 느꼈엄

  • 참조와 값의 차이: 배열 참조를 유지하면서 내용만 변경하는 패턴
  • 깊은 복사의 중요성: JSON.parse(JSON.stringify())를 통한 완전한 복제
  • 테스트 독립성의 중요성: 병렬 실행 시 상태 공유 문제

구현 과정에서의 기술적 도전과 해결:

기술적 도전보다는 구현하기에 급급해서 구현을 하는 것 하나하나가 도전이었습니다..

코드 품질

특히 만족스러운 구현:

이번 과제에서 가장 만족스러웠던 부분은 makeEventHelper() 함수를 utils에 구현한 것입니다. 처음에는 각 테스트마다 이벤트 객체를 일일이 생성하면서 코드 중복이 심했는데, 헬퍼 함수를 만들어 테스트 데이터 생성을 표준화하니 코드량이 크게 줄어들고 가독성도 좋아졌습니다.

또한 handlersUtils.ts를 통해 모든 events 배열 접근을 공통화 한 부분입니다. 처음에는 테스트 파일에서 events.push(), events.length = 0 같이 직접 조작을 했는데, 이를 setupMockHandlerCreation(), clearEvents() 같은 명확한 함수로 추상화하니 테스트 코드의 의도가 훨씬 명확해졌습니다.

리팩토링이 필요한 부분:

일부 테스트에서 DOM 쿼리 로직이 반복되는 것도 개선이 필요합니다. 예를 들어 이벤트 리스트에서 특정 텍스트를 가진 요소를 찾는 로직이 여러 테스트에서 반복되는데, 이를 커스텀 헬퍼 함수로 추출하면 테스트 코드가 더 간결해질 것 같습니다.

App 컴포넌트가 여전히 복잡도가 높다는 점이 아쉽습니다. 모든 상태 관리와 렌더링 로직이 하나의 컴포넌트에 집중되어 있어 코드를 읽기가 쉽지 않았고, 테스트를 작성할 때도 전체 App을 렌더링해야 해서 무거웠습니다.

코드 설계 관련 고민과 결정:

handlerUtils가 가장 고민된 부분이고 여전히 해소되지 않는 부분입니다.

학습 효과 분석

가장 큰 배움이 있었던 부분:

msw를 직접 사용해보며 구현이 생각보다 어렵지 않아 적용해 볼 수 있을 것 같았습니다.

비동기 테스트에서의 타이밍 제어도 큰 배움이었습니다. vi.advanceTimersByTime()을 사용할 때 act()로 감싸야 하는 이유, waitFor와 findBy*의 차이점 등을 실제로 부딪히면서 배우니 훨씬 이해가 잘 되었습니다.

테스트를 위한 다양한 도구들도 처음 접해봤는데, 특히 vitest-ui로 테스트 결과를 시각적으로 확인하고, vitest-preview로 렌더링 결과를 브라우저에서 확인할 수 있다는 점이 신선했습니다. debug() 함수로 컴포넌트 상태를 디버깅하는 것도 개발 중 큰 도움이 되었습니다.

추가 학습이 필요한 영역:

MSW를 더 깊이 있게 활용하는 방법을 배우고 싶습니다. 이번 과제에서는 기본적인 CRUD 모킹만 했는데, 실무에서는 네트워크 에러 상황, 지연 시뮬레이션, 페이지네이션 등 더 복잡한 시나리오를 모킹해야 할 것 같습니다.

HTML element와 접근성 관련해서도 부족함을 많이 느꼈습니다. 테스트를 작성하면서 어떤 속성으로 요소를 찾아야 할지 매번 고민되었습니다.

실무 적용 가능성:

MSW를 활용한 API 모킹은 프론트엔드 개발 초기 단계에서 정말 유용할 것 같습니다. 백엔드 API가 완성되지 않아도 프론트엔드 개발을 진행할 수 있고, 테스트도 외부 의존성 없이 독립적으로 실행할 수 있다는 점이 좋아보였습니다.

또한 기존 코드를 수정하거나 리팩토링할 때 테스트 코드가 있으면 훨씬 안전하게 작업할 수 있겠다는 생각이 들었습니다. 아무래도 테스트하는데 시간이 많이 드는데 처음에 한번 작성해두면 이후로 지속적으로 테스트 가능해보였습니다.

과제 피드백

과제에서 좋았던 부분:

Easy에서 단순 반복적인 테스트가 많아 처음에는 갯수에 압도되었지만 단위 테스트를 작성하면서 다양한 비교방법에 대해 익힐 수 있는 시간이어서 좋았습니다.

MSW와 Vitest와 각종 툴을 사용해 볼 수 있는 이었습니다. 간단한 설정들과 직접 해보면서 테스트 적용 방향성에 대해 조금은 알게 되었습니다.

과제에서 모호하거나 애매했던 부분:

초기에 handlersUtils의 함수명(setupMockHandlerCreation, setupMockHandlerUpdating 등)과 용도가 명확하지 않아서 혼란스러웠습니다. 주석에 힌트가 있긴 했지만, 함수를 어떻게 구현해야 할지 방향을 잡기가 쉽지 않았습니다.

또한 Medium 과제를 해결하려면 Hard 과제(테스트 격리)도 함께 해결해야 했다는 점이 조금 어려웠습니다. Medium 도 완료하지 못한채로 Hard 과제가 선행되어야 한다는게 부담감으로 다가왔습니다.

리뷰 받고 싶은 내용

  1. 테스트 격리 방식의 적절성

    현재 handlersUtils에서 단순히 깊은 복사(JSON.parse(JSON.stringify()))를 통해 초기 데이터를 보존하고 있습니다. 이 방식이 적절한지, 더 나은 방법이 있는지 피드백 부탁드립니다.

    추가로, 테스트 코드를 작성할 때 재사용성을 고려하여 매직 넘버나 문자열을 상수로 추출하는 것을 선호하는데, 테스트 코드에서도 이런 패턴이 권장되는지 궁금합니다. 예를 들어:

    const expectedDate = '2025-10-21';
expect(targetBox?.textContent).toContain(expectedTitle);
expect(targetBox?.textContent).toContain(expectedDate);

    어느 방식이 더 좋은 테스트 코드인지 조언 부탁드립니다.

  2. MSW 핸들러에서의 직접 접근

    handlers.ts에서 events 배열을 직접 조작하는 것이 괜찮은지 의견 부탁드립니다. 현재는 핸들러는 직접 접근을 허용하고 테스트만 handlersUtils를 통하도록 했는데, 일관성을 위해 모든 접근을 handlersUtils로 통일하는 것이 더 나을까요?

궁금한 점

  1. MSW의 실무 활용 범위

    이번 과제에서 MSW를 테스트에 활용했는데, 예시를 찾아보니 테스트뿐 아니라 개발 환경에서도 모킹 용도로 사용되는 것 같습니다. 실무에서는 MSW를 어느 범위까지 활용하시나요? 개발 중에 백엔드 API를 기다리는 동안 MSW로 모킹해서 개발하는 것이 일반적인지, 아니면 주로 테스트에만 사용하시는지 궁금합니다. Storybook과 함께 사용하는 경우도 있나요?

  2. 테스트를 위한 extension

    테스트를 진행하면서 HTML 요소를 찾거나, 테스트를 하나씩 실행하고 기다려야 하는 등 개발 경험이 그리 좋지 않다고 느꼈습니다. 특히 전체 App을 렌더링하는 통합 테스트는 실행 시간이 오래 걸려서 피드백 루프가 느렸습니다.

    실무에서 테스트 진행 시 자주 사용하는 도구나 팁이 있을까요? 예를 들어 테스트 실행 속도를 개선하는 방법, HTML 요소를 쉽게 찾는 브라우저 확장 프로그램, 또는 테스트 디버깅을 돕는 도구 등이 있다면 소개해주시면 감사하겠습니다.

  3. 테스트 전략과 균형

    현재는 통합 테스트 위주로 작성했는데, 단위 테스트와 통합 테스트, E2E 테스트의 비율은 어느 정도가 적절한가요? 프로젝트 규모나 팀 상황에 따라 다르겠지만, 일반적으로 권장되는 테스트 피라미드가 있는지 궁금합니다. 또한 E2E 테스트는 프로젝트 어느 시점부터 도입하는 것이 좋을까요?

  4. 어느시점에 getBy, findBy 이고 await를 써야 하는지 여전히 어렵습니다.

과제를 수행하면서 느낀점

과제 시작 전 생각

처음 과제를 받았을 때는 솔직히 막막했습니다. 테스트 코드 작성 경험이 많지 않아서 어디서부터 시작해야 할지 감이 잡히지 않았습니다. 특히 App 컴포넌트 하나에 모든 로직이 집중되어 있어서 코드 흐름을 파악하는 것조차 쉽지 않았습니다. 스크롤을 한참 내려야 할 정도로 긴 코드를 보면서 "이걸 다 테스트해야 하나..." 하는 부담감이 컸습니다.

그래도 일단 실행은 시켜보자는 마음으로 vitest-ui를 켰는데, 빨간 불이 가득한 테스트 목록을 보고 더 걱정이 되었습니다. 하지만 Easy 난이도부터 하나씩 해결하다 보니 점차 감이 잡혔고, 테스트 도구들(vitest-preview, debug() 등)을 활용하면서 재미를 느끼기 시작했습니다.

진행 우선순위를 이렇게 정했습니다:

  1. Easy 난이도 단위 테스트부터 처리 (성공 경험 쌓기)
    • 커스텀 훅: useCalendarView, useSearch
    • Utils: eventUtils, fetchHolidays, notificationUtils, timeValidation
    • 테스트용 헬퍼 모듈 생성: makeEventHelper
  2. Mock 테스트 방법 학습 및 적용
  3. Medium/Hard 통합 테스트 (본격적인 도전)

과제 제출 후 생각

어려웠던 점:

렌더링 테스트가 실행하는 데 너무 오래 걸려서 답답했습니다. 코드를 수정하고 테스트가 통과하는지 확인하는 피드백 루프가 느려서, 한 번 실수하면 다시 결과를 보기까지 시간이 오래 걸렸습니다. App 전체를 항상 렌더링해야 하는 구조 때문에 테스트가 무거웠던 것 같습니다.

HTML 요소를 찾는 것도 생각보다 어려웠습니다. "이 버튼을 어떤 속성으로 찾아야 하지?", "role로 찾아야 하나, testid로 찾아야 하나?" 같은 고민을 계속했습니다. 브라우저 개발자 도구에서 보이는 요소를 테스트 코드에서 찾으려니 속성 이름이 직관적이지 않아서 시행착오가 많았습니다.

가장 힘들었던 건 테스트 간 싱크가 안 맞는 문제였습니다. 어떤 테스트는 혼자 실행하면 통과하는데 전체 테스트를 돌리면 실패하는 상황이 반복되었습니다. 처음에는 왜 그런지 몰라서 한참을 헤맸는데, 나중에 전역 상태 오염 문제라는 걸 알게 되었습니다.

성장한 부분:

테스트 작성이 단순히 "기능이 작동하는지 확인"하는 것을 넘어서, 코드의 구조와 설계를 개선하는 도구가 될 수 있다는 걸 깨달았습니다. 테스트하기 어려운 코드는 대부분 복잡도가 높거나 역할이 명확하지 않은 코드였고, 테스트를 작성하면서 자연스럽게 리팩토링 포인트를 발견하게 되었습니다.

비동기 처리와 타이머 제어에 대한 이해도가 크게 높아졌습니다. 실제로 시간이 흐르는 걸 기다리는 대신 가짜 타이머로 제어하는 방법, Promise와 async/await를 테스트하는 방법 등을 배우면서 JavaScript의 비동기 동작 원리도 더 깊이 이해하게 되었습니다.

무엇보다 코드 격리와 캡슐화의 실질적인 가치를 몸소 체험한 것이 가장 큰 수확입니다. "왜 전역 변수를 피해야 하는가", "왜 함수로 추상화해야 하는가" 같은 원칙들이 이제는 단순한 이론이 아니라 실제로 필요한 이유가 있는 실용적인 가이드라인으로 느껴집니다.

5점 만점에 4점

테스트 병렬 실행 격리 문제는 처음 접해봐서 어려웠지만, 해결 과정에서 많이 배웠습니다.

과제 피드백

지영님 수고하셨습니다.

Q. 테스트 격리 방식의 적절성 현재 handlersUtils에서 단순히 깊은 복사(JSON.parse(JSON.stringify()))를 통해 초기 데이터를 보존하고 있습니다. 이 방식이 적절한지, 더 나은 방법이 있는지 피드백 부탁드립니다. 추가로, 테스트 코드를 작성할 때 재사용성을 고려하여 매직 넘버나 문자열을 상수로 추출하는 것을 선호하는데, 테스트 코드에서도 이런 패턴이 권장되는지 궁금합니다.  예를 들어: const expectedDate = '2025-10-21'; expect(targetBox?.textContent).toContain(expectedTitle); expect(targetBox?.textContent).toContain(expectedDate);

A. 깊은 복사를 위해 JSON을 사용하는 것은 저도 종종 사용했던 방법이긴 하지만 structuredClone를 쓰는 것을 권장합니다~ 매직넘버를 변수에 담아쓰는것은 아주 강력하게 권장하고 테스트 코드에서도 충분히 의미가 있을 것 같은데요. 적어주신 expectedDate, expectedTItle은 그냥 expected만 썼을때와 큰 차이는 없어 보여요. 예를들어 selectedDate 이렇게 좀더 의미 있는 변수명이여야 효과가 있을 것 같습니다. 그런데 이미 디스크립션으로 설명이 되기때문에 expected만 쓰는것도 나쁘지 않을 것 같아요.  Q. MSW 핸들러에서의 직접 접근 handlers.ts에서 events 배열을 직접 조작하는 것이 괜찮은지 의견 부탁드립니다. 현재는 핸들러는 직접 접근을 허용하고 테스트만 handlersUtils를 통하도록 했는데, 일관성을 위해 모든 접근을 handlersUtils로 통일하는 것이 더 나을까요?

A. 이건 사용하기 나름인것 같은데요handler.ts와 handlersUtils의 목적을 지영님이 새로 정의해도 될것 같아요. 말씀하신대로 일관성 측면에서 통일하는 것도 괜찮지만 발제해주신 코치님의 의도는 조금 다릅니다 ㅎㅎ 이건 솔루션 코드를 보시면서 의도가 파악해보시면 어떨가 싶어요 근데 그건 뭐 정답이 아니고요 이 과제엔 그런 의도로 만들어진것일 뿐입니다~

Q. MSW의 실무 활용 범위 A. 넵 맞습니다. 실무에서 실제 API를 받기전에 백엔드와 API 스펙협의를 통해 미리 스펙을 파악해두고 MSW로 모킹해서 병렬 개발합니다. 모든 회사가 그런다고는 할 수 없겠지만, 저는 그렇게하는것을 권장하고 아마 많은 회사가 그럴것 같습니다.

Q. 테스트를 위한 extension A. 속도는 통합테스트에 가까워질수록 느려지고 유닛테스트에 가까워질수록 빠릅니다. 이건 어쩔수 없는 트레이드 오프같아요~ 사람 손으로 해도 비슷하니까욥. 다만 불필요한 속도 저하가 가끔 있을 수 있으니 디버깅하는 테스트 코드도 개선할 필요는 있을 것 같아요. HTML을 찾거나 테스트에 유용한도구는 아시다시피 testing-library가 있습니다 :)

Q. 테스트 전략과 균형 A. 비율은 중요하지 않은것 같습니다. 테스트 전략을 짜기 전에 중요한 것은 적절하게 어플리케이션이 목적과 역할에 맞게 레이어로 추상화되어 분리되어야 합니다. 그리고 이 레이어 별로 유닛테스트(통합) 혹은 E2E테스트 중 어떻게 테스트를 진행할지 정하는 것입니다.

Q. 어느시점에 getBy, findBy 이고 await를 써야 하는지 여전히 어렵습니다. A. getBy와 findBy 차이는, 찾아봤는대 없다면 getBy는 바로 에러를 던지고 findBy는 프로미스롤 반환해 일정 기간 기다리고 없으면 reject한다는 차이가 있습니다.
